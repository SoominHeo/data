<p><b>Ascii85</b>, also called <b>Base85</b>, is a form of <a href="/wiki/Binary-to-text_encoding" title="Binary-to-text encoding">binary-to-text encoding</a> developed by Paul E. Rutter for the <a class="new" href="/w/index.php?title=Btoa&amp;action=edit&amp;redlink=1" title="Btoa (page does not exist)">btoa</a> utility. By using five <a href="/wiki/ASCII" title="ASCII">ASCII</a> characters to represent four bytes of <a href="/wiki/Binary_data" title="Binary data">binary data</a> (making the encoded size ¹⁄₄ larger than the original, assuming eight bits per ASCII character), it is more efficient than <a class="mw-redirect" href="/wiki/Uuencode" title="Uuencode">uuencode</a> or <a href="/wiki/Base64" title="Base64">Base64</a>, which use four characters to represent three bytes of data (¹⁄₃ increase, assuming eight bits per ASCII character).</p>
<p>Its main modern use is in <a href="/wiki/Adobe_Systems" title="Adobe Systems">Adobe</a>'s <a href="/wiki/PostScript" title="PostScript">PostScript</a> and <a href="/wiki/Portable_Document_Format" title="Portable Document Format">Portable Document Format</a> file formats.</p>

<h2><span class="mw-headline" id="Basic_idea">Basic idea</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ascii85&amp;action=edit&amp;section=1" title="Edit section: Basic idea">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The basic need for a binary-to-text encoding comes from a need to communicate arbitrary <a href="/wiki/Binary_data" title="Binary data">binary data</a> over preexisting <a href="/wiki/Communications_protocol" title="Communications protocol">communications protocols</a> that were designed to carry only <a class="mw-redirect" href="/wiki/Human-readable" title="Human-readable">human-readable</a> text. Those communication protocols may only be 7-bit safe (and within that avoid certain ASCII control codes), and may require <a class="mw-redirect" href="/wiki/Line_break_(computing)" title="Line break (computing)">line breaks</a> at certain maximum intervals, and may not maintain <a class="mw-redirect" href="/wiki/Whitespace_(computer_science)" title="Whitespace (computer science)">whitespace</a>. Thus, only the 95 <a href="/wiki/ASCII#ASCII_printable_characters" title="ASCII">printable ASCII characters</a> are "safe" to use to convey data.</p>
<p>Four bytes can represent 2<sup>32</sup> = 4,294,967,296 possible values. Five <a href="/wiki/Radix" title="Radix">radix</a>-85 digits provide 85<sup>5</sup> = 4,437,053,125 possible values, enough to provide for a unique representation for each possible 32-bit value. Because five radix-84 digits only provide 84<sup>5</sup> = 4,182,119,424 representable values, 85 is the minimum possible integral base that will represent four bytes in five characters, hence its choice.</p>
<p>When encoding, each group of 4 bytes is taken as a 32-bit binary number, most significant byte first (Ascii85 uses a <a class="mw-redirect" href="/wiki/Big-endian" title="Big-endian">big-endian</a> convention). This is converted, by repeatedly dividing by 85 and taking the remainder, into 5 radix-85 digits. Then each digit (again, most significant first) is encoded as an ASCII printable character by adding 33 to it, giving the ASCII characters 33 ("<code>!</code>") through 117 ("<code>u</code>").</p>
<p>Because all-zero data is quite common, an exception is made for the sake of <a href="/wiki/Data_compression" title="Data compression">data compression</a>, and an all-zero group is encoded as a single character "<code>z</code>" instead of "<code>!!!!!</code>".</p>
<p>Groups of characters that decode to a value greater than <span class="nowrap">2<sup>32</sup> − 1</span> (encoded as "<code>s8W-!</code>") will cause a decoding error, as will "<code>z</code>" characters in the middle of a group. White space between the characters is ignored and may occur anywhere to accommodate line-length limitations.</p>
<p>One disadvantage of Ascii85 is that encoded data may contain <a href="/wiki/Escape_character" title="Escape character">escape characters</a> such as backslash and quote, which have special meaning in many programming languages and in some text-based protocols. Other base-85 encodings like Z85 are designed to be safe in source code.<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup></p>
<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ascii85&amp;action=edit&amp;section=2" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="btoa_version">btoa version</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ascii85&amp;action=edit&amp;section=3" title="Edit section: btoa version">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The original btoa program always encoded full groups (padding the source as necessary), with a prefix line of "xbtoa Begin", and suffix line of "xbtoa End", followed by the original file length (in decimal and <a href="/wiki/Hexadecimal" title="Hexadecimal">hexadecimal</a>) and three 32-bit <a href="/wiki/Checksum" title="Checksum">checksums</a>. The decoder needs to use the file length to see how much of the group was padding. The initial proposal for btoa encoding used an encoding alphabet starting at the ASCII space character through "t" inclusive, but this was replaced with an encoding alphabet of "!" to "u" to avoid "problems with some mailers (stripping off trailing blanks)."<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> This program also introduced the special "<code>z</code>" short form for an all-zero group. Version 4.2 added a "<code>y</code>" exception for a group of all ASCII <a href="/wiki/Space_(punctuation)" title="Space (punctuation)">space</a> characters (0x20202020).</p>
<h3><span class="mw-headline" id="ZMODEM_version">ZMODEM version</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ascii85&amp;action=edit&amp;section=4" title="Edit section: ZMODEM version">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>"ZMODEM Pack-7 encoding" encodes groups of 4 octets into groups of 5 printable ASCII characters, similar to Ascii85 (or perhaps exactly the same?). When <a href="/wiki/ZMODEM" title="ZMODEM">ZMODEM</a> programs send pre-compressed 8-bit data files over <a href="/wiki/8-bit_clean" title="8-bit clean">7-bit data channels</a>, it uses "ZMODEM Pack-7 encoding".<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup></p>
<h3><span class="mw-headline" id="Adobe_version">Adobe version</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ascii85&amp;action=edit&amp;section=5" title="Edit section: Adobe version">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Adobe adopted the basic btoa encoding, but with slight changes, and gave it the name Ascii85. The characters used are the ASCII characters 33 (!) through 117 (u) inclusive (to represent the base-85 digits 0 through 84), together with the letter z (as a special case to represent a 32-bit 0 value), and white space is ignored. Adobe uses the delimiter "<code>~&gt;</code>" to mark the end of an Ascii85-encoded string, and represents the length by truncating the final group: If the last block of source bytes contains fewer than 4 bytes, the block is padded with up to three null bytes before encoding. After encoding, as many bytes as were added as padding are removed from the end of the output.</p>
<p>The reverse is applied when decoding: The last block is padded to 5 bytes with the Ascii85 character "<code>u</code>", and as many bytes as were added as padding are omitted from the end of the output (see example).</p>
<p>NOTE: The padding is not arbitrary. Converting from binary to base 64 only regroups bits and does not change them or their order (a high bit in binary does not affect the low bits in the base64 representation). In converting a binary number to base85 (85 is <i>not</i> a power of two) high bits do affect the low order base85 digits and conversely. Padding the binary low (with zero bits) while encoding and padding the base85 value high (with 'u's) in decoding assures that the high order bits are preserved (the zero padding in the binary gives enough room so that a small addition is trapped and there is no "carry" to the high bits).</p>
<p><br/>
In Ascii85-encoded blocks, whitespace and line-break characters may be present anywhere, including in the middle of a 5-character block, but they must be silently ignored.</p>
<p>Adobe's specification does not support the "<code>y</code>" exception.</p>
<h3><span class="mw-headline" id="ZeroMQ_Version_.28Z85.29">ZeroMQ Version (Z85)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ascii85&amp;action=edit&amp;section=6" title="Edit section: ZeroMQ Version (Z85)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Z85, the <a href="/wiki/ZeroMQ" title="ZeroMQ">ZeroMQ</a> base-85 encoding algorithm, is a string-safe variant of base85. By avoiding the double-quote, single-quote, and backslash characters, Z85-encoded data can be better embedded in <a class="mw-redirect" href="/wiki/Command-line_interpreter" title="Command-line interpreter">command-line interpreter</a> strings. Z85 uses the characters <tt>0</tt>...<tt>9</tt>, <tt>a</tt>...<tt>z</tt>, <tt>A</tt>...<tt>Z</tt>, <tt>.</tt>, <tt>-</tt>, <tt>:</tt>, <tt>+</tt>, <tt>=</tt>, <tt>^</tt>, <tt>!</tt>, <tt>/</tt>, <tt>*</tt>, <tt>?</tt>, <tt>&amp;</tt>, <tt>&lt;</tt>, <tt>&gt;</tt>, <tt>(</tt>, <tt>)</tt>, <tt>[</tt>, <tt>]</tt>, <tt>{</tt>, <tt>}</tt>, <tt>@</tt>, <tt>%</tt>, <tt>$</tt>, <tt>#</tt>.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></p>

<h3><span class="mw-headline" id="Compatibility">Compatibility</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ascii85&amp;action=edit&amp;section=8" title="Edit section: Compatibility">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The Ascii85 encoding is compatible with 7-bit and 8-bit <a href="/wiki/MIME" title="MIME">MIME</a>, while having less overhead than <a href="/wiki/Base64" title="Base64">Base64</a>.</p>
<p>One potential compatibility issue of Ascii85 is that 'single' and "double" quotation marks, &lt;angle&gt; brackets, and ampersands (&amp;) cannot be used unescaped in markup languages like XML or SGML.</p>
<h2><span class="mw-headline" id="RFC_1924_version">RFC 1924 version</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ascii85&amp;action=edit&amp;section=9" title="Edit section: RFC 1924 version">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Published on <a href="/wiki/April_Fools%27_Day_Request_for_Comments" title="April Fools' Day Request for Comments">April 1, 1996</a>, informational <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc1924" rel="nofollow">RFC 1924</a>: "A Compact Representation of IPv6 Addresses" by <a href="/wiki/Kevin_Robert_Elz" title="Kevin Robert Elz">Robert Elz</a> suggests a base-85 encoding of <a href="/wiki/IPv6" title="IPv6">IPv6</a> addresses. This differs from the scheme used above in that he proposes a different set of 85 ASCII characters, and proposes to do all arithmetic on the 128-bit number, converting it to a single 20-digit base-85 number (internal whitespace not allowed), rather than breaking it into four 32-bit groups.</p>
<p>The proposed character set is, in order, <code>0</code>–<code>9</code>, <code>A</code>–<code>Z</code>, <code>a</code>–<code>z</code>, and then the 23 characters <code>!#$%&amp;()*+-;&lt;=&gt;?@^_`{|}~</code>. The highest possible representable address, 2<sup>128</sup>−1 = 74×85<sup>19</sup> + 53×85<sup>18</sup> + 5×85<sup>17</sup> + ..., would be encoded as <code>=r54lj&amp;NUUO~Hi%c2ym0</code>.</p>
<p>This character set excludes the characters <code>"',./:[\] </code>, making it suitable for use in <a href="/wiki/JSON" title="JSON">JSON</a> strings (where <code>"</code> and <code>\</code> would require escaping). However, for <a class="mw-redirect" href="/wiki/SGML" title="SGML">SGML</a>-based protocols, notably including <a href="/wiki/XML" title="XML">XML</a>, string escapes may still be required (to accommodate <code>&lt;</code>, <code>&gt;</code> and <code>&amp;</code>).</p>
