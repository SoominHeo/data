<h1 class="firstHeading" id="firstHeading" lang="en">Windows Driver Model</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">

<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><script>function mfTempOpenSection(id){var block=document.getElementById("mf-section-"+id);block.className+=" open-block";block.previousSibling.className+=" open-block";}</script><p>In <a href="/wiki/Computing" title="Computing">computing</a>, the <b>Windows Driver Model</b> (<b>WDM</b>) –  also known at one point as the <b>Win32 Driver Model</b> –  is a framework for <a href="/wiki/Device_driver" title="Device driver">device drivers</a> that was introduced with <a href="/wiki/Windows_98" title="Windows 98">Windows 98</a> and <a href="/wiki/Windows_2000" title="Windows 2000">Windows 2000</a> to replace <a href="/wiki/VxD" title="VxD">VxD</a>, which was used on older versions of Windows such as <a href="/wiki/Windows_95" title="Windows 95">Windows 95</a> and <a href="/wiki/Windows_3.1x" title="Windows 3.1x">Windows 3.1</a>, as well as the <a href="/wiki/Windows_NT#Driver_models" title="Windows NT">Windows NT Driver Model</a>.</p>

<h2><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Driver_Model&amp;action=edit&amp;section=1" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>WDM drivers are layered in a complex hierarchy and communicate with each other via <a class="mw-redirect" href="/wiki/I/O_request_packets" title="I/O request packets">I/O request packets</a> (IRPs). The Microsoft Windows Driver Model unified driver models for the Windows 98 and Windows 2000 product lines by standardizing requirements and reducing the amount of code that needed to be written. WDM drivers will not run on operating systems earlier than Windows 98 or Windows 2000, such as Windows 95, Windows NT 4.0 and Windows 3.1. By conforming to WDM, drivers can be <a class="mw-redirect" href="/wiki/Binary_compatible" title="Binary compatible">binary compatible</a> and <a href="/wiki/Source_code_compatibility" title="Source code compatibility">source-compatible</a> across Windows 98, Windows 98 Second Edition, <a class="mw-redirect" href="/wiki/Windows_Me" title="Windows Me">Windows Me</a>, Windows 2000, <a href="/wiki/Windows_XP" title="Windows XP">Windows XP</a>, <a href="/wiki/Windows_Server_2003" title="Windows Server 2003">Windows Server 2003</a> and <a href="/wiki/Windows_Vista" title="Windows Vista">Windows Vista</a> (for backwards compatibility) on <a href="/wiki/X86" title="X86">x86</a>-based computers. WDM drivers are designed to be <a href="/wiki/Forward_compatibility" title="Forward compatibility">forward-compatible</a> so that a WDM driver can run on a version of Windows newer than what the driver was initially written for, but doing that would mean that the driver cannot take advantage of any new features introduced with the new version. WDM is generally not <a href="/wiki/Backward_compatibility" title="Backward compatibility">backward-compatible</a>, that is, a WDM driver is not guaranteed to run on any older version of Windows. For example, <a href="/wiki/Windows_XP" title="Windows XP">Windows XP</a> can use a driver written for <a href="/wiki/Windows_2000" title="Windows 2000">Windows 2000</a> but will not make use of any of the new WDM features that were introduced in Windows XP. However, a driver written for Windows XP may or may not load on Windows 2000.</p>
<p>WDM exists in the intermediary layer of Windows 2000 <a class="mw-redirect" href="/wiki/Kernel-mode" title="Kernel-mode">kernel-mode</a> drivers and was introduced to increase the functionality and ease of writing drivers for Windows. Although WDM was mainly designed to be <a class="mw-redirect" href="/wiki/Binary_compatible" title="Binary compatible">binary</a> and <a href="/wiki/Source_code_compatibility" title="Source code compatibility">source</a> compatible between <a href="/wiki/Windows_98" title="Windows 98">Windows 98</a> and Windows 2000, this may not always be desired and so specific drivers can be developed for either operating system. WDM drivers can be classified into the following types and sub-types:</p>
<h3><span class="mw-headline" id="Device_function_drivers">Device function drivers</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Driver_Model&amp;action=edit&amp;section=2" title="Edit section: Device function drivers">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>A <b>function driver</b> is the main driver for a device. A function driver is typically written by the device vendor and is required (unless the device is being used in raw mode). A function driver can service one or more devices.</p>
<ul>
<li><b><a href="/wiki/Class_driver" title="Class driver">Class drivers</a>:</b> These are types of <i>function drivers</i> and can be thought of as built-in <i>framework</i> drivers that <a href="/wiki/Miniport" title="Miniport">miniport</a> and other <a href="/wiki/Class_driver" title="Class driver">class drivers</a> can be built on top of. The <a href="/wiki/Class_driver" title="Class driver">class drivers</a> provide interfaces between different levels of the WDM architecture. Common functionality between different classes of drivers can be written into the class driver and used by other class and <a href="/wiki/Miniport" title="Miniport">miniport</a> drivers. The lower edge of the class driver will have its interface exposed to the <a href="/wiki/Miniport" title="Miniport">miniport</a> driver, while the upper edge of top level class drivers is operating system specific. Class drivers can be dynamically loaded and unloaded as well. They can do class specific functions that are not hardware or <a href="/wiki/Bus_(computing)" title="Bus (computing)">bus</a>-specific (with the exception of bus-type class drivers) and in fact sometimes only do class specific functions like <a href="/wiki/Enumeration" title="Enumeration">enumeration</a>.</li>
<li><b><a href="/wiki/Miniport" title="Miniport">Miniport</a> drivers:</b> These are also <i>function drivers</i> for <a href="/wiki/USB" title="USB">USB</a>, <a href="/wiki/Sound_card" title="Sound card">audio</a>, <a href="/wiki/SCSI" title="SCSI">SCSI</a> and <a class="mw-redirect" href="/wiki/Network_card" title="Network card">network adapters</a>. They should usually be source and binary compatible between Windows 98 and Windows 2000 and are hardware specific but control access to the hardware through a specific bus <a href="/wiki/Class_driver" title="Class driver">class driver</a>.</li>
</ul>
<p><i>"The device class-specific driver models are typically structured as a port driver written by Microsoft paired with a miniport driver written by an independent hardware vendor. The port driver does much of the work required for the device class, and the miniport driver supports device-specific characteristics."<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup></i></p>
<h3><span class="mw-headline" id="Bus_drivers">Bus drivers</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Driver_Model&amp;action=edit&amp;section=3" title="Edit section: Bus drivers">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>A <b><a href="/wiki/Bus_(computing)" title="Bus (computing)">bus</a> driver</b> services a bus controller, adapter, or bridge. Microsoft provides bus drivers for most common buses, such as <a class="mw-redirect" href="/wiki/Peripheral_Component_Interconnect" title="Peripheral Component Interconnect">PCI</a>, <a class="mw-redirect" href="/wiki/Plug-And-Play" title="Plug-And-Play">PnPISA</a>, <a href="/wiki/SCSI" title="SCSI">SCSI</a>, <a href="/wiki/USB" title="USB">USB</a> and <a class="mw-redirect" href="/wiki/FireWire" title="FireWire">FireWire</a>. Each software vendor can create their own bus drivers if needed. A bus driver can service more than one bus if there is more than one bus of the same type on the machine.</p>
<h3><span class="mw-headline" id="Filter_drivers">Filter drivers</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Driver_Model&amp;action=edit&amp;section=4" title="Edit section: Filter drivers">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><b><a href="/wiki/Filter_driver" title="Filter driver">Filter drivers</a></b> are optional drivers that add value to or modify the behavior of a device and may be non-device drivers. A <a href="/wiki/Filter_driver" title="Filter driver">filter driver</a> can also service one or more devices. Upper level filter drivers sit above the primary driver for the device (the function driver), while lower level filter drivers sit below the function driver and above the bus driver.</p>
<ul>
<li><b><a href="/wiki/Windows_service" title="Windows service">Driver service</a>:</b> This is a type of kernel-level filter driver implemented as a <a href="/wiki/Windows_service" title="Windows service">Windows service</a> that enables applications to work with devices.</li>
</ul>
<h2><span class="mw-headline" id="VxD.2C_WDM_and_Windows_98">VxD, WDM and Windows 98</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Driver_Model&amp;action=edit&amp;section=5" title="Edit section: VxD, WDM and Windows 98">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p><a href="/wiki/Windows_98" title="Windows 98">Windows 98</a> based operating systems (<a href="/wiki/Windows_98" title="Windows 98">Windows 98</a>, <a href="/wiki/Windows_98" title="Windows 98">Windows 98 Second Edition</a>, and <a class="mw-redirect" href="/wiki/Windows_Me" title="Windows Me">Windows Me</a>) are able to use both WDM and <a href="/wiki/VxD" title="VxD">VxD</a> (Virtual device driver) driver standards. Both drivers models can provide unique and different features for the same hardware. However, usually the newer WDM standard provides more features. For example, if a <a href="/wiki/TV_tuner_card" title="TV tuner card">TV tuner card</a> using a <a href="/wiki/VxD" title="VxD">VxD</a> driver is able to capture images at a resolution of 384 x 288 pixels, the same TV Tuner card with the WDM driver model may be able to capture at a resolution of 768 x 576 pixels. This can be attributed to the new <a href="/wiki/Broadcast_Driver_Architecture" title="Broadcast Driver Architecture">Broadcast Driver Architecture</a> model which is part of WDM.</p>
<h2><span class="mw-headline" id="Criticism">Criticism</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Driver_Model&amp;action=edit&amp;section=6" title="Edit section: Criticism">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The Windows Driver Model, while a significant improvement over the <a href="/wiki/VxD" title="VxD">VxD</a> and Windows NT driver model used before it, has been criticised by driver software developers <a class="external autonumber" href="http://www.wd-3.com/archive/FrameworkIntro.htm" rel="nofollow">[1]</a>, most significantly for the following:</p>
<ul>
<li>Interactions with <a href="/wiki/Power_management" title="Power management">power management</a> events and <a href="/wiki/Plug_and_play" title="Plug and play">plug and play</a> are difficult. This leads to a variety of situations where Windows machines cannot go to <a href="/wiki/Sleep_mode" title="Sleep mode">sleep</a> or wake up correctly due to bugs in driver code.</li>
<li>I/O cancellation is almost impossible to get right.<sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a class="mw-redirect" href="/wiki/Wikipedia:Disputed_statement" title="Wikipedia:Disputed statement"><span title="The material near this tag is possibly inaccurate or nonfactual. (June 2015)">dubious</span></a> <span class="metadata">– <a href="/wiki/Talk:Windows_Driver_Model#Dubious" title="Talk:Windows Driver Model">discuss</a></span></i>]</sup></li>
<li>Thousands of lines of support code are required for every driver.<sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a class="mw-redirect" href="/wiki/Wikipedia:Disputed_statement" title="Wikipedia:Disputed statement"><span title="The material near this tag is possibly inaccurate or nonfactual. (June 2015)">dubious</span></a> <span class="metadata">– <a href="/wiki/Talk:Windows_Driver_Model#Dubious" title="Talk:Windows Driver Model">discuss</a></span></i>]</sup></li>
<li>No support for writing pure <a class="mw-redirect" href="/wiki/User-mode" title="User-mode">user-mode</a> drivers.</li>
</ul>
<p>There were also a number of concerns about the quality of documentation and samples that Microsoft provided.</p>
<p>Because of these issues, Microsoft has released a new framework incorporated on top of WDM, called the <a class="mw-redirect" href="/wiki/Windows_Driver_Foundation" title="Windows Driver Foundation">Windows Driver Foundation</a>, which includes <a href="/wiki/Kernel-Mode_Driver_Framework" title="Kernel-Mode Driver Framework">Kernel-Mode Driver Framework</a> (KMDF) and <a href="/wiki/User-Mode_Driver_Framework" title="User-Mode Driver Framework">User-Mode Driver Framework</a> (UMDF). <a href="/wiki/Windows_Vista" title="Windows Vista">Windows Vista</a> supports both pure WDM and the newer <a class="mw-redirect" href="/wiki/Windows_Driver_Foundation" title="Windows Driver Foundation">Windows Driver Foundation</a>. KMDF is also available for download for Windows XP and even Windows 2000, while UMDF is available for Windows XP and above.</p>
