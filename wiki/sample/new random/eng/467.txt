<p>A <b>COM file</b> is a type of simple <a class="mw-redirect" href="/wiki/Executable_file" title="Executable file">executable file</a>. On the <a class="mw-redirect" href="/wiki/Digital_Equipment" title="Digital Equipment">Digital Equipment</a> <a href="/wiki/Operating_system" title="Operating system">operating systems</a> of the 1970s, <code>.COM</code> was used as a <a href="/wiki/Filename_extension" title="Filename extension">filename extension</a> for <a href="/wiki/Text_file" title="Text file">text files</a> containing commands to be issued to the operating system (similar to a <a href="/wiki/Batch_file" title="Batch file">batch file</a>).<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> With the introduction of <a href="/wiki/CP/M" title="CP/M">CP/M</a> (a <a href="/wiki/Microcomputer" title="Microcomputer">microcomputer</a> operating system), the type of files commonly associated with COM extension changed to that of executable files. This convention was later carried over to <a href="/wiki/MS-DOS" title="MS-DOS">MS-DOS</a>. Even when complemented by the more general <a href="/wiki/.exe" title=".exe">.exe</a> <a href="/wiki/File_format" title="File format">file format</a> for executables, the compact COM files remain viable and frequently used in MS-DOS.</p>
<p>The <code>.COM</code> file name extension has no relation to the <a href="/wiki/.com" title=".com">.com</a> (for "commercial") top-level Internet domain name. However, this similarity in name has been exploited by malicious <a href="/wiki/Computer_virus" title="Computer virus">computer virus</a> writers.</p>

<h2><span class="mw-headline" id="MS-DOS_binary_format">MS-DOS binary format</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=COM_file&amp;action=edit&amp;section=1" title="Edit section: MS-DOS binary format">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The COM format is the original binary executable format used in <a href="/wiki/CP/M" title="CP/M">CP/M</a> and <a href="/wiki/DOS" title="DOS">MS-DOS</a>. It is very simple; it has no header (with the exception of CP/MÂ 3 files), and contains no standard <a href="/wiki/Metadata" title="Metadata">metadata</a>, only code and data. This simplicity exacts a price: the <a href="/wiki/Object_code" title="Object code">binary</a> has a maximum size of 65,280 (FF00<a href="/wiki/Hexadecimal" title="Hexadecimal">h</a>) bytes (256Â bytes short of 64Â KB) and stores all its <a href="/wiki/Code_segment" title="Code segment">code</a> and <a href="/wiki/Data_segment" title="Data segment">data</a> in one segment.</p>
<p>Since it lacks <a class="mw-redirect" href="/wiki/Relocation_(computer_science)" title="Relocation (computer science)">relocation</a> information, it is <a href="/wiki/Loader_(computing)" title="Loader (computing)">loaded</a> by the operating system at a pre-set address, at offset 0100h immediately following the <a class="mw-redirect" href="/wiki/Program_segment_prefix" title="Program segment prefix">PSP</a>, where it is executed (hence the limitation of the executable's size): the <a href="/wiki/Entry_point" title="Entry point">entry point</a> is fixed at 0100h. This was not an issue on early <a href="/wiki/8-bit" title="8-bit">8-bit</a> machines because of how the <a href="/wiki/X86_memory_segmentation" title="X86 memory segmentation">segmentation</a> model works, but it is the main reason why the format fell into disuse soon after the introduction of <a href="/wiki/16-bit" title="16-bit">16-</a> and then <a href="/wiki/32-bit" title="32-bit">32-bit</a> processors with their much larger, <a class="mw-redirect" href="/wiki/Memory_segment" title="Memory segment">segmented</a> memory.</p>
<p>In the <a href="/wiki/Intel_8080" title="Intel 8080">Intel 8080</a> CPU architecture, only 65,536 bytes of memory could be addressed (address range 0000h to FFFFh). Under CP/M, the first 256 bytes of this memory, from 0000h to 00FFh were reserved for system use by the <a href="/wiki/Zero_page" title="Zero page">zero page</a>, and any user program had to be loaded at exactly 0100h to be executed. COM files fit this model perfectly. Note that before the introduction of <a href="/wiki/MP/M" title="MP/M">MP/M</a> and <a class="mw-redirect" href="/wiki/Concurrent_CP/M" title="Concurrent CP/M">Concurrent CP/M</a>, there was no possibility of running more than one program or command at a time: the program loaded at 0100h was run, and no other.</p>
<p>Although the file format is the same in MS-DOS and CP/M, .COM files for the two operating systems are not compatible; MS-DOS COM files contain <a href="/wiki/X86" title="X86">x86</a> instructions and possibly MS-DOS <a href="/wiki/System_call" title="System call">system calls</a>, while CP/M COM files contain <a href="/wiki/Intel_8080" title="Intel 8080">8080</a> instructions (programs restricted to certain machines could also contain additional instructions for <a href="/wiki/Intel_8085" title="Intel 8085">8085</a> or <a href="/wiki/Zilog_Z80" title="Zilog Z80">Z80</a>) and CP/M system calls.</p>
<p>It is possible to make a .COM file which will <a href="/wiki/Fat_binary#Combined_COM-style_binaries_for_CP.2FM-80_and_DOS" title="Fat binary">run under both operating systems</a>. There is no true compatibility at the instruction level; the instructions at the <a href="/wiki/Entry_point" title="Entry point">entry point</a> are chosen to be equal in functionality but different in both operating systems, and make program execution jump to the section for the operating system in use. It is basically two different programs with the same functionality in a single file, preceded by code selecting the one to use.</p>
<p>Under CP/MÂ 3, if the first byte of a COM file is C9h, there is a 256-byte header; since C9h corresponds to the <a href="/wiki/Intel_8080" title="Intel 8080">8080</a> instruction <code>RET</code>, this means that the COM file will immediately terminate if run on an earlier version of CP/M that does not support this extension. (Because the instruction sets of the 8085 and Z80 are supersets of the 8080 instruction set, this works on all three processors.) C9h is an <a class="mw-redirect" href="/wiki/Invalid_opcode" title="Invalid opcode">invalid opcode</a> on the 8088/8086, and it will cause an <a class="new" href="/w/index.php?title=INT_6&amp;action=edit&amp;redlink=1" title="INT 6 (page does not exist)">INTÂ 6</a> exception in <a class="mw-redirect" href="/wiki/V86_mode" title="V86 mode">v86 mode</a> since the <a href="/wiki/Intel_80386" title="Intel 80386">386</a>. Since C9h is the opcode for LEAVE since the <a href="/wiki/Intel_80188" title="Intel 80188">80188</a>/<a href="/wiki/Intel_80186" title="Intel 80186">80186</a> and therefore not used as the first instruction in a valid program, the executable loader in some versions of DOS rejects COM files that start with C9h, avoiding a crash.</p>
<p>Files may have names ending in .COM, but not be in the simple format described above; this is indicated by a <a href="/wiki/Magic_number_(programming)" title="Magic number (programming)">magic number</a> at the start of the file. For example, the <a href="/wiki/COMMAND.COM" title="COMMAND.COM">COMMAND.COM</a> file in <a class="mw-redirect" href="/wiki/DR_DOS_6.0" title="DR DOS 6.0">DRÂ DOSÂ 6.0</a> is actually in <a class="mw-redirect" href="/wiki/DOS_executable" title="DOS executable">DOS executable</a> format, indicated by the first two bytes being <i>MZ</i> (4Dh 5Ah), the initials of <a href="/wiki/Mark_Zbikowski" title="Mark Zbikowski">Mark Zbikowski</a>.</p>
<h2><span class="mw-headline" id="Large_programs">Large programs</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=COM_file&amp;action=edit&amp;section=2" title="Edit section: Large programs">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In <a href="/wiki/MS-DOS" title="MS-DOS">MS-DOS</a> and compatible <a href="/wiki/DOS" title="DOS">DOSes</a>, there is no <a href="/wiki/Memory_management" title="Memory management">memory management</a> provided for COM files by the <a href="/wiki/Loader_(computing)" title="Loader (computing)">loader</a> or execution environment. All memory is simply available to the COM file. After execution, the operating system command shell, <a href="/wiki/COMMAND.COM" title="COMMAND.COM">COMMAND.COM</a>, is reloaded. This leaves the possibilities that the COM file can either be very simple, using a single <a href="/wiki/Memory_segmentation" title="Memory segmentation">segment</a>, or arbitrarily complex, providing its own memory management system. An example of a complex program is COMMAND.COM, the MS-DOS shell, which provided a loader to load other COM or <a class="mw-redirect" href="/wiki/EXE" title="EXE">EXE</a> programs. In the .COM system, larger programs (up to the available memory size) can be loaded and run, but the system loader assumes that all code and data is in the first segment, and it is up to the .COM program to provide any further organization. Programs larger than available memory, or large <a href="/wiki/Data_segment" title="Data segment">data segments</a>, can be handled by <a class="mw-redirect" href="/wiki/Dynamic_linking" title="Dynamic linking">dynamic linking</a>, if the necessary code is included in the .COM program. The advantage of using the .COM rather than .EXE format is that the binary image is usually smaller and easier to program using an <a href="/wiki/Assembly_language#Assembler" title="Assembly language">assembler</a>.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> Once <a href="/wiki/Compiler" title="Compiler">compilers</a> and <a href="/wiki/Linker_(computing)" title="Linker (computing)">linkers</a> of sufficient power became available, it was no longer advantageous to use the .COM format for complex programs.</p>
<h2><span class="mw-headline" id="Platform_support">Platform support</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=COM_file&amp;action=edit&amp;section=3" title="Edit section: Platform support">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The format is still <a href="/wiki/Executable" title="Executable">executable</a> on many modern <a href="/wiki/Windows_NT" title="Windows NT">Windows NT</a>-based <a class="mw-redirect" href="/wiki/Platform_(computing)" title="Platform (computing)">platforms</a>, but it is run in an MS-DOS-emulating subsystem, <a href="/wiki/Virtual_DOS_machine" title="Virtual DOS machine">NTVDM</a>, which is not present in <a class="mw-redirect" href="/wiki/64-bit" title="64-bit">64-bit</a> variants. COM files can be executed also on DOS emulators such as <a href="/wiki/DOSBox" title="DOSBox">DOSBox</a>, on any platform supported by these emulators.</p>
<h2><span class="mw-headline" id="Use_for_compatibility_reasons">Use for compatibility reasons</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=COM_file&amp;action=edit&amp;section=4" title="Edit section: Use for compatibility reasons">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p><a href="/wiki/Windows_NT" title="Windows NT">Windows NT</a>-based operating systems use the .com extension for a small number of commands carried over from <a href="/wiki/MS-DOS" title="MS-DOS">MS-DOS</a> days although they are in fact presently implemented as <a href="/wiki/.exe" title=".exe">.exe</a> files. The operating system will recognize the .exe file header and execute them correctly despite their technically incorrect .com extension. (In fact any .exe file can be renamed .com and still execute correctly.) The use of the original .com extensions for these commands ensures compatibility with older MS-DOS batch files that may refer to them with their full original filenames. These commands are <code>chcp.com</code>, <code>discomp.com</code>, <code>diskcopy.com</code>, <code>format.com</code>, <code>mode.com</code>, <code>more.com</code> and <code>tree.com</code>.</p>
<h2><span class="mw-headline" id="Execution_preference">Execution preference</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=COM_file&amp;action=edit&amp;section=5" title="Edit section: Execution preference">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In MS-DOS, if a directory contains both a COM file and an <a class="mw-redirect" href="/wiki/EXE" title="EXE">EXE</a> file with same name, when no extension is specified the COM file is preferentially selected for execution. For example, if a directory in the <a class="mw-redirect" href="/wiki/Path_(variable)" title="Path (variable)">system path</a> contains two files named <code>foo.com</code> and <code>foo.exe</code>, the following would execute <code>foo.com</code>:</p>
<pre>
C:\&gt;foo
</pre>
<p>A user wishing to run <code>foo.exe</code> can explicitly use the complete filename:</p>
<pre>
C:\&gt;foo.exe
</pre>
<p>Taking advantage of this default behaviour, <a href="/wiki/Computer_virus" title="Computer virus">virus</a> writers and other malicious programmers have used names like <code>notepad.com</code> for their creations, hoping that if it is placed in the same directory as the corresponding EXE file, a command or batch file may accidentally trigger their program instead of the text editor <a class="mw-redirect" href="/wiki/Notepad_(Windows)" title="Notepad (Windows)"><code>notepad.exe</code></a>. Again, note that these .com file may in fact contain a .exe format executable.</p>
<p>On <a href="/wiki/Windows_NT" title="Windows NT">Windows NT</a> and derivatives (<a href="/wiki/Windows_2000" title="Windows 2000">Windows 2000</a>, <a href="/wiki/Windows_XP" title="Windows XP">Windows XP</a>, <a href="/wiki/Windows_Vista" title="Windows Vista">Windows Vista</a>, and <a href="/wiki/Windows_7" title="Windows 7">Windows 7</a>), the <tt>PATHEXT</tt> variable is used to override the order of preference (and acceptable extensions) for calling files without specifying the extension from the command line. The default value still places <code>.com</code> files before <code>.exe</code> files. This closely resembles a feature previously found in JP Software's line of extended command line processors <a href="/wiki/4DOS" title="4DOS">4DOS</a>, <a href="/wiki/4OS2" title="4OS2">4OS2</a>, and <a class="mw-redirect" href="/wiki/4NT" title="4NT">4NT</a>.</p>
<h2><span class="mw-headline" id="Malicious_usage_of_the_.com_extension">Malicious usage of the .com extension</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=COM_file&amp;action=edit&amp;section=6" title="Edit section: Malicious usage of the .com extension">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Some computer virus writers have hoped to take advantage of modern computer users' likely lack of knowledge of the <tt>.com</tt> file extension and associated binary format, along with their more likely familiarity with the <a href="/wiki/.com" title=".com">.com</a> Internet domain name. E-mail has been sent with attachment names similar to "www.example.com". Unwary <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a> users clicking on such an attachment would expect to begin browsing a site named <code>http://www.example.com/</code>, but instead would run the attached binary command file named <code>www.example</code>, giving it full permission to do to their machine whatever its author had in mind.</p>
<p>Note that there is nothing malicious about the COM file format itself; this is an exploitation of the coincidental name collision between .com <i>com</i>mand files and .com <i>com</i>mercial web sites.</p>
