<p><b>XSD (XML Schema Definition)</b>, a recommendation of the World Wide Web Consortium (<a class="mw-redirect" href="/wiki/W3C" title="W3C">W3C</a>), specifies how to formally describe the elements in an Extensible Markup Language (<a href="/wiki/XML" title="XML">XML</a>) document. It can be used by programmers to verify each piece of item content in a document. They can check if it adheres to the description of the element it is placed in.<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup></p>
<p>Like all XML schema languages, XSD can be used to express a set of rules to which an XML document must conform in order to be considered "valid" according to that schema. However, unlike most other schema languages, XSD was also designed with the intent that determination of a document's validity would produce a collection of information adhering to specific <a href="/wiki/Data_type" title="Data type">data types</a>. Such a post-validation <i><a href="/wiki/XML_Information_Set" title="XML Information Set">infoset</a></i> can be useful in the development of XML document processing software.</p>

<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=XML_Schema_(W3C)&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p><b>XML Schema</b>, published as a <a class="mw-redirect" href="/wiki/W3C_recommendation" title="W3C recommendation">W3C recommendation</a> in May 2001,<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> is one of several <a href="/wiki/XML_schema" title="XML schema">XML schema languages</a>. It was the first separate schema language for <a href="/wiki/XML" title="XML">XML</a> to achieve Recommendation status by the W3C. Because of confusion between XML Schema as a specific W3C specification, and the use of the same term to describe schema languages in general, some parts of the user community referred to this language as <b>WXS</b>, an initialism for W3C XML Schema, while others referred to it as <b>XSD</b>, an initialism for XML Schema Definition.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup><sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup> In Version 1.1 the W3C has chosen to adopt XSD as the preferred name, and that is the name used in this article.</p>
<p>In its appendix of references, the XSD specification acknowledges the influence of <a class="mw-redirect" href="/wiki/Document_Type_Definition" title="Document Type Definition">DTDs</a> and other early XML schema efforts such as <a class="mw-redirect" href="/wiki/DDML" title="DDML">DDML</a>, <a href="/wiki/Schema_for_Object-Oriented_XML" title="Schema for Object-Oriented XML">SOX</a>, XML-Data, and <a href="/wiki/XDR_Schema" title="XDR Schema">XDR</a>. It has adopted features from each of these proposals but is also a compromise among them. Of those languages, XDR and SOX continued to be used and supported for a while after XML Schema was published. A number of <a href="/wiki/Microsoft" title="Microsoft">Microsoft</a> products supported XDR until the release of <a href="/wiki/MSXML" title="MSXML">MSXML</a> 6.0 (which dropped XDR in favor of XML Schema) in December 2006.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> <a href="/wiki/Commerce_One" title="Commerce One">Commerce One</a>, Inc. supported its SOX schema language until declaring bankruptcy in late 2004.</p>
<p>The most obvious features offered in XSD that are not available in XML's native <a class="mw-redirect" href="/wiki/Document_Type_Definition" title="Document Type Definition">Document Type Definitions</a> (DTDs) are <a class="mw-redirect" href="/wiki/XML_Namespace" title="XML Namespace">namespace</a> awareness and datatypes, that is, the ability to define element and attribute content as containing values such as integers and dates rather than arbitrary text.</p>
<p>The XSD 1.0 specification was originally published in 2001, with a second edition following in 2004 to correct large numbers of errors. XSD 1.1 became a <a href="/wiki/World_Wide_Web_Consortium#W3C_Recommendation_.28REC.29" title="World Wide Web Consortium">W3C Recommendation</a> in <a class="external text" href="http://www.w3.org/News/2012#entry-9412" rel="nofollow">April 2012</a>.</p>
<h2><span class="mw-headline" id="Schemas_and_schema_documents">Schemas and schema documents</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=XML_Schema_(W3C)&amp;action=edit&amp;section=2" title="Edit section: Schemas and schema documents">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Technically, a <b>schema</b> is an abstract collection of metadata, consisting of a set of <b>schema components</b>: chiefly element and attribute declarations and complex and simple type definitions. These components are usually created by processing a collection of <b>schema documents</b>, which contain the source language definitions of these components. In popular usage, however, a schema document is often referred to as a schema.</p>
<p>Schema documents are organized by namespace: all the named schema components belong to a target namespace, and the target namespace is a property of the schema document as a whole. A schema document may <i>include</i> other schema documents for the same namespace, and may <i>import</i> schema documents for a different namespace.</p>
<p>When an instance document is validated against a schema (a process known as <i>assessment</i>), the schema to be used for validation can either be supplied as a parameter to the validation engine, or it can be referenced directly from the instance document using two special attributes, <code>xsi:schemaLocation</code> and <code>xsi:noNamespaceSchemaLocation</code>. (The latter mechanism requires the client invoking validation to trust the document sufficiently to know that it is being validated against the correct schema. "xsi" is the conventional prefix for the namespace "<a class="external free" href="http://www.w3.org/2001/XMLSchema-instance" rel="nofollow">http://www.w3.org/2001/XMLSchema-instance</a>".)</p>
<p>XML Schema Documents usually have the filename extension ".xsd". A unique <a class="mw-redirect" href="/wiki/Internet_media_type" title="Internet media type">Internet Media Type</a> is not yet registered for XSDs, so "application/xml" or "text/xml" should be used, as per <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc3023" rel="nofollow">RFC 3023</a>.</p>
<h2><span class="mw-headline" id="Schema_components">Schema components</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=XML_Schema_(W3C)&amp;action=edit&amp;section=3" title="Edit section: Schema components">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The main components of a schema are:</p>
<ul>
<li><b>Element declarations</b>, which define properties of elements. These include the element name and target namespace. An important property is the type of the element, which constrains what attributes and children the element can have. In XSD 1.1, the type of the element may be conditional on the values of its attributes. An element may belong to a substitution group; if element E is in the substitution group of element H, then wherever the schema permits H to appear, E may appear in its place. Elements may have integrity constraints: uniqueness constraints determining that particular values must be unique within the subtree rooted at an element, and referential constraints determining that values must match the identifier of some other element. Element declarations may be global or local, allowing the same name to be used for unrelated elements in different parts of an instance document.</li>
<li><b>Attribute declarations</b>, which define properties of attributes. Again the properties include the attribute name and target namespace. The attribute type constrains the values that the attribute may take. An attribute declaration may also include a default value or a fixed value (which is then the only value the attribute may take.)</li>
<li><b>Simple and complex types</b>. These are described in the following section.</li>
<li><b>Model group</b> and <b>attribute group</b> definitions. These are essentially macros: named groups of elements and attributes that can be reused in many different type definitions.</li>
<li>An <b>attribute use</b> represents the relationship of a complex type and an attribute declaration, and indicates whether the attribute is mandatory or optional when it is used in that type.</li>
<li>An <b>element particle</b> similarly represents the relationship of a complex type and an element declaration, and indicates the minimum and maximum number of times the element may appear in the content. As well as element particles, content models can include <b>model group</b> particles, which act like non-terminals in a grammar: they define the choice and repetition units within the sequence of permitted elements. In addition, <b>wildcard</b> particles are allowed, which permit a set of different elements (perhaps any element provided it is in a certain namespace).</li>
</ul>
<p>Other more specialized components include annotations, assertions, notations, and the <b>schema component</b> which contains information about the schema as a whole.</p>
<h2><span class="mw-headline" id="Types">Types</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=XML_Schema_(W3C)&amp;action=edit&amp;section=4" title="Edit section: Types">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Complex types describe the permitted content of an element, including its element and text children and its attributes. A complex type definition consists of a set of attribute uses and a content model. Varieties of content model include element-only content, in which no text may appear (other than whitespace, or text enclosed by a child element); simple content, in which text is allowed but child elements are not; empty content, in which neither text nor child elements are allowed; and mixed content, which permits both elements and text to appear. A complex type can be derived from another complex type by restriction (disallowing some elements, attributes, or values that the base type permits) or by extension (allowing additional attributes and elements to appear). In XSD 1.1, a complex type may be constrained by assertions — <a href="/wiki/XPath_2.0" title="XPath 2.0">XPath 2.0</a> expressions evaluated against the content that must evaluate to true.</p>
<p>Simple types (also called data types) constrain the textual values that may appear in an element or attribute. This is one of the more significant ways in which XML Schema differs from DTDs. For example, an attribute might be constrained to hold only a valid date or a decimal number.</p>
<p>XSD provides a set of 19 <a href="/wiki/Primitive_data_type" title="Primitive data type">primitive data types</a> (<code>anyURI</code>, <code>base64Binary</code>, <code>boolean</code>, <code>date</code>, <code>dateTime</code>, <code>decimal</code>, <code>double</code>, <code>duration</code>, <code>float</code>, <code>hexBinary</code>, <code>gDay</code>, <code>gMonth</code>, <code>gMonthDay</code>, <code>gYear</code>, <code>gYearMonth</code>, <code>NOTATION</code>, <code>QName</code>, <code>string</code>, and <code>time</code>). It allows new data types to be constructed from these primitives by three mechanisms:</p>
<ul>
<li>restriction (reducing the set of permitted values),</li>
<li>list (allowing a sequence of values), and</li>
<li>union (allowing a choice of values from several types).</li>
</ul>
<p>Twenty-five derived types are defined within the specification itself, and further derived types can be defined by users in their own schemas.</p>
<p>The mechanisms available for restricting data types include the ability to specify minimum and maximum values, regular expressions, constraints on the length of strings, and constraints on the number of digits in decimal values. XSD 1.1 again adds assertions, the ability to specify an arbitrary constraint by means of an <a href="/wiki/XPath_2.0" title="XPath 2.0">XPath 2.0</a> expression.</p>
<h2><span class="mw-headline" id="Post-Schema-Validation_Infoset">Post-Schema-Validation Infoset</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=XML_Schema_(W3C)&amp;action=edit&amp;section=5" title="Edit section: Post-Schema-Validation Infoset">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>After XML Schema-based validation, it is possible to express an XML document's structure and content in terms of the <a href="/wiki/Data_model" title="Data model">data model</a> that was implicit during validation. The XML Schema data model includes:</p>
<ul>
<li>The vocabulary (element and attribute names)</li>
<li>The content model (relationships and structure)</li>
<li>The data types</li>
</ul>
<p>This collection of information is called the Post-Schema-Validation Infoset (PSVI). The PSVI gives a valid XML document its "type" and facilitates treating the document as an object, using <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a> (OOP) paradigms.</p>

<h2><span class="mw-headline" id="Secondary_uses_for_XML_Schemas">Secondary uses for XML Schemas</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=XML_Schema_(W3C)&amp;action=edit&amp;section=6" title="Edit section: Secondary uses for XML Schemas">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The primary reason for defining an XML schema is to formally describe an XML document; however the resulting schema has a number of other uses that go beyond simple validation.</p>
<h3><span class="mw-headline" id="Code_generation">Code generation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=XML_Schema_(W3C)&amp;action=edit&amp;section=7" title="Edit section: Code generation">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The schema can be used to generate code, referred to as <a class="mw-redirect" href="/wiki/XML_Data_Binding" title="XML Data Binding">XML Data Binding</a>. This code allows contents of XML documents to be treated as objects within the programming environment.</p>
<h3><span class="mw-headline" id="Generation_of_XML_file_structure_documentation">Generation of XML file structure documentation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=XML_Schema_(W3C)&amp;action=edit&amp;section=8" title="Edit section: Generation of XML file structure documentation">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The schema can be used to generate human-readable documentation of an XML file structure; this is especially useful where the authors have made use of the annotation elements. No formal standard exists for documentation generation, but a number of tools are available, such as the <a href="/wiki/Xs3p" title="Xs3p">Xs3p</a> stylesheet, that will produce high quality readable HTML and printed material.</p>
<h2><span class="mw-headline" id="Criticism">Criticism</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=XML_Schema_(W3C)&amp;action=edit&amp;section=9" title="Edit section: Criticism">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Although XML Schema is successful in that it has been widely adopted and largely achieves what it set out to, it has been the subject of a great deal of severe criticism, perhaps more so than any other W3C Recommendation. Good summaries of the criticisms are provided by James Clark,<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup> Anders Møller and Michael Schwartzbach,<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup> Rick Jelliffe<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> and David Webber.<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup></p>
<p>General problems:</p>
<ul>
<li>It is too complicated (the spec is several hundred pages in a very technical language), so it is hard to use by non-experts – but many non-experts need schemas to describe data formats. The W3C Recommendation itself is extremely difficult to read. Most users find <a class="external text" href="http://www.w3.org/TR/xmlschema-0/" rel="nofollow">W3Cs XML Schema Primer</a> much easier to understand.</li>
<li>XSD lacks any formal mathematical specification. (This makes it difficult to reason about schemas, for example to prove that a modification to a schema is backwards compatible)</li>
<li>There are many surprises in the language, for example that restriction of elements works differently from restriction of attributes.</li>
</ul>
<p>Practical limitations of expressibility:</p>
<ul>
<li>XSD offers very weak support for unordered content.</li>
<li>XSD cannot require a specific <i>root element</i> (so extra information is required to validate even the simplest documents).</li>
<li>When describing <i>mixed content</i>, the character data cannot be constrained in any way (not even a set of valid characters can be specified).</li>
<li><i>Content and attribute declarations cannot depend on attributes or element context</i> (this was also listed as a central problem of DTD).</li>
<li>It is <i>not 100% self-describing</i> (as a trivial example, see the previous point), even though that was an initial design requirement.</li>
<li><i>Defaults</i> cannot be specified separately from the declarations (this makes it hard to make families of schemas that only differ in the default values); <i>element defaults</i> can only be character data (not containing markup).</li>
</ul>
<p>Technical problems:</p>
<ul>
<li>Although it technically is namespace-conformant, it does not seem to follow the namespace spirit (e.g. "unqualified locals").</li>
<li>XSD 1.0 provided no facilities to state that the value or presence of one attribute is dependent on the values or presence of other attributes (so-called <i>co-occurrence constraints</i>). This has been fixed in XSD 1.1.</li>
<li>The set of XSD datatypes on offer is highly arbitrary.<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup></li>
</ul>
<ul>
<li>The two tasks of validation and augmentation (adding type information and default values) should be kept separate.</li>
</ul>
<h2><span class="mw-headline" id="Version_1.1">Version 1.1</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=XML_Schema_(W3C)&amp;action=edit&amp;section=10" title="Edit section: Version 1.1">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>XSD 1.1 became a <a href="/wiki/World_Wide_Web_Consortium#W3C_Recommendation_.28REC.29" title="World Wide Web Consortium">W3C Recommendation</a> in <a class="external text" href="http://www.w3.org/News/2012#entry-9412" rel="nofollow">April 2012</a>, which means it is an approved W3C specification.</p>
<p>Significant new features in XSD 1.1 are:</p>
<ul>
<li>The ability to define assertions against the document content by means of <a href="/wiki/XPath_2.0" title="XPath 2.0">XPath 2.0</a> expressions (an idea borrowed from <a href="/wiki/Schematron" title="Schematron">Schematron</a>).</li>
<li>The ability to select the type against which an element will be validated based on the values of the element's attributes ("conditional type assignment").</li>
<li>Relaxing the rules whereby explicit elements in a content model must not match wildcards also allowed by the model.</li>
<li>The ability to specify wildcards (for both elements and attributes) that apply to all types in the schema, so that they all implement the same extensibility policy.</li>
</ul>
<p>Until the Proposed Recommendation draft, XSD 1.1 also proposed the addition of a new numeric data type, precisionDecimal. This proved controversial, and was therefore dropped from the specification at a late stage of development.</p>
