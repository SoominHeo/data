<p><b>Ascii85</b> 또는 <b>Base85</b> 불리는 이것은 Paul E. Rutter가 유틸리티 <a class="extiw" href="https://en.wikipedia.org/wiki/btoa" title="en:btoa">en:btoa</a>를 위해 만든 <a class="extiw" href="https://en.wikipedia.org/wiki/binary-to-text_encoding" title="en:binary-to-text encoding">en:binary-to-text encoding</a> 방법으로 5개의 <a class="mw-redirect mw-disambig" href="/wiki/ASCII" title="ASCII">ASCII</a> 문자로 4 바이트의 <a href="/wiki/%EC%9D%B4%EC%A7%84_%EB%8D%B0%EC%9D%B4%ED%84%B0" title="이진 데이터">이진 데이터</a>를 표현한다. ASCII 한 문자가 8비트라면 이 방법으로 하면 원본이 ¹⁄₄만큼 더 커진다. 이것은 4개의 문자로 3 바이트를 표현해서 ¹⁄₃만큼 더 커지는 <a class="extiw" href="https://en.wikipedia.org/wiki/uuencode" title="en:uuencode">en:uuencode</a>나 <a href="/wiki/%EB%B2%A0%EC%9D%B4%EC%8A%A464" title="베이스64">베이스64</a>보다 효율적이다.</p>
<p>이것은 <a href="/wiki/%EC%96%B4%EB%8F%84%EB%B9%84_%EC%8B%9C%EC%8A%A4%ED%85%9C%EC%A6%88" title="어도비 시스템즈">Adobe</a>의 <a href="/wiki/%ED%8F%AC%EC%8A%A4%ED%8A%B8%EC%8A%A4%ED%81%AC%EB%A6%BD%ED%8A%B8" title="포스트스크립트">포스트스크립트</a>와 <a href="/wiki/PDF" title="PDF">PDF</a> 파일 형식에 사용된다.</p>

<h2><span class="mw-headline" id=".EA.B8.B0.EB.B3.B8_.EA.B5.AC.EC.83.81">기본 구상</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ascii85&amp;action=edit&amp;section=1" title="부분 편집: 기본 구상">편집</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>binary-to-text encoding은 사람이 읽을 수 있는(<a class="extiw" href="https://en.wikipedia.org/wiki/human-readable" title="en:human-readable">en:human-readable</a>) 문자만을 전송할 수 있는 기존 통신 프로토콜 위에서 <a href="/wiki/%EC%9D%B4%EC%A7%84_%EB%8D%B0%EC%9D%B4%ED%84%B0" title="이진 데이터">이진 데이터</a>를 다루기 위해서 필요했다. 이런 환경에서 문자는 7 비트만을 사용하고 그 마저도 일부 제어 문자를 빼야하므로 데이터의 전송에 사용할 수 있는 문자는 95개 뿐이다.</p>
<p>4 바이트는 2<sup>32</sup> = 4,294,967,296 가지의 값을 가지고, 5자리 85 진수(<a class="extiw" href="https://en.wikipedia.org/wiki/radix" title="en:radix">en:radix</a>)는 85<sup>5</sup> = 4,437,053,125 가지의 값이 가능하므로 이걸로 32비트 값을 충분히 표현할 수 있다. 5자리의 84 진수는 84<sup>5</sup> = 4,182,119,424 가지 값만을 가지기 때문에 85가 5자리로 4바이트를 표현해 낼 수 있는 최소값이라 이것이 선택되었다.</p>
<p>인코딩은 4바이트를 <a href="/wiki/%EC%97%94%EB%94%94%EC%96%B8" title="엔디언">big-endian</a>으로 묶어 32비트 수를 얻는 것에서 시작한다. 이것을 85진수로 바꾼다. 85를 연속해서 나누면서 나머지들을 얻는 과정이다. 그렇게 5자리의 85진수가 얻어지면, 각 자리에 33를 더해서 33 ("<code>!</code>")에서 117 ("<code>u</code>")까지의 ASCII 문자로 바꾸는 것이다.</p>
<p>데이터가 모두 0인 경우는 상당히 흔하기 때문에 예외적으로 데이터 압축을 위해서 "<code>z</code>" 한 문자로 인코딩한다. "<code>!!!!!</code>"가 아니다.</p>
<p>디코딩했을 때 <span class="nowrap">2<sup>32</sup> − 1</span>(인코딩 했을 때 "<code>s8W-!</code>")보다 큰 수는 에러다. as will "<code>z</code>" characters in the middle of a group. White space between the characters is ignored and may occur anywhere to accommodate line-length limitations.</p>
<p>Ascii85의 한 가지 단점은 인코딩 데이터가 backslash나 quote 같은 <a class="extiw" href="https://en.wikipedia.org/wiki/escape_character" title="en:escape character">en:escape character</a>를 포함할 수 있다는 것인데, 이것들은 많은 프로그래밍 언어나 일부 문자 기반의 프로토콜에서 특별한 의미를 가지므로 문제가 될 수 있다. Z85는 소스 코드에서도 안전하도록 설계된 것이다. <sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup></p>
<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ascii85&amp;action=edit&amp;section=2" title="부분 편집: History">편집</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="btoa_version">btoa version</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ascii85&amp;action=edit&amp;section=3" title="부분 편집: btoa version">편집</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The original btoa program always encoded full groups (padding the source as necessary), with a prefix line of "xbtoa Begin", and suffix line of "xbtoa End", followed by the original file length (in decimal and <a class="new" href="/w/index.php?title=Hexadecimal&amp;action=edit&amp;redlink=1" title="Hexadecimal (없는 문서)">hexadecimal</a>) and three 32-bit <a class="new" href="/w/index.php?title=Checksum&amp;action=edit&amp;redlink=1" title="Checksum (없는 문서)">checksums</a>. The decoder needs to use the file length to see how much of the group was padding. The initial proposal for btoa encoding used an encoding alphabet starting at the ASCII space character through "t" inclusive, but this was replaced with an encoding alphabet of "!" to "u" to avoid "problems with some mailers (stripping off trailing blanks)."<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> This program also introduced the special "<code>z</code>" short form for an all-zero group. Version 4.2 added a "<code>y</code>" exception for a group of all ASCII <a class="new" href="/w/index.php?title=Space_(punctuation)&amp;action=edit&amp;redlink=1" title="Space (punctuation) (없는 문서)">space</a> characters (0x20202020).</p>
<h3><span class="mw-headline" id="ZMODEM_version">ZMODEM version</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ascii85&amp;action=edit&amp;section=4" title="부분 편집: ZMODEM version">편집</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>"ZMODEM Pack-7 encoding" encodes groups of 4 octets into groups of 5 printable ASCII characters, similar to Ascii85 (or perhaps exactly the same?). When <a class="mw-redirect" href="/wiki/ZMODEM" title="ZMODEM">ZMODEM</a> programs send pre-compressed 8-bit data files over <a class="new" href="/w/index.php?title=8-bit_clean&amp;action=edit&amp;redlink=1" title="8-bit clean (없는 문서)">7-bit data channels</a>, it uses "ZMODEM Pack-7 encoding".<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup></p>
<h3><span class="mw-headline" id="Adobe_version">Adobe version</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ascii85&amp;action=edit&amp;section=5" title="부분 편집: Adobe version">편집</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Adobe adopted the basic btoa encoding, but with slight changes, and gave it the name Ascii85. The characters used are the ASCII characters 33 (!) through 117 (u) inclusive (to represent the base-85 digits 0 through 84), together with the letter z (as a special case to represent a 32-bit 0 value), and white space is ignored. Adobe uses the delimiter "<code>~&gt;</code>" to mark the end of an Ascii85-encoded string, and represents the length by truncating the final group: If the last block of source bytes contains fewer than 4 bytes, the block is padded with up to three null bytes before encoding. After encoding, as many bytes as were added as padding are removed from the end of the output.</p>
<p>The reverse is applied when decoding: The last block is padded to 5 bytes with the Ascii85 character "<code>u</code>", and as many bytes as were added as padding are omitted from the end of the output (see example).</p>
<p>NOTE: The padding is not arbitrary. Converting from binary to base 64 only regroups bits and does not change them or their order (a high bit in binary does not affect the low bits in the base64 representation). In converting a binary number to base85 (85 is <i>not</i> a power of two) high bits do affect the low order base85 digits and conversely. Padding the binary low (with zero bits) while encoding and padding the base85 value high (with 'u's) in decoding assures that the high order bits are preserved (the zero padding in the binary gives enough room so that a small addition is trapped and there is no "carry" to the high bits).</p>
<p><br/>
In Ascii85-encoded blocks, whitespace and line-break characters may be present anywhere, including in the middle of a 5-character block, but they must be silently ignored.</p>
<p>Adobe's specification does not support the "<code>y</code>" exception.</p>
<h3><span class="mw-headline" id="ZeroMQ_Version_.28Z85.29">ZeroMQ Version (Z85)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ascii85&amp;action=edit&amp;section=6" title="부분 편집: ZeroMQ Version (Z85)">편집</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Z85, the <a class="new" href="/w/index.php?title=ZeroMQ&amp;action=edit&amp;redlink=1" title="ZeroMQ (없는 문서)">ZeroMQ</a> base-85 encoding algorithm, is a string-safe variant of base85. By avoiding the double-quote, single-quote, and backslash characters, Z85-encoded data can be better embedded in <a class="new" href="/w/index.php?title=Command-line_interpreter&amp;action=edit&amp;redlink=1" title="Command-line interpreter (없는 문서)">command-line interpreter</a> strings. Z85 uses the characters <tt>0</tt>...<tt>9</tt>, <tt>a</tt>...<tt>z</tt>, <tt>A</tt>...<tt>Z</tt>, <tt>.</tt>, <tt>-</tt>, <tt>:</tt>, <tt>+</tt>, <tt>=</tt>, <tt>^</tt>, <tt>!</tt>, <tt>/</tt>, <tt>*</tt>, <tt>?</tt>, <tt>&amp;</tt>, <tt>&lt;</tt>, <tt>&gt;</tt>, <tt>(</tt>, <tt>)</tt>, <tt>[</tt>, <tt>]</tt>, <tt>{</tt>, <tt>}</tt>, <tt>@</tt>, <tt>%</tt>, <tt>$</tt>, <tt>#</tt>.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></p>
<h3><span class="mw-headline" id="Example_for_Ascii85">Example for Ascii85</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ascii85&amp;action=edit&amp;section=7" title="부분 편집: Example for Ascii85">편집</a><span class="mw-editsection-bracket">]</span></span></h3>

<h3><span class="mw-headline" id="Compatibility">Compatibility</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ascii85&amp;action=edit&amp;section=8" title="부분 편집: Compatibility">편집</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The Ascii85 encoding is compatible with 7-bit and 8-bit <a href="/wiki/MIME" title="MIME">MIME</a>, while having less overhead than <a class="mw-redirect" href="/wiki/Base64" title="Base64">Base64</a>.</p>
<p>One potential compatibility issue of Ascii85 is that 'single' and "double" quotation marks, &lt;angle&gt; brackets, and ampersands (&amp;) cannot be used unescaped in markup languages like XML or SGML.</p>
<h2><span class="mw-headline" id="RFC_1924_version">RFC 1924 version</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Ascii85&amp;action=edit&amp;section=9" title="부분 편집: RFC 1924 version">편집</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Published on <a class="new" href="/w/index.php?title=April_Fools%27_Day_Request_for_Comments&amp;action=edit&amp;redlink=1" title="April Fools' Day Request for Comments (없는 문서)">April 1, 1996</a>, informational <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc1924" rel="nofollow">RFC 1924</a>: "A Compact Representation of IPv6 Addresses" by <a class="new" href="/w/index.php?title=Kevin_Robert_Elz&amp;action=edit&amp;redlink=1" title="Kevin Robert Elz (없는 문서)">Robert Elz</a> suggests a base-85 encoding of <a href="/wiki/IPv6" title="IPv6">IPv6</a> addresses. This differs from the scheme used above in that he proposes a different set of 85 ASCII characters, and proposes to do all arithmetic on the 128-bit number, converting it to a single 20-digit base-85 number (internal whitespace not allowed), rather than breaking it into four 32-bit groups.</p>
<p>The proposed character set is, in order, <code>0</code>–<code>9</code>, <code>A</code>–<code>Z</code>, <code>a</code>–<code>z</code>, and then the 23 characters <code>!#$%&amp;()*+-;&lt;=&gt;?@^_`{|}~</code>. The highest possible representable address, 2<sup>128</sup>−1 = 74×85<sup>19</sup> + 53×85<sup>18</sup> + 5×85<sup>17</sup> + ..., would be encoded as <code>=r54lj&amp;NUUO~Hi%c2ym0</code>.</p>
<p>While the RFC chose a different character set in order to prevent the use of certain problematic characters <code>"',./:[\]</code>, it still requires escaping for SGML-based protocols, notably for XML.</p>
