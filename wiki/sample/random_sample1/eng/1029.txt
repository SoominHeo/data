<div class="hatnote" role="note">"Hurd" redirects here. For other uses, see <a class="mw-disambig" href="/wiki/Hurd_(disambiguation)" title="Hurd (disambiguation)">Hurd (disambiguation)</a>.</div>
<p><b>GNU Hurd</b> (usually referred to as <b>The Hurd</b> or just <b>Hurd</b>) is the multiserver <a href="/wiki/Microkernel" title="Microkernel">microkernel</a> written as part of <a href="/wiki/GNU" title="GNU">GNU</a>. It has been under development since 1990 by the <a href="/wiki/GNU_Project" title="GNU Project">GNU Project</a> of the <a href="/wiki/Free_Software_Foundation" title="Free Software Foundation">Free Software Foundation</a>, designed as a replacement for the <a href="/wiki/Unix" title="Unix">Unix</a> kernel,<sup class="reference" id="cite_ref-What_Is_the_GNU_Hurd_3-0"><a href="#cite_note-What_Is_the_GNU_Hurd-3">[3]</a></sup> and released as <a href="/wiki/Free_software" title="Free software">free software</a> under the <a href="/wiki/GNU_General_Public_License" title="GNU General Public License">GNU General Public License</a>.</p>
<p>GNU Hurd consists of a set of protocols and <a href="/wiki/Server_(computing)" title="Server (computing)">server processes</a> (or <a class="mw-redirect" href="/wiki/Daemon_(computer_software)" title="Daemon (computer software)">daemons</a>, in <a href="/wiki/Unix" title="Unix">Unix</a> terminology) that run on the <a href="/wiki/GNU_Mach" title="GNU Mach">GNU Mach</a> <a href="/wiki/Microkernel" title="Microkernel">microkernel</a>.<sup class="reference" id="cite_ref-What_Is_the_GNU_Hurd_3-1"><a href="#cite_note-What_Is_the_GNU_Hurd-3">[3]</a></sup> The Hurd aims to surpass the <a href="/wiki/Unix" title="Unix">Unix</a> kernel in functionality, security, and stability, while remaining largely compatible with it. The GNU Project chose the multiserver microkernel<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup> for the operating system, due to perceived advantages over the traditional Unix <a href="/wiki/Monolithic_kernel" title="Monolithic kernel">monolithic kernel</a> architecture.<sup class="reference" id="cite_ref-advantages_5-0"><a href="#cite_note-advantages-5">[5]</a></sup></p>

<h2><span class="mw-headline" id="Name_and_logo">Name and logo</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GNU_Hurd&amp;action=edit&amp;section=1" title="Edit section: Name and logo">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In December 1991 the primary architect of the Hurd described the name as a <a href="/wiki/Mutual_recursion" title="Mutual recursion">mutually</a> <a href="/wiki/Recursive_acronym" title="Recursive acronym">recursive acronym</a>:<sup class="reference" id="cite_ref-Koen_Vervloesem.2C_2010_6-0"><a href="#cite_note-Koen_Vervloesem.2C_2010-6">[6]</a></sup></p>
<blockquote class="templatequote">
<p>It's time [to] explain the meaning of "Hurd". "Hurd" stands for "Hird of Unix-Replacing Daemons". And, then, "Hird" stands for "Hurd of Interfaces Representing Depth". We have here, to my knowledge, the first software to be named by a pair of mutually recursive acronyms.</p>
<div class="templatequotecite"><cite>— <a href="/wiki/Thomas_Bushnell" title="Thomas Bushnell">Thomas (then Michael) Bushnell</a></cite></div>
</blockquote>
<p>As both <i>hurd</i> and <i>hird</i> are <a class="mw-redirect" href="/wiki/Homophones" title="Homophones">homophones</a> of the English word <i>herd</i>, the full name <i>GNU Hurd</i> is also a play on the words <i><a href="/wiki/Herd" title="Herd">herd</a> of <a href="/wiki/Wildebeest" title="Wildebeest">gnus</a></i>, reflecting how the kernel works.<sup class="reference" id="cite_ref-hurd-name_7-0"><a href="#cite_note-hurd-name-7">[7]</a></sup> The logo is called the <i>Hurd boxes</i> and it also reflects on architecture. The logo is a graph where nodes represent the Hurd kernel's servers and directed edges are <a href="/wiki/Inter-process_communication" title="Inter-process communication">IPC</a> messages.<sup class="reference" id="cite_ref-Koen_Vervloesem.2C_2010_6-1"><a href="#cite_note-Koen_Vervloesem.2C_2010-6">[6]</a></sup></p>
<h2><span class="mw-headline" id="Development_history">Development history</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GNU_Hurd&amp;action=edit&amp;section=2" title="Edit section: Development history">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p><a href="/wiki/Richard_Stallman" title="Richard Stallman">Richard Stallman</a> founded the <a class="mw-redirect" href="/wiki/GNU_project" title="GNU project">GNU project</a> in September 1983 with an aim to create a <a href="/wiki/Free_software" title="Free software">free</a> <a href="/wiki/GNU" title="GNU">GNU</a> operating system. Initially the components required for kernel development were written: <a href="/wiki/Text_editor" title="Text editor">editors</a>, <a href="/wiki/Unix_shell" title="Unix shell">shell</a>, <a href="/wiki/Compiler" title="Compiler">compiler</a> and all the others. By 1989, the <a class="mw-redirect" href="/wiki/GNU_GPL" title="GNU GPL">GNU GPL</a> came into being and the only major component missing was the kernel.<sup class="reference" id="cite_ref-Richard_Hillesley.2C_2010_8-0"><a href="#cite_note-Richard_Hillesley.2C_2010-8">[8]</a></sup><sup class="reference" id="cite_ref-linux-and-gnu_9-0"><a href="#cite_note-linux-and-gnu-9">[9]</a></sup></p>
<p>Development on the Hurd began in 1990 after an abandoned kernel attempt in 1986, based on the research <a class="mw-redirect" href="/wiki/Trix_(operating_system)" title="Trix (operating system)">TRIX</a> operating system developed by Professor Steve Ward and his group at <a href="/wiki/Massachusetts_Institute_of_Technology" title="Massachusetts Institute of Technology">MIT's</a> <a class="mw-redirect" href="/wiki/Laboratory_for_Computer_Science" title="Laboratory for Computer Science">Laboratory for Computer Science</a> (LCS).<sup class="reference" id="cite_ref-hurd-hist_10-0"><a href="#cite_note-hurd-hist-10">[10]</a></sup> According to <a href="/wiki/Thomas_Bushnell" title="Thomas Bushnell">Thomas Bushnell</a>, the initial Hurd architect, their early plan was to adapt the <a class="mw-redirect" href="/wiki/4.4BSD" title="4.4BSD">4.4BSD</a>-Lite kernel and, in hindsight, "It is now perfectly obvious to me that this would have succeeded splendidly and the world would be a very different place today".<sup class="reference" id="cite_ref-hurd-and-bsdi_11-0"><a href="#cite_note-hurd-and-bsdi-11">[11]</a></sup> In 1987 Richard Stallman proposed using the <a href="/wiki/Mach_(kernel)" title="Mach (kernel)">Mach microkernel</a> developed at <a href="/wiki/Carnegie_Mellon_University" title="Carnegie Mellon University">Carnegie Mellon University</a>. Work on this was delayed for three years due to uncertainty over whether CMU would release the Mach code under a suitable license.<sup class="reference" id="cite_ref-hurd-hist_10-1"><a href="#cite_note-hurd-hist-10">[10]</a></sup></p>
<p>With the release of the <a href="/wiki/Linux_kernel" title="Linux kernel">Linux kernel</a> in 1991, the primary user of GNU's <a class="mw-redirect" href="/wiki/Userland_(computing)" title="Userland (computing)">userland</a> components soon became operating systems based on the Linux kernel (<a href="/wiki/Linux_distribution" title="Linux distribution">Linux distributions</a>), prompting the coining of the term <a href="/wiki/GNU/Linux_naming_controversy" title="GNU/Linux naming controversy"><i>GNU/Linux</i></a>.</p>
<p>Development of the Hurd has proceeded slowly. Despite an optimistic announcement by Stallman in 2002 predicting a release of GNU/Hurd later that year,<sup class="reference" id="cite_ref-gnu-loose-linux_12-0"><a href="#cite_note-gnu-loose-linux-12">[12]</a></sup> the Hurd is still not considered suitable for production environments. Development in general has not met expectations, and there are still a significant number of bugs and missing features.<sup class="reference" id="cite_ref-hurd-status_13-0"><a href="#cite_note-hurd-status-13">[13]</a></sup> This has resulted in a poorer product than many (including Stallman) had expected.<sup class="reference" id="cite_ref-fsf-future-of-freedom_14-0"><a href="#cite_note-fsf-future-of-freedom-14">[14]</a></sup> In 2010, after twenty years under development, Stallman said that he was "not very optimistic about the GNU Hurd. It makes some progress, but to be really superior it would require solving a lot of deep problems", but added that "finishing it is not crucial" for the GNU system because a free kernel already existed (<a href="/wiki/Linux" title="Linux">Linux</a>), and completing Hurd would not address the main remaining problem for a free operating system: device support.<sup class="reference" id="cite_ref-reddit-interview-stallman_15-0"><a href="#cite_note-reddit-interview-stallman-15">[15]</a></sup></p>
<p>The <a href="/wiki/Debian" title="Debian">Debian</a> project, among others, have worked on the Hurd project to produce binary distributions of <a href="/wiki/Debian_GNU/Hurd" title="Debian GNU/Hurd">Hurd-based GNU operating systems</a> for <a href="/wiki/IBM_PC_compatible" title="IBM PC compatible">IBM PC compatible</a> systems.</p>
<p>On August 20, 2015, amid the <a href="/wiki/Google_Summer_of_Code" title="Google Summer of Code">Google Summer of Code</a>, it was announced that <a href="/wiki/GNU_Guix" title="GNU Guix">GNU Guix</a> had been ported to GNU Hurd,<sup class="reference" id="cite_ref-gsoc_16-0"><a href="#cite_note-gsoc-16">[16]</a></sup> making it the first native package manager on the Hurd.<sup class="reference" id="cite_ref-phoro_17-0"><a href="#cite_note-phoro-17">[17]</a></sup></p>
<h2><span class="mw-headline" id="Architecture">Architecture</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GNU_Hurd&amp;action=edit&amp;section=3" title="Edit section: Architecture">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="thumb tright">

<p>Unlike most <a href="/wiki/Unix-like" title="Unix-like">Unix-like</a> kernels, the Hurd uses a <a class="mw-redirect" href="/wiki/Server%E2%80%93client_architecture" title="Server–client architecture">server–client architecture</a>, built on a <a href="/wiki/Microkernel" title="Microkernel">microkernel</a> that is responsible for providing the most basic <a class="mw-redirect" href="/wiki/Kernel_service" title="Kernel service">kernel services</a> – coordinating access to the <a href="/wiki/Computer_hardware" title="Computer hardware">hardware</a>: the <a href="/wiki/Central_processing_unit" title="Central processing unit">CPU</a> (through <a href="/wiki/Process_management_(computing)" title="Process management (computing)">process management</a> and <a href="/wiki/Scheduling_(computing)" title="Scheduling (computing)">scheduling</a>), <a class="mw-redirect" href="/wiki/Random_access_memory" title="Random access memory">RAM</a> (via <a href="/wiki/Memory_management" title="Memory management">memory management</a>), and other various <a href="/wiki/Input/output" title="Input/output">input/output</a> devices (via <a href="/wiki/I/O_scheduling" title="I/O scheduling">I/O scheduling</a>) for sound, graphics, mass storage, etc. In theory the microkernel design would allow for all device drivers to be built as servers working in <a href="/wiki/User_space" title="User space">user space</a>, but today most drivers of this kind are still contained in the <a href="/wiki/GNU_Mach" title="GNU Mach">GNU Mach</a> <a class="mw-redirect" href="/wiki/Kernel_space" title="Kernel space">kernel space</a>.<sup class="reference" id="cite_ref-device-drivers-in-mach_18-0"><a href="#cite_note-device-drivers-in-mach-18">[18]</a></sup></p>
<p>According to Hurd developers the main advantage of microkernel-based design is the ability to extend the system: developing a new module would not require in depth knowledge of the rest of the kernel and a bug in one module would not crash the entire system. Hurd provides a concept of <i>translators</i>, a framework of modules used to extend a file system functionality.<sup class="reference" id="cite_ref-Doeppner2010_19-0"><a href="#cite_note-Doeppner2010-19">[19]</a></sup></p>
<p>From early on, the Hurd was developed to use <a href="/wiki/GNU_Mach" title="GNU Mach">GNU Mach</a> as the microkernel. This was a technical decision made by <a href="/wiki/Richard_Stallman" title="Richard Stallman">Richard Stallman</a>, who thought it would speed up the work by saving a large part of it. He has admitted that he was wrong about that.<sup class="reference" id="cite_ref-lt-defence-rh_20-0"><a href="#cite_note-lt-defence-rh-20">[20]</a></sup> Other Unix-like systems working on the <a class="mw-redirect" href="/wiki/Mach_kernel" title="Mach kernel">Mach microkernel</a> include <a class="mw-redirect" href="/wiki/OSF/1" title="OSF/1">OSF/1</a>, <a href="/wiki/Lites" title="Lites">Lites</a>, and <a href="/wiki/MkLinux" title="MkLinux">MkLinux</a>. <a href="/wiki/MacOS" title="MacOS">macOS</a> and <a href="/wiki/NeXTSTEP" title="NeXTSTEP">NeXTSTEP</a> use <a href="/wiki/Hybrid_kernel" title="Hybrid kernel">hybrid kernels</a> based on Mach.</p>
<h3><span class="mw-headline" id="Other_microkernels">Other microkernels</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GNU_Hurd&amp;action=edit&amp;section=4" title="Edit section: Other microkernels">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>From 2004 onward, various efforts were launched to port the Hurd to more modern microkernels. The <a href="/wiki/L4_microkernel_family" title="L4 microkernel family">L4 microkernel</a> was the original choice in 2004, but progress slowed to a halt. Nevertheless, during 2005, Hurd developer Neal Walfield finished the initial memory management framework for the L4/Hurd port, and Marcus Brinkmann ported essential parts of <a class="mw-redirect" href="/wiki/Glibc" title="Glibc">glibc</a>; namely, getting the process startup code working, allowing programs to run, thus allowing the first user programs (trivial ones such as the <a class="mw-redirect" href="/wiki/Hello_world_program" title="Hello world program">hello world program</a>) in C to run.</p>
<p>Since 2005 Brinkmann and Walfield started researching <a href="/wiki/Coyotos" title="Coyotos">Coyotos</a> as a new kernel for HURD.<sup class="reference" id="cite_ref-shapiro-comment_21-0"><a href="#cite_note-shapiro-comment-21">[21]</a></sup><sup class="reference" id="cite_ref-l4-and-coyotos-mess_22-0"><a href="#cite_note-l4-and-coyotos-mess-22">[22]</a></sup> In 2006, Brinkmann met with Jonathan Shapiro (a primary architect of the Coyotos Operating System) to aid in and discuss the use of the Coyotos kernel for GNU/Hurd. In further discussion HURD developers realised that Coyotos (as well as other similar kernels) are not suitable for HURD<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (December 2013)">citation needed</span></a></i>]</sup>.</p>
<p>In 2007, Hurd developers Neal Walfield and Marcus Brinkmann gave a critique of the Hurd architecture, known as "the critique",<sup class="reference" id="cite_ref-critique-mach_23-0"><a href="#cite_note-critique-mach-23">[23]</a></sup> and a proposal for how a future system may be designed, known as "the position paper".<sup class="reference" id="cite_ref-position-paper_24-0"><a href="#cite_note-position-paper-24">[24]</a></sup> In 2008, Neal Walfield began working on the Viengoos microkernel as a modern native kernel for HURD. As of 2009<sup class="plainlinks noprint asof-tag update" style="display:none;"><a class="external text" href="//en.wikipedia.org/w/index.php?title=GNU_Hurd&amp;action=edit">[update]</a></sup>, development on Viengoos is paused due to Walfield lacking time to work on it.<sup class="reference" id="cite_ref-viengoos_25-0"><a href="#cite_note-viengoos-25">[25]</a></sup></p>
<p>In the meantime, others have continued working on the Mach variant of Hurd.<sup class="reference" id="cite_ref-what-happend-l4-coyotos_26-0"><a href="#cite_note-what-happend-l4-coyotos-26">[26]</a></sup></p>

<p>A number of traditional Unix concepts are replaced or extended in the Hurd.</p>
<p>Under Unix, every running program has an associated <a class="mw-redirect" href="/wiki/User_id" title="User id">user id</a>, which normally corresponds to the user that started the <a href="/wiki/Process_(computing)" title="Process (computing)">process</a>. This id largely dictates the actions permitted to the program. No outside process can change the user id of a running program. A Hurd process, on the other hand, runs under a <i>set</i> of user ids, which can contain multiple ids, one, or none. A sufficiently privileged process can add and remove ids to another process. For example, there is a password server that will hand out ids in return for a correct login password.</p>
<p>Regarding the <a href="/wiki/File_system" title="File system">file system</a>, a suitable program can be designated as a <i>translator</i> for a single file or a whole directory hierarchy. Every access to the translated file, or files below a hierarchy in the second case, is in fact handled by the program. For example, a file translator may simply redirect read and write operations to another file, like a Unix <a href="/wiki/Symbolic_link" title="Symbolic link">symbolic link</a>. The effect of Unix <i><a href="/wiki/Mount_(computing)" title="Mount (computing)">mounting</a></i> is achieved by setting up a filesystem translator (using the "settrans" command). Translators can also be used to provide services to the user. For example, the <a class="new" href="/w/index.php?title=Ftpfs&amp;action=edit&amp;redlink=1" title="Ftpfs (page does not exist)">ftpfs</a> translator allows a user to encapsulate remote FTP sites within a directory. Then, standard tools such as <a href="/wiki/Ls" title="Ls">ls</a>, <a href="/wiki/Cp_(Unix)" title="Cp (Unix)">cp</a>, and <a href="/wiki/Rm_(Unix)" title="Rm (Unix)">rm</a> can be used to manipulate files on the remote system. Even more powerful translators are ones such as <a href="/wiki/UnionFS" title="UnionFS">UnionFS</a>, which allows a user to unify multiple directories into one; thus listing the unified directory reveals the contents of all the directories.</p>
<p>The Hurd requires a <a href="/wiki/Multiboot_Specification" title="Multiboot Specification">multiboot</a>-compliant <a href="/wiki/Booting" title="Booting">boot loader</a>, such as <a href="/wiki/GNU_GRUB" title="GNU GRUB">GRUB</a>.</p>
<h3><span class="mw-headline" id="Architecture_of_the_servers">Architecture of the servers</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GNU_Hurd&amp;action=edit&amp;section=6" title="Edit section: Architecture of the servers">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>According to the Debian documentation there are 24 servers (18 core servers and 6 file system servers) named as follows:<sup class="reference" id="cite_ref-debian-hurd-doc_27-0"><a href="#cite_note-debian-hurd-doc-27">[27]</a></sup></p>
<h4><span class="mw-headline" id="Core_servers">Core servers</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GNU_Hurd&amp;action=edit&amp;section=7" title="Edit section: Core servers">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<ul>
<li><b>auth</b> (<a href="/wiki/Authentication_server" title="Authentication server">authentication server</a>): Receives requests and passwords from programs and gives them an ID, which changes the privileges of the program.</li>
<li><b>crash</b> (<a href="/wiki/Crash_(computing)" title="Crash (computing)">crash</a> server): Handles all fatal errors.</li>
<li><b>exec</b> (execution server): Translates an executable image (currently <a href="/wiki/Executable_and_Linkable_Format" title="Executable and Linkable Format">ELF</a> and <a href="/wiki/A.out" title="A.out">a.out</a> are supported) to a runnable image in memory.</li>
<li><b>fifo</b> (<a href="/wiki/FIFO_(computing_and_electronics)" title="FIFO (computing and electronics)">FIFO</a> translator): Implements named pipes.</li>
<li><b>new-fifo</b> (new FIFO server): An alternate server for named pipes.</li>
<li><b>firmlink</b> (the firmlink translator): Implements firmlinks ‒ "half-way between a <a href="/wiki/Symbolic_link" title="Symbolic link">symbolic link</a> and a <a href="/wiki/Hard_link" title="Hard link">hard link</a>".<sup class="reference" id="cite_ref-28"><a href="#cite_note-28">[28]</a></sup></li>
<li><b>fwd</b> (forward server): Forwards requests to other servers, used by fifo and symlink servers.</li>
<li><b>hostmux</b> (host multiplexer server)</li>
<li><b>ifsock</b> (server for sockets interface): Helps with UNIX domain socket addresses.</li>
<li><b>init</b> (<a href="/wiki/Init" title="Init">init</a> server): Basic system booting and configuration.</li>
<li><b>magic</b> (<a href="/wiki/Magic_(programming)" title="Magic (programming)">magic</a> server): Signals that a name lookup must be resolved internally by a process when the result involves the process's state.</li>
<li><b>null</b> (null server): Implements <a class="mw-redirect" href="/wiki//dev/null" title="/dev/null">/dev/null</a> and /dev/zero.</li>
<li><b>pfinet</b> (pfinet server): Implements the PF_INET protocol family.</li>
<li><b>pflocal</b> (pflocal server): Implements <a class="mw-redirect" href="/wiki/UNIX_domain_sockets" title="UNIX domain sockets">UNIX domain sockets</a>.</li>
<li><b>proc</b> (process server): Assigns <a href="/wiki/Process_identifier" title="Process identifier">PIDs</a> and manages process-level actions.</li>
<li><b>symlink</b> (<a href="/wiki/Symbolic_link" title="Symbolic link">symbolic link</a> translator): Implements symbolic links for filesystems that don't support them.</li>
<li><b>term</b> (terminal server): A <a href="/wiki/POSIX" title="POSIX">POSIX</a> terminal.</li>
<li><b>usermux</b> (user multiplexer server): Invokes user-specific translators.</li>
</ul>
<h4><span class="mw-headline" id="Filesystem_servers">Filesystem servers</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GNU_Hurd&amp;action=edit&amp;section=8" title="Edit section: Filesystem servers">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<dl>
<dt>ext2fs</dt>
<dd>The <a href="/wiki/Ext2" title="Ext2">ext2</a> filesystem translator. It receives disk blocks from the microkernel and gives files and directories to the applications.</dd>
<dt>isofs</dt>
<dd>The translator for the <a href="/wiki/ISO_9660" title="ISO 9660">ISO 9660</a> filesystem. Translates blocks of a CD or DVD to files and directories for the applications.</dd>
<dt>nfs</dt>
<dd>See <a class="mw-redirect" href="/wiki/Network_File_System_(protocol)" title="Network File System (protocol)">Network File System</a>.</dd>
<dt>ufs</dt>
<dd>Translator for the BSD filesystem of the same name, <a href="/wiki/Unix_File_System" title="Unix File System">UFS</a>.</dd>
<dt>ftpfs</dt>
<dd><a class="mw-redirect" href="/wiki/File_transfer_protocol" title="File transfer protocol">File transfer protocol</a> filesystem translator.</dd>
<dt>storeio</dt>
<dd>The storage translator.</dd>
</dl>
<p>The servers collectively implement the <a href="/wiki/POSIX" title="POSIX">POSIX</a> <a class="mw-redirect" href="/wiki/Applications_programming_interface" title="Applications programming interface">API</a>, with each server implementing a part of the interface. For instance, the various filesystem servers each implement the filesystem calls. The storage server will work as a wrapping layer, similar to the block layer of Linux. The equivalent of <a href="/wiki/Virtual_file_system" title="Virtual file system">VFS</a> of <a href="/wiki/Linux" title="Linux">Linux</a> is achieved by libdiskfs and libpager libraries.</p>
