As the <a href="/wiki/GNU_C_Library" title="GNU C Library">GNU C Library</a> serves as a wrapper for <a href="/wiki/Linux_kernel" title="Linux kernel">Linux kernel</a> <a href="/wiki/System_call" title="System call">system calls</a>, so do the libraries bundled in <a href="/wiki/GLib" title="GLib">GLib</a> (GObject, <a class="mw-redirect" href="/wiki/Glib" title="Glib">Glib</a>, <a class="new" href="/w/index.php?title=GModule&amp;action=edit&amp;redlink=1" title="GModule (page does not exist)">GModule</a>, <a class="new" href="/w/index.php?title=GThread&amp;action=edit&amp;redlink=1" title="GThread (page does not exist)">GThread</a> and <a href="/wiki/GIO_(software)" title="GIO (software)">GIO</a>) serve as further wrappers for their specific tasks.</div>
</div>
</div>
<p>The <b>GLib Object System</b>, or <b>GObject</b>, is a <a href="/wiki/Free_software" title="Free software">free</a> <a class="mw-redirect" href="/wiki/Software_library" title="Software library">software library</a> providing a portable <a class="mw-redirect" href="/wiki/Object_system" title="Object system">object system</a> and transparent cross-language interoperability. GObject is designed for use both directly in <a href="/wiki/C_(programming_language)" title="C (programming language)">C</a> programs to provide object-oriented C-based APIs and through <a href="/wiki/Language_binding" title="Language binding">bindings</a> to other languages to provide transparent cross-language interoperability, e.g. <a class="mw-redirect" href="/wiki/PyGObject" title="PyGObject">PyGObject</a>.</p>
<p></p>

<h2><span class="mw-headline" id="GObject_Introspection">GObject Introspection</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GObject&amp;action=edit&amp;section=1" title="Edit section: GObject Introspection">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<ul>
<li><span class="url"><a class="external text" href="https://wiki.gnome.org/Projects/GObjectIntrospection" rel="nofollow">wiki<wbr></wbr>.gnome<wbr></wbr>.org<wbr></wbr>/Projects<wbr></wbr>/GObjectIntrospection</a></span> GObject introspection is a middleware layer between C libraries (using GObject) and language bindings, cf. <a href="/wiki/List_of_language_bindings_for_GTK%2B" title="List of language bindings for GTK+">List of language bindings for GTK+</a>.</li>
</ul>
<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GObject&amp;action=edit&amp;section=2" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Depending only on <a href="/wiki/GLib" title="GLib">GLib</a> and <a class="mw-redirect" href="/wiki/Libc" title="Libc">libc</a>, GObject is a cornerstone of <a href="/wiki/GNOME" title="GNOME">GNOME</a> and is used throughout <a href="/wiki/GTK%2B" title="GTK+">GTK+</a>, <a href="/wiki/Pango" title="Pango">Pango</a>, <a href="/wiki/Accessibility_Toolkit" title="Accessibility Toolkit">ATK</a>, and most higher-level <a href="/wiki/GNOME" title="GNOME">GNOME</a> libraries like <a href="/wiki/GStreamer" title="GStreamer">GStreamer</a> and applications. Prior to GTK+ 2.0, code similar to GObject was part of the GTK+ codebase. (The name “GObject” was not yet in use — the common baseclass was called <code>GtkObject</code>.)</p>
<p>At the release of GTK+ 2.0, the object system was extracted into a separate library due to its general utility. In the process, most non-<a class="mw-redirect" href="/wiki/GUI" title="GUI">GUI</a>-specific parts of the <code>GtkObject</code> class were moved up into <code>GObject</code>, the new common baseclass. Having existed as a separate library since March 11, 2002 (the release date of GTK+ 2.0), the GObject library is now used by many non-GUI programs such as <a class="mw-redirect" href="/wiki/Command-line" title="Command-line">command-line</a> and <a href="/wiki/Server_(computing)" title="Server (computing)">server</a> applications.</p>
<h2><span class="mw-headline" id="Relation_to_GLib">Relation to GLib</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GObject&amp;action=edit&amp;section=3" title="Edit section: Relation to GLib">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Though GObject has its own separate set of documentation<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup> and is usually compiled into its own <a class="mw-redirect" href="/wiki/Shared_library" title="Shared library">shared library</a> file, the source code for GObject resides in the <a href="/wiki/GLib" title="GLib">GLib</a> source tree and is distributed along with GLib. For this reason, GObject uses the GLib version numbers and is typically packaged together with GLib (for example, <a href="/wiki/Debian" title="Debian">Debian</a> puts GObject in its <code>libglib2.0</code> package family).</p>
<h2><span class="mw-headline" id="The_type_system">The type system</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GObject&amp;action=edit&amp;section=4" title="Edit section: The type system">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>At the most basic level of the GObject framework lies a generic and dynamic <a href="/wiki/Type_system" title="Type system">type system</a> called GType. The GType system holds a runtime description of all objects allowing <a href="/wiki/Glue_code" title="Glue code">glue code</a> to facilitate multiple language bindings. The type system can handle any <a class="mw-redirect" href="/wiki/Single_inheritance" title="Single inheritance">singly inherited</a> class structure, in addition to <i>non-classed</i> types such as <a href="/wiki/Opaque_pointer" title="Opaque pointer">opaque pointers</a>, <a href="/wiki/String_(computer_science)" title="String (computer science)">strings</a>, and variously sized <a href="/wiki/Integer" title="Integer">integers</a> and <a class="mw-redirect" href="/wiki/Floating_point_number" title="Floating point number">floating point numbers</a>.</p>
<p>The type system knows how to copy, assign, and destroy values belonging to any of the registered types. This is trivial for types like integers, but many complex objects are <a href="/wiki/Reference_counting" title="Reference counting">reference-counted</a>, while some are complex but not reference-counted. When the type system “copies” a reference-counted object, it will typically just increase its reference count, whereas when copying a complex, non-reference-counted object (such as a string), it will typically create an actual copy by <a class="mw-redirect" href="/wiki/Memory_allocation" title="Memory allocation">allocating memory</a>.</p>
<p>This basic functionality is used for implementing <code>GValue</code>, a type of generic container that can hold values of any type known by the type system. Such containers are particularly useful when interacting with dynamically typed language environments in which all native values reside in such <a href="/wiki/Tagged_union" title="Tagged union">type-tagged</a> containers.</p>
<h3><span class="mw-headline" id="Fundamental_types">Fundamental types</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GObject&amp;action=edit&amp;section=5" title="Edit section: Fundamental types">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Types that do not have any associated <a class="mw-redirect" href="/wiki/Class_(programming)" title="Class (programming)">classes</a> are called <i>non-classed</i>. These types, together with all types that correspond to some form of <a class="new" href="/w/index.php?title=Root_class&amp;action=edit&amp;redlink=1" title="Root class (page does not exist)">root class</a>, are known as <i>fundamental types</i>: the types from which all other types are derived. These make up a relatively closed set, but although the average user is not expected to create his own fundamental types, the possibility does exist and has been exploited to create custom <a href="/wiki/Class_hierarchy" title="Class hierarchy">class hierarchies</a> — i.e., class hierarchies not based on the <code>GObject</code> class.</p>
<p>As of GLib 2.9.2,<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup> the <i>non-classed</i> built-in fundamental types are</p>
<ul>
<li>an empty type, corresponding to C’s <code>void</code> (<code>G_TYPE_NONE</code>);</li>
<li>types corresponding to C’s signed and unsigned <code>char</code>, <code>int</code>, <code>long</code>, and 64-bit integers (<code>G_TYPE_CHAR</code>, <code>G_TYPE_UCHAR</code>, <code>G_TYPE_INT</code>, <code>G_TYPE_UINT</code>, <code>G_TYPE_LONG</code>, <code>G_TYPE_ULONG</code>, <code>G_TYPE_INT64</code>, and <code>G_TYPE_UINT64</code>);</li>
<li>a boolean type (<code>G_TYPE_BOOLEAN</code>);</li>
<li>an enumeration type and a “flags” type, both corresponding to C’s <code>enum</code> type, but differing in that the latter is only used for <a href="/wiki/Bit_field" title="Bit field">bit fields</a> (<code>G_TYPE_ENUM</code> and <code>G_TYPE_FLAGS</code>);</li>
<li>types for single- and double-precision <a class="mw-redirect" href="/wiki/IEEE_float" title="IEEE float">IEEE floats</a>, corresponding to C’s <code>float</code> and <code>double</code> (<code>G_TYPE_FLOAT</code> and <code>G_TYPE_DOUBLE</code>);</li>
<li>a string type, corresponding to C’s <code>char *</code> (<code>G_TYPE_STRING</code>);</li>
<li>an opaque pointer type, corresponding to C’s <code>void *</code> (<code>G_TYPE_POINTER</code>).</li>
</ul>
<p>The <i>classed</i> built-in fundamental types are</p>
<ul>
<li>a base class type for instances of <code>GObject</code>, the root of the standard class inheritance tree (<code>G_TYPE_OBJECT</code>)</li>
<li>a base interface type, analoguous to the base class type but representing the root of the standard <i>interface</i> inheritance tree (<code>G_TYPE_INTERFACE</code>)</li>
<li>a type for <a class="mw-redirect" href="/wiki/Boxed_type" title="Boxed type">boxed</a> structures, which are used to wrap simple value objects or foreign objects in reference-counted “boxes” (<code>G_TYPE_BOXED</code>)</li>
<li>a type for “parameter specification objects,” which are used in GObject to describe <a href="/wiki/Metadata" title="Metadata">metadata</a> for object properties (<code>G_TYPE_PARAM</code>).</li>
</ul>
<p>Types that can be instantiated automatically by the type system are called <i>instantiable</i>. An important characteristic of these types is that the first bytes of any instance always contain a pointer to the <i>class structure</i> (a form of <a class="mw-redirect" href="/wiki/Virtual_table" title="Virtual table">virtual table</a>) associated to the type of the instance. For this reason, any instantiable type must be classed. Contrapositively, any non-classed type (such as <i>integer</i> or <i>string</i>) must be non-instantiable. On the other hand, most classed types are instantiable, but some, such as interface types, are not.</p>
<h3><span class="mw-headline" id="Derived_types">Derived types</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GObject&amp;action=edit&amp;section=6" title="Edit section: Derived types">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The types that are derived from the built-in GObject fundamental types fall roughly into four categories:</p>
<dl>
<dt>Enumerated types and “flags” types </dt>
<dd>In general, every enumerated type and every integer-based bitfield type (i.e., every <code>enum</code> type) that one wishes to use in some way that is related to the object system — for example, as the type of an object property — should be registered with the type system. Typically, the initialization code that takes care of registering these types is generated by an automated tool called <code>glib-mkenums</code><sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> and stored in a separate file.</dd>
<dt>Boxed types </dt>
<dd>Some data structures that are too simple to be made full-fledged class types (with all the overhead incurred) may still need to be registered with the type system. For example, we might have a class to which we want to add a <code>background-color</code> property, whose values should be instances of a structure that looks like <code class="mw-highlight" dir="ltr" id="" style=""><span class="k">struct</span> <span class="n">color</span> <span class="p">{</span> <span class="kt">int</span> <span class="n">r</span><span class="p">,</span> <span class="n">g</span><span class="p">,</span> <span class="n">b</span><span class="p">;</span> <span class="p">}</span></code>. To avoid having to subclass <code>GObject</code>, we can create a <a class="mw-redirect" href="/wiki/Boxed_type" title="Boxed type">boxed type</a> to represent this structure, and provide functions for copying and freeing. GObject ships with a handful of boxed types wrapping simple GLib data types. Another use for boxed types is as a way to wrap foreign objects in a tagged container that the type system can identify and will know how to copy and free.</dd>
<dt>Opaque pointer types </dt>
<dd>Sometimes, for objects that need to be neither copied or reference-counted nor freed, even a boxed type would be <a href="/wiki/Overkill_(term)" title="Overkill (term)">overkill</a>. While such objects can be used in GObject by simply treating them as opaque pointers (<code>G_TYPE_POINTER</code>), it is often a good idea to create a derived pointer type, documenting the fact that the pointers should reference a particular kind of object, even though nothing else is said about it.</dd>
<dt>Class and interface types </dt>
<dd>Most types in a GObject application will be classes — in the normal object-oriented sense of the word — derived directly or indirectly from the root class, <code>GObject</code>. There are also interfaces, which, unlike classic <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>-style interfaces, can contain implemented methods. GObject interfaces can thus be described as <a href="/wiki/Mixin" title="Mixin">mixins</a>.</dd>
</dl>
<h2><span class="mw-headline" id="Messaging_system">Messaging system</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GObject&amp;action=edit&amp;section=7" title="Edit section: Messaging system">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The GObject messaging system consists of two complementary parts: <i>closures</i> and <i>signals</i>.</p>
<dl>
<dt>Closures </dt>
<dd>A GObject closure is a generalized version of a <a class="mw-redirect" href="/wiki/Callback_(computer_science)" title="Callback (computer science)">callback</a>. Support exists for closures written in C and C++, as well as arbitrary languages (when bindings are provided). This allows code written in (for example) Python and Java to be invoked via a GObject closure.</dd>
</dl>
<dl>
<dt>Signals </dt>
<dd>Signals are the primary mechanism by which closures are invoked. Objects register signal listeners with the type system, specifying a mapping between a given signal and a given closure. Upon emission of a registered signal, that signal's closure is invoked. In GTK+, all native GUI events (such as mouse motion and keyboard actions) can generate GObject signals for listeners to potentially act upon.</dd>
</dl>
<h2><span class="mw-headline" id="Class_implementation">Class implementation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GObject&amp;action=edit&amp;section=8" title="Edit section: Class implementation">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Each GObject class is implemented by at least two structures: the <i>class structure</i> and the <i>instance structure</i>.</p>
<dl>
<dt>The class structure </dt>
<dd>The class structure corresponds to the <a class="mw-redirect" href="/wiki/Vtable" title="Vtable">vtable</a> of a C++ class. It must begin with the class structure of the superclass. Following that, it will hold a set of function pointers — one for each <a class="mw-redirect" href="/wiki/Virtual_method" title="Virtual method">virtual method</a> of the class. Class-specific variables can be used to emulate class members.</dd>
<dt>The instance structure </dt>
<dd>The instance structure, which will exist in one copy per object instance, must begin with the instance structure of the <a class="mw-redirect" href="/wiki/Superclass_(computer_science)" title="Superclass (computer science)">superclass</a> (this ensures that all instances begin with a pointer to the class structure, since all fundamental instantiable types share this property). After the data belonging to the superclass, the structure can hold any instance-specific variables, corresponding to C++ member variables.</dd>
</dl>
<p>Since a C structure cannot have access modifiers like “public”, “protected”, or “private”, a common technique is to include a pointer to the private data — conventionally called <code>_priv</code> — in the instance structure. The <i>private structure</i> can be declared in the public header file, but defined only in the implementation file, with the effect that the private data is opaque to users, but transparent to the implementor. If the private structure is registered with GType, it will be automatically allocated by the object system. Indeed, it is not even necessary to include the <code>_priv</code> pointer, if one is willing to use the incantation <code>G_TYPE_INSTANCE_GET_PRIVATE</code> every time the private data is needed.</p>
<p>The main drawback of the GObject framework is its verbosity. Large amounts of <a href="/wiki/Boilerplate_(text)#Boilerplate_code" title="Boilerplate (text)">boilerplate</a> code, such as manual definitions of type casting macros and obscure type registration incantations, are necessary to create a new class. The GObject Builder, or <a class="external text" href="http://www.jirka.org/gob.html" rel="nofollow">GOB2</a>, is a tool that attempts to remedy this problem by offering a template syntax reminiscent of Java. Code written using GOB2 is pre-processed into <a class="mw-redirect" href="/wiki/Vanilla_(computing)" title="Vanilla (computing)">vanilla</a> C code prior to compilation. Another compiler-to-C for the GObject type system is <a href="/wiki/Vala_(programming_language)" title="Vala (programming language)">Vala</a>, which uses a <a href="/wiki/C_Sharp_(programming_language)" title="C Sharp (programming language)">C#</a>-style syntax.</p>
<h2><span class="mw-headline" id="Usage">Usage</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GObject&amp;action=edit&amp;section=9" title="Edit section: Usage">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The combination of C and GObject is used in many successful <a href="/wiki/Free_software" title="Free software">free software</a> projects, such as the <a href="/wiki/GNOME" title="GNOME">GNOME</a> desktop, the <a href="/wiki/GTK%2B" title="GTK+">GTK+</a> toolkit and the <a href="/wiki/GIMP" title="GIMP">GIMP</a> image manipulation program.</p>
<p>Though many GObject applications are written entirely in C, the GObject system maps well into the native object systems of many other languages, like <a href="/wiki/C%2B%2B" title="C++">C++</a>, <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>, <a class="mw-redirect" href="/wiki/Ruby_programming_language" title="Ruby programming language">Ruby</a>, <a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a>, <a href="/wiki/Common_Lisp" title="Common Lisp">Common Lisp</a>, and <a href="/wiki/.NET_Framework" title=".NET Framework">.NET</a>/<a href="/wiki/Mono_(software)" title="Mono (software)">Mono</a>. As a result, it is usually relatively painless to create <a href="/wiki/Language_binding" title="Language binding">language bindings</a> for well-written libraries that use the GObject framework.</p>
<p>Writing GObject code in C in the first place, however, is relatively verbose. The library takes a good deal of time to learn, and programmers with experience in <a class="mw-redirect" href="/wiki/High-level" title="High-level">high-level</a> object-oriented languages often find it somewhat tedious to work with GObject in C. For example, creating a subclass (even just a subclass of <code>GObject</code>) can require writing and/or copying large amounts of <a href="/wiki/Boilerplate_code" title="Boilerplate code">boilerplate code</a>.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup> However, using <a href="/wiki/Vala_(programming_language)" title="Vala (programming language)">Vala</a>, a language that is designed primarily to work with GObject and which converts to C, is likely to make working with GObject or writing GObject based libraries nicer.</p>
<p>Although they are not really <a class="mw-redirect" href="/wiki/First-class_object" title="First-class object">first-class objects</a> (there are no actual metatypes in GType), <a href="/wiki/Metaobject" title="Metaobject">metaobjects</a> like classes and interfaces are created by GObject applications at runtime, and provide good support for <a class="mw-redirect" href="/wiki/Introspection_(computer_science)" title="Introspection (computer science)">introspection</a>. The introspective capabilities are used by language bindings and user interface design applications like <a href="/wiki/Glade_Interface_Designer" title="Glade Interface Designer">Glade</a> to allow doing things like loading a <a class="mw-redirect" href="/wiki/Shared_library" title="Shared library">shared library</a> that provides a GObject class—usually some kind of <a href="/wiki/Web_widget" title="Web widget">widget</a>, in the case of Glade—and then obtain a list of all properties of the class, complete with type information and documentation strings.</p>
<h2><span class="mw-headline" id="Comparisons_to_other_object_systems">Comparisons to other object systems</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GObject&amp;action=edit&amp;section=10" title="Edit section: Comparisons to other object systems">edit</a><span class="mw-editsection-bracket">]</span></span></h2>

<p>Since GObject provides a mostly complete object system for C, it can be seen as an alternative to C-derived languages such as <a href="/wiki/C%2B%2B" title="C++">C++</a> and <a href="/wiki/Objective-C" title="Objective-C">Objective-C</a>. (Though both also offer many other features beyond just their respective object systems.) An easily observed difference between C++ and GObject is that GObject (like Java) does not support <a href="/wiki/Multiple_inheritance" title="Multiple inheritance">multiple inheritance</a>.</p>
<p>GObject's use of <a href="/wiki/GLib" title="GLib">GLib</a>'s g_malloc() memory allocation function will cause the program to exit unconditionally upon memory exhaustion, unlike the C library's <a class="mw-redirect" href="/wiki/Malloc" title="Malloc">malloc</a>(), C++'s <a class="mw-redirect" href="/wiki/New_(C%2B%2B)" title="New (C++)">new</a>, and other common memory allocators which allow a program to cope with or even fully recover from out-of-memory situations without simply crashing. This tends to work against including GObject in software where resilience in the face of limited memory is important, or where very many or very large objects are commonly handled. The g_try_new() can be used when a memory allocation is more likely to fail (for a large object for example), but this cannot grant that the allocation will not fail elsewhere in the code.</p>
<p>Another important difference is that while C++ and Objective-C are separate languages, GObject is strictly a library and as such does not introduce any new syntax or compiler intelligence. For example, when writing GObject-based C code, it is frequently necessary to perform explicit <a class="mw-redirect" href="/wiki/Cast_(computer_science)" title="Cast (computer science)">upcasting</a>. Hence, “C with GObject”, considered as a language separate from plain C, is a strict superset of plain C — like Objective C, but unlike C++.</p>
<p>On platforms where there is no standard <a href="/wiki/Application_binary_interface" title="Application binary interface">ABI</a> that works across all C++ compilers (which is not usually the case, since either the Itanium ABI or the Microsoft ABI are usually followed), a library compiled with one C++ compiler is not always able to call a library compiled with a different one. If such compatibility is required, the C++ methods must be exported as plain C functions, partly defeating the purpose of the C++ object system. The problem occurs in part because different C++ compilers use different kinds of <a href="/wiki/Name_mangling" title="Name mangling">name mangling</a> to ensure the uniqueness of all exported symbols. (This is necessary because, for example, two different classes may have identically named member functions, one function name may be <a class="mw-redirect" href="/wiki/Method_overloading" title="Method overloading">overloaded</a> multiple times, or identically named functions may appear in different <a class="mw-redirect" href="/wiki/Namespaces" title="Namespaces">namespaces</a>, but in <a href="/wiki/Object_code" title="Object code">object code</a> these overlaps are not allowed.) In contrast, since C does not support any form of overloading or namespacing, authors of C libraries will typically use explicit prefixes to ensure the global uniqueness of their exported names. Hence, despite being object-oriented, a GObject-based library written in C will always use the same external symbol names regardless of which compiler is used.</p>
<p>Perhaps the most profound difference is GObject’s emphasis on signals (called <a href="/wiki/Event-driven_programming" title="Event-driven programming">events</a> in other languages). This emphasis derives from the fact that GObject was specifically designed to meet the needs of a GUI toolkit. Whilst there are signal libraries for most object-oriented languages out there, in the case of GObject it is built into the object system. Because of this, a typical GObject application will tend to use signals to a much larger extent than a non-GObject application would, making GObject <a class="mw-redirect" href="/wiki/Software_componentry" title="Software componentry">components</a> much more <a href="/wiki/Information_hiding" title="Information hiding">encapsulated</a> and reusable than the ones using plain C++ or Java. If using <a class="new" href="/w/index.php?title=Glibmm&amp;action=edit&amp;redlink=1" title="Glibmm (page does not exist)">glibmm</a>/<a href="/wiki/Gtkmm" title="Gtkmm">gtkmm</a>, the official C++ wrappers to Glib/GTK+ respectively, the sibling project <a href="/wiki/Libsigc%2B%2B" title="Libsigc++">libsigc++</a> allows easy use of underlying GObject signals using standard C++. Of course, other implementations of signals are available on almost all platforms, although sometimes an extra library is needed, such as Boost.Signals2 for C++.</p>
