<h1 class="firstHeading" id="firstHeading" lang="en">GUID Partition Table</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>

Diagram illustrating the layout of the GPT scheme. In this example, each logical block is 512 bytes in size, and each partition entry is 128 bytes, and the corresponding partition entries are assumed to be located in <a href="/wiki/Logical_block_addressing" title="Logical block addressing">LBA</a> 2-33. <p><b>GUID Partition Table</b> (<b>GPT</b>) is a standard for the layout of the <a href="/wiki/Partition_table" title="Partition table">partition table</a> on a physical <a class="mw-redirect" href="/wiki/Computer_storage_device" title="Computer storage device">storage device</a> used in a desktop or server PC, such as a <a href="/wiki/Hard_disk_drive" title="Hard disk drive">hard disk drive</a> or <a href="/wiki/Solid-state_drive" title="Solid-state drive">solid-state drive</a>, using <a class="mw-redirect" href="/wiki/Globally_unique_identifier" title="Globally unique identifier">globally unique identifiers</a> (GUID). Although it forms a part of the <a href="/wiki/Unified_Extensible_Firmware_Interface" title="Unified Extensible Firmware Interface">Unified Extensible Firmware Interface</a> (UEFI) standard (<a href="/wiki/Unified_EFI_Forum" title="Unified EFI Forum">Unified EFI Forum</a> proposed replacement for the <a class="mw-redirect" href="/wiki/IBM_PC" title="IBM PC">PC</a> <a href="/wiki/BIOS" title="BIOS">BIOS</a>), it is also used on some BIOS systems because of the limitations of <a href="/wiki/Master_boot_record" title="Master boot record">master boot record</a> (MBR) partition tables, which use 32 bits for storing <a href="/wiki/Logical_block_addressing" title="Logical block addressing">logical block addresses</a> (LBA) and size information on a traditionally 512 byte <a href="/wiki/Disk_sector" title="Disk sector">disk sector</a>.</p>
<p>All modern PC <a href="/wiki/Operating_system" title="Operating system">operating systems</a> support GPT. Some, including <a class="mw-redirect" href="/wiki/OS_X" title="OS X">OS X</a> and <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a> on x86, support booting from GPT partitions only on systems with EFI firmware, but <a href="/wiki/FreeBSD" title="FreeBSD">FreeBSD</a> and most <a href="/wiki/Linux_distribution" title="Linux distribution">Linux distributions</a> can boot from GPT partitions on systems with both legacy BIOS firmware interface and EFI.</p>

<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GUID_Partition_Table&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="hatnote" role="note">Main articles: <a href="/wiki/INT_13H" title="INT 13H">INT 13H</a> and <a class="mw-redirect" href="/wiki/Enhanced_BIOS" title="Enhanced BIOS">Enhanced BIOS</a></div>
<p>The widespread MBR partitioning scheme, dating from the early 1980s, imposed limitations that affect the use of modern hardware. One of the main limitations is the usage of 32 bits for storing block addresses and quantity information. For hard disks with 512-byte sectors, the MBR partition table entries allow up to a maximum of 2 <a href="/wiki/Tebibyte" title="Tebibyte">TiB</a> (2<sup>32</sup> × 512 <a href="/wiki/Byte" title="Byte">bytes</a>).<sup class="reference" id="cite_ref-UEFIDrivePartitionFAQ_1-0"><a href="#cite_note-UEFIDrivePartitionFAQ-1">[1]</a></sup></p>
<p><a href="/wiki/Intel" title="Intel">Intel</a> therefore developed a new partition table format in the late 1990s as part of what eventually became UEFI. As of 2010<sup class="plainlinks noprint asof-tag update" style="display:none;"><a class="external text" href="//en.wikipedia.org/w/index.php?title=GUID_Partition_Table&amp;action=edit">[update]</a></sup>, GPT forms a subset of the UEFI specification.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> GPT allocates 64 bits for logical block addresses, therefore allowing a maximum disk size of 2<sup>64</sup> sectors. For disks with 512-byte sectors, maximum size is 9.4 <a href="/wiki/Zettabyte" title="Zettabyte">ZB</a> (9.4 × 10<sup>21</sup> bytes) or 8 <a class="mw-redirect" href="/wiki/ZiB" title="ZiB">ZiB</a> (9,444,732,965,739,290,427,392 bytes, coming from 18,446,744,073,709,551,616 (2<sup>64</sup>) sectors × 512 (2<sup>9</sup>) bytes per sector).<sup class="reference" id="cite_ref-UEFIDrivePartitionFAQ_1-1"><a href="#cite_note-UEFIDrivePartitionFAQ-1">[1]</a></sup><sup class="reference" id="cite_ref-IBMLargeDrivesGPT_3-0"><a href="#cite_note-IBMLargeDrivesGPT-3">[3]</a></sup></p>
<h2><span class="mw-headline" id="Features">Features</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GUID_Partition_Table&amp;action=edit&amp;section=2" title="Edit section: Features">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>MBR-based partition table schemes insert the partitioning information for (usually) four "primary" partitions in the MBR (which on a BIOS system is also the container for code that begins the process of booting the system). In a GPT, the first sector of the disk is reserved for a "protective MBR" such that booting a BIOS-based computer from a GPT disk is supported, but the bootloader and operating system must both be GPT-aware. Regardless of the sector size, the GPT <a href="/wiki/Header_(computing)" title="Header (computing)">header</a> begins on the second logical block of the device.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (November 2016)">citation needed</span></a></i>]</sup></p>
<p>Like modern MBRs, GPTs use <a href="/wiki/Logical_block_addressing" title="Logical block addressing">logical block addressing</a> (LBA) in place of the historical <a href="/wiki/Cylinder-head-sector" title="Cylinder-head-sector">cylinder-head-sector</a> (CHS) addressing. The protective MBR is contained in LBA 0, the GPT header is in LBA 1, and the GPT header has a <a href="/wiki/Pointer_(computer_programming)" title="Pointer (computer programming)">pointer</a> to the partition table, or Partition Entry Array, typically LBA 2. The UEFI specification stipulates that a minimum of 16,384 bytes, regardless of sector size, be allocated for the Partition Entry Array.<sup class="reference" id="cite_ref-uefi_spec_4-0"><a href="#cite_note-uefi_spec-4">[4]</a></sup> On a disk having 512-byte sectors, a partition entry array size of 16,384 bytes and the minimum size of 128 bytes for each partition entry, LBA 34 is the first usable sector on the disk.</p>
<p>Hard-disk manufacturers are transitioning to <a href="/wiki/Advanced_Format" title="Advanced Format">4,096-byte sectors</a>. The first such drives continued to present 512-byte physical sectors to the OS, so degraded performance could result when the drive's physical <a class="mw-redirect" href="/wiki/4_KB_sector_alignment" title="4 KB sector alignment">4 KB sector boundaries</a> did not coincide with the 4 KB logical blocks, clusters and virtual memory pages common in many operating systems and file systems. This was a particular problem on writes, when the drive is forced to perform two read-modify-write operations to satisfy a single misaligned 4 KB write operation.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup></p>
<p>For <a href="/wiki/Backward_compatibility" title="Backward compatibility">backward compatibility</a> with most legacy operating systems such as <a href="/wiki/DOS" title="DOS">DOS</a>, <a href="/wiki/OS/2" title="OS/2">OS/2</a>, and versions of Windows before <a href="/wiki/Windows_Vista" title="Windows Vista">Vista</a>, MBR partitions must always start on <a class="mw-redirect" href="/wiki/Track_partition_alignment" title="Track partition alignment">track boundaries</a> according to the traditional CHS addressing scheme and end on a <a class="mw-redirect" href="/wiki/Cylinder_partition_alignment" title="Cylinder partition alignment">cylinder boundary</a>. This is also true of partitions with emulated CHS geometries (as reflected by the BIOS and the CHS sectors entries in the MBR partition table) or partitions accessed only via LBA. <a class="mw-redirect" href="/wiki/Extended_partition" title="Extended partition">Extended partitions</a> must start on cylinder boundaries as well. This typically causes the first <a class="mw-redirect" href="/wiki/Primary_partition" title="Primary partition">primary partition</a> to start at LBA 63 on disks accessed via LBA, leaving a gap of 62 sectors with MBR-based disks, sometimes called "MBR gap", "boot track", or "embedding area". That otherwise unused disk space is commonly used by bootloaders such as <a href="/wiki/GNU_GRUB" title="GNU GRUB">GRUB</a> for storing their second stages.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup></p>
<h2><span class="mw-headline" id="MBR_variants">MBR variants</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GUID_Partition_Table&amp;action=edit&amp;section=3" title="Edit section: MBR variants">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Protective_MBR_.28LBA_0.29"><span id="PROTECTIVE-MBR"></span>Protective MBR (LBA 0)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GUID_Partition_Table&amp;action=edit&amp;section=4" title="Edit section: Protective MBR (LBA 0)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>For limited backward compatibility, the space of the legacy MBR is still reserved in the GPT specification, but it is now used in a way that prevents MBR-based disk utilities from misrecognizing and possibly overwriting GPT disks. This is referred to as a <i>protective MBR</i>.<sup class="reference" id="cite_ref-IBMLargeDrivesGPT_3-1"><a href="#cite_note-IBMLargeDrivesGPT-3">[3]</a></sup></p>
<p>A single partition type of <span style="font-family: monospace, monospace;"><a href="/wiki/Partition_type#PID_EEh" title="Partition type">EEh</a></span>, encompassing the entire GPT drive (where "entire" actually means as much of the drive as can be represented in an MBR), is indicated and identifies it as GPT. Operating systems and tools which cannot read GPT disks will generally recognize the disk as containing one partition of unknown type and no empty space, and will typically refuse to modify the disk unless the user explicitly requests and confirms the deletion of this partition. This minimizes accidental erasures.<sup class="reference" id="cite_ref-IBMLargeDrivesGPT_3-2"><a href="#cite_note-IBMLargeDrivesGPT-3">[3]</a></sup> Furthermore, GPT-aware OSes may check the protective MBR and if the enclosed partition type is not of type <span style="font-family: monospace, monospace;">EEh</span> or if there are multiple partitions defined on the target device, the OS may refuse to manipulate the partition table.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup></p>
<p>While the MBR and protective MBR layouts were defined around 512 bytes per sector, the actual sector size can be larger on various media such as <a class="mw-redirect" href="/wiki/MO_disk" title="MO disk">MO disks</a> or hard disks with <a href="/wiki/Advanced_Format" title="Advanced Format">Advanced Format</a>. Extra space in the MBR typically remains unused.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (November 2016)">citation needed</span></a></i>]</sup></p>
<p>If the actual size of the disk exceeds the maximum partition size representable using the legacy 32-bit LBA entries in the MBR partition table, the recorded size of this partition is clipped at the maximum, thereby ignoring the rest of disk. This amounts to a maximum reported size of 2 TB, assuming a disk with 512 bytes per sector (see <a class="mw-redirect" href="/wiki/512e" title="512e">512e</a>). It would result in 16 TB with 4 KB sectors (<a class="mw-redirect" href="/wiki/4Kn" title="4Kn">4Kn</a>), but since many older operating systems and tools are hard wired for a sector size of 512 bytes or are limited to 32-bit calculations, exceeding the 2 TB limit could cause compatibility problems.<sup class="reference" id="cite_ref-IBMLargeDrivesGPT_3-3"><a href="#cite_note-IBMLargeDrivesGPT-3">[3]</a></sup></p>
<h3><span class="mw-headline" id="Hybrid_MBR_.28LBA_0_.2B_GPT.29">Hybrid MBR (LBA 0 + GPT)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=GUID_Partition_Table&amp;action=edit&amp;section=5" title="Edit section: Hybrid MBR (LBA 0 + GPT)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In operating systems that support GPT-based boot through BIOS services rather than EFI, the first sector is also still used to store the first stage of the bootloader code, but modified to recognize GPT partitions. The bootloader in the MBR must not assume a sector size of 512 bytes.<sup class="reference" id="cite_ref-IBMLargeDrivesGPT_3-4"><a href="#cite_note-IBMLargeDrivesGPT-3">[3]</a></sup></p>

<p>The partition table header defines the usable blocks on the disk. It also defines the number and size of the partition entries that make up the partition table. The EFI stipulates a minimum of 16,384 bytes be reserved for the partition table array, so there are 128 partition entries reserved, each 128 bytes long.</p>
<p>The header contains the disk GUID. It records its own size and location (always LBA 1) and the size and location of the secondary GPT header and table (always the last sectors on the disk). Importantly, it also contains a <a class="mw-redirect" href="/wiki/CRC32" title="CRC32">CRC32</a> checksum for itself and for the partition table, which may be verified by the firmware, bootloader, or operating system on boot. Because of this, hex editors should not be used to modify the contents of the GPT. Such modification would render the checksum invalid. In this case, the primary GPT may be overwritten with the secondary one by disk recovery software. If both GPTs contain invalid checksums, many bootloaders (those governed by an integrity model in particular) and operating systems will refuse to work with the disk until the corrupted partition tables are repaired or removed.</p>

<p>The values for current and backup LBAs of the primary header should be the second sector of the disk (LBA 1) and the last sector of the disk, respectively. The secondary header at the end of the disk identifies its own table of partition entries, which is located directly before that header.</p>
<p>Since the primary header must be located at LBA 1, it will not necessarily be physically contiguous with the MBR; on an <a href="/wiki/Advanced_Format" title="Advanced Format">Advanced Format</a> disk with 4 KB sectors, the header will be located at the byte 4096 from the beginning of the disk, leaving a gap of unused space between it and the MBR. On such a disk, the byte 512 that directly follows the MBR is still part of LBA 0. However, a disk with 512-byte sectors will store its GPT header at byte 512 because, as such, that position corresponds to LBA 1.</p>

<p>After the header, the Partition Entry Array describes partitions, using a minimum size of 128 bytes for each entry block.<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[8]</a></sup> The starting location of the array on disk, and the size of each entry, are given in the GPT header. The first 16 bytes of each entry designate the partition type's globally unique identifier (GUID). For example, the GUID for an <a href="/wiki/EFI_system_partition" title="EFI system partition">EFI system partition</a> is <span style="font-family: monospace, monospace;">C12A7328-F81F-11D2-BA4B-00A0C93EC93B</span>. The second 16 bytes are a GUID unique to the partition. Then follow the starting and ending 64 bit LBAs, partition attributes, and the 36 character (max) <a href="/wiki/Unicode" title="Unicode">Unicode</a> partition name. As is the nature and purpose of GUIDs, no central registry is needed to ensure the uniqueness of the GUID partition type designators.</p>
<p>Also, the sector size must not be assumed to be hard-wired to 512 bytes per sector in calculations (see <a href="/wiki/Advanced_Format" title="Advanced Format">Advanced Format</a>), that is, there can be more than four partition entries in a single sector, and (with much larger partition table entries possible in the future) it is possible to have a sector hold only a fraction of a partition entry. Except for the first two sectors (LBA 0 and LBA 1), the GPT specification describes the size and organization of a data structure, but intentionally does <i>not</i> specify the number of sectors on disk that are used to store it.</p>

<div class="reflist columns references-column-width" style="-moz-column-width: 30em; -webkit-column-width: 30em; column-width: 30em; list-style-type: lower-alpha;">
<ol class="references">
<li id="cite_note-8"><span class="mw-cite-backlink"><b><a href="#cite_ref-8">^</a></b></span> <span class="reference-text">Adding <code>ULL</code> suffix to an integer constant makes it of type <code>unsigned long long int</code>.</span></li>
<li id="cite_note-23"><span class="mw-cite-backlink"><b><a href="#cite_ref-23">^</a></b></span> <span class="reference-text">Only if using its service pack 1 or 2</span></li>
<li id="cite_note-24"><span class="mw-cite-backlink"><b><a href="#cite_ref-24">^</a></b></span> <span class="reference-text">In a multi-disk setup, non-UEFI bootloader (boot drive) requires MBR-based partitioning, while a system drive can use GUID partitioning.</span></li>
<li id="cite_note-25"><span class="mw-cite-backlink"><b><a href="#cite_ref-25">^</a></b></span> <span class="reference-text">The GUIDs in this table are written assuming a <a class="mw-redirect" href="/wiki/Little-endian" title="Little-endian">little-endian</a> <a class="mw-redirect" href="/wiki/Byte_order" title="Byte order">byte order</a>. For example, the GUID for an EFI System partition is written as <span style="font-family: monospace, monospace;">C12A7328-F81F-11D2-BA4B-00A0C93EC93B</span> here, which corresponds to the 16 byte sequence <span style="font-family: monospace, monospace;">28h 73h 2Ah C1h 1Fh F8h D2h 11h BAh 4Bh 00h A0h C9h 3Eh C9h 3Bh</span> –  only the first three blocks are byte-swapped.</span></li>
<li id="cite_note-26"><span class="mw-cite-backlink"><b><a href="#cite_ref-26">^</a></b></span> <span class="reference-text">The formation of this GUID does not follow the GUID definition; it is formed by using the <a href="/wiki/ASCII" title="ASCII">ASCII</a> codes for the string "<span style="font-family: monospace, monospace;">Hah!IdontNeedEFI</span>". Such formation of "GUID" value breaks down the guaranteed uniqueness of GUID.</span></li>
<li id="cite_note-manufacturerbootpartition-30"><span class="mw-cite-backlink">^ <a href="#cite_ref-manufacturerbootpartition_30-0"><sup><i><b>a</b></i></sup></a> <a href="#cite_ref-manufacturerbootpartition_30-1"><sup><i><b>b</b></i></sup></a></span> <span class="reference-text">Some computer manufacturers have their own GUIDs for partitions that are analogous to the EFI System Partition, but that hold boot loaders to launch manufacturer-specific recovery tools.<sup class="reference" id="cite_ref-29"><a href="#cite_note-29">[24]</a></sup></span></li>
<li id="cite_note-linwin-32"><span class="mw-cite-backlink">^ <a href="#cite_ref-linwin_32-0"><sup><i><b>a</b></i></sup></a> <a href="#cite_ref-linwin_32-1"><sup><i><b>b</b></i></sup></a></span> <span class="reference-text">Previously, Linux used the same GUID for the data partitions as Windows (Basic data partition: <span style="font-family: monospace, monospace;">EBD0A0A2-B9E5-4433-87C0-68B6B72699C7</span>). Linux never had a separate unique partition type GUID defined for its data partitions. This created problems when dual-booting Linux and Windows in UEFI-GPT setup. The new GUID (Linux filesystem data: <span style="font-family: monospace, monospace;">0FC63DAF-8483-4772-8E79-3D69D8477DE4</span>) was defined jointly by GPT fdisk and GNU Parted developers.<sup class="reference" id="cite_ref-RodSmithMLpost_31-0"><a href="#cite_note-RodSmithMLpost-31">[25]</a></sup> It is identified as type code <span style="font-family: monospace, monospace;">0x8300</span> in GPT fdisk. (See definitions in <a class="external text" href="http://gptfdisk.git.sourceforge.net/git/gitweb.cgi?p=gptfdisk/gptfdisk;a=blob_plain;f=parttypes.cc;hb=HEAD" rel="nofollow">gdisk's parttypes.cc</a>)</span></li>
<li id="cite_note-solmac-34"><span class="mw-cite-backlink">^ <a href="#cite_ref-solmac_34-0"><sup><i><b>a</b></i></sup></a> <a href="#cite_ref-solmac_34-1"><sup><i><b>b</b></i></sup></a></span> <span class="reference-text">The GUID for <code>/usr</code> on Solaris is used as a generic GUID for ZFS by OS X.</span></li>
<li id="cite_note-netbsd-36"><span class="mw-cite-backlink">^ <a href="#cite_ref-netbsd_36-0"><sup><i><b>a</b></i></sup></a> <a href="#cite_ref-netbsd_36-1"><sup><i><b>b</b></i></sup></a></span> <span class="reference-text">NetBSD and MidnightBSD had used the FreeBSD GUIDs before their unique GUIDs were created.</span></li>

<li id="cite_note-2"><span class="mw-cite-backlink"><b><a href="#cite_ref-2">^</a></b></span> <span class="reference-text"><cite class="citation journal">Nikkel, Bruce J. (September 2009). The MBR, located in sector zero, contains the initial boot code and a four entry partition table (Microsoft, 1983). Intended to solve booting and partitioning limitations with newer hardware, a replacement for both the BIOS and the MBR partition table was developed by Intel in the late 1990s (Intel, 2000). This is now called the Unified EFI (UEFI, 2008 UEFI Forum. Unified extensible firmware interface specification version 2.2 2008.UEFI, 2008) specification, and managed by the UEFI Forum (UEFI, 2009). A subset of this specification includes GPT, intended to replace the DOS/MBR partition tables.</q></cite><span class="Z3988" title="ctx_ver=Z39.88-2004&amp;rfr_id=info%3Asid%2Fen.wikipedia.org%3AGUID+Partition+Table&amp;rft.atitle=Forensic+analysis+of+GPT+disks+and+GUID+partition+tables&amp;rft.aufirst=Bruce+J.&amp;rft.aulast=Nikkel&amp;rft.date=2009-09&amp;rft.genre=article&amp;rft_id=info%3Adoi%2F10.1016%2Fj.diin.2009.07.001&amp;rft.issue=1-2&amp;rft.jtitle=Digital+Investigation&amp;rft.pages=39-47&amp;rft_val_fmt=info%3Aofi%2Ffmt%3Akev%3Amtx%3Ajournal&amp;rft.volume=6"><span style="display:none;"> </span></span></span></li>
