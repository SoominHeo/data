<p>The <b>ICO file format</b> is an <a class="mw-redirect" href="/wiki/Image_file_format" title="Image file format">image file format</a> for <a class="mw-redirect" href="/wiki/Computer_icon" title="Computer icon">computer icons</a> in <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a>. ICO files contain one or more small images at multiple sizes and <a href="/wiki/Color_depth" title="Color depth">color depths</a>, such that they may be <a href="/wiki/Image_scaling" title="Image scaling">scaled</a> appropriately. In Windows, all <a href="/wiki/Executable" title="Executable">executables</a> that display an icon to the user, on the desktop, in the Start Menu, or in Windows Explorer, must carry the icon in ICO format.</p>
<p>The <b>CUR file format</b> is an almost identical image file format for non-animated <a class="mw-redirect" href="/wiki/Cursor_(computing)" title="Cursor (computing)">cursors</a> in Microsoft Windows. The only differences between these two file formats are the bytes used to identify them and the addition of a hotspot in the CUR format header; the hotspot is defined as the pixel offset (in x,y coordinates) from the top-left corner of the cursor image where the user is actually pointing the mouse.</p>

<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ICO_(file_format)&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Icons introduced in <a href="/wiki/Windows_1.0" title="Windows 1.0">Windows 1.0</a> were 32×32 pixels in size and were monochrome.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> Support for 16 colors was introduced in <a href="/wiki/Windows_3.0" title="Windows 3.0">Windows 3.0</a>.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (January 2015)">citation needed</span></a></i>]</sup></p>
<p><a class="mw-redirect" href="/wiki/Win32" title="Win32">Win32</a> introduced support for storing icon images of up to <a href="/wiki/RGB_color_model#The_24-bit_RGB_representation" title="RGB color model">16.7 million colors (Truecolor)</a> and up to 256x256 pixels in dimensions.<sup class="reference" id="cite_ref-win95_3-0"><a href="#cite_note-win95-3">[3]</a></sup> Windows 95 also introduced a new Device Independent Bitmap (DIB) engine.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup> However, 256 color was the default icon color depth in Windows 95. It was possible to enable <a href="/wiki/RGB_color_model#16-bit_RGB_.28Highcolor.29" title="RGB color model">65535 color (Highcolor)</a> icons by either modifying the <i>Shell Icon BPP</i> value in the registry<sup class="reference" id="cite_ref-win95_3-1"><a href="#cite_note-win95-3">[3]</a></sup><sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> or by purchasing <a href="/wiki/Microsoft_Plus!" title="Microsoft Plus!">Microsoft Plus!</a> for Windows 95. The <i>Shell Icon Size</i> value allows using larger icons in place of 32×32 icons and the <i>Shell Small Icon Size</i> value allows using custom sizes in place of 16×16 icons.<sup class="reference" id="cite_ref-win95_3-2"><a href="#cite_note-win95-3">[3]</a></sup> Thus, a single icon file could store images of any size from 1×1 pixel up to 256×256 pixels (including non-square sizes) with 2(Rarely used),16, 256, 65535, or 16.7 million colors; but the shell could not display very large sized icons. The notification area of the Windows <a href="/wiki/Taskbar" title="Taskbar">taskbar</a> was limited to 16 color icons by default until <a class="mw-redirect" href="/wiki/Windows_Me" title="Windows Me">Windows Me</a> when it was updated to support high color icons.</p>
<p><a href="/wiki/Windows_XP" title="Windows XP">Windows XP</a> added support for <a href="/wiki/RGB_color_model#32-bit_RGBA_.28RGB_plus_alpha_channel.29" title="RGB color model">32-bit color</a> (16.7 million colors plus alpha channel transparency) icon images, thus allowing semitransparent areas like <a href="/wiki/Drop_shadow" title="Drop shadow">shadows</a>, <a href="/wiki/Spatial_anti-aliasing" title="Spatial anti-aliasing">anti-aliasing</a>, and glass-like effects to be drawn in an icon. Windows XP, by default, employs 48×48 pixel icons in <a class="mw-redirect" href="/wiki/Windows_Explorer" title="Windows Explorer">Windows Explorer</a>. Windows XP can be forced to use icons as large as 256×256 by modifying the <i>Shell icon size</i> value but this would cause all 32×32 icons throughout the shell to be upscaled.<sup class="reference" id="cite_ref-win95_3-3"><a href="#cite_note-win95-3">[3]</a></sup> Microsoft only recommended icon sizes up to 48×48 pixels for Windows XP.<sup class="reference" id="cite_ref-winxp_6-0"><a href="#cite_note-winxp-6">[6]</a></sup> Windows XP can downscale 256×256 and larger icons if no closer image size is available.<sup class="reference" id="cite_ref-win95_3-4"><a href="#cite_note-win95-3">[3]</a></sup></p>
<p><a href="/wiki/Windows_Vista" title="Windows Vista">Windows Vista</a> added a 256×256-pixel icon view to Windows Explorer, as well as support for the <a href="/wiki/Data_compression" title="Data compression">compressed</a> <a href="/wiki/Portable_Network_Graphics" title="Portable Network Graphics">PNG</a> format. Although compression is not required, Microsoft recommends that all 256×256 icons in ICO files should be stored in PNG format to reduce the overall size of the file.<sup class="reference" id="cite_ref-vista_7-0"><a href="#cite_note-vista-7">[7]</a></sup> The Windows Vista Explorer supports smoothly scaling icons to non-standard sizes which are rendered on the fly even if an image is not present for that size in the icon file. The Windows Vista shell adds a slider for "zooming" the icon sizes in and out. With users using higher resolutions and high DPI modes, larger icon formats (such as 256×256) are recommended.<sup class="reference" id="cite_ref-vista_7-1"><a href="#cite_note-vista-7">[7]</a></sup></p>
<h2><span class="mw-headline" id="MIME_type">MIME type</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ICO_(file_format)&amp;action=edit&amp;section=2" title="Edit section: MIME type">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>While the <a href="/wiki/Internet_Assigned_Numbers_Authority" title="Internet Assigned Numbers Authority">IANA</a>-registered <a class="mw-redirect" href="/wiki/MIME_type" title="MIME type">MIME type</a> for ICO files is <code>image/vnd.microsoft.icon</code>,<sup class="reference" id="cite_ref-Butcher_8-0"><a href="#cite_note-Butcher-8">[8]</a></sup> it was submitted to IANA in 2003 by a third party and is not recognised by Microsoft software, which uses <code>image/x-icon</code> instead.<sup class="reference" id="cite_ref-ericlaw_9-0"><a href="#cite_note-ericlaw-9">[9]</a></sup> Erroneous types <code>image/ico</code>, <code>image/icon</code>, <code>text/ico</code> and <code>application/ico</code> have also been seen in use.<sup class="reference" id="cite_ref-Butcher_8-1"><a href="#cite_note-Butcher-8">[8]</a></sup></p>
<h2><span class="mw-headline" id="Icon_resource_structure">Icon resource structure</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ICO_(file_format)&amp;action=edit&amp;section=3" title="Edit section: Icon resource structure">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>An ICO or CUR file is made up of an ICONDIR ("Icon directory") structure, containing an ICONDIRENTRY structure for each image in the file, followed by a contiguous block of all image bitmap data (which may be in either <a href="/wiki/BMP_file_format" title="BMP file format">Windows BMP</a> format, excluding the BITMAPFILEHEADER structure, or in <a href="/wiki/Portable_Network_Graphics" title="Portable Network Graphics">PNG</a> format, stored in its entirety).<sup class="reference" id="cite_ref-win95_3-5"><a href="#cite_note-win95-3">[3]</a></sup></p>
<p>Images with less than 32 bits of color depth follow a particular format: the image is encoded as a single image consisting of a color mask (the "XOR mask") together with an opacity mask (the "AND mask").<sup class="reference" id="cite_ref-winxp_6-1"><a href="#cite_note-winxp-6">[6]</a></sup> The XOR mask must precede the AND mask inside the bitmap data; if the image is stored in bottom-up order (which it most likely is), the XOR mask would be drawn below the AND mask. The AND mask is 1 bit per pixel, regardless of the color depth specified by the BMP header, and specifies which pixels are fully transparent and which are fully opaque. The XOR mask conforms to the bit depth specified in the BMP header and specifies the numerical color or palette value for each pixel. Together, the AND mask and XOR mask make for a non-transparent image representing an image with 1-bit transparency; they also allow for inversion of the background. The height for the image in the ICONDIRENTRY structure of the ICO/CUR file takes on that of the intended image dimensions (after the masks are composited), whereas the height in the BMP header takes on that of the two mask images combined (before they are composited). Therefore, the masks must each be of the same dimensions, and the height specified in the BMP header must be exactly twice the height specified in the ICONDIRENTRY structure.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (January 2015)">citation needed</span></a></i>]</sup></p>
<p>32-bit images (including 32-bit BITMAPINFOHEADER-format BMP images<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[Notes 1]</a></sup>) are specifically a 24-bit image with the addition of an 8-bit channel for <a href="/wiki/Alpha_compositing" title="Alpha compositing">alpha compositing</a>. Thus, in 32-bit images, the AND mask is not required, but recommended for consideration. Windows XP and higher will use a 32-bit image in less than <a class="mw-redirect" href="/wiki/True_Color" title="True Color">True color</a> mode by constructing an AND mask based on the alpha channel (if one does not reside with the image already) if no 24-bit version of the image is supplied in the ICO/CUR file. However, earlier versions of Windows interpret all pixels with 100% opacity unless an AND mask is supplied with the image. Supplying a custom AND mask will also allow for tweaking and hinting by the icon author. Even if the AND mask is not supplied, if the image is in Windows BMP format, the BMP header must still specify a doubled height.</p>

<p><b>Referenced image data</b></p>
<p>All image data referenced by entries in the image directory proceed directly after the image directory. It is customary practice to store them in the same order as defined in the image directory.</p>
<p>Recall that if an image is stored in BMP format, it must exclude the opening BITMAPFILEHEADER structure, whereas if it is stored in PNG format, it must be stored in its entirety.</p>
<h3><span class="mw-headline" id="PNG_format">PNG format</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ICO_(file_format)&amp;action=edit&amp;section=5" title="Edit section: PNG format">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The ability to read <a href="/wiki/Portable_Network_Graphics" title="Portable Network Graphics">PNG</a> images from ICO and CUR format images was introduced in <a href="/wiki/Windows_Vista" title="Windows Vista">Windows Vista</a>.<sup class="reference" id="cite_ref-vista_7-2"><a href="#cite_note-vista-7">[7]</a></sup> A PNG image can be stored in the image in the same way as done for a standard Windows BMP format image, with the exception that the PNG image must be stored in its entirety, with its file header.</p>
<h2><span class="mw-headline" id="Icon_library">Icon library</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ICO_(file_format)&amp;action=edit&amp;section=6" title="Edit section: Icon library">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>An <b>icon library</b> is a way to package Windows icons. It is typically a 16-bit <a href="/wiki/New_Executable" title="New Executable">New Executable</a> or a 32-bit <a href="/wiki/Portable_Executable" title="Portable Executable">Portable Executable</a> binary file having .ICL extension with icon <a href="/wiki/Resource_(Windows)" title="Resource (Windows)">resources</a> being the packaged icons. <a href="/wiki/Windows_Vista" title="Windows Vista">Windows Vista</a> and later versions do not support viewing icons from 16-bit (<a href="/wiki/New_Executable" title="New Executable">New Executable</a>) files.<sup class="reference" id="cite_ref-14"><a href="#cite_note-14">[10]</a></sup></p>

<h2><span class="mw-headline" id="Notes">Notes</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ICO_(file_format)&amp;action=edit&amp;section=8" title="Edit section: Notes">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<ol class="references">
<li id="cite_note-10"><span class="mw-cite-backlink"><b><a href="#cite_ref-10">^</a></b></span> <span class="reference-text">The classic BITMAPINFOHEADER bitmap format supports storing images with 32 bits per pixel. When saved as a standalone .BMP file, "the high byte in each [pixel] is not used". However, when this same data is stored inside a ICO or CUR file, Windows XP (the first Windows version to support ICO/CUR files with more than 1 bit of transparency) and above interpret this byte as an alpha value.</span></li>
<li id="cite_note-11"><span class="mw-cite-backlink"><b><a href="#cite_ref-11">^</a></b></span> <span class="reference-text">Although Microsoft's technical documentation states that this value must be zero, the icon encoder built into .NET (System.Drawing.Icon.Save) sets this value to 255. It appears that the operating system ignores this value altogether.</span></li>
<li id="cite_note-12"><span class="mw-cite-backlink"><b><a href="#cite_ref-12">^</a></b></span> <span class="reference-text">Setting the color planes to 0 or 1 is treated equivalently by the operating system, but if the color planes are set higher than 1, this value should be multiplied by the bits per pixel to determine the final color depth of the image. It is unknown if the various Windows operating system versions are resilient to different color plane values.</span></li>
<li id="cite_note-13"><span class="mw-cite-backlink"><b><a href="#cite_ref-13">^</a></b></span> <span class="reference-text">The bits per pixel might be set to zero, but can be inferred from the other data; specifically, if the bitmap is not PNG compressed, then the bits per pixel can be calculated based on the length of the bitmap data relative to the size of the image. If the bitmap is PNG compressed, the bits per pixel are stored within the PNG data. It is unknown if the various Windows operating system versions contain logic to infer the bit depth for all possibilities if this value is set to zero.</span></li>
</ol>
