<h1 class="firstHeading" id="firstHeading" lang="en">NaN</h1>

<p>In <a href="/wiki/Computing" title="Computing">computing</a>, <b>NaN</b>, standing for <b>not a number</b>, is a numeric <a href="/wiki/Data_type" title="Data type">data type</a> value representing an undefined or unrepresentable value, especially in <a href="/wiki/Floating_point" title="Floating point">floating-point</a> calculations. Systematic use of NaNs was introduced by the <a href="/wiki/IEEE_floating_point" title="IEEE floating point">IEEE 754</a> floating-point standard in 1985, along with the representation of other non-finite quantities like <a href="/wiki/Infinity" title="Infinity">infinities</a>.</p>
<p>Two separate kinds of NaNs are provided, termed quiet NaNs and signaling NaNs. Quiet NaNs are used to propagate errors resulting from invalid operations or values, whereas signaling NaNs can support advanced features such as mixing numerical and <a href="/wiki/Symbolic_computation" title="Symbolic computation">symbolic computation</a> or other extensions to basic floating-point arithmetic. For example, <a href="/wiki/Indeterminate_form" title="Indeterminate form">0/0</a> is undefined as a <a href="/wiki/Real_number" title="Real number">real number</a>, and so represented by NaN; the square root of a negative number is <a href="/wiki/Imaginary_number" title="Imaginary number">imaginary</a>, and thus not representable as a real floating-point number, and so is represented by NaN; and NaNs may be used to represent missing values in computations.<sup class="reference" id="cite_ref-idl_1-0"><a href="#cite_note-idl-1">[1]</a></sup><sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup></p>

<p>In floating-point calculations, NaN is not the same as <a href="/wiki/Extended_real_number_line" title="Extended real number line">infinity</a>, although both are typically handled as special cases in floating-point representations of real numbers as well as in floating-point operations. An invalid operation is also not the same as an <a href="/wiki/Arithmetic_overflow" title="Arithmetic overflow">arithmetic overflow</a> (which might return an infinity) or an <a href="/wiki/Arithmetic_underflow" title="Arithmetic underflow">arithmetic underflow</a> (which would return the smallest <a href="/wiki/Normal_number_(computing)" title="Normal number (computing)">normal number</a>, a <a href="/wiki/Denormal_number" title="Denormal number">denormal number</a>, or <a class="mw-redirect" href="/wiki/0_(number)" title="0 (number)">zero</a>).</p>
<p><a href="/wiki/IEEE_floating_point" title="IEEE floating point">IEEE 754</a> NaNs are represented with the exponent field filled with ones (like infinity values), and some non-zero number in the <a href="/wiki/Significand" title="Significand">significand</a> (to make them distinct from infinity values); this representation allows the definition of multiple distinct NaN values, depending on which bits are set in the significand, but also on the value of the leading sign bit (not all applications are required to provide distinct semantics for those distinct NaN values).</p>
<p>For example, a bit-wise <a class="mw-redirect" href="/wiki/IEEE_floating-point" title="IEEE floating-point">IEEE floating-point</a> standard single precision (32-bit) NaN would be: <span style="white-space:nowrap;">s111 1111 1xxx xxxx xxxx xxxx xxxx xxxx</span> where <i>s</i> is the sign (most often ignored in applications) and <i>x</i> is non-zero (the value zero encodes infinities). Some bits from <i>x</i> (usually and preferably the first one) are used to determine the type of NaN: <b>quiet NaN</b> or <b>signaling NaN</b>. The remaining bits encode a <b>payload</b> (most often ignored in applications).</p>
<p>Floating-point operations other than ordered comparisons normally propagate a quiet NaN (<i>qNaN</i>). Floating-point operations on a signaling NaN (<i>sNaN</i>) signal the invalid operation exception, the default exception action is then the same as for qNaN operands and they produce a qNaN if producing a floating-point result.</p>
<p>A comparison with a NaN always returns an <i>unordered result</i> even when comparing with itself. The comparison predicates are either signaling or non-signaling; the signaling versions signal the invalid operation exception for such comparisons. The equality and inequality predicates are non-signaling so <i>x</i> = <i>x</i> returning false can be used to test if <i>x</i> is a quiet NaN. The other standard comparison predicates are all signaling if they receive a NaN operand, the standard also provides non-signaling versions of these other predicates. The predicate <i>isNaN(x)</i> determines if a value is a NaN and never signals an exception, even if <i>x</i> is a signaling NaN.</p>
<p>The propagation of quiet NaNs through arithmetic operations allows errors to be detected at the end of a sequence of operations without extensive testing during intermediate stages. However, note that depending on the language and the function, NaNs can silently be removed in expressions that would give a constant result for all other floating-point values e.g. NaN^0, which may be defined as 1, so in general a later test for a set INVALID flag is needed to detect all cases where NaNs are introduced <sup class="reference" id="cite_ref-LectureNotes_3-0"><a href="#cite_note-LectureNotes-3">[3]</a></sup> (see <a href="#Function_definition">Function definition</a> below for further details).</p>
<p>In section 6.2 of the revised <a href="/wiki/IEEE_floating_point" title="IEEE floating point">IEEE 754-2008</a> standard there are two anomalous functions (the <b>maxnum</b> and <b>minnum</b> functions that return the maximum of two operands that are expected to be numbers) that favor numbers — if just one of the operands is a NaN then the value of the other operand is returned.</p>
<h3><span class="mw-headline" id="Operations_generating_NaN">Operations generating NaN</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=NaN&amp;action=edit&amp;section=2" title="Edit section: Operations generating NaN">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>There are three kinds of operations that can return NaN:<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></p>
<ul>
<li>Operations with a NaN as at least one operand.</li>
<li><a class="mw-redirect" href="/wiki/Indeterminate_forms" title="Indeterminate forms">Indeterminate forms</a>:
<ul>
<li>The divisions 0/0 and ±∞/±∞.</li>
<li>The multiplications 0×±∞ and ±∞×0.</li>
<li>The additions ∞ + (−∞), (−∞) + ∞ and equivalent subtractions.</li>
<li>The standard has alternative functions for powers:
<ul>
<li>The standard <tt>pow</tt> function and the integer exponent <tt>pown</tt> function define 0<sup>0</sup>, 1<sup>∞</sup>, and ∞<sup>0</sup> as 1.</li>
<li>The <tt>powr</tt> function defines all three indeterminate forms as invalid operations and so returns NaN.</li>
</ul>
</li>
</ul>
</li>
<li>Real operations with <a href="/wiki/Complex_number" title="Complex number">complex</a> results, for example:
<ul>
<li>The square root of a negative number.</li>
<li>The logarithm of a negative number.</li>
<li>The inverse sine or cosine of a number that is less than −1 or greater than +1.</li>
</ul>
</li>
</ul>
<p>NaNs may also be explicitly assigned to variables, typically as a representation for missing values. Prior to the IEEE standard, programmers often used a special value (such as −99999999) to represent undefined or missing values, but there was no guarantee that they would be handled consistently or correctly.<sup class="reference" id="cite_ref-idl_1-1"><a href="#cite_note-idl-1">[1]</a></sup></p>
<p>NaNs are not necessarily generated in all the above cases. If an operation can produce an exception condition and traps are not masked then the operation will cause a trap instead.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> If an operand is a quiet NaN, and there isn't also a signaling NaN operand, then there is no exception condition and the result is a quiet NaN. Explicit assignments will not cause an exception even for signaling NaNs.</p>
<h3><span class="mw-headline" id="Quiet_NaN">Quiet NaN</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=NaN&amp;action=edit&amp;section=3" title="Edit section: Quiet NaN">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Quiet NaNs, or qNaNs, do not raise any additional exceptions as they propagate through most operations. The exceptions are where the NaN cannot simply be passed through unchanged to the output, such as in format conversions or certain comparison operations (which do not "expect" a NaN input).</p>
<h3><span class="mw-headline" id="Signaling_NaN">Signaling NaN</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=NaN&amp;action=edit&amp;section=4" title="Edit section: Signaling NaN">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Signaling NaNs, or sNaNs, are special forms of a NaN that when consumed by most operations should raise the invalid operation exception and then, if appropriate, be "quieted" into a qNaN that may then propagate. They were introduced in <a href="/wiki/IEEE_floating_point" title="IEEE floating point">IEEE 754</a>. There have been several ideas for how these might be used:</p>
<ul>
<li>Filling uninitialized memory with signaling NaNs would produce the invalid operation exception if the data is used before it is initialized</li>
<li>Using an sNaN as a placeholder for a more complicated <a href="/wiki/Object_(computer_science)" title="Object (computer science)">object</a>, such as:
<ul>
<li>A representation of a number that has <a href="/wiki/Arithmetic_underflow" title="Arithmetic underflow">underflowed</a></li>
<li>A representation of a number that has <a href="/wiki/Arithmetic_overflow" title="Arithmetic overflow">overflowed</a></li>
<li>Number in a higher precision format</li>
<li>A <a href="/wiki/Complex_number" title="Complex number">complex number</a></li>
</ul>
</li>
</ul>
<p>When encountered a trap handler could decode the sNaN and return an index to the computed result. In practice this approach is faced with many complications. The treatment of the <a href="/wiki/Sign_bit" title="Sign bit">sign bit</a> of NaNs for some simple operations (such as <a href="/wiki/Absolute_value" title="Absolute value">absolute value</a>) is different from that for arithmetic operations. Traps are not required by the standard. There are other approaches to this sort of problem that would be more portable.</p>
<h2><span class="mw-headline" id="Function_definition">Function definition</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=NaN&amp;action=edit&amp;section=5" title="Edit section: Function definition">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>There are differences of opinion about the proper definition for the result of a numeric <a href="/wiki/Subroutine" title="Subroutine">function</a> that receives a quiet NaN as input. One view is that the NaN should propagate to the output of the function in all cases to propagate the indication of an error. Another view, and the one taken by the <a href="/wiki/C99" title="C99">ISO C99</a> and <a href="/wiki/IEEE_floating_point" title="IEEE floating point">IEEE 754-2008</a> standards in general, is that if the function has multiple arguments and the output is uniquely determined by all the non-NaN inputs (including infinity), then that value should be the result. Thus for example the value returned by hypot(±∞,qNaN) and hypot(qNaN,±∞) is +∞.</p>
<p>The problem is particularly acute for the <a href="/wiki/Exponentiation" title="Exponentiation">exponentiation</a> function pow(x,y) = x<sup>y</sup>. The expressions 0<sup>0</sup>, ∞<sup>0</sup> and 1<sup>∞</sup> are considered <a href="/wiki/Indeterminate_form" title="Indeterminate form">indeterminate forms</a> when they occur as limits (just like ∞ × 0), and the question of whether <a href="/wiki/Exponentiation#Zero_to_the_power_of_zero" title="Exponentiation">zero to the zero power</a> should be defined as 1 has divided opinion.</p>
<p>If the output is considered as undefined when a parameter is undefined, then pow(1,qNaN) should produce a qNaN. However, <a class="mw-redirect" href="/wiki/Math.h" title="Math.h">math libraries</a> have typically returned 1 for pow(1,y) for any <a href="/wiki/Real_number" title="Real number">real number</a> y, and even when y is an <a href="/wiki/Extended_real_number_line" title="Extended real number line">infinity</a>. Similarly, they produce 1 for pow(x,0) even when x is 0 or an infinity. The rationale for returning the value 1 for the indeterminate forms was that the value of functions at singular points can be taken as a particular value if that value is in the limit the value<sup class="noprint Inline-Template" style="margin-left:0.1em; white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Please_clarify" title="Wikipedia:Please clarify"><span title="The text near this tag may need clarification or removal of jargon. (October 2011)">clarification needed</span></a></i>]</sup> for all but a vanishingly small part of a ball around the limit value of the parameters. The 2008 version of the <a href="/wiki/IEEE_floating_point" title="IEEE floating point">IEEE 754</a> standard says that pow(1,qNaN) and pow(qNaN,0) should both return 1 since they return 1 whatever else is used instead of quiet NaN. Moreover, ISO C99, and later IEEE 754-2008, chose to specify pow(−1,±∞) = 1 instead of qNaN; the reason of this choice is given in the C rationale:<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup> "Generally, C99 eschews a NaN result where a numerical value is useful. [...] The result of pow(−2,∞) is +∞, because all large positive floating-point values are even integers."</p>
<p>To satisfy those wishing a more strict interpretation of how the power function should act, the 2008 standard defines two additional power functions: pown(x,n), where the exponent must be an integer, and powr(x,y), which returns a NaN whenever a parameter is a NaN or the exponentiation would give an <a href="/wiki/Indeterminate_form" title="Indeterminate form">indeterminate form</a>.</p>
<h2><span class="mw-headline" id="Integer_NaN">Integer NaN</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=NaN&amp;action=edit&amp;section=6" title="Edit section: Integer NaN">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Most fixed-size <a href="/wiki/Integer" title="Integer">integer</a> formats do not have any way of explicitly indicating invalid data. Converting NaN to an integer type, or performing an integer operation whose floating-point equivalent would produce NaN, usually throws an <a href="/wiki/Exception_handling" title="Exception handling">exception</a>. In <a class="mw-redirect" href="/wiki/Java_programming_language" title="Java programming language">Java</a>, such operations throw instances of <code>java.lang.ArithmeticException</code>.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup> In <a class="mw-redirect" href="/wiki/C_programming_language" title="C programming language">C</a>, they lead to <a href="/wiki/Undefined_behavior" title="Undefined behavior">undefined behavior</a>.</p>
<p><a href="/wiki/Perl" title="Perl">Perl</a>'s <span style="font-family: monospace, monospace;">Math::BigInt</span> package uses "NaN" for the result of strings that don't represent valid integers.<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup></p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="gp">&gt;</span> perl -mMath::BigInt -e <span class="s2">"print Math::BigInt-&gt;new('foo')"</span>
<span class="go">NaN</span>
</pre></div>
<h2><span class="mw-headline" id="Display">Display</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=NaN&amp;action=edit&amp;section=7" title="Edit section: Display">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Different operating systems and programming languages may have different string representations of NaN.</p>
<pre>
 nan
 NaN
 NaN%
 NAN
 NaNQ
 NaNS
 qNaN
 sNaN
 1.#SNAN
 1.#QNAN
 -1.#IND
</pre>
<p>Since, in practice, encoded NaNs have both a sign and optional 'diagnostic information' (sometimes called a <i>payload</i>), these will often be found in string representations of NaNs, too, for example:</p>
<pre>
 -NaN
  NaN12345
 -sNaN12300
 -NaN(s1234)
</pre>
<p>(other variants exist)</p>
<h2><span class="mw-headline" id="Encoding">Encoding</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=NaN&amp;action=edit&amp;section=8" title="Edit section: Encoding">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In <a href="/wiki/IEEE_floating_point" title="IEEE floating point">IEEE 754</a> standard-conforming floating-point storage formats, NaNs are identified by specific, pre-defined bit patterns unique to NaNs. The sign bit does not matter. Binary format NaNs are represented with the exponential field filled with ones (like infinity values), and some non-zero number in the significand field (to make them distinct from infinity values). The original IEEE 754 standard from 1985 (<a href="/wiki/IEEE_floating_point" title="IEEE floating point">IEEE 754-1985</a>) only described binary floating-point formats, and did not specify how the signaling/quiet state was to be tagged. In practice, the most significant bit of the significand field determined whether a NaN is signaling or quiet. Two different implementations, with reversed meanings, resulted:</p>
<ul>
<li>most processors (including those of the <a href="/wiki/Intel" title="Intel">Intel</a> and <a class="mw-redirect" href="/wiki/AMD" title="AMD">AMD</a>'s <a href="/wiki/X86" title="X86">x86</a> family, the <a class="mw-redirect" href="/wiki/Motorola_68000_family" title="Motorola 68000 family">Motorola 68000 family</a>, the <a href="/wiki/AIM_alliance" title="AIM alliance">AIM</a> <a href="/wiki/PowerPC" title="PowerPC">PowerPC</a> family, the <a href="/wiki/ARM_architecture" title="ARM architecture">ARM</a> family, and the <a href="/wiki/Sun_Microsystems" title="Sun Microsystems">Sun</a> <a href="/wiki/SPARC" title="SPARC">SPARC</a> family) set the signaling/quiet bit to non-zero if the NaN is quiet, and to zero if the NaN is signaling. Thus, on these processors, the bit represents an 'is_quiet' flag;</li>
<li>in NaNs generated by the <a href="/wiki/PA-RISC" title="PA-RISC">PA-RISC</a> and <a class="mw-redirect" href="/wiki/MIPS_architecture" title="MIPS architecture">MIPS</a> processors, the signaling/quiet bit is zero if the NaN is quiet, and non-zero if the NaN is signaling. Thus, on these processors, the bit represents an 'is_signaling' flag.</li>
</ul>
<p>The former choice has been preferred as it allows the implementation to quiet a signaling NaN by just setting the signaling/quiet bit to 1. The reverse is not possible with the latter choice because setting the signaling/quiet bit to 0 could yield an infinity.</p>
<p>The 2008 revision of the IEEE 754 standard (<a href="/wiki/IEEE_floating_point" title="IEEE floating point">IEEE 754-2008</a>) makes formal recommendations for the encoding of the signaling/quiet state.</p>
<ul>
<li>For binary formats, the most significant bit of the significand field should be an 'is_quiet' flag. I.e. this bit is non-zero if the NaN is quiet, and zero if the NaN is signaling.</li>
<li>For decimal formats, whether binary or decimal encoded, a NaN is identified by having the top five bits of the combination field after the sign bit set to ones. The sixth bit of the field is the 'is_quiet' flag. The standard follows the interpretation as an 'is_signaling' flag. I.e. the signaling/quiet bit is zero if the NaN is quiet, and non-zero if the NaN is signaling. A signaling NaN is quieted by clearing this sixth bit.</li>
</ul>
<p>The state/value of the remaining bits (i.e. other than the ones used to identify a NaN as NaN, including the quiet/signaling bits) are not defined by the standard. This value is called the 'payload' of the NaN. If an operation has a single NaN input and propagates it to the output, the result NaN's payload should be that of the input NaN (this is not always possible for binary formats when the signaling/quiet state is encoded by an 'is_signaling' flag, as explained above). If there are multiple NaN inputs, the result NaN's payload should be from one of the input NaNs; the standard does not specify which.</p>
