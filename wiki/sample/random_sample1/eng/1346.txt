<p><b>OSx86</b> (from the names <a class="mw-redirect" href="/wiki/OS_X" title="OS X">OS X</a> and <a href="/wiki/X86" title="X86">x86</a>) is a collaborative <a class="mw-redirect" href="/wiki/Hacker_(hobbyist)" title="Hacker (hobbyist)">hacking</a> project to run the <span class="nowrap">OS X</span> <a href="/wiki/Operating_system" title="Operating system">operating system</a> on non-<a href="/wiki/Apple_Inc." title="Apple Inc.">Apple</a> personal computers with x86 architecture on <a href="/wiki/X86-64" title="X86-64">x86-64</a> processors. The effort started soon after the June 2005 <a class="mw-redirect" href="/wiki/Worldwide_Developers_Conference" title="Worldwide Developers Conference">Worldwide Developers Conference</a> announcement that Apple would be <a class="mw-redirect" href="/wiki/Apple%E2%80%93Intel_transition" title="Apple–Intel transition">moving</a> its personal computers from <a href="/wiki/PowerPC" title="PowerPC">PowerPC</a> to <a href="/wiki/Intel" title="Intel">Intel</a> microprocessors.</p>
<p>A computer built to run this type of <span class="nowrap">OS X</span> is also known as a <b>Hackintosh</b>,<sup class="reference" id="cite_ref-bmr_1-0"><a href="#cite_note-bmr-1">[1]</a></sup> a <a href="/wiki/Portmanteau" title="Portmanteau">Portmanteau</a> of the word "hack" and the name of Apple's main model of computers, <a href="/wiki/Macintosh" title="Macintosh">Macintosh</a>. Hackintoshed notebook computers are also called "Hackbooks".<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup></p>
<p>The Apple <a href="/wiki/Software_license" title="Software license">software license</a> does not allow <span class="nowrap">OS X</span> to be used on a computer that is not "Apple-branded".<sup class="reference" id="cite_ref-EULA_3-0"><a href="#cite_note-EULA-3">[3]</a></sup> The legality of this form of <a href="/wiki/Tying_(commerce)" title="Tying (commerce)">tying</a> is disputed by companies such as <a class="mw-redirect" href="/wiki/Psystar" title="Psystar">Psystar</a>, <a class="new" href="/w/index.php?title=Bizon_computer&amp;action=edit&amp;redlink=1" title="Bizon computer (page does not exist)">Bizon computer</a>, <a href="/wiki/PearC" title="PearC">PearC</a> and <a class="new" href="/w/index.php?title=MacPC&amp;action=edit&amp;redlink=1" title="MacPC (page does not exist)">MacPC</a> who have attempted to release products using Mac OS on non-Apple machines. However, while the methods Apple uses to prevent <span class="nowrap">OS X</span> from being installed on non-Apple hardware are protected from commercial circumvention in the <a href="/wiki/United_States" title="United States">United States</a> by the <a href="/wiki/Digital_Millennium_Copyright_Act" title="Digital Millennium Copyright Act">Digital Millennium Copyright Act</a> (DMCA),<sup class="reference" id="cite_ref-pcworld1_4-0"><a href="#cite_note-pcworld1-4">[4]</a></sup> specific changes to the law regarding the concept of <a href="/wiki/Privilege_escalation#Jailbreaking" title="Privilege escalation">jailbreaking</a> has thrown such and similar circumvention methods into a grey area when carried out by <a class="mw-redirect" href="/wiki/End-users" title="End-users">end-users</a> for personal use.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup><sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Verifiability" title="Wikipedia:Verifiability"><span title="The material near this tag failed verification of its source citation(s). (September 2012)">not in citation given</span></a></i>]</sup></p>

<h3><span class="mw-headline" id="Mac_OS_X_v10.4_.22Tiger.22">Mac OS X v10.4 "Tiger"</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=OSx86&amp;action=edit&amp;section=2" title='Edit section: Mac OS X v10.4 "Tiger"'>edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>On June 6, 2005, Apple announced their plans to switch to <a href="/wiki/Intel" title="Intel">Intel</a> processors at their <a href="/wiki/Apple_Worldwide_Developers_Conference" title="Apple Worldwide Developers Conference">Worldwide Developers Conference</a> and released a Developer Transition Kit to selected developers at a cost of $999.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup> Efforts immediately began to attempt to run Mac OS X on non-Apple hardware, but developers quickly found themselves with an error message saying that the PC hardware configurations were not supported.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup></p>
<p>Apple released on January 10, 2006 <span class="nowrap">Mac OS X</span> 10.4.4 with the first generation of Intel-based Macs, the iMac and the MacBook Pro. These machines used <a class="mw-redirect" href="/wiki/Extensible_Firmware_Interface" title="Extensible Firmware Interface">Extensible Firmware Interface</a> (EFI) platform firmware instead of the older style <a href="/wiki/BIOS" title="BIOS">BIOS</a> found on most x86 motherboards at the time. On February 14, 2006, an initial "hack" of <span class="nowrap">Mac OS X</span> v10.4.4 was released on the Internet by a programmer with the pseudonym <i>crg92</i>.<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> Within hours Apple released the 10.4.5 update,<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup> which was patched again by the same author within two weeks.<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup> On April 3, 2006 Apple released their 10.4.6 update<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup> and again patches were released within two weeks that allowed users to install most of this update on non-Apple computers, although this did not include the updated kernel in 10.4.6. In June, the hackers released the 10.4.7 <span class="nowrap">Mac OS X</span> update for non-Apple computers using the 10.4.4 kernel.</p>
<p>Up to the release of the 10.4.8 update, all OSx86 patches used the 10.4.4 kernel with the rest of the operating system at version 10.4.8. However, the newer frameworks relied on the newer kernels and this led to users of 10.4.8 encountering many problems. Apple also started making more use of <a href="/wiki/SSE3" title="SSE3">SSE3</a> instructions on their hardware making it even more difficult for users with CPUs supporting only SSE2 (such as older <a href="/wiki/Pentium_4" title="Pentium 4">Pentium 4s</a>) to get a fully compatible system running. To solve this problem, hackers from the community released kernels where those instructions were emulated with <a href="/wiki/SSE2" title="SSE2">SSE2</a> equivalents, although this produced a performance penalty.</p>
<p>Throughout the years, many "distros" were released for download over the Internet. These distros were copies of the Mac OS X installer disc modified to include additional components necessary to make the OS run on the non-Apple hardware. A prominent member of the community, JaS, released many distros of Mac OS X Tiger containing patched kernels. Some other popular distros are iATKOS, Kalyway, iPC, iDeneb, and Niresh. Distros have fallen out of favour as the OSx86 community grew, as new bootloaders were developed which made it possible to use actual copies of the OS X Installer.</p>
<h3><span class="mw-headline" id="Mac_OS_X_v10.5_.22Leopard.22">Mac OS X v10.5 "Leopard"</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=OSx86&amp;action=edit&amp;section=3" title='Edit section: Mac OS X v10.5 "Leopard"'>edit</a><span class="mw-editsection-bracket">]</span></span></h3>

<p>As early as <span class="nowrap">Mac OS X</span> v10.5 build 9A466 the community has maintained a version of Leopard that can run on non-Apple hardware. A hacker by the handle of BrazilMac created one of the earliest patching processes which made it convenient for users to install <span class="nowrap">Mac OS X</span> onto 3rd party hardware by using a legally obtained, retail version of Apple <span class="nowrap">Mac OS X</span>. It was utilizing the BrazilMac patch that many effortless distros of Mac OSx86 came to fruition. Five of the most popular builds go by the name JaS, Kalyway, iATKOS, iPC and iDeneb - although more recently these builds are on the way out as the Boot-132 method (described below) gains popularity. However, all of these compilations rely on the work of kernel hackers made by Lorem (build 9A466), SynthetiX (builds 9A499, 9A527 and 9A559), ToH (builds 9A581, 9B13 and 9B18) and more recently a group calling themselves StageXNU (now called Voodoo) (Darwin 9.4.0). Their contributions trickled down into the various Mac OSx86 installers, readily available on the Internet. They continue to be refined and updated builds released, not just to maintain compatibility with Apple releases but an ever-increasing number of third-party components. The OSx86 community has been quick to make the necessary modifications to enable Apple's latest releases to run on non-Apple hardware. Within hours of Leopard's release, an AMD/Intel SSE2/3 Kernel Patcher was created which removed the <a class="mw-redirect" href="/wiki/HPET" title="HPET">HPET</a> requirement from an original untouched mach_kernel file, a core component of the Mac OS.</p>
<h3><span class="mw-headline" id="Mac_OS_X_v10.6_.22Snow_Leopard.22">Mac OS X v10.6 "Snow Leopard"</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=OSx86&amp;action=edit&amp;section=4" title='Edit section: Mac OS X v10.6 "Snow Leopard"'>edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>When "Snow Leopard" was released, Russian hacker netkas created a version of Chameleon which can boot Mac OS X v10.6. The main problem was that many people were forced to modify DSDT or use kexts due to some specific issues. As soon as possible modbin and dmitrik released test versions of kernel that allow to boot Snow Leopard on AMD machines. Stable XNU kernels for v10.6 were released by Qoopz and Pcj. There are some popular builds based on Retail by the name Universal (Intel only), Hazard, and iAtkos. Since v10.6.2 Nawcom, Qoopz, and Andy Vandijck have been working on Legacy kernel for unsupported CPUs.</p>
<h3><span class="mw-headline" id="Mac_OS_X_v10.7_.22Lion.22">Mac OS X v10.7 "Lion"</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=OSx86&amp;action=edit&amp;section=5" title='Edit section: Mac OS X v10.7 "Lion"'>edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>When Apple released the Developer Preview 1, a Russian Hackintosh developer usr-sse2 was the first who created a method to install Lion. The method consists of deploying Mac OS X v10.7 image on a flash drive, and booting from it via XPC UEFI Bootloader (See <a href="/wiki/OSx86#DUET" title="OSx86">DUET</a> below). After some changes were made to the Chameleon source code, booting of Lion was possible with an updated version of Chameleon. After a while Dmitrik also known as Bronzovka had luck with creating a kernel which supported AMD systems; After a few months (10.7.3 V2 With AMD Support) and iAtkos L2 (10.7.2 Only Intel) were released . <sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup></p>
<h3><span class="mw-headline" id="Mac_OS_X_v10.8_.22Mountain_Lion.22">Mac OS X v10.8 "Mountain Lion"</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=OSx86&amp;action=edit&amp;section=6" title='Edit section: Mac OS X v10.8 "Mountain Lion"'>edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Shortly after the release of Developer Preview 1, some unknown developers managed to install this version of OSX to their PC by using a modified version of the Chameleon Bootloader. This version was released via the main project starting at version r1997<sup class="reference" id="cite_ref-13"><a href="#cite_note-13">[13]</a></sup> to the general public. Due to the problems sourced during the Lion era,<sup class="noprint Inline-Template" style="margin-left:0.1em; white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Please_clarify" title="Wikipedia:Please clarify"><span title="The text near this tag may need clarification or removal of jargon. (September 2012)">clarification needed</span></a></i>]</sup> other ways of installing and required patches were never made public, which leaves the scene in an unknown state towards Mountain Lion.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (September 2012)">citation needed</span></a></i>]</sup></p>
<p>Since the retail release of Mountain Lion several users have reported successful setups using installers purchased from the <a href="/wiki/Mac_App_Store" title="Mac App Store">Mac App Store</a>, along with updated versions of Chameleon and other tools including distros. Niresh's Distro (10.8 Intel only) was first released and then was updated to 10.8.2 (With AMD and Intel) and 10.8.5 (With UEFI Support, AMD and Intel Support) versions; iAtkos ML2 was released after Niresh's Release.<sup class="reference" id="cite_ref-14"><a href="#cite_note-14">[14]</a></sup><sup class="reference" id="cite_ref-15"><a href="#cite_note-15">[15]</a></sup> <sup class="reference" id="cite_ref-16"><a href="#cite_note-16">[16]</a></sup> <sup class="reference" id="cite_ref-17"><a href="#cite_note-17">[17]</a></sup></p>
<h3><span class="mw-headline" id="Mac_OS_X_v10.9_.22Mavericks.22">Mac OS X v10.9 "Mavericks"</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=OSx86&amp;action=edit&amp;section=7" title='Edit section: Mac OS X v10.9 "Mavericks"'>edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Multiple new kernels for Hackintosh 10.9 are in the works, although there still are minor issues with most of them. Most of these kernels aim to allow users to run Mavericks on AMD and older Intel CPUs, which lack certain instruction sets of the latest Intel CPUs. Significant efforts have been made to emulate instruction sets like <a href="/wiki/SSSE3" title="SSSE3">SSSE3</a>, which are not present on AMD K10-based CPUs, and older Intel CPUs, like the Intel Core Duo. AMD's latest CPUs, from the '<a href="/wiki/Bulldozer_(microarchitecture)" title="Bulldozer (microarchitecture)">Bulldozer</a>' architecture onwards, contain almost all the latest instruction sets, and hence, some kernels with full <a href="/wiki/SSE4" title="SSE4">SSE4</a> support have also been released. After two months, Niresh's Distro was released for Mavericks which supports AMD CPUs and latest Intel CPUs. It also has a custom kernel that allows Intel Atom processors to boot into Mavericks. Niresh's was the only free distro which was released for Mavericks, since the iAtkos Team decided to release their Mavericks distro for specific hardware on donation basis.<sup class="reference" id="cite_ref-18"><a href="#cite_note-18">[18]</a></sup> <sup class="reference" id="cite_ref-19"><a href="#cite_note-19">[19]</a></sup></p>

<p>After the initial release of OSX Yosemite 10.10 BETA, various developers took on the role of updating their bootloaders for the system. Members of OSx86 forum Insanelymac set to update the EFI Bootloader Chameleon for this new OS release. Some time later, Niresh (a standalone OSx86 developer) released a standalone tool known as Yosemite Zone which would automatically install the new OS and other various features on a non-Apple device with minimal input. This method consisted of <a href="/wiki/BitTorrent" title="BitTorrent">torrenting</a> an OSX 10.10 DMG onto a USB Flash Drive with MacPwn Vanilla Installation. Unibeast was updated to support Yosemite,<sup class="reference" id="cite_ref-20"><a href="#cite_note-20">[20]</a></sup> and a distribution of Yosemite Zone was released with AMD processor support. A vanilla installation of Yosemite is possible via Insanelymac's Pandora Box Beta 2.0 and UniBeast. This type installation uses as few kexts (drivers) as possible in addition to using an unaltered version of the OS X installation app, and is preferred over distributions.<sup class="reference" id="cite_ref-21"><a href="#cite_note-21">[21]</a></sup><sup class="reference" id="cite_ref-22"><a href="#cite_note-22">[22]</a></sup><sup class="reference" id="cite_ref-23"><a href="#cite_note-23">[23]</a></sup></p>
<h3><span class="mw-headline" id="OS_X_El_Capitan_.2810.11.0-10.11.6.29">OS X El Capitan (10.11.0-10.11.6)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=OSx86&amp;action=edit&amp;section=9" title="Edit section: OS X El Capitan (10.11.0-10.11.6)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Both Clover and Chameleon were updated to be compatible with <a href="/wiki/OS_X_El_Capitan" title="OS X El Capitan">El Capitan</a>. UniBeast<sup class="reference" id="cite_ref-26"><a href="#cite_note-26">[26]</a></sup> and MacPwn<sup class="reference" id="cite_ref-27"><a href="#cite_note-27">[27]</a></sup> were updated to support it.<sup class="reference" id="cite_ref-28"><a href="#cite_note-28">[28]</a></sup></p>

<p>Apple does not authorize the use of Mac OS X on any x86 PC other than the ones it has developed itself. The company used technical means (although not the <a href="/wiki/Trusted_Platform_Module" title="Trusted Platform Module">Trusted Platform Module</a>, or TPM, as has been widely mis-reported<sup class="reference" id="cite_ref-29"><a href="#cite_note-29">[29]</a></sup>), to tie Mac OS to the systems it distributed to developers after announcing its switch to Intel's chips.<sup class="reference" id="cite_ref-30"><a href="#cite_note-30">[30]</a></sup></p>
<p>The <span class="nowrap">Mac OS X</span> <a class="mw-redirect" href="/wiki/EULA" title="EULA">EULA</a> forbids installations of <span class="nowrap">Mac OS X</span> on a "non-Apple-branded computer".<sup class="reference" id="cite_ref-EULA_3-1"><a href="#cite_note-EULA-3">[3]</a></sup> On July 3, 2008, Apple filed a lawsuit against <a href="/wiki/Psystar_Corporation" title="Psystar Corporation">Psystar Corporation</a><sup class="reference" id="cite_ref-31"><a href="#cite_note-31">[31]</a></sup> for violating this restriction, among other claims.<sup class="reference" id="cite_ref-32"><a href="#cite_note-32">[32]</a></sup> Apple claimed Psystar "violated the Digital Millennium Copyright Act (DMCA) by dodging copy-protection technologies Apple uses to protect <span class="nowrap">Mac OS X</span>." Apple employs technological protection measures that effectively control access to Apple's copyrighted works. Specifically, Apple charged Psystar with acquiring or creating code that "avoids, bypasses, removes, descrambles, decrypts, deactivates or impairs a technological protection measure without Apple's authority for the purpose of gaining unauthorized access to Apple's copyrighted works."<sup class="reference" id="cite_ref-33"><a href="#cite_note-33">[33]</a></sup> This brief revealed that Apple considers the methods that it uses to prevent <span class="nowrap">Mac OS X</span> from being installed on non-Apple hardware to be protected by the <a href="/wiki/Digital_Millennium_Copyright_Act" title="Digital Millennium Copyright Act">Digital Millennium Copyright Act</a> (DMCA).</p>
<p>On November 13, 2009, the court granted Apple's motion for summary judgment and found Apple's copyrights were violated as well as the DMCA when Psystar installed Apple's operating system on non-Apple computers. A hearing on remedies was set for December 14.<sup class="reference" id="cite_ref-pcworld1_4-1"><a href="#cite_note-pcworld1-4">[4]</a></sup><sup class="reference" id="cite_ref-34"><a href="#cite_note-34">[34]</a></sup></p>
<p>On January 14, 2009, the Gadget Lab site of <i>Wired Magazine</i> posted a video tutorial for installing <span class="nowrap">Mac OS X</span> on an MSI Wind netbook, but removed it following a complaint from Apple.<sup class="reference" id="cite_ref-35"><a href="#cite_note-35">[35]</a></sup> Textual instructions remain, but include an EULA violation disclaimer.<sup class="reference" id="cite_ref-36"><a href="#cite_note-36">[36]</a></sup></p>
<p>On May 15, 2012, the case <a href="/wiki/Apple_Inc." title="Apple Inc.">Apple</a> vs. <a href="/wiki/Psystar_Corporation" title="Psystar Corporation">Psystar Corporation</a> finally ended. The court ruled that Psystar has "violated Apple's exclusive reproduction right, distribution right, and right to create derivative works,"<sup class="reference" id="cite_ref-37"><a href="#cite_note-37">[37]</a></sup> putting an end to the case.</p>
<h2><span class="mw-headline" id="Hacking_approaches">Hacking approaches</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=OSx86&amp;action=edit&amp;section=12" title="Edit section: Hacking approaches">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Kernel_hacks">Kernel hacks</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=OSx86&amp;action=edit&amp;section=13" title="Edit section: Kernel hacks">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>When copies of Mac OS X Tiger started running on non-Apple hardware, it was found that some processors were unable to run the OS. <a href="/wiki/Rosetta_(software)" title="Rosetta (software)">Rosetta</a>, a binary translator which made it possible to run PowerPC programs on Intel processors, (and later the kernel itself) required the support of the <a href="/wiki/SSE3" title="SSE3">SSE3</a> instruction set. To circumvent this, programmers in the community released patched kernels, which included support for emulating SSE3 instructions using <a href="/wiki/SSE2" title="SSE2">SSE2</a> equivalents. In October 2005, Apple released update 10.4.3 to developers that required <a href="/wiki/NX_bit" title="NX bit">NX bit</a> microprocessor support;<sup class="reference" id="cite_ref-38"><a href="#cite_note-38">[38]</a></sup> however, patches were released to circumvent this as well.<sup class="reference" id="cite_ref-39"><a href="#cite_note-39">[39]</a></sup> Patched kernels were also later released which supported AMD processors.</p>
<p>When Mac OS X Leopard released on October 26, 2007, patches were created to remove the <a href="/wiki/High_Precision_Event_Timer" title="High Precision Event Timer">HPET</a> requirement from the kernel. Efforts were also made to emulate the <a href="/wiki/SSSE3" title="SSSE3">SSSE3</a> instruction set for processors that did not support it. The kernel used by OS X Mavericks made use of SSSE3 instructions, requiring those patches.</p>
<h3><span class="mw-headline" id="Boot_loaders_and_emulators">Boot loaders and emulators</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=OSx86&amp;action=edit&amp;section=14" title="Edit section: Boot loaders and emulators">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<h4><span class="mw-headline" id="EFI_emulation">EFI emulation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=OSx86&amp;action=edit&amp;section=15" title="Edit section: EFI emulation">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p><a class="mw-redirect" href="/wiki/Extensible_Firmware_Interface" title="Extensible Firmware Interface">Extensible Firmware Interface</a> (EFI) is a specification that defines a software interface between an operating system and platform firmware. Since this method generally does not require copying or modification of <span class="nowrap">Mac OS X</span>, it is considered<sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Manual_of_Style/Words_to_watch#Unsupported_attributions" title="Wikipedia:Manual of Style/Words to watch"><span title="The material near this tag may use weasel words or too-vague attribution. (August 2014)">by whom?</span></a></i>]</sup> to be the legal way of installing <span class="nowrap">Mac OS X</span> on non-Apple computers (despite being untested in courts).</p>
<p>In early November 2007, a group of hackers (fronted by a Russian hacker known as Netkas), using an already modified Boot-132 source root from David Elliot (also known as dfe),<sup class="reference" id="cite_ref-40"><a href="#cite_note-40">[40]</a></sup> developed a method<sup class="reference" id="cite_ref-41"><a href="#cite_note-41">[41]</a></sup> of emulating an EFI environment using a specially modified Darwin bootloader.<sup class="reference" id="cite_ref-42"><a href="#cite_note-42">[42]</a></sup> In practical terms, this meant that regular PCs meeting a minimum set of hardware requirements could now be "seen" as real Macintosh computers by the OS, allowing the use of unmodified, "stock" Apple kernels (as long as the CPU supports it) and thus giving a more transparent and reliable operation. Several methods for real world deployment of this innovative solution have arisen around the Internet. An explanation of this achievement along with a usage guide was provided by the website DigitMemo.com.<sup class="reference" id="cite_ref-43"><a href="#cite_note-43">[43]</a></sup></p>
<p>True EFI emulation was a highly sought after asset for the OSx86 community. Previous efforts based upon Apple's open source Darwin Project and Hackintosh gurus allowed users to use <span class="nowrap">Mac OS X</span> on normal PCs, with patched kernels/kernel modules which simply bypassed EFI. Using the EFI patch, a Hackintosh could boot off "<a href="/wiki/Vanilla_software" title="Vanilla software">vanilla</a>" (unmodified) <span class="nowrap">Mac OS X</span> kernels and use vanilla kernel extensions. This not only allowed the system to be compatible with future system updates, but also offered increased stability. This method also circumvents one aspect of Apple's End User License Agreement, which states that the modification of non-Open Source components of the OS is forbidden.<sup class="reference" id="cite_ref-EULA_3-2"><a href="#cite_note-EULA-3">[3]</a></sup></p>
<p>In mid-2008, a new commercial product, EFi-X, was released that claims to allow full, simple booting off official Leopard install disks, and a subsequent install, without any patching required, but this is possibly a repackaging of Boot-132 technology in a <a href="/wiki/USB" title="USB">USB</a>-attached device.<sup class="reference" id="cite_ref-44"><a href="#cite_note-44">[44]</a></sup> <a class="mw-redirect" href="/wiki/Rebel_EFI" title="Rebel EFI">Rebel EFI</a> is another commercial product that also seems to use Open Source software.<sup class="reference" id="cite_ref-45"><a href="#cite_note-45">[45]</a></sup></p>
<p>It was thought<sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Manual_of_Style/Words_to_watch#Unsupported_attributions" title="Wikipedia:Manual of Style/Words to watch"><span title="The material near this tag may use weasel words or too-vague attribution. (August 2014)">by whom?</span></a></i>]</sup> that Windows 7's support of EFI would result in PC motherboards replacing BIOS with EFI. MSI announced the Efinity mainboard in early 2008. As of 2011, EFI-based computers have entered the market, however none can natively boot Mac OS X due to the lack of a <a class="mw-redirect" href="/wiki/HFS%2B" title="HFS+">HFS+</a> driver in the EFI implementation.</p>
<h4><span class="mw-headline" id="Boot-132">Boot-132</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=OSx86&amp;action=edit&amp;section=16" title="Edit section: Boot-132">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Boot-132 is a bootloader provided by Apple for loading the XNU kernel.<sup class="reference" id="cite_ref-46"><a href="#cite_note-46">[46]</a></sup> In mid-2008, a new modified BOOT-132 came on to the scene.<sup class="reference" id="cite_ref-47"><a href="#cite_note-47">[47]</a></sup> This method allows users to conduct the Leopard-based OSx86 installation using a stock, retail-purchased copy of <span class="nowrap">Mac OS X</span> Leopard and eradicates the necessity of a hacked installation like JaS or Kalyway (mentioned previously). The Boot-132 bootloader essentially preloads an environment on the system from which Leopard can boot and operate. The bootloader stores the necessary files (kext files) in a .img collection or simply a folder. The luxury of this new installation method includes the ability to boot and install from a retail Leopard DVD and update straight from Apple without breaking the DMCA. The only possible problem here is that it breaks the <span class="nowrap">Mac OS X</span> EULA.<sup class="reference" id="cite_ref-48"><a href="#cite_note-48">[48]</a></sup></p>
<p>The bootloader behaves like the <a href="/wiki/Linux" title="Linux">Linux</a> kernel: one can use an <a class="new" href="/w/index.php?title=Mboot&amp;action=edit&amp;redlink=1" title="Mboot (page does not exist)">mboot</a>-compatible (a patched <a class="mw-redirect" href="/wiki/Syslinux" title="Syslinux">syslinux</a> was used for the hack) bootloader which tells boot-dfe about the .img file (the <a class="mw-redirect" href="/wiki/Ramdisk" title="Ramdisk">ramdisk</a> or <a href="/wiki/Initrd" title="Initrd">initrd</a>, as it's known by Linux users), and boot-dfe will then use the kexts (or mkext) from it. This new boot-dfe has been tested with the retail Leopard DVD, and it can boot, install, run Leopard without having to build a modified DVD.</p>
<h4><span class="mw-headline" id="Chameleon">Chameleon</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=OSx86&amp;action=edit&amp;section=17" title="Edit section: Chameleon">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Since the early developer builds of Mac OS X v10.6, members of the OSx86 community had been booting the new operating system using yet another bootloader called PC EFI provided by Russian hacker Netkas or the bootloader of the Voodoo team's Chameleon. Chameleon is based on David Elliot's Boot-132. The bootloader supports ACPI, SMBIOS, graphics, ethernet, and some other injections. It allows to boot up <span class="nowrap">Mac OS X</span> on non-Macintosh hardware. Chameleon supports a lot of AMD as well as Nvidia graphics cards. There are a lot of branches of it by different developers. Chameleon is currently in the 2.1 development state.</p>
<h4><span class="mw-headline" id="Clover">Clover</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=OSx86&amp;action=edit&amp;section=18" title="Edit section: Clover">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>FakeEFI was invented by David Elliot many years ago and operates on the assumption that EFI already did its work. It leaves activity traces (boot-args and tables tree) and EfiRuntime routines in a simple form in memory and starts the kernel mach_kernel.</p>
<p>RealEFI theoretically should be flashed instead of the BIOS but alternatively a loadable EFI can be used for those with a BIOS motherboard. This system was invented by Intel and is currently in active open source development at <a href="/wiki/Unified_Extensible_Firmware_Interface#Intel_EFI" title="Unified Extensible Firmware Interface">TianoCore</a>. The bootloader is named DUET and it is capable of loading EFI but it is not designed to load Mac OS X, so, one more step was needed, adapt DUET to the requirements of Mac OS X. Also, newer motherboards already contain EFI but it is not suitable for running a Macintosh.</p>
<p>Beginning in March 2011, Slice discussed his idea with other community members, resulting in the development of a bootloader that can do both: emulate an EFI firmware of one's choice or use a Real UEFI firmware to boot Mac OS X.</p>
<h3><span class="mw-headline" id="Live_DVD">Live DVD</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=OSx86&amp;action=edit&amp;section=19" title="Edit section: Live DVD">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In March 2007, the OSx86 community made some significant progress with the development of a <a class="mw-redirect" href="/wiki/Live_DVD" title="Live DVD">Live DVD</a>. The Live DVD allows booting to a working system with Mac OS X v10.4.8.</p>
<p>On January 2, 2009, InsanelyMac's Live DVD team published a new method by which a Mac OS X v10.5.x Live DVD could be produced, allowing users to boot a fully working <span class="nowrap">Mac OS X</span> desktop from a DVD or USB flash drive.<sup class="reference" id="cite_ref-49"><a href="#cite_note-49">[49]</a></sup> The method was more reliable than previous methods because it manipulated Apple's existing Netboot and Imageboot functionalities and behaved as if the system were running off a network disk. It was easier to produce; requiring only a single script to be added to an existing installation. Distributions of the live DVD have been made since its inception. Since then, it is notable that this method has been shown to work on normal Apple Mac hardware.</p>
<h3><span class="mw-headline" id="Virtual_machine">Virtual machine</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=OSx86&amp;action=edit&amp;section=20" title="Edit section: Virtual machine">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>It is also possible to run the Mac OS as a <a href="/wiki/Virtual_machine" title="Virtual machine">virtual machine</a> inside other operating systems installed on standard PC hardware, by using <a href="/wiki/Virtualization" title="Virtualization">virtualization</a> software such as <a href="/wiki/Oracle_Corporation" title="Oracle Corporation">Oracle's</a> <a href="/wiki/VirtualBox" title="VirtualBox">VirtualBox</a><sup class="reference" id="cite_ref-50"><a href="#cite_note-50">[50]</a></sup><sup class="reference" id="cite_ref-51"><a href="#cite_note-51">[51]</a></sup> (though this is not officially supported by Oracle<sup class="reference" id="cite_ref-52"><a href="#cite_note-52">[52]</a></sup>). It is also possible to install Mac OS X on VMware but this is not supported. This method typically utilizes some of the modified OS software listed above to perform the actual install, or requires a modified version of the hypervisor that unlocks native OS X support.<sup class="reference" id="cite_ref-53"><a href="#cite_note-53">[53]</a></sup> <sup class="reference" id="cite_ref-54"><a href="#cite_note-54">[54]</a></sup> In most cases, a <a class="new" href="/w/index.php?title=Virtual_machine_image&amp;action=edit&amp;redlink=1" title="Virtual machine image (page does not exist)">virtual machine image</a> with OS X preinstalled is the easiest way to run OS X on a non-Apple computer.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (August 2016)">citation needed</span></a></i>]</sup></p>
