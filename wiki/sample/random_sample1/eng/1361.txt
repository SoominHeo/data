<p><b>PCX</b>, standing for <i>PiCture eXchange</i>, is an <a class="mw-redirect" href="/wiki/Graphics_file_formats" title="Graphics file formats">image file format</a> developed by the now-defunct <a href="/wiki/ZSoft_Corporation" title="ZSoft Corporation">ZSoft Corporation</a> of <a href="/wiki/Marietta,_Georgia" title="Marietta, Georgia">Marietta, Georgia</a>, <a href="/wiki/United_States" title="United States">United States</a>. It was the native file format for <a href="/wiki/PC_Paintbrush" title="PC Paintbrush">PC Paintbrush</a> and became one of the first widely accepted <a href="/wiki/MS-DOS" title="MS-DOS">DOS</a> imaging standards, although it has since been succeeded by more sophisticated image formats, such as <a href="/wiki/BMP_file_format" title="BMP file format">BMP</a>, <a href="/wiki/JPEG" title="JPEG">JPEG</a>, and <a href="/wiki/Portable_Network_Graphics" title="Portable Network Graphics">PNG</a>. PCX files commonly stored palette-indexed images ranging from 2 or 4 colors to 16 and 256 colors, although the format has been extended to record true-color (24-bit) images as well.<sup class="reference" id="cite_ref-pcx_2-0"><a href="#cite_note-pcx-2">[2]</a></sup></p>

<h2><span class="mw-headline" id="PCX_image_formats">PCX image formats</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=PCX&amp;action=edit&amp;section=1" title="Edit section: PCX image formats">edit</a><span class="mw-editsection-bracket">]</span></span></h2>

<p>PCX was designed during the early development of PC display hardware and most of the formats it supported are no longer used, Table A shows a list of the most commonly used PCX formats. Contemporary image editing programs may not read PCX files that match older hardware.</p>
<p>PCX is supported by common image processing software including <a href="/wiki/ACDSee" title="ACDSee">ACDSee</a>, <a href="/wiki/GIMP" title="GIMP">GIMP</a>, <a href="/wiki/ImageMagick" title="ImageMagick">ImageMagick</a>, <a href="/wiki/IrfanView" title="IrfanView">IrfanView</a>, <a href="/wiki/LView" title="LView">LView</a>, <a href="/wiki/Netpbm" title="Netpbm">Netpbm</a>, <a href="/wiki/PaintShop_Pro" title="PaintShop Pro">PaintShop Pro</a>, <a class="mw-redirect" href="/wiki/Photoshop_Elements" title="Photoshop Elements">Photoshop</a>, <a href="/wiki/Microsoft_Visio" title="Microsoft Visio">Visio</a>, <a class="mw-redirect" href="/wiki/PMview" title="PMview">PMview</a>, and <a href="/wiki/XnView" title="XnView">XnView</a>.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup><sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup> In version 2.1.4 <a href="/wiki/FFmpeg" title="FFmpeg">FFmpeg</a> could encode and decode the PCX pixel formats <i>rgb24, rgb8, bgr8, rgb4_byte, bgr4_byte, gray, pal8,</i> and <i>monob</i>.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup></p>
<p>There is a multi-page version of PCX, used by some computer <a href="/wiki/Fax" title="Fax">fax</a> and document management programs, with file extension <code>.dcx</code>. A DCX file consists of a header introducing a set of following PCX files.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup></p>
<h2><span class="mw-headline" id="PCX_file_format">PCX file format</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=PCX&amp;action=edit&amp;section=2" title="Edit section: PCX file format">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>PCX files were designed for use on IBM-compatible PCs and always use <a href="/wiki/Endianness" title="Endianness">little endian</a> byte ordering. A PCX file has three main sections, in the following order</p>
<ol>
<li>128-byte header</li>
<li>image data</li>
<li>(optional) 256-color palette</li>
</ol>
<p>The PCX file header contains an identifier byte (value 10), a version number, image dimensions, 16 palette colors, number color planes, bit depth of each plane, and a value for compression method. PCX version numbers range from 0 to 5, this originally denoted the version of the <a href="/wiki/PC_Paintbrush" title="PC Paintbrush">PC Paintbrush</a> program used to create the PCX file. The header always has space for 16 colors though the number of colors used depends upon the bit depth of the image.<sup class="noprint Inline-Template" style="margin-left:0.1em; white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Please_clarify" title="Wikipedia:Please clarify"><span title="The text near this tag may need clarification or removal of jargon. (January 2015)">clarification needed</span></a></i>]</sup> The header is 74 bytes long and the image data begins 128 bytes after the start of the file, the 54 bytes between are not used.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup></p>
<p>All PCX files use the same compression scheme and the compression value is always 1. No other values have been defined and there are no uncompressed PCX files. One source claims that 0 (uncompressed) is <i>allowed, but not much software supports it</i>.<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup></p>

<p>PCX image data is stored in rows or scan lines in top-down order. If the image has multiple planes, these are stored by plane within row, such that all the red data for row 0 are followed by all the green data for row 0, then all the blue data, then alpha data. This pattern is repeated for each line as shown in Table B.</p>
<p>When an image is less than 8 bits per pixel, each line is padded to the next byte boundary. For example, if an image has 1 plane of 1-bit data (monochrome) with a width of 22 pixels, each row will be 3 bytes long, having 24 bits per row with 2 bits unused.</p>
<h3><span class="mw-headline" id="Image_data_compression">Image data compression</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=PCX&amp;action=edit&amp;section=4" title="Edit section: Image data compression">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>PCX image data are compressed using <a href="/wiki/Run-length_encoding" title="Run-length encoding">run-length encoding</a> (RLE), a simple <a href="/wiki/Lossless_compression" title="Lossless compression">lossless compression</a> algorithm that collapses a series of three or more consecutive bytes with identical values into a two-byte pair. The two most-significant bits of a byte are used to determine whether the given data represent a single <a href="/wiki/Pixel" title="Pixel">pixel</a> of a given palette index or color value, or an RLE pair representing a series of several pixels of a single value:</p>
<ol>
<li>if both bits are 1, the byte is interpreted as the run length. This leaves 6 bits for the actual run length value, i.e. a value range of 0-63</li>
<li>in any other case, the byte is interpreted as a single pixel value. This leaves all value for which bit #7 and bit #8 are not 1 at the same time. This requirement is not met by all values of 192 (binary 11000000) and above.</li>
</ol>
<p>Compared to the maximum run length of 128, possible with <a href="/wiki/Truevision_TGA" title="Truevision TGA">TGA</a> RLE compression, the PCX run-length encoding offers a larger single-pixel value range, while the maximum run length is restricted to 63.</p>
<p>Due to the use of the two most-significant bits as flags, pixel values from 192 to 255 (with their most-significant bit already set) must be stored in an RLE byte pair, even when they only occur one or two pixels in succession, whereas color indexes 0 to 191 can be stored directly <i>or</i> in RLE byte pairs (whichever is more space-efficient); therefore, the actual compression ratio could be optimized with proper sorting of palette entries, though this is not feasible where the file must share its color palette with other images. For example, a palette could be optimized with the most commonly used colors occurring in palette positions 0 to 191 and the least common colors allocated to the remaining quarter of the palette.</p>
<p>Another inefficiency with the RLE algorithm is that it is possible to store chunks with a length of 0, which allows whitespace in the file. This allowed PCX files to be decompressed slightly faster on the processors it was originally intended for. This quirk could be used for <a href="/wiki/Steganography" title="Steganography">steganography</a>.</p>
<p>The PCX compression algorithm requires very little processor power or <a class="mw-redirect" href="/wiki/Random_access_memory" title="Random access memory">memory</a> to apply, a significant concern with the computer systems when it was designed. As computers and display hardware grow more sophisticated, the PCX algorithm becomes less space-efficient. Compression algorithms used by newer image formats are more efficient when compressing images such as photographs, and <a class="mw-redirect" href="/wiki/Dithering" title="Dithering">dithered</a> or otherwise complex graphics.</p>
<h3><span class="mw-headline" id="Color_palette">Color palette</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=PCX&amp;action=edit&amp;section=5" title="Edit section: Color palette">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>A PCX file has space in its header for a 16 color palette. When 256-color VGA hardware became available there was not enough space for the palette in a PCX file; even the 54 unused bytes after the header would not be enough. The solution chosen was to put the palette at the end of the file, along with a marker byte to confirm its existence.</p>
<p>If a PCX file has a 256-color palette, it is found 768 bytes from the end of the file. In this case the value in the byte preceding the palette should be 12 (0x0C). The palette is stored as a sequence of RGB triples; its usable length is defined by the number of colors in the image. Colors values in a PCX palette always use 8 bits, regardless of the bit depth of the image.</p>
