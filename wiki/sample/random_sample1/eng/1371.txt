<h1 class="firstHeading" id="firstHeading" lang="en">Point-to-point protocol over Ethernet</h1>

<p>The <b>Point-to-Point Protocol over Ethernet</b> (<b>PPPoE</b>) is a <a class="mw-redirect" href="/wiki/Network_protocol" title="Network protocol">network protocol</a> for <a href="/wiki/Encapsulation_(networking)" title="Encapsulation (networking)">encapsulating</a> <a href="/wiki/Point-to-Point_Protocol" title="Point-to-Point Protocol">PPP</a> frames inside <a href="/wiki/Ethernet" title="Ethernet">Ethernet</a> frames. It appeared in 1999, in the context of the boom of <a class="mw-redirect" href="/wiki/DSL" title="DSL">DSL</a> as the solution for tunneling packets over the DSL connection to the <a class="mw-redirect" href="/wiki/ISP" title="ISP">ISP</a>'s <a href="/wiki/Internet_Protocol" title="Internet Protocol">IP</a> network, and from there to the rest of the Internet. A 2005 networking book noted that "Most DSL providers use PPPoE, which provides <a href="/wiki/Authentication" title="Authentication">authentication</a>, <a href="/wiki/Encryption" title="Encryption">encryption</a>, and <a href="/wiki/Data_compression" title="Data compression">compression</a>."<sup class="reference" id="cite_ref-Boney2005_1-0"><a href="#cite_note-Boney2005-1">[1]</a></sup> Typical use of PPPoE involves leveraging the PPP facilities for authenticating the user with a username and password, predominately via the <a class="mw-redirect" href="/wiki/Password_authentication_protocol" title="Password authentication protocol">PAP</a> protocol and less often via <a href="/wiki/Challenge-Handshake_Authentication_Protocol" title="Challenge-Handshake Authentication Protocol">CHAP</a>.<sup class="reference" id="cite_ref-GoldenDedieu2007_2-0"><a href="#cite_note-GoldenDedieu2007-2">[2]</a></sup></p>
<p>On the <a href="/wiki/Customer-premises_equipment" title="Customer-premises equipment">customer-premises equipment</a>, PPPoE may be implemented either in a unified <a href="/wiki/Residential_gateway" title="Residential gateway">residential gateway</a> device that handles both DSL <a href="/wiki/Modem" title="Modem">modem</a> and <a href="/wiki/IP_routing" title="IP routing">IP routing</a> functions or in the case of a simple <a href="/wiki/DSL_modem" title="DSL modem">DSL modem</a> (without routing support), PPPoE may be handled behind it on a separate Ethernet-only router or even directly on a user's computer. (Support for PPPoE is present in most operating systems, ranging from <a href="/wiki/Windows_XP" title="Windows XP">Windows XP</a>,<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup> <a href="/wiki/Linux" title="Linux">Linux</a><sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup> to <a class="mw-redirect" href="/wiki/Mac_OS_X" title="Mac OS X">Mac OS X</a>.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup>) More recently, some <a class="mw-redirect" href="/wiki/GPON" title="GPON">GPON</a>-based (instead of DSL-based) residential gateways also use PPPoE, although the status of PPPoE in the GPON standards is marginal.</p>
<p>PPPoE was developed by <a href="/wiki/UUNET" title="UUNET">UUNET</a>, <a href="/wiki/Redback_Networks" title="Redback Networks">Redback Networks</a> (now Ericsson) and RouterWare (now <a href="/wiki/Wind_River_Systems" title="Wind River Systems">Wind River Systems</a>) <sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup> and is available as an informational <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc2516" rel="nofollow">RFC 2516</a>.</p>
<p>In the world of DSL, PPPoE was commonly understood to be running on top of <a href="/wiki/Asynchronous_Transfer_Mode" title="Asynchronous Transfer Mode">ATM</a> (or DSL) as the underlying transport, although no such limitation exists in the PPPoE protocol itself. Other usage scenarios are sometimes distinguished by tacking as a suffix another underlying transport. For example, PPPoEoE, when the transport is Ethernet itself, as in the case of <a href="/wiki/Metro_Ethernet" title="Metro Ethernet">Metro Ethernet</a> networks. (In this notation, the original use of PPPoE would be labeled PPPoEoA, although it should not be confused with <a class="mw-redirect" href="/wiki/PPPoA" title="PPPoA">PPPoA</a>, which is a different encapsulation protocol.)</p>
<p>PPPoE has been described in some books as a "<a class="mw-redirect" href="/wiki/Layer_2.5" title="Layer 2.5">layer 2.5</a>" protocol,<sup class="reference" id="cite_ref-GoldenDedieu2007_2-1"><a href="#cite_note-GoldenDedieu2007-2">[2]</a></sup><sup class="reference" id="cite_ref-Beck2005_7-0"><a href="#cite_note-Beck2005-7">[7]</a></sup> in some rudimentary sense similar to <a class="mw-redirect" href="/wiki/MPLS" title="MPLS">MPLS</a> because it can be used to distinguish different IP flows sharing an Ethernet infrastructure, although the lack of PPPoE switches making routing decision based on PPPoE headers limits applicability in that respect.<sup class="reference" id="cite_ref-Beck2005_7-1"><a href="#cite_note-Beck2005-7">[7]</a></sup></p>

<h2><span class="mw-headline" id="Original_rationale">Original rationale</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Point-to-point_protocol_over_Ethernet&amp;action=edit&amp;section=1" title="Edit section: Original rationale">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In late 1998 the DSL service model had a chicken-and-egg problem. ADSL technology had been proposed a decade earlier.<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> Potential <a href="/wiki/Redback_Networks" title="Redback Networks">equipment vendors</a> and <a href="/wiki/UUNET" title="UUNET">carriers</a> alike recognized that broadband such as <a href="/wiki/Cable_modem" title="Cable modem">cable modem</a> or <a class="mw-redirect" href="/wiki/DSL" title="DSL">DSL</a> would eventually replace <a class="mw-redirect" href="/wiki/Dialup" title="Dialup">dialup</a> service, but the hardware (both customer premises and LEC) faced a significant <a href="/wiki/Economies_of_scale" title="Economies of scale">low-quantity</a> <a href="/wiki/Barriers_to_entry" title="Barriers to entry">cost barrier</a>. Initial estimates for low-quantity deployment of DSL showed costs in the $300–$500 range for a DSL modem and $300/mo access fee from the telco<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (December 2011)">citation needed</span></a></i>]</sup> which was well beyond what a home user would pay. Thus the initial focus was on small &amp; home business customers for whom a <a href="/wiki/Digital_Signal_1" title="Digital Signal 1">T1 line</a> (at the time $800–$1500 per month) was not economical, but who needed more than dialup or <a class="mw-redirect" href="/wiki/ISDN" title="ISDN">ISDN</a> could deliver. If enough of these customers paved the way, quantities would drive the prices down to where the home-use dialup user might be interested: more like $50 for the modem and $50/mo for the access.</p>
<h3><span class="mw-headline" id="Different_usage_profile">Different usage profile</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Point-to-point_protocol_over_Ethernet&amp;action=edit&amp;section=2" title="Edit section: Different usage profile">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The problem was that small business customers had a different usage profile than a home-use dialup user, including: connecting an entire LAN to the internet; Providing services on a local LAN accessible from the far side of the connection; Simultaneous access to multiple external data sources, such as a company VPN and a general purpose ISP; Continuous usage throughout the workday, or even around the clock.</p>
<p>These requirements didn't lend themselves to the connection establishment lag of a dialup process nor its one-computer-to-one-ISP model, nor even the many-to-one that <a href="/wiki/Network_address_translation" title="Network address translation">NAT</a> + dialup provided. A new model was required.</p>
<p>PPPoE is used mainly either:</p>
<ul>
<li>with PPPoE-speaking Internet <a class="mw-redirect" href="/wiki/DSL" title="DSL">DSL</a> services where a PPPoE-speaking <a href="/wiki/Modem" title="Modem">modem</a>-<a href="/wiki/Router_(computing)" title="Router (computing)">router</a> (<a href="/wiki/Residential_gateway" title="Residential gateway">residential gateway</a>) connects to the DSL service. Here both ISP and modem-router need to speak PPPoE. (Note that in this case, the PPPoE-over-DSL side of things is occasionally referred to as <i>PPPoEoA</i>, for ‘PPPoE over <a href="/wiki/Asynchronous_Transfer_Mode" title="Asynchronous Transfer Mode">ATM</a>’.)</li>
<li>or when a PPPoE-speaking <i>DSL modem</i> is connected to a PPPoE-speaking Ethernet-only router using an Ethernet cable.</li>
</ul>
<h3><span class="mw-headline" id="Time_to_market:_simpler_is_better">Time to market: simpler is better</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Point-to-point_protocol_over_Ethernet&amp;action=edit&amp;section=3" title="Edit section: Time to market: simpler is better">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>A problem with creating a completely new protocol to fill these needs was time. The equipment was available immediately, as was the service, and a whole new protocol stack (<a href="/wiki/Microsoft" title="Microsoft">Microsoft</a> at the time was advocating fiber-based atm-cells-to-the-desktop,<sup class="reference" id="cite_ref-Partners_9-0"><a href="#cite_note-Partners-9">[9]</a></sup> and <a class="mw-redirect" href="/wiki/L2TP" title="L2TP">L2TP</a> was brewing as well, but was not near completion) would take so long to implement that the window of opportunity might slip by. Several decisions were made to simplify implementation and standardization in an effort to deliver a complete solution quickly.</p>
<h4><span class="mw-headline" id="Reuse_existing_software_stacks">Reuse existing software stacks</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Point-to-point_protocol_over_Ethernet&amp;action=edit&amp;section=4" title="Edit section: Reuse existing software stacks">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>PPPoE hoped to merge the widespread Ethernet infrastructure with the ubiquitous PPP, allowing vendors to reuse their existing software and deliver products in the very near term. Essentially all operating systems at the time had a PPP stack, and the design of PPPoE allowed for a simple shim at the line-encoding stage to convert from PPP to PPPoE<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (January 2017)">citation needed</span></a></i>]</sup>.</p>
<h4><span class="mw-headline" id="Simplify_hardware_requirements">Simplify hardware requirements</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Point-to-point_protocol_over_Ethernet&amp;action=edit&amp;section=5" title="Edit section: Simplify hardware requirements">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Competing WAN technologies (T1, ISDN) required a <a href="/wiki/Router_(computing)" title="Router (computing)">router</a> on the customer premises. PPPoE used a different Ethernet frame type, which allowed the DSL hardware to function as simply a <a href="/wiki/Bridging_(networking)" title="Bridging (networking)">bridge</a>, passing some frames to the WAN and ignoring the others. Implementation of such a bridge is multiple orders of magnitude simpler than a router.</p>
<h4><span class="mw-headline" id="Informational_RFC">Informational RFC</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Point-to-point_protocol_over_Ethernet&amp;action=edit&amp;section=6" title="Edit section: Informational RFC">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p><a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc2516" rel="nofollow">RFC 2516</a> was initially released as an <a class="mw-redirect" href="/wiki/Request_for_comments#Status_.22informational.22" title="Request for comments">informational</a> (rather than <a class="mw-redirect" href="/wiki/Internet_standards" title="Internet standards">standards-track</a>) RFC for the same reason: the adoption period for a standards-track RFC was prohibitively long.</p>
<h3><span class="mw-headline" id="Success">Success</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Point-to-point_protocol_over_Ethernet&amp;action=edit&amp;section=7" title="Edit section: Success">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>PPPoE was initially designed to provide a small LAN with individual independent connections to the internet at large, but also such that the protocol itself would be lightweight enough that it wouldn't impinge on the hoped-for home usage market when it finally arrived. While success on the second matter may be debated (some complain that 8 bytes per packet is too much) PPPoE clearly succeeded in bringing sufficient volume to drive the price for service down to what a home user would pay. It remains the dominant DSL connectivity mechanism as of 2011, more than a decade later.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (December 2013)">citation needed</span></a></i>]</sup></p>
<h2><span class="mw-headline" id="PPPoE_stages">PPPoE stages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Point-to-point_protocol_over_Ethernet&amp;action=edit&amp;section=8" title="Edit section: PPPoE stages">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The PPPoE has two distinct stages:</p>
<h3><span class="mw-headline" id="PPPoE_discovery">PPPoE discovery</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Point-to-point_protocol_over_Ethernet&amp;action=edit&amp;section=9" title="Edit section: PPPoE discovery">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Since traditional PPP connections are established between two end points over a serial link or over an ATM virtual circuit that has already been established during dial-up, all PPP frames sent on the wire are sure to reach the other end. But Ethernet networks are multi-access where each node in the network can access every other node. An Ethernet frame contains the hardware address of the destination node (<a href="/wiki/MAC_address" title="MAC address">MAC address</a>). This helps the frame reach the intended destination.</p>
<p>Hence before exchanging PPP control packets to establish the connection over Ethernet, the MAC address of the two end points should be known to each other so that they can be encoded in these control packets. The PPPoE Discovery stage does exactly this. In addition it also helps establish a Session ID that can be used for further exchange of packets.</p>
<h3><span class="mw-headline" id="PPP_session">PPP session</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Point-to-point_protocol_over_Ethernet&amp;action=edit&amp;section=10" title="Edit section: PPP session">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Once the MAC address of the peer is known and a session has been established, the Session stage will start.</p>
<h2><span class="mw-headline" id="PPPoE_Discovery_.28PPPoED.29">PPPoE Discovery (PPPoED)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Point-to-point_protocol_over_Ethernet&amp;action=edit&amp;section=11" title="Edit section: PPPoE Discovery (PPPoED)">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Although traditional PPP is a peer-to-peer protocol, PPPoE is inherently a client-server relationship since multiple hosts can connect to a service provider over a single physical connection.</p>
<p>The Discovery process consists of four steps between the host computer which acts as the client and the access concentrator at the internet service provider's end acts as the server. They are outlined below. The fifth and last step is the way to close an existing session.</p>
<h3><span class="mw-headline" id="Client_to_server:_Initiation_.28PADI.29">Client to server: Initiation (PADI)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Point-to-point_protocol_over_Ethernet&amp;action=edit&amp;section=12" title="Edit section: Client to server: Initiation (PADI)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><b>PADI</b> stands for PPPoE Active Discovery Initiation.<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup></p>
<p>If a user wants to "dial up" to the Internet using DSL, then their computer first must find the DSL access concentrator (DSL-AC) at the user's <a href="/wiki/Internet_service_provider" title="Internet service provider">Internet service provider</a>'s <a href="/wiki/Point_of_presence" title="Point of presence">point of presence</a> (POP). Communication over Ethernet is only possible via <a href="/wiki/MAC_address" title="MAC address">MAC addresses</a>. As the computer does not know the MAC address of the DSL-AC, it sends out a PADI packet via an Ethernet <a class="mw-redirect" href="/wiki/Broadcasting_(networks)" title="Broadcasting (networks)">broadcast</a> (MAC: ff:ff:ff:ff:ff:ff). This PADI packet contains the MAC address of the computer sending it.</p>
<p>Example of a PADI-packet:</p>
<pre>
Frame 1 (44 bytes on wire, 44 bytes captured)
Ethernet II, Src: 00:50:da:42:d7:df, Dst: ff:ff:ff:ff:ff:ff
PPP-over-Ethernet Discovery
  Version: 1
  Type 1
  Code Active Discovery Initiation (PADI)
  Session ID: 0000
  Payload Length: 24
PPPoE Tags
  Tag: Service-Name
  Tag: Host-Uniq
    Binary Data: (16 bytes)
</pre>
<p><i>Src.</i> (=source) holds the MAC address of the computer sending the PADI.<br/>
<i>Dst.</i> (=destination) is the Ethernet broadcast address.<br/>
The PADI packet can be received by more than one DSL-AC. Only DSL-AC equipment that can serve the "Service-Name" tag should reply.</p>
<h3><span class="mw-headline" id="Server_to_client:_Offer_.28PADO.29">Server to client: Offer (PADO)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Point-to-point_protocol_over_Ethernet&amp;action=edit&amp;section=13" title="Edit section: Server to client: Offer (PADO)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><b>PADO</b> stands for PPPoE Active Discovery Offer.<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup></p>
<p>Once the user's computer has sent the PADI packet, the DSL-AC replies with a PADO packet, using the MAC address supplied in the PADI. The PADO packet contains the MAC address of the DSL-AC, its name (e.g. LEIX11-erx for the T-Com DSL-AC in <a href="/wiki/Leipzig" title="Leipzig">Leipzig</a>) and the name of the service. If more than one POP's DSL-AC replies with a PADO packet, the user's computer selects the DSL-AC for a particular POP using the supplied name or service.</p>
<p>Here is an example of a PADO packet:</p>
<pre>
Frame 2 (60 bytes on wire, 60 bytes captured)
Ethernet II, Src: 00:0e:40:7b:f3:8a, Dst: 00:50:da:42:d7:df
PPP-over-Ethernet Discovery
  Version: 1
  Type 1
  Code Active Discovery Offer (PADO)
  Session ID: 0000
  Payload Length: 36
PPPoE Tags
  Tag: AC-Name
    String Data: IpzbrOOl
  Tag: Host-Uniq
    Binary Data: (16 bytes)
</pre>
<p><i>AC-Name -&gt; String data</i> holds the AC name, in this case “Ipzbr001” (the Arcor DSL-AC in Leipzig)<br/>
<i>Src.</i> holds the MAC address of the DSL-AC.<br/>
The MAC address of the DSL-AC also reveals the manufacturer of the DSL-AC (in this case <a class="mw-redirect" href="/wiki/Nortel_Networks" title="Nortel Networks">Nortel Networks</a>).</p>
<h3><span class="mw-headline" id="Client_to_server:_request_.28PADR.29">Client to server: request (PADR)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Point-to-point_protocol_over_Ethernet&amp;action=edit&amp;section=14" title="Edit section: Client to server: request (PADR)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><b>PADR</b> stands for PPPoE active discovery request.<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup></p>
<p>A PADR packet is sent by the user's computer to the DSL-AC following receipt of an acceptable PADO packet from the DSL-AC. It confirms acceptance of the offer of a PPPoE connection made by the DSL-AC issuing the PADO packet.</p>
<h3><span class="mw-headline" id="Server_to_client:_session-confirmation_.28PADS.29">Server to client: session-confirmation (PADS)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Point-to-point_protocol_over_Ethernet&amp;action=edit&amp;section=15" title="Edit section: Server to client: session-confirmation (PADS)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><b>PADS</b> stands for PPPoE Active Discovery Session-confirmation.<sup class="reference" id="cite_ref-13"><a href="#cite_note-13">[13]</a></sup></p>
<p>The PADR packet above is confirmed by the DSL-AC with a PADS packet, and a Session ID is given out with it. The connection with the DSL-AC for that POP has now been fully established.</p>
<h3><span class="mw-headline" id="Either_end_to_other_end:_termination_.28PADT.29">Either end to other end: termination (PADT)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Point-to-point_protocol_over_Ethernet&amp;action=edit&amp;section=16" title="Edit section: Either end to other end: termination (PADT)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><b>PADT</b> stands for PPPoE Active Discovery Termination.<sup class="reference" id="cite_ref-14"><a href="#cite_note-14">[14]</a></sup> This packet terminates the connection to the POP. It may be sent either from the user's computer or from the DSL-AC.</p>
<h2><span class="mw-headline" id="Protocol_overhead">Protocol overhead</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Point-to-point_protocol_over_Ethernet&amp;action=edit&amp;section=17" title="Edit section: Protocol overhead">edit</a><span class="mw-editsection-bracket">]</span></span></h2>

<p>PPPoE over ATM has the highest overhead of the popular DSL delivery methods.<sup class="reference" id="cite_ref-15"><a href="#cite_note-15">[15]</a></sup><sup class="noprint Inline-Template" style="margin-left:0.1em; white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Please_clarify" title="Wikipedia:Please clarify"><span title="The text near this tag may need clarification or removal of jargon. (December 2013)">clarification needed</span></a></i>]</sup></p>
<p><b>On ATM/DSL (aka PPPoEoA):</b> The amount of overhead added by PPPoEoA on the DSL Link depends on the packet size because of (i) the absorbing effect of ATM cell-padding which completely cancels out overheads of PPPoEoA in some cases, (ii) the PPPoE overhead can cause an entire additional 53 byte ATM cell to be required, and (iii) IP fragmentation can be induced (which also invokes the first two effects).<sup class="reference" id="cite_ref-16"><a href="#cite_note-16">[16]</a></sup> However ignoring ATM and IP fragmentation for the moment, the basic additional protocol header overheads for <b><a class="mw-redirect" href="/wiki/AAL5" title="AAL5">AAL5</a> payload</b> are: 2 bytes (for PPP) + 6 (for PPPoE) + 18 (Ethernet MAC, variable) + 10 (<a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc2684" rel="nofollow">RFC 2684</a> LLC, variable) = <b>36 bytes</b>. For very small packets this overhead may be even greater because of possible Ethernet frame padding. However more realistically, the typical DSL/ATM overhead is either zero or 53 bytes as in case (ii) where the ATM last-cell payload maximum of 48-8 bytes is exceeded, so requiring an additional 53 byte ATM cell. The 36 byte figure deduced earlier can be slightly reduced by discarding the Ethernet FCS which loses 4 bytes of overhead, bringing the total down to 32.</p>
<p>Compare this with a vastly more header-efficient protocol, PPPoA <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc2364" rel="nofollow">RFC 2364</a> VC-MUX, with a fixed 2 byte overhead inside the AAL5 payload on the DSL side.</p>
<p><b>On Ethernet:</b> On an Ethernet LAN, overhead for PPP + PPPoE is a fixed 2 + 6 = <b>8 bytes</b> as before, unless IP fragmentation is produced.</p>
<h3><span class="mw-headline" id="MTU.2FMRU">MTU/MRU</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Point-to-point_protocol_over_Ethernet&amp;action=edit&amp;section=18" title="Edit section: MTU/MRU">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>When the DSL modem sends and receives PPPoE-containing Ethernet frames <i>across the Ethernet link to the router</i> (or PPPoE-speaking single PC), there is an overhead of 8 bytes (2 for PPP, 6 for PPPoE) added within the payload of the Ethernet frame. This added overhead can mean that a reduced limit (so-called <i>‘<a href="/wiki/Maximum_transmission_unit" title="Maximum transmission unit">MTU</a>’</i> or <i>‘<a href="/wiki/Maximum_transmission_unit" title="Maximum transmission unit">MRU</a>’</i>) of 1492 bytes is imposed on the length of IP packets sent or received, as opposed to the usual 1500 bytes for Ethernet networks. Some devices support <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc4638" rel="nofollow">RFC 4638</a>, which allows negotiation for the use of 1508-byte long, non-standard Ethernet frames, sometimes called ‘baby <a class="mw-redirect" href="/wiki/Jumbo_frames" title="Jumbo frames">jumbo frames</a>’, so allowing a full 1500 byte payload to be transported over a PPPoE connection. This capability is advantageous because of the many companies who (incorrectly) block all <a href="/wiki/Internet_Control_Message_Protocol" title="Internet Control Message Protocol">ICMP</a> traffic from exiting their network, a bad practice which prevents <a class="mw-redirect" href="/wiki/Path_MTU_discovery" title="Path MTU discovery">path MTU discovery</a> from working correctly and which can cause problems for users accessing such networks who have an MTU of less than 1500 bytes.</p>
<p><b>How PPPoE fits in the DSL Internet access architecture</b></p>
<p>The transport protocol used on the telephone network is <a href="/wiki/Asynchronous_Transfer_Mode" title="Asynchronous Transfer Mode">ATM</a>. The DSL modem encapsulates PPP packets inside ATM cells and sends them over the WAN. There are several encapsulation methods.</p>
<h3><span class="mw-headline" id="PPPoE_to_PPPoA">PPPoE to PPPoA</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Point-to-point_protocol_over_Ethernet&amp;action=edit&amp;section=19" title="Edit section: PPPoE to PPPoA">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In this diagram, the scenario is the same as before with two exceptions: (i) A <b>more efficient modem</b> architecture is shown, the modem being a PPPoE-to-<a href="/wiki/Point-to-Point_Protocol_over_ATM" title="Point-to-Point Protocol over ATM">PPPoA</a> protocol converter. (ii) The service provider offers a PPPoA service and does not understand PPPoE. There is no PPPoEoA in this protocol chain.</p>
<p>In this alternative topology, PPPoE is merely a means of connecting DSL-modems to an Ethernet-only router (again, or to a single host PC). Here it is not concerned with the mechanism employed by an ISP to offer broadband services.</p>
<p>When transmitting packets bound for the internet, the PPPoE-speaking Ethernet router sends Ethernet frames to the (also PPPoE-speaking) DSL modem. The modem extracts PPP frames from within the received PPPoE frames, and sends the PPP frames onwards to the DSLAM by encapsulating them according to <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc2364" rel="nofollow">RFC 2364</a> (PPPoA), thus converting PPPoE into PPPoA.</p>

<h2><span class="mw-headline" id="Quirks">Quirks</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Point-to-point_protocol_over_Ethernet&amp;action=edit&amp;section=20" title="Edit section: Quirks">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Since the point-to-point connection established has a <a class="mw-redirect" href="/wiki/MTU_(networking)" title="MTU (networking)">MTU</a> lower than that of standard Ethernet (typically 1492 vs Ethernet's 1500), it can sometimes cause problems when <a href="/wiki/Path_MTU_Discovery" title="Path MTU Discovery">Path MTU Discovery</a> is defeated by poorly configured <a class="mw-redirect" href="/wiki/Firewall_(networking)" title="Firewall (networking)">firewalls</a>. Although higher MTUs are becoming more common in providers' networks, usually the workaround is to use TCP MSS (Maximum Segment Size) "clamping" or "rewrite", whereby the access concentrator rewrites the MSS to ensure TCP peers send smaller datagrams. Although TCP MSS clamping solves the MTU issue for TCP, other protocols such as ICMP and UDP may still be affected.</p>
<p><a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc4638" rel="nofollow">RFC 4638</a> allows PPPoE devices to negotiate an MTU of greater than 1492 if the underlying Ethernet layer is capable of <a class="mw-redirect" href="/wiki/Jumbo_frames" title="Jumbo frames">jumbo frames</a>.</p>
<p>Some vendors (<a href="/wiki/Cisco_Systems" title="Cisco Systems">Cisco</a><sup class="reference" id="cite_ref-cisco.com_17-0"><a href="#cite_note-cisco.com-17">[17]</a></sup> and <a href="/wiki/Juniper_Networks" title="Juniper Networks">Juniper</a>,<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (December 2013)">citation needed</span></a></i>]</sup> for example) distinguish PPPoE[oA] from PPPoEoE (PPPoE over Ethernet), which is PPPoE running directly over Ethernet or other <a href="/wiki/IEEE_802" title="IEEE 802">IEEE 802</a> networks or over Ethernet <a href="/wiki/Bridging_(networking)" title="Bridging (networking)">bridged</a> over <a href="/wiki/Asynchronous_Transfer_Mode" title="Asynchronous Transfer Mode">ATM</a>, in order to distinguish it from PPPoEoA (PPPoE over ATM), which is PPPoE running over an ATM virtual circuit using <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc2684" rel="nofollow">RFC 2684</a> and <a href="/wiki/Subnetwork_Access_Protocol" title="Subnetwork Access Protocol">SNAP</a> encapsulation of PPPoE.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (December 2013)">citation needed</span></a></i>]</sup> (PPPoEoA is not the same as <a href="/wiki/Point-to-Point_Protocol_over_ATM" title="Point-to-Point Protocol over ATM">Point-to-Point Protocol over ATM</a> (PPPoA), which doesn't use SNAP).</p>
<p>According to a Cisco document "PPPoEoE is a variant of PPPoE where the Layer 2 transport protocol is now Ethernet or 802.1q VLAN instead of ATM. This encapsulation method is generally found in <a href="/wiki/Metro_Ethernet" title="Metro Ethernet">Metro Ethernet</a> or Ethernet digital subscriber line access multiplexer (DSLAM) environments. The common deployment model is that this encapsulation method is typically found in multi-tenant buildings or hotels. By delivering Ethernet to the subscriber, the available bandwidth is much more abundant and the ease of further service delivery is increased."<sup class="reference" id="cite_ref-cisco.com_17-1"><a href="#cite_note-cisco.com-17">[17]</a></sup></p>
<p>It is possible to find DSL modems, such as the Draytek Vigor 120, where PPPoE is confined to the ethernet link between a DSL modem and a partnering router, and the ISP does not speak PPPoE at all (but rather <a class="mw-redirect" href="/wiki/PPPoA" title="PPPoA">PPPoA</a>).<sup class="reference" id="cite_ref-18"><a href="#cite_note-18">[18]</a></sup></p>
<h2><span class="mw-headline" id="Post-DSL_uses_and_some_alternatives_in_these_contexts">Post-DSL uses and some alternatives in these contexts</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Point-to-point_protocol_over_Ethernet&amp;action=edit&amp;section=21" title="Edit section: Post-DSL uses and some alternatives in these contexts">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>A certain method of using PPPoE in conjunction with <a class="mw-redirect" href="/wiki/GPON" title="GPON">GPON</a> (which involves creating a <a class="mw-redirect" href="/wiki/VLAN" title="VLAN">VLAN</a> via <a class="new" href="/w/index.php?title=OMCI&amp;action=edit&amp;redlink=1" title="OMCI (page does not exist)">OMCI</a>) has been patented by <a href="/wiki/ZTE" title="ZTE">ZTE</a>.<sup class="reference" id="cite_ref-19"><a href="#cite_note-19">[19]</a></sup></p>
<p>PPPoE over GPON is reportedly used by retail service providers such as <a href="/wiki/Internode_(ISP)" title="Internode (ISP)">Internode</a> of Australia's <a href="/wiki/National_Broadband_Network" title="National Broadband Network">National Broadband Network</a>,<sup class="reference" id="cite_ref-20"><a href="#cite_note-20">[20]</a></sup> Romania's <a href="/wiki/RCS_%26_RDS" title="RCS &amp; RDS">RCS &amp; RDS</a> (for their "Fiberlink" customers — GPON is sold as Ethernet ports in <a href="/wiki/Multi-family_residential" title="Multi-family residential">MDUs</a>).,<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (December 2013)">citation needed</span></a></i>]</sup> <a href="/wiki/Orange_S.A." title="Orange S.A.">Orange</a> France<sup class="reference" id="cite_ref-21"><a href="#cite_note-21">[21]</a></sup> and Philippines' <a href="/wiki/Globe_Telecom" title="Globe Telecom">Globe Telecom</a>.<sup class="reference" id="cite_ref-22"><a href="#cite_note-22">[22]</a></sup> <a class="mw-redirect" href="/wiki/Verizon" title="Verizon">Verizon</a>'s <a class="mw-redirect" href="/wiki/FIOS" title="FIOS">FIOS</a> product uses <a class="mw-redirect" href="/wiki/DHCP" title="DHCP">DHCP</a> in some states and PPPoE in others.<sup class="reference" id="cite_ref-23"><a href="#cite_note-23">[23]</a></sup></p>
<p><a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc6934" rel="nofollow">RFC 6934</a> "Applicability of Access Node Control Mechanism to PON based Broadband Networks", which argues for the use of <a class="new" href="/w/index.php?title=Access_Node_Control_Protocol&amp;action=edit&amp;redlink=1" title="Access Node Control Protocol (page does not exist)">Access Node Control Protocol</a> in PONs for—among other things—authenticating subscriber access and managing their IP addresses, and the first author of which is a Verizon employee, excludes PPPoE as an acceptable encapsulation for GPON: "The protocol encapsulation on BPON is based on multi-protocol encapsulation over ATM Adaptation Layer 5 (AAL5), defined in [RFC2684]. This covers PPP over Ethernet (PPPoE, defined in [RFC2516]) or IP over Ethernet (IPoE). The protocol encapsulation on GPON is always IPoE."<sup class="reference" id="cite_ref-24"><a href="#cite_note-24">[24]</a></sup></p>
<p>The <a href="/wiki/10G-PON" title="10G-PON">10G-PON</a> (XG-PON) standard (<a class="mw-redirect" href="/wiki/G.987" title="G.987">G.987</a>) provides for <a class="mw-redirect" href="/wiki/802.1X" title="802.1X">802.1X</a> mutual authentication of the ONU and OLT, besides the OMCI method carried forward from <a href="/wiki/G.984" title="G.984">G.984</a>.<sup class="reference" id="cite_ref-25"><a href="#cite_note-25">[25]</a></sup> G.987 also adds support for authenticating other <a href="/wiki/Customer-premises_equipment" title="Customer-premises equipment">customer-premises equipment</a> beyond the ONU (e.g. in a MDU), although this is limited to Ethernet ports, also handled via 802.1X. (The ONU is supposed snoop <a href="/wiki/Extensible_Authentication_Protocol" title="Extensible Authentication Protocol">EAP</a>-encapsulated <a href="/wiki/RADIUS" title="RADIUS">RADIUS</a> messages in this scenario and determine if the authentication was successful or not.)<sup class="reference" id="cite_ref-26"><a href="#cite_note-26">[26]</a></sup> There is some modicum support for PPPoE specified in the OMCI standards, but only in terms of the ONU being able to filter and add VLAN tags for traffic based on its encapsulation (and other parameters), which includes PPPoE among the protocols that ONU must be able to discern.<sup class="reference" id="cite_ref-27"><a href="#cite_note-27">[27]</a></sup></p>
<p>The <a href="/wiki/Broadband_Forum" title="Broadband Forum">Broadband Forum</a>'s TR-200 "Using <a class="mw-redirect" href="/wiki/Ethernet_passive_optical_network" title="Ethernet passive optical network">EPON</a> in the Context of TR-101" (2011), which also pertains to <a href="/wiki/10G-EPON" title="10G-EPON">10G-EPON</a>, says "The OLT and the multiple-subscriber ONU MUST be able to perform the PPPoE Intermediate Agent function, as specified in Section 3.9.2/TR-101."<sup class="reference" id="cite_ref-28"><a href="#cite_note-28">[28]</a></sup></p>
<p>A book on <a href="/wiki/Ethernet_in_the_first_mile" title="Ethernet in the first mile">Ethernet in the first mile</a> notes that DHCP can obviously be used instead of PPPoE to configure a host for an IP session, although it points out that DHCP is not a complete replacement for PPPoE if some encapsulation is also desired (although VLAN bridges can fulfill this function) and that furthermore DHCP does not provide (subscriber) authentication, suggesting that IEEE 802.1X is also needed for a "complete solution" sans PPPoE.<sup class="reference" id="cite_ref-Beck2005b_29-0"><a href="#cite_note-Beck2005b-29">[29]</a></sup> (This book assumes that PPPoE is leveraged for other features of PPP besides encapsulation, including <a class="mw-redirect" href="/wiki/IPCP" title="IPCP">IPCP</a> for host configuration, and <a class="mw-redirect" href="/wiki/Password_authentication_protocol" title="Password authentication protocol">PAP</a> or <a href="/wiki/Challenge-Handshake_Authentication_Protocol" title="Challenge-Handshake Authentication Protocol">CHAP</a> for authentication.)</p>
<p>There are security reasons to use PPPoE in a (non-DSL/ATM) shared-medium environment, such as <a class="mw-redirect" href="/wiki/Power_line_communication" title="Power line communication">power line communication</a> networks, in order to create separate tunnels for each customer.<sup class="reference" id="cite_ref-Carcelle2009_30-0"><a href="#cite_note-Carcelle2009-30">[30]</a></sup></p>
