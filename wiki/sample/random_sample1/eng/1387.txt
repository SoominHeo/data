<p><b>Perforce Helix</b> (originally just called Perforce) is a commercial, proprietary <a class="mw-redirect" href="/wiki/Revision_control" title="Revision control">revision control</a> system developed by <a href="/wiki/Perforce_Software" title="Perforce Software">Perforce Software</a>.</p>

<h2><span class="mw-headline" id="Architecture">Architecture</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Perforce_Helix&amp;action=edit&amp;section=1" title="Edit section: Architecture">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The Perforce Helix server manages a central <a href="/wiki/Database" title="Database">database</a> and a master <a class="extiw" href="https://en.wiktionary.org/wiki/repository" title="wikt:repository">repository</a> of <a href="/wiki/Computer_file" title="Computer file">file</a> versions.</p>
<h3><span class="mw-headline" id="Server">Server</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Perforce_Helix&amp;action=edit&amp;section=2" title="Edit section: Server">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<h4><span class="mw-headline" id="Metadata">Metadata</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Perforce_Helix&amp;action=edit&amp;section=3" title="Edit section: Metadata">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>The Perforce Helix database is proprietary, preconfigured, and self-installed. It stores system-related <a href="/wiki/Metadata" title="Metadata">metadata</a> (file state, file attributes, branching and merging history, changelists, change descriptions, users, groups, labels, etc.). Files are identified by namespace (i.e., by OS-neutral filenames). File content itself is not stored in the database. <a href="/wiki/MD5" title="MD5">MD5</a> hashes of file content are stored in the database, however, and can be used to verify repository file integrity.</p>
<p>Database tables are stored as binary files. <a class="mw-redirect" href="/wiki/Transaction_checkpoint" title="Transaction checkpoint">Checkpoints and journals</a> are written as text files that can be compressed and offloaded. A database that has been corrupted by hardware failure or other catastrophe can be recovered from the most recent journal and checkpoint. Administrators must plan for disaster recovery by configuring database journaling and setting up regular checkpoints.</p>
<h4><span class="mw-headline" id="Depot">Depot</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Perforce_Helix&amp;action=edit&amp;section=4" title="Edit section: Depot">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Versioned file content is stored in a master directory hierarchy whose top levels are called "depots". Text file revisions are stored as <a href="/wiki/Revision_Control_System" title="Revision Control System">RCS</a> deltas,<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[notes 1]</a></sup> and binary file revisions are stored in their entirety.</p>
<p>The encoding used for text files in the repository is either <a href="/wiki/ASCII" title="ASCII">ASCII</a> or <a href="/wiki/UTF-8" title="UTF-8">UTF-8</a>,<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[2]</a></sup> depending on Perforce server configuration. Repository files are not encrypted. Revisions that are branches or copies of other revisions are virtual copies within the repository. All revisions are preserved by default; limits can be set on the number of revisions preserved. Obsolete revisions and files can be purged by the administrator. Repository files must be included in regular system backups.</p>
<h3><span class="mw-headline" id="Clients">Clients</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Perforce_Helix&amp;action=edit&amp;section=5" title="Edit section: Clients">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Clients to the Perforce Helix system fall into roughly five categories: <a class="mw-redirect" href="/wiki/Git_(software)" title="Git (software)">Git</a>, <a href="/wiki/Command_(computing)" title="Command (computing)">command</a>, <a class="mw-redirect" href="/wiki/GUI" title="GUI">GUI</a>, <a href="/wiki/World_Wide_Web" title="World Wide Web">web</a>, and <a href="/wiki/Plug-in_(computing)" title="Plug-in (computing)">plugin</a>. The Perforce system can make part or all of its content available as Git repositories. Users of Git and of other clients can work with the same file content and history. Git commits are visible to users of other clients as Perforce changelists, and vice versa. Users submit changed files together in <a class="mw-redirect" href="/wiki/Source_Code_Management#Common_vocabulary" title="Source Code Management">changelists</a>, which are applied as <a href="/wiki/Atomic_commit" title="Atomic commit">atomic commits</a>.</p>
<p>The original command interface is P4, the <a class="mw-redirect" href="/wiki/Command_line_interface" title="Command line interface">command-line</a> client. P4 can be used in any command <a href="/wiki/Shell_(computing)" title="Shell (computing)">shell</a> or <a href="/wiki/Scripting_language" title="Scripting language">script</a>. It produces <a class="mw-redirect" href="/wiki/Human-readable" title="Human-readable">human-readable</a> output by default, but can also produce <a href="/wiki/Tag_(metadata)" title="Tag (metadata)">tagged</a> text output and <a href="/wiki/Marshalling_(computer_science)" title="Marshalling (computer science)">marshalled</a> <a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a> and <a href="/wiki/Ruby_(programming_language)" title="Ruby (programming language)">Ruby</a> output. Native C++ and Java APIs to the Perforce client commands are also available, as are <a href="/wiki/Lua_(programming_language)" title="Lua (programming language)">Lua</a>, <a href="/wiki/Perl" title="Perl">Perl</a>, Python, PHP, Objective-C, and Ruby extensions built upon the C++ API.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[3]</a></sup></p>
<p>The command interfaces support the system's complete client functionality and can be used with OS-native <a href="/wiki/Filename" title="Filename">filename</a> syntax, as well as with Perforce's OS-neutral filename syntax.</p>
<p>Two GUI clients are available for users, the <a href="/wiki/Cross-platform" title="Cross-platform">cross-platform</a>, <a class="mw-redirect" href="/wiki/Qt_(toolkit)" title="Qt (toolkit)">Qt</a>-based P4V, and the <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Windows</a>-only P4Win (in legacy maintenance mode since 2008<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[4]</a></sup>). Both support the majority of end-user operations. An administration GUI client, P4Admin, supports a subset of administrative operations.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[5]</a></sup> P4Admin, like P4V, is <a href="/wiki/Cross-platform" title="Cross-platform">cross-platform</a> and <a class="mw-redirect" href="/wiki/Qt_(toolkit)" title="Qt (toolkit)">Qt</a>-based. P4V and P4Admin can be extended with applets written in JavaScript and HTML.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[6]</a></sup></p>
<p>Plugins are provided for the Eclipse<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[7]</a></sup> and Visual Studio<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[8]</a></sup> IDEs. Ultimate version of Intellij IDEA has its own Perforce integration plugin.<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[9]</a></sup></p>
<p>A web interface is provided by P4Web,<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[10]</a></sup> a program that is both a Perforce client and a stand-alone <a class="mw-redirect" href="/wiki/HTTP" title="HTTP">HTTP</a> <a class="mw-redirect" href="/wiki/Daemon_(computer_software)" title="Daemon (computer software)">daemon</a>. P4Web can be run as a shared <a href="/wiki/Web_server" title="Web server">web server</a> to provide read-only access to the Perforce file repository and metadata. It can also be run on a user's machine, enabling <a href="/wiki/Web_browser" title="Web browser">web browsers</a> to become the interface to client operations on the local machine.</p>
<p>The plugin interfaces are behind-the-scenes programs that integrate Perforce client functionality into third-party software. Perforce plugins are available for <a class="mw-redirect" href="/wiki/Desktop_environments" title="Desktop environments">desktop environments</a>, <a href="/wiki/Software_development" title="Software development">software development</a> tools, <a href="/wiki/Digital_asset" title="Digital asset">digital asset</a> development tools, <a href="/wiki/Software_build" title="Software build">software build</a> tools, <a href="/wiki/Code_review" title="Code review">code review</a> systems, <a href="/wiki/Defect_tracking" title="Defect tracking">defect tracking</a> systems, <a href="/wiki/Office_automation" title="Office automation">office automation</a> tools, <a href="/wiki/SQL" title="SQL">SQL</a> clients, and <a class="mw-redirect" href="/wiki/FTP" title="FTP">FTP</a> clients.</p>
<h3><span class="mw-headline" id="Distributed_and_remote_revision_control">Distributed and remote revision control</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Perforce_Helix&amp;action=edit&amp;section=6" title="Edit section: Distributed and remote revision control">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Perforce Helix has four mechanisms for providing revision control to distributed or remote teams; these mechanisms can be used independently or in combination. The first is a <a href="/wiki/Proxy_server" title="Proxy server">proxy server</a> that caches frequently read versions in order to reduce file access times for remote users. This mechanism accommodates closed development organizations where a centrally controlled file repository and a universally accessible database are desired.</p>
<p>The second mechanism, known as <i>remote depots</i>, lets users connected to one server access file versions managed by other servers. With remote depots, each organization has control of its own server and makes parts or all of its repository visible to other servers. This mechanism is used for loosely coupled development organizations where a <a href="/wiki/Peer-to-peer" title="Peer-to-peer">peer-to-peer</a> approach is desired.</p>
<p>The third mechanism, known as <i>replication</i>,<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[11]</a></sup> mirrors all (or some) of the repository data to another server. Replicas provide faster response time for remote users.</p>
<p>Finally, Perforce can be replicated to Git repositories, using the standard Git protocol and commands.</p>

<h2><span class="mw-headline" id="Concurrency_model">Concurrency model</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Perforce_Helix&amp;action=edit&amp;section=8" title="Edit section: Concurrency model">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The system offers a hybrid of <i>merge</i> and <i>lock</i> concurrency models.<sup class="reference" id="cite_ref-18"><a href="#cite_note-18">[notes 3]</a></sup> As with similar systems, users do not have to lock files in order to work on them and are required to resolve concurrent, committed changes before submitting their work. Users may optionally lock files to ensure that they won't have to resolve concurrent changes.</p>
<p>However, the Perforce model is slightly different from those of similar systems in that users are expected to let the system know in advance which files they intend to change, even if they don't mean to lock them. Giving advance notice puts files in a <i>pending changelist</i> that can be submitted to the server. It also enables the system to alert other users working on the same files. Thus users can tell when they are working in parallel and can take the opportunity to coordinate with one another before making changes that could otherwise be difficult to merge.</p>
<p>Perforce enforces this advanced notification requirement loosely by setting read-only permission on workspace files as it fetches them from the repository. Users can bypass the requirement, by choice or by necessity (when working offline, for example), simply by hijacking file permissions and modifying files as they see fit. It is up to the user, in these cases, to remember to use Perforce to reconcile offline work and put hijacked files in a pending changelist so they can be submitted. (It is also up to users to leave hijacked files writable after changing them. A read-only file that is not in a pending changelist is assumed by Perforce to be a candidate for update by replacement.)</p>
<h2><span class="mw-headline" id="Branching_and_merging">Branching and merging</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Perforce_Helix&amp;action=edit&amp;section=9" title="Edit section: Branching and merging">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>A file is uniquely identified by its complete filename, e.g., <code>//depot/trunk/src/item.cpp</code>. Any non-deleted revision of a file can be branched. Perforce Helix uses <i>inter-file branching</i>,<sup class="reference" id="cite_ref-19"><a href="#cite_note-19">[16]</a></sup> wherein branching creates a new file with a new name. For example, <code>my/index.php</code> may be branched into <code>your/index.php</code> and each file may then evolve independently. Repository paths are typically designated as containers for branched sets of files. For example, files in the <code>//depot/trunk</code> path may be branched as a set into a new <code>//depot/rel1.0</code> path, resulting in two sets of files evolving independently and between which changes can be merged.</p>
<p>In Perforce Helix, the operation that merges changes from one branch to another is called <i>integration</i>. Integration propagates changes from a set of donor files into a set of corresponding target files; optional <i>branch views</i> can store customized donor–target mappings. By default, integration propagates all outstanding donor changes. Donor changes can be limited or cherry-picked by changelist, date, label, filename, or filename pattern-matching. The system records all integrations, uses them to select common ancestors for file merging, and does not by default perform redundant or unnecessary integrations.</p>
<p>Merging is actually only one of three possible outcomes of an integration. The others are <i>ignoring</i> (aka "blocking") and <i>copying</i> (aka "promoting"). Merging is used to keep one set of files up to date with another. For example, a development branch may be kept up to date with its trunk through repeated merging. Ignoring disqualifies changes in one set of files from future integration into another. It is often used when a development branch must be up to date with, and yet divergent from, its trunk. Copying is typically used to promote the content of an up-to-date development branch into a trunk.</p>
<p>Branching also supports renamed and moved files. The 'move' command branches originals to new files and deletes the originals. A branched file is no different from an added file; branched files are peers, not offshoots, of their originals. The system keeps track of file origins, however, and refers to them when displaying the history of renamed files.</p>
<p>Perforce streams<sup class="reference" id="cite_ref-20"><a href="#cite_note-20">[17]</a></sup> offer a way to capture more information about a branch, including ancestry and included paths. This information is used to provide guidelines for branching and merging operations.</p>
<h2><span class="mw-headline" id="Access_control_and_security">Access control and security</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Perforce_Helix&amp;action=edit&amp;section=10" title="Edit section: Access control and security">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The Perforce Helix server stores file content in a master repository that, when properly installed, is inaccessible to users. User access to files is controlled by one or more Perforce <a href="/wiki/Superuser" title="Superuser">superusers</a>. A range of file access protection levels can be granted. Protections can be set for repository file <a href="/wiki/Path_(computing)" title="Path (computing)">paths</a>, <a href="/wiki/User_(computing)" title="User (computing)">users</a>, <a href="/wiki/Group_(computing)" title="Group (computing)">groups</a>, and <a href="/wiki/IP_address" title="IP address">IP address</a> <a href="/wiki/Subnetwork" title="Subnetwork">subnets</a>. The server can maintain an audit log of client access events for <a class="mw-redirect" href="/wiki/Sarbanes-Oxley_Act" title="Sarbanes-Oxley Act">Sarbanes-Oxley Act</a> (SOX) and other compliance requirements.</p>
<p>User authentication is controlled by the Perforce system administrator. <a href="/wiki/Password_strength" title="Password strength">Password strength</a> is configurable; <a href="/wiki/Ticket_(IT_security)" title="Ticket (IT security)">ticket</a>-based authentication can be configured as well. <i>Triggers</i> (custom scripts or programs that run at predefined events) can be set on many but not all Perforce user commands and used to extend user authentication (with <a class="mw-redirect" href="/wiki/LDAP" title="LDAP">LDAP</a> or <a href="/wiki/Single_sign-on" title="Single sign-on">SSO</a>, for example), to block or allow user commands, and to constrain or normalize file modifications. Triggers are run by the Perforce server and do not have access to client machines or workspaces.</p>
<p>Perforce Helix, like most version control systems, does not encrypt file content in the master repository or on user machines.<sup class="reference" id="cite_ref-21"><a href="#cite_note-21">[18]</a></sup><sup class="reference" id="cite_ref-22"><a href="#cite_note-22">[19]</a></sup><sup class="reference" id="cite_ref-23"><a href="#cite_note-23">[20]</a></sup> Perforce versions prior to 2012.1 cannot encrypt file content sent over the network. A <a href="/wiki/Tunneling_protocol" title="Tunneling protocol">tunneling protocol</a> (like <a class="mw-redirect" href="/wiki/VPN" title="VPN">VPN</a> or <a href="/wiki/Secure_Shell" title="Secure Shell">SSH</a>) must be used to secure network transfers with those versions.<sup class="reference" id="cite_ref-24"><a href="#cite_note-24">[21]</a></sup></p>
<p>The Perforce Helix client completely trusts the server, including writing arbitrary files anywhere in the local filesystem, and therefore running arbitrary code from the server.<sup class="reference" id="cite_ref-25"><a href="#cite_note-25">[22]</a></sup> That means the server has complete control over the client user's account, including reading and writing and sending all non-source code files of the user. In environments where the Perforce server is managed by a third party, this poses a significant threat to the client's security and privacy.</p>
<h2><span class="mw-headline" id="Availability">Availability</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Perforce_Helix&amp;action=edit&amp;section=11" title="Edit section: Availability">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Use of the Perforce Helix server is unrestricted and free for up to 5<sup class="reference" id="cite_ref-26"><a href="#cite_note-26">[23]</a></sup> users, 20 workspaces and unlimited files, or unlimited users and up to 1,000 files, without a license. A license must be purchased for more users or workspaces; licenses may be purchased in perpetuity or on a subscription basis. The Perforce versioning engine, clients, plugin software, tools, and APIs are also freely available.<sup class="reference" id="cite_ref-27"><a href="#cite_note-27">[24]</a></sup></p>
<p>Free licenses are available for <a class="mw-redirect" href="/wiki/Open_source" title="Open source">open-source</a> software development, school or classroom projects, and trial/evaluation periods. Use of the Perforce client and plugin software is unrestricted, as is online access to Perforce technical documentation.<sup class="reference" id="cite_ref-28"><a href="#cite_note-28">[25]</a></sup></p>
<p>The server and client software are released as pre-built <a href="/wiki/Executable" title="Executable">executables</a> for <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a>, <a href="/wiki/MacOS" title="MacOS">macOS</a>, <a href="/wiki/Linux" title="Linux">Linux</a>, <a href="/wiki/Solaris_(operating_system)" title="Solaris (operating system)">Solaris</a>, <a href="/wiki/FreeBSD" title="FreeBSD">FreeBSD</a>, and other <a href="/wiki/Operating_system" title="Operating system">operating systems</a>.</p>
