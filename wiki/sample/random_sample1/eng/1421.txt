<p><b>RPM Package Manager</b> (<b>RPM</b>) (originally <b>Red Hat Package Manager</b>; now a <a href="/wiki/Recursive_acronym" title="Recursive acronym">recursive acronym</a>) is a <a class="mw-redirect" href="/wiki/Package_management_system" title="Package management system">package management system</a>.<sup class="reference" id="cite_ref-max_5-0"><a href="#cite_note-max-5">[5]</a></sup> The name RPM refers to the following: the <tt>.rpm</tt> <a href="/wiki/File_format" title="File format">file format</a>, files in the .rpm file format, <a href="/wiki/Software" title="Software">software</a> packaged in such files, and the package manager program itself. RPM was intended primarily for <a href="/wiki/Linux_distribution" title="Linux distribution">Linux distributions</a>; the file format is the baseline package format of the <a href="/wiki/Linux_Standard_Base" title="Linux Standard Base">Linux Standard Base</a>.</p>
<p>Even though it was created for use in <a href="/wiki/Red_Hat_Linux" title="Red Hat Linux">Red Hat Linux</a>, RPM is now used in many <a href="/wiki/Linux_distribution" title="Linux distribution">Linux distributions</a>. It has also been ported to some other <a href="/wiki/Operating_system" title="Operating system">operating systems</a>, such as <a href="/wiki/NetWare" title="NetWare">Novell NetWare</a> (as of version 6.5 SP3) and <a href="/wiki/IBM_AIX" title="IBM AIX">IBM's AIX</a> (as of version 4).</p>
<p>An RPM package can contain an arbitrary set of files. The larger part of RPM files encountered are “binary RPMs” (or BRPMs) containing the compiled version of some software. There are also “source RPMs” (or SRPMs) files containing the <a href="/wiki/Source_code" title="Source code">source code</a> used to produce a package. These have an appropriate tag in the file header that distinguishes them from normal (B)RPMs, causing them to be extracted to /usr/src on installation. SRPMs customarily carry the file extension “.src.rpm” (.spm on file systems limited to 3 extension characters, e.g. old DOS <a href="/wiki/File_Allocation_Table" title="File Allocation Table">FAT</a>).</p>

<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=RPM_Package_Manager&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>RPM was originally written in 1997 by Erik Troan and Marc Ewing,<sup class="reference" id="cite_ref-author_1-2"><a href="#cite_note-author-1">[1]</a></sup> based on <code>pms</code>, <code>rpp</code>, and <code>pm</code> experiences.</p>
<p><code>pm</code> was written by Rik Faith and Doug Hoffman in May 1995 for Red Hat Software, its design and implementations influenced greatly by <code>pms</code>, a package management system by Faith and Kevin Martin in the fall of 1993 for the Bogus Linux Distribution. <code>pm</code> preserves the "<a href="/wiki/Pristine_Sources" title="Pristine Sources">Pristine Sources</a> + patches" paradigm of <code>pms</code>, while adding features and eliminating arbitrary limitations present in the implementation. <code>pm</code> provides greatly enhanced database support for tracking and verifying installed packages<sup class="reference" id="cite_ref-Max_RPM-Ch1_4-1"><a href="#cite_note-Max_RPM-Ch1-4">[4]</a></sup><sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup><sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup></p>
<h2><span class="mw-headline" id="Features">Features</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=RPM_Package_Manager&amp;action=edit&amp;section=2" title="Edit section: Features">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>For a <a href="/wiki/System_administrator" title="System administrator">system administrator</a> performing software installation and maintenance, the use of package management rather than manual building has advantages such as simplicity, consistency and the ability for these processes to be automated and non-interactive.</p>
<p>Features of RPM include:</p>
<ul>
<li>RPM packages can be cryptographically verified with <a href="/wiki/GNU_Privacy_Guard" title="GNU Privacy Guard">GPG</a> and <a href="/wiki/MD5" title="MD5">MD5</a></li>
<li>Original source archive(s) (e.g. <tt>.tar.gz</tt>, <tt>.tar.bz2</tt>) are included in SRPMs, making verification easier</li>
<li>PatchRPMs and DeltaRPMs, the RPM equivalent of a <a href="/wiki/Patch_(computing)" title="Patch (computing)">patch</a> file, can incrementally update RPM-installed software</li>
<li>Automatic build-time dependency evaluation.</li>
</ul>
<h2><span class="mw-headline" id="Local_operations">Local operations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=RPM_Package_Manager&amp;action=edit&amp;section=3" title="Edit section: Local operations">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Packages may come from within a particular distribution (for example <a href="/wiki/Red_Hat_Enterprise_Linux" title="Red Hat Enterprise Linux">Red Hat Enterprise Linux</a>) or be built for it by other parties (for example <a href="/wiki/RPM_Fusion" title="RPM Fusion">RPM Fusion</a> for Fedora).<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> Circular dependencies among mutually dependent RPMs (so-called "<a href="/wiki/Dependency_hell" title="Dependency hell">dependency hell</a>") can be problematic;<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup> in such cases a single installation command needs to specify all the relevant packages.</p>
<h3><span class="mw-headline" id="Repositories">Repositories</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=RPM_Package_Manager&amp;action=edit&amp;section=4" title="Edit section: Repositories">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>RPMs are often collected centrally in one or more <a href="/wiki/Software_repository" title="Software repository">repositories</a> on the internet. A site often has its own RPM repositories which may either act as local mirrors of such internet repositories or be locally maintained collections of useful RPMs.</p>
<h3><span class="mw-headline" id="Front_ends">Front ends</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=RPM_Package_Manager&amp;action=edit&amp;section=5" title="Edit section: Front ends">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Several <a href="/wiki/Front_and_back_ends" title="Front and back ends">front-ends</a> to RPM ease the process of obtaining and installing RPMs from repositories and help in resolving their dependencies. These include:</p>
<ul>
<li><a href="/wiki/Yellowdog_Updater,_Modified" title="Yellowdog Updater, Modified">yum</a> used in <a href="/wiki/Fedora_(operating_system)" title="Fedora (operating system)">Fedora</a>, <a href="/wiki/CentOS" title="CentOS">CentOS</a> 5, <a href="/wiki/Red_Hat_Enterprise_Linux" title="Red Hat Enterprise Linux">Red Hat Enterprise Linux</a> 5 and above, <a href="/wiki/Scientific_Linux" title="Scientific Linux">Scientific Linux</a>, <a href="/wiki/Yellow_Dog_Linux" title="Yellow Dog Linux">Yellow Dog Linux</a> and <a href="/wiki/Oracle_Linux" title="Oracle Linux">Oracle Linux</a></li>
<li><a href="/wiki/DNF_(software)" title="DNF (software)">DNF</a>, introduced in <a href="/wiki/Fedora_(operating_system)" title="Fedora (operating system)">Fedora</a> 18, default since <a class="mw-redirect" href="/wiki/List_of_Fedora_releases#Fedora_22" title="List of Fedora releases">22</a>.</li>
<li><a href="/wiki/Up2date" title="Up2date">up2date</a> used in <a href="/wiki/Red_Hat_Enterprise_Linux" title="Red Hat Enterprise Linux">Red Hat Enterprise Linux</a>, <a href="/wiki/CentOS" title="CentOS">CentOS</a> 3 and 4, and <a href="/wiki/Oracle_Linux" title="Oracle Linux">Oracle Linux</a></li>
<li><a class="mw-redirect" href="/wiki/Zypper" title="Zypper">Zypper</a> used in <a href="/wiki/MeeGo" title="MeeGo">MeeGo</a>,<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup> <a href="/wiki/OpenSUSE" title="OpenSUSE">openSUSE</a> and <a href="/wiki/SUSE_Linux_distributions" title="SUSE Linux distributions">SUSE Linux Enterprise</a></li>
<li><a href="/wiki/Urpmi" title="Urpmi">urpmi</a> used in <a href="/wiki/Mandriva_Linux" title="Mandriva Linux">Mandriva Linux</a>, <a href="/wiki/ROSA_Linux" title="ROSA Linux">ROSA Linux</a> and <a href="/wiki/Mageia" title="Mageia">Mageia</a></li>
<li><a class="mw-redirect" href="/wiki/Apt-rpm" title="Apt-rpm">apt-rpm</a>, a port of Debian's <a href="/wiki/Advanced_Packaging_Tool" title="Advanced Packaging Tool">Advanced Packaging Tool</a> (APT) used in <a class="new" href="/w/index.php?title=Ark_Linux&amp;action=edit&amp;redlink=1" title="Ark Linux (page does not exist)">Ark Linux</a>,<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup> <a href="/wiki/PCLinuxOS" title="PCLinuxOS">PCLinuxOS</a> and <a href="/wiki/ALT_Linux" title="ALT Linux">ALT Linux</a></li>
<li><a class="mw-redirect" href="/wiki/Smart_Package_Manager" title="Smart Package Manager">Smart Package Manager</a>, used in Unity Linux, available for many distributions including <a href="/wiki/Fedora_(operating_system)" title="Fedora (operating system)">Fedora</a>.</li>
<li><code>rpmquery</code>, a command-line utility available in (for example) Red Hat Enterprise Linux</li>
</ul>
<h3><span class="mw-headline" id="Local_RPM_installation_database">Local RPM installation database</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=RPM_Package_Manager&amp;action=edit&amp;section=6" title="Edit section: Local RPM installation database">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Working behind the scenes of the package manager is the RPM database, stored in <code>/var/lib/rpm</code>. It uses <a href="/wiki/Berkeley_DB" title="Berkeley DB">Berkeley DB</a> as its back-end. It consists of a single database (<code>Packages</code>) containing all of the meta information of the installed rpms. Multiple databases are created for indexing purposes, replicating data to speed up queries. The database is used to keep track of all files that are changed and created when a user (using RPM) installs a package, thus enabling the user (via RPM) to reverse the changes and remove the package later. If the database gets corrupted (which is possible if the RPM client is <a class="mw-redirect" href="/wiki/Kill_(Unix)" title="Kill (Unix)">killed</a>), the index databases can be recreated with the <code>rpm --rebuilddb</code> command.<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup></p>
<h2><span class="mw-headline" id="Description">Description</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=RPM_Package_Manager&amp;action=edit&amp;section=7" title="Edit section: Description">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Whilst the RPM format is the same across different <a href="/wiki/Linux_distribution" title="Linux distribution">Linux distributions</a>, the detailed conventions and guidelines may vary across them.</p>
<h3><span class="mw-headline" id="Package_filename_and_label">Package filename and label</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=RPM_Package_Manager&amp;action=edit&amp;section=8" title="Edit section: Package filename and label">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>An RPM is delivered in a single file, normally in the format:</p>
<dl>
<dd><tt>&lt;name&gt;-&lt;version&gt;-&lt;release&gt;.&lt;architecture&gt;.rpm</tt></dd>
</dl>
<p>such as:</p>
<dl>
<dd><tt>libgnomeuimm-2.0-2.0.0-3.i386.rpm</tt></dd>
</dl>
<p>where <tt>&lt;name&gt;</tt> is <tt>libgnomeuimm</tt>, <tt>&lt;version&gt;</tt> is <tt>2.0</tt>, <tt>&lt;release&gt;</tt> is <tt>2.0.0-3</tt>, and <tt>&lt;architecture&gt;</tt> is <tt>i386</tt>.</p>
<p>Source code may also be distributed in RPM packages in which case the <tt>&lt;architecture&gt;</tt> part is specified as <tt>src</tt> as in, <tt>libgnomeuimm-2.0-2.0.0-3.<b>src</b>.rpm</tt></p>
<p>RPMs with the <tt>noarch.rpm</tt> extension refer to packages which do not depend on a certain computer's architecture. These include graphics and text for another program to use, and programs written in interpreted programming languages such as <a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a> programs and <a href="/wiki/Shell_script" title="Shell script">shell scripts</a>.</p>
<p>The RPM contents also include a <i>package label</i>, which contains the following pieces of information:</p>
<ul>
<li>software name</li>
<li>software version (the version taken from original <a href="/wiki/Upstream_(software_development)" title="Upstream (software development)">upstream</a> source of the software)</li>
<li>package release (the number of times the package has been rebuilt using the same version of the software). This field is also often used for indicating the specific distribution the package is intended for by appending strings like "mdv" (formerly, "mdk") (<a href="/wiki/Mandriva_Linux" title="Mandriva Linux">Mandriva Linux</a>), "mga" (<a href="/wiki/Mageia" title="Mageia">Mageia</a>), "fc4" (<a href="/wiki/Fedora_(operating_system)" title="Fedora (operating system)">Fedora Core</a> 4), "rhl9" (Red Hat Linux 9), "suse100" (<a class="mw-redirect" href="/wiki/SUSE_Linux" title="SUSE Linux">SUSE Linux</a> 10.0) etc.</li>
<li>architecture for which the package was built (i386, i686, x86_64, ppc, etc.)</li>
</ul>
<p>The package label fields do not need to match the filename.</p>
<h3><span class="mw-headline" id="Library_packaging">Library packaging</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=RPM_Package_Manager&amp;action=edit&amp;section=9" title="Edit section: Library packaging">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Libraries are distributed in two separate packages for each version. One contains the precompiled code for use at run-time, while the second one contains the related development files such as headers, etc. Those packages have "-devel" appended to their name field. The system administrator should ensure that the versions of the binary and development packages match.</p>
<h3><span class="mw-headline" id="Format">Format</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=RPM_Package_Manager&amp;action=edit&amp;section=10" title="Edit section: Format">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The format is binary and consists of four sections:<sup class="reference" id="cite_ref-max_5-1"><a href="#cite_note-max-5">[5]</a></sup></p>
<ul>
<li>The lead, which identifies the file as an RPM file and contains some obsolete headers.</li>
<li>The signature, which can be used to ensure integrity and/or authenticity.</li>
<li>The header, which contains <a href="/wiki/Metadata" title="Metadata">metadata</a> including package name, version, architecture, file list, etc.</li>
<li>A file archive (the <a href="/wiki/Payload_(computing)" title="Payload (computing)">payload</a>), which usually is in <a href="/wiki/Cpio" title="Cpio">cpio</a> format, compressed with <a href="/wiki/Gzip" title="Gzip">gzip</a>. The <tt>rpm2cpio</tt> tool enables retrieval of the cpio file without needing to install the RPM package.<sup class="reference" id="cite_ref-13"><a href="#cite_note-13">[13]</a></sup>
<ul>
<li>More recent versions of RPM can also use <a href="/wiki/Bzip2" title="Bzip2">bzip2</a>, <a href="/wiki/Lzip" title="Lzip">lzip</a>,<sup class="reference" id="cite_ref-14"><a href="#cite_note-14">[14]</a></sup> <a class="mw-redirect" href="/wiki/Lzma" title="Lzma">lzma</a>, or <a href="/wiki/Xz" title="Xz">xz</a> compression.</li>
<li>RPM 5.0 format supports using <a href="/wiki/Xar_(archiver)" title="Xar (archiver)">xar</a> for archiving.</li>
</ul>
</li>
</ul>
<h3><span class="mw-headline" id="SPEC_file">SPEC file</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=RPM_Package_Manager&amp;action=edit&amp;section=11" title="Edit section: SPEC file">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The "Recipe" for creating an RPM package is a spec file. Spec files end in the ".spec" suffix and contain the package name, version, RPM revision number, steps to build, install, and clean a package, and a changelog. Multiple packages can be built from a single RPM spec file, if desired. RPM packages are created from RPM spec files using the rpmbuild tool.</p>
<p>Spec files are usually distributed within SRPM files, which contain the spec file packaged along with the source code.</p>
<h3><span class="mw-headline" id="SRPM">SRPM</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=RPM_Package_Manager&amp;action=edit&amp;section=12" title="Edit section: SRPM">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>A typical RPM is pre-compiled software ready for direct installation. The corresponding source code can also be distributed. This is done in an SRPM, which also includes the "SPEC" file describing the software and how it is built. The SRPM also allows the user to compile, and perhaps modify, the code itself.</p>
<p>A software package may contain only scripts that are architecture-independent. In such a case only an SRPM may be available; this is still an installable RPM.</p>
<h2><span class="mw-headline" id="Forks">Forks</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=RPM_Package_Manager&amp;action=edit&amp;section=13" title="Edit section: Forks">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>As of June 2010<sup class="plainlinks noprint asof-tag update" style="display:none;"><a class="external text" href="//en.wikipedia.org/w/index.php?title=RPM_Package_Manager&amp;action=edit">[update]</a></sup>, there are two versions of RPM in development: one led by the Fedora Project and Red Hat, and the other by a separate group led by a previous <a href="/wiki/Software_maintainer" title="Software maintainer">maintainer</a> of RPM, a former employee of Red Hat.</p>
<h3><span class="mw-headline" id="RPM.org">RPM.org</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=RPM_Package_Manager&amp;action=edit&amp;section=14" title="Edit section: RPM.org">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The <i>rpm.org</i> community's first major code revision was in July 2007; version 4.8 was released in January 2010, version 4.9 in March 2011, 4.10 in May 2012, 4.11 in January 2013, 4.12 in September 2014 and 4.13 in July 2015.</p>
<p>This version is used by distributions such as <a href="/wiki/Fedora_(operating_system)" title="Fedora (operating system)">Fedora</a>, <a href="/wiki/Red_Hat_Enterprise_Linux" title="Red Hat Enterprise Linux">Red Hat Enterprise Linux</a>, <a href="/wiki/OpenSUSE" title="OpenSUSE">openSUSE</a> and <a class="mw-redirect" href="/wiki/SUSE_Linux_Enterprise" title="SUSE Linux Enterprise">SUSE Linux Enterprise</a>, <a class="mw-redirect" href="/wiki/Unity_Linux" title="Unity Linux">Unity Linux</a>, <a href="/wiki/Mageia" title="Mageia">Mageia</a>,<sup class="reference" id="cite_ref-15"><a href="#cite_note-15">[15]</a></sup> and formerly <a href="/wiki/Mandriva_Linux" title="Mandriva Linux">Mandriva</a> (until 2010).</p>
<h3><span class="mw-headline" id="RPM_v5">RPM v5</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=RPM_Package_Manager&amp;action=edit&amp;section=15" title="Edit section: RPM v5">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Jeff Johnson, the RPM maintainer since 1999, continued development efforts together with participants from several other distributions. RPM version 5 was released in May 2007.</p>
<p>This version is used by distributions such as <a class="mw-redirect" href="/wiki/Wind_River_Linux" title="Wind River Linux">Wind River Linux</a>, Rosa Linux, and <a href="/wiki/OpenMandriva_Lx" title="OpenMandriva Lx">OpenMandriva Lx</a> (former <a href="/wiki/Mandriva_Linux" title="Mandriva Linux">Mandriva Linux</a> which switched to rpm5 in 2011<sup class="reference" id="cite_ref-16"><a href="#cite_note-16">[16]</a></sup>) and also by the <a href="/wiki/OpenPKG" title="OpenPKG">OpenPKG</a> project which provides packages for other common UNIX-platforms. OpenMandriva Lx considered switching back to rpm.org<sup class="reference" id="cite_ref-17"><a href="#cite_note-17">[17]</a></sup> before folding.</p>
