<p><b>ReiserFS</b> is a general-purpose, <a href="/wiki/Journaling_file_system" title="Journaling file system">journaled computer file system</a> formerly designed and implemented by a team at <a href="/wiki/Namesys" title="Namesys">Namesys</a> led by <a href="/wiki/Hans_Reiser" title="Hans Reiser">Hans Reiser</a>. ReiserFS is currently supported on <a href="/wiki/Linux" title="Linux">Linux</a> (without quota support). Introduced in version 2.4.1 of the <a href="/wiki/Linux_kernel" title="Linux kernel">Linux kernel</a>, it was the first journaling file system to be included in the standard kernel. ReiserFS is the default file system on the <a class="new" href="/w/index.php?title=Elive&amp;action=edit&amp;redlink=1" title="Elive (page does not exist)">Elive</a>, <a href="/wiki/Xandros" title="Xandros">Xandros</a>, <a href="/wiki/Linspire" title="Linspire">Linspire</a>, and <a href="/wiki/YOPER" title="YOPER">YOPER</a> <a href="/wiki/Linux_distribution" title="Linux distribution">Linux distributions</a>. ReiserFS was the default file system in <a href="/wiki/Novell" title="Novell">Novell</a>'s SUSE Linux Enterprise until Novell decided to move to <a href="/wiki/Ext3" title="Ext3">ext3</a> on October 12, 2006 for future releases.<sup class="reference" id="cite_ref-shankland1_3-0"><a href="#cite_note-shankland1-3">[3]</a></sup></p>
<p>Namesys considered ReiserFS (now occasionally referred to as Reiser3) stable and feature-complete and, with the exception of security updates and critical bug fixes, ceased development on it to concentrate on its successor, <a href="/wiki/Reiser4" title="Reiser4">Reiser4</a>. Namesys went out of business in 2008 after <a href="/wiki/Hans_Reiser" title="Hans Reiser">Hans Reiser</a> was charged with the murder of his wife (and later convicted and sent to prison). Nevertheless, volunteers continue to work on the open source project.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></p>

<h2><span class="mw-headline" id="Features">Features</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ReiserFS&amp;action=edit&amp;section=1" title="Edit section: Features">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>At the time of its introduction, ReiserFS offered features that had not been available in existing Linux file systems:</p>
<ul>
<li><a href="/wiki/Metadata" title="Metadata">Metadata</a>-only <a href="/wiki/Journaling_file_system" title="Journaling file system">journaling</a> (also block journaling, since Linux 2.6.8), its most-publicized advantage over what was the stock Linux file system at the time, <a href="/wiki/Ext2" title="Ext2">ext2</a>.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (January 2017)">citation needed</span></a></i>]</sup></li>
<li>Online resizing (growth only), with or without an underlying volume manager such as <a href="/wiki/Logical_Volume_Manager_(Linux)" title="Logical Volume Manager (Linux)">LVM</a>.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (January 2017)">citation needed</span></a></i>]</sup> Since then, Namesys has also provided tools to resize (both grow and shrink) ReiserFS file systems offline.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (January 2017)">citation needed</span></a></i>]</sup></li>
<li><a href="/wiki/Block_suballocation#Tail_packing" title="Block suballocation">Tail packing</a>, a scheme to reduce <a href="/wiki/Fragmentation_(computing)#Internal_fragmentation" title="Fragmentation (computing)">internal fragmentation</a>. Tail packing, however, can have a significant performance impact. Reiser4 may have improved this by packing tails where it does not hurt performance.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup></li>
</ul>
<h2><span class="mw-headline" id="Performance">Performance</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ReiserFS&amp;action=edit&amp;section=2" title="Edit section: Performance">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Compared with <a href="/wiki/Ext2" title="Ext2">ext2</a> and <a href="/wiki/Ext3" title="Ext3">ext3</a> in version 2.4 of the Linux kernel, when dealing with files under 4 <a class="mw-redirect" href="/wiki/KiB" title="KiB">KiB</a> and with tail packing enabled, ReiserFS may<sup class="noprint Inline-Template" style="margin-left:0.1em; white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Manual_of_Style/Words_to_watch#Unsupported_attributions" title="Wikipedia:Manual of Style/Words to watch"><span title="The material near this tag may use weasel words or too-vague attribution. (October 2016)">according to whom?</span></a></i>]</sup> be faster. This was said<sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Manual_of_Style/Words_to_watch#Unsupported_attributions" title="Wikipedia:Manual of Style/Words to watch"><span title="The material near this tag may use weasel words or too-vague attribution. (October 2016)">by whom?</span></a></i>]</sup> to be of great benefit in <a href="/wiki/Usenet" title="Usenet">Usenet</a> news spools, HTTP caches, mail delivery systems, and other applications where performance with small files is critical. However, in practice<sup class="noprint Inline-Template" style="margin-left:0.1em; white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Manual_of_Style/Words_to_watch#Unsupported_attributions" title="Wikipedia:Manual of Style/Words to watch"><span title="The material near this tag may use weasel words or too-vague attribution. (October 2016)">according to whom?</span></a></i>]</sup> news spools use a feature called cycbuf, which holds articles in one large file; fast HTTP caches and several <a class="mw-redirect" href="/wiki/Revision_control" title="Revision control">revision control</a> systems use a similar approach, nullifying these performance advantages. For email servers, ReiserFS was problematic due to semantic problems explained below. Also, ReiserFS had a problem with very fast filesystem aging when compared to other filesystems — in several usage scenarios filesystem performance decreased dramatically with time.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (August 2014)">citation needed</span></a></i>]</sup></p>
<p>Before Linux 2.6.33,<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup> ReiserFS heavily used the <a href="/wiki/Giant_lock" title="Giant lock">big kernel lock</a> (BKL) — a global kernel-wide lock — which does not scale very well<sup class="reference" id="cite_ref-mahoney-proposal_7-0"><a href="#cite_note-mahoney-proposal-7">[7]</a></sup><sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> for systems with multiple cores, as the critical code parts are only ever executed by one core at a time.</p>
<h2><span class="mw-headline" id="Criticism">Criticism</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ReiserFS&amp;action=edit&amp;section=3" title="Edit section: Criticism">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Some directory operations (including <tt>unlink</tt>(2)) are not <a href="/wiki/Synchronization" title="Synchronization">synchronous</a> on ReiserFS, which can result in data corruption with applications relying heavily on file-based locks (such as <a href="/wiki/Message_transfer_agent" title="Message transfer agent">mail transfer agents</a> <a href="/wiki/Qmail" title="Qmail">qmail</a><sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup> and <a href="/wiki/Postfix_(software)" title="Postfix (software)">Postfix</a><sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup>) if the machine halts before it has synchronized the disk.<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup></p>
<p>There are no programs to specifically <a href="/wiki/Defragmentation" title="Defragmentation">defragment</a> a ReiserFS file system, although tools have been written to automatically copy the contents of fragmented files hoping that more contiguous blocks of free space can be found. However, a "repacker" tool was planned for the next Reiser4 file system to deal with file fragmentation.<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup></p>
<h3><span class="mw-headline" id="fsck">fsck</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ReiserFS&amp;action=edit&amp;section=4" title="Edit section: fsck">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The tree rebuild process of ReiserFS's <a href="/wiki/Fsck" title="Fsck">fsck</a> has attracted much criticism:<sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Manual_of_Style/Words_to_watch#Unsupported_attributions" title="Wikipedia:Manual of Style/Words to watch"><span title="The material near this tag may use weasel words or too-vague attribution. (January 2017)">by whom?</span></a></i>]</sup> If the file system becomes so badly corrupted that its internal tree is unusable, performing a tree rebuild operation may further corrupt existing files or introduce new entries with unexpected contents,<sup class="reference" id="cite_ref-13"><a href="#cite_note-13">[13]</a></sup> but this action is not part of normal operation or a normal file system check and has to be explicitly initiated and confirmed by the administrator.</p>
<p>ReiserFS v3 images should not be stored on a ReiserFS v3 <a href="/wiki/Disk_partitioning" title="Disk partitioning">partition</a> (e.g., backups or <a href="/wiki/Disk_image" title="Disk image">disk images</a> for emulators) without transforming them (e.g., by compressing or encrypting) in order to avoid confusing the rebuild. Reformatting an existing ReiserFS v3 partition can also leave behind data that could confuse the rebuild operation and make files from the old system reappear. This also allows malicious users to intentionally store files that will confuse the rebuilder. As the metadata is always in a consistent state after a file system check, <i>corruption</i> here means that contents of files are merged in unexpected ways with the contained file system's metadata. The ReiserFS successor, Reiser4, fixes this problem.</p>
<h3><span class="mw-headline" id="Earlier_issues">Earlier issues</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ReiserFS&amp;action=edit&amp;section=5" title="Edit section: Earlier issues">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>ReiserFS in versions of the Linux kernel before 2.4.16 were considered unstable by Namesys and not recommended for production use, especially in conjunction with <a href="/wiki/Network_File_System" title="Network File System">NFS</a>.<sup class="reference" id="cite_ref-14"><a href="#cite_note-14">[14]</a></sup></p>
<p>Early implementations of ReiserFS (prior to that in Linux 2.6.2) were also susceptible to out-of-order write hazards. But the current journaling implementation in ReiserFS is now on par with that of <a href="/wiki/Ext3" title="Ext3">ext3</a>'s "ordered" journaling level.</p>
<h3><span class="mw-headline" id="Move_away_from_ReiserFS_to_ext3">Move away from ReiserFS to ext3</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ReiserFS&amp;action=edit&amp;section=6" title="Edit section: Move away from ReiserFS to ext3">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Jeff Mahoney of SUSE wrote a post on 14 September 2006 proposing to move from ReiserFS to <a href="/wiki/Ext3" title="Ext3">ext3</a> for the default installation file system.<sup class="reference" id="cite_ref-mahoney-proposal_7-1"><a href="#cite_note-mahoney-proposal-7">[7]</a></sup> Some reasons he mentioned were scalability, "performance problems with <a href="/wiki/Extended_file_attributes" title="Extended file attributes">extended attributes</a> and <a href="/wiki/Access_control_list" title="Access control list">ACLs</a>", "a small and shrinking development community", and that "<a href="/wiki/Reiser4" title="Reiser4">Reiser4</a> is not an incremental update and requires a reformat, which is unreasonable for most people."<sup class="reference" id="cite_ref-mahoney-proposal_7-2"><a href="#cite_note-mahoney-proposal-7">[7]</a></sup> On October 4 he wrote a response comment on a blog in order to clear up some issues.<sup class="reference" id="cite_ref-mahoney-comment29_15-0"><a href="#cite_note-mahoney-comment29-15">[15]</a></sup> He wrote that his proposal for the switch was unrelated to Reiser's "legal troubles" (i.e., Hans Reiser <a href="/wiki/Hans_Reiser#Trial_and_conviction" title="Hans Reiser">murdering his wife</a>)<sup class="reference" id="cite_ref-16"><a href="#cite_note-16">[16]</a></sup> Mahoney wrote he "was concerned that people would make a connection where none existed" and that "the timing is entirely coincidental and the motivation is unrelated."<sup class="reference" id="cite_ref-mahoney-comment29_15-1"><a href="#cite_note-mahoney-comment29-15">[15]</a></sup></p>
<p>On Oct 12, 2006, <a href="/wiki/Novell" title="Novell">Novell</a> officially announced that <a href="/wiki/SUSE_Linux_distributions#SUSE_Linux_Enterprise" title="SUSE Linux distributions">SUSE Linux Enterprise</a> would switch from ReiserFS to ext3.<sup class="reference" id="cite_ref-shankland1_3-1"><a href="#cite_note-shankland1-3">[3]</a></sup></p>
<h2><span class="mw-headline" id="Design">Design</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ReiserFS&amp;action=edit&amp;section=7" title="Edit section: Design">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>ReiserFS stores file metadata ("stat items"), <a href="/wiki/Directory_(computing)" title="Directory (computing)">directory</a> entries ("directory items"), <a href="/wiki/Inode" title="Inode">inode</a> block lists ("indirect items"), and tails of <a href="/wiki/Computer_file" title="Computer file">files</a> ("direct items") in a single, combined <a href="/wiki/B%2B_tree" title="B+ tree">B+ tree</a> keyed by a universal object ID. Disk blocks allocated to nodes of the tree are "formatted internal blocks". Blocks for leaf nodes (in which items are packed end-to-end) are "formatted leaf blocks". All other blocks are "unformatted blocks" containing file contents. Directory items with too many entries or indirect items which are too long to fit into a node spill over into the right leaf neighbour. Block allocation is tracked by <a href="/wiki/Free_space_bitmap" title="Free space bitmap">free space bitmaps</a> in fixed locations.</p>
<p>By contrast, <a href="/wiki/Ext2" title="Ext2">ext2</a> and other Berkeley <a href="/wiki/Unix_File_System" title="Unix File System">FFS</a>-like file systems of that time simply used a fixed formula for computing inode locations, hence limiting the number of files they may contain.<sup class="reference" id="cite_ref-17"><a href="#cite_note-17">[17]</a></sup> Most such file systems also store directories as simple lists of entries, which makes directory lookups and updates <a href="/wiki/Time_complexity#Linear_time" title="Time complexity">linear time</a> operations and degrades performance on very large directories. The single <a href="/wiki/B%2B_tree" title="B+ tree">B+ tree</a> design in ReiserFS avoids both of these problems due to better scalability properties.</p>
