<p><b>SHA-3</b> (<b>Secure Hash Algorithm 3</b>), a subset of the cryptographic primitive family <b>Keccak</b> (<span class="nowrap"><span class="IPA nopopups"><a href="/wiki/Help:IPA_for_English" title="Help:IPA for English">/<span style="border-bottom:1px dotted"><span title="/ˈ/ primary stress follows">ˈ</span><span title="'k' in 'kind'">k</span><span title="/ɛ/ short 'e' in 'bed'">ɛ</span><span title="'t' in 'tie'">t</span><span title="/ʃ/ 'sh' in 'shy'">ʃ</span><span title="/æ/ short 'a' in 'bad'">æ</span><span title="'k' in 'kind'">k</span></span>/</a></span></span>, or <span class="nowrap"><span class="IPA nopopups"><a href="/wiki/Help:IPA_for_English" title="Help:IPA for English">/<span style="border-bottom:1px dotted"><span title="'k' in 'kind'">k</span><span title="/ɛ/ short 'e' in 'bed'">ɛ</span><span title="'t' in 'tie'">t</span><span title="/ʃ/ 'sh' in 'shy'">ʃ</span><span title="/ɑː/ 'a' in 'father'">ɑː</span><span title="'k' in 'kind'">k</span></span>/</a></span></span>),<sup class="reference" id="cite_ref-nist_1-0"><a href="#cite_note-nist-1">[1]</a></sup><sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup><sup class="reference" id="cite_ref-drdobbs_3-0"><a href="#cite_note-drdobbs-3">[3]</a></sup> is a <a href="/wiki/Cryptographic_hash_function" title="Cryptographic hash function">cryptographic hash function</a> designed by Guido Bertoni, <a href="/wiki/Joan_Daemen" title="Joan Daemen">Joan Daemen</a>, Michaël Peeters, and <a href="/wiki/Gilles_Van_Assche" title="Gilles Van Assche">Gilles Van Assche</a>, building upon <a href="/wiki/RadioGat%C3%BAn" title="RadioGatún">RadioGatún</a>. SHA-3 is a member of the <a href="/wiki/Secure_Hash_Algorithm" title="Secure Hash Algorithm">Secure Hash Algorithm</a> family. The SHA-3 standard was released by <a class="mw-redirect" href="/wiki/NIST" title="NIST">NIST</a> on August 5, 2015.<sup class="reference" id="cite_ref-nist.gov_4-0"><a href="#cite_note-nist.gov-4">[4]</a></sup><sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> The <a href="/wiki/Reference_implementation" title="Reference implementation">reference implementation</a> <a href="/wiki/Source_code" title="Source code">source code</a> was dedicated to <a href="/wiki/Public_domain" title="Public domain">public domain</a> via <a class="mw-redirect" href="/wiki/CC0" title="CC0">CC0</a> <a href="/wiki/Waiver" title="Waiver">waiver</a>.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup></p>

<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=SHA-3&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The Keccak algorithm is the work of Guido Bertoni, <a href="/wiki/Joan_Daemen" title="Joan Daemen">Joan Daemen</a> (who also co-designed the <a class="mw-redirect" href="/wiki/Rijndael" title="Rijndael">Rijndael</a> cipher with <a href="/wiki/Vincent_Rijmen" title="Vincent Rijmen">Vincent Rijmen</a>), Michael Peeters, and <a href="/wiki/Gilles_Van_Assche" title="Gilles Van Assche">Gilles Van Assche</a>. It is based on earlier hash function designs <a class="mw-redirect" href="/wiki/PANAMA" title="PANAMA">PANAMA</a> and <a href="/wiki/RadioGat%C3%BAn" title="RadioGatún">RadioGatún</a>. PANAMA was designed by Daemen and Craig Clapp in 1998. RadioGatún, a successor of PANAMA, was designed by Daemen, Peeters, and Van Assche, and was presented at the NIST Hash Workshop in 2006.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup></p>
<p>In 2006 <a class="mw-redirect" href="/wiki/NIST" title="NIST">NIST</a> started to organize the <a href="/wiki/NIST_hash_function_competition" title="NIST hash function competition">NIST hash function competition</a> to create a new hash standard, SHA-3. SHA-3 is not meant to replace <a href="/wiki/SHA-2" title="SHA-2">SHA-2</a>, as no significant attack on SHA-2 has been demonstrated. Because of the successful attacks on <a href="/wiki/MD5" title="MD5">MD5</a> and <a class="mw-redirect" href="/wiki/SHA-0" title="SHA-0">SHA-0</a> and theoretical attacks on <a href="/wiki/SHA-1" title="SHA-1">SHA-1</a>,<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> NIST perceived a need for an alternative, dissimilar cryptographic hash, which became SHA-3.</p>
<p>After a setup period, admissions were to be submitted by the end of 2008. Keccak was accepted as one of the 51 candidates. In July 2009, 14 algorithms were selected for the second round. Keccak advanced to the last round in December 2010.<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup></p>
<p>During the competition, entrants were permitted to "tweak" their algorithms to address issues that were discovered. Changes that have been made to Keccak are:<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup><sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup></p>
<ul>
<li>The number of rounds was increased from <span class="nowrap">12 + ℓ</span> to <span class="nowrap">12 + 2ℓ</span> to be more conservative about security.</li>
<li>The message padding was changed from a more complex scheme to the simple 10<sup>*</sup>1 pattern described below.</li>
<li>The rate <i>r</i> was increased to the security limit, rather than rounding down to the nearest power of 2.</li>
</ul>
<p>On October 2, 2012, Keccak was selected as the winner of the competition.<sup class="reference" id="cite_ref-nist_1-1"><a href="#cite_note-nist-1">[1]</a></sup></p>
<p>In 2014, the NIST published a draft <a href="/wiki/Federal_Information_Processing_Standards" title="Federal Information Processing Standards">FIPS</a> 202 "SHA-3 Standard: Permutation-Based Hash and Extendable-Output Functions".<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup> FIPS 202 was approved on August 5, 2015.<sup class="reference" id="cite_ref-13"><a href="#cite_note-13">[13]</a></sup></p>
<p>On August 5, 2015 NIST announced that SHA-3 had become a hashing standard.<sup class="reference" id="cite_ref-14"><a href="#cite_note-14">[14]</a></sup></p>
<h2><span class="mw-headline" id="Design">Design</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=SHA-3&amp;action=edit&amp;section=2" title="Edit section: Design">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="thumb tright">
<div class="thumbinner" style="width:302px;"><a class="image" href="/wiki/File:SpongeConstruction.svg"><img alt="Illustration of the sponge construction" class="thumbimage" data-file-height="232" data-file-width="512" height="136" src="//upload.wikimedia.org/wikipedia/commons/thumb/7/70/SpongeConstruction.svg/300px-SpongeConstruction.svg.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/7/70/SpongeConstruction.svg/450px-SpongeConstruction.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/7/70/SpongeConstruction.svg/600px-SpongeConstruction.svg.png 2x" width="300"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:SpongeConstruction.svg" title="Enlarge"></a></div>
The sponge construction for hash functions. <i>p<sub>i</sub></i> are input, <i>z<sub>i</sub></i> are hashed output. The unused "capacity" <i>c</i> should be twice the desired resistance to <a href="/wiki/Collision_attack" title="Collision attack">collision</a> or <a href="/wiki/Preimage_attack" title="Preimage attack">preimage attacks</a>.</div>
</div>
</div>
<p>SHA-3 uses the <a href="/wiki/Sponge_function" title="Sponge function">sponge construction</a>,<sup class="reference" id="cite_ref-15"><a href="#cite_note-15">[15]</a></sup><sup class="reference" id="cite_ref-16"><a href="#cite_note-16">[16]</a></sup> in which data is "absorbed" into the sponge, then the result is "squeezed" out. In the absorbing phase, message blocks are <a href="/wiki/Exclusive_or" title="Exclusive or">XORed</a> into a subset of the state, which is then transformed as a whole. In the "squeeze" phase, output blocks are read from the same subset of the state, alternated with state transformations. The size of the part of the state that is written and read is called the "rate" (often denoted <i>r</i>), and the size of the part that is untouched by input/output is called the "capacity" (often denoted <i>c</i>). The capacity determines the security of the scheme. The maximum <a href="/wiki/Security_level" title="Security level">security level</a> is half the capacity.</p>
<p>In SHA-3, the state consists of a <span class="nowrap">5 × 5</span> array of 64-bit words, 1600 bits total. Keccak is also defined for smaller power-of-2 word sizes <i>w</i> down to 1 bit (25 bits total state). Small state sizes can be used to test cryptanalytic attacks, and intermediate state sizes (from <span class="nowrap"><i>w</i> = 8</span>, 200 bits, to <span class="nowrap"><i>w</i> = 32</span>, 800 bits) can be used in practical, lightweight applications.<sup class="reference" id="cite_ref-ketje_17-0"><a href="#cite_note-ketje-17">[17]</a></sup><sup class="reference" id="cite_ref-keyak_18-0"><a href="#cite_note-keyak-18">[18]</a></sup></p>
<p>The block transformation is a permutation that uses <b><a href="/wiki/Bitwise_operation#XOR" title="Bitwise operation">xor</a></b>, <b><a class="mw-redirect" href="/wiki/Binary_and" title="Binary and">and</a></b> and <b><a class="mw-redirect" href="/wiki/Binary_NOT" title="Binary NOT">not</a></b> operations, and designed for easy implementation in both software and hardware. The authors claim 12.5 <a href="/wiki/Cycles_per_byte" title="Cycles per byte">cycles per byte</a><sup class="reference" id="cite_ref-19"><a href="#cite_note-19">[19]</a></sup> on an <a href="/wiki/Intel_Core_2" title="Intel Core 2">Intel Core 2</a> CPU. However, in <a href="/wiki/Application-specific_integrated_circuit" title="Application-specific integrated circuit">hardware implementations</a>, it is notably faster than all other finalists.<sup class="reference" id="cite_ref-20"><a href="#cite_note-20">[20]</a></sup></p>
<p>Keccak's authors have proposed additional, not-yet-standardized uses for the function, including an <a href="/wiki/Authenticated_encryption" title="Authenticated encryption">authenticated encryption</a> system and a "tree" hashing scheme for faster hashing on certain architectures.<sup class="reference" id="cite_ref-21"><a href="#cite_note-21">[21]</a></sup></p>
<h2><span class="mw-headline" id="The_block_permutation">The block permutation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=SHA-3&amp;action=edit&amp;section=3" title="Edit section: The block permutation">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>This is defined for any power-of-two <a href="/wiki/Word_(computer_architecture)" title="Word (computer architecture)">word</a> size, <span class="texhtml"><i>w</i> = 2<sup><i>ℓ</i></sup></span> bits. The main SHA-3 submission uses 64-bit words, <span class="texhtml"><i>ℓ</i> = 6</span>.</p>
<p>The state can be considered to be a <span class="texhtml">5 × 5 × <i>w</i></span> array of bits. Let <span class="texhtml"><i>a</i>[<i>i</i>][<i> j</i>][<i>k</i>]</span> be bit <span class="texhtml">(5<i>i</i> + <i>j</i>) × <i>w</i> + <i>k</i></span> of the input, using a <a class="mw-redirect" href="/wiki/Little-endian" title="Little-endian">little-endian</a> bit numbering convention and <a class="mw-redirect" href="/wiki/Row-major" title="Row-major">row-major</a> indexing. I.e. <span class="texhtml mvar" style="font-style:italic;">i</span> selects the row, <span class="texhtml mvar" style="font-style:italic;">j</span> the column, and <span class="texhtml mvar" style="font-style:italic;">k</span> the bit.</p>
<p>Index arithmetic is performed modulo 5 for the first two dimensions and modulo <span class="texhtml mvar" style="font-style:italic;">w</span> for the third.</p>

To be precise, <span class="texhtml"><i>a</i>[<i>i</i>][<i> j</i>][<i>k</i>] ← <i>a</i>[<i>i</i>][<i> j</i>][<i>k</i>] ⊕ parity(a[0...4][<i> j</i>−1][<i>k</i>]) ⊕ parity(a[0...4][<i> j</i>+1][<i>k</i>−1])</span></dd>

<h2><span class="mw-headline" id="Hashing_variable-length_messages">Hashing variable-length messages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=SHA-3&amp;action=edit&amp;section=4" title="Edit section: Hashing variable-length messages">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>To ensure the message can be evenly divided into <span class="texhtml mvar" style="font-style:italic;">r</span>-bit blocks, padding is required. Keccak uses the pattern 10<sup><a href="/wiki/Kleene_star" title="Kleene star">*</a></sup>1: a 1 bit, zero or more 0 bits (maximum <span class="texhtml"><i>r</i> − 1</span>), and a final 1 bit. The final 1 bit is required for the security proof to work for sponges of different rates, that is, different hash variants (multi-rate padding). Without it, different hash variants of the same short message would be the same up to truncation.</p>
<p>To compute a hash, initialize the state to 0, pad the input, and break it into <span class="texhtml mvar" style="font-style:italic;">r</span>-bit pieces. Absorb the input into the state; that is, for each piece, XOR it into the state and then apply the block permutation.</p>
<p>After the final block permutation, the desired number of bits <span class="texhtml mvar" style="font-style:italic;">n</span> is squeezed. For the SHA3 instances, <span class="texhtml mvar" style="font-style:italic;">r</span> is always greater than <span class="texhtml mvar" style="font-style:italic;">n</span>, thus there is never a need for additional block permutations in the squeezing phase. The leading <span class="texhtml mvar" style="font-style:italic;">n</span> bits of the state are the desired hash. However, arbitrary output length may be useful in applications such as <a href="/wiki/Optimal_asymmetric_encryption_padding" title="Optimal asymmetric encryption padding">optimal asymmetric encryption padding</a>.</p>
<h2><span class="mw-headline" id="Instances">Instances</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=SHA-3&amp;action=edit&amp;section=5" title="Edit section: Instances">edit</a><span class="mw-editsection-bracket">]</span></span></h2>

<p>The SHA3 instances are the drop-in replacements for SHA2, with identical security claims. SHAKE instances are so called XOF's, Extendable Output Functions. For example, SHAKE128(M, 256) can be used as a hash function with a 256 bit length and 128 bit overall security.</p>
<p>Note that all instances append some bits to the message. Since 10<sup>*</sup>1 padding always adds at least two bits, in byte aligned libraries there are always six unused zero bits. Therefore, these appended extra bits never make the padded message longer.</p>
<h2><span class="mw-headline" id="NIST_announcement_controversy">NIST announcement controversy</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=SHA-3&amp;action=edit&amp;section=6" title="Edit section: NIST announcement controversy">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In February 2013 at the RSA Conference, and then in August 2013 at CHES, NIST announced they would select different values for the capacity, i.e. the security parameter, for the SHA-3 standard, compared to the submission.<sup class="reference" id="cite_ref-rsa2013_22-0"><a href="#cite_note-rsa2013-22">[22]</a></sup><sup class="reference" id="cite_ref-ches2013_23-0"><a href="#cite_note-ches2013-23">[23]</a></sup> The changes caused some turmoil.</p>
<p>In September 2013, <a href="/wiki/Daniel_J._Bernstein" title="Daniel J. Bernstein">Daniel J. Bernstein</a> suggested on the <a class="mw-redirect" href="/wiki/NIST" title="NIST">NIST</a> hash-forum mailing list<sup class="reference" id="cite_ref-24"><a href="#cite_note-24">[24]</a></sup> to strengthen the security to the 576-bit capacity that was originally proposed as the default Keccak.<sup class="reference" id="cite_ref-25"><a href="#cite_note-25">[25]</a></sup> In late September, the Keccak team responded by stating that they proposed 128-bit security by setting <span class="nowrap"><i>c</i> = 256</span> as an option already in their SHA-3 proposal.<sup class="reference" id="cite_ref-26"><a href="#cite_note-26">[26]</a></sup> However, in the light of the negative response from the cryptographic community, they proposed raising the capacity to 512 bits for all instances.<sup class="reference" id="cite_ref-27"><a href="#cite_note-27">[27]</a></sup></p>
<p>In early October 2013, <a href="/wiki/Bruce_Schneier" title="Bruce Schneier">Bruce Schneier</a> criticized NIST's decision on the basis of its possible detrimental effects on the acceptance of the algorithm, saying:</p>
<blockquote class="templatequote">
<p>There is too much mistrust in the air. NIST risks publishing an algorithm that no one will trust and no one (except those forced) will use.<sup class="reference" id="cite_ref-schneier_28-0"><a href="#cite_note-schneier-28">[28]</a></sup></p>
</blockquote>
<p>Paul Crowley, a senior developer at an independent software development company, expressed his support of the decision, saying that Keccak is supposed to be tunable and there is no reason for different security levels within one primitive. He also added:</p>
<blockquote class="templatequote">
<p>Yes, it’s a bit of a shame for the competition that they demanded a certain security level for entrants, then went to publish a standard with a different one. But there’s nothing that can be done to fix that now, except re-opening the competition. Demanding that they stick to their mistake doesn’t improve things for anyone.<sup class="reference" id="cite_ref-29"><a href="#cite_note-29">[29]</a></sup></p>
</blockquote>
<p>There was also some confusion that internal changes were made to Keccak. The Keccak team clarified this, stating that NIST's proposal for SHA-3 is a subset of the Keccak family, for which one can generate test vectors using their reference code submitted to the contest, and that this proposal was the result of a series of discussions between them and the NIST hash team.<sup class="reference" id="cite_ref-30"><a href="#cite_note-30">[30]</a></sup> Also, <a href="/wiki/Bruce_Schneier" title="Bruce Schneier">Bruce Schneier</a> corrected his earlier statement, saying:</p>
<blockquote class="templatequote">
<p>I misspoke when I wrote that NIST made "internal changes" to the algorithm. That was sloppy of me. The Keccak permutation remains unchanged. What NIST proposed was reducing the hash function's capacity in the name of performance. One of Keccak's nice features is that it's highly tunable.<sup class="reference" id="cite_ref-schneier_28-1"><a href="#cite_note-schneier-28">[28]</a></sup></p>
</blockquote>
<p>In response to the controversy, in November 2013 John Kelsey of NIST proposed to go back to the original <span class="nowrap"><i>c</i> = 2<i>n</i></span> proposal for all SHA-2 drop-in replacement instances.<sup class="reference" id="cite_ref-31"><a href="#cite_note-31">[31]</a></sup> These changes were confirmed in the April 2014 draft.<sup class="reference" id="cite_ref-draft201404_32-0"><a href="#cite_note-draft201404-32">[32]</a></sup> This proposal was implemented in the final release standard in August 2015.<sup class="reference" id="cite_ref-nist.gov_4-1"><a href="#cite_note-nist.gov-4">[4]</a></sup></p>
