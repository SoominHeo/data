<h1 class="firstHeading" id="firstHeading" lang="en">SQL injection</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">

<p><b>SQL injection</b> is a <a href="/wiki/Code_injection" title="Code injection">code injection</a> technique, used to <a href="/wiki/Attack_(computing)" title="Attack (computing)">attack</a> data-driven applications, in which nefarious <a href="/wiki/SQL" title="SQL">SQL</a> statements are inserted into an entry field for execution (e.g. to dump the database contents to the attacker).<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> SQL injection must exploit a <a class="mw-redirect" href="/wiki/Security_vulnerability" title="Security vulnerability">security vulnerability</a> in an application's software, for example, when user input is either incorrectly filtered for <a href="/wiki/String_literal" title="String literal">string literal</a> <a href="/wiki/Escape_sequence" title="Escape sequence">escape characters</a> embedded in SQL statements or user input is not <a class="mw-redirect" href="/wiki/Strongly-typed_programming_language" title="Strongly-typed programming language">strongly typed</a> and unexpectedly executed. SQL injection is mostly known as an attack <a href="/wiki/Vector_(malware)" title="Vector (malware)">vector</a> for websites but can be used to attack any type of SQL database.</p>
<p>SQL injection attacks allow attackers to spoof identity, tamper with existing data, cause repudiation issues such as voiding transactions or changing balances, allow the complete disclosure of all data on the system, destroy the data or make it otherwise unavailable, and become administrators of the database server.</p>
<p>In a 2012 study, it was observed that the average web application received 4 attack campaigns per month, and retailers received twice as many attacks as other industries.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup></p>
<p></p>

<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=SQL_injection&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The first public discussions of SQL injection started appearing around 1998;<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup> for example, a 1998 article in <a class="mw-redirect" href="/wiki/Phrack_Magazine" title="Phrack Magazine">Phrack Magazine</a>.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></p>
<h2><span class="mw-headline" id="Form">Form</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=SQL_injection&amp;action=edit&amp;section=2" title="Edit section: Form">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>SQL injection (SQLI) is considered one of the top 10 web application vulnerabilities of 2007 and 2010 by the <a href="/wiki/OWASP" title="OWASP">Open Web Application Security Project</a>.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> In 2013, SQLI was rated the number one attack on the OWASP top ten.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup> There are four main sub-classes of SQL injection:</p>
<ul>
<li>Classic SQLI</li>
<li>Blind or Inference SQL injection</li>
<li><a class="mw-redirect" href="/wiki/Database_management_system" title="Database management system">Database management system</a>-specific SQLI</li>
<li>Compounded SQLI</li>
</ul>
<dl>
<dd>
<ul>
<li>SQL injection + insufficient authentication<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup></li>
<li>SQL injection + <a class="mw-redirect" href="/wiki/DDoS" title="DDoS">DDoS</a> attacks<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup></li>
<li>SQL injection + <a href="/wiki/DNS_hijacking" title="DNS hijacking">DNS hijacking</a><sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup></li>
<li>SQL injection + <a href="/wiki/Cross-site_scripting" title="Cross-site scripting">XSS</a><sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup></li>
</ul>
</dd>
</dl>
<p>The <a href="/wiki/Storm_Worm" title="Storm Worm">Storm Worm</a> is one representation of Compounded SQLI.<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup></p>
<p>This classification represents the state of SQLI, respecting its evolution until 2010—further refinement is underway.<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup></p>
<h2><span class="mw-headline" id="Technical_implementations">Technical implementations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=SQL_injection&amp;action=edit&amp;section=3" title="Edit section: Technical implementations">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Incorrectly_filtered_escape_characters">Incorrectly filtered escape characters</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=SQL_injection&amp;action=edit&amp;section=4" title="Edit section: Incorrectly filtered escape characters">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>This form of SQL injection occurs when user input is not filtered for <a href="/wiki/Escape_character" title="Escape character">escape characters</a> and is then passed into an SQL statement. This results in the potential manipulation of the statements performed on the database by the end-user of the application.</p>
<p>The following line of code illustrates this vulnerability:</p>
<pre>
statement = "<code class="mw-highlight" dir="ltr"><span class="k">SELECT</span> <span class="o">*</span> <span class="k">FROM</span> <span class="n">users</span> <span class="k">WHERE</span> <span class="n">name</span> <span class="o">=</span> <span class="err">'</span></code>" + userName + "<code class="mw-highlight" dir="ltr"><span class="err">'</span><span class="p">;</span></code>"
</pre>
<p>This SQL code is designed to pull up the records of the specified username from its table of users. However, if the "userName" variable is crafted in a specific way by a malicious user, the SQL statement may do more than the code author intended. For example, setting the "userName" variable as:</p>
<pre>
' OR '1'='1
</pre>
<p>or using comments to even block the rest of the query (there are three types of SQL comments<sup class="reference" id="cite_ref-13"><a href="#cite_note-13">[13]</a></sup>). All three lines have a space at the end:</p>
<pre>
' OR '1'='1' --
' OR '1'='1' ({
' OR '1'='1' /* 
</pre>
<p>. renders one of the following SQL statements by the parent language:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">SELECT</span> <span class="o">*</span> <span class="k">FROM</span> <span class="n">users</span> <span class="k">WHERE</span> <span class="n">name</span> <span class="o">=</span> <span class="s1">''</span> <span class="k">OR</span> <span class="s1">'1'</span><span class="o">=</span><span class="s1">'1'</span><span class="p">;</span>
</pre></div>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">SELECT</span> <span class="o">*</span> <span class="k">FROM</span> <span class="n">users</span> <span class="k">WHERE</span> <span class="n">name</span> <span class="o">=</span> <span class="s1">''</span> <span class="k">OR</span> <span class="s1">'1'</span><span class="o">=</span><span class="s1">'1'</span> <span class="c1">-- ';</span>
</pre></div>
<p>If this code were to be used in an authentication procedure then this example could be used to force the selection of every data field (*) from <i>all</i> users rather than from one specific user name as the coder intended, because the evaluation of '1'='1' is always true (<a href="/wiki/Short-circuit_evaluation" title="Short-circuit evaluation">short-circuit evaluation</a>).</p>
<p>The following value of "userName" in the statement below would cause the deletion of the "users" table as well as the selection of all data from the "userinfo" table (in essence revealing the information of every user), using an <a class="mw-redirect" href="/wiki/API" title="API">API</a> that allows multiple statements:</p>
<pre>
a';<code class="mw-highlight" dir="ltr"><span class="k">DROP</span> <span class="k">TABLE</span> <span class="n">users</span><span class="p">;</span> <span class="k">SELECT</span> <span class="o">*</span> <span class="k">FROM</span> <span class="n">userinfo</span> <span class="k">WHERE</span> <span class="s1">'t'</span> <span class="o">=</span> <span class="s1">'t'</span></code>
</pre>
<p>This input renders the final SQL statement as follows and specified:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">SELECT</span> <span class="o">*</span> <span class="k">FROM</span> <span class="n">users</span> <span class="k">WHERE</span> <span class="n">name</span> <span class="o">=</span> <span class="s1">'a'</span><span class="p">;</span><span class="k">DROP</span> <span class="k">TABLE</span> <span class="n">users</span><span class="p">;</span> <span class="k">SELECT</span> <span class="o">*</span> <span class="k">FROM</span> <span class="n">userinfo</span> <span class="k">WHERE</span> <span class="s1">'t'</span> <span class="o">=</span> <span class="s1">'t'</span><span class="p">;</span>
</pre></div>
<p>While most SQL server implementations allow multiple statements to be executed with one call in this way, some SQL APIs such as <a href="/wiki/PHP" title="PHP">PHP</a>'s <code>mysql_query()</code> function do not allow this for security reasons. This prevents attackers from injecting entirely separate queries, but doesn't stop them from modifying queries.</p>
<h3><span class="mw-headline" id="Incorrect_type_handling">Incorrect type handling</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=SQL_injection&amp;action=edit&amp;section=5" title="Edit section: Incorrect type handling">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>This form of SQL injection occurs when a <b>user-supplied</b> field is not <a class="mw-redirect" href="/wiki/Strongly_typed" title="Strongly typed">strongly typed</a> or is not checked for <a href="/wiki/Data_type" title="Data type">type</a> constraints. This could take place when a numeric field is to be used in a SQL statement, but the programmer makes no checks to validate that the user supplied input is numeric. For example:</p>
<pre>
statement := "<code class="mw-highlight" dir="ltr"><span class="k">SELECT</span> <span class="o">*</span> <span class="k">FROM</span> <span class="n">userinfo</span> <span class="k">WHERE</span> <span class="n">id</span> <span class="o">=</span></code>" + a_variable + ";"
</pre>
<p>It is clear from this statement that the author intended a_variable to be a number correlating to the "id" field. However, if it is in fact a <a href="/wiki/String_(computer_science)" title="String (computer science)">string</a> then the <a class="mw-redirect" href="/wiki/End-user" title="End-user">end-user</a> may manipulate the statement as they choose, thereby bypassing the need for escape characters. For example, setting a_variable to</p>
<pre>
1;DROP TABLE users
</pre>
<p>will drop (delete) the "users" table from the database, since the SQL becomes:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">SELECT</span> <span class="o">*</span> <span class="k">FROM</span> <span class="n">userinfo</span> <span class="k">WHERE</span> <span class="n">id</span><span class="o">=</span><span class="mi">1</span><span class="p">;</span> <span class="k">DROP</span> <span class="k">TABLE</span> <span class="n">users</span><span class="p">;</span>
</pre></div>
<h3><span class="mw-headline" id="Blind_SQL_injection">Blind SQL injection</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=SQL_injection&amp;action=edit&amp;section=6" title="Edit section: Blind SQL injection">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Blind SQL Injection is used when a web application is vulnerable to an SQL injection but the results of the injection are not visible to the attacker. The page with the vulnerability may not be one that displays data but will display differently depending on the results of a logical statement injected into the legitimate SQL statement called for that page. This type of attack has traditionally been considered time-intensive because a new statement needed to be crafted for each bit recovered, and depending on its structure, the attack may consist of many unsuccessful requests. Recent advancements have allowed each request to recover multiple bits, with no unsuccessful requests, allowing for more consistent and efficient extraction. <sup class="reference" id="cite_ref-14"><a href="#cite_note-14">[14]</a></sup> There are several tools that can automate these attacks once the location of the vulnerability and the target information has been established.<sup class="reference" id="cite_ref-15"><a href="#cite_note-15">[15]</a></sup></p>
<h4><span class="mw-headline" id="Conditional_responses">Conditional responses</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=SQL_injection&amp;action=edit&amp;section=7" title="Edit section: Conditional responses">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>One type of blind SQL injection forces the database to evaluate a logical statement on an ordinary application screen. As an example, a book review website uses a <a href="/wiki/Query_string" title="Query string">query string</a> to determine which book review to display. So the <a class="mw-redirect" href="/wiki/URL" title="URL">URL</a> <code>http://books.example.com/showReview.php?ID=5</code> would cause the server to run the query</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">SELECT</span> <span class="o">*</span> <span class="k">FROM</span> <span class="n">bookreviews</span> <span class="k">WHERE</span> <span class="n">ID</span> <span class="o">=</span> <span class="s1">'Value(ID)'</span><span class="p">;</span>
</pre></div>
<p>from which it would populate the review page with data from the review with <a href="/wiki/Identifier" title="Identifier">ID</a> 5, stored in the <a href="/wiki/Table_(database)" title="Table (database)">table</a> bookreviews. The query happens completely on the server; the user does not know the names of the database, table, or fields, nor does the user know the query string. The user only sees that the above URL returns a book review. A <a class="mw-redirect" href="/wiki/Hacker_(computer_security)" title="Hacker (computer security)">hacker</a> can load the URLs <code><code class="mw-highlight" dir="ltr"><span class="n">http</span><span class="p">:</span><span class="o">//</span><span class="n">books</span><span class="p">.</span><span class="n">example</span><span class="p">.</span><span class="n">com</span><span class="o">/</span><span class="n">showReview</span><span class="p">.</span><span class="n">php</span><span class="o">?</span><span class="n">ID</span><span class="o">=</span><span class="mi">5</span> <span class="k">OR</span> <span class="mi">1</span><span class="o">=</span><span class="mi">1</span></code></code> and <code><code class="mw-highlight" dir="ltr"><span class="n">http</span><span class="p">:</span><span class="o">//</span><span class="n">books</span><span class="p">.</span><span class="n">example</span><span class="p">.</span><span class="n">com</span><span class="o">/</span><span class="n">showReview</span><span class="p">.</span><span class="n">php</span><span class="o">?</span><span class="n">ID</span><span class="o">=</span><span class="mi">5</span> <span class="k">AND</span> <span class="mi">1</span><span class="o">=</span><span class="mi">2</span></code></code>, which may result in queries</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">SELECT</span> <span class="o">*</span> <span class="k">FROM</span> <span class="n">bookreviews</span> <span class="k">WHERE</span> <span class="n">ID</span> <span class="o">=</span> <span class="s1">'5'</span> <span class="k">OR</span> <span class="s1">'1'</span><span class="o">=</span><span class="s1">'1'</span><span class="p">;</span>
<span class="k">SELECT</span> <span class="o">*</span> <span class="k">FROM</span> <span class="n">bookreviews</span> <span class="k">WHERE</span> <span class="n">ID</span> <span class="o">=</span> <span class="s1">'5'</span> <span class="k">AND</span> <span class="s1">'1'</span><span class="o">=</span><span class="s1">'2'</span><span class="p">;</span>
</pre></div>
<p>respectively. If the original review loads with the "1=1" URL and a blank or error page is returned from the "1=2" URL, and the returned page has not been created to alert the user the input is invalid, or in other words, has been caught by an input test script, the site is likely vulnerable to a SQL injection attack as the query will likely have passed through successfully in both cases. The hacker may proceed with this query string designed to reveal the version number of <a href="/wiki/MySQL" title="MySQL">MySQL</a> running on the server: <code><code class="mw-highlight" dir="ltr"><span class="n">http</span><span class="p">:</span><span class="o">//</span><span class="n">books</span><span class="p">.</span><span class="n">example</span><span class="p">.</span><span class="n">com</span><span class="o">/</span><span class="n">showReview</span><span class="p">.</span><span class="n">php</span><span class="o">?</span><span class="n">ID</span><span class="o">=</span><span class="mi">5</span> <span class="k">AND</span> <span class="nf">substring</span><span class="p">(</span><span class="o">@@</span><span class="n">version</span><span class="p">,</span> <span class="mi">1</span><span class="p">,</span> <span class="nf">INSTR</span><span class="p">(</span><span class="o">@@</span><span class="n">version</span><span class="p">,</span> <span class="s1">'.'</span><span class="p">)</span> <span class="o">-</span> <span class="mi">1</span><span class="p">)</span><span class="o">=</span><span class="mi">4</span></code></code>, which would show the book review on a server running MySQL 4 and a blank or error page otherwise. The hacker can continue to use code within query strings to glean more information from the server until another avenue of attack is discovered or his or her goals are achieved.<sup class="reference" id="cite_ref-16"><a href="#cite_note-16">[16]</a></sup><sup class="reference" id="cite_ref-17"><a href="#cite_note-17">[17]</a></sup></p>
<h3><span class="mw-headline" id="Second_order_SQL_injection">Second order SQL injection</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=SQL_injection&amp;action=edit&amp;section=8" title="Edit section: Second order SQL injection">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Second order SQL injection occurs when submitted values contain malicious commands that are stored rather than executed immediately. In some cases, the application may correctly encode an SQL statement and store it as valid SQL. Then, another part of that application without controls to protect against SQL injection might execute that stored SQL statement. This attack requires more knowledge of how submitted values are later used. Automated web application security scanners would not easily detect this type of SQL injection and may need to be manually instructed where to check for evidence that it is being attempted.</p>
<h2><span class="mw-headline" id="Mitigation">Mitigation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=SQL_injection&amp;action=edit&amp;section=9" title="Edit section: Mitigation">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>An SQL injection is a well known attack and easily prevented by simple measures. After an apparent SQL injection attack on <a href="/wiki/TalkTalk_Group" title="TalkTalk Group">Talktalk</a> in 2015, the BBC reported that security experts were stunned that such a large company would be vulnerable to it.<sup class="reference" id="cite_ref-18"><a href="#cite_note-18">[18]</a></sup></p>
<h3><span class="mw-headline" id="Parameterized_statements">Parameterized statements</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=SQL_injection&amp;action=edit&amp;section=10" title="Edit section: Parameterized statements">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Prepared_statement" title="Prepared statement">Prepared statement</a></div>
<p>With most development platforms, parameterized statements that work with parameters can be used (sometimes called placeholders or <a class="mw-redirect" href="/wiki/Bind_variable" title="Bind variable">bind variables</a>) instead of embedding user input in the statement. A placeholder can only store a value of the given type and not an arbitrary SQL fragment. Hence the SQL injection would simply be treated as a strange (and probably invalid) parameter value.</p>
<p>In many cases, the SQL statement is fixed, and each parameter is a <a class="mw-redirect" href="/wiki/Scalar_(computing)" title="Scalar (computing)">scalar</a>, not a <a href="/wiki/Table_(database)" title="Table (database)">table</a>. The user input is then assigned (bound) to a parameter.<sup class="reference" id="cite_ref-19"><a href="#cite_note-19">[19]</a></sup></p>
<h4><span class="mw-headline" id="Enforcement_at_the_coding_level">Enforcement at the coding level</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=SQL_injection&amp;action=edit&amp;section=11" title="Edit section: Enforcement at the coding level">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Using <a href="/wiki/Object-relational_mapping" title="Object-relational mapping">object-relational mapping</a> libraries avoids the need to write SQL code. The ORM library in effect will generate parameterized SQL statements from object-oriented code.</p>
<h3><span class="mw-headline" id="Escaping">Escaping</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=SQL_injection&amp;action=edit&amp;section=12" title="Edit section: Escaping">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>A straightforward, though error-prone way to prevent injections is to escape characters that have a special meaning in SQL. The manual for an SQL DBMS explains which characters have a special meaning, which allows creating a comprehensive <a href="/wiki/Blacklist_(computing)" title="Blacklist (computing)">blacklist</a> of characters that need translation. For instance, every occurrence of a single quote (<code>'</code>) in a parameter must be replaced by two single quotes (<code>''</code>) to form a valid SQL string literal. For example, in <a href="/wiki/PHP" title="PHP">PHP</a> it is usual to escape parameters using the function <code>mysqli_real_escape_string();</code> before sending the SQL query:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="nv">$mysqli</span> <span class="o">=</span> <span class="k">new</span> <span class="nx">mysqli</span><span class="p">(</span><span class="s1">'hostname'</span><span class="p">,</span> <span class="s1">'db_username'</span><span class="p">,</span> <span class="s1">'db_password'</span><span class="p">,</span> <span class="s1">'db_name'</span><span class="p">);</span>
<span class="nv">$query</span> <span class="o">=</span> <span class="nb">sprintf</span><span class="p">(</span><span class="s2">"SELECT * FROM `Users` WHERE UserName='%s' AND Password='%s'"</span><span class="p">,</span>
                  <span class="nv">$mysqli</span><span class="o">-&gt;</span><span class="na">real_escape_string</span><span class="p">(</span><span class="nv">$username</span><span class="p">),</span>
                  <span class="nv">$mysqli</span><span class="o">-&gt;</span><span class="na">real_escape_string</span><span class="p">(</span><span class="nv">$password</span><span class="p">));</span>
<span class="nv">$mysqli</span><span class="o">-&gt;</span><span class="na">query</span><span class="p">(</span><span class="nv">$query</span><span class="p">);</span>
</pre></div>
<p>This function prepends backslashes to the following characters: \x00, \n, \r, \, ', " and \x1a. This function is normally used to make data safe before sending a query to <a href="/wiki/MySQL" title="MySQL">MySQL</a>.<sup class="reference" id="cite_ref-20"><a href="#cite_note-20">[20]</a></sup><br/>
There are other functions for many database types in PHP such as pg_escape_string() for <a href="/wiki/PostgreSQL" title="PostgreSQL">PostgreSQL</a>. The function <code>addslashes(string $str)</code> works for escaping characters, and is used especially for querying on databases that do not have escaping functions in PHP. It returns a string with backslashes before characters that need to be quoted in database queries, etc. These characters are single quote ('), double quote ("), backslash (\) and NUL (the NULL byte).<sup class="reference" id="cite_ref-21"><a href="#cite_note-21">[21]</a></sup><br/>
Routinely passing escaped strings to SQL is error prone because it is easy to forget to escape a given string. Creating a transparent layer to secure the input can reduce this error-proneness, if not entirely eliminate it.<sup class="reference" id="cite_ref-22"><a href="#cite_note-22">[22]</a></sup></p>
<h3><span class="mw-headline" id="Pattern_check">Pattern check</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=SQL_injection&amp;action=edit&amp;section=13" title="Edit section: Pattern check">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Integer, float or boolean,string parameters can be checked if their value is valid representation for the given type. Strings that must follow some strict pattern (date, UUID, alphanumeric only, etc.) can be checked if they match this pattern.</p>
<h3><span class="mw-headline" id="Database_permissions">Database permissions</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=SQL_injection&amp;action=edit&amp;section=14" title="Edit section: Database permissions">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Limiting the permissions on the database logon used by the web application to only what is needed may help reduce the effectiveness of any SQL injection attacks that exploit any bugs in the web application.</p>
<p>For example, on <a href="/wiki/Microsoft_SQL_Server" title="Microsoft SQL Server">Microsoft SQL Server</a>, a database logon could be restricted from selecting on some of the system tables which would limit exploits that try to insert JavaScript into all the text columns in the database.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="n">deny</span> <span class="k">select</span> <span class="k">on</span> <span class="n">sys</span><span class="p">.</span><span class="n">sysobjects</span> <span class="k">to</span> <span class="n">webdatabaselogon</span><span class="p">;</span>
<span class="n">deny</span> <span class="k">select</span> <span class="k">on</span> <span class="n">sys</span><span class="p">.</span><span class="n">objects</span> <span class="k">to</span> <span class="n">webdatabaselogon</span><span class="p">;</span>
<span class="n">deny</span> <span class="k">select</span> <span class="k">on</span> <span class="n">sys</span><span class="p">.</span><span class="n">tables</span> <span class="k">to</span> <span class="n">webdatabaselogon</span><span class="p">;</span>
<span class="n">deny</span> <span class="k">select</span> <span class="k">on</span> <span class="n">sys</span><span class="p">.</span><span class="n">views</span> <span class="k">to</span> <span class="n">webdatabaselogon</span><span class="p">;</span>
<span class="n">deny</span> <span class="k">select</span> <span class="k">on</span> <span class="n">sys</span><span class="p">.</span><span class="n">packages</span> <span class="k">to</span> <span class="n">webdatabaselogon</span><span class="p">;</span>
</pre></div>
<h2><span class="mw-headline" id="Examples">Examples</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=SQL_injection&amp;action=edit&amp;section=15" title="Edit section: Examples">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<ul>
<li>In February 2002, Jeremiah Jacks discovered that Guess.com was vulnerable to an SQL injection attack, permitting anyone able to construct a properly-crafted URL to pull down 200,000+ names, credit card numbers and expiration dates in the site's customer database.<sup class="reference" id="cite_ref-23"><a href="#cite_note-23">[23]</a></sup></li>
<li>On November 1, 2005, a teenaged hacker used SQL injection to break into the site of a <a href="/wiki/Taiwan" title="Taiwan">Taiwanese</a> information security magazine from the Tech Target group and steal customers' information.<sup class="reference" id="cite_ref-24"><a href="#cite_note-24">[24]</a></sup></li>
<li>On January 13, 2006, <a href="/wiki/Russia" title="Russia">Russian</a> computer criminals broke into a <a href="/wiki/Government_of_Rhode_Island" title="Government of Rhode Island">Rhode Island government</a> website and allegedly stole credit card data from individuals who have done business online with state agencies.<sup class="reference" id="cite_ref-25"><a href="#cite_note-25">[25]</a></sup></li>
<li>On March 29, 2006, a hacker discovered an SQL injection flaw in an official <a href="/wiki/Government_of_India" title="Government of India">Indian government</a>'s <a href="/wiki/Tourism_in_India" title="Tourism in India">tourism</a> site.<sup class="reference" id="cite_ref-26"><a href="#cite_note-26">[26]</a></sup></li>
<li>On June 29, 2007, a computer criminal defaced the <a href="/wiki/Microsoft" title="Microsoft">Microsoft</a> UK website using SQL injection.<sup class="reference" id="cite_ref-27"><a href="#cite_note-27">[27]</a></sup><sup class="reference" id="cite_ref-28"><a href="#cite_note-28">[28]</a></sup> UK website <i><a href="/wiki/The_Register" title="The Register">The Register</a></i> quoted a Microsoft <a href="/wiki/Spokesperson" title="Spokesperson">spokesperson</a> acknowledging the problem.</li>
<li>On September 19, 2007 and January 26, 2009 the Turkish hacker group "m0sted" used SQL injection to exploit Microsoft's SQL Server to hack web servers belonging to <a href="/wiki/McAlester_Army_Ammunition_Plant" title="McAlester Army Ammunition Plant">McAlester Army Ammunition Plant</a> and the <a href="/wiki/United_States_Army_Corps_of_Engineers" title="United States Army Corps of Engineers">US Army Corps of Engineers</a> respectively.<sup class="reference" id="cite_ref-29"><a href="#cite_note-29">[29]</a></sup></li>
<li>In January 2008, tens of thousands of PCs were infected by an automated SQL injection attack that exploited a vulnerability in application code that uses <a href="/wiki/Microsoft_SQL_Server" title="Microsoft SQL Server">Microsoft SQL Server</a> as the database store.<sup class="reference" id="cite_ref-chinesefarm_30-0"><a href="#cite_note-chinesefarm-30">[30]</a></sup></li>
<li>In July 2008, <a href="/wiki/Kaspersky_Lab" title="Kaspersky Lab">Kaspersky</a>'s <a href="/wiki/Malaysia" title="Malaysia">Malaysian</a> site was hacked by the "m0sted" hacker group using SQL injection.</li>
<li>On April 13, 2008, the <a href="/wiki/Sex_offender_registries_in_the_United_States" title="Sex offender registries in the United States">Sexual and Violent Offender Registry</a> of <a href="/wiki/Oklahoma" title="Oklahoma">Oklahoma</a> shut down its website for "<a class="mw-redirect" href="/wiki/Routine_maintenance" title="Routine maintenance">routine maintenance</a>" after being informed that 10,597 <a href="/wiki/Social_Security_number" title="Social Security number">Social Security numbers</a> belonging to <a href="/wiki/Sex_offender" title="Sex offender">sex offenders</a> had been downloaded via an SQL injection attack<sup class="reference" id="cite_ref-31"><a href="#cite_note-31">[31]</a></sup></li>
<li>In May 2008, a <a href="/wiki/Server_farm" title="Server farm">server farm</a> inside <a href="/wiki/China" title="China">China</a> used automated queries to <a href="/wiki/Google_Search" title="Google Search">Google's search engine</a> to identify <a href="/wiki/Microsoft_SQL_Server" title="Microsoft SQL Server">SQL server</a> websites which were vulnerable to the attack of an automated SQL injection tool.<sup class="reference" id="cite_ref-chinesefarm_30-1"><a href="#cite_note-chinesefarm-30">[30]</a></sup><sup class="reference" id="cite_ref-attackspecifics_32-0"><a href="#cite_note-attackspecifics-32">[32]</a></sup></li>
<li>In 2008, at least April through August, a sweep of attacks began exploiting the SQL injection vulnerabilities of Microsoft's <a href="/wiki/Internet_Information_Services" title="Internet Information Services">IIS web server</a> and <a href="/wiki/Microsoft_SQL_Server" title="Microsoft SQL Server">SQL Server database server</a>. The attack does not require guessing the name of a table or column, and corrupts all text columns in all tables in a single request.<sup class="reference" id="cite_ref-broad_inject_specifics_33-0"><a href="#cite_note-broad_inject_specifics-33">[33]</a></sup> A HTML string that references a <a href="/wiki/Malware" title="Malware">malware</a> <a href="/wiki/JavaScript" title="JavaScript">JavaScript</a> file is appended to each value. When that database value is later displayed to a website visitor, the script attempts several approaches at gaining control over a visitor's system. The number of exploited web pages is estimated at 500,000.<sup class="reference" id="cite_ref-broad_inject_numbers_34-0"><a href="#cite_note-broad_inject_numbers-34">[34]</a></sup></li>
<li>On August 17, 2009, the <a href="/wiki/United_States_Department_of_Justice" title="United States Department of Justice">United States Department of Justice</a> charged an American citizen, <a href="/wiki/Albert_Gonzalez" title="Albert Gonzalez">Albert Gonzalez</a>, and two unnamed Russians with the theft of 130 million credit card numbers using an SQL injection attack. In reportedly "the biggest case of <a href="/wiki/Identity_theft" title="Identity theft">identity theft</a> in American history", the man stole cards from a number of corporate victims after researching their <a href="/wiki/Payment_processor" title="Payment processor">payment processing systems</a>. Among the companies hit were credit card processor <a href="/wiki/Heartland_Payment_Systems" title="Heartland Payment Systems">Heartland Payment Systems</a>, convenience store chain <a href="/wiki/7-Eleven" title="7-Eleven">7‑Eleven</a>, and supermarket chain <a class="mw-redirect" href="/wiki/Hannaford_Brothers" title="Hannaford Brothers">Hannaford Brothers</a>.<sup class="reference" id="cite_ref-35"><a href="#cite_note-35">[35]</a></sup></li>
<li>In December 2009, an attacker breached a <a href="/wiki/RockYou" title="RockYou">RockYou</a> plaintext database containing the <a href="/wiki/Encryption" title="Encryption">unencrypted</a> usernames and passwords of about 32 million users using an SQL injection attack.<sup class="reference" id="cite_ref-36"><a href="#cite_note-36">[36]</a></sup></li>
<li>On July 2010, a South American security researcher who goes by the <a href="/wiki/User_(computing)" title="User (computing)">handle</a> "Ch Russo" obtained sensitive user information from popular <a href="/wiki/BitTorrent" title="BitTorrent">BitTorrent</a> site <a href="/wiki/The_Pirate_Bay" title="The Pirate Bay">The Pirate Bay</a>. He gained access to the site's administrative control panel and exploited a SQL injection vulnerability that enabled him to collect user account information, including <a href="/wiki/IP_address" title="IP address">IP addresses</a>, <a href="/wiki/MD5" title="MD5">MD5</a> <a href="/wiki/Cryptographic_hash_function" title="Cryptographic hash function">password hashes</a> and records of which torrents individual users have uploaded.<sup class="reference" id="cite_ref-37"><a href="#cite_note-37">[37]</a></sup></li>
<li>From July 24 to 26, 2010, attackers from <a href="/wiki/Japan" title="Japan">Japan</a> and <a href="/wiki/China" title="China">China</a> used an SQL injection to gain access to customers' credit card data from Neo Beat, an <a href="/wiki/Osaka" title="Osaka">Osaka</a>-based company that runs a large online supermarket site. The attack also affected seven business partners including supermarket chains Izumiya Co, Maruetsu Inc, and Ryukyu Jusco Co. The theft of data affected a reported 12,191 customers. As of August 14, 2010 it was reported that there have been more than 300 cases of credit card information being used by third parties to purchase goods and services in China.</li>
<li>On September 19 during the <a href="/wiki/Swedish_general_election,_2010" title="Swedish general election, 2010">2010 Swedish general election</a> a voter attempted a code injection by hand writing SQL commands as part of a <a href="/wiki/Write-in_candidate" title="Write-in candidate">write‑in</a> vote.<sup class="reference" id="cite_ref-38"><a href="#cite_note-38">[38]</a></sup></li>
<li>On November 8, 2010 the British <a href="/wiki/Royal_Navy" title="Royal Navy">Royal Navy</a> website was compromised by a Romanian hacker named TinKode using SQL injection.<sup class="reference" id="cite_ref-39"><a href="#cite_note-39">[39]</a></sup><sup class="reference" id="cite_ref-40"><a href="#cite_note-40">[40]</a></sup></li>
<li>On February 5, 2011 <a href="/wiki/HBGary" title="HBGary">HBGary</a>, a technology security firm, was broken into by <a href="/wiki/LulzSec" title="LulzSec">LulzSec</a> using a SQL injection in their CMS-driven website<sup class="reference" id="cite_ref-41"><a href="#cite_note-41">[41]</a></sup></li>
<li>On March 27, 2011, <a class="external text" href="http://www.mysql.com" rel="nofollow">mysql.com</a>, the official homepage for <a href="/wiki/MySQL" title="MySQL">MySQL</a>, was compromised by a hacker using SQL blind injection<sup class="reference" id="cite_ref-42"><a href="#cite_note-42">[42]</a></sup></li>
<li>On April 11, 2011, <a href="/wiki/Barracuda_Networks" title="Barracuda Networks">Barracuda Networks</a> was compromised using an SQL injection flaw. <a href="/wiki/Email_address" title="Email address">Email addresses</a> and usernames of employees were among the information obtained.<sup class="reference" id="cite_ref-43"><a href="#cite_note-43">[43]</a></sup></li>
<li>Over a period of 4 hours on April 27, 2011, an automated SQL injection attack occurred on <a class="mw-redirect" href="/wiki/Broadband_Reports" title="Broadband Reports">Broadband Reports</a> website that was able to extract 8% of the username/password pairs: 8,000 random accounts of the 9,000 active and 90,000 old or inactive accounts.<sup class="reference" id="cite_ref-DSLReports_44-0"><a href="#cite_note-DSLReports-44">[44]</a></sup><sup class="reference" id="cite_ref-Cnet_News_45-0"><a href="#cite_note-Cnet_News-45">[45]</a></sup><sup class="reference" id="cite_ref-The_Tech_Herald_46-0"><a href="#cite_note-The_Tech_Herald-46">[46]</a></sup></li>
<li>On June 1, 2011, "<a class="mw-redirect" href="/wiki/Hacktivist" title="Hacktivist">hacktivists</a>" of the group <a href="/wiki/LulzSec" title="LulzSec">LulzSec</a> were accused of using SQLI to steal <a href="/wiki/Coupon" title="Coupon">coupons</a>, download keys, and passwords that were stored in plaintext on <a href="/wiki/Sony" title="Sony">Sony</a>'s website, accessing the personal information of a million users.<sup class="reference" id="cite_ref-47"><a href="#cite_note-47">[47]</a></sup><sup class="reference" id="cite_ref-48"><a href="#cite_note-48">[48]</a></sup></li>
<li>In June 2011, <a href="/wiki/PBS" title="PBS">PBS</a> was hacked, mostly likely through use of SQL injection; the full process used by hackers to execute SQL injections was described in this <a class="external text" href="http://blog.imperva.com/2011/05/pbs-breached-how-hackers-probably-did-it.html" rel="nofollow">Imperva</a> blog.<sup class="reference" id="cite_ref-PBS_Breached_-_How_Hackers_Probably_Did_It_49-0"><a href="#cite_note-PBS_Breached_-_How_Hackers_Probably_Did_It-49">[49]</a></sup></li>
<li>In May 2012, the website for <i><a href="/wiki/Wurm_Online" title="Wurm Online">Wurm Online</a></i>, a <a href="/wiki/Massively_multiplayer_online_game" title="Massively multiplayer online game">massively multiplayer online game</a>, was shut down from an SQL injection while the site was being updated.<sup class="reference" id="cite_ref-50"><a href="#cite_note-50">[50]</a></sup></li>
<li><a href="/wiki/2012_Yahoo!_Voices_hack" title="2012 Yahoo! Voices hack">In July 2012</a> a hacker group was reported to have stolen 450,000 login credentials from <a href="/wiki/Yahoo!" title="Yahoo!">Yahoo!</a>. The logins were stored in <a href="/wiki/Plain_text" title="Plain text">plain text</a> and were allegedly taken from a Yahoo <a href="/wiki/Subdomain" title="Subdomain">subdomain</a>, <a href="/wiki/Yahoo!_Voices" title="Yahoo! Voices">Yahoo! Voices</a>. The group breached Yahoo's security by using a "<a href="/wiki/Set_operations_(SQL)#UNION_operator" title="Set operations (SQL)">union</a>-based SQL injection technique".<sup class="reference" id="cite_ref-51"><a href="#cite_note-51">[51]</a></sup><sup class="reference" id="cite_ref-52"><a href="#cite_note-52">[52]</a></sup></li>
<li>On October 1, 2012, a hacker group called "Team GhostShell" published the personal records of students, faculty, employees, and alumni from 53 universities including <a class="mw-redirect" href="/wiki/Harvard" title="Harvard">Harvard</a>, <a href="/wiki/Princeton_University" title="Princeton University">Princeton</a>, <a class="mw-redirect" href="/wiki/Stanford" title="Stanford">Stanford</a>, <a class="mw-redirect" href="/wiki/Cornell" title="Cornell">Cornell</a>, <a href="/wiki/Johns_Hopkins_University" title="Johns Hopkins University">Johns Hopkins</a>, and the <a href="/wiki/University_of_Zurich" title="University of Zurich">University of Zurich</a> on <a href="/wiki/Pastebin" title="Pastebin">pastebin.com</a>. The hackers claimed that they were trying to "raise awareness towards the changes made in today’s education", bemoaning changing education laws in Europe and increases in <a href="/wiki/College_tuition_in_the_United_States" title="College tuition in the United States">tuition in the United States</a>.<sup class="reference" id="cite_ref-53"><a href="#cite_note-53">[53]</a></sup></li>
<li>In February 2013, a group of Maldivian hackers, hacked the website "UN-Maldives" using SQL Injection.</li>
<li>On June 27, 2013, hacker group "<a href="/wiki/RedHack" title="RedHack">RedHack</a>" breached Istanbul Administration Site.<sup class="reference" id="cite_ref-54"><a href="#cite_note-54">[54]</a></sup> They claimed that, they’ve been able to erase people's debts to water, gas, Internet, electricity, and telephone companies. Additionally, they published admin user name and password for other citizens to log in and clear their debts early morning. They announced the news from Twitter.<sup class="reference" id="cite_ref-55"><a href="#cite_note-55">[55]</a></sup></li>
<li>On November 4, 2013, hacktivist group "RaptorSwag" allegedly compromised 71 Chinese government databases using an SQL injection attack on the Chinese Chamber of International Commerce. The leaked data was posted publicly in cooperation with <a href="/wiki/Anonymous_(group)" title="Anonymous (group)">Anonymous</a>.<sup class="reference" id="cite_ref-56"><a href="#cite_note-56">[56]</a></sup></li>
<li>On February 2, 2014, AVS TV had 40,000 accounts leaked by a hacking group called @deletesec <sup class="reference" id="cite_ref-57"><a href="#cite_note-57">[57]</a></sup></li>
<li>On February 21, 2014, United Nations Internet Governance Forum had 3,215 account details leaked.<sup class="reference" id="cite_ref-58"><a href="#cite_note-58">[58]</a></sup></li>
<li>On February 21, 2014, Hackers of a group called @deletesec hacked Spirol International after allegedly threatening to have the hackers arrested for reporting the security vulnerability. 70,000 user details were exposed over this conflict.<sup class="reference" id="cite_ref-59"><a href="#cite_note-59">[59]</a></sup></li>
<li>On March 7, 2014, officials at Johns Hopkins University publicly announced that their Biomedical Engineering Servers had become victim to an SQL injection attack carried out by an Anonymous hacker named "Hooky" and aligned with hacktivist group "RaptorSwag". The hackers compromised personal details of 878 students and staff, posting a <a class="external text" href="http://pastebin.com/UG4fYnby" rel="nofollow">press release</a> and the leaked data on the internet.<sup class="reference" id="cite_ref-60"><a href="#cite_note-60">[60]</a></sup></li>
<li>In August 2014, <a href="/wiki/Milwaukee" title="Milwaukee">Milwaukee</a>-based computer security company Hold Security disclosed that it uncovered <a href="/wiki/2014_Russian_hacker_password_theft" title="2014 Russian hacker password theft">a theft of confidential information</a> from nearly 420,000 websites through SQL injections.<sup class="reference" id="cite_ref-61"><a href="#cite_note-61">[61]</a></sup> <i><a href="/wiki/The_New_York_Times" title="The New York Times">The New York Times</a></i> confirmed this finding by hiring a security expert to check the claim.<sup class="reference" id="cite_ref-62"><a href="#cite_note-62">[62]</a></sup></li>
<li>In October 2015, an SQL injection attack was used to steal the personal details of 156,959 customers from British telecommunications company <a href="/wiki/TalkTalk_Group" title="TalkTalk Group">Talk Talk's</a> servers, exploiting a vulnerability in a legacy web portal<sup class="reference" id="cite_ref-63"><a href="#cite_note-63">[63]</a></sup></li>
</ul>
<h2><span class="mw-headline" id="In_popular_culture">In popular culture</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=SQL_injection&amp;action=edit&amp;section=16" title="Edit section: In popular culture">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<ul>
<li>Unauthorized login to web sites by means of SQL injection forms the basis of one of the subplots in <a class="mw-redirect" href="/wiki/J.K._Rowling" title="J.K. Rowling">J.K. Rowling</a>'s novel <i><a href="/wiki/The_Casual_Vacancy" title="The Casual Vacancy">The Casual Vacancy</a></i>, published in 2012.</li>
<li>An <i><a href="/wiki/Xkcd" title="Xkcd">xkcd</a></i> cartoon involved a character "Robert'); DROP TABLE students;--" named to carry out a SQL injection. As a result of this cartoon, SQL injection is sometimes informally referred to as 'Bobby Tables'.<sup class="reference" id="cite_ref-64"><a href="#cite_note-64">[64]</a></sup><sup class="reference" id="cite_ref-65"><a href="#cite_note-65">[65]</a></sup></li>
<li>In 2014, an individual in Poland legally renamed his business to <i>Dariusz Jakubowski x'; DROP TABLE users; SELECT '1</i> in an attempt to disrupt operation of spammers’ <a href="/wiki/Web_scraping" title="Web scraping">harvesting bots</a>.<sup class="reference" id="cite_ref-66"><a href="#cite_note-66">[66]</a></sup></li>
<li>The 2015 game <a href="/wiki/Hacknet" title="Hacknet">Hacknet</a> has a hacking program called SQL_MemCorrupt. It is described as injecting a table entry that causes a corruption error in a SQL database, then queries said table, causing a SQL database crash and core dump.</li>
</ul>
