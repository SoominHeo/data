<div class="hatnote" role="note">For other uses of "Trunk", see <a class="mw-redirect mw-disambig" href="/wiki/Trunk_(disambiguation)" title="Trunk (disambiguation)">Trunk (disambiguation)</a>.</div>
<p>In the field of <a href="/wiki/Software_development" title="Software development">software development</a>, <i><b>trunk</b></i> refers to the unnamed <a class="mw-redirect" href="/wiki/Branching_(software)" title="Branching (software)">branch</a> (version) of a file tree under <a class="mw-redirect" href="/wiki/Revision_control" title="Revision control">revision control</a>. The trunk is usually meant to be the base of a project on which development progresses. If developers are working exclusively on the trunk, it always contains the latest <a href="/wiki/State_of_the_art" title="State of the art">cutting-edge</a> version of the project, but therefore may also be the most unstable version. Another approach is to split a branch off the trunk, implement changes in that branch and merge the changes back into the trunk when the branch has proven to be stable and working. Depending on development mode and <a href="/wiki/Commit_(data_management)" title="Commit (data management)">commit</a> policy the trunk may contain the most stable or the least stable or something-in-between version. Other terms for trunk include <i>baseline,</i> <i>mainline,</i> and <i>master,</i> though in some cases these are used with similar but distinct senses – see <a class="mw-redirect" href="/wiki/Revision_control#Common_vocabulary" title="Revision control">Revision control: Common vocabulary</a>. The trunk is also sometimes loosely referred to as HEAD, but properly head refers not to a branch, but to the most recent commit on a given branch, and both the trunk and each named branch has its own head.<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup></p>
<p>Often main developer work takes place in the trunk and stable versions are branched, and occasional bug-fixes are merged from branches to the trunk. When development of future versions is done in non-trunk branches, it is usually done for projects that do not change often, or where a change is expected to take a long time to develop until it will be ready for incorporating in the trunk.</p>
