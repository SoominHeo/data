<h1 class="firstHeading" id="firstHeading" lang="en">UTF-16</h1>

<p><b>UTF-16</b> (16-<a href="/wiki/Bit" title="Bit">bit</a> <a href="/wiki/Unicode" title="Unicode">Unicode</a> Transformation Format) is a <a href="/wiki/Character_encoding" title="Character encoding">character encoding</a> capable of encoding all 1,112,064 possible characters in Unicode. The encoding is <a href="/wiki/Variable-width_encoding" title="Variable-width encoding">variable-length</a>, as <a href="/wiki/Code_point" title="Code point">code points</a> are encoded with one or two 16-bit <i>code units</i>. (also see <a href="/wiki/Comparison_of_Unicode_encodings" title="Comparison of Unicode encodings">Comparison of Unicode encodings</a> for a comparison of <a href="/wiki/UTF-8" title="UTF-8">UTF-8</a>, -16 &amp; <a href="/wiki/UTF-32" title="UTF-32">-32</a>)</p>
<p>UTF-16 developed from an earlier fixed-width 16-bit encoding known as <b>UCS-2</b> (for 2-byte Universal Character Set) once it became clear that 16 bits were not sufficient for Unicode's user community.<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup></p>
<p></p>

<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=UTF-16&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In the late 1980s, work began on developing a uniform encoding for a "Universal Character Set" (UCS) that would replace earlier language-specific encodings with one coordinated system. The goal was to include all required characters from most of the world's languages, as well as symbols from technical domains such as science, mathematics, and music. The original idea was to expand the typical 256-character encodings requiring 1 byte per character with an encoding using 2<sup>16</sup> = 65,536 values requiring 2 bytes per character. Two groups worked on this in parallel, the <a class="mw-redirect" href="/wiki/IEEE" title="IEEE">IEEE</a> and the <a href="/wiki/Unicode_Consortium" title="Unicode Consortium">Unicode Consortium</a>, the latter representing mostly manufacturers of computing equipment. The two groups attempted to synchronize their character assignments, so that the developing encodings would be mutually compatible. The early 2-byte encoding was usually called "Unicode", but is now called "UCS-2".</p>
<p>Early in this process,<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (September 2015)">citation needed</span></a></i>]</sup> however, it became increasingly clear that 2<sup>16</sup> characters would not suffice, and IEEE introduced a larger 31-bit space with an encoding (UCS-4) that would require 4 bytes per character. This was resisted by the Unicode Consortium, both because 4 bytes per character wasted a lot of disk space and memory, and because some manufacturers were already heavily invested in 2-byte-per-character technology. The UTF-16 encoding scheme was developed as a compromise to resolve this impasse in version 2.0 of the Unicode standard in July 1996<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> and is fully specified in <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc2781" rel="nofollow">RFC 2781</a> published in 2000 by the <a class="mw-redirect" href="/wiki/IETF" title="IETF">IETF</a>.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup><sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup><sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup></p>
<p>In UTF-16, code points greater or equal to 2<sup>16</sup> are encoded using <i>two</i> 16-bit code units. The standards organizations chose the largest block available of un-allocated 16-bit code points to use as these code units, and code points from this range are not individually encodable in UTF-16 (and not legally encodable in any UTF encoding).</p>
<p>UTF-16 is specified in the latest versions of both the international standard <a class="mw-redirect" href="/wiki/ISO/IEC_10646" title="ISO/IEC 10646">ISO/IEC 10646</a> and the Unicode Standard.</p>
<h2><span class="mw-headline" id="Description">Description</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=UTF-16&amp;action=edit&amp;section=2" title="Edit section: Description">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="U.2B0000_to_U.2BD7FF_and_U.2BE000_to_U.2BFFFF">U+0000 to U+D7FF and U+E000 to U+FFFF</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=UTF-16&amp;action=edit&amp;section=3" title="Edit section: U+0000 to U+D7FF and U+E000 to U+FFFF">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Both UTF-16 and UCS-2 encode code points in this range as single 16-bit code units that are numerically equal to the corresponding code points. These code points in the <a href="/wiki/Plane_(Unicode)#Basic_Multilingual_Plane" title="Plane (Unicode)">Basic Multilingual Plane</a> (BMP) are the <i>only</i> code points that can be represented in UCS-2.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (November 2015)">citation needed</span></a></i>]</sup> As of Unicode 9.0 some modern non-latin Asian, Middle-eastern and African scripts fall outside this range.</p>
<h3><span class="mw-headline" id="U.2B10000_to_U.2B10FFFF">U+10000 to U+10FFFF</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=UTF-16&amp;action=edit&amp;section=4" title="Edit section: U+10000 to U+10FFFF">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Code points from the other planes (called <a href="/wiki/Plane_(Unicode)#Supplementary_Multilingual_Plane" title="Plane (Unicode)">Supplementary Planes</a>) are encoded as two 16-bit code units called <i>surrogate pairs</i>, by the following scheme:</p>

<ul>
<li>0x010000 is subtracted from the code point, leaving a 20-bit number in the range 0x000000..0x0FFFFF.</li>
<li>The top ten bits (a number in the range 0x0000..0x03FF) are added to 0xD800 to give the first 16-bit code unit or <i>high surrogate</i>, which will be in the range <span style="color:#00AA00;">0xD800..0xDBFF</span>.</li>
<li>The low ten bits (also in the range 0x0000..0x03FF) are added to 0xDC00 to give the second 16-bit code unit or <i>low surrogate</i>, which will be in the range <span style="color:#0070FF;">0xDC00..0xDFFF</span>.</li>
</ul>
<p>(High and low surrogates are also known as "leading" and "trailing" surrogates, respectively, analogous to the leading and trailing bytes of UTF-8.<sup class="reference" id="cite_ref-Unicode7Ch3s8_6-0"><a href="#cite_note-Unicode7Ch3s8-6">[6]</a></sup> Note that "high" surrogates have lower code-unit numbers than "low" surrogates.)</p>
<p>Since the ranges for the high surrogates, low surrogates, and valid BMP characters are disjoint, it is not possible for a surrogate to match a BMP character, or for (parts of) two adjacent characters to look like a legal surrogate pair. This simplifies searches a great deal. It also means that UTF-16 is <i>self-synchronizing</i> on 16-bit words: whether a code unit starts a character can be determined without examining earlier code units. UTF-8 shares these advantages, but many earlier multi-byte encoding schemes (such as <a href="/wiki/Shift_JIS" title="Shift JIS">Shift JIS</a> and other Asian multi byte encodings) did not allow unambiguous searching and could only be synchronized by re-parsing from the start of the string (UTF-16 is not self-synchronizing if one byte is lost or if traversal starts at a random byte).</p>
<p>Because the most commonly used characters are all in the BMP, handling of surrogate pairs is often not thoroughly tested. This leads to persistent bugs and potential security holes, even in popular and well-reviewed application software (e.g. CVE-2008-2938, CVE-2012-2135).<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup></p>
<p>The <a href="/wiki/Plane_(Unicode)" title="Plane (Unicode)">Supplementary Planes</a> contain <a href="/wiki/Emoji" title="Emoji">Emoji</a>, historic scripts, less used symbols, less used Chinese ideographs and some more.</p>
<h3><span class="mw-headline" id="U.2BD800_to_U.2BDFFF">U+D800 to U+DFFF</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=UTF-16&amp;action=edit&amp;section=5" title="Edit section: U+D800 to U+DFFF">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The Unicode standard permanently reserves these code point values for UTF-16 encoding of the high and low surrogates, and they will never be assigned a character, so there should be no reason to encode them. The official Unicode standard says that no UTF forms, including UTF-16, can encode these code points.</p>
<p>However UCS-2, UTF-8, and <a href="/wiki/UTF-32" title="UTF-32">UTF-32</a> can encode these code points in trivial and obvious ways, and large amounts of software does so even though the standard states that such arrangements should be treated as encoding errors. It is possible to unambiguously encode them in UTF-16 by using a code unit equal to the code point, as long as no sequence of two code units can be interpreted as a legal surrogate pair (that is, as long as a high surrogate is never followed by a low surrogate). The majority of UTF-16 encoder and decoder implementations translate between encodings as though this were the case<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (October 2011)">citation needed</span></a></i>]</sup> and Windows allows such sequences in filenames.</p>
<h3><span class="mw-headline" id="Examples">Examples</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=UTF-16&amp;action=edit&amp;section=6" title="Edit section: Examples">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Consider the encoding of U+10437 (𐐷):</p>
<ul>
<li>Subtract 0x10000 from 0x10437. The result is 0x00437, <span style="color: #00AA00;">0000 0000 01</span><span style="color: #0070FF;">00 0011 0111</span>.</li>
<li>Split this into the high 10-bit value and the low 10-bit value: <span style="color: #00AA00;">0000000001</span> and <span style="color: #0070FF;">0000110111</span>.</li>
<li>Add 0xD800 to the high value to form the <span style="color: #00AA00;">high surrogate</span>: 0xD800 + 0x0001 = <span style="color: #00AA00;">0xD801</span>.</li>
<li>Add 0xDC00 to the low value to form the <span style="color: #0070FF;">low surrogate</span>: 0xDC00 + 0x0037 = <span style="color: #0070FF;">0xDC37</span>.</li>
</ul>

<h2><span class="mw-headline" id="Byte_order_encoding_schemes"><span id="UTF-16LE"></span><span id="UTF-16BE"></span>Byte order encoding schemes</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=UTF-16&amp;action=edit&amp;section=7" title="Edit section: Byte order encoding schemes">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>UTF-16 and UCS-2 produce a sequence of 16-bit code units. Since most communication and storage protocols are defined for bytes, and each unit thus takes two 8-bit bytes, the order of the bytes may depend on the <a href="/wiki/Endianness" title="Endianness">endianness</a> (byte order) of the computer architecture.</p>
<p>To assist in recognizing the byte order of code units, <b>UTF-16</b> allows a <a class="mw-redirect" href="/wiki/Byte_Order_Mark" title="Byte Order Mark">Byte Order Mark</a> (BOM), a code point with the value U+FEFF, to precede the first actual coded value.<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[nb 1]</a></sup> (U+FEFF is the invisible <a class="mw-redirect" href="/wiki/Zero-width_non-breaking_space" title="Zero-width non-breaking space">zero-width non-breaking space</a>/ZWNBSP character.)<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[nb 2]</a></sup> If the endian architecture of the decoder matches that of the encoder, the decoder detects the 0xFEFF value, but an opposite-endian decoder interprets the BOM as the non-character value U+FFFE reserved for this purpose. This incorrect result provides a hint to perform byte-swapping for the remaining values. If the BOM is missing, <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc2781" rel="nofollow">RFC 2781</a> says that big-endian encoding should be assumed. (In practice, due to Windows using little-endian order by default, many applications similarly assume little-endian encoding by default.) If there is no BOM, one method of recognizing a UTF-16 encoding is searching for the space character (U+0020) which is very common in texts in most languages.</p>
<p>The standard also allows the byte order to be stated explicitly by specifying <b>UTF-16BE</b> or <b>UTF-16LE</b> as the encoding type. When the byte order is specified explicitly this way, a BOM is specifically <i>not</i> supposed to be prepended to the text, and a U+FEFF at the beginning should be handled as a ZWNBSP character. Many applications ignore the BOM code at the start of any Unicode encoding. Web browsers often use a BOM as a hint in determining the character encoding.<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[8]</a></sup></p>
<p>For <a href="/wiki/Internet" title="Internet">Internet</a> protocols, <a href="/wiki/Internet_Assigned_Numbers_Authority" title="Internet Assigned Numbers Authority">IANA</a> has approved "UTF-16", "UTF-16BE", and "UTF-16LE" as the names for these encodings (the names are case insensitive). The aliases <b>UTF_16</b> or <b>UTF16</b> may be meaningful in some programming languages or software applications, but they are not standard names in Internet protocols.</p>
<p>Similar designations, <b>UCS-2</b>, <b>UCS-2BE</b> and <b>UCS-2LE</b>, are used to imitate the UTF-16 labels and behaviour. However, "UCS-2 should now be considered obsolete. It no longer refers to an encoding form in either 10646 or the Unicode Standard."<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[9]</a></sup></p>
<h2><span class="mw-headline" id="Usage">Usage</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=UTF-16&amp;action=edit&amp;section=8" title="Edit section: Usage">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>UTF-16 is used for text in the OS API in <a href="/wiki/Windows_2000" title="Windows 2000">Microsoft Windows 2000</a>/<a href="/wiki/Windows_XP" title="Windows XP">XP</a>/<a class="mw-redirect" href="/wiki/Windows_2003" title="Windows 2003">2003</a>/<a href="/wiki/Windows_Vista" title="Windows Vista">Vista</a>/<a href="/wiki/Windows_7" title="Windows 7">7</a>/<a href="/wiki/Windows_8" title="Windows 8">8</a>/<a class="mw-redirect" href="/wiki/Windows_CE" title="Windows CE">CE</a>.<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[10]</a></sup> Older <a href="/wiki/Windows_NT" title="Windows NT">Windows NT</a> systems (prior to Windows 2000) only support UCS-2.<sup class="reference" id="cite_ref-13"><a href="#cite_note-13">[11]</a></sup> In Windows XP, no code point above U+FFFF is included in any font delivered with Windows for European languages.<sup class="reference" id="cite_ref-14"><a href="#cite_note-14">[12]</a></sup><sup class="reference" id="cite_ref-15"><a href="#cite_note-15">[13]</a></sup> Files and network data tend to be a mix of UTF-16, UTF-8, and legacy byte encodings.</p>
<p><a href="/wiki/IBM" title="IBM">IBM</a> <a href="/wiki/IBM_System_i" title="IBM System i">iSeries</a> systems designate <a href="/wiki/Code_page" title="Code page">code page</a> <a href="/wiki/CCSID" title="CCSID">CCSID</a> 13488 for UCS-2 character encoding, CCSID 1200 for UTF-16 encoding, and CCSID 1208 for UTF-8 encoding.<sup class="reference" id="cite_ref-16"><a href="#cite_note-16">[14]</a></sup></p>
<p>UTF-16 is used by the <a href="/wiki/Binary_Runtime_Environment_for_Wireless" title="Binary Runtime Environment for Wireless">Qualcomm BREW</a> operating systems; the <a href="/wiki/.NET_Framework" title=".NET Framework">.NET</a> environments; and the <a class="mw-redirect" href="/wiki/Qt_(toolkit)" title="Qt (toolkit)">Qt</a> cross-platform graphical <a href="/wiki/Widget_toolkit" title="Widget toolkit">widget toolkit</a>.</p>
<p><a href="/wiki/Symbian" title="Symbian">Symbian OS</a> used in Nokia S60 handsets and Sony Ericsson <a href="/wiki/UIQ" title="UIQ">UIQ</a> handsets uses UCS-2. <a href="/wiki/IPhone" title="IPhone">iPhone</a> handsets use UTF-16 for <a href="/wiki/Short_Message_Service" title="Short Message Service">Short Message Service</a> instead of UCS-2 described in the <a href="/wiki/GSM_03.38" title="GSM 03.38">3GPP TS 23.038</a> (<a href="/wiki/GSM" title="GSM">GSM</a>) and <a class="new" href="/w/index.php?title=IS-637&amp;action=edit&amp;redlink=1" title="IS-637 (page does not exist)">IS-637</a> (<a href="/wiki/CDMA2000" title="CDMA2000">CDMA</a>) standards.<sup class="reference" id="cite_ref-17"><a href="#cite_note-17">[15]</a></sup></p>
<p>The <a href="/wiki/Joliet_(file_system)" title="Joliet (file system)">Joliet file system</a>, used in <a href="/wiki/CD-ROM" title="CD-ROM">CD-ROM</a> media, encodes file names using UCS-2BE (up to sixty-four Unicode characters per file name).</p>
<p>The <a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a> language environment officially only uses UCS-2 internally since version 2.0, but the UTF-8 decoder to "Unicode" produces correct UTF-16. Since Python 2.2, "wide" builds of Unicode are supported which use UTF-32 instead;<sup class="reference" id="cite_ref-18"><a href="#cite_note-18">[16]</a></sup> these are primarily used on Linux. Python 3.3 no longer ever uses UTF-16, instead an encoding that gives the most compact representation for the given string is chosen from ASCII/Latin-1, UCS-2, and UTF-32.<sup class="reference" id="cite_ref-19"><a href="#cite_note-19">[17]</a></sup></p>
<p><a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a> originally used UCS-2, and added UTF-16 supplementary character support in <a href="/wiki/Java_Platform,_Standard_Edition" title="Java Platform, Standard Edition">J2SE 5.0</a>.</p>
<p><a href="/wiki/JavaScript" title="JavaScript">JavaScript</a> implementations may use UCS-2 or UTF-16. <sup class="reference" id="cite_ref-mathiasbynens.be_20-0"><a href="#cite_note-mathiasbynens.be-20">[18]</a></sup> As of ES2015, string methods and regular expression flags have been added to the language that permit handling strings from an encoding-agnostic perspective.</p>
<p>In many languages, quoted strings need a new syntax for quoting non-BMP characters, if the language's <code>"\uXXXX"</code> syntax explicitly limits itself to 4 hex digits. The most common (used by <a href="/wiki/C%2B%2B" title="C++">C++</a>, <a href="/wiki/C_Sharp_(programming_language)" title="C Sharp (programming language)">C#</a>, <a href="/wiki/D_(programming_language)" title="D (programming language)">D</a>, and several other languages) is to use an upper-case 'U' with 8 hex digits such as <code>"\U0001D11E"</code>.<sup class="reference" id="cite_ref-21"><a href="#cite_note-21">[19]</a></sup> In Java 7 regular expressions, <a href="/wiki/International_Components_for_Unicode" title="International Components for Unicode">ICU</a>, and Perl, the syntax <code>"\x{1D11E}"</code> must be used; similarly, in <a href="/wiki/ECMAScript" title="ECMAScript">ECMAScript</a> 2015 (JavaScript), the escape format is <code>"\u{1D11E}"</code>. In many other cases (such as Java outside of regular expressions),<sup class="reference" id="cite_ref-22"><a href="#cite_note-22">[20]</a></sup> the only way to get non-BMP characters is to enter the surrogate halves individually, for example: <code>"\uD834\uDD1E"</code> for U+1D11E.</p>
<p>String implementations based on UTF-16 or UCS-2 typically return lengths and allow indexing in terms of <i>code units</i>, not code points. It’s important to emphasise that neither code points nor code units correspond to anything an end user might recognise as a “character”; the things users identify as characters may in general consist of a base code point and a sequence of combining characters (or be a sequence of code points of other kind, for example Hangul conjoining jamos) — Unicode refers to this as a <i>grapheme cluster</i><sup class="reference" id="cite_ref-23"><a href="#cite_note-23">[21]</a></sup> — and as such, applications dealing with Unicode strings, whatever the encoding, have to cope with the fact that they cannot arbitrarily split and combine strings.</p>
