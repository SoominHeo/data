<p><b>Valgrind</b> <span class="nowrap"><span class="IPA nopopups"><a href="/wiki/Help:IPA_for_English" title="Help:IPA for English">/<span style="border-bottom:1px dotted"><span title="/ˈ/ primary stress follows">ˈ</span><span title="'v' in 'vie'">v</span><span title="/æ/ short 'a' in 'bad'">æ</span><span title="'l' in 'lie'">l</span><span title="'g' in 'guy'">ɡ</span><span title="'r' in 'rye'">r</span><span title="/ɪ/ short 'i' in 'bid'">ɪ</span><span title="'n' in 'no'">n</span><span title="'d' in 'dye'">d</span></span>/</a></span></span> is a <a href="/wiki/Programming_tool" title="Programming tool">programming tool</a> for <a href="/wiki/Memory_debugger" title="Memory debugger">memory debugging</a>, <a href="/wiki/Memory_leak" title="Memory leak">memory leak</a> detection, and <a href="/wiki/Profiling_(computer_programming)" title="Profiling (computer programming)">profiling</a>.</p>
<p>Valgrind was originally designed to be a <a href="/wiki/Free_software" title="Free software">free</a> <a class="mw-redirect" href="/wiki/Memory_debugging" title="Memory debugging">memory debugging</a> tool for <a href="/wiki/Linux" title="Linux">Linux</a> on <a href="/wiki/X86" title="X86">x86</a>, but has since evolved to become a generic framework for creating dynamic analysis tools such as checkers and profilers.</p>

<h2><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Valgrind&amp;action=edit&amp;section=1" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Valgrind is in essence a <a href="/wiki/Virtual_machine" title="Virtual machine">virtual machine</a> using <a href="/wiki/Just-in-time_compilation" title="Just-in-time compilation">just-in-time</a> (JIT) compilation techniques, including <a href="/wiki/Dynamic_recompilation" title="Dynamic recompilation">dynamic recompilation</a>. Nothing from the original program ever gets run directly on the host <a href="/wiki/Central_processing_unit" title="Central processing unit">processor</a>. Instead, Valgrind first translates the program into a temporary, simpler form called Intermediate Representation (IR), which is a processor-neutral, <a href="/wiki/Static_single_assignment_form" title="Static single assignment form">SSA</a>-based form. After the conversion, a <i>tool</i> (see below) is free to do whatever transformations it would like on the IR, before Valgrind translates the IR back into machine code and lets the host processor run it. Even though it could use <a class="mw-redirect" href="/wiki/Dynamic_translation" title="Dynamic translation">dynamic translation</a> (that is, the host and target processors are from different architectures), it doesn't. Valgrind recompiles <a href="/wiki/Binary_code" title="Binary code">binary code</a> to run on host and target (or simulated) CPUs of the same architecture. It also includes a <a class="mw-redirect" href="/wiki/GDB" title="GDB">GDB</a> stub to allow debugging of the target program as it runs in Valgrind, with "monitor commands" that allow you to query the Valgrind tool for various sorts of information.</p>
<p>A considerable amount of performance is lost in these transformations (and usually, the code the tool inserts); usually, code run with Valgrind and the "none" tool (which does nothing to the IR) runs at 1/4 to 1/5 of the speed of the normal program.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup><sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></p>
<h2><span class="mw-headline" id="Tools">Tools</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Valgrind&amp;action=edit&amp;section=2" title="Edit section: Tools">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="hatnote" role="note">See also: <a href="/wiki/Memory_debugger" title="Memory debugger">Memory debugger</a> and <a href="/wiki/Profiling_(computer_programming)" title="Profiling (computer programming)">Profiling (computer programming)</a></div>
<h3><span class="mw-headline" id="Memcheck">Memcheck</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Valgrind&amp;action=edit&amp;section=3" title="Edit section: Memcheck">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>There are multiple tools included with Valgrind (and several external ones). The default (and most used) tool is <i>Memcheck</i>. Memcheck inserts extra <a href="/wiki/Instrumentation_(computer_programming)" title="Instrumentation (computer programming)">instrumentation</a> code around almost all instructions, which keeps track of the <i>validity</i> (all unallocated memory starts as invalid or "undefined", until it is initialized into a deterministic state, possibly from other memory) and <i>addressability</i> (whether the memory address in question points to an allocated, non-freed memory block), stored in the so-called <i>V bits</i> and <i>A bits</i>, respectively. As data is moved around or manipulated, the instrumentation code keeps track of the A and V bits so they are always correct on a single-bit level.</p>
<p>In addition, Memcheck replaces the standard C <a class="mw-redirect" href="/wiki/Malloc" title="Malloc">memory allocator</a> with its own implementation, which also includes <i>memory guards</i> around all allocated blocks (with the A bits set to "invalid"). This feature enables Memcheck to detect <a href="/wiki/Off-by-one_error" title="Off-by-one error">off-by-one errors</a> where a program reads or writes outside an allocated block by a small amount. The problems Memcheck can detect and warn about include the following:</p>
<ul>
<li>Use of uninitialized memory</li>
<li>Reading/writing memory after it has been <code><a class="mw-redirect" href="/wiki/Free_(programming)" title="Free (programming)">free</a></code>'d</li>
<li>Reading/writing off the end of <code><a class="mw-redirect" href="/wiki/Malloc" title="Malloc">malloc</a></code>'d blocks</li>
<li><a href="/wiki/Memory_leak" title="Memory leak">Memory leaks</a></li>
</ul>
<p>The price of this is lost performance. Programs running under Memcheck usually run from twenty to thirty times slower<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> than running outside Valgrind and use more memory (there is a memory penalty per-allocation). Thus, few developers run their code under Memcheck (or any other Valgrind tool) all the time. They most commonly use such tools either to trace down some specific bug, or to verify there are no latent bugs (of the kind Memcheck can detect) in the code.</p>
<h3><span class="mw-headline" id="Other_tools">Other tools</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Valgrind&amp;action=edit&amp;section=4" title="Edit section: Other tools">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In addition to Memcheck, Valgrind has several other tools:<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup></p>
<ul>
<li><i>None</i>, runs the code in the virtual machine without performing any analysis and thus has the smallest possible CPU and memory overhead of all tools. Since valgrind itself provides a trace back from a <a href="/wiki/Segmentation_fault" title="Segmentation fault">segmentation fault</a>, the <i>none</i> tool provides this traceback at minimal overhead.</li>
<li><i>Addrcheck</i>, similar to Memcheck but with much smaller CPU and memory overhead, thus catching fewer types of bug. Addrcheck has been removed as of version 3.2.0.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup></li>
<li><i>Massif</i>, a <a class="mw-redirect" href="/wiki/Dynamic_memory_allocation" title="Dynamic memory allocation">heap</a> <a href="/wiki/Profiling_(computer_programming)" title="Profiling (computer programming)">profiler</a>. The separate <a class="mw-redirect" href="/wiki/GUI" title="GUI">GUI</a> massif-visualizer visualizes output from Massif.</li>
<li><i>Helgrind</i> and <i>DRD</i>, detect <a href="/wiki/Race_condition" title="Race condition">race conditions</a> in <a class="mw-redirect" href="/wiki/Thread_(computer_science)" title="Thread (computer science)">multithreaded code</a></li>
<li><i>Cachegrind</i>, a <a href="/wiki/CPU_cache" title="CPU cache">cache</a> profiler. The separate GUI KCacheGrind visualizes output from Cachegrind.</li>
<li><i>Callgrind</i>, a <a class="mw-redirect" href="/wiki/Callgraph" title="Callgraph">callgraph</a> analyzer created by Josef Weidendorfer was added to Valgrind as of version 3.2.0. KCacheGrind can visualize output from Callgrind as well as Cachegrind.</li>
<li><i>exp-sgcheck</i> (named <i>exp-ptrcheck</i> prior to version 3.7), an experimental tool to find stack and global array overrun errors which Memcheck cannot find.<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> Some code results in false positives from this tool.<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup></li>
<li><i>exp-dhat</i>, dynamic heap analysis tool which analyzes how much memory is allocated and for how long as well as patterns of memory usage.</li>
<li><i>exp-bbv</i>, a performance simulator that extrapolates performance from a small sample set.</li>
</ul>
<p>There are also several externally developed tools available. One such tool is ThreadSanitizer, another detector of <a href="/wiki/Race_condition" title="Race condition">race conditions</a>.<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup><sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup></p>
<h2><span class="mw-headline" id="Platforms_supported">Platforms supported</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Valgrind&amp;action=edit&amp;section=5" title="Edit section: Platforms supported">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>As of version 3.4.0, Valgrind supports <a href="/wiki/Linux" title="Linux">Linux</a> on <a href="/wiki/X86" title="X86">x86</a>, <a href="/wiki/X86-64" title="X86-64">x86-64</a> and <a href="/wiki/PowerPC" title="PowerPC">PowerPC</a>. Support for <span class="nowrap"><a href="/wiki/MacOS" title="MacOS">OS X</a></span> was added in version 3.5.0.<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup> Support for Linux on <a href="/wiki/ARM_architecture" title="ARM architecture">ARMv7</a> (used for example in certain <a class="mw-redirect" href="/wiki/Smartphones" title="Smartphones">smartphones</a>) was added in version 3.6.0.<sup class="reference" id="cite_ref-13"><a href="#cite_note-13">[13]</a></sup> Support for <a href="/wiki/Solaris_(operating_system)" title="Solaris (operating system)">Solaris</a> was added in version 3.11.0.<sup class="reference" id="cite_ref-relnotes_2-1"><a href="#cite_note-relnotes-2">[2]</a></sup> There are unofficial ports to other UNIX-like platforms (like <a href="/wiki/FreeBSD" title="FreeBSD">FreeBSD</a>,<sup class="reference" id="cite_ref-14"><a href="#cite_note-14">[14]</a></sup> <a href="/wiki/OpenBSD" title="OpenBSD">OpenBSD</a>,<sup class="reference" id="cite_ref-15"><a href="#cite_note-15">[15]</a></sup> and <a href="/wiki/NetBSD" title="NetBSD">NetBSD</a><sup class="reference" id="cite_ref-16"><a href="#cite_note-16">[16]</a></sup>). From version 3.7.0 the <a href="/wiki/ARM_architecture" title="ARM architecture">ARM</a>/<a href="/wiki/Android_(operating_system)" title="Android (operating system)">Android</a> platform support was added.<sup class="reference" id="cite_ref-relnotes_2-2"><a href="#cite_note-relnotes-2">[2]</a></sup></p>
<p>Since version 3.9.0 there is support for Linux on <a class="mw-redirect" href="/wiki/MIPS_architecture#MIPS64" title="MIPS architecture">MIPS64</a> little and big endian, for MIPS DSP ASE on MIPS32, for <a class="mw-redirect" href="/wiki/S390x" title="S390x">s390x</a> Decimal Floating Point instructions, for <a href="/wiki/POWER8" title="POWER8">POWER8</a> (<a href="/wiki/Power_Architecture" title="Power Architecture">Power ISA 2.07</a>) instructions, for Intel <a href="/wiki/Advanced_Vector_Extensions" title="Advanced Vector Extensions">AVX2</a> instructions, for Intel Transactional Synchronization Extensions, both RTM and HLE and initial support for Hardware Transactional Memory on POWER.<sup class="reference" id="cite_ref-Valgrind_News_1-1"><a href="#cite_note-Valgrind_News-1">[1]</a></sup></p>
<h2><span class="mw-headline" id="History_and_development">History and development</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Valgrind&amp;action=edit&amp;section=6" title="Edit section: History and development">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>It is named after the main entrance to <a href="/wiki/Valhalla" title="Valhalla">Valhalla</a> in <a href="/wiki/Norse_mythology" title="Norse mythology">Norse mythology</a>.<sup class="reference" id="cite_ref-17"><a href="#cite_note-17">[17]</a></sup></p>
<p>The original author of Valgrind is <a href="/wiki/Julian_Seward" title="Julian Seward">Julian Seward</a>, who in 2006 won a <a class="mw-redirect" href="/wiki/Google-O%27Reilly_Open_Source_Award" title="Google-O'Reilly Open Source Award">Google-O'Reilly Open Source Award</a> for his work on Valgrind.<sup class="reference" id="cite_ref-18"><a href="#cite_note-18">[18]</a></sup><sup class="reference" id="cite_ref-19"><a href="#cite_note-19">[19]</a></sup></p>
<p>Several others have also made significant contributions, including Cerion Armour-Brown, Jeremy Fitzhardinge, Tom Hughes, Nicholas Nethercote, Paul Mackerras, Dirk Mueller, Bart Van Assche, Josef Weidendorfer, and Robert Walsh.<sup class="reference" id="cite_ref-20"><a href="#cite_note-20">[20]</a></sup></p>
<p>It is used by a number of Linux-based projects.<sup class="reference" id="cite_ref-21"><a href="#cite_note-21">[21]</a></sup></p>
<h2><span class="mw-headline" id="Limitations_of_Memcheck">Limitations of Memcheck</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Valgrind&amp;action=edit&amp;section=7" title="Edit section: Limitations of Memcheck">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In addition to the performance penalty, an important limitation of Memcheck is its inability to detect all cases of bounds errors in the use of static or stack-allocated data.<sup class="reference" id="cite_ref-22"><a href="#cite_note-22">[22]</a></sup> The following code will pass the <i>Memcheck</i> tool in Valgrind without incident, despite containing the errors described in the comments:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
  <span class="kt">int</span> <span class="n">Static</span><span class="p">[</span><span class="mi">5</span><span class="p">];</span>
  
  <span class="kt">int</span> <span class="nf">func</span><span class="p">(</span><span class="kt">void</span><span class="p">)</span>
  <span class="p">{</span>
    <span class="kt">int</span> <span class="n">Stack</span><span class="p">[</span><span class="mi">5</span><span class="p">];</span>
  
    <span class="n">Static</span><span class="p">[</span><span class="mi">5</span><span class="p">]</span> <span class="o">=</span> <span class="mi">0</span><span class="p">;</span>  <span class="cm">/* Error - Static[0] to Static[4] exist, Static[5] is out of bounds */</span>
    <span class="n">Stack</span> <span class="p">[</span><span class="mi">5</span><span class="p">]</span> <span class="o">=</span> <span class="mi">0</span><span class="p">;</span>  <span class="cm">/* Error - Stack[0] to Stack[4] exist, Stack[5] is out of bounds */</span>
    
    <span class="k">return</span> <span class="mi">0</span><span class="p">;</span>
  <span class="p">}</span>
</pre></div>
<p>The experimental valgrind tool exp-sgcheck has been written to address this limitation in Memcheck. It will detect array overrun errors, provided the first access to an array is within the array bounds. Note that exp-sgcheck will not detect the array overrun in the code above, since the first access to an array is out of bounds, but it will detect the array overrun error in the following code.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
  <span class="kt">void</span> <span class="nf">func</span><span class="p">(</span><span class="kt">void</span><span class="p">)</span>
  <span class="p">{</span>
    <span class="kt">int</span> <span class="n">i</span><span class="p">,</span> <span class="n">Stack</span><span class="p">[</span><span class="mi">5</span><span class="p">];</span>

    <span class="k">for</span> <span class="p">(</span><span class="n">i</span> <span class="o">=</span> <span class="mi">0</span><span class="p">;</span> <span class="n">i</span> <span class="o">&lt;=</span> <span class="mi">5</span><span class="p">;</span> <span class="n">i</span><span class="o">++</span><span class="p">)</span>
        <span class="n">Stack</span> <span class="p">[</span><span class="n">i</span><span class="p">]</span> <span class="o">=</span> <span class="mi">0</span><span class="p">;</span>        <span class="cm">/* Within bounds for i=0..4, out of bounds error when i=5 */</span>
  <span class="p">}</span>
</pre></div>
<p>The inability to detect all errors involving the access of stack allocated data is especially noteworthy since <a href="/wiki/Buffer_overflow" title="Buffer overflow">certain types of stack errors</a> make software <a href="/wiki/Vulnerability_(computing)" title="Vulnerability (computing)">vulnerable</a> to the classic <a href="/wiki/Stack_buffer_overflow" title="Stack buffer overflow">stack smashing exploit</a>.</p>
