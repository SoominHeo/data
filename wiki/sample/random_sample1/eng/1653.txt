<p><b>WebP</b> is an <a href="/wiki/Image_file_formats" title="Image file formats">image format</a> employing both <a href="/wiki/Lossy_compression" title="Lossy compression">lossy</a><sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup> and <a href="/wiki/Lossless_compression" title="Lossless compression">lossless</a> compression. It is currently developed by <a href="/wiki/Google" title="Google">Google</a>, based on technology acquired with the purchase of <a href="/wiki/On2_Technologies" title="On2 Technologies">On2 Technologies</a>.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup></p>

<div class="thumbcaption">

<a href="/wiki/Lenna" title="Lenna">Lenna</a> test image compressed using version 0.30 lossy WebP compression at quality 75 with accompanying RGB histograms. Note the breaks in the histogram of the compressed image compared to the source.</div>

<p>As a derivative of the <a href="/wiki/VP8" title="VP8">VP8</a> video format, it is a sister project to the <a href="/wiki/WebM" title="WebM">WebM</a> multimedia container format.<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> WebP-related software is released under a <a href="/wiki/BSD_licenses" title="BSD licenses">BSD license</a>.<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup></p>
<p>The format was first announced in 30 September 2010 as a new <a href="/wiki/Open_standard" title="Open standard">open standard</a> for lossily compressed true-color graphics on the web, producing smaller files of comparable image quality to the older <a href="/wiki/JPEG" title="JPEG">JPEG</a> scheme.<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup> On October 3, 2011 Google announced WebP support for animation, <a href="/wiki/ICC_profile" title="ICC profile">ICC profile</a>, <a href="/wiki/Extensible_Metadata_Platform" title="Extensible Metadata Platform">XMP</a> metadata and tiling (compositing very large images from maximum 16384×16384 tiles).<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup></p>
<p>On 18 November 2011 Google began to experiment with <a href="/wiki/Lossless_compression" title="Lossless compression">lossless compression</a> and support for <a href="/wiki/Transparency_(graphic)" title="Transparency (graphic)">transparency</a> (<a href="/wiki/Alpha_compositing" title="Alpha compositing">alpha channel</a>) in both lossless and lossy modes; support has been enabled by default in libwebp 0.2.0 (16 August 2012).<sup class="reference" id="cite_ref-lossless-announce_12-0"><a href="#cite_note-lossless-announce-12">[12]</a></sup><sup class="reference" id="cite_ref-13"><a href="#cite_note-13">[13]</a></sup> According to Google's measurements, a conversion from <a href="/wiki/Portable_Network_Graphics" title="Portable Network Graphics">PNG</a> to WebP results in a 45% reduction in file size when starting with PNGs found on the web, and a 28% reduction compared to PNGs that are recompressed with <a href="/wiki/Pngcrush" title="Pngcrush">pngcrush</a> and <a href="/wiki/PNGOUT" title="PNGOUT">PNGOUT</a>.<sup class="reference" id="cite_ref-14"><a href="#cite_note-14">[14]</a></sup></p>

<h2><span class="mw-headline" id="Animated_WebP">Animated WebP</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=WebP&amp;action=edit&amp;section=1" title="Edit section: Animated WebP">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Google has proposed using WebP for animated images as an alternative to the popular <a class="mw-redirect" href="/wiki/Graphics_Interchange_Format" title="Graphics Interchange Format">GIF</a> format, citing the advantages of 24-bit color with transparency, combining frames with lossy and lossless compression in the same animation, and as well as support for seeking to specific frames.<sup class="reference" id="cite_ref-15"><a href="#cite_note-15">[15]</a></sup> Google reports a 64% reduction in file size for images converted from animated GIFs to lossy WebP, and a 19% reduction when converted to lossless WebP.</p>

<p>WebP's lossy compression algorithm is based on the <a class="mw-redirect" href="/wiki/Intra-frame" title="Intra-frame">intra-frame</a> coding of the <a href="/wiki/VP8" title="VP8">VP8</a> video format<sup class="reference" id="cite_ref-glaser_16-0"><a href="#cite_note-glaser-16">[16]</a></sup> and the <a href="/wiki/Resource_Interchange_File_Format" title="Resource Interchange File Format">Resource Interchange File Format</a> (RIFF) as a <a class="mw-redirect" href="/wiki/Container_format_(digital)" title="Container format (digital)">container format</a>.<sup class="reference" id="cite_ref-Announcement-in-chromium_2-1"><a href="#cite_note-Announcement-in-chromium-2">[2]</a></sup> As such, it is a block-based transformation scheme with eight bits of <a href="/wiki/Color_depth" title="Color depth">color depth</a> and a <a class="mw-redirect" href="/wiki/Luminance-chrominance_model" title="Luminance-chrominance model">luminance-chrominance model</a> with <a href="/wiki/Chroma_subsampling" title="Chroma subsampling">chroma subsampling</a> by a ratio of 1:2 (<a href="/wiki/YCbCr" title="YCbCr">YCbCr</a> 4:2:0).<sup class="reference" id="cite_ref-vp8-bitstream_17-0"><a href="#cite_note-vp8-bitstream-17">[17]</a></sup> Without further content, the mandatory RIFF container has an <a class="mw-redirect" href="/wiki/Computational_overhead" title="Computational overhead">overhead</a> of only twenty bytes, though it can also hold additional <a href="/wiki/Metadata" title="Metadata">metadata</a>.<sup class="reference" id="cite_ref-Announcement-in-chromium_2-2"><a href="#cite_note-Announcement-in-chromium-2">[2]</a></sup> The side length of WebP images is limited to 16383 pixels.<sup class="reference" id="cite_ref-faq_5-1"><a href="#cite_note-faq-5">[5]</a></sup></p>
<p>WebP is based on block prediction. Each block is predicted on the values from the three blocks above it and from one block to the left of it (block decoding is done in raster-scan order: left to right and top to bottom). There are four basic modes of block prediction: horizontal, vertical, DC (one color), and TrueMotion. Mispredicted data and non-predicted blocks are compressed in a 4×4 pixel sub-block with a <a href="/wiki/Discrete_cosine_transform" title="Discrete cosine transform">discrete cosine transform</a> or a <a class="mw-redirect" href="/wiki/Walsh%E2%80%93Hadamard_transform" title="Walsh–Hadamard transform">Walsh–Hadamard transform</a>. Both transforms are done with <a href="/wiki/Fixed-point_arithmetic" title="Fixed-point arithmetic">fixed-point arithmetic</a> to avoid rounding errors. The output is compressed with <a href="/wiki/Entropy_encoding" title="Entropy encoding">entropy encoding</a>.<sup class="reference" id="cite_ref-vp8-bitstream_17-1"><a href="#cite_note-vp8-bitstream-17">[17]</a></sup> WebP also has explicit support for parallel decoding.<sup class="reference" id="cite_ref-vp8-bitstream_17-2"><a href="#cite_note-vp8-bitstream-17">[17]</a></sup></p>
<p>The <a href="/wiki/Reference_implementation" title="Reference implementation">reference implementation</a> consists of converter software in the form of a <a href="/wiki/Command-line_interface" title="Command-line interface">command-line</a> program for Linux (cwebp) and a <a href="/wiki/Library_(computing)" title="Library (computing)">programming library</a> for the decoding, the same as for WebM. The open source community quickly managed to port the converter to other platforms, such as Windows.<sup class="reference" id="cite_ref-webp-windows_18-0"><a href="#cite_note-webp-windows-18">[18]</a></sup></p>
<p>WebP’s lossless compression uses advanced techniques such as dedicated entropy codes for different color channels, exploiting 2D locality of backward reference distances and a color cache of recently used colors. This complements basic techniques such as dictionary coding, Huffman coding and color indexing transform.<sup class="reference" id="cite_ref-lossless-announce_12-1"><a href="#cite_note-lossless-announce-12">[12]</a></sup></p>
<h2><span class="mw-headline" id="Support">Support</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=WebP&amp;action=edit&amp;section=3" title="Edit section: Support">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Amongst web browsers, <a href="/wiki/Google_Chrome" title="Google Chrome">Google Chrome</a>, <a href="/wiki/Opera_(web_browser)" title="Opera (web browser)">Opera</a> and <a href="/wiki/Pale_Moon_(web_browser)" title="Pale Moon (web browser)">Pale Moon</a> natively support WebP while <a href="/wiki/Mozilla" title="Mozilla">Mozilla</a> has announced plans to implement WebP natively in <a href="/wiki/Firefox" title="Firefox">Firefox</a>.<sup class="reference" id="cite_ref-19"><a href="#cite_note-19">[19]</a></sup><sup class="reference" id="cite_ref-20"><a href="#cite_note-20">[20]</a></sup><sup class="reference" id="cite_ref-21"><a href="#cite_note-21">[21]</a></sup><sup class="reference" id="cite_ref-22"><a href="#cite_note-22">[22]</a></sup> All <a href="/wiki/WebM" title="WebM">WebM</a>-compatible browsers can also display WebP via a JavaScript <a href="/wiki/Shim_(computing)" title="Shim (computing)">shim</a>.<sup class="reference" id="cite_ref-23"><a href="#cite_note-23">[23]</a></sup><sup class="reference" id="cite_ref-24"><a href="#cite_note-24">[24]</a></sup> WebP can also be displayed in all major browsers using the WebPJS JavaScript library, although support in Internet Explorer 6 and above is achieved using <a href="/wiki/Adobe_Flash" title="Adobe Flash">Flash</a>.<sup class="reference" id="cite_ref-25"><a href="#cite_note-25">[25]</a></sup></p>
<p>Amongst graphics software, <a href="/wiki/Picasa" title="Picasa">Picasa</a> (from version 3.9),<sup class="reference" id="cite_ref-26"><a href="#cite_note-26">[26]</a></sup> <a href="/wiki/PhotoLine" title="PhotoLine">PhotoLine</a>,<sup class="reference" id="cite_ref-27"><a href="#cite_note-27">[27]</a></sup> <a href="/wiki/Pixelmator" title="Pixelmator">Pixelmator</a>,<sup class="reference" id="cite_ref-28"><a href="#cite_note-28">[28]</a></sup> <a href="/wiki/ImageMagick" title="ImageMagick">ImageMagick</a>,<sup class="reference" id="cite_ref-29"><a href="#cite_note-29">[29]</a></sup> <a href="/wiki/XnView" title="XnView">XnView</a>,<sup class="reference" id="cite_ref-30"><a href="#cite_note-30">[30]</a></sup> <a href="/wiki/IrfanView" title="IrfanView">IrfanView</a>,<sup class="reference" id="cite_ref-31"><a href="#cite_note-31">[31]</a></sup> <a href="/wiki/GDAL" title="GDAL">GDAL</a> <sup class="reference" id="cite_ref-32"><a href="#cite_note-32">[32]</a></sup> and Aseprite<sup class="reference" id="cite_ref-33"><a href="#cite_note-33">[33]</a></sup> all natively support WebP. Telegraphics has released a free plug-in that enables WebP support in <a href="/wiki/Adobe_Photoshop" title="Adobe Photoshop">Adobe Photoshop</a>.<sup class="reference" id="cite_ref-34"><a href="#cite_note-34">[34]</a></sup> The plugin was last updated in 2013 so it saves using an older revision of the WebP format. A more current beta Photoshop plugin has also been released by fnordware.<sup class="reference" id="cite_ref-35"><a href="#cite_note-35">[35]</a></sup> <a href="/wiki/GIMP" title="GIMP">GIMP</a><sup class="reference" id="cite_ref-36"><a href="#cite_note-36">[36]</a></sup> and <a href="/wiki/Paint.NET" title="Paint.NET">Paint.NET</a><sup class="reference" id="cite_ref-37"><a href="#cite_note-37">[37]</a></sup> support WebP via plugins. Google has also released a plug-in for Microsoft Windows that enables WebP support in <a href="/wiki/Windows_Photo_Viewer" title="Windows Photo Viewer">Windows Photo Viewer</a>, <a href="/wiki/Microsoft_Office_2010" title="Microsoft Office 2010">Microsoft Office 2010</a>, <a href="/wiki/FastPictureViewer" title="FastPictureViewer">FastPictureViewer</a>,<sup class="reference" id="cite_ref-38"><a href="#cite_note-38">[38]</a></sup> and any other application that uses <a href="/wiki/Windows_Imaging_Component" title="Windows Imaging Component">Windows Imaging Component</a>.<sup class="reference" id="cite_ref-WebP_WIC_Codec_39-0"><a href="#cite_note-WebP_WIC_Codec-39">[39]</a></sup></p>
<p><a href="/wiki/FFmpeg" title="FFmpeg">FFmpeg</a> linked with the VP8/VP9 reference codec library <i><a href="/wiki/Libvpx" title="Libvpx">libvpx</a></i> can extract VP8 key frames from <a href="/wiki/WebM" title="WebM">WebM</a> media and a script can then add the WebP <a href="/wiki/Resource_Interchange_File_Format" title="Resource Interchange File Format">RIFF</a> header and the NUL pad byte for odd frame lengths. Meanwhile, FFmpeg supports <i>libwebp</i> directly.</p>
<p><a href="/wiki/Gmail" title="Gmail">Gmail</a> and <a href="/wiki/Picasa_Web_Albums" title="Picasa Web Albums">Picasa Web Albums</a> (both Google <a href="/wiki/Web_application" title="Web application">web applications</a>) support WebP. Support for WebP is also planned for <a href="/wiki/Google_App_Engine" title="Google App Engine">Google App Engine</a>. The <i>Instant Previews</i> feature of <a href="/wiki/Google_Search" title="Google Search">Google Search</a> currently uses WebP internally to reduce disk space used by previews.<sup class="reference" id="cite_ref-Chromium_WebP_40-0"><a href="#cite_note-Chromium_WebP-40">[40]</a></sup> <a href="/wiki/Android_(operating_system)" title="Android (operating system)">Android</a> 4.0 supports encoding and decoding WebP images (via bitmap and <a href="/wiki/Skia_Graphics_Engine" title="Skia Graphics Engine">Skia</a>).<sup class="reference" id="cite_ref-41"><a href="#cite_note-41">[41]</a></sup> SDL_image supports the format since 1.2.11.</p>
<p><a href="/wiki/Telegram_(software)" title="Telegram (software)">Telegram Messenger</a> uses WebP for their Stickers, claiming they are displayed 5 times faster compared to the other formats usually used in messaging apps.<sup class="reference" id="cite_ref-42"><a href="#cite_note-42">[42]</a></sup></p>
<p>In early beta versions of macOS Sierra and iOS 10, Apple added WebP support.<sup class="reference" id="cite_ref-Apple_CNET_43-0"><a href="#cite_note-Apple_CNET-43">[43]</a></sup> In the GM seed version of macOS Sierra released 7 September 2016, WebP support has been removed.</p>
<h2><span class="mw-headline" id="Restrictions">Restrictions</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=WebP&amp;action=edit&amp;section=4" title="Edit section: Restrictions">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Like VP8 on which it is based, former lossy WebP only supports 8-bit YUV 4:2:0 format,<sup class="reference" id="cite_ref-44"><a href="#cite_note-44">[44]</a></sup> which may cause color loss on images with thin contrast elements (such as in <a href="/wiki/Pixel_art" title="Pixel art">pixel art</a> and <a href="/wiki/Computer_graphics_(computer_science)" title="Computer graphics (computer science)">computer graphics</a>) and ghosting in <a href="/wiki/Anaglyph_3D" title="Anaglyph 3D">anaglyph</a>. To overcome this restriction, new lossless WebP supports VP8L encoding that works exclusively with 8-bit ARGB color space.<sup class="reference" id="cite_ref-45"><a href="#cite_note-45">[45]</a></sup><sup class="reference" id="cite_ref-46"><a href="#cite_note-46">[46]</a></sup></p>
<h2><span class="mw-headline" id="Criticism">Criticism</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=WebP&amp;action=edit&amp;section=5" title="Edit section: Criticism">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In September 2010, Fiona Glaser, a developer of the <a href="/wiki/X264" title="X264">x264</a> encoder, wrote a very early critique of WebP.<sup class="reference" id="cite_ref-glaser_16-1"><a href="#cite_note-glaser-16">[16]</a></sup> Comparing different encodings (JPEG, x264, and WebP) of a reference image, she stated that the quality of the WebP-encoded result was the worst of the three, mostly because of blurriness on the image. Her main remark was that "libvpx, a much more powerful encoder than ffmpeg's jpeg encoder, loses because it tries too hard to optimize for <a href="/wiki/Peak_signal-to-noise_ratio" title="Peak signal-to-noise ratio">PSNR</a>" (peak signal-to-noise ratio), arguing instead that "good <a href="/wiki/Subjective_video_quality" title="Subjective video quality">psycho-visual optimizations</a> are more important than anything else for compression."<sup class="reference" id="cite_ref-glaser_16-2"><a href="#cite_note-glaser-16">[16]</a></sup></p>
<p>Pascal Massimino, developer of the <i>cwebp</i> encoder, reports improvements to the WebP encoder with a number of defects resolved since the preview releases. As WebP (and the WebM standard upon which it is based) is an open-source format<sup class="reference" id="cite_ref-47"><a href="#cite_note-47">[47]</a></sup> under active development, issues can be expected to both resolve and regress as they are addressed on an ad-hoc basis.</p>
<p>In October 2013, Josh Aas from Mozilla Research published a comprehensive study of current lossy encoding techniques<sup class="reference" id="cite_ref-48"><a href="#cite_note-48">[48]</a></sup> and was not able to conclude WebP outperformed JPEG by any significant margin.<sup class="reference" id="cite_ref-49"><a href="#cite_note-49">[49]</a></sup></p>
