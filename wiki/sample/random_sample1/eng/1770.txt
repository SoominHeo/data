<p>In the <a class="mw-redirect" href="/wiki/80386" title="80386">80386</a> <a href="/wiki/Microprocessor" title="Microprocessor">microprocessor</a> and later, <b>virtual 8086 mode</b> (also called <b>virtual real mode</b>, <b>V86-mode</b> or <b>VM86</b>) allows the execution of <a href="/wiki/Real_mode" title="Real mode">real mode</a> applications that are <a href="/wiki/Protected_mode#Real_mode_application_compatibility" title="Protected mode">incapable of running</a> directly in <a href="/wiki/Protected_mode" title="Protected mode">protected mode</a> while the processor is running a protected mode operating system. It is a <a href="/wiki/Hardware_virtualization" title="Hardware virtualization">hardware virtualization</a> technique that allowed multiple 8086 processors to be emulated by the 386 chip; it emerged from the painful experiences with the 80286 <a href="/wiki/Protected_mode" title="Protected mode">protected mode</a>, which by itself was not suitable enough to run concurrent real mode applications well.<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup></p>
<p>VM86 mode uses a <a href="/wiki/X86_memory_segmentation" title="X86 memory segmentation">segmentation</a> scheme identical to that of real mode (for compatibility reasons) which creates 20-bit linear addresses in the same manner as 20-bit physical addresses are created in real mode, but are subject to protected mode's memory <a href="/wiki/Paging" title="Paging">paging</a> mechanism.</p>

<h2><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Virtual_8086_mode&amp;action=edit&amp;section=1" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The virtual 8086 mode is a mode for a <a href="/wiki/Task_state_segment" title="Task state segment">protected-mode task</a>. Consequently, the processor can switch between VM86 and non-VM86 tasks, which is what enables multitasking of legacy (MS-DOS) applications.</p>
<p>To use virtual 8086 mode, an operating system sets up a virtual 8086 mode monitor, which is a program that manages the real-mode program and emulates or filters access to system hardware and software resources. The monitor must run at <a href="/wiki/Privilege_level" title="Privilege level">privilege level</a> 0 and in protected mode. Only the 8086 program runs in VM86 mode and at privilege level 3. When the real-mode program attempts to do things like access certain I/O ports to use hardware devices or access certain regions in its memory space, the CPU traps these events and calls the V86 monitor, which examines what the real mode program is trying to do and either acts as a proxy to interface with the hardware, emulates the intended function the real-mode program was trying to access, or terminates the real-mode program if it is trying to do something that cannot either be allowed or be adequately supported (such as reboot the machine, set a video display into a mode that is not supported by the hardware and is not emulated, or write over operating system code).</p>
<p>The V86 monitor can also deny permission gently by emulating the failure of a requested operation—for example, it can make a disk drive always appear not ready when in fact it has not even checked the drive but simply will not permit the real-mode program to access it. Also, the V86 monitor can do things like map memory pages, intercept calls and interrupts, and preempt the real-mode program, allowing real-mode programs to be multitasked like protected-mode programs. By intercepting the hardware and software I/O of the real-mode program and tracking the state that the V86 program expects, it can allow multiple programs to share the same hardware without interfering with each other.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[a]</a></sup> So V86 mode provides a way for real-mode programs designed for a single-tasking environment (like MS-DOS<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[b]</a></sup>) to run concurrently in a multitasking environment.</p>
<h2><span class="mw-headline" id="Usage">Usage</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Virtual_8086_mode&amp;action=edit&amp;section=2" title="Edit section: Usage">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>It is used to execute certain <a href="/wiki/DOS" title="DOS">DOS</a> programs in <a class="mw-redirect" href="/wiki/Windows/386" title="Windows/386">Windows/386</a>, <a href="/wiki/Windows_3.1x" title="Windows 3.1x">Windows 3.x</a>, <a href="/wiki/Windows_9x" title="Windows 9x">Windows 9x</a>/<a class="mw-redirect" href="/wiki/Windows_Me" title="Windows Me">Me</a>, and <a href="/wiki/OS/2" title="OS/2">OS/2</a> 2.x and later through <a href="/wiki/Virtual_DOS_machine" title="Virtual DOS machine">virtual DOS machines</a>, in <a class="mw-redirect" href="/wiki/SCO_UNIX" title="SCO UNIX">SCO UNIX</a> through <a href="/wiki/Merge_(software)" title="Merge (software)">Merge</a>, and in <a href="/wiki/Linux" title="Linux">Linux</a> through <a href="/wiki/DOSEMU" title="DOSEMU">DOSEMU</a>. (Other <a href="/wiki/DOS" title="DOS">DOS</a> programs which use protected mode execute using user mode under the emulator.) <a class="mw-redirect" href="/wiki/NTVDM" title="NTVDM">NTVDM</a> in <a class="mw-redirect" href="/wiki/X86_architecture" title="X86 architecture">x86</a> Windows NT based operating systems also use VM86 mode,<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[2]</a></sup> but with very limited direct hardware access.</p>
<h2><span class="mw-headline" id="Memory_addressing_and_interrupts">Memory addressing and interrupts</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Virtual_8086_mode&amp;action=edit&amp;section=3" title="Edit section: Memory addressing and interrupts">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The most common problem by running 8086 code from protected mode is <a class="mw-redirect" href="/wiki/Memory_addressing" title="Memory addressing">memory addressing</a> which is totally different between <a href="/wiki/Protected_mode" title="Protected mode">protected mode</a> and <a href="/wiki/Real_mode" title="Real mode">real mode</a>. As mentioned, by working under VM86 mode the <a href="/wiki/X86_memory_segmentation" title="X86 memory segmentation">segmentation</a> mechanism is reconfigured to work just like under real mode, but the <a href="/wiki/Paging" title="Paging">paging</a> mechanism is still active, and it is transparent to the real mode code; thus, <a href="/wiki/Memory_protection" title="Memory protection">memory protection</a> is still applicable, and so is the isolation of the address space.</p>
<p>When interrupts (both hardware, software and int instruction) occur, the processor switches off the VM86 mode and returns to work in full protected mode to handle the interrupt. And before servicing the interrupt, the DS, ES, FS, and GS registers are pushed on the new stack and zeroed.</p>
<h2><span class="mw-headline" id="Virtual_8086_mode_enhancements_.28VME.29"><span id="VME"></span>Virtual 8086 mode enhancements (VME)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Virtual_8086_mode&amp;action=edit&amp;section=4" title="Edit section: Virtual 8086 mode enhancements (VME)">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The Pentium architecture added a number of enhancements to the virtual 8086 mode. These were however documented by Intel only starting with the subsequent <a href="/wiki/P6_(microarchitecture)" title="P6 (microarchitecture)">P6 (microarchitecture)</a>;<sup class="reference" id="cite_ref-Shanley1998_5-0"><a href="#cite_note-Shanley1998-5">[3]</a></sup> their more recent formal name is Virtual 8086 Mode Enhancements, abbreviated VME;<sup class="reference" id="cite_ref-cpuid_6-0"><a href="#cite_note-cpuid-6">[4]</a></sup> (Older documentation may use "Extensions" as the last word in the VME acronym expansion.<sup class="reference" id="cite_ref-Shanley1998_5-1"><a href="#cite_note-Shanley1998-5">[3]</a></sup> Some later Intel 486 chips also support it.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[5]</a></sup><sup class="reference" id="cite_ref-pvi1_8-0"><a href="#cite_note-pvi1-8">[6]</a></sup>) The enhancements address mainly the 8086 virtualization overhead, with a particular focus on (virtual) interrupts.<sup class="reference" id="cite_ref-Shanley1998_5-2"><a href="#cite_note-Shanley1998-5">[3]</a></sup><sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[7]</a></sup> Before the extensions were publicly documented in the P6 documentation, the official documentation referred to the famed <a href="/wiki/Appendix_H" title="Appendix H">Appendix H</a>, which was omitted from the public documentation and shared only with selected partners under <a href="/wiki/Non-disclosure_agreement" title="Non-disclosure agreement">NDA</a>.</p>
<p>Activating VME is done by setting bit number 0 (0x1 in value) of <a href="/wiki/Control_register#CR4" title="Control register">CR4</a>. Because the VME interrupt speed-up enhancements were found useful for non-VM86 protected tasks, they can also be enabled separately by setting only bit number 1 (0x2 in value), which is called PVI (Protected Mode Virtual Interrupts).<sup class="reference" id="cite_ref-Shanley1998_5-3"><a href="#cite_note-Shanley1998-5">[3]</a></sup><sup class="reference" id="cite_ref-pvi1_8-1"><a href="#cite_note-pvi1-8">[6]</a></sup> Detecting whether a processor supports VME (including PVI) is done using the <a class="mw-redirect" href="/wiki/Cpuid" title="Cpuid">cpuid</a> instruction, by testing the value of second bit (bit number 1, 0x2 in value) in EDX register, which is set if VME is supported by the processor.<sup class="reference" id="cite_ref-cpuid_6-1"><a href="#cite_note-cpuid-6">[4]</a></sup><sup class="reference" id="cite_ref-Shanley1998_5-4"><a href="#cite_note-Shanley1998-5">[3]</a></sup> In Linux, this latter bit is reported as the <tt>vme</tt> <a class="mw-redirect" href="/wiki/CPU_flag_(x86)" title="CPU flag (x86)">flag</a> in the <tt>/proc/<a class="mw-redirect" href="/wiki/Cpuinfo" title="Cpuinfo">cpuinfo</a></tt> file, under the "flags" section.</p>
<p>In virtual 8086 mode, the basic idea is that when IOPL is less than 3, PUSHF/POPF/STI/CLI/INT/IRET instructions will treat the value of VIF in the real 32-bit EFLAGS register as the value of IF in the simulated 16-bit FLAGS register (32-bit PUSHFD/POPFD continues to GP fault). VIP will cause a GP fault on the setting of simulated IF, directing the OS to process any pending interrupts. PVI is the same idea but only affects CLI/STI instructions.</p>
<h2><span class="mw-headline" id="64-bit_and_VMX_support">64-bit and VMX support</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Virtual_8086_mode&amp;action=edit&amp;section=5" title="Edit section: 64-bit and VMX support">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Support for virtual 8086 mode is not available in <a href="/wiki/X86-64" title="X86-64">x86-64</a> <a href="/wiki/Long_mode" title="Long mode">long mode</a>, although it is still present on 64-bit capable processors running in <a href="/wiki/X86-64#Legacy_mode" title="X86-64">legacy mode</a>.</p>
<p>The addition of <a class="mw-redirect" href="/wiki/VT-x" title="VT-x">VT-x</a> has added back the ability to run virtual 8086 mode from x86-64 long mode, but it has to be done by transitioning the (physical) processor to VMX root mode and launching a logical (virtual) processor itself running in virtual 8086 mode.<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[8]</a></sup></p>
<p><a href="/wiki/Westmere_(microarchitecture)" title="Westmere (microarchitecture)">Westmere</a> and later Intel processors usually<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[9]</a></sup> can start the logical processor directly in real mode using the "unrestricted guest" feature (which itself requires <a class="mw-redirect" href="/wiki/Extended_Page_Table" title="Extended Page Table">Extended Page Tables</a>); this method removes the need to resort to the nested virtual 8086 mode simply to run the legacy BIOS for booting.<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[10]</a></sup><sup class="reference" id="cite_ref-13"><a href="#cite_note-13">[11]</a></sup></p>

<ol class="references">
<li id="cite_note-2"><span class="mw-cite-backlink"><b><a href="#cite_ref-2">^</a></b></span> <span class="reference-text">For example, if one program writes to a display, then another program gets control and writes to the same display, and then the first program gets control back, it will try to use the display as if the second program had not changed it. The V86 monitor can intercept the display writes, keep track of the display state for each program, and switch the real display between them according to which program the user has selected to interact with presently. The V86 monitor emulates independent displays for each program using only one real display.</span></li>
<li id="cite_note-3"><span class="mw-cite-backlink"><b><a href="#cite_ref-3">^</a></b></span> <span class="reference-text">MS-DOS is mentioned because it was especially the extensive library of existing MS-DOS programs that Intel had in mind when they designed V86 mode.</span></li>
