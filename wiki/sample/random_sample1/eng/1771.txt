<h1 class="firstHeading" id="firstHeading" lang="en">Virtual storage access method</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
</div>
<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><script>function mfTempOpenSection(id){var block=document.getElementById("mf-section-"+id);block.className+=" open-block";block.previousSibling.className+=" open-block";}</script><p><b>Virtual storage access method</b> (<b>VSAM</b>) is an <a href="/wiki/IBM" title="IBM">IBM</a> <a class="mw-redirect" href="/wiki/Direct_access_storage_device" title="Direct access storage device">DASD</a> file storage <a href="/wiki/Access_method" title="Access method">access method</a>, first used in the <a href="/wiki/OS/VS1" title="OS/VS1">OS/VS1</a>, OS/VS2 <a href="/wiki/OS/VS2_(SVS)" title="OS/VS2 (SVS)">Release 1</a> (SVS) and <a href="/wiki/MVS" title="MVS">Release 2</a> (MVS) operating systems, later used throughout the <a class="mw-redirect" href="/wiki/Multiple_Virtual_Storage" title="Multiple Virtual Storage">Multiple Virtual Storage</a> (MVS) architecture and now in <a href="/wiki/Z/OS" title="Z/OS">z/OS</a>. Originally a <a href="/wiki/Record-oriented_filesystem" title="Record-oriented filesystem">record-oriented filesystem</a>.<sup class="reference" id="cite_ref-notfile_2-0"><a href="#cite_note-notfile-2">[NB 2]</a></sup>, VSAM comprises four<sup class="reference" id="cite_ref-notfile_2-1"><a href="#cite_note-notfile-2">[NB 2]</a></sup> <a href="/wiki/Data_set_(IBM_mainframe)" title="Data set (IBM mainframe)">data set</a> <i>organizations</i>: <a href="/wiki/Key_Sequenced_Data_Set" title="Key Sequenced Data Set">Key Sequenced Data Set</a> (KSDS), <a href="/wiki/Relative_Record_Data_Set" title="Relative Record Data Set">Relative Record Data Set</a> (RRDS), <a href="/wiki/Entry_Sequenced_Data_Set" title="Entry Sequenced Data Set">Entry Sequenced Data Set</a> (ESDS) and <a class="new" href="/w/index.php?title=Linear_Data_Set&amp;action=edit&amp;redlink=1" title="Linear Data Set (page does not exist)">Linear Data Set</a> (LDS). The KSDS, RRDS and ESDS organizations contain records, while the LDS organization (added later to VSAM) simply contains a sequence of pages with no intrinsic record structure, for use as a <a href="/wiki/Memory-mapped_file" title="Memory-mapped file">memory-mapped file</a>.</p>
<p>IBM uses the term <i>data set</i> in official documentation as a synonym of <i>file</i>, and <i><a class="mw-redirect" href="/wiki/Direct_access_storage_device" title="Direct access storage device">Direct access storage device</a></i> (<i>DASD</i>) because it supported other devices similar to <i>disk drives</i>.</p>
<p>VSAM records can be of fixed or variable length. They are organised in fixed-size blocks called Control Intervals (CIs), and then into larger divisions called Control Areas (CAs). Control Interval sizes are measured in bytes — for example 4 <a href="/wiki/Kilobyte" title="Kilobyte">kilobytes</a> — while Control Area sizes are measured in disk tracks or cylinders. Control Intervals are the units of transfer between disk and computer so a read request will read one complete Control Interval. Control Areas are the units of allocation so, when a VSAM data set is defined, an integral number of Control Areas will be allocated.</p>
<p>The Access Method Services utility program <a href="/wiki/IBM_mainframe_utility_programs#IDCAMS" title="IBM mainframe utility programs">IDCAMS</a> is commonly used to manipulate ("delete and define") VSAM data sets.</p>
<p>Custom programs can access VSAM datasets through Data Definition (DD) statements in <a href="/wiki/Job_Control_Language" title="Job Control Language">Job Control Language</a> (JCL), via dynamic allocation or in online regions such as in <a href="/wiki/CICS" title="CICS">Customer Information Control System</a> (CICS).</p>
<p>Both <a class="mw-redirect" href="/wiki/IMS/DB" title="IMS/DB">IMS/DB</a> and <a href="/wiki/IBM_DB2" title="IBM DB2">DB2</a> are implemented on top of VSAM and use its underlying <a href="/wiki/Data_structure" title="Data structure">data structures</a>.</p>
<p></p>

<h2><span class="mw-headline" id="VSAM_files">VSAM files</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Virtual_storage_access_method&amp;action=edit&amp;section=1" title="Edit section: VSAM files">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The physical organization of VSAM data sets differs considerably from the organizations used by other access methods, as follows.</p>
<p>VSAM data sets consist of control intervals (CI) and control areas (CA). The size of the CI and CA is normally determined by the access method, and the way in which they are used is not visible to the user. There will be a fixed number of control intervals in each control area.</p>
<p>A control interval normally contains multiple records. The records are stored within the control interval starting from the low address upwards. Control information is stored at the other end of the control interval, starting from the high address and moving downwards. The space between the records and the control information is free space. The control information comprises two types of entry: a control interval descriptor field (CIDF) which is always present, and record descriptor fields (RDF) which are present when there are records within the control interval and describe the length of the associated record. Free space within a CI is always contiguous.</p>
<p>When records are inserted into a control interval, they are placed in the correct order relative to other records. This may require records to be moved out of the way inside the control interval. Conversely, when a record is deleted, later records are moved down so that the free space remains contiguous. If there is not enough free space in a control interval for a record to be inserted, the control interval is split. Roughly half the records are stored in the original control interval while the remaining records are moved into a new control interval. The new control interval is taken from a pool of free control intervals within the same control area as the original control interval. If there is no remaining free control interval within that control area, the control area itself is split and the control intervals are distributed equally between the old and the new control areas.</p>
<p>You can use three types of record-orientated file organization with VSAM (the contents of linear data sets have no record structure):</p>
<h3><span class="mw-headline" id="Sequential_VSAM_organization">Sequential VSAM organization</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Virtual_storage_access_method&amp;action=edit&amp;section=2" title="Edit section: Sequential VSAM organization">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Further information: <a href="/wiki/Entry_Sequenced_Data_Set" title="Entry Sequenced Data Set">Entry Sequenced Data Set</a></div>
<p>An ESDS may have an index defined to it to enable access via keys, by defining an Alternate Index.Records in ESDS are stored in order in which they are written by address access.Records are loaded irrespective of their contents and their byte addresses cannot be changed.</p>
<h3><span class="mw-headline" id="Indexed_VSAM_organization">Indexed VSAM organization</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Virtual_storage_access_method&amp;action=edit&amp;section=3" title="Edit section: Indexed VSAM organization">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Further information: <a href="/wiki/Key_Sequenced_Data_Set" title="Key Sequenced Data Set">Key Sequenced Data Set</a></div>
<p>A KSDS has two parts: the index component and the data component. These may be stored on separate disk volumes.</p>
<p>While a basic KSDS only has one key (the primary key), alternate indices may be defined to permit the use of additional fields as secondary keys. An Alternate Index (AIX) is itself a KSDS.</p>
<p>The data structure used by a KSDS is nowadays known as a <a href="/wiki/B%2B_tree" title="B+ tree">B+ tree</a>.</p>
<h3><span class="mw-headline" id="Relative_VSAM_organization">Relative VSAM organization</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Virtual_storage_access_method&amp;action=edit&amp;section=4" title="Edit section: Relative VSAM organization">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Further information: <a href="/wiki/Relative_Record_Data_Set" title="Relative Record Data Set">Relative Record Data Set</a></div>
<p>An RRDS may have an index defined to it to enable access via keys, by defining an Alternate Index.</p>
<h3><span class="mw-headline" id="Linear_VSAM_organization">Linear VSAM organization</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Virtual_storage_access_method&amp;action=edit&amp;section=5" title="Edit section: Linear VSAM organization">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Further information: <a class="new" href="/w/index.php?title=Linear_Data_Set&amp;action=edit&amp;redlink=1" title="Linear Data Set (page does not exist)">Linear Data Set</a></div>
<p>An LDS is an unstructured VSAM dataset with a control interval size of a multiple of 4K. It is used by certain system services.</p>
<h2><span class="mw-headline" id="VSAM_Data_Access_Techniques">VSAM Data Access Techniques</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Virtual_storage_access_method&amp;action=edit&amp;section=6" title="Edit section: VSAM Data Access Techniques">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>There are four types of access techniques for VSAM data:</p>
<ul>
<li>Local Shared Resources (LSR), is optimised for "random" or direct access. LSR access is easy to achieve from <a href="/wiki/CICS" title="CICS">CICS</a>.</li>
<li>Global Shared Resources (GSR)</li>
<li>Non-Shared Resources (NSR), which is optimised for sequential access. NSR access has historically been easier to use than LSR for batch programs.</li>
<li>Distributed File Management (DFM), an implementation of a <a href="/wiki/Distributed_Data_Management_Architecture" title="Distributed Data Management Architecture">Distributed Data Management Architecture</a> server, enables programs on remote computers to create, manage, and access VSAM files.</li>
</ul>
<h2><span class="mw-headline" id="Sharing_VSAM_data">Sharing VSAM data</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Virtual_storage_access_method&amp;action=edit&amp;section=7" title="Edit section: Sharing VSAM data">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Sharing of VSAM data between CICS regions can be done by VSAM Record-Level Sharing (RLS). This adds record caching and, more importantly, record locking. Logging and commit processing remain the responsibility of CICS which means that sharing of VSAM data outside a CICS environment is severely restricted.</p>
<p>Sharing between CICS regions and batch jobs requires <a class="new" href="/w/index.php?title=Transactional_VSAM&amp;action=edit&amp;redlink=1" title="Transactional VSAM (page does not exist)">Transactional VSAM</a>, DFSMStvs. This is an optional program that builds on VSAM RLS by adding logging and two-phase commit, using underlying z/OS system services. This permits generalised sharing of VSAM data.</p>
<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Virtual_storage_access_method&amp;action=edit&amp;section=8" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>VSAM was introduced as a replacement for older access methods and was intended to add function, to be easier to use and to overcome problems of performance and device-dependence. VSAM was introduced in the 1970s when IBM announced <a href="/wiki/Virtual_memory" title="Virtual memory">virtual storage</a> operating systems (DOS/VS, OS/VS1 and OS/VS2) for its new System/370 series, as successors of the <a href="/wiki/DOS/360_and_successors" title="DOS/360 and successors">DOS/360</a> and <a href="/wiki/OS/360_and_successors" title="OS/360 and successors">OS/360</a> operating systems running on its System/360 computer series. While backwards compatibility was maintained, the older access methods suffered from performance problems due to the address translation required for virtual storage.</p>
<p>The KSDS organization was designed to replace <a href="/wiki/ISAM" title="ISAM">ISAM</a>, the Indexed Sequential Access Method. Changes in disk technology had meant that searching for data in ISAM data sets had become very inefficient. It was also difficult to move ISAM data sets as there were embedded pointers to physical disk locations which became invalid if the data set was moved. IBM also provided a compatibility interface to allow programs coded to use ISAM to use a KSDS instead.</p>
<p>The RRDS organization was designed to replace <a class="mw-redirect" href="/wiki/BDAM" title="BDAM">BDAM</a>, the Basic Direct Access Method. In some cases, BDAM data sets contained embedded pointers which prevented them from being moved. However, most BDAM data sets did not and the incentive to move from BDAM to VSAM RRDS was much less compelling than that to move from ISAM to VSAM KSDS.</p>
<p>Linear data sets were added later, followed by VSAM RLS and then Transactional VSAM.</p>
