<p>In <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a>, the <b>open/closed principle</b> states "<i>software entities (classes, modules, functions, etc.) should be open for extension, but closed for modification</i>";<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> that is, such an entity can allow its behaviour to be extended without modifying its <a href="/wiki/Source_code" title="Source code">source code</a>.</p>
<p>The name <i>open/closed principle</i> has been used in two ways. Both ways use <a class="mw-redirect" href="/wiki/Inheritance_(computer_science)" title="Inheritance (computer science)">inheritance</a> to resolve the apparent dilemma, but the goals, techniques, and results are different.</p>
<p></p>

<h2><span class="mw-headline" id="Meyer.27s_open.2Fclosed_principle">Meyer's open/closed principle</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Open/closed_principle&amp;action=edit&amp;section=1" title="Edit section: Meyer's open/closed principle">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p><a href="/wiki/Bertrand_Meyer" title="Bertrand Meyer">Bertrand Meyer</a> is generally credited for having originated the term <i>open/closed principle</i>,<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> which appeared in his 1988 book <cite><i><a href="/wiki/Object-Oriented_Software_Construction" title="Object-Oriented Software Construction">Object Oriented Software Construction</a></i></cite>.</p>
<blockquote>
<ul>
<li>A module will be said to be open if it is still available for extension. For example, it should be possible to add fields to the data structures it contains, or new elements to the set of functions it performs.</li>
<li>A module will be said to be closed if it is available for use by other modules. This assumes that the module has been given a well-defined, stable description (the interface in the sense of information hiding).<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup></li>
</ul>
</blockquote>
<p>At the time Meyer was writing, adding fields or functions to a library inevitably required changes to any programs depending on that library. Meyer's proposed solution to this dilemma relied on the notion of object-oriented <a class="mw-redirect" href="/wiki/Inheritance_(computer_science)" title="Inheritance (computer science)">inheritance</a> (specifically <a class="mw-redirect" href="/wiki/Implementation_inheritance" title="Implementation inheritance">implementation inheritance</a>):</p>
<blockquote>
<p>A class is closed, since it may be compiled, stored in a library, baselined, and used by client classes. But it is also open, since any new class may use it as parent, adding new features. When a descendant class is defined, there is no need to change the original or to disturb its clients.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></p>
</blockquote>
<h2><span class="mw-headline" id="Polymorphic_open.2Fclosed_principle">Polymorphic open/closed principle</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Open/closed_principle&amp;action=edit&amp;section=2" title="Edit section: Polymorphic open/closed principle">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>During the 1990s, the open/closed principle became popularly redefined to refer to the use of <a class="mw-redirect" href="/wiki/Abstraction_(computer_science)" title="Abstraction (computer science)">abstracted</a> interfaces, where the implementations can be changed and multiple implementations could be created and <a href="/wiki/Polymorphism_(computer_science)" title="Polymorphism (computer science)">polymorphically</a> substituted for each other.</p>
<p>In contrast to Meyer's usage, this definition advocates inheritance from <a class="mw-redirect" href="/wiki/Abstract_base_class" title="Abstract base class">abstract base classes</a>. Interface specifications can be reused through inheritance but implementation need not be. The existing interface is closed to modifications and new implementations must, at a minimum, implement that interface.</p>
<p><a class="mw-redirect" href="/wiki/Robert_C._Martin" title="Robert C. Martin">Robert C. Martin</a>'s 1996 article "The Open-Closed Principle"<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> was one of the seminal writings to take this approach. In 2001 <a href="/wiki/Craig_Larman" title="Craig Larman">Craig Larman</a> related the open/closed principle to the pattern by <a href="/wiki/Alistair_Cockburn" title="Alistair Cockburn">Alistair Cockburn</a> called <i>Protected Variations</i>, and to the <a href="/wiki/David_Parnas" title="David Parnas">David Parnas</a> discussion of <i><a href="/wiki/Information_hiding" title="Information hiding">information hiding</a></i>.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup></p>
