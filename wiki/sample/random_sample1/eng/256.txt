<h1 class="firstHeading" id="firstHeading" lang="en">Static library</h1>

<p>In <a href="/wiki/Computer_science" title="Computer science">computer science</a>, a <b>static library</b> or <b>statically-linked library</b> is a set of routines, external functions and variables which are resolved in a caller at <a class="mw-redirect" href="/wiki/Compile-time" title="Compile-time">compile-time</a> and copied into a target application by a <a href="/wiki/Compiler" title="Compiler">compiler</a>, <a href="/wiki/Linker_(computing)" title="Linker (computing)">linker</a>, or binder, producing an <a href="/wiki/Object_file" title="Object file">object file</a> and a <a href="/wiki/Standalone_program" title="Standalone program">stand-alone</a> <a href="/wiki/Executable" title="Executable">executable</a>.<sup class="reference" id="cite_ref-Static_Libraries_1-0"><a href="#cite_note-Static_Libraries-1">[1]</a></sup> This executable and the process of compiling it are both known as a <a href="/wiki/Static_build" title="Static build">static build</a> of the program. Historically, libraries could only be <i>static</i>. Static libraries are either merged with other static libraries and <a href="/wiki/Object_file" title="Object file">object files</a> during building/linking to form a single <a href="/wiki/Executable" title="Executable">executable</a>, or they may be loaded at <a href="/wiki/Run_time_(program_lifecycle_phase)" title="Run time (program lifecycle phase)">run-time</a> into the <a href="/wiki/Address_space" title="Address space">address space</a> of the loaded executable at a <a class="mw-redirect" href="/wiki/Static_memory_allocation" title="Static memory allocation">static memory offset</a> determined at compile-time/link-time.</p>

<h2><span class="mw-headline" id="Advantages_and_disadvantages">Advantages and disadvantages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Static_library&amp;action=edit&amp;section=1" title="Edit section: Advantages and disadvantages">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>There are several advantages to statically linking libraries with an executable instead of <a class="mw-redirect" href="/wiki/Dynamic_linking" title="Dynamic linking">dynamically linking</a> them. The most significant is that the application can be certain that all its libraries are present and that they are the correct version. This avoids dependency problems, known colloquially as <a href="/wiki/DLL_Hell" title="DLL Hell">DLL Hell</a> or more generally <a href="/wiki/Dependency_hell" title="Dependency hell">dependency hell</a>. Static linking can also allow the application to be contained in a single executable file, simplifying distribution and installation.</p>
<p>With static linking, it is enough to include those parts of the library that are directly and indirectly referenced by the target executable (or target library). With dynamic libraries, the entire library is loaded, as it is not known in advance which functions will be invoked by applications. Whether this advantage is significant in practice depends on the structure of the library.</p>
<p>In static linking, the size of the executable becomes greater than in dynamic linking, as the library code is stored <i>within the executable</i> rather than in separate files. But if library files are counted as part of the application then the total size will be similar, or even smaller if the compiler eliminates the unused symbols. On <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a> it is common to include the library files an application needs with the application.<sup class="reference" id="cite_ref-endofdllhell_2-0"><a href="#cite_note-endofdllhell-2">[2]</a></sup> On <a href="/wiki/Unix-like" title="Unix-like">Unix-like</a> systems this is less common as <a class="mw-redirect" href="/wiki/Package_management_system" title="Package management system">package management systems</a> can be used to ensure the correct library files are available. This allows the library files to be shared between many applications leading to space savings. It also allows the library to be updated to fix bugs and security flaws without updating the applications that use the library. In practice, many executables (especially those targeting Microsoft Windows) use both static and dynamic libraries.</p>
<h2><span class="mw-headline" id="Linking_and_loading">Linking and loading</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Static_library&amp;action=edit&amp;section=2" title="Edit section: Linking and loading">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Any static library function can call a function or procedure in another static library. The <a href="/wiki/Linker_(computing)" title="Linker (computing)">linker</a> and loader handle this the same way as for kinds of other <a href="/wiki/Object_file" title="Object file">object files</a>. Static library files may be linked at <a href="/wiki/Run_time_(program_lifecycle_phase)" title="Run time (program lifecycle phase)">run time</a> by a <a class="mw-redirect" href="/wiki/Linking_loader" title="Linking loader">linking loader</a> (e.g., the <a href="/wiki/X_Window_System" title="X Window System">X11</a> module loader). However, whether such a process can be called <i>static linking</i> is controversial.</p>
<h2><span class="mw-headline" id="Creating_static_libraries_in_C.2FC.2B.2B">Creating static libraries in C/C++</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Static_library&amp;action=edit&amp;section=3" title="Edit section: Creating static libraries in C/C++">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Static libraries can be easily created in <a href="/wiki/C_(programming_language)" title="C (programming language)">C</a> or in <a href="/wiki/C%2B%2B" title="C++">C++</a>. These two languages provide <a class="mw-redirect" href="/wiki/Storage_classes_in_C" title="Storage classes in C">storage-class specifiers</a> for indicating external or internal linkage, in addition to providing other features. To create such a library, the exported functions/procedures and other objects variables must be specified for <i><a class="mw-redirect" href="/wiki/External_linkage" title="External linkage">external linkage</a></i> (i.e. by not using the C <a href="/wiki/Static_(keyword)" title="Static (keyword)"><code>static</code> keyword</a>). Static library filenames usually have a ".a" extension on <a href="/wiki/Unix-like" title="Unix-like">Unix-like</a> systems<sup class="reference" id="cite_ref-Static_Libraries_1-1"><a href="#cite_note-Static_Libraries-1">[1]</a></sup> and ".lib" on <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a>.</p>
<p>For example, to create an archive from files <tt>class1.o</tt>, <tt>class2.o</tt>, <tt>class3.o</tt>, the following command would be used:<sup class="reference" id="cite_ref-Static_Libraries_1-2"><a href="#cite_note-Static_Libraries-1">[1]</a></sup></p>
<pre>
ar rcs libclass.a class1.o class2.o class3.o
</pre>
<p>to compile a program that depends on <tt>class1.o</tt>, <tt>class2.o</tt>, and <tt>class3.o</tt> one could do:</p>
<pre>
cc main.c libclass.a
</pre>
<p>or (if libclass.a is placed in standard library path, like <tt>/usr/local/lib</tt>)</p>
<pre>
cc main.c -lclass
</pre>
<p>or (during linking)</p>
<pre>
ld ... main.o -lclass ...
</pre>
<p>instead of:</p>
<pre>
cc main.c class1.o class2.o class3.o
</pre>
