<p><b>Git</b> (<span class="nowrap"><span class="IPA nopopups"><a href="/wiki/Help:IPA_for_English" title="Help:IPA for English">/<span style="border-bottom:1px dotted"><span title="'g' in 'guy'">ɡ</span><span title="/ɪ/ short 'i' in 'bid'">ɪ</span><span title="'t' in 'tie'">t</span></span>/</a></span></span><sup class="reference" id="cite_ref-:0_6-0"><a href="#cite_note-:0-6">[6]</a></sup>) is a <a class="mw-redirect" href="/wiki/Version_control_system" title="Version control system">version control system</a> (VCS) for tracking changes in <a href="/wiki/Computer_file" title="Computer file">computer files</a> and coordinating work on those files among multiple people. It is primarily used for <a href="/wiki/Software_development" title="Software development">software development</a>,<sup class="reference" id="cite_ref-effcomp_7-0"><a href="#cite_note-effcomp-7">[7]</a></sup> but it can be used to keep track of changes in any files. As a <a class="mw-redirect" href="/wiki/Distributed_revision_control" title="Distributed revision control">distributed revision control</a> system it is aimed at speed,<sup class="reference" id="cite_ref-kernel_SCM_saga_8-0"><a href="#cite_note-kernel_SCM_saga-8">[8]</a></sup> data integrity,<sup class="reference" id="cite_ref-integrity_goals_9-0"><a href="#cite_note-integrity_goals-9">[9]</a></sup> and support for distributed, non-linear workflows.<sup class="reference" id="cite_ref-linusGoogleTalk_10-0"><a href="#cite_note-linusGoogleTalk-10">[10]</a></sup></p>
<p>Git was created by <a href="/wiki/Linus_Torvalds" title="Linus Torvalds">Linus Torvalds</a> in 2005 for development of the <a href="/wiki/Linux_kernel" title="Linux kernel">Linux kernel</a>, with other kernel developers contributing to its initial development.<sup class="reference" id="cite_ref-pro-git-1.2_11-0"><a href="#cite_note-pro-git-1.2-11">[11]</a></sup> Its current maintainer is <a href="/wiki/Junio_Hamano" title="Junio Hamano">Junio Hamano</a>.</p>
<p>As with most other distributed version control systems, and unlike most <a class="mw-redirect" href="/wiki/Client%E2%80%93server" title="Client–server">client–server</a> systems, every Git <a href="/wiki/Directory_(computing)" title="Directory (computing)">directory</a> on every <a href="/wiki/Node_(networking)" title="Node (networking)">computer</a> is a full-fledged <a href="/wiki/Repository_(version_control)" title="Repository (version control)">repository</a> with complete history and full version tracking abilities, independent of network access or a central server.<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup></p>
<p>Like the Linux kernel, Git is <a href="/wiki/Free_software" title="Free software">free software</a> distributed under the terms of the <a href="/wiki/GNU_General_Public_License" title="GNU General Public License">GNU General Public License</a> version 2.</p>

<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Git&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Git development began in April 2005, after many developers of the <a href="/wiki/Linux_kernel" title="Linux kernel">Linux kernel</a> gave up access to <a href="/wiki/BitKeeper" title="BitKeeper">BitKeeper</a>, a proprietary <a class="mw-redirect" href="/wiki/Source_control_management" title="Source control management">source control management</a> (SCM) system that they had formerly used to maintain the project.<sup class="reference" id="cite_ref-linux.com44147_13-0"><a href="#cite_note-linux.com44147-13">[13]</a></sup> The copyright holder of BitKeeper, <a href="/wiki/Larry_McVoy" title="Larry McVoy">Larry McVoy</a>, had withdrawn free use of the product after claiming that <a href="/wiki/Andrew_Tridgell" title="Andrew Tridgell">Andrew Tridgell</a> had <a href="/wiki/Reverse_engineering" title="Reverse engineering">reverse-engineered</a> the BitKeeper protocols.<sup class="reference" id="cite_ref-infoworld_bit_keeper_14-0"><a href="#cite_note-infoworld_bit_keeper-14">[14]</a></sup></p>
<p><a href="/wiki/Linus_Torvalds" title="Linus Torvalds">Linus Torvalds</a> wanted a distributed system that he could use like BitKeeper, but none of the available free systems met his needs, especially for performance. Torvalds cited an example of a source-control management system needing 30 seconds to apply a patch and update all associated metadata, and noted that this would not scale to the needs of Linux kernel development, where syncing with fellow maintainers could require 250 such actions at once. For his design criteria, he specified that patching should take no more than three seconds,<sup class="reference" id="cite_ref-kernel_SCM_saga_8-1"><a href="#cite_note-kernel_SCM_saga-8">[8]</a></sup> and added three more points:</p>
<ul>
<li>Take <a href="/wiki/Concurrent_Versions_System" title="Concurrent Versions System">Concurrent Versions System</a> (CVS) as an example of what <i>not</i> to do; if in doubt, make the exact opposite decision<sup class="reference" id="cite_ref-linusGoogleTalk_10-1"><a href="#cite_note-linusGoogleTalk-10">[10]</a></sup></li>
<li>Support a distributed, BitKeeper-like workflow<sup class="reference" id="cite_ref-linusGoogleTalk_10-2"><a href="#cite_note-linusGoogleTalk-10">[10]</a></sup></li>
<li>Include very strong safeguards against corruption, either accidental or malicious<sup class="reference" id="cite_ref-integrity_goals_9-1"><a href="#cite_note-integrity_goals-9">[9]</a></sup></li>
</ul>
<p>These criteria eliminated every then-extant <a class="mw-redirect" href="/wiki/Version_control_system" title="Version control system">version control system</a> except <a href="/wiki/Monotone_(software)" title="Monotone (software)">Monotone</a>. Performance considerations excluded it, too.<sup class="reference" id="cite_ref-linusGoogleTalk_10-3"><a href="#cite_note-linusGoogleTalk-10">[10]</a></sup> So immediately after the 2.6.12-rc2 Linux kernel development release, Torvalds set out to write his own system.<sup class="reference" id="cite_ref-linusGoogleTalk_10-4"><a href="#cite_note-linusGoogleTalk-10">[10]</a></sup></p>
<p>Torvalds quipped about the name <i>git</i> (which means <i>unpleasant person</i> in <a href="/wiki/British_English" title="British English">British English</a> slang): "I'm an egotistical bastard, and I name all my projects after myself. First '<a href="/wiki/Linux_kernel" title="Linux kernel">Linux</a>', now 'git'."<sup class="reference" id="cite_ref-whythegitname_15-0"><a href="#cite_note-whythegitname-15">[15]</a></sup><sup class="reference" id="cite_ref-16"><a href="#cite_note-16">[16]</a></sup> The <a href="/wiki/Man_page" title="Man page">man page</a> describes Git as "the stupid content tracker".<sup class="reference" id="cite_ref-17"><a href="#cite_note-17">[17]</a></sup> The readme file of the source code elaborates further:<sup class="reference" id="cite_ref-18"><a href="#cite_note-18">[18]</a></sup></p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
The name "git" was given by Linus Torvalds when he wrote the very
first version. He described the tool as "the stupid content tracker"
and the name as (depending on your way):

 - random three-letter combination that is pronounceable, and not
   actually used by any common UNIX command.  The fact that it is a
   mispronunciation of "get" may or may not be relevant.
 - stupid. contemptible and despicable. simple. Take your pick from the
   dictionary of slang.
 - "global information tracker": you're in a good mood, and it actually
   works for you. Angels sing, and a light suddenly fills the room.
 - "g*dd*mn idiotic truckload of sh*t": when it breaks
</pre></div>
<p>The development of Git began on 3 April 2005.<sup class="reference" id="cite_ref-selfhost_19-0"><a href="#cite_note-selfhost-19">[19]</a></sup> Torvalds announced the project on 6 April;<sup class="reference" id="cite_ref-20"><a href="#cite_note-20">[20]</a></sup> it became <a href="/wiki/Self-hosting" title="Self-hosting">self-hosting</a> as of 7 April.<sup class="reference" id="cite_ref-selfhost_19-1"><a href="#cite_note-selfhost-19">[19]</a></sup> The first merge of multiple branches took place on 18 April.<sup class="reference" id="cite_ref-21"><a href="#cite_note-21">[21]</a></sup> Torvalds achieved his performance goals; on 29 April, the nascent Git was benchmarked recording patches to the Linux kernel tree at the rate of 6.7 per second.<sup class="reference" id="cite_ref-22"><a href="#cite_note-22">[22]</a></sup> On 16 June Git managed the kernel 2.6.12 release.<sup class="reference" id="cite_ref-23"><a href="#cite_note-23">[23]</a></sup></p>
<p>Torvalds turned over <a href="/wiki/Software_maintenance" title="Software maintenance">maintenance</a> on 26 July 2005 to <a href="/wiki/Junio_Hamano" title="Junio Hamano">Junio Hamano</a>, a major contributor to the project.<sup class="reference" id="cite_ref-24"><a href="#cite_note-24">[24]</a></sup> Hamano was responsible for the 1.0 release on 21 December 2005, and remains the project's maintainer.<sup class="reference" id="cite_ref-25"><a href="#cite_note-25">[25]</a></sup></p>

<h2><span class="mw-headline" id="Design">Design</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Git&amp;action=edit&amp;section=2" title="Edit section: Design">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Git's design was inspired by <a href="/wiki/BitKeeper" title="BitKeeper">BitKeeper</a> and <a href="/wiki/Monotone_(software)" title="Monotone (software)">Monotone</a>.<sup class="reference" id="cite_ref-26"><a href="#cite_note-26">[26]</a></sup><sup class="reference" id="cite_ref-a_quick_challenge_27-0"><a href="#cite_note-a_quick_challenge-27">[27]</a></sup> Git was originally designed as a low-level version control system engine on top of which others could write front ends, such as <a href="/wiki/Cogito_(software)" title="Cogito (software)">Cogito</a> or <a class="new" href="/w/index.php?title=StGIT&amp;action=edit&amp;redlink=1" title="StGIT (page does not exist)">StGIT</a>.<sup class="reference" id="cite_ref-a_quick_challenge_27-1"><a href="#cite_note-a_quick_challenge-27">[27]</a></sup> The core Git project has since become a complete version control system that is usable directly.<sup class="reference" id="cite_ref-bare_url_28-0"><a href="#cite_note-bare_url-28">[28]</a></sup> While strongly influenced by BitKeeper, Torvalds deliberately avoided conventional approaches, leading to a unique design.<sup class="reference" id="cite_ref-29"><a href="#cite_note-29">[29]</a></sup></p>
<h3><span class="mw-headline" id="Characteristics">Characteristics</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Git&amp;action=edit&amp;section=3" title="Edit section: Characteristics">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Git's design is a synthesis of Torvalds's experience with Linux in maintaining a large distributed development project, along with his intimate knowledge of file system performance gained from the same project and the urgent need to produce a working system in short order. These influences led to the following implementation choices:</p>
<dl>
<dt>Strong support for non-linear development</dt>
<dd>Git supports rapid branching and merging, and includes specific tools for visualizing and navigating a non-linear development history. In Git, a core assumption is that a change will be merged more often than it is written, as it is passed around to various reviewers. In Git, branches are very lightweight: a branch is only a reference to one commit. With its parental commits, the full branch structure can be constructed.</dd>
<dt>Distributed development</dt>
<dd>Like <a href="/wiki/Darcs" title="Darcs">Darcs</a>, <a href="/wiki/BitKeeper" title="BitKeeper">BitKeeper</a>, <a href="/wiki/Mercurial" title="Mercurial">Mercurial</a>, <a href="/wiki/SVK" title="SVK">SVK</a>, <a class="mw-redirect" href="/wiki/Bazaar_(software)" title="Bazaar (software)">Bazaar</a>, and <a href="/wiki/Monotone_(software)" title="Monotone (software)">Monotone</a>, Git gives each developer a local copy of the full development history, and changes are copied from one such repository to another. These changes are imported as added development branches, and can be merged in the same way as a locally developed branch.</dd>
<dt>Compatibility with extant systems and protocols</dt>
<dd>Repositories can be published via <a href="/wiki/Hypertext_Transfer_Protocol" title="Hypertext Transfer Protocol">Hypertext Transfer Protocol</a> (HTTP), <a href="/wiki/File_Transfer_Protocol" title="File Transfer Protocol">File Transfer Protocol</a> (FTP), <a href="/wiki/Rsync" title="Rsync">rsync</a> (until Git 2.8.0<sup class="reference" id="cite_ref-30"><a href="#cite_note-30">[30]</a></sup>), or a Git protocol over either a plain socket, or <a href="/wiki/Secure_Shell" title="Secure Shell">Secure Shell</a> (ssh). Git also has a CVS server emulation, which enables the use of extant CVS clients and IDE plugins to access Git repositories. <a href="/wiki/Apache_Subversion" title="Apache Subversion">Subversion</a> and <a class="mw-redirect" href="/wiki/Svk" title="Svk">svk</a> repositories can be used directly with git-svn.</dd>
<dt>Efficient handling of large projects</dt>
<dd>Torvalds has described Git as being very fast and scalable,<sup class="reference" id="cite_ref-31"><a href="#cite_note-31">[31]</a></sup> and performance tests done by Mozilla<sup class="reference" id="cite_ref-32"><a href="#cite_note-32">[32]</a></sup> showed it was an <a href="/wiki/Order_of_magnitude" title="Order of magnitude">order of magnitude</a> faster than some version control systems, and fetching version history from a locally stored repository can be one hundred times faster than fetching it from the remote server.<sup class="reference" id="cite_ref-33"><a href="#cite_note-33">[33]</a></sup></dd>
<dt>Cryptographic authentication of history</dt>
<dd>The Git history is stored in such a way that the ID of a particular version (a <i>commit</i> in Git terms) depends upon the complete development history leading up to that commit. Once it is published, it is not possible to change the old versions without it being noticed. The structure is similar to a <a href="/wiki/Merkle_tree" title="Merkle tree">Merkle tree</a>, but with added data at the nodes and leaves.<sup class="reference" id="cite_ref-34"><a href="#cite_note-34">[34]</a></sup> (<a href="/wiki/Mercurial" title="Mercurial">Mercurial</a> and <a href="/wiki/Monotone_(software)" title="Monotone (software)">Monotone</a> also have this property.)</dd>
<dt>Toolkit-based design</dt>
<dd>Git was designed as a set of programs written in <a href="/wiki/C_(programming_language)" title="C (programming language)">C</a>, and several shell scripts that provide wrappers around those programs.<sup class="reference" id="cite_ref-35"><a href="#cite_note-35">[35]</a></sup> Although most of those scripts have since been rewritten in C for speed and portability, the design remains, and it is easy to chain the components together.<sup class="reference" id="cite_ref-36"><a href="#cite_note-36">[36]</a></sup></dd>
<dt>Pluggable merge strategies</dt>
<dd>As part of its toolkit design, Git has a well-defined model of an incomplete merge, and it has multiple algorithms for completing it, culminating in telling the user that it is unable to complete the merge automatically and that manual editing is needed.</dd>
<dt><a href="/wiki/Garbage_(computer_science)" title="Garbage (computer science)">Garbage</a> accumulates until collected</dt>
<dd>Aborting operations or backing out changes will leave useless dangling objects in the database. These are generally a small fraction of the continuously growing history of wanted objects. Git will automatically perform <a href="/wiki/Garbage_collection_(computer_science)" title="Garbage collection (computer science)">garbage collection</a> when enough loose objects have been created in the repository. Garbage collection can be called explicitly using <code>git gc --prune</code>.<sup class="reference" id="cite_ref-37"><a href="#cite_note-37">[37]</a></sup></dd>
<dt>Periodic explicit object packing</dt>
<dd>Git stores each newly created object as a separate file. Although individually compressed, this takes a great deal of space and is inefficient. This is solved by the use of <i>packs</i> that store a large number of objects in one file (or network byte stream) called <i>packfile</i>, <a href="/wiki/Delta_encoding" title="Delta encoding">delta-compressed</a> among themselves. Packs are compressed using the <a href="/wiki/Heuristic_(computer_science)" title="Heuristic (computer science)">heuristic</a> that files with the same name are probably similar, but do not depend on it for correctness. A corresponding index file is created for each packfile, telling the offset of each object in the packfile. Newly created objects (newly added history) are still stored singly, and periodic repacking is needed to maintain space efficiency. The process of packing the repository can be very computationally costly. By allowing objects to exist in the repository in a loose, but quickly generated format, Git allows the costly pack operation to be deferred until later, when time matters less, e.g., the end of a work day. Git does periodic repacking automatically but manual repacking is also possible with the <tt>git gc</tt> command. For data integrity, both packfile and its index have <a href="/wiki/SHA-1" title="SHA-1">SHA-1</a> checksum inside, and also the file name of packfile contains a SHA-1 checksum. To check integrity, run the <tt>git fsck</tt> command.</dd>
</dl>
<p>Another property of Git is that it snapshots directory trees of files. The earliest systems for tracking versions of source code, <a href="/wiki/Source_Code_Control_System" title="Source Code Control System">Source Code Control System</a> (SCCS) and <a href="/wiki/Revision_Control_System" title="Revision Control System">Revision Control System</a> (RCS), worked on individual files and emphasized the space savings to be gained from <a href="/wiki/Interleaved_deltas" title="Interleaved deltas">interleaved deltas</a> (SCCS) or <a href="/wiki/Delta_encoding" title="Delta encoding">delta encoding</a> (RCS) the (mostly similar) versions. Later revision control systems maintained this notion of a file having an identity across multiple revisions of a project. However, Torvalds rejected this concept.<sup class="reference" id="cite_ref-38"><a href="#cite_note-38">[38]</a></sup> Consequently, Git does not explicitly record file revision relationships at any level below the source code tree.</p>
<p>Implicit revision relationships have some significant consequences:</p>
<ul>
<li>It is slightly more costly to examine the change history of one file than the whole project.<sup class="reference" id="cite_ref-39"><a href="#cite_note-39">[39]</a></sup> To obtain a history of changes affecting a given file, Git must walk the global history and then determine whether each change modified that file. This method of examining history does, however, let Git produce with equal efficiency a single history showing the changes to an arbitrary set of files. For example, a subdirectory of the source tree plus an associated global header file is a very common case.</li>
<li>Renames are handled implicitly rather than explicitly. A common complaint with <a href="/wiki/Concurrent_Versions_System" title="Concurrent Versions System">CVS</a> is that it uses the name of a file to identify its revision history, so moving or renaming a file is not possible without either interrupting its history, or renaming the history and thereby making the history inaccurate. Most post-CVS revision control systems solve this by giving a file a unique long-lived name (analogous to an <a href="/wiki/Inode" title="Inode">inode</a> number) that survives renaming. Git does not record such an identifier, and this is claimed as an advantage.<sup class="reference" id="cite_ref-40"><a href="#cite_note-40">[40]</a></sup><sup class="reference" id="cite_ref-41"><a href="#cite_note-41">[41]</a></sup> <a href="/wiki/Source_code" title="Source code">Source code</a> files are sometimes split or merged, or simply renamed,<sup class="reference" id="cite_ref-42"><a href="#cite_note-42">[42]</a></sup> and recording this as a simple rename would freeze an inaccurate description of what happened in the (immutable) history. Git addresses the issue by detecting renames while browsing the history of snapshots rather than recording it when making the snapshot.<sup class="reference" id="cite_ref-43"><a href="#cite_note-43">[43]</a></sup> (Briefly, given a file in revision <i>N,</i> a file of the same name in revision <i>N−1</i> is its default ancestor. However, when there is no like-named file in revision <i>N−1,</i> Git searches for a file that existed only in revision <i>N−1</i> and is very similar to the new file.) However, it does require more <a href="/wiki/Central_processing_unit" title="Central processing unit">central processing unit</a> (CPU) intensive work every time history is reviewed, and several options to adjust the heuristics. This mechanism does not always work; sometimes a file that is renamed with changes in the same commit is read as a deletion of the old file and the creation of a new file. Developers can work around this limitation by committing the rename and changes separately.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (September 2016)">citation needed</span></a></i>]</sup></li>
</ul>
<p>Git implements several merging strategies; a non-default can be selected at merge time:<sup class="reference" id="cite_ref-44"><a href="#cite_note-44">[44]</a></sup></p>
<ul>
<li><i>resolve</i>: the traditional <a class="mw-redirect" href="/wiki/Three-way_merge" title="Three-way merge">three-way merge</a> algorithm.</li>
<li><i>recursive</i>: This is the default when pulling or merging one branch, and is a variant of the three-way merge algorithm.
<blockquote class="templatequote">
<p>When there are more than one common ancestors that can be used for three-way merge, it creates a merged tree of the common ancestors and uses that as the reference tree for the three-way merge. This has been reported to result in fewer merge conflicts without causing mis-merges by tests done on prior merge commits taken from Linux 2.6 kernel development history. Also, this can detect and handle merges involving renames.</p>
<div class="templatequotecite"><cite>— Linus Torvalds<sup class="reference" id="cite_ref-45"><a href="#cite_note-45">[45]</a></sup></cite></div>
</blockquote>
</li>
</ul>
<ul>
<li><i>octopus</i>: This is the default when merging more than two heads.</li>
</ul>
<h3><span class="mw-headline" id="Data_structures">Data structures</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Git&amp;action=edit&amp;section=4" title="Edit section: Data structures">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Git's primitives are not inherently a <a class="mw-redirect" href="/wiki/Source_code_management" title="Source code management">source code management</a> (SCM) system. Torvalds explains,<sup class="reference" id="cite_ref-46"><a href="#cite_note-46">[46]</a></sup></p>
<blockquote class="templatequote">
<p>In many ways you can just see git as a filesystem –  it's <a href="/wiki/Content-addressable_storage" title="Content-addressable storage">content-addressable</a>, and it has a notion of versioning, but I really really designed it coming at the problem from the viewpoint of a <i>filesystem</i> person (hey, kernels is what I do), and I actually have absolutely <i>zero</i> interest in creating a traditional SCM system.</p>
</blockquote>
<p>From this initial design approach, Git has developed the full set of features expected of a traditional SCM,<sup class="reference" id="cite_ref-bare_url_28-1"><a href="#cite_note-bare_url-28">[28]</a></sup> with features mostly being created as needed, then refined and extended over time.</p>
<div class="thumb tright">
<div class="thumbinner" style="width:222px;"><a class="image" href="/wiki/File:Git_operations.svg"><img alt="" class="thumbimage" data-file-height="621" data-file-width="760" height="180" src="//upload.wikimedia.org/wikipedia/commons/thumb/d/d8/Git_operations.svg/220px-Git_operations.svg.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/d/d8/Git_operations.svg/330px-Git_operations.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/d/d8/Git_operations.svg/440px-Git_operations.svg.png 2x" width="220"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Git_operations.svg" title="Enlarge"></a></div>
Some data flows and storage levels in the Git revision control system.</div>
</div>
</div>
<p>Git has two <a href="/wiki/Data_structure" title="Data structure">data structures</a>: a mutable <i>index</i> (also called <i>stage</i> or <i>cache</i>) that caches information about the working directory and the next revision to be committed; and an immutable, append-only <i>object database.</i></p>
<p>The index serves as connection point between the object database and the working tree.</p>
<p>The object database contains four types of objects:</p>
<ul>
<li>A <i>blob</i> (<a href="/wiki/Binary_large_object" title="Binary large object">binary large object</a>) is the content of a <a href="/wiki/Computer_file" title="Computer file">file</a>. Blobs have no proper file name, time stamps, or other metadata. (A blob's name internally is a hash of its content.)</li>
<li>A <i>tree</i> object is the equivalent of a directory. It contains a list of file names, each with some type bits and a reference to a blob or tree object that is that file, symbolic link, or directory's contents. These objects are a snapshot of the source tree. (In whole, this comprises a <a class="mw-redirect" href="/wiki/Merkle_Tree" title="Merkle Tree">Merkle Tree</a>, meaning that only a single hash for the root tree is sufficient and actually used in commits to exactly pinpoint to the exact state of whole tree structures of any number of sub-directories and files.)</li>
<li>A <i>commit</i> object links tree objects together into a history. It contains the name of a tree object (of the top-level source directory), a time stamp, a log message, and the names of zero or more parent commit objects.</li>
<li>A <i>tag</i> object is a container that contains a reference to another object and can hold added meta-data related to another object. Most commonly, it is used to store a <a href="/wiki/Digital_signature" title="Digital signature">digital signature</a> of a commit object corresponding to a particular release of the data being tracked by Git.</li>
</ul>
<p>Each object is identified by a SHA-1 <a href="/wiki/Cryptographic_hash_function" title="Cryptographic hash function">hash</a> of its contents. Git computes the hash, and uses this value for the object's name. The object is put into a directory matching the first two characters of its hash. The rest of the hash is used as the file name for that object.</p>
<p>Git stores each revision of a file as a unique blob. The relationships between the blobs can be found through examining the tree and commit objects. Newly added objects are stored in their entirety using <a href="/wiki/Zlib" title="Zlib">zlib</a> compression. This can consume a large amount of disk space quickly, so objects can be combined into <i>packs</i>, which use <a href="/wiki/Delta_encoding" title="Delta encoding">delta compression</a> to save space, storing blobs as their changes relative to other blobs.</p>
<p>Git servers typically listen on <a class="mw-redirect" href="/wiki/TCP_and_UDP_port" title="TCP and UDP port">TCP port</a> 9418.<sup class="reference" id="cite_ref-47"><a href="#cite_note-47">[47]</a></sup></p>
<h3><span class="mw-headline" id="References">References</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Git&amp;action=edit&amp;section=5" title="Edit section: References">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Every object in the Git database which is not referred to may be cleaned up by using a garbage collection command, or automatically. An object may be referenced by another object, or an explicit reference. Git knows different types of references. The commands to create, move, and delete references vary. "git show-ref" lists all references. Some types are:</p>
<ul>
<li><i>heads</i>: refers to an object locally</li>
<li><i>remotes</i>: refers to an object which exists in a remote repository</li>
<li><i>stash</i>: refers to an object not yet committed</li>
<li><i>meta</i>: e.g. a configuration in a bare repository, user rights; the refs/meta/config namespace was introduced resp gets used by <a href="/wiki/Gerrit_(software)" title="Gerrit (software)">Gerrit</a><sup class="noprint Inline-Template" style="margin-left:0.1em; white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Please_clarify" title="Wikipedia:Please clarify"><span title="The text near this tag may need clarification or removal of jargon. (June 2014)">clarification needed</span></a></i>]</sup><sup class="reference" id="cite_ref-48"><a href="#cite_note-48">[48]</a></sup></li>
<li><i>tags</i>: see above</li>
</ul>
<h2><span class="mw-headline" id="Implementations">Implementations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Git&amp;action=edit&amp;section=6" title="Edit section: Implementations">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="thumb tright">
<div class="thumbinner" style="width:222px;"><a class="image" href="/wiki/File:GNOME_gitg.png"><img alt="" class="thumbimage" data-file-height="1408" data-file-width="2002" height="155" src="//upload.wikimedia.org/wikipedia/commons/thumb/9/93/GNOME_gitg.png/220px-GNOME_gitg.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/9/93/GNOME_gitg.png/330px-GNOME_gitg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/9/93/GNOME_gitg.png/440px-GNOME_gitg.png 2x" width="220"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:GNOME_gitg.png" title="Enlarge"></a></div>
<a href="/wiki/GNOME_Core_Applications" title="GNOME Core Applications">gitg</a> is a graphical front-end using GTK+</div>
</div>
</div>
<p>Git is primarily developed on <a href="/wiki/Linux" title="Linux">Linux</a>, although it also supports most major operating systems including <a href="/wiki/Berkeley_Software_Distribution" title="Berkeley Software Distribution">BSD</a>, <a href="/wiki/Solaris_(operating_system)" title="Solaris (operating system)">Solaris</a>, <a href="/wiki/MacOS" title="MacOS">macOS</a>, and <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Windows</a>.<sup class="reference" id="cite_ref-49"><a href="#cite_note-49">[49]</a></sup></p>
<p>The first Microsoft Windows "port" of Git was primarily a Linux emulation framework that hosts the Linux version. Installing Git under Windows creates a similarly named Program Files directory containing the MinGW port of the GNU Compiler Collection, Perl 5, msys2.0, itself a fork of Cygwin, a Unix-like emulation environment for Windows, and various other Windows ports or emulations of Linux utilities and libraries. Currently native Windows builds of Git are distributed as 32 and 64-bit installers.<sup class="reference" id="cite_ref-50"><a href="#cite_note-50">[50]</a></sup></p>
<p>The JGit implementation of Git is a pure <a href="/wiki/Java_(software_platform)" title="Java (software platform)">Java</a> software library, designed to be embedded in any Java application. JGit is used in the <a href="/wiki/Gerrit_(software)" title="Gerrit (software)">Gerrit</a> code review tool and in EGit, a <a class="new" href="/w/index.php?title=Git_client&amp;action=edit&amp;redlink=1" title="Git client (page does not exist)">Git client</a> for the <a href="/wiki/Eclipse_(software)" title="Eclipse (software)">Eclipse</a> IDE.<sup class="reference" id="cite_ref-51"><a href="#cite_note-51">[51]</a></sup></p>
<p>The Dulwich implementation of Git is a pure <a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a> software component for Python 2.<sup class="reference" id="cite_ref-52"><a href="#cite_note-52">[52]</a></sup></p>
<p>The libgit2 implementation of Git is an ANSI C software library with no other dependencies, which can be built on multiple platforms including Windows, Linux, macOS, and BSD.<sup class="reference" id="cite_ref-53"><a href="#cite_note-53">[53]</a></sup> It has bindings for many programming languages, including <a href="/wiki/Ruby_(programming_language)" title="Ruby (programming language)">Ruby</a>, Python, and <a href="/wiki/Haskell_(programming_language)" title="Haskell (programming language)">Haskell</a>.<sup class="reference" id="cite_ref-54"><a href="#cite_note-54">[54]</a></sup><sup class="reference" id="cite_ref-55"><a href="#cite_note-55">[55]</a></sup><sup class="reference" id="cite_ref-56"><a href="#cite_note-56">[56]</a></sup></p>
<p>JS-Git is a <a href="/wiki/JavaScript" title="JavaScript">JavaScript</a> implementation of a subset of Git.<sup class="reference" id="cite_ref-57"><a href="#cite_note-57">[57]</a></sup></p>
<h2><span class="mw-headline" id="Git_server">Git server</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Git&amp;action=edit&amp;section=7" title="Edit section: Git server">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>As Git is a distributed version control system, it can be used as a server out of the box. Dedicated Git server software helps, amongst other features, to add access control, display the contents of a Git repository via the web, and help managing multiple repositories. Remote file store and shell access: A Git repository can be cloned to a shared file system, and accessed by other persons. It can also be accessed via remote shell just by having the Git software installed and allowing a user to log in.<sup class="reference" id="cite_ref-58"><a href="#cite_note-58">[58]</a></sup></p>
<div class="hatnote" role="note">See also: <a href="/wiki/Comparison_of_source_code_hosting_facilities" title="Comparison of source code hosting facilities">Comparison of source code hosting facilities</a></div>
<h2><span class="mw-headline" id="Adoption">Adoption</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Git&amp;action=edit&amp;section=8" title="Edit section: Adoption">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The <a href="/wiki/Eclipse_Foundation" title="Eclipse Foundation">Eclipse Foundation</a> reported in its annual community survey that as of May 2014, Git is now the most widely used source code management tool, with 42.9% of professional software developers reporting that they use Git as their primary source control system<sup class="reference" id="cite_ref-ianskerrett.wordpress.com_59-0"><a href="#cite_note-ianskerrett.wordpress.com-59">[59]</a></sup> compared with 36.3% in 2013, 32% in 2012; or for Git responses excluding use of <a href="/wiki/GitHub" title="GitHub">GitHub</a>: 33.3% in 2014, 30.3% in 2013, 27.6% in 2012 and 12.8% in 2011.<sup class="reference" id="cite_ref-60"><a href="#cite_note-60">[60]</a></sup> Open source directory <a class="mw-redirect" href="/wiki/Black_Duck_Open_Hub" title="Black Duck Open Hub">Black Duck Open Hub</a> reports a similar uptake among open source projects.<sup class="reference" id="cite_ref-61"><a href="#cite_note-61">[61]</a></sup> The Stack Overflow developer survey reported in 2015 that 69.3% of developers use Git; 36.9% use Subversion; 12.2% use TFS; and 7.9% use Mercurial.<sup class="reference" id="cite_ref-62"><a href="#cite_note-62">[62]</a></sup></p>
<p>The UK IT jobs website itjobswatch.co.uk reports that as of late September 2016, 29.27% of UK permanent software development job openings have cited Git,<sup class="reference" id="cite_ref-63"><a href="#cite_note-63">[63]</a></sup> ahead of 12.17% for Microsoft <a href="/wiki/Team_Foundation_Server" title="Team Foundation Server">Team Foundation Server</a>,<sup class="reference" id="cite_ref-64"><a href="#cite_note-64">[64]</a></sup> 10.60% for <a href="/wiki/Apache_Subversion" title="Apache Subversion">Subversion</a>,<sup class="reference" id="cite_ref-65"><a href="#cite_note-65">[65]</a></sup> 1.30% for <a href="/wiki/Mercurial" title="Mercurial">Mercurial</a>,<sup class="reference" id="cite_ref-66"><a href="#cite_note-66">[66]</a></sup> and 0.48% for <a class="mw-redirect" href="/wiki/Visual_SourceSafe" title="Visual SourceSafe">Visual SourceSafe</a>.<sup class="reference" id="cite_ref-67"><a href="#cite_note-67">[67]</a></sup></p>
<h2><span class="mw-headline" id="Security">Security</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Git&amp;action=edit&amp;section=9" title="Edit section: Security">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Git does not provide access control mechanisms, but was designed for operation with other tools that specialize in access control.<sup class="reference" id="cite_ref-68"><a href="#cite_note-68">[68]</a></sup></p>
<p>On 17 December 2014, an exploit was found affecting the <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Windows</a> and <a href="/wiki/MacOS" title="MacOS">Mac</a> versions of the <a class="new" href="/w/index.php?title=Git_client&amp;action=edit&amp;redlink=1" title="Git client (page does not exist)">Git client</a>. An attacker could perform <a href="/wiki/Arbitrary_code_execution" title="Arbitrary code execution">arbitrary code execution</a> on a target computer with Git installed by creating a malicious Git tree (directory) named <i>.git</i> (a directory in Git repositories that stores all the data of the repository) in a different case (such as .GIT or .Git, needed because Git doesn't allow the all-lowercase version of <i>.git</i> to be created manually) with malicious files in the <i>.git/hooks</i> subdirectory (a folder with executable files that Git runs) on a repository that the attacker made or on a repository that the attacker can modify. If a Windows or Mac user <i>pulls</i> (downloads) a version of the repository with the malicious directory, then switches to that directory, the .git directory will be overwritten (due to the case-insensitive trait of the Windows and Mac filesystems) and the malicious executable files in <i>.git/hooks</i> may be run, which results in the attacker's commands being executed. An attacker could also modify the <i>.git/config</i> configuration file, which allows the attacker to create malicious Git aliases (aliases for Git commands or external commands) or modify extant aliases to execute malicious commands when run. The vulnerability was patched in version 2.2.1 of Git, released on 17 December 2014, and announced on the next day.<sup class="reference" id="cite_ref-69"><a href="#cite_note-69">[69]</a></sup><sup class="reference" id="cite_ref-70"><a href="#cite_note-70">[70]</a></sup></p>
<p>Git version 2.6.1, released on 29 September 2015, contained a patch for a security vulnerability (CVE-2015-7545)<sup class="reference" id="cite_ref-71"><a href="#cite_note-71">[71]</a></sup> which allowed arbitrary code execution.<sup class="reference" id="cite_ref-git-v2.6.1-commit_72-0"><a href="#cite_note-git-v2.6.1-commit-72">[72]</a></sup> The vulnerability was exploitable if an attacker could convince a victim to clone a specific URL, as the arbitrary commands were embedded in the URL itself.<sup class="reference" id="cite_ref-seclists-CVE-2015-7545_73-0"><a href="#cite_note-seclists-CVE-2015-7545-73">[73]</a></sup> An attacker could use the exploit via a <a href="/wiki/Man-in-the-middle_attack" title="Man-in-the-middle attack">man-in-the-middle attack</a> if the connection was unencrypted,<sup class="reference" id="cite_ref-seclists-CVE-2015-7545_73-1"><a href="#cite_note-seclists-CVE-2015-7545-73">[73]</a></sup> as they could redirect the user to a URL of their choice. Recursive clones were also vulnerable, since they allowed the controller of a repository to specify arbitrary URLs via the gitmodules file.<sup class="reference" id="cite_ref-seclists-CVE-2015-7545_73-2"><a href="#cite_note-seclists-CVE-2015-7545-73">[73]</a></sup></p>
<p>Git uses <a href="/wiki/SHA-1" title="SHA-1">SHA-1</a> hashes internally. Linus Torvalds has responded that the hash was mostly to guard against accidental corruption, and the security a cryptographically secure hash gives was just an accidental side effect, with the main security being signing elsewhere.<sup class="reference" id="cite_ref-74"><a href="#cite_note-74">[74]</a></sup><sup class="reference" id="cite_ref-75"><a href="#cite_note-75">[75]</a></sup></p>
