<h1 class="firstHeading" id="firstHeading" lang="en">Include directive</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
</div>
<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><script>function mfTempOpenSection(id){var block=document.getElementById("mf-section-"+id);block.className+=" open-block";block.previousSibling.className+=" open-block";}</script><p>Many <a href="/wiki/Programming_language" title="Programming language">programming languages</a> and other computer files have a <a href="/wiki/Directive_(programming)" title="Directive (programming)">directive</a>, often called <b><code>include</code></b> (as well as <b><code>copy</code></b> and <b><code>import</code></b>), that causes the contents of a second file to be inserted into the original file. These included files are called copybooks or header files. They are often used to define the physical layout of program data, pieces of procedural code and/or <a href="/wiki/Forward_declaration" title="Forward declaration">forward declarations</a> while promoting <a class="mw-redirect" href="/wiki/Encapsulation_(object-oriented_programming)" title="Encapsulation (object-oriented programming)">encapsulation</a> and the reuse of code.</p>

<h2><span class="mw-headline" id="Purpose">Purpose</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Include_directive&amp;action=edit&amp;section=1" title="Edit section: Purpose">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The include directive allows <a href="/wiki/Library_(computing)" title="Library (computing)">libraries</a> of code to be developed which help to:</p>
<ul>
<li>ensure that everyone uses the same version of a data layout definition or procedural code throughout a program.</li>
<li>easily cross-reference where components are used in a system.</li>
<li>easily change programs when needed (only one master file to change).</li>
<li>save time by not needing to code extensive data layouts (minor, but useful).</li>
</ul>
<p>An example situation which benefits from the use of an include directive is when referring to functions in a different file. Suppose we have a function <code>add</code> in one file, which is then declared (with a <a href="/wiki/Function_prototype" title="Function prototype">function prototype</a>) and then referred to in a second source file as follows:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="kt">int</span> <span class="nf">add</span><span class="p">(</span><span class="kt">int</span><span class="p">,</span> <span class="kt">int</span><span class="p">);</span>

<span class="kt">int</span> <span class="nf">triple</span><span class="p">(</span><span class="kt">int</span> <span class="n">x</span><span class="p">)</span>
<span class="p">{</span>
    <span class="k">return</span> <span class="n">add</span><span class="p">(</span><span class="n">x</span><span class="p">,</span> <span class="n">add</span><span class="p">(</span><span class="n">x</span><span class="p">,</span> <span class="n">x</span><span class="p">));</span>
<span class="p">}</span>
</pre></div>
<p>One drawback of this method is that the prototype must be present in all files that use the function. Another drawback is that if the return type or arguments of the function are changed, these prototypes will have to be updated. Putting the prototype in a single, separate file avoids these problems. Assuming the prototype is moved to the file <code>add.h</code>, the second source file can then become:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="cp">#include</span> <span class="cpf">"add.h"</span>

<span class="kt">int</span> <span class="nf">triple</span><span class="p">(</span><span class="kt">int</span> <span class="n">x</span><span class="p">)</span>
<span class="p">{</span>
    <span class="k">return</span> <span class="n">add</span><span class="p">(</span><span class="n">x</span><span class="p">,</span> <span class="n">add</span><span class="p">(</span><span class="n">x</span><span class="p">,</span><span class="n">x</span><span class="p">));</span>
<span class="p">}</span>
</pre></div>
<p>Now, every time the code is compiled, the latest function prototypes in <code>add.h</code> will be included in the files using them, avoiding potentially disastrous errors.</p>
<h2><span class="mw-headline" id="Language_Support">Language Support</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Include_directive&amp;action=edit&amp;section=2" title="Edit section: Language Support">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="C.2FC.2B.2B">C/C++</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Include_directive&amp;action=edit&amp;section=3" title="Edit section: C/C++">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In <a href="/wiki/C_(programming_language)" title="C (programming language)">C</a> and <a href="/wiki/C%2B%2B" title="C++">C++</a>, the <code>#include</code> <a class="mw-redirect" href="/wiki/Preprocessor_directive" title="Preprocessor directive">preprocessor directive</a> causes the <a href="/wiki/Compiler" title="Compiler">compiler</a> to replace that line with the entire text of the contents of the named source file (if included in quotes: "") or named header (if included in angle brackets: &lt;&gt;);<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> note that a header need not be a file.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> Inclusion continues recursively on these included contents, up to an implementation-defined nesting limit. Headers need not have names corresponding to files: in C++ standard headers are typically identified with words, like "vector", hence <code>#include &lt;vector&gt;</code> while in C standard headers have identifiers in the form of filenames with a ".h" extension, as in <code>#include &lt;stdio.h&gt;</code>. A "source file" can be any file, with a name of any form, but is most commonly named with a ".h" extension and called a "header file" (sometimes ".hpp" to distinguish C++ headers), though files with .c, .cc, and .cpp extensions may also be included (particularly in the <a href="/wiki/Single_Compilation_Unit" title="Single Compilation Unit">Single Compilation Unit</a> technique), and sometimes other extensions are used.</p>
<p>These two forms of <code>#include</code> directive can determine which header or source file to include in an implementation-defined way. In practice, what is usually done is that the angle-brackets form searches for <i>source files</i> in a standard system directory (or set of directories), and then searches for source files in local or project-specific paths (specified on the command line, in an environment variable, or in a <a href="/wiki/Makefile" title="Makefile">Makefile</a> or other build file), while the form with quotes does not search in a standard system directory, only searching in local or project-specific paths.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup> In case there is no clash, the angle-brackets form can also be used to specify project-specific includes, but this is considered poor form. The fact that headers need not correspond to files is primarily an implementation technicality, and used to omit the .h extension in including C++ standard headers; in common use "header" means "header file".</p>
<p>For example:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="cp">#include</span> <span class="cpf">&lt;stdio.h&gt;  // Include the contents of the standard header 'stdio.h' (probably a file 'stdio.h').</span>
<span class="cp">#include</span> <span class="cpf">&lt;vector&gt;  // Include the contents of the standard header 'vector' (probably a file 'vector.h').</span>
<span class="cp">#include</span> <span class="cpf">"user_defined.h"  // Include the contents of the file 'user_defined.h'.</span>
</pre></div>
<p>In C and C++, problems may be faced if two (or more) include files both in turn include the same third file. One solution is to avoid include files from including any other files, possibly requiring the programmer to manually add extra include directives to the original file. Another solution is to use <a href="/wiki/Include_guard" title="Include guard">include guards</a>.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></p>
<h3><span class="mw-headline" id="COBOL">COBOL</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Include_directive&amp;action=edit&amp;section=4" title="Edit section: COBOL">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/COBOL" title="COBOL">COBOL</a> (and also <a class="mw-redirect" href="/wiki/RPG_IV" title="RPG IV">RPG IV</a>) allows programmers to copy copybooks into the source of the program in a similar way to header files, but it also allows replacing certain text in them with other text. The COBOL keyword for inclusion is <code>COPY</code>, and replacement is done using the <code>REPLACING ... BY ...</code> clause. An include directive has been present in COBOL since COBOL 60, but changed from the original <code>INCLUDE</code><sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> to <code>COPY</code> by 1968.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup></p>
<h3><span class="mw-headline" id="Fortran">Fortran</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Include_directive&amp;action=edit&amp;section=5" title="Edit section: Fortran">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/Fortran" title="Fortran">Fortran</a> does not require header files <i>per se</i>. However, Fortran 90 and later has two related features: <code>include</code> statements and modules. The former can be used to share a common file containing procedure interfaces, much like a C header, although the specification of an interface is not required for all varieties of Fortran procedures. This approach is not commonly used; instead procedures are generally grouped into modules that can then be referenced with a <code>use</code> statement within other regions of code. For modules, header-type interface information is automatically generated by the compiler, and typically put into separate module files, although some compilers have placed this information directly into object files. The language specification itself does not mandate the creation of any extra files, even though module procedure interfaces are almost universally propagated in this manner.</p>
<h3><span class="mw-headline" id="PHP">PHP</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Include_directive&amp;action=edit&amp;section=6" title="Edit section: PHP">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In <a href="/wiki/PHP" title="PHP">PHP</a>, the <code>include</code> directive causes another PHP file to be included and evaluated.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup> Similar commands are <code>require</code>, which upon failure to include will produce a <a class="mw-redirect" href="/wiki/Fatal_error" title="Fatal error">fatal error</a> and halt the script,<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> and <code>include_once</code> and <code>require_once</code>, which cause a file to not be included or required again if it has already been included or required, avoiding the C's double inclusion problem.</p>
<h3><span class="mw-headline" id="Other_languages">Other languages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Include_directive&amp;action=edit&amp;section=7" title="Edit section: Other languages">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>There are many forms of the include directive, such as:</p>
<ul>
<li><code>/COPY <i>QCPYLESRC</i>,<i>QBC</i></code> (RPG IV – first argument is the filename, second argument is the copybook)</li>
<li><code>include ...</code> (<a href="/wiki/Fortran" title="Fortran">Fortran</a>, <a href="/wiki/Microsoft_Macro_Assembler" title="Microsoft Macro Assembler">MASM</a>)</li>
<li><code>%include ...</code> (PL/I)</li>
<li><code>&lt;!--#include ... --&gt;</code> (HTML <a href="/wiki/Server_Side_Includes" title="Server Side Includes">SSI</a>)</li>
<li><code>&lt;%@ include ... %&gt;</code> (<a href="/wiki/JavaServer_Pages" title="JavaServer Pages">JSP</a>).</li>
</ul>
<p>Modern languages (e.g. <a href="/wiki/Haskell_(programming_language)" title="Haskell (programming language)">Haskell</a> and <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>) tend to avoid copybooks or includes, preferring <a href="/wiki/Modular_programming" title="Modular programming">modules</a> and import/export systems for <a href="/wiki/Namespace" title="Namespace">namespace</a> control. Some of these languages (such as Java and <a href="/wiki/C_Sharp_(programming_language)" title="C Sharp (programming language)">C#</a>) do not use forward declarations and, instead, identifiers are recognized automatically from <a class="mw-redirect" href="/wiki/Source_file" title="Source file">source files</a> and read directly from <a class="mw-redirect" href="/wiki/Dynamic_library" title="Dynamic library">dynamic library</a> symbols, meaning header files are not needed.</p>
