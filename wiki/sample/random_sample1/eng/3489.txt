<p>In <a href="/wiki/Computing" title="Computing">computing</a>, <b>multitasking</b> is a concept of performing multiple <a href="/wiki/Task_computing" title="Task computing">tasks</a> (also known as <a href="/wiki/Process_(computing)" title="Process (computing)">processes</a>) over a certain period of time by executing them <a href="/wiki/Concurrent_computing" title="Concurrent computing">concurrently</a>. New tasks start and interrupt already started ones before they have reached completion, instead of executing the tasks sequentially so each started task needs to reach its end before a new one is started. As a result, a computer executes segments of multiple tasks in an interleaved manner, while the tasks share common processing resources such as <a href="/wiki/Central_processing_unit" title="Central processing unit">central processing units</a> (CPUs) and <a class="mw-redirect" href="/wiki/Main_memory" title="Main memory">main memory</a>.</p>
<p>Multitasking does <i>not</i> necessarily mean that multiple tasks are executing at exactly the same time (simultaneously). In other words, multitasking does not imply <a href="/wiki/Parallel_computing" title="Parallel computing">parallel execution</a>, but it does mean that more than one task can be part-way through execution at the same time, and that more than one task is advancing over a given period of time.<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> Even on <a class="mw-redirect" href="/wiki/Multiprocessor" title="Multiprocessor">multiprocessor</a> or <a class="mw-redirect" href="/wiki/Multicore" title="Multicore">multicore</a> computers, which have multiple CPUs/cores so more than one task can be executed at once (physically, one per CPU or core), multitasking allows many more tasks to be run than there are CPUs.</p>
<p>In the case of a computer with a single CPU, only one task is said to be running at any point in time, meaning that the CPU is actively executing instructions for that task. Multitasking solves the problem by <a href="/wiki/Scheduling_(computing)" title="Scheduling (computing)">scheduling</a> which task may be the one running at any given time, and when another waiting task gets a turn. The act of reassigning a CPU from one task to another one is called a <a href="/wiki/Context_switch" title="Context switch">context switch</a>; the illusion of parallelism is achieved when context switches occur frequently enough. Operating systems may adopt one of many different scheduling strategies, which generally fall into the following categories:<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (July 2013)">citation needed</span></a></i>]</sup></p>
<ul>
<li>In <a class="mw-redirect" href="/wiki/Multiprogramming" title="Multiprogramming">multiprogramming</a> systems, the running task keeps running until it performs an operation that requires waiting for an external event (e.g. reading from a tape) or until the computer's scheduler forcibly swaps the running task out of the CPU. Multiprogramming systems are designed to maximize CPU usage.</li>
<li>In <a href="/wiki/Time-sharing" title="Time-sharing">time-sharing</a> systems, the running task is required to relinquish the CPU, either voluntarily or by an external event such as a <a class="mw-redirect" href="/wiki/Hardware_interrupt" title="Hardware interrupt">hardware interrupt</a>. Time sharing systems are designed to allow several programs to execute apparently simultaneously.</li>
<li>In <a href="/wiki/Real-time_computing" title="Real-time computing">real-time</a> systems, some waiting tasks are guaranteed to be given the CPU when an external event occurs. Real time systems are designed to control mechanical devices such as industrial robots, which require timely processing.</li>
</ul>
<p>The term "multitasking" has become an international term, as the same word is used in many other languages such as German, Italian, Dutch, Danish and Norwegian.</p>

<h2><span class="mw-headline" id="Multiprogramming">Multiprogramming</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Computer_multitasking&amp;action=edit&amp;section=1" title="Edit section: Multiprogramming">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In the early days of computing, <a href="/wiki/CPU_time" title="CPU time">CPU time</a> was expensive, and <a href="/wiki/Peripheral" title="Peripheral">peripherals</a> were very slow. When the computer ran a program that needed access to a peripheral, the central processing unit (CPU) would have to stop executing program instructions while the peripheral processed the data. This was usually very inefficient.</p>
<p>The first computer using a multiprogramming system was the British <i><a href="/wiki/LEO_(computer)#Applications_and_successors" title="LEO (computer)">Leo III</a></i> owned by <a href="/wiki/J._Lyons_and_Co." title="J. Lyons and Co.">J. Lyons and Co.</a> During <a href="/wiki/Batch_processing" title="Batch processing">batch processing</a>, several different programs were loaded in the computer memory, and the first one began to run. When the first program reached an instruction waiting for a peripheral, the context of this program was stored away, and the second program in memory was given a chance to run. The process continued until all programs finished running.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (July 2013)">citation needed</span></a></i>]</sup></p>
<p>The use of multiprogramming was enhanced by the arrival of <a href="/wiki/Virtual_memory" title="Virtual memory">virtual memory</a> and <a href="/wiki/Virtual_machine" title="Virtual machine">virtual machine</a> technology, which enabled individual programs to make use of memory and operating system resources as if other concurrently running programs were, for all practical purposes, non-existent and invisible to them.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (July 2013)">citation needed</span></a></i>]</sup></p>
<p>Multiprogramming doesn't give any guarantee that a program will run in a timely manner. Indeed, the very first program may very well run for hours without needing access to a peripheral. As there were no users waiting at an interactive terminal, this was no problem: users handed in a deck of punched cards to an operator, and came back a few hours later for printed results. Multiprogramming greatly reduced wait times when multiple batches were being processed.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (July 2013)">citation needed</span></a></i>]</sup></p>
<h2><span class="mw-headline" id="Cooperative_multitasking"><span id="COOP"></span><span id="Cooperative_multitasking.2Ftime-sharing"></span>Cooperative multitasking</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Computer_multitasking&amp;action=edit&amp;section=2" title="Edit section: Cooperative multitasking">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="hatnote" role="note">Main article: <a class="mw-redirect" href="/wiki/Non-preemptive_multitasking" title="Non-preemptive multitasking">Non-preemptive multitasking</a></div>
<p>The expression "time sharing" usually designated computers shared by interactive users at terminals, such as IBM's <a href="/wiki/Time_Sharing_Option" title="Time Sharing Option">TSO</a>, and <a href="/wiki/CP/CMS" title="CP/CMS">VM/CMS</a>. The term "time-sharing" is no longer commonly used, having been replaced by "multitasking", following the advent of personal computers and workstations rather than shared interactive systems.</p>
<p>Early multitasking systems used applications that voluntarily ceded time to one another. This approach, which was eventually supported by many computer <a href="/wiki/Operating_system" title="Operating system">operating systems</a>, is known today as cooperative multitasking. Although it is now rarely used in larger systems except for specific applications such as <a href="/wiki/CICS" title="CICS">CICS</a> or the <a class="mw-redirect" href="/wiki/JES2" title="JES2">JES2</a> subsystem, cooperative multitasking was once the scheduling scheme employed by <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a> (prior to <a href="/wiki/Windows_95" title="Windows 95">Windows 95</a> and <a href="/wiki/Windows_NT" title="Windows NT">Windows NT</a>) and <a href="/wiki/Classic_Mac_OS" title="Classic Mac OS">Classic Mac OS</a> (prior to <a class="mw-redirect" href="/wiki/Mac_OS_X" title="Mac OS X">Mac OS X</a>) in order to enable multiple applications to be run simultaneously. <a href="/wiki/Windows_9x" title="Windows 9x">Windows 9x</a> also used cooperative multitasking, but only for 16-bit legacy applications, much the same way as pre-<a class="mw-redirect" href="/wiki/Mac_OS_X_v10.5" title="Mac OS X v10.5">Leopard</a> <a href="/wiki/PowerPC" title="PowerPC">PowerPC</a> versions of Mac OS X used it for <a class="mw-redirect" href="/wiki/Classic_(Mac_OS_X)" title="Classic (Mac OS X)">Classic</a> applications. The network operating system <a href="/wiki/NetWare" title="NetWare">NetWare</a> used cooperative multitasking up to NetWare 6.5. Cooperative multitasking is still used today on <a href="/wiki/RISC_OS" title="RISC OS">RISC OS</a> systems.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup></p>
<p>As a cooperatively multitasked system relies on each process regularly giving up time to other processes on the system, one poorly designed program can consume all of the CPU time for itself, either by performing extensive calculations or by <a class="mw-redirect" href="/wiki/Busy_wait" title="Busy wait">busy waiting</a>; both would cause the whole system to <a href="/wiki/Hang_(computing)" title="Hang (computing)">hang</a>. In a server environment, this is a hazard that makes the entire environment unacceptably fragile.</p>
<h2><span class="mw-headline" id="Preemptive_multitasking">Preemptive multitasking</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Computer_multitasking&amp;action=edit&amp;section=3" title="Edit section: Preemptive multitasking">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="hatnote" role="note">Main article: <a href="/wiki/Preemption_(computing)" title="Preemption (computing)">Preemption (computing)</a></div>
<p>Preemptive multitasking allows the computer system to more reliably guarantee to each process a regular "slice" of operating time. It also allows the system to deal rapidly with important external events like incoming data, which might require the immediate attention of one or another process. Operating systems were developed to take advantage of these hardware capabilities and run multiple processes preemptively. Preemptive multitasking <a href="/wiki/PDP-8#Programming_facilities" title="PDP-8">was supported on DEC's PDP-8 computers</a>, and implemented in <a href="/wiki/OS/360_and_successors#MFT" title="OS/360 and successors">OS/360 MFT</a> in 1967, in <a class="mw-redirect" href="/wiki/MULTICS" title="MULTICS">MULTICS</a> (1964), and <a href="/wiki/Unix" title="Unix">Unix</a> (1969); it is a core feature of all <a href="/wiki/Unix-like" title="Unix-like">Unix-like</a> operating systems, such as <a href="/wiki/Linux" title="Linux">Linux</a>, <a href="/wiki/Solaris_(operating_system)" title="Solaris (operating system)">Solaris</a> and <a href="/wiki/Berkeley_Software_Distribution" title="Berkeley Software Distribution">BSD</a> with its <a href="/wiki/Comparison_of_BSD_operating_systems" title="Comparison of BSD operating systems">derivatives</a>.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup></p>
<p>At any specific time, processes can be grouped into two categories: those that are waiting for input or output (called "<a href="/wiki/I/O_bound" title="I/O bound">I/O bound</a>"), and those that are fully utilizing the CPU ("<a class="mw-redirect" href="/wiki/CPU_bound" title="CPU bound">CPU bound</a>"). In primitive systems, the software would often "<a href="/wiki/Polling_(computer_science)" title="Polling (computer science)">poll</a>", or "<a href="/wiki/Busy_waiting" title="Busy waiting">busywait</a>" while waiting for requested input (such as disk, keyboard or network input). During this time, the system was not performing useful work. With the advent of interrupts and preemptive multitasking, I/O bound processes could be "blocked", or put on hold, pending the arrival of the necessary data, allowing other processes to utilize the CPU. As the arrival of the requested data would generate an interrupt, blocked processes could be guaranteed a timely return to execution.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (July 2013)">citation needed</span></a></i>]</sup></p>
<p>The earliest preemptive multitasking OS available to home users was <a href="/wiki/Sinclair_QDOS" title="Sinclair QDOS">Sinclair QDOS</a> on the <a href="/wiki/Sinclair_QL" title="Sinclair QL">Sinclair QL</a>, released in 1984, but very few people bought the machine. Commodore's powerful <a href="/wiki/Amiga" title="Amiga">Amiga</a>, released the following year, was the first commercially successful home computer to use the technology, and its multimedia abilities make it a clear ancestor of contemporary multitasking personal computers. <a href="/wiki/Microsoft" title="Microsoft">Microsoft</a> made preemptive multitasking a core feature of their flagship operating system in the early 1990s when developing <a href="/wiki/Windows_NT_3.1" title="Windows NT 3.1">Windows NT 3.1</a> and then <a href="/wiki/Windows_95" title="Windows 95">Windows 95</a>. It was later adopted on the Apple Macintosh by <a class="mw-redirect" href="/wiki/Mac_OS_X" title="Mac OS X">Mac OS X</a> that, as a <a href="/wiki/Unix-like" title="Unix-like">Unix-like</a> operating system, uses preemptive multitasking for all native applications.</p>
<p>A similar model is used in <a href="/wiki/Windows_9x" title="Windows 9x">Windows 9x</a> and the <a href="/wiki/Windows_NT" title="Windows NT">Windows NT family</a>, where native 32-bit applications are multitasked preemptively, and legacy 16-bit <a href="/wiki/Windows_3.1x" title="Windows 3.1x">Windows 3.x</a> programs are multitasked cooperatively within a single process, although in the NT family it is possible to force a 16-bit application to run as a separate preemptively multitasked process.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup> 64-bit editions of Windows, both for the <a href="/wiki/X86-64" title="X86-64">x86-64</a> and <a href="/wiki/Itanium" title="Itanium">Itanium</a> architectures, no longer provide support for legacy 16-bit applications, and thus provide preemptive multitasking for all supported applications.</p>
<h2><span class="mw-headline" id="Real_time">Real time</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Computer_multitasking&amp;action=edit&amp;section=4" title="Edit section: Real time">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Another reason for multitasking was in the design of <a href="/wiki/Real-time_computing" title="Real-time computing">real-time computing</a> systems, where there are a number of possibly unrelated external activities needed to be controlled by a single processor system. In such systems a hierarchical interrupt system is coupled with process prioritization to ensure that key activities were given a greater share of available <a class="mw-redirect" href="/wiki/Process_time" title="Process time">process time</a>.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (July 2013)">citation needed</span></a></i>]</sup></p>
<h2><span class="mw-headline" id="Multithreading">Multithreading</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Computer_multitasking&amp;action=edit&amp;section=5" title="Edit section: Multithreading">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>As multitasking greatly improved the throughput of computers, programmers started to implement applications as sets of cooperating processes (e. g., one process gathering input data, one process processing input data, one process writing out results on disk). This, however, required some tools to allow processes to efficiently exchange data.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (July 2013)">citation needed</span></a></i>]</sup></p>
<p><a class="mw-redirect" href="/wiki/Thread_(computer_science)" title="Thread (computer science)">Threads</a> were born from the idea that the most efficient way for cooperating processes to exchange data would be to share their entire memory space. Thus, threads are effectively processes that run in the same memory context and share other resources with their <a href="/wiki/Parent_process" title="Parent process">parent processes</a>, such as open files. Threads are described as <i>lightweight processes</i> because switching between threads does not involve changing the memory context.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup><sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup><sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup></p>
<p>While threads are scheduled preemptively, some operating systems provide a variant to threads, named <i><a href="/wiki/Fiber_(computer_science)" title="Fiber (computer science)">fibers</a></i>, that are scheduled cooperatively. On operating systems that do not provide fibers, an application may implement its own fibers using repeated calls to worker functions. Fibers are even more lightweight than threads, and somewhat easier to program with, although they tend to lose some or all of the benefits of threads on <a href="/wiki/Multiprocessing" title="Multiprocessing">machines with multiple processors</a>.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (August 2007)">citation needed</span></a></i>]</sup></p>
<p>Some systems directly support <a class="mw-redirect" href="/wiki/Multithreading_(computer_hardware)" title="Multithreading (computer hardware)">multithreading in hardware</a>.</p>
<h2><span class="mw-headline" id="Memory_protection">Memory protection</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Computer_multitasking&amp;action=edit&amp;section=6" title="Edit section: Memory protection">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="hatnote" role="note">Main article: <a href="/wiki/Memory_protection" title="Memory protection">Memory protection</a></div>
<p>Essential to any multitasking system is to safely and effectively share access to system resources. Access to memory must be strictly managed to ensure that no process can inadvertently or deliberately read or write to memory locations outside of the process's address space. This is done for the purpose of general system stability and data integrity, as well as data security.</p>
<p>In general, memory access management is a responsibility of the operating system kernel, in combination with hardware mechanisms that provide supporting functionalities, such as a <a href="/wiki/Memory_management_unit" title="Memory management unit">memory management unit</a> (MMU). If a process attempts to access a memory location outside of its memory space, the MMU denies the request and signals the kernel to take appropriate actions; this usually results in forcibly terminating the offending process. Depending on the software and kernel design and the specific error in question, the user may receive an access violation error message such as "segmentation fault".</p>
<p>In a well designed and correctly implemented multitasking system, a given process can never directly access memory that belongs to another process. An exception to this rule is in the case of shared memory; for example, in the <a href="/wiki/UNIX_System_V" title="UNIX System V">System V</a> inter-process communication mechanism the kernel allocates memory to be mutually shared by multiple processes. Such features are often used by database management software such as PostgreSQL.</p>
<p>Inadequate memory protection mechanisms, either due to flaws in their design or poor implementations, allow for security vulnerabilities that may be potentially exploited by malicious software.</p>
<h2><span class="mw-headline" id="Memory_swapping">Memory swapping</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Computer_multitasking&amp;action=edit&amp;section=7" title="Edit section: Memory swapping">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Use of a <a href="/wiki/Virtual_memory" title="Virtual memory">swap file</a> or swap partition is a way for the operating system to provide more memory than is physically available by keeping portions of the primary memory in <a class="mw-redirect" href="/wiki/Secondary_storage" title="Secondary storage">secondary storage</a>. While multitasking and memory swapping are two completely unrelated techniques, they are very often used together, as swapping memory allows more tasks to be loaded at the same time. Typically, a multitasking system allows another process to run when the running process hits a point where it has to wait for some portion of memory to be reloaded from secondary storage.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (July 2013)">citation needed</span></a></i>]</sup></p>
<h2><span class="mw-headline" id="Programming">Programming</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Computer_multitasking&amp;action=edit&amp;section=8" title="Edit section: Programming">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Processes that are entirely independent are not much trouble to program in a multitasking environment. Most of the complexity in multitasking systems comes from the need to share computer resources between tasks and to synchronize the operation of co-operating tasks.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (July 2013)">citation needed</span></a></i>]</sup></p>
<p>Various <a href="/wiki/Concurrent_computing" title="Concurrent computing">concurrent computing</a> techniques are used to avoid potential problems caused by multiple tasks attempting to access the same resource.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (July 2013)">citation needed</span></a></i>]</sup></p>
<p>Bigger systems were sometimes built with a central processor(s) and some number of <a href="/wiki/Channel_I/O" title="Channel I/O">I/O processors</a>, a kind of asymmetric <a href="/wiki/Multiprocessing" title="Multiprocessing">multiprocessing</a>.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (July 2013)">citation needed</span></a></i>]</sup></p>
<p>Over the years, multitasking systems have been refined. Modern operating systems generally include detailed mechanisms for prioritizing processes, while <a href="/wiki/Symmetric_multiprocessing" title="Symmetric multiprocessing">symmetric multiprocessing</a> has introduced new complexities and capabilities.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (July 2013)">citation needed</span></a></i>]</sup> In other words, if you were in need of a smaller explanation, it is the task of doing two (or more) things at one time.</p>
