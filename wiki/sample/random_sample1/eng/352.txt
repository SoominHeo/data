<p>In <a href="/wiki/Computing" title="Computing">computing</a>, <b>JSON</b> (canonically pronounced <span class="nowrap"><span class="IPA nopopups"><a href="/wiki/Help:IPA_for_English" title="Help:IPA for English">/<span style="border-bottom:1px dotted"><span title="/ˈ/ primary stress follows">ˈ</span><span title="/dʒ/ 'j' in 'jam'">dʒ</span><span title="/eɪ/ long 'a' in 'base'">eɪ</span><span title="'s' in 'sigh'">s</span><span title="/ən/ 'on' in 'button'">ən</span></span>/</a></span></span> <span title="English pronunciation respelling"><a href="/wiki/Help:Pronunciation_respelling_key" title="Help:Pronunciation respelling key"><i><b><span class="smallcaps"><span style="font-variant: small-caps; text-transform: lowercase;">JAY</span></span></b>-sən</i></a></span>;<sup class="reference" id="cite_ref-Pronunciation_1-0"><a href="#cite_note-Pronunciation-1">[1]</a></sup> sometimes <b>JavaScript Object Notation</b>) is an <a href="/wiki/Open_standard" title="Open standard">open-standard</a> <a href="/wiki/File_format#Chunk-based_formats" title="File format">format</a> that uses <a class="mw-redirect" href="/wiki/Human-readable" title="Human-readable">human-readable</a> text to transmit data objects consisting of <a href="/wiki/Attribute%E2%80%93value_pair" title="Attribute–value pair">attribute–value pairs</a>. It is the most common data format used for <a href="/wiki/Asynchronous_I/O" title="Asynchronous I/O">asynchronous</a> browser/server communication, largely replacing <a href="/wiki/XML" title="XML">XML</a> which is used by <a href="/wiki/Ajax_(programming)" title="Ajax (programming)">Ajax</a>.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (December 2016)">citation needed</span></a></i>]</sup></p>
<p>JSON is a <a href="/wiki/Language-independent_specification" title="Language-independent specification">language-independent</a> data format. It derives from <a href="/wiki/JavaScript" title="JavaScript">JavaScript</a>, but as of 2017<sup class="plainlinks noprint asof-tag update" style="display:none;"><a class="external text" href="//en.wikipedia.org/w/index.php?title=JSON&amp;action=edit">[update]</a></sup> many <a class="mw-redirect" href="/wiki/Programming_languages" title="Programming languages">programming languages</a> include code to generate and <a class="mw-redirect" href="/wiki/Parse" title="Parse">parse</a> JSON-format data. The official <a class="mw-redirect" href="/wiki/Internet_media_type" title="Internet media type">Internet media type</a> for JSON is <code>application/json</code>. JSON filenames use the extension <code>.json</code>.</p>
<p><a href="/wiki/Douglas_Crockford" title="Douglas Crockford">Douglas Crockford</a> originally specified the JSON format in the early 2000s; two competing standards, <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc7159" rel="nofollow">RFC 7159</a> and <a class="external text" href="https://www.ecma-international.org/publications/standards/Ecma-404.htm" rel="nofollow">ECMA-404</a>, defined it in 2013. The ECMA standard describes only the allowed syntax, whereas the RFC covers some security and interoperability considerations.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup></p>

<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>JSON grew out of a need for <a href="/wiki/State_(computer_science)" title="State (computer science)">stateful</a>, real-time server-to-browser communication without using browser plugins such as <a href="/wiki/Adobe_Flash" title="Adobe Flash">Flash</a> or Java applets, which were the dominant methods in the early 2000s.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (December 2016)">citation needed</span></a></i>]</sup></p>
<p><a href="/wiki/Douglas_Crockford" title="Douglas Crockford">Douglas Crockford</a> was<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup> the first to specify and popularize the JSON format. The acronym was coined at State Software, a company co-founded by Crockford, <a href="/wiki/Chip_Morningstar" title="Chip Morningstar">Chip Morningstar</a> and Robert F. Napiltonia in April 2001 and funded by Tesla Ventures. The co-founders agreed to build a system that used standard browser capabilities and provided an <a href="/wiki/Abstraction_layer" title="Abstraction layer">abstraction layer</a> for Web developers to create stateful Web applications that had a persistent duplex connection to a Web server by holding two <a href="/wiki/Hypertext_Transfer_Protocol" title="Hypertext Transfer Protocol">HTTP</a> connections open and recycling them before standard browser time-outs if no further data were exchanged. The idea for the State Application Framework was developed by Morningstar at State Software.<sup class="reference" id="cite_ref-Morningstar_4-0"><a href="#cite_note-Morningstar-4">[4]</a></sup><sup class="reference" id="cite_ref-State_Software_5-0"><a href="#cite_note-State_Software-5">[5]</a></sup> JSON was used in a project at Communities.com for Cartoon Network, which used a plug-in with a proprietary messaging format to manipulate <a href="/wiki/Dynamic_HTML" title="Dynamic HTML">DHTML</a> elements (this system is also owned by 3DO). Upon discovery of early <a href="/wiki/Ajax_(programming)" title="Ajax (programming)">Ajax</a> capabilities, digiGroups, Noosh, and others used frames to pass information into the user browsers' visual field without refreshing a Web application's visual context, realizing real-time rich Web applications using only the standard HTTP, HTML and JavaScript capabilities of Netscape 4.0.5+ and IE 5+. Crockford then found that JavaScript could be used as an object-based messaging format for such a system. The system was sold to <a href="/wiki/Sun_Microsystems" title="Sun Microsystems">Sun Microsystems</a>, <a href="/wiki/Amazon.com" title="Amazon.com">Amazon.com</a> and <a href="/wiki/Electronic_Data_Systems" title="Electronic Data Systems">EDS</a>. The <a class="external text" href="http://json.org/" rel="nofollow">JSON.org</a> Web site was launched in 2002. In December 2005, <a href="/wiki/Yahoo!" title="Yahoo!">Yahoo!</a> began offering some of its <a href="/wiki/Web_service" title="Web service">Web services</a> in JSON.<sup class="reference" id="cite_ref-yahoo_6-0"><a href="#cite_note-yahoo-6">[6]</a></sup> <a href="/wiki/Google" title="Google">Google</a> started offering JSON feeds for its <a href="/wiki/GData" title="GData">GData</a> web protocol in December 2006.<sup class="reference" id="cite_ref-Google_7-0"><a href="#cite_note-Google-7">[7]</a></sup></p>
<p>JSON was originally intended to be a subset of the <a href="/wiki/JavaScript" title="JavaScript">JavaScript</a> scripting language (specifically, Standard <a href="/wiki/Ecma_International" title="Ecma International">ECMA</a>-262 3rd Edition—December 1999<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup>) and is commonly used with Javascript, but it is a <a href="/wiki/Language-independent_specification" title="Language-independent specification">language-independent</a> data format. Code for <a href="/wiki/Parsing" title="Parsing">parsing</a> and generating JSON data is readily available in many <a class="mw-redirect" href="/wiki/Programming_languages" title="Programming languages">programming languages</a>. JSON's Web site lists JSON <a href="/wiki/Language_binding" title="Language binding">libraries</a> by language.</p>
<p>Though JSON was originally advertised and believed to be a strict subset of JavaScript and ECMAScript,<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup> it inadvertently allows some unescaped characters in strings that are illegal in JavaScript and ECMAScript strings. See <a href="#Data_portability_issues">Data portability issues</a> below.</p>
<p>JSON itself became an ECMA international standard in 2013 as the <i>ECMA-404 standard</i>.<sup class="reference" id="cite_ref-ecma2013_10-0"><a href="#cite_note-ecma2013-10">[10]</a></sup> In the same year <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc7158" rel="nofollow">RFC 7158</a> used ECMA-404 as reference. In 2014 <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc7159" rel="nofollow">RFC 7159</a> became the main reference for JSON's internet uses (ex. MIME application/json), and obsoletes <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc4627" rel="nofollow">RFC 4627</a> and <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc7158" rel="nofollow">RFC 7158</a> (but preserving ECMA-262 and ECMA-404 as main references).</p>
<h2><span class="mw-headline" id="Data_types.2C_syntax_and_example">Data types, syntax and example</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=2" title="Edit section: Data types, syntax and example">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>JSON's basic data types are:</p>
<ul>
<li>Number: a signed decimal number that may contain a fractional part and may use exponential <a class="mw-redirect" href="/wiki/E_notation" title="E notation">E notation</a>, but cannot include non-numbers like <a href="/wiki/NaN" title="NaN">NaN</a>. The format makes no distinction between integer and floating-point. JavaScript uses a <a href="/wiki/Double-precision_floating-point_format" title="Double-precision floating-point format">double-precision floating-point format</a> for all its numeric values, but other languages implementing JSON may encode numbers differently.</li>
<li><a href="/wiki/String_(computer_science)" title="String (computer science)">String</a>: a sequence of zero or more <a href="/wiki/Unicode" title="Unicode">Unicode</a> characters. Strings are delimited with double-quotation marks and support a backslash <a href="/wiki/Escape_character" title="Escape character">escaping</a> syntax.</li>
<li><a class="mw-redirect" href="/wiki/Boolean_datatype" title="Boolean datatype">Boolean</a>: either of the values <code>true</code> or <code>false</code></li>
<li><a href="/wiki/Array_data_structure" title="Array data structure">Array</a>: an <a href="/wiki/List_(abstract_data_type)" title="List (abstract data type)">ordered list</a> of zero or more values, each of which may be of any type. Arrays use <a class="mw-redirect" href="/wiki/Square_bracket" title="Square bracket">square bracket</a> notation with elements being comma-separated.</li>
<li>Object: an unordered collection of name/value pairs where the names (also called keys) are strings. Since objects are intended to represent <a href="/wiki/Associative_array" title="Associative array">associative arrays</a>,<sup class="reference" id="cite_ref-ecma2013_10-1"><a href="#cite_note-ecma2013-10">[10]</a></sup> it is recommended, though not required,<sup class="reference" id="cite_ref-rfc7519_11-0"><a href="#cite_note-rfc7519-11">[11]</a></sup> that each key is unique within an object. Objects are delimited with <a class="mw-redirect" href="/wiki/Braces_(punctuation)#Braces" title="Braces (punctuation)">curly brackets</a> and use commas to separate each pair, while within each pair the colon ':' character separates the key or name from its value.</li>
<li><code><a href="/wiki/Nullable_type" title="Nullable type">null</a></code>: An empty value, using the word <code>null</code></li>
</ul>
<p>Limited <a href="/wiki/Whitespace_character" title="Whitespace character">whitespace</a> is allowed and ignored around or between syntactic elements (values and punctuation, but not within a string value). Only four specific characters are considered whitespace for this purpose: space, horizontal tab, line feed, and carriage return. In particular, the <a href="/wiki/Byte_order_mark" title="Byte order mark">byte order mark</a> must not be generated by a conforming implementation (though it may be accepted when parsing JSON). JSON does not provide any syntax for comments.</p>
<p>Early versions of JSON (such as specified by <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc4627" rel="nofollow">RFC 4627</a>) required that a valid JSON "document" must consist of only an object or an array type, which could contain other types within them. This restriction was removed starting with <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc7158" rel="nofollow">RFC 7158</a>, so that a JSON document may consist entirely of any possible JSON typed value. (Note that <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc7159" rel="nofollow">RFC 7159</a>, dated March 2014, now obsoletes both 4627 and 7158.)</p>

<h3><span class="mw-headline" id="Data_portability_issues">Data portability issues</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=4" title="Edit section: Data portability issues">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Although Douglas Crockford originally claimed that JSON is a strict subset of JavaScript, his specification actually allows valid JSON documents that are invalid JavaScript. Specifically, JSON allows the <a href="/wiki/Unicode#Newlines" title="Unicode">Unicode line terminators</a> <span class="nowrap"><span style="font-family: monospace, monospace;">U+2028</span></span> <span class="smallcaps" style="font-variant: small-caps; font-size: smaller;">LINE SEPARATOR</span> and <span class="nowrap"><span style="font-family: monospace, monospace;">U+2029</span></span> <span class="smallcaps" style="font-variant: small-caps; font-size: smaller;">PARAGRAPH SEPARATOR</span> to appear unescaped in quoted strings, while JavaScript does not.<sup class="reference" id="cite_ref-json-2028_12-0"><a href="#cite_note-json-2028-12">[12]</a></sup> This is a consequence of JSON disallowing only "control characters". For maximum portability, these characters should be backslash-escaped. This subtlety is important when generating <a href="/wiki/JSONP" title="JSONP">JSONP</a>.</p>
<p>JSON documents can be encoded in <a href="/wiki/UTF-8" title="UTF-8">UTF-8</a>, <a href="/wiki/UTF-16" title="UTF-16">UTF-16</a> or <a href="/wiki/UTF-32" title="UTF-32">UTF-32</a>, the default encoding being UTF-8.<sup class="reference" id="cite_ref-ecma2013_10-2"><a href="#cite_note-ecma2013-10">[10]</a></sup> These encodings support the full Unicode character set, including those characters outside the <a href="/wiki/Plane_(Unicode)#Basic_Multilingual_Plane" title="Plane (Unicode)">Basic Multilingual Plane</a> (U+10000 to U+10FFFF). However, if escaped, those characters must be written using <a href="/wiki/UTF-16" title="UTF-16">UTF-16</a> surrogate pairs, a detail missed by some JSON parsers. For example, to include the <a href="/wiki/Emoji" title="Emoji">Emoji</a> character <span class="nowrap"><span style="font-family: monospace, monospace;">U+1F602</span></span> <span style="font-size:125%">😂</span> <span class="smallcaps" style="font-variant: small-caps; font-size: smaller;">FACE WITH TEARS OF JOY</span> in JSON:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="p">{</span> <span class="s2">"face"</span><span class="o">:</span> <span class="s2">"😂"</span> <span class="p">}</span>
<span class="c1">// or</span>
<span class="p">{</span> <span class="s2">"face"</span><span class="o">:</span> <span class="s2">"\uD83D\uDE02"</span> <span class="p">}</span>
</pre></div>
<p>Numbers in JSON are agnostic with regard to their representation within programming languages. No differentiation is made between an integer and floating-point value: some implementations may treat <code>42</code>, <code>42.0</code>, and <code>4.2E+1</code> as the same number while others may not. Furthermore, no requirements are made regarding implementation issues such as <a href="/wiki/Arithmetic_overflow" title="Arithmetic overflow">overflow</a>, <a href="/wiki/Arithmetic_underflow" title="Arithmetic underflow">underflow</a>, loss of precision, or rounding. Additionally, JSON says nothing about the treatment of <a class="mw-redirect" href="/wiki/Signed_zeros" title="Signed zeros">signed zeros</a>: whether <code>0.0</code> is distinct from <code>-0.0</code>. Most implementations that use the <a class="mw-redirect" href="/wiki/IEEE_754" title="IEEE 754">IEEE 754</a> floating-point standard, including JavaScript, preserve signed zeros; but not all JSON implementations may do so.</p>
<h3><span class="mw-headline" id="Using_JSON_in_JavaScript">Using JSON in JavaScript</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=5" title="Edit section: Using JSON in JavaScript">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Since JSON was derived from JavaScript and its syntax is (mostly) a subset of the language, it is possible to use the JavaScript <code class="mw-highlight" dir="ltr" id="" style=""><span class="nb">eval</span><span class="p">()</span></code> function to parse JSON data. Due to the issue with parsing of Unicode line terminators discussed in the preceding section, the eval function needs to perform a string replacement.<sup class="reference" id="cite_ref-json-2028_12-1"><a href="#cite_note-json-2028-12">[12]</a></sup></p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="kd">var</span> <span class="nx">p</span> <span class="o">=</span> <span class="nb">eval</span><span class="p">(</span><span class="s1">'('</span> <span class="o">+</span> <span class="nx">json_string</span><span class="p">.</span><span class="nx">replace</span><span class="p">(</span><span class="sr">/\u2028/g</span><span class="p">,</span><span class="s1">'\\u2028'</span><span class="p">).</span><span class="nx">replace</span><span class="p">(</span><span class="sr">/\u2029/g</span><span class="p">,</span> <span class="s1">'\\u2029'</span><span class="p">)</span> <span class="o">+</span> <span class="s1">')'</span><span class="p">);</span>
</pre></div>
<p>This is unsafe if the string is untrusted. Instead, a JSON parser library or JavaScript's native JSON support should be used for reading and writing JSON.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="kd">var</span> <span class="nx">p</span> <span class="o">=</span> <span class="nx">JSON</span><span class="p">.</span><span class="nx">parse</span><span class="p">(</span><span class="nx">json_string</span><span class="p">);</span>
</pre></div>
<p>A correctly implemented JSON parser only accepts valid JSON, preventing potentially malicious code from being inadvertently executed.</p>
<p>Since 2010, web browsers such as <a href="/wiki/Firefox" title="Firefox">Firefox</a> and <a href="/wiki/Internet_Explorer" title="Internet Explorer">Internet Explorer</a> have included support for parsing JSON. As native browser support is more efficient and secure than <code class="mw-highlight" dir="ltr" id="" style=""><span class="nb">eval</span><span class="p">()</span></code>, native JSON support is included in Edition 5 of the <a href="/wiki/ECMAScript" title="ECMAScript">ECMAScript</a> standard.<sup class="reference" id="cite_ref-13"><a href="#cite_note-13">[13]</a></sup></p>
<h3><span class="mw-headline" id="Unsupported_native_data_types">Unsupported native data types</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=6" title="Edit section: Unsupported native data types">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/JavaScript_syntax" title="JavaScript syntax">JavaScript syntax</a> defines several native data types that are not included in the JSON standard:<sup class="reference" id="cite_ref-rfc7519_11-1"><a href="#cite_note-rfc7519-11">[11]</a></sup> Map, Set, Date, Error, Regular Expression, Function, Promise, and <code>undefined</code>.<sup class="reference" id="cite_ref-undefinedType_14-0"><a href="#cite_note-undefinedType-14">[note 1]</a></sup> These JavaScript data types must be represented by some other data format, with the programs on both ends agreeing on how to convert between the types. As of 2011<sup class="plainlinks noprint asof-tag update" style="display:none;"><a class="external text" href="//en.wikipedia.org/w/index.php?title=JSON&amp;action=edit">[update]</a></sup>, there are some de facto standards, <i>e.g.</i>, converting from Date to String, but none universally recognized.<sup class="reference" id="cite_ref-15"><a href="#cite_note-15">[14]</a></sup><sup class="reference" id="cite_ref-16"><a href="#cite_note-16">[15]</a></sup> Other languages may have a different set of native types that must be <a href="/wiki/Serialization" title="Serialization">serialized</a> carefully to deal with this type of conversion.</p>
<h2><span class="mw-headline" id="Schema_and_metadata">Schema and metadata</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=7" title="Edit section: Schema and metadata">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="JSON_Schema">JSON Schema</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=8" title="Edit section: JSON Schema">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><span id="Schema"></span> JSON Schema<sup class="reference" id="cite_ref-17"><a href="#cite_note-17">[16]</a></sup> specifies a JSON-based format to define the structure of JSON data for validation, documentation, and interaction control. A JSON Schema provides a contract for the JSON data required by a given application, and how that data can be modified.</p>
<p>JSON Schema is based on the concepts from <a href="/wiki/XML_Schema_(W3C)" title="XML Schema (W3C)">XML Schema</a> (XSD), but is JSON-based. The JSON data schema can be used to validate JSON data. As in XSD, the same serialization/deserialization tools can be used both for the schema and data. The schema is self-describing.</p>
<p>JSON Schema was an <a href="/wiki/Internet_Draft" title="Internet Draft">Internet Draft</a>, most recently version 4, which expired on August 4, 2013.<sup class="reference" id="cite_ref-18"><a href="#cite_note-18">[17]</a></sup> There are several validators available for different programming languages,<sup class="reference" id="cite_ref-19"><a href="#cite_note-19">[18]</a></sup> each with varying levels of conformance. There are no standard file extension, but some suggested <code>.schema.json</code>.<sup class="reference" id="cite_ref-20"><a href="#cite_note-20">[19]</a></sup></p>
<p>Example JSON Schema (draft 4):</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="p">{</span>
  <span class="nt">"$schema"</span><span class="p">:</span> <span class="s2">"http://json-schema.org/schema#"</span><span class="p">,</span>
  <span class="nt">"title"</span><span class="p">:</span> <span class="s2">"Product"</span><span class="p">,</span>
  <span class="nt">"type"</span><span class="p">:</span> <span class="s2">"object"</span><span class="p">,</span>
  <span class="nt">"required"</span><span class="p">:</span> <span class="p">[</span><span class="s2">"id"</span><span class="p">,</span> <span class="s2">"name"</span><span class="p">,</span> <span class="s2">"price"</span><span class="p">],</span>
  <span class="nt">"properties"</span><span class="p">:</span> <span class="p">{</span>
    <span class="nt">"id"</span><span class="p">:</span> <span class="p">{</span>
      <span class="nt">"type"</span><span class="p">:</span> <span class="s2">"number"</span><span class="p">,</span>
      <span class="nt">"description"</span><span class="p">:</span> <span class="s2">"Product identifier"</span>
    <span class="p">},</span>
    <span class="nt">"name"</span><span class="p">:</span> <span class="p">{</span>
      <span class="nt">"type"</span><span class="p">:</span> <span class="s2">"string"</span><span class="p">,</span>
      <span class="nt">"description"</span><span class="p">:</span> <span class="s2">"Name of the product"</span>
    <span class="p">},</span>
    <span class="nt">"price"</span><span class="p">:</span> <span class="p">{</span>
      <span class="nt">"type"</span><span class="p">:</span> <span class="s2">"number"</span><span class="p">,</span>
      <span class="nt">"minimum"</span><span class="p">:</span> <span class="mi">0</span>
    <span class="p">},</span>
    <span class="nt">"tags"</span><span class="p">:</span> <span class="p">{</span>
      <span class="nt">"type"</span><span class="p">:</span> <span class="s2">"array"</span><span class="p">,</span>
      <span class="nt">"items"</span><span class="p">:</span> <span class="p">{</span>
        <span class="nt">"type"</span><span class="p">:</span> <span class="s2">"string"</span>
      <span class="p">}</span>
    <span class="p">},</span>
    <span class="nt">"stock"</span><span class="p">:</span> <span class="p">{</span>
      <span class="nt">"type"</span><span class="p">:</span> <span class="s2">"object"</span><span class="p">,</span>
      <span class="nt">"properties"</span><span class="p">:</span> <span class="p">{</span>
        <span class="nt">"warehouse"</span><span class="p">:</span> <span class="p">{</span>
          <span class="nt">"type"</span><span class="p">:</span> <span class="s2">"number"</span>
        <span class="p">},</span>
        <span class="nt">"retail"</span><span class="p">:</span> <span class="p">{</span>
          <span class="nt">"type"</span><span class="p">:</span> <span class="s2">"number"</span>
        <span class="p">}</span>
      <span class="p">}</span>
    <span class="p">}</span>
  <span class="p">}</span>
<span class="p">}</span>
</pre></div>
<p>The JSON Schema above can be used to test the validity of the JSON code below:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>

  <span class="nt">"id"</span><span class="p">:</span> <span class="mi">1</span><span class="p">,</span>
  <span class="nt">"name"</span><span class="p">:</span> <span class="s2">"Foo"</span><span class="p">,</span>
  <span class="nt">"price"</span><span class="p">:</span> <span class="mi">123</span><span class="p">,</span>
  <span class="nt">"tags"</span><span class="p">:</span> <span class="p">[</span>
    <span class="s2">"Bar"</span><span class="p">,</span>
    <span class="s2">"Eek"</span>
  <span class="p">],</span>
  <span class="nt">"stock"</span><span class="p">:</span> <span class="p">{</span>
    <span class="nt">"warehouse"</span><span class="p">:</span> <span class="mi">300</span><span class="p">,</span>

  <span class="p">}</span>

<h2><span class="mw-headline" id="MIME_type">MIME type</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=9" title="Edit section: MIME type">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The official <a class="mw-redirect" href="/wiki/MIME_type" title="MIME type">MIME type</a> for JSON text is "<code>application/json</code>".<sup class="reference" id="cite_ref-21"><a href="#cite_note-21">[20]</a></sup> Although most modern implementations have adopted the official MIME type, many applications continue to provide legacy support for other MIME types. Many service providers, browsers, servers, web applications, libraries, frameworks, and APIs use, expect, or recognize the (unofficial) MIME type "<code>text/json</code>" or the content-type "<code>text/javascript</code>". Notable examples include the Google Search API,<sup class="reference" id="cite_ref-faraday-stack-pull-2_22-0"><a href="#cite_note-faraday-stack-pull-2-22">[21]</a></sup> Yahoo!,<sup class="reference" id="cite_ref-faraday-stack-pull-2_22-1"><a href="#cite_note-faraday-stack-pull-2-22">[21]</a></sup><sup class="reference" id="cite_ref-23"><a href="#cite_note-23">[22]</a></sup> Flickr,<sup class="reference" id="cite_ref-faraday-stack-pull-2_22-2"><a href="#cite_note-faraday-stack-pull-2-22">[21]</a></sup> Facebook API,<sup class="reference" id="cite_ref-24"><a href="#cite_note-24">[23]</a></sup> <a href="/wiki/Lift_(web_framework)" title="Lift (web framework)">Lift framework</a>,<sup class="reference" id="cite_ref-25"><a href="#cite_note-25">[24]</a></sup> Dojo Toolkit 0.4,<sup class="reference" id="cite_ref-26"><a href="#cite_note-26">[25]</a></sup> etc.</p>
<h2><span class="mw-headline" id="Applications">Applications</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=10" title="Edit section: Applications">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="JSON-RPC">JSON-RPC</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=11" title="Edit section: JSON-RPC">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/JSON-RPC" title="JSON-RPC">JSON-RPC</a> is a <a href="/wiki/Remote_procedure_call" title="Remote procedure call">remote procedure call</a> (RPC) protocol built on JSON, as a replacement for <a href="/wiki/XML-RPC" title="XML-RPC">XML-RPC</a> or <a href="/wiki/SOAP" title="SOAP">SOAP</a>. It is a simple protocol that defines only a handful of data types and commands. JSON-RPC lets a system send notifications (information to the server that does not require a response) and multiple calls to the server that can be answered out of order. Example of a JSON-RPC 2.0 request and response using positional parameters.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="o">--&gt;</span> <span class="p">{</span><span class="s2">"jsonrpc"</span><span class="o">:</span> <span class="s2">"2.0"</span><span class="p">,</span> <span class="s2">"method"</span><span class="o">:</span> <span class="s2">"subtract"</span><span class="p">,</span> <span class="s2">"params"</span><span class="o">:</span> <span class="p">[</span><span class="mi">42</span><span class="p">,</span> <span class="mi">23</span><span class="p">],</span> <span class="s2">"id"</span><span class="o">:</span> <span class="mi">1</span><span class="p">}</span>
<span class="o">&lt;--</span> <span class="p">{</span><span class="s2">"jsonrpc"</span><span class="o">:</span> <span class="s2">"2.0"</span><span class="p">,</span> <span class="s2">"result"</span><span class="o">:</span> <span class="mi">19</span><span class="p">,</span> <span class="s2">"id"</span><span class="o">:</span> <span class="mi">1</span><span class="p">}</span>
</pre></div>
<h3><span class="mw-headline" id="AJAJ">AJAJ</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=12" title="Edit section: AJAJ">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a class="mw-redirect" href="/wiki/Asynchronous_JavaScript_and_JSON" title="Asynchronous JavaScript and JSON">Asynchronous JavaScript and JSON</a> (or AJAJ) refers to the same <a href="/wiki/Dynamic_web_page" title="Dynamic web page">dynamic web page</a> methodology as <a href="/wiki/Ajax_(programming)" title="Ajax (programming)">Ajax</a>, but instead of <a href="/wiki/XML" title="XML">XML</a>, JSON is the data format. AJAJ is a web development technique that provides for the ability of a <a class="mw-redirect" href="/wiki/Webpage" title="Webpage">webpage</a> to request new data after it has loaded into the <a href="/wiki/Web_browser" title="Web browser">web browser</a>. Typically it renders new data from the server in response to user actions on that webpage. For example, what the user types into a <a href="/wiki/Text_box" title="Text box">search box</a>, <a href="/wiki/Client-side_scripting" title="Client-side scripting">client-side code</a> then sends to the server, which immediately responds with a <a href="/wiki/Drop-down_list" title="Drop-down list">drop-down list</a> of matching <a href="/wiki/Database" title="Database">database</a> items.</p>
<p>The following JavaScript code is an example of a <a href="/wiki/Client-side" title="Client-side">client</a> using <a href="/wiki/XMLHttpRequest" title="XMLHttpRequest">XMLHttpRequest</a> to request data in JSON format from a server. (The <a href="/wiki/Server-side" title="Server-side">server-side</a> programming is omitted; it must be set up to service requests to the <code>url</code> containing a JSON-formatted string.)</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="kd">var</span> <span class="nx">my_JSON_object</span><span class="p">;</span>
<span class="kd">var</span> <span class="nx">http_request</span> <span class="o">=</span> <span class="k">new</span> <span class="nx">XMLHttpRequest</span><span class="p">();</span>
<span class="nx">http_request</span><span class="p">.</span><span class="nx">open</span><span class="p">(</span><span class="s2">"GET"</span><span class="p">,</span> <span class="nx">url</span><span class="p">,</span> <span class="kc">true</span><span class="p">);</span>
<span class="nx">http_request</span><span class="p">.</span><span class="nx">responseType</span> <span class="o">=</span> <span class="s2">"json"</span><span class="p">;</span>
<span class="nx">http_request</span><span class="p">.</span><span class="nx">onreadystatechange</span> <span class="o">=</span> <span class="kd">function</span> <span class="p">()</span> <span class="p">{</span>
  <span class="kd">var</span> <span class="nx">done</span> <span class="o">=</span> <span class="mi">4</span><span class="p">,</span> <span class="nx">ok</span> <span class="o">=</span> <span class="mi">200</span><span class="p">;</span>
  <span class="k">if</span> <span class="p">(</span><span class="nx">http_request</span><span class="p">.</span><span class="nx">readyState</span> <span class="o">===</span> <span class="nx">done</span> <span class="o">&amp;&amp;</span> <span class="nx">http_request</span><span class="p">.</span><span class="nx">status</span> <span class="o">===</span> <span class="nx">ok</span><span class="p">)</span> <span class="p">{</span>
    <span class="nx">my_JSON_object</span> <span class="o">=</span> <span class="nx">http_request</span><span class="p">.</span><span class="nx">response</span><span class="p">;</span>
  <span class="p">}</span>
<span class="p">};</span>
<span class="nx">http_request</span><span class="p">.</span><span class="nx">send</span><span class="p">(</span><span class="kc">null</span><span class="p">);</span>
</pre></div>
<h2><span class="mw-headline" id="Security_issues">Security issues</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=13" title="Edit section: Security issues">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>JSON is intended as a <a class="mw-redirect" href="/wiki/Data_serialization" title="Data serialization">data serialization</a> format. However, its design as a non-strict subset of the JavaScript scripting language poses several security concerns. These concerns center on the use of a JavaScript interpreter to execute JSON text dynamically as embedded JavaScript. This exposes a program to errant or malicious scripts. This is a serious issue when dealing with data retrieved from the Internet. This easy and popular but risky technique exploits JSON's compatibility with the JavaScript <code class="mw-highlight" dir="ltr" id="" style=""><span class="nb">eval</span><span class="p">()</span></code> function, which is described below.</p>
<h3><span class="mw-headline" id="JavaScript_eval.28.29">JavaScript <code class="mw-highlight" dir="ltr" id="" style=""><span class="nb">eval</span><span class="p">()</span></code></span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=14" title="Edit section: JavaScript eval()">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Some developers mistakenly believe that JSON-formatted text is also syntactically legal JavaScript code though certain valid JSON strings are actually not valid JavaScript code.<sup class="reference" id="cite_ref-Magnus_Holm_27-0"><a href="#cite_note-Magnus_Holm-27">[26]</a></sup> Because of that, they believe an easy way for a JavaScript program to parse JSON-formatted data would be to use the built-in JavaScript <code class="mw-highlight" dir="ltr" id="" style=""><span class="nb">eval</span><span class="p">()</span></code> function, which was designed to evaluate JavaScript <a href="/wiki/Expression_(computer_science)" title="Expression (computer science)">expressions</a>. Rather than using a JSON-specific parser, the JavaScript interpreter itself is thus used to execute the JSON data, producing native JavaScript objects. This technique is risky, however, if there is any chance that the JSON data might contain arbitrary JavaScript code, which would then be executed also.</p>
<p>Unless precautions are taken to validate the data first, the eval technique is subject to <a href="/wiki/Vulnerability_(computing)" title="Vulnerability (computing)">security vulnerabilities</a> when the data and the entire JavaScript environment are not within the control of a single <a href="/wiki/Trusted_system" title="Trusted system">trusted source</a>. For example, if the data is itself not trusted, it is subject to malicious JavaScript <a href="/wiki/Code_injection" title="Code injection">code injection</a> attacks. Such breaches of trust also can create vulnerabilities for <a href="/wiki/Data_theft" title="Data theft">data theft</a>, <a href="/wiki/Digital_identity" title="Digital identity">authentication forgery</a>, and other potential misuses of data and resources.</p>
<p>The RFC that defines JSON (<a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc4627" rel="nofollow">RFC 4627</a>) suggests using the following code to validate JSON using a <a href="/wiki/Regular_expression" title="Regular expression">regular expression</a> before evaluating it (the variable 'text' is the input JSON):<sup class="reference" id="cite_ref-28"><a href="#cite_note-28">[27]</a></sup></p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="kd">var</span> <span class="nx">my_JSON_object</span> <span class="o">=</span> <span class="o">!</span><span class="p">(</span><span class="sr">/[^,:{}\[\]0-9.\-+Eaeflnr-u \n\r\t]/</span><span class="p">.</span><span class="nx">test</span><span class="p">(</span>
  <span class="nx">text</span><span class="p">.</span><span class="nx">replace</span><span class="p">(</span><span class="sr">/"(\\.|[^"\\])*"/g</span><span class="p">,</span> <span class="s1">''</span><span class="p">)))</span> <span class="o">&amp;&amp;</span> <span class="nb">eval</span><span class="p">(</span><span class="s1">'('</span> <span class="o">+</span> <span class="nx">text</span> <span class="o">+</span> <span class="s1">')'</span><span class="p">);</span>
</pre></div>
<p>However, this validation is now known to be insufficient.<sup class="reference" id="cite_ref-29"><a href="#cite_note-29">[28]</a></sup></p>
<p>A new function, <code>JSON.parse()</code>, was thus developed as a safer alternative to <code>eval</code>. It is specifically intended to process JSON data and not JavaScript. It was originally planned for inclusion in the Fourth Edition of the <a href="/wiki/ECMAScript" title="ECMAScript">ECMAScript</a> standard,<sup class="reference" id="cite_ref-30"><a href="#cite_note-30">[29]</a></sup> but this did not occur. It was first added to the Fifth Edition,<sup class="reference" id="cite_ref-31"><a href="#cite_note-31">[30]</a></sup> and is now supported by the major browsers given below. For older ones, a compatible JavaScript library is available at JSON.org.</p>
<p>An additional issue when parsing JSON using the <code class="mw-highlight" dir="ltr" id="" style=""><span class="nb">eval</span><span class="p">()</span></code> function is that there are some Unicode characters that are valid in JSON strings but invalid in JavaScript, so additional escaping may be needed in some cases.<sup class="reference" id="cite_ref-Magnus_Holm_27-1"><a href="#cite_note-Magnus_Holm-27">[26]</a></sup></p>
<h3><span class="mw-headline" id="Native_encoding_and_decoding_in_browsers">Native encoding and decoding in browsers</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=15" title="Edit section: Native encoding and decoding in browsers">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Web browsers now have or plan to have native JSON encoding and decoding. This eliminates the <code class="mw-highlight" dir="ltr" id="" style=""><span class="nb">eval</span><span class="p">()</span></code> security problem above, and can increase performance compared to the JavaScript libraries commonly used before. As of June 2009, the following browsers have native JSON support, via <code>JSON.parse()</code> and <code>JSON.stringify()</code>:</p>
<ul>
<li><a class="mw-redirect" href="/wiki/Mozilla_Firefox_3.5" title="Mozilla Firefox 3.5">Mozilla Firefox 3.5</a>+<sup class="reference" id="cite_ref-32"><a href="#cite_note-32">[31]</a></sup></li>
<li>Microsoft <a href="/wiki/Internet_Explorer" title="Internet Explorer">Internet Explorer</a> 8+<sup class="reference" id="cite_ref-33"><a href="#cite_note-33">[32]</a></sup></li>
<li><a href="/wiki/Opera_(web_browser)" title="Opera (web browser)">Opera 10.5</a>+<sup class="reference" id="cite_ref-34"><a href="#cite_note-34">[33]</a></sup></li>
<li><a href="/wiki/WebKit" title="WebKit">WebKit</a>-based browsers (Apple Safari)<sup class="reference" id="cite_ref-35"><a href="#cite_note-35">[34]</a></sup></li>
<li><a class="mw-redirect" href="/wiki/Blink_(layout_engine)" title="Blink (layout engine)">Blink</a>-based browsers (e.g. Google Chrome, Opera)</li>
</ul>
<p>At least five popular JavaScript libraries have committed to use native JSON, if available:</p>
<ul>
<li><a href="/wiki/YUI_Library" title="YUI Library">YUI Library</a><sup class="reference" id="cite_ref-36"><a href="#cite_note-36">[35]</a></sup></li>
<li><a href="/wiki/Prototype_JavaScript_Framework" title="Prototype JavaScript Framework">Prototype</a><sup class="reference" id="cite_ref-37"><a href="#cite_note-37">[36]</a></sup></li>
<li><a href="/wiki/JQuery" title="JQuery">jQuery</a><sup class="reference" id="cite_ref-38"><a href="#cite_note-38">[37]</a></sup></li>
<li><a href="/wiki/Dojo_Toolkit" title="Dojo Toolkit">Dojo Toolkit</a><sup class="reference" id="cite_ref-39"><a href="#cite_note-39">[38]</a></sup></li>
<li><a href="/wiki/MooTools" title="MooTools">MooTools</a><sup class="reference" id="cite_ref-40"><a href="#cite_note-40">[39]</a></sup></li>
</ul>
<h3><span class="mw-headline" id="Implementation-specific_issues">Implementation-specific issues</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=16" title="Edit section: Implementation-specific issues">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Various JSON parser implementations suffered in the past<sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Manual_of_Style/Dates_and_numbers#Chronological_items" title="Wikipedia:Manual of Style/Dates and numbers"><span title="The time period mentioned near this tag is ambiguous. (September 2016)">when?</span></a></i>]</sup> from <a href="/wiki/Denial-of-service_attack" title="Denial-of-service attack">denial-of-service attack</a> and <a href="/wiki/Mass_assignment_vulnerability" title="Mass assignment vulnerability">mass assignment vulnerability</a>.<sup class="reference" id="cite_ref-41"><a href="#cite_note-41">[40]</a></sup><sup class="reference" id="cite_ref-42"><a href="#cite_note-42">[41]</a></sup></p>
<h2><span class="mw-headline" id="Object_references">Object references</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=17" title="Edit section: Object references">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The JSON standard does not support object <a href="/wiki/Reference_(computer_science)" title="Reference (computer science)">references</a>, but an <a class="mw-redirect" href="/wiki/IETF" title="IETF">IETF</a> draft standard for JSON-based object references exists.<sup class="reference" id="cite_ref-43"><a href="#cite_note-43">[42]</a></sup> The <a href="/wiki/Dojo_Toolkit" title="Dojo Toolkit">Dojo Toolkit</a> supports object references using standard JSON; specifically, the <code>dojox.json.ref</code> module provides support for several forms of referencing including <a href="/wiki/Circular_reference" title="Circular reference">circular</a>, multiple, inter-message, and <a href="/wiki/Lazy_evaluation" title="Lazy evaluation">lazy</a> referencing.<sup class="reference" id="cite_ref-44"><a href="#cite_note-44">[43]</a></sup><sup class="reference" id="cite_ref-45"><a href="#cite_note-45">[44]</a></sup><sup class="reference" id="cite_ref-46"><a href="#cite_note-46">[45]</a></sup> Alternatively, non-standard solutions exist such as the use of Mozilla JavaScript Sharp Variables. However this functionality became obsolete with JavaScript 1.8.5 and was removed in Firefox version 12.<sup class="reference" id="cite_ref-47"><a href="#cite_note-47">[46]</a></sup></p>
<h2><span class="mw-headline" id="Comparison_with_other_formats">Comparison with other formats</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=18" title="Edit section: Comparison with other formats">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="hatnote" role="note">See also: <a href="/wiki/Comparison_of_data_serialization_formats" title="Comparison of data serialization formats">Comparison of data serialization formats</a></div>
<p>JSON is promoted as a low-overhead alternative to XML as both of these formats have widespread support for creation, reading, and decoding in the real-world situations where they are commonly used.<sup class="reference" id="cite_ref-48"><a href="#cite_note-48">[47]</a></sup> Apart from XML, examples could include <a href="/wiki/OGDL" title="OGDL">OGDL</a>, <a href="/wiki/YAML" title="YAML">YAML</a> and <a href="/wiki/Comma-separated_values" title="Comma-separated values">CSV</a>. Also, <a href="/wiki/Protocol_Buffers" title="Protocol Buffers">Google Protocol Buffers</a> can fill this role, although it is not a data interchange language.</p>
<h3><span class="mw-headline" id="YAML">YAML</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=19" title="Edit section: YAML">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/YAML" title="YAML">YAML</a> version 1.2 is a superset of JSON; prior versions were "not strictly compatible". For example, escaping a slash (/) with a backslash (\) is valid in JSON, but was not valid in YAML. (This is common practice when injecting JSON into HTML to protect against <a href="/wiki/Cross-site_scripting" title="Cross-site scripting">cross-site scripting attacks</a>.) Nonetheless, many YAML parsers can natively parse the output from many JSON encoders.<sup class="reference" id="cite_ref-YAML_Version_1.2_49-0"><a href="#cite_note-YAML_Version_1.2-49">[48]</a></sup></p>
<h3><span class="mw-headline" id="XML">XML</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=20" title="Edit section: XML">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/XML" title="XML">XML</a> has been used to describe structured data and to serialize objects. Various XML-based protocols exist to represent the same kind of data structures as JSON for the same kind of data interchange purposes. Data can be encoded in XML several ways. The most expansive form using tag pairs results in a much larger representation than JSON, but if data is stored in attributes and 'short tag' form where the closing tag is replaced with '/&gt;', the representation is often about the same size as JSON or just a little larger. If the data is compressed using an algorithm like <a href="/wiki/Gzip" title="Gzip">gzip</a>, there is little difference because compression is good at saving space when a pattern is repeated.</p>
<p>XML also has the concept of <a href="/wiki/XML_schema" title="XML schema">schema</a>. This permits strong typing, user-defined types, predefined tags, and formal structure, allowing for formal validation of an XML stream in a portable way. There is, however, an <a class="mw-redirect" href="/wiki/IETF" title="IETF">IETF</a> draft proposal for a schema system for JSON <a class="external autonumber" href="http://json-schema.org/documentation.html" rel="nofollow">[1]</a>.</p>
<p>XML supports comments, but JSON does not.<sup class="reference" id="cite_ref-Saternos45_50-0"><a href="#cite_note-Saternos45-50">[49]</a></sup></p>

<p>The properties can also be serialized using attributes instead of tags:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="nt">&lt;person</span> <span class="na">firstName=</span><span class="s">"John"</span> <span class="na">lastName=</span><span class="s">"Smith"</span> <span class="na">age=</span><span class="s">"25"</span><span class="nt">&gt;</span>
  <span class="nt">&lt;address</span> <span class="na">streetAddress=</span><span class="s">"21 2nd Street"</span> <span class="na">city=</span><span class="s">"New York"</span> <span class="na">state=</span><span class="s">"NY"</span> <span class="na">postalCode=</span><span class="s">"10021"</span> <span class="nt">/&gt;</span>

<p>The XML encoding <i>may</i> therefore be comparable in length to the equivalent JSON encoding. A wide range of XML processing technologies exist, from the <a href="/wiki/Document_Object_Model" title="Document Object Model">Document Object Model</a> to <a href="/wiki/XPath" title="XPath">XPath</a> and <a href="/wiki/XSLT" title="XSLT">XSLT</a>. XML can also be styled for immediate display using <a href="/wiki/Cascading_Style_Sheets" title="Cascading Style Sheets">CSS</a>. <a href="/wiki/XHTML" title="XHTML">XHTML</a> is a form of XML so that elements can be passed in this form ready for direct insertion into webpages using client-side scripting.</p>

<h2><span class="mw-headline" id="Notes">Notes</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=26" title="Edit section: Notes">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="reflist" style="list-style-type: decimal;">
<ol class="references">
<li id="cite_note-undefinedType-14"><span class="mw-cite-backlink"><b><a href="#cite_ref-undefinedType_14-0">^</a></b></span> <span class="reference-text">The <code>undefined</code> type was left out of the JSON standard, and one finds suggestions that <code>null</code> be used instead. In fact, the current standard says that for a sparse array such as:</span>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="reference-text"> <span class="kd">var</span> <span class="nx">v</span> <span class="o">=</span> <span class="p">[</span><span class="mi">0</span><span class="p">];</span>
 <span class="nx">v</span><span class="p">[</span><span class="mi">3</span><span class="p">]</span> <span class="o">=</span> <span class="mi">3</span><span class="p">;</span>
</span>
</pre></div>
<p>which behaves in JavaScript as if it were:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
 <span class="kd">var</span> <span class="nx">vx</span> <span class="o">=</span> <span class="p">[</span><span class="mi">0</span><span class="p">,</span> <span class="kc">undefined</span><span class="p">,</span> <span class="kc">undefined</span><span class="p">,</span> <span class="mi">3</span><span class="p">];</span>
</pre></div>
<p>with the <code>undefined</code> entries being only implicit rather than explicit, should translate to JSON as if it were:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
 <span class="kd">var</span> <span class="nx">vx</span> <span class="o">=</span> <span class="p">[</span><span class="mi">0</span><span class="p">,</span> <span class="kc">null</span><span class="p">,</span> <span class="kc">null</span><span class="p">,</span> <span class="mi">3</span><span class="p">];</span>
</pre></div>
<p>with explicit <code>null</code> fillers for the undefined entries.</p>
<p>Furthermore, in JavaScript <code>{a: undefined}</code> often behaves the same as <code>{}</code>. Both translate as <code>"{}"</code> in JSON. However <code>undefined</code> as an explicit property value does have use in JavaScript inheritance situations such as:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
 <span class="kd">var</span> <span class="nx">x</span> <span class="o">=</span> <span class="p">{</span><span class="nx">a</span><span class="o">:</span> <span class="mi">1</span><span class="p">};</span>
 <span class="kd">var</span> <span class="nx">xi</span> <span class="o">=</span> <span class="nb">Object</span><span class="p">.</span><span class="nx">create</span><span class="p">(</span><span class="nx">x</span><span class="p">);</span>
 <span class="nx">xi</span><span class="p">.</span><span class="nx">a</span> <span class="o">=</span> <span class="kc">undefined</span><span class="p">;</span>
</pre></div>
<p>where the inheritance of <code>x</code>'s property <code>a</code> is overridden in <code>xi</code> and makes it pretty much behave as if nothing was inherited. <code>JSON.stringify</code> itself ignores inherited values - it only translates the enumerable own properties as given by <code>Object.keys(y)</code>. The default stringification, while not encoding inheritance, can (except for <code>undefined</code> values) encode enough of an object to reconstruct it in an environment that knows what inheritance it should have. To encode JavaScript objects that contain explicit <code>undefined</code> values a convention for representing <code>undefined</code> must be established, such as mapping it to the string <code>"UNDEFINED"</code>. One can then pass <code>JSON.stringify</code> the optional <code>replacer</code> argument to translate with this convention:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
 <span class="kd">var</span> <span class="nx">y</span> <span class="o">=</span> <span class="p">{</span><span class="nx">a</span><span class="o">:</span> <span class="kc">undefined</span><span class="p">};</span>
 <span class="kd">var</span> <span class="nx">ys</span> <span class="o">=</span> <span class="nx">JSON</span><span class="p">.</span><span class="nx">stringify</span><span class="p">(</span><span class="nx">y</span><span class="p">,</span>
  <span class="kd">function</span> <span class="p">(</span><span class="nx">k</span><span class="p">,</span> <span class="nx">v</span><span class="p">){</span><span class="k">return</span> <span class="p">(</span><span class="nx">v</span> <span class="o">===</span> <span class="kc">undefined</span><span class="p">)</span> <span class="o">?</span> <span class="s2">"UNDEFINED"</span> <span class="o">:</span> <span class="nx">v</span><span class="p">});</span>
</pre></div>
<p>Converting this JSON back into JavaScript is not as straightforward. While <code>JSON.parse</code> can take an optional <code>reviver</code> argument that is, essentially, the inverse of a <code>replacer</code>, it can't be used in this situation. If that function returns <code>undefined</code>, the <code>JSON.parse</code> logic interprets this to mean to not define a property rather than define one with a <code>undefined</code> value. Instead one has to explicitly post process the result from <code>JSON.parse</code> replacing each <code>"UNDEFINED"</code> with <code>undefined</code>.</p>
</li>
</ol>
</div>
