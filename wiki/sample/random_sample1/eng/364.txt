<p><b>Audio Video Interleaved</b> (also <b>Audio Video Interleave</b>), known by its initials <b>AVI</b>, is a multimedia <a class="mw-redirect" href="/wiki/Container_format_(digital)" title="Container format (digital)">container format</a> introduced by <a href="/wiki/Microsoft" title="Microsoft">Microsoft</a> in November 1992 as part of its <a href="/wiki/Video_for_Windows" title="Video for Windows">Video for Windows</a> software. AVI <a href="/wiki/Computer_file" title="Computer file">files</a> can contain both audio and video data in a file container that allows synchronous audio-with-video playback. Like the <a href="/wiki/DVD-Video" title="DVD-Video">DVD video format</a>, AVI files support multiple streaming audio and video, although these features are seldom used. Most AVI files also use the file format extensions developed by the <a href="/wiki/Matrox" title="Matrox">Matrox</a> OpenDML group in February 1996. These files are supported by Microsoft, and are unofficially called "AVI 2.0".</p>

<h2><span class="mw-headline" id="Format">Format</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Audio_Video_Interleave&amp;action=edit&amp;section=1" title="Edit section: Format">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>AVI is a derivative of the <a class="mw-redirect" href="/wiki/RIFF_(File_format)" title="RIFF (File format)">Resource Interchange File Format</a> (RIFF), which divides a file's data into blocks, or "chunks." Each "chunk" is identified by a <a href="/wiki/FourCC" title="FourCC">FourCC</a> tag. An AVI file takes the form of a single "chunk" in a RIFF formatted file, which is then subdivided into two mandatory "chunks" and one optional "chunk".</p>
<p>The first sub-chunk is identified by the "hdrl" tag. This sub-chunk is the file header and contains metadata about the video, such as its width, height and <a href="/wiki/Frame_rate" title="Frame rate">frame rate</a>. The second sub-chunk is identified by the "movi" tag. This chunk contains the actual audio/visual data that make up the AVI movie. The third optional sub-chunk is identified by the "idx1" tag which indexes the <a href="/wiki/Offset_(computer_science)" title="Offset (computer science)">offsets</a> of the data chunks within the file.</p>
<p>By way of the RIFF format, the audio-visual data contained in the "movi" chunk can be encoded or decoded by software called a <a href="/wiki/Codec" title="Codec">codec</a>, which is an abbreviation for (en)coder/decoder. Upon creation of the file, the codec translates between raw data and the (compressed) data format used inside the chunk. An AVI file may carry audio/visual data inside the chunks in virtually any compression scheme, including Full Frame (Uncompressed), Intel Real Time (<a href="/wiki/Indeo" title="Indeo">Indeo</a>), <a href="/wiki/Cinepak" title="Cinepak">Cinepak</a>, <a href="/wiki/Motion_JPEG" title="Motion JPEG">Motion JPEG</a>, Editable <a class="mw-redirect" href="/wiki/MPEG" title="MPEG">MPEG</a>, VDOWave, ClearVideo / <a href="/wiki/RealVideo" title="RealVideo">RealVideo</a>, <a href="/wiki/QPEG" title="QPEG">QPEG</a>, and <a href="/wiki/MPEG-4" title="MPEG-4">MPEG-4</a> Video.</p>
<h2><span class="mw-headline" id="Metadata">Metadata</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Audio_Video_Interleave&amp;action=edit&amp;section=2" title="Edit section: Metadata">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>As a derivative of the <a href="/wiki/Resource_Interchange_File_Format" title="Resource Interchange File Format">Resource Interchange File Format</a> (RIFF), AVI files are commonly tagged with <a href="/wiki/Metadata" title="Metadata">metadata</a> in the INFO chunk. In addition, AVI files can embed <a href="/wiki/Extensible_Metadata_Platform" title="Extensible Metadata Platform">Extensible Metadata Platform</a> (XMP). By design, any RIFF file can legally include additional chunks of data, each identified by a four-character code; software which does not understand that particular code should skip the chunk. As such, it is theoretically possible to expand any RIFF file format, including AVI, to support almost any conceivable metadata. Some of the limitations of AVI in modern use relate to a lack of standardization in this metadata (see Limitations below).</p>
<h2><span class="mw-headline" id="Limitations">Limitations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Audio_Video_Interleave&amp;action=edit&amp;section=3" title="Edit section: Limitations">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Since its introduction in the early 90s, new computer video techniques have been introduced which the original AVI specification did not anticipate.</p>
<ul>
<li>AVI does not provide a standardized way to encode <a href="/wiki/Aspect_ratio" title="Aspect ratio">aspect ratio</a> information, with the result that players cannot select the right aspect ratio automatically (though it may be possible to do so manually).<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup></li>
<li>There are several competing approaches to including a <a class="mw-redirect" href="/wiki/Time_code" title="Time code">time code</a> in AVI files, which affects usability of the format in film and television post-production, although it is widely used. For WAV audio files, <a href="/wiki/Broadcast_Wave_Format" title="Broadcast Wave Format">Broadcast Wave</a> extensions were designed to standardize post-production metadata, but an equivalent for AVI files has not emerged.</li>
<li>AVI was not intended to contain video using any compression technique that requires access to future video frame data beyond the current frame. Approaches exist to support modern video compression techniques (such as <a href="/wiki/MPEG-4" title="MPEG-4">MPEG-4</a>) that rely on this function, although this is beyond the intent of the original specification and may cause problems with playback software which does not anticipate this use.</li>
<li>AVI cannot contain some specific types of <a href="/wiki/Variable_bitrate" title="Variable bitrate">variable bitrate</a> (VBR) data reliably (such as MP3 audio at sample rates below 32 kHz).</li>
<li>Overhead for AVI files at the resolutions and frame rates normally used to encode standard definition feature films is about 5 MB per hour of video, the significance of which varies with the application.</li>
</ul>
<p>More recent container formats (such as <a href="/wiki/Matroska" title="Matroska">Matroska</a>, <a href="/wiki/Ogg" title="Ogg">Ogg</a> and <a class="mw-redirect" href="/wiki/MP4" title="MP4">MP4</a>) solve all these problems, although software is freely available to both create and correctly replay AVI files which use the techniques described here.</p>
<h2><span class="mw-headline" id="DV_AVI">DV AVI</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Audio_Video_Interleave&amp;action=edit&amp;section=4" title="Edit section: DV AVI">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>DV AVI is a type of AVI file where the video has been compressed to conform with <a href="/wiki/DV" title="DV">DV</a> standards. There are two types of DV-AVI files:</p>
<ul>
<li>Type 1: The multiplexed Audio-Video is kept in its original multiplexing and saved together into the Video section of the AVI file
<ul>
<li>Does not waste much space (audio is saved uncompressed, but even uncompressed audio is tiny compared to the video part of DV), but Windows applications based on the <a href="/wiki/Video_for_Windows" title="Video for Windows">VfW</a> <a href="/wiki/Application_programming_interface" title="Application programming interface">API</a> do not support it.</li>
</ul>
</li>
<li>Type 2: Like type 1, but audio is also saved as an additional audio stream into the file.
<ul>
<li>Supported by VfW applications, at the price of a small increase in file size.</li>
</ul>
</li>
</ul>
<p>Type 1 is actually the newer of the two types. Microsoft made the "type" designations, and decided to name their older VfW-compatible version "Type 2", which only furthered confusion about the two types. In the late 1990s through early 2000s, most professional-level DV software, including non-linear editing programs, only supported Type 1. One notable exception was <a class="mw-redirect" href="/wiki/Adobe_Premiere" title="Adobe Premiere">Adobe Premiere</a>, which only supported Type 2. High-end FireWire controllers usually captured to Type 1 only, while "consumer" level controllers usually captured to Type 2 only. Software is and was available for converting Type 1 AVIs to Type 2, and vice versa, but this is a time-consuming process.</p>
<p>Many current FireWire controllers still only capture to one or the other type. However, almost all current DV software supports both Type 1 and Type 2 editing and rendering, including Adobe Premiere. Thus, many of today's users are unaware of the fact that there are two types of DV AVI files. In any event, the debate continues as to which – Type 1 or Type 2 – if either, is better.</p>
