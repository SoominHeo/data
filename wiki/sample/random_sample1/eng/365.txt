<p>A <b>JAR</b> (<b>J</b>ava <b>AR</b>chive) is a <a href="/wiki/Package_format" title="Package format">package</a> <a href="/wiki/File_format" title="File format">file format</a> typically used to aggregate many <a href="/wiki/Java_class_file" title="Java class file">Java class files</a> and associated <a href="/wiki/Metadata" title="Metadata">metadata</a> and resources (text, images, etc.) into one file for distribution.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup></p>
<p>JAR files are <a href="/wiki/Archive_file" title="Archive file">archive files</a> with which include a Java-specific <a href="/wiki/Manifest_file" title="Manifest file">manifest file</a>. They are built on the <a class="mw-redirect" href="/wiki/ZIP_(file_format)" title="ZIP (file format)">ZIP format</a> and typically have a <code>.jar</code> <a class="mw-redirect" href="/wiki/File_extension" title="File extension">file extension</a>.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></p>

<h2><span class="mw-headline" id="Design">Design</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JAR_(file_format)&amp;action=edit&amp;section=1" title="Edit section: Design">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>A JAR file allows Java runtimes to efficiently <a href="/wiki/Software_deployment" title="Software deployment">deploy</a> an entire application, including its classes and their associated resources, in a single request. JAR file elements may be compressed, shortening download times.</p>
<p>A JAR file may contain a manifest file, located at <code>META-INF/MANIFEST.MF</code>. The entries in the manifest file describe how to use the JAR file. For instance, a <a href="/wiki/Classpath_(Java)" title="Classpath (Java)">Classpath</a> entry can be used to specify other JAR files to load with the JAR.</p>
<h2><span class="mw-headline" id="Extraction">Extraction</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JAR_(file_format)&amp;action=edit&amp;section=2" title="Edit section: Extraction">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The contents of a JAR file may be extracted using any standard decompression software, or the <code>jar</code> command line utility: "<code>jar -xf foo.jar</code>".</p>
<h2><span class="mw-headline" id="Security">Security</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JAR_(file_format)&amp;action=edit&amp;section=3" title="Edit section: Security">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Developers can <a href="/wiki/Digital_signature" title="Digital signature">digitally sign</a> JAR files. In that case, the signature information becomes part of the embedded manifest file. The JAR itself is not signed, but instead every file inside the archive is listed along with its checksum; it is these checksums that are signed. Multiple entities may sign the JAR file, changing the JAR file itself with each signing, although the signed files themselves remain valid. When the Java runtime loads signed JAR files, it can validate the signatures and refuse to load classes that do not match the signature. It can also support 'sealed' packages, in which the <a class="mw-redirect" href="/wiki/Classloader" title="Classloader">Classloader</a> will only permit Java classes to be loaded into the same package if they are all signed by the same entities. This prevents malicious code from being inserted into an existing package, and so gaining access to package-scoped classes and data.</p>
<p>The content of JAR files may be <a class="mw-redirect" href="/wiki/Obfuscated_code" title="Obfuscated code">obfuscated</a> to make <a href="/wiki/Reverse_engineering" title="Reverse engineering">reverse engineering</a> more difficult.</p>
<h2><span class="mw-headline" id="Executable_JAR_files">Executable JAR files</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JAR_(file_format)&amp;action=edit&amp;section=4" title="Edit section: Executable JAR files">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>An executable Java program can be packaged in a JAR file, along with any libraries the program uses. Executable JAR files have the manifest specifying the <a href="/wiki/Entry_point" title="Entry point">entry point</a> class with <code>Main-Class: myPrograms.MyClass</code> and an explicit Class-Path (and the -cp argument is ignored). Some operating systems can run these directly when clicked. The typical invocation is <code>java -jar foo.jar</code> from a command line.</p>
<p>Native launchers can be created on most platforms. For instance, <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a> users who prefer having Windows <a class="mw-redirect" href="/wiki/EXE" title="EXE">EXE</a> files can use tools such as <a href="/wiki/JSmooth" title="JSmooth">JSmooth</a>, Launch4J, WinRun4J or <a href="/wiki/Nullsoft_Scriptable_Install_System" title="Nullsoft Scriptable Install System">Nullsoft Scriptable Install System</a> to wrap single JAR files into executables.</p>
<h2><span class="mw-headline" id="Manifest">Manifest</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JAR_(file_format)&amp;action=edit&amp;section=5" title="Edit section: Manifest">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>A manifest file is a metadata file contained within a JAR.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup><sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup> It defines extension and package-related data. It contains <a class="mw-redirect" href="/wiki/Name-value_pair" title="Name-value pair">name-value pairs</a> organized in sections. If a JAR file is intended to be used as an executable file, the manifest file specifies the main class of the application. The manifest file is named <code>MANIFEST.MF</code>.</p>
<h3><span class="mw-headline" id="Specifications">Specifications</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JAR_(file_format)&amp;action=edit&amp;section=6" title="Edit section: Specifications">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The manifest appears at the canonical location <code>META-INF/MANIFEST.MF</code>.<sup class="reference" id="cite_ref-jarspec_7-0"><a href="#cite_note-jarspec-7">[7]</a></sup> There can be only one manifest file in an archive and it must be at that location.</p>
<p>The content of the manifest file in a JAR file created with version 1.0 of the <a href="/wiki/Java_Development_Kit" title="Java Development Kit">Java Development Kit</a> is the following.</p>
<pre>
Manifest-Version: 1.0
</pre>
<p>The name is separated from its value by a colon. The default manifest shows that it conforms to version 1.0 of the manifest specification.</p>
<p>The manifest can contain information about the other files that are packaged in the archive. Manifest contents depend on the intended use for the JAR file. The default manifest file makes no assumptions about what information it should record about other files, so its single line contains data only about itself.</p>
<h4><span class="mw-headline" id="Special-Purpose_Manifest_Headers">Special-Purpose Manifest Headers</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JAR_(file_format)&amp;action=edit&amp;section=7" title="Edit section: Special-Purpose Manifest Headers">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>JAR files created only for the purpose of archiving do not use the <code>MANIFEST.MF</code> file.</p>
<p>Most uses of JAR files go beyond simple archiving and compression and require special information in the manifest file.</p>
<h4><span class="mw-headline" id="Applications">Applications</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JAR_(file_format)&amp;action=edit&amp;section=8" title="Edit section: Applications">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>If an application is contained in a JAR file, the <a class="mw-redirect" href="/wiki/Java_Virtual_Machine" title="Java Virtual Machine">Java Virtual Machine</a> needs to know the application's entry point. An entry point is any class with a <code>public static void main(String[] args)</code> method. This information is provided in the Main-Class header, which has the general form:</p>
<pre>
Main-Class: com.example.MyClassName
</pre>
<p>In this example <code>com.example.MyClassName.main()</code> executes at application launch.</p>
<h4><span class="mw-headline" id="Package_Sealing">Package Sealing</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JAR_(file_format)&amp;action=edit&amp;section=9" title="Edit section: Package Sealing">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Optionally, a package within a JAR file can be sealed, which means that all classes defined in that package are archived in the same JAR file. A package might be sealed to ensure version consistency among the classes in the software or as a security measure.</p>
<p>To seal a package, a Name entry needs to appear, followed by a Sealed header, such as:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="na">Name</span><span class="o">:</span> <span class="s">myCompany/myPackage/</span>
<span class="na">Sealed</span><span class="o">:</span> <span class="s">true</span>
</pre></div>
<p>The Name header's value is the package's relative pathname. Note that it ends with a '/' to distinguish it from a filename. Any headers following a Name header, without any intervening blank lines, apply to the file or package specified in the Name header. In the above example, because the <code>Sealed</code> header occurs after the <code>Name: myCompany/myPackage</code> header with no intervening blank lines, the <code>Sealed</code> header applies (only) to the package <code>myCompany/myPackage</code>.</p>
<h3><span class="mw-headline" id="Package_Versioning">Package Versioning</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JAR_(file_format)&amp;action=edit&amp;section=10" title="Edit section: Package Versioning">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Several manifest headers hold versioning information. One set of headers can be assigned to each package. The versioning headers appear directly beneath the Name header for the package. This example shows all the versioning headers:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="na">Name</span><span class="o">:</span> <span class="s">java/util/</span>
<span class="na">Specification-Title</span><span class="o">:</span> <span class="s">"Java Utility Classes"</span>
<span class="na">Specification-Version</span><span class="o">:</span> <span class="s">"1.2"</span>
<span class="na">Specification-Vendor</span><span class="o">:</span> <span class="s">"Sun Microsystems, Inc.".</span>
<span class="na">Implementation-Title</span><span class="o">:</span> <span class="s">"java.util"</span>
<span class="na">Implementation-Version</span><span class="o">:</span> <span class="s">"build57"</span>
<span class="na">Implementation-Vendor</span><span class="o">:</span> <span class="s">"Sun Microsystems, Inc."</span>
</pre></div>
<h3><span class="mw-headline" id="Dependencies">Dependencies</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JAR_(file_format)&amp;action=edit&amp;section=11" title="Edit section: Dependencies">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The <code>MANIFEST.MF</code> file can be used to specify all the classes that must be loaded for an application to be able to run.<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup></p>
<p>Note that Class-Path entries are delimited with spaces, not with the system path delimiter:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="na">Class-Path</span><span class="o">:</span> <span class="s">. pkg1.jar path/to/pkg2.jar</span>
</pre></div>
<h2><span class="mw-headline" id="Apache_Ant_Zip.2FJAR_support">Apache Ant Zip/JAR support</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JAR_(file_format)&amp;action=edit&amp;section=12" title="Edit section: Apache Ant Zip/JAR support">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The <a href="/wiki/Apache_Ant" title="Apache Ant">Apache Ant</a> build tool has its own package to read and write Zip and JAR archives, including support for <a href="/wiki/Unix" title="Unix">Unix</a> <a class="mw-redirect" href="/wiki/Filesystem" title="Filesystem">filesystem</a> extensions. The org.apache.tools.zip package is released under the <a href="/wiki/Apache_Software_Foundation" title="Apache Software Foundation">Apache Software Foundation</a> license and is designed to be usable outside Ant.</p>
<h2><span class="mw-headline" id="Related_formats">Related formats</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JAR_(file_format)&amp;action=edit&amp;section=13" title="Edit section: Related formats">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Several related file formats build on the JAR format:</p>
<ul>
<li><a class="mw-redirect" href="/wiki/WAR_(Sun_file_format)" title="WAR (Sun file format)">WAR</a> (Web application archive) files, also Java archives, store <a href="/wiki/XML" title="XML">XML</a> files, Java classes, <a href="/wiki/JavaServer_Pages" title="JavaServer Pages">JavaServer Pages</a> and other objects for Web Applications.</li>
<li><a class="mw-redirect" href="/wiki/Resource_Adapter" title="Resource Adapter">RAR</a> (resource adapter archive) files (not to be confused with the <a class="mw-redirect" href="/wiki/RAR_file_format" title="RAR file format">RAR file format</a>), also Java archives, store XML files, Java classes and other objects for <a class="mw-redirect" href="/wiki/J2EE_Connector_Architecture" title="J2EE Connector Architecture">J2EE Connector Architecture</a> (JCA) applications.</li>
<li><a href="/wiki/EAR_(file_format)" title="EAR (file format)">EAR</a> (enterprise archive) files provide composite Java archives that combine XML files, Java classes and other objects including JAR, WAR and RAR Java archive files for Enterprise Applications.</li>
<li><a class="new" href="/w/index.php?title=Service_Archive&amp;action=edit&amp;redlink=1" title="Service Archive (page does not exist)">SAR</a> (service archive) is similar to EAR. It provides a <tt>service.xml</tt> file and accompanying JAR files.</li>
<li><a class="mw-redirect" href="/wiki/APK_(file_format)" title="APK (file format)">APK</a> (Android Application Package), a variant of the Java archive format, is used for <a href="/wiki/Android_(operating_system)" title="Android (operating system)">Android</a> applications.<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup></li>
<li>PAR (Plan ARchive) - supported by <a href="/wiki/Virgo_(software)" title="Virgo (software)">Eclipse Virgo</a> <a href="/wiki/OSGi" title="OSGi">OSGi</a> application server, allows the deployment of multi-bundle OSGi applications as a single archive and provides isolation from other PAR-based applications deployed in the same server.</li>
<li>KAR (Karaf ARchive) - supported by <a class="mw-redirect" href="/wiki/Apache_Karaf" title="Apache Karaf">Apache Karaf</a> OSGi application server, allows the deployment of multi-bundle, multi-feature OSGi applications.</li>
</ul>
