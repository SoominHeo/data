<p>The <b>Data Distribution Service</b> for real-time systems (<b>DDS</b>) is an <a href="/wiki/Object_Management_Group" title="Object Management Group">Object Management Group</a> (OMG) <a class="mw-redirect" href="/wiki/Machine-to-machine" title="Machine-to-machine">machine-to-machine</a> (sometimes called <a href="/wiki/Middleware" title="Middleware">middleware</a>) standard that aims to enable <a href="/wiki/Scalability" title="Scalability">scalable</a>, <a href="/wiki/Real-time_computing" title="Real-time computing">real-time</a>, <a class="mw-redirect" href="/wiki/Safety-critical" title="Safety-critical">dependable</a>, <a href="/wiki/Many-task_computing" title="Many-task computing">high-performance</a> and <a class="mw-redirect" href="/wiki/Interoperable" title="Interoperable">interoperable</a> <a href="/wiki/Data_exchange" title="Data exchange">data exchanges</a> using a <a href="/wiki/Publish%E2%80%93subscribe_pattern" title="Publish–subscribe pattern">publish–subscribe pattern</a>. DDS addresses the needs of applications like <a href="/wiki/Trader_(finance)" title="Trader (finance)">financial trading</a>, <a class="mw-redirect" href="/wiki/Air-traffic_control" title="Air-traffic control">air-traffic control</a>, <a href="/wiki/Smart_grid" title="Smart grid">smart grid</a> management, and other <a href="/wiki/Big_data" title="Big data">big data</a> applications. The standard is used in applications such as smartphone operating systems,<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> transportation systems and vehicles,<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> <a href="/wiki/Software-defined_radio" title="Software-defined radio">software-defined radio</a>, and by healthcare providers. DDS was promoted for use in the <a href="/wiki/Internet_of_things" title="Internet of things">Internet of things</a>.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup></p>
<p></p>

<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_Distribution_Service&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>A few proprietary DDS systems had been available for several years. Starting in 2001, two vendors, the US government contractor Real-Time Innovations and the French <a href="/wiki/Thales_Group" title="Thales Group">Thales Group</a> teamed up to create the DDS specification, which was subsequently approved by the <a href="/wiki/Object_Management_Group" title="Object Management Group">Object Management Group</a> (OMG) resulting in version 1.0 in published in December 2004.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup> Version 1.1 was published in December 2005,<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> 1.2 in January 2007,<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup> and 1.4 in April 2015.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup> DDS is covered by several US patents,<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup><sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup><sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup><sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup> among others.</p>
<p>The DDS specification describes two levels of interfaces:</p>
<ul>
<li>A lower data-centric publish-subscribe (DCPS) level that is targeted towards the efficient delivery of the proper information to the proper recipients.</li>
<li>An optional higher data local reconstruction layer (DLRL), which allows for a simple integration of DDS into the <a href="/wiki/Application_layer" title="Application layer">application layer</a>.</li>
</ul>
<p>Other related standards followed the initial core document. The Real-time Publish-Subscribe Wire Protocol DDS Interoperability Wire Protocol Specification ensured that information published on a topic using one vendor's DDS implementation is consumable by one or more subscribers using the same or different vendor's DDS implementations. Although the specification is targeted at the DDS community, its use is not limited. Versions 2.0 was published in April 2008, version 2.1 in November 2010, and 2.2 in September 2014.<sup class="reference" id="cite_ref-RtpsRef_12-0"><a href="#cite_note-RtpsRef-12">[12]</a></sup></p>
<p>DDS for Lightweight <a class="mw-redirect" href="/wiki/CORBA_Component_Model" title="CORBA Component Model">CCM</a> (dds4ccm) offers an architectural pattern that separates the business logic from the non-functional properties. A 2012 extension added support for streams.<sup class="reference" id="cite_ref-13"><a href="#cite_note-13">[13]</a></sup> A Java 5 Language PSM for DDS defined a Java 5 language binding, referred to as a Platform Specific Model (PSM) for DDS. It specified only the Data-Centric Publish-Subscribe (DCPS) portion of the DDS specification; Additionally, it encompasses the DDS APIs introduced by DDS-XTypes and DDS-CCM. DDS-PSM-Cxx defines the ISO/IEC C++<sup class="reference" id="cite_ref-14"><a href="#cite_note-14">[14]</a></sup> PSM language binding, referred to as a Platform Specific Model (PSM) for DDS. It provides a new C++ API for programming DDS that is more natural to a C++ programmer.<sup class="reference" id="cite_ref-15"><a href="#cite_note-15">[15]</a></sup> The specification provides mappings for the <a href="/wiki/Application_programming_interface" title="Application programming interface">application programming interface</a> (API) specified in DDS-XTypes, and accessing <a href="/wiki/Quality_of_service" title="Quality of service">quality of service</a> (QoS) profiles specified in DDS-CCM.</p>
<p>Extensible and Dynamic Topic Types for DDS (DDS-XTypes) provided support for data-centric publish-subscribe communication where topics are defined with specific data structures. To be <i>extensible</i>, DDS topics use data types defined before compile time and used throughout the DDS global data space. This model is desirable when static type checking is useful.<sup class="reference" id="cite_ref-16"><a href="#cite_note-16">[16]</a></sup> A <a href="/wiki/Unified_Modeling_Language" title="Unified Modeling Language">Unified Modeling Language</a> (UML) profile specified DDS domains and topics to be part of analysis and design modeling.<sup class="reference" id="cite_ref-17"><a href="#cite_note-17">[17]</a></sup> This specification also defined how to publish and subscribe objects without first describing the types in another language, such as XML or OMG IDL.<sup class="reference" id="cite_ref-18"><a href="#cite_note-18">[18]</a></sup> An <a class="mw-redirect" href="/wiki/Interface_definition_language" title="Interface definition language">interface definition language</a> (IDL) was specified in 2014 independently from the <a href="/wiki/Common_Object_Request_Broker_Architecture" title="Common Object Request Broker Architecture">Common Object Request Broker Architecture</a> (CORBA) specification chapter 3. This IDL 3.5 was compatible with the CORBA 3 specification, but extracted as its own specification so that it can evolve independently from CORBA.<sup class="reference" id="cite_ref-19"><a href="#cite_note-19">[19]</a></sup></p>
<p>Starting with DDS version 1.4 in 2015, the optional DLRL layer was moved to a separate specification.<sup class="reference" id="cite_ref-20"><a href="#cite_note-20">[20]</a></sup></p>
<h2><span class="mw-headline" id="Architecture">Architecture</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_Distribution_Service&amp;action=edit&amp;section=2" title="Edit section: Architecture">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Model">Model</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_Distribution_Service&amp;action=edit&amp;section=3" title="Edit section: Model">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>DDS is networking <a href="/wiki/Middleware" title="Middleware">middleware</a> that simplifies complex <a href="/wiki/Computer_network_programming" title="Computer network programming">network programming</a>. It implements a <a href="/wiki/Publish%E2%80%93subscribe_pattern" title="Publish–subscribe pattern">publish–subscribe pattern</a> for sending and receiving data, events, and commands among the <a href="/wiki/Node_(networking)" title="Node (networking)">nodes</a>. Nodes that produce information (publishers) create "topics" (e.g., temperature, location, pressure) and publish "samples". DDS delivers the samples to subscribers that declare an interest in that topic.</p>
<p>DDS handles transfer chores: message addressing, <a href="/wiki/Serialization" title="Serialization">data marshalling and demarshalling</a> (so subscribers can be on different platforms from the publisher), delivery, flow control, retries, etc. Any node can be a publisher, subscriber, or both simultaneously.</p>
<p>The DDS publish-subscribe model virtually eliminates complex network programming for distributed applications.</p>
<p>DDS supports mechanisms that go beyond the basic publish-subscribe model. The key benefit is that applications that use DDS for their communications are decoupled. Little design time need be spent on handling their mutual interactions. In particular, the applications never need information about the other participating applications, including their existence or locations. DDS transparently handles message delivery without requiring intervention from the user applications, including:</p>
<ul>
<li>determining who should receive the messages</li>
<li>where recipients are located</li>
<li>what happens if messages cannot be delivered</li>
</ul>
<p>DDS allows the user to specify <a href="/wiki/Quality_of_service" title="Quality of service">quality of service</a> (QoS) parameters to configure discovery and behavior mechanisms up-front. By exchanging messages anonymously, DDS simplifies distributed applications and encourages modular, well-structured programs. DDS also automatically handles hot-swapping redundant publishers if the primary fails. Subscribers always get the sample with the highest priority whose data is still valid (that is, whose publisher-specified validity period has not expired). It automatically switches back to the primary when it recovers, too.</p>
<h3><span class="mw-headline" id="Interoperability">Interoperability</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_Distribution_Service&amp;action=edit&amp;section=4" title="Edit section: Interoperability">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Both commercial and <a href="/wiki/Open-source_software" title="Open-source software">open-source software</a> implementations of DDS are available. These include <a href="/wiki/Application_programming_interface" title="Application programming interface">application programming interfaces</a> (APIs) and libraries of implementations in <a href="/wiki/Ada_(programming_language)" title="Ada (programming language)">Ada</a>, <a href="/wiki/C_(programming_language)" title="C (programming language)">C</a>, <a href="/wiki/C%2B%2B" title="C++">C++</a>, <a href="/wiki/C_Sharp_(programming_language)" title="C Sharp (programming language)">C#</a>, <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>, <a href="/wiki/Scala_(programming_language)" title="Scala (programming language)">Scala</a>, <a href="/wiki/Lua_(programming_language)" title="Lua (programming language)">Lua</a>, <a href="/wiki/Pharo" title="Pharo">Pharo</a> and <a href="/wiki/Ruby_(programming_language)" title="Ruby (programming language)">Ruby</a>. Some implementations are shown in the <a href="#table">table at the end of this article</a>.</p>
<p>DDS vendors participated in interoperability demonstrations at the OMG Spring technical meetings from 2009 to 2013.<sup class="reference" id="cite_ref-2009DemoNotes_21-0"><a href="#cite_note-2009DemoNotes-21">[21]</a></sup><sup class="reference" id="cite_ref-2010DemoNotes_22-0"><a href="#cite_note-2010DemoNotes-22">[22]</a></sup><sup class="reference" id="cite_ref-2011DemoNotes_23-0"><a href="#cite_note-2011DemoNotes-23">[23]</a></sup><sup class="reference" id="cite_ref-2012DemoNotes_24-0"><a href="#cite_note-2012DemoNotes-24">[24]</a></sup><sup class="reference" id="cite_ref-2013DemoNotes_25-0"><a href="#cite_note-2013DemoNotes-25">[25]</a></sup><sup class="reference" id="cite_ref-26"><a href="#cite_note-26">[26]</a></sup></p>
<p>During demos, each vendor published and subscribed to each other's topics using a test suite called the shapes demo. For example, one vendor publishes information about a shape and the other vendors can subscribe to the topic and display the results on their own shapes display. Each vendor takes turns publishing the information and the other subscribe. Two things made the demos possible: the DDS-I or Real-Time Publish-Subscribe (RTPS) protocol,<sup class="reference" id="cite_ref-RtpsRef_12-1"><a href="#cite_note-RtpsRef-12">[12]</a></sup> and the agreement to use a common model.</p>

<p>In March 2009, three vendors (Real-Time Innovations, Inc., PrismTech and Twin Oaks) demonstrated interoperability between the individual, independent products that implemented the OMG Real-time Publish-Subscribe protocol version 2.1 from January 2009. The demonstration included the discovery of each other's publishers and subscribers on different OS Platforms (<a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a> and <a href="/wiki/Linux" title="Linux">Linux</a>) and supported <a href="/wiki/Multicast" title="Multicast">multicast</a> and <a href="/wiki/Unicast" title="Unicast">unicast</a> network communications.<sup class="reference" id="cite_ref-2009DemoNotes_21-1"><a href="#cite_note-2009DemoNotes-21">[21]</a></sup></p>
<p>By March 2013, six more companies joined the interoperability demonstration: Object Computing Inc. (OCI, OpenDDS), Electronics and Telecommunications Research Institute (ETRI), <a href="/wiki/IBM" title="IBM">IBM</a>, <a href="/wiki/Kongsberg_Gruppen" title="Kongsberg Gruppen">Kongsberg</a>, Milsoft, and RemedyIT.<sup class="reference" id="cite_ref-2013DemoNotes_25-1"><a href="#cite_note-2013DemoNotes-25">[25]</a></sup></p>
<p>The DDS interoperability demonstration used scenarios such as:</p>
<ul>
<li>Basic connectivity to network using <a href="/wiki/Internet_Protocol" title="Internet Protocol">Internet Protocol</a> (IP)</li>
<li>Discovery of publishers and subscribers</li>
<li>Quality of service (QoS) Compatibility between requester and offerer</li>
<li>Delay-tolerant networking</li>
<li>Multiple topics and instances of topics</li>
<li>Exclusive ownerships of topics</li>
<li>Content filtering of topic data including time and geographic</li>
</ul>
