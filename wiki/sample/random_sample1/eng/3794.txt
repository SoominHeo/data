<h1 class="firstHeading" id="firstHeading" lang="en">Database engine</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
</div>
<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><script>function mfTempOpenSection(id){var block=document.getElementById("mf-section-"+id);block.className+=" open-block";block.previousSibling.className+=" open-block";}</script><p>A <b>database engine</b> (or <b>storage engine</b>) is the underlying software component that a <a class="mw-redirect" href="/wiki/Database_management_system" title="Database management system">database management system</a> (DBMS) uses to <a href="/wiki/Create,_read,_update_and_delete" title="Create, read, update and delete">create, read, update and delete</a> (CRUD) <a href="/wiki/Data" title="Data">data</a> from a <a href="/wiki/Database" title="Database">database</a>. Most database management systems include their own <a href="/wiki/Application_programming_interface" title="Application programming interface">application programming interface</a> (API) that allows the user to interact with their underlying engine without going through the user interface of the DBMS.</p>
<p>The term "database engine" is frequently used interchangeably with "<a href="/wiki/Database_server" title="Database server">database server</a>" or "database management system". A 'database instance' refers to the processes and memory structures of the running <b>database engine</b>.</p>
<p></p>

<h2><span class="mw-headline" id="Storage_engines">Storage engines</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database_engine&amp;action=edit&amp;section=1" title="Edit section: Storage engines">edit</a><span class="mw-editsection-bracket">]</span></span></h2>

<h2><span class="mw-headline" id="Design_considerations">Design considerations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database_engine&amp;action=edit&amp;section=2" title="Edit section: Design considerations">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Database bits are laid out in storage in data structures and groupings that can take advantage of both known effective algorithms to retrieve and manipulate them and the storage own properties. Typically the storage itself is designed to meet requirements of various areas that extensively utilize storage, including databases. A DBMS in operation always simultaneously utilizes several storage types (e.g., memory, and external storage), with respective layout methods.</p>
<p>In principle the database storage can be viewed as a <a class="mw-redirect" href="/wiki/Linear_address_space" title="Linear address space">linear address space</a>, where every bit of data has its unique address in this address space. In practice, only a very small percentage of addresses are kept as initial reference points (which also requires storage); most data is accessed by indirection using displacement calculations (distance in bits from the reference points) and data structures which define access paths (using pointers) to all needed data in an effective manner, optimized for the needed data access operations.</p>
<h3><span class="mw-headline" id="Database_storage_hierarchy">Database storage hierarchy</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database_engine&amp;action=edit&amp;section=3" title="Edit section: Database storage hierarchy">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>A database, while in operation, resides simultaneously in several types of storage, forming a <a class="mw-redirect" href="/wiki/Storage_hierarchy" title="Storage hierarchy">storage hierarchy</a>. By the nature of contemporary computers most of the database part inside a computer that hosts the DBMS resides (partially replicated) in volatile storage. Data (pieces of the database) that are being processed/manipulated reside inside a processor, possibly in <a href="/wiki/CPU_cache" title="CPU cache">processor's caches</a>. These data are being read from/written to memory, typically through a computer <a href="/wiki/Bus_(computing)" title="Bus (computing)">bus</a> (so far typically volatile storage components). Computer memory is communicating data (transferred to/from) external storage, typically through standard storage interfaces or networks (e.g., <a class="mw-redirect" href="/wiki/Fibre_channel" title="Fibre channel">fibre channel</a>, <a href="/wiki/ISCSI" title="ISCSI">iSCSI</a>). A <a href="/wiki/Disk_array" title="Disk array">storage array</a>, a common external storage unit, typically has storage hierarchy of its own, from a fast cache, typically consisting of (volatile and fast) <a class="mw-redirect" href="/wiki/DRAM" title="DRAM">DRAM</a>, which is connected (again via standard interfaces) to drives, possibly with different speeds, like <a href="/wiki/USB_flash_drive" title="USB flash drive">flash drives</a> and magnetic <a class="mw-redirect" href="/wiki/Disk_drive" title="Disk drive">disk drives</a> (non-volatile). The drives may be connected to <a href="/wiki/Magnetic_tape" title="Magnetic tape">magnetic tapes</a>, on which typically the least active parts of a large database may reside, or database backup generations.</p>
<p>Typically a correlation exists currently between storage speed and price, while the faster storage is typically volatile.</p>
<h3><span class="mw-headline" id="Data_structures">Data structures</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database_engine&amp;action=edit&amp;section=4" title="Edit section: Data structures">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Database_storage_structures" title="Database storage structures">Database storage structures</a></div>
<p>A data structure is an abstract construct that embeds data in a well defined manner. An efficient data structure allows to manipulate the data in efficient ways. The data manipulation may include data insertion, deletion, updating and retrieval in various modes. A certain data structure type may be very effective in certain operations, and very ineffective in others. A data structure type is selected upon DBMS development to best meet the operations needed for the types of data it contains. Type of data structure selected for a certain task typically also takes into consideration the type of storage it resides in (e.g., speed of access, minimal size of storage chunk accessed, etc.). In some DBMSs database administrators have the flexibility to select among options of data structures to contain user data for performance reasons. Sometimes the data structures have selectable parameters to tune the database performance.</p>
<p>Databases may store data in many data structure types.<sup class="reference" id="cite_ref-Physical_Database_Design_1-0"><a href="#cite_note-Physical_Database_Design-1">[1]</a></sup> Common examples are the following:</p>
<ul>
<li>ordered/unordered <a href="/wiki/Flat_file_database" title="Flat file database">flat files</a></li>
<li><a href="/wiki/Hash_table" title="Hash table">hash tables</a></li>
<li><a href="/wiki/B%2B_tree" title="B+ tree">B+ trees</a></li>
<li><a href="/wiki/ISAM" title="ISAM">ISAM</a></li>
<li><a href="/wiki/Heap_(data_structure)" title="Heap (data structure)">heaps</a></li>
</ul>
<h3><span class="mw-headline" id="Data_orientation_and_clustering">Data orientation and clustering</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database_engine&amp;action=edit&amp;section=5" title="Edit section: Data orientation and clustering">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In contrast to conventional row-orientation, relational databases can also be <a href="/wiki/Column-oriented_DBMS" title="Column-oriented DBMS">column-oriented</a> or <a class="new" href="/w/index.php?title=Correlational_database&amp;action=edit&amp;redlink=1" title="Correlational database (page does not exist)">correlational</a> in the way they store data in any particular structure.</p>
<p>In general, substantial performance improvement is gained if different types of database objects that are usually utilized together are laid in storage in proximity, being "clustered". This usually allows to retrieve needed related objects from storage in minimum number of input operations (each sometimes substantially time consuming). Even for in-memory databases clustering provides performance advantage due to common utilization of large caches for input-output operations in memory, with similar resulting behavior.</p>
<p>For example, it may be beneficial to cluster a record of an "item" in stock with all its respective "order" records. The decision of whether to cluster certain objects or not depends on the objects' utilization statistics, object sizes, caches sizes, storage types, etc.</p>
<h3><span class="mw-headline" id="Database_indexing">Database indexing</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Database_engine&amp;action=edit&amp;section=6" title="Edit section: Database indexing">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Database_index" title="Database index">Database index</a></div>
<p>Indexing is a technique some storage engines use for improving database performance. The many types of indexes share the common property that they reduce the need to examine every entry when running a query. In large databases, this can reduce query time/cost by orders of magnitude. The simplest form of index is a sorted list of values that can be searched using a <a class="mw-redirect" href="/wiki/Binary_search" title="Binary search">binary search</a> with an adjacent reference to the location of the entry, analogous to the index in the back of a book. The same data can have multiple indexes (an employee database could be indexed by last name and hire date).</p>
<p>Indexes affect performance, but not results. Database designers can add or remove indexes without changing application logic, reducing maintenance costs as the database grows and database usage evolves. Indexes can speed up data access, but they consume space in the database, and must be updated each time the data is altered. Indexes therefore can speed data access but slow data maintenance. These two properties determine whether a given index is worth the cost.</p>
