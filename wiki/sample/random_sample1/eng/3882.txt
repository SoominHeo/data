<h1 class="firstHeading" id="firstHeading" lang="en">DOS</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">

<p><b>DOS</b> <span class="nowrap"><span class="IPA nopopups"><a href="/wiki/Help:IPA_for_English" title="Help:IPA for English">/<span style="border-bottom:1px dotted"><span title="'d' in 'dye'">d</span><span title="/ɒ/ short 'o' in 'body'">ɒ</span><span title="'s' in 'sigh'">s</span></span>/</a></span></span>, short for <i>disk operating system</i>,<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> is an <a href="/wiki/Acronym" title="Acronym">acronym</a> used to describe the family of several very similar <a class="mw-redirect" href="/wiki/Command-line" title="Command-line">command-line</a> computer <a href="/wiki/Operating_system" title="Operating system">operating systems</a> for <a href="/wiki/X86" title="X86">x86</a>-based computers: primarily <a href="/wiki/MS-DOS" title="MS-DOS">MS-DOS</a> and <a class="mw-redirect" href="/wiki/PC_DOS" title="PC DOS">PC DOS</a>, but also <a href="/wiki/DR-DOS" title="DR-DOS">DR-DOS</a>, <a href="/wiki/FreeDOS" title="FreeDOS">FreeDOS</a>, <a class="mw-redirect" href="/wiki/ROM-DOS" title="ROM-DOS">ROM-DOS</a>, and <a href="/wiki/PTS-DOS" title="PTS-DOS">PTS-DOS</a>. None of these systems were officially named "DOS", and indeed "DOS" is a general term for <a href="/wiki/Disk_operating_system" title="Disk operating system">disk operating system</a>.</p>
<p><a href="/wiki/MS-DOS" title="MS-DOS">MS-DOS</a> dominated the <a href="/wiki/IBM_PC_compatible" title="IBM PC compatible">IBM PC compatible</a> (PC and PC Compatible) market between 1981 and 1995, and <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a> (<a href="/wiki/Windows_95" title="Windows 95">95</a>, <a href="/wiki/Windows_98" title="Windows 98">98</a>, and <a class="mw-redirect" href="/wiki/Windows_Me" title="Windows Me">Millennium Edition</a>), still ran on top of it until about 2001.</p>
<p>Dozens of other operating systems also use the acronym "DOS", including the 1960s <a href="/wiki/DOS/360_and_successors" title="DOS/360 and successors">DOS/360</a>. Others are <a href="/wiki/Apple_DOS" title="Apple DOS">Apple DOS</a>, <a href="/wiki/Apple_ProDOS" title="Apple ProDOS">Apple ProDOS</a>, <a href="/wiki/Atari_DOS" title="Atari DOS">Atari DOS</a>, <a href="/wiki/Commodore_DOS" title="Commodore DOS">Commodore DOS</a>, and <a href="/wiki/TRSDOS" title="TRSDOS">TRSDOS</a>, and <a href="/wiki/AmigaDOS" title="AmigaDOS">AmigaDOS</a>. See <a class="mw-redirect" href="/wiki/List_of_DOS_operating_systems#Other_operating_systems" title="List of DOS operating systems">List of DOS operating systems § Other operating systems</a>.</p>

<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DOS&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="hatnote" role="note">Further information: <a href="/wiki/Comparison_of_DOS_operating_systems" title="Comparison of DOS operating systems">Comparison of DOS operating systems</a> and <a href="/wiki/Timeline_of_DOS_operating_systems" title="Timeline of DOS operating systems">Timeline of DOS operating systems</a></div>
<h3><span class="mw-headline" id="Origins">Origins</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DOS&amp;action=edit&amp;section=2" title="Edit section: Origins">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/IBM_PC_DOS" title="IBM PC DOS">IBM PC DOS</a> (and the separately sold <a href="/wiki/MS-DOS" title="MS-DOS">MS-DOS</a>) and its predecessor, <a href="/wiki/86-DOS" title="86-DOS">86-DOS</a>, resembled <a href="/wiki/Digital_Research" title="Digital Research">Digital Research</a>'s <a href="/wiki/CP/M" title="CP/M">CP/M</a>—the dominant disk operating system for 8-bit <a href="/wiki/Intel_8080" title="Intel 8080">Intel 8080</a> and <a href="/wiki/Zilog_Z80" title="Zilog Z80">Zilog Z80</a> based microcomputers. DOS instead ran on <a href="/wiki/Intel_8086" title="Intel 8086">Intel 8086</a> 16-bit processors. Starting with <a class="mw-redirect" href="/wiki/MS-DOS_1.28" title="MS-DOS 1.28">MS-DOS 1.28</a><sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (October 2016)">citation needed</span></a></i>]</sup> and <a class="mw-redirect" href="/wiki/PC_DOS_2.0" title="PC DOS 2.0">PC DOS 2.0</a> the operating system incorporated various features inspired by <a href="/wiki/Xenix" title="Xenix">Xenix</a>, Microsoft's variant of <a href="/wiki/Unix" title="Unix">Unix</a>.</p>
<p>When IBM introduced the <a class="mw-redirect" href="/wiki/IBM_PC" title="IBM PC">IBM PC</a>, built with the <a href="/wiki/Intel_8088" title="Intel 8088">Intel 8088</a> microprocessor, they needed an operating system. Seeking an 8088-compatible build of CP/M, IBM initially approached Microsoft CEO <a href="/wiki/Bill_Gates" title="Bill Gates">Bill Gates</a> (possibly believing that Microsoft owned CP/M due to the Microsoft <a href="/wiki/Z-80_SoftCard" title="Z-80 SoftCard">Z-80 SoftCard</a>, which allowed CP/M to run on an <a href="/wiki/Apple_II" title="Apple II">Apple II</a><sup class="reference" id="cite_ref-rolander_2-0"><a href="#cite_note-rolander-2">[2]</a></sup>). IBM was sent to Digital Research, and a meeting was set up. However, the initial negotiations for the use of CP/M broke down; Digital Research wished to sell CP/M on a royalty basis, while IBM sought a single license, and to change the name to "PC DOS". Digital Research founder <a href="/wiki/Gary_Kildall" title="Gary Kildall">Gary Kildall</a> refused, and IBM withdrew.<sup class="reference" id="cite_ref-rolander_2-1"><a href="#cite_note-rolander-2">[2]</a></sup><sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup></p>
<p>IBM again approached Bill Gates. Gates in turn approached <a href="/wiki/Seattle_Computer_Products" title="Seattle Computer Products">Seattle Computer Products</a>. There, programmer <a href="/wiki/Tim_Paterson" title="Tim Paterson">Tim Paterson</a> had developed a variant of <a class="mw-redirect" href="/wiki/CP/M-80" title="CP/M-80">CP/M-80</a>, intended as an internal product for testing SCP's new <a href="/wiki/16-bit" title="16-bit">16-bit</a> <a href="/wiki/Intel_8086" title="Intel 8086">Intel 8086</a> <a href="/wiki/Central_processing_unit" title="Central processing unit">CPU</a> card for the <a href="/wiki/S-100_bus" title="S-100 bus">S-100 bus</a>. The system was initially named <a href="/wiki/Seattle_Computer_Products" title="Seattle Computer Products">QDOS</a> (Quick and Dirty Operating System), before being made commercially available as <a href="/wiki/86-DOS" title="86-DOS">86-DOS</a>. Microsoft purchased 86-DOS, allegedly for $50,000. This became Microsoft Disk Operating System, MS-DOS, introduced in 1981.<sup class="reference" id="cite_ref-mshist_4-0"><a href="#cite_note-mshist-4">[4]</a></sup></p>
<p>Within a year Microsoft licensed MS-DOS to over 70 other companies,<sup class="reference" id="cite_ref-freiberger19820823_5-0"><a href="#cite_note-freiberger19820823-5">[5]</a></sup> which supplied the operating system for their own hardware, sometimes under their own names. Microsoft later required the use of the MS-DOS name, with the exception of the IBM variant. IBM continued to develop their version, <a class="mw-redirect" href="/wiki/PC_DOS" title="PC DOS">PC DOS</a>, for the IBM PC.<sup class="reference" id="cite_ref-mshist_4-1"><a href="#cite_note-mshist-4">[4]</a></sup> Digital Research became aware that an operating system similar to CP/M was being sold by IBM (under the same name that IBM insisted upon for CP/M), and threatened legal action. IBM responded by offering an agreement: they would give PC consumers a choice of PC DOS or <a href="/wiki/CP/M-86" title="CP/M-86">CP/M-86</a>, Kildall's 8086 version. Side-by-side, CP/M cost almost $200 more than PC DOS, and sales were low. CP/M faded, with MS-DOS and PC DOS becoming the marketed operating system for PCs and PC compatibles.<sup class="reference" id="cite_ref-rolander_2-2"><a href="#cite_note-rolander-2">[2]</a></sup></p>
<p>Microsoft originally sold MS-DOS only to <a class="mw-redirect" href="/wiki/Original_equipment_manufacturers" title="Original equipment manufacturers">original equipment manufacturers</a> (OEMs). One major reason for this was that not all early PCs were 100% <a href="/wiki/IBM_PC_compatible" title="IBM PC compatible">IBM PC compatible</a>. DOS was structured such that there was a separation between the system specific device driver code (<a href="/wiki/IO.SYS" title="IO.SYS">IO.SYS</a>) and the DOS kernel (<a href="/wiki/MSDOS.SYS" title="MSDOS.SYS">MSDOS.SYS</a>). Microsoft provided an OEM Adaptation Kit (OAK) which allowed OEMs to customize the device driver code to their particular system. By the early 1990s, most PCs adhered to IBM PC standards so Microsoft began selling MS-DOS in retail with MS-DOS 5.0.</p>
<p>In the mid-1980s Microsoft developed a <a href="/wiki/MS-DOS_4.0_(multitasking)" title="MS-DOS 4.0 (multitasking)">multitasking version of DOS</a>.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup><sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup> This version of DOS is generally referred to as "European MS-DOS 4" because it was developed for <a href="/wiki/International_Computers_Limited" title="International Computers Limited">ICL</a> and licensed to several European companies. This version of DOS supports preemptive multitasking, shared memory, device helper services and <a href="/wiki/New_Executable" title="New Executable">New Executable</a> ("NE") format executables. None of these features were used in later versions of DOS, but they were used to form the basis of the OS/2 1.0 kernel. This version of DOS is distinct from the widely released PC DOS 4.0 which was developed by IBM and based upon DOS 3.3.</p>
<p>Digital Research attempted to regain the market lost from CP/M-86, initially with <a class="mw-redirect" href="/wiki/Concurrent_DOS" title="Concurrent DOS">Concurrent DOS</a>, <a href="/wiki/FlexOS" title="FlexOS">FlexOS</a> and <a href="/wiki/DOS_Plus" title="DOS Plus">DOS Plus</a> (both compatible with both MS-DOS and CP/M-86 software), later with <a href="/wiki/Multiuser_DOS" title="Multiuser DOS">Multiuser DOS</a> (compatible with both MS-DOS and CP/M-86 software) and <a class="mw-redirect" href="/wiki/DR_DOS" title="DR DOS">DR DOS</a> (compatible with MS-DOS software). Digital Research was bought by <a href="/wiki/Novell" title="Novell">Novell</a>, and DR DOS became <a class="mw-redirect" href="/wiki/Novell_DOS" title="Novell DOS">Novell DOS</a> 7; later, it was part of <a href="/wiki/Caldera_(company)" title="Caldera (company)">Caldera</a> (under the names <a class="mw-redirect" href="/wiki/OpenDOS" title="OpenDOS">OpenDOS</a> and DR-DOS 7.02/7.03), <a href="/wiki/Lineo" title="Lineo">Lineo</a>, and <a href="/wiki/DeviceLogics" title="DeviceLogics">DeviceLogics</a>.</p>
<p><a href="/wiki/Gordon_Letwin" title="Gordon Letwin">Gordon Letwin</a> wrote in 1995 that "DOS was, when we first wrote it, a one-time throw-away product intended to keep IBM happy so that they'd buy our languages".<sup class="reference" id="cite_ref-letwin19950817_8-0"><a href="#cite_note-letwin19950817-8">[8]</a></sup> Microsoft expected that it would be an interim solution before Xenix. The company planned to over time improve MS-DOS so it would be almost indistinguishable from single-user Xenix, or <a class="mw-redirect" href="/wiki/XEDOS" title="XEDOS">XEDOS</a>, which would also run on the <a href="/wiki/Motorola_68000" title="Motorola 68000">Motorola 68000</a>, <a class="mw-redirect" href="/wiki/Zilog_Z-8000" title="Zilog Z-8000">Zilog Z-8000</a>, and <a class="mw-redirect" href="/wiki/LSI-11" title="LSI-11">LSI-11</a>; they would be <a class="mw-redirect" href="/wiki/Upward_compatible" title="Upward compatible">upwardly compatible</a> with Xenix, which <i><a class="mw-redirect" href="/wiki/BYTE" title="BYTE">BYTE</a></i> in 1983 described as "the multi-user MS-DOS of the future".<sup class="reference" id="cite_ref-morgan198201_9-0"><a href="#cite_note-morgan198201-9">[9]</a></sup><sup class="reference" id="cite_ref-fiedler198310_10-0"><a href="#cite_note-fiedler198310-10">[10]</a></sup></p>
<p>IBM, however, did not want to replace DOS.<sup class="reference" id="cite_ref-howitt19841210_11-0"><a href="#cite_note-howitt19841210-11">[11]</a></sup> After <a href="/wiki/AT%26T" title="AT&amp;T">AT&amp;T</a> began selling Unix, Microsoft and IBM began developing <a href="/wiki/OS/2" title="OS/2">OS/2</a> as an alternative.<sup class="reference" id="cite_ref-letwin19950817_8-1"><a href="#cite_note-letwin19950817-8">[8]</a></sup> The two companies later had a series of disagreements over two successor operating systems to DOS, OS/2 and Microsoft's Windows.<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup> They split development of their DOS systems as a result.<sup class="reference" id="cite_ref-13"><a href="#cite_note-13">[13]</a></sup> The last retail version of MS-DOS was MS-DOS 6.22, after this MS-DOS became part of the Windows 95/98/Me bundle. The last retail version of PC DOS was PC DOS 2000 (a.k.a. PC DOS 7 revision 1) though IBM did later develop PC DOS 7.10 for OEMs and internal use.</p>
<p>The <a href="/wiki/FreeDOS" title="FreeDOS">FreeDOS</a> project began 26 June 1994, when Microsoft announced it would no longer sell or support MS-DOS. <a href="/wiki/Jim_Hall_(programmer)" title="Jim Hall (programmer)">Jim Hall</a> then posted a manifesto proposing the development of an open-source replacement. Within a few weeks, other programmers including <a href="/wiki/Pat_Villani" title="Pat Villani">Pat Villani</a> and Tim Norman joined the project. A kernel, the COMMAND.COM command line interpreter (shell) and core utilities were created by pooling code they had written or found available. There were several official pre-release distributions of FreeDOS before the FreeDOS 1.0 distribution was released on 3 September 2006. Made available under the <a href="/wiki/GNU_General_Public_License" title="GNU General Public License">GNU General Public License</a> (GPL), FreeDOS does not require license fees or royalties.<sup class="reference" id="cite_ref-14"><a href="#cite_note-14">[14]</a></sup><sup class="reference" id="cite_ref-15"><a href="#cite_note-15">[15]</a></sup></p>
<h3><span class="mw-headline" id="Decline">Decline</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DOS&amp;action=edit&amp;section=3" title="Edit section: Decline">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/History_of_Microsoft_Windows" title="History of Microsoft Windows">History of Microsoft Windows</a></div>
<p>Early versions of <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a> ran on top of a separate version of DOS.<sup class="reference" id="cite_ref-emulate_16-0"><a href="#cite_note-emulate-16">[16]</a></sup> By the early 1990s, Windows graphical shell saw heavy use on new DOS systems. In 1995, <a href="/wiki/Windows_95" title="Windows 95">Windows 95</a> was bundled as a standalone operating system that did not require a separate DOS license. Windows 95 (and Windows 98 and ME, that followed it) took over as the default <a class="mw-redirect" href="/wiki/OS_kernel" title="OS kernel">OS kernel</a>, though the MS-DOS component remained for compatibility. With Windows 95 and 98, but not ME, the MS-DOS component could be run without starting Windows.<sup class="reference" id="cite_ref-17"><a href="#cite_note-17">[17]</a></sup><sup class="reference" id="cite_ref-various-77715_18-0"><a href="#cite_note-various-77715-18">[18]</a></sup><sup class="reference" id="cite_ref-various-42623_19-0"><a href="#cite_note-various-42623-19">[19]</a></sup> With DOS no longer required to use Windows, the majority of PC users stopped using it directly.</p>

<p>Available DOS systems in 2012 are <a href="/wiki/FreeDOS" title="FreeDOS">FreeDOS</a>, <a href="/wiki/DR-DOS" title="DR-DOS">DR-DOS</a>, <a class="mw-redirect" href="/wiki/ROM-DOS" title="ROM-DOS">ROM-DOS</a>, <a href="/wiki/PTS-DOS" title="PTS-DOS">PTS-DOS</a>, <a class="new" href="/w/index.php?title=RxDOS&amp;action=edit&amp;redlink=1" title="RxDOS (page does not exist)">RxDOS</a> and <a class="mw-redirect" href="/wiki/REAL/32" title="REAL/32">REAL/32</a>. Some computer manufacturers, including <a href="/wiki/Dell" title="Dell">Dell</a> and <a href="/wiki/Hewlett-Packard" title="Hewlett-Packard">HP</a>, sell computers with FreeDOS as the <a href="/wiki/Original_equipment_manufacturer" title="Original equipment manufacturer">OEM</a> operating system.<sup class="reference" id="cite_ref-20"><a href="#cite_note-20">[20]</a></sup><sup class="reference" id="cite_ref-21"><a href="#cite_note-21">[21]</a></sup></p>
<h4><span class="mw-headline" id="Embedded_systems">Embedded systems</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DOS&amp;action=edit&amp;section=5" title="Edit section: Embedded systems">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>DOS's structure of accessing hardware directly makes it ideal for use in embedded devices. The final versions of DR-DOS are still aimed at this market.<sup class="reference" id="cite_ref-22"><a href="#cite_note-22">[22]</a></sup> ROM-DOS was used as the embedded system on the <a class="mw-redirect" href="/wiki/Canon_PowerShot_Pro" title="Canon PowerShot Pro">Canon PowerShot Pro</a> 70.<sup class="reference" id="cite_ref-23"><a href="#cite_note-23">[23]</a></sup></p>
<h4><span class="mw-headline" id="Emulation">Emulation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DOS&amp;action=edit&amp;section=6" title="Edit section: Emulation">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Under <a href="/wiki/Linux" title="Linux">Linux</a>, it is also possible to run copies of DOS and many of its clones under <i><a href="/wiki/DOSEMU" title="DOSEMU">DOSEMU</a></i>, a Linux-native <a href="/wiki/Virtual_machine" title="Virtual machine">virtual machine</a> for running DOS programs at near native speed. There are a number of other <a href="/wiki/Emulator" title="Emulator">emulators</a> for running DOS under various versions of UNIX, even on non-<a href="/wiki/X86" title="X86">x86</a> platforms, such as <a href="/wiki/DOSBox" title="DOSBox">DOSBox</a>.<sup class="reference" id="cite_ref-DOSBox_24-0"><a href="#cite_note-DOSBox-24">[24]</a></sup><sup class="reference" id="cite_ref-25"><a href="#cite_note-25">[25]</a></sup></p>
<p>DOS emulators are a popular way to run games or other DOS software under <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a>, as modern versions lack full compatibility with DOS. One of the best-known is <a href="/wiki/DOSBox" title="DOSBox">DOSBox</a>, designed for legacy gaming (e.g. <a href="/wiki/King%27s_Quest" title="King's Quest">King's Quest</a>, <a href="/wiki/Doom_(1993_video_game)" title="Doom (1993 video game)">Doom</a>) on modern operating systems.<sup class="reference" id="cite_ref-emulate_16-1"><a href="#cite_note-emulate-16">[16]</a></sup><sup class="reference" id="cite_ref-DOSBox_24-1"><a href="#cite_note-DOSBox-24">[24]</a></sup> Another alternative is running DOS applications in a copy of a DOS operating system on a <a href="/wiki/Category:X86_emulators" title="Category:X86 emulators">PC emulator</a>. This approach provides better compatibility at the cost of increased overhead.<sup class="reference" id="cite_ref-26"><a href="#cite_note-26">[26]</a></sup></p>
<h2><span class="mw-headline" id="Design">Design</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DOS&amp;action=edit&amp;section=7" title="Edit section: Design">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>All MS-DOS-type operating systems run on machines with the <a href="/wiki/Intel" title="Intel">Intel</a> <a href="/wiki/X86" title="X86">x86</a> or compatible <a class="mw-redirect" href="/wiki/CPU" title="CPU">CPUs</a>, mainly the <a class="mw-redirect" href="/wiki/IBM_PC" title="IBM PC">IBM PC</a> and <a href="/wiki/IBM_PC_compatible" title="IBM PC compatible">compatibles</a>. Machine-dependent versions of MS-DOS were produced for many non-IBM-compatible <a href="/wiki/X86" title="X86">x86</a>-based machines, with variations from relabelling of the <a href="/wiki/Microsoft" title="Microsoft">Microsoft</a> distribution under the manufacturer's name, to versions specifically designed to work with non-IBM-PC-compatible hardware. For as long as application programs used DOS APIs instead of direct hardware access, they could thereby also run on non-IBM-PC compatible machines. In 1985, Digital Research also had a version of <a class="mw-redirect" href="/wiki/Concurrent_DOS_68K" title="Concurrent DOS 68K">Concurrent DOS 68K</a> for use on Motorola 68000 CPUs, and the original <a href="/wiki/FreeDOS" title="FreeDOS">FreeDOS</a> kernel <a class="mw-redirect" href="/wiki/DOS-C" title="DOS-C">DOS-C</a> derived from <a class="mw-redirect" href="/wiki/DOS/NT" title="DOS/NT">DOS/NT</a>, also for Motorola CPUs, in the early 1990s. While these systems resembled the DOS architecture, applications were not binary compatible due to the incompatible instruction sets of these non-x86-CPUs. However, applications written in high-level languages could be ported easily.</p>
<p>DOS is a single-user, single-tasking operating system with basic <a class="mw-redirect" href="/wiki/Kernel_(computer_science)" title="Kernel (computer science)">kernel</a> functions that are <a class="mw-redirect" href="/wiki/Reentrant_(subroutine)" title="Reentrant (subroutine)">non-reentrant</a>: only one program at a time can use them and DOS itself has no functionality to allow more than one program to execute at a time. The DOS kernel provides <a href="/wiki/MS-DOS_API" title="MS-DOS API">various functions for programs</a> (an <i>application program interface</i>), like character I/O, file management, memory management, program loading and termination.</p>
<p>DOS by default provides a primitive ability for <a href="/wiki/Shell_script" title="Shell script">shell scripting</a>, via <i>batch files</i> (with the <a href="/wiki/Filename_extension" title="Filename extension">filename extension</a> <code>.BAT</code>). These are text files that can be created in any text editor. They are executed in the same fashion as compiled programs, and run each line of the batch file as a command. Batch files can also make use of several internal commands, such as <a class="mw-redirect" href="/wiki/GOTO" title="GOTO">GOTO</a> and <a class="mw-redirect" href="/wiki/Conditional_(programming)" title="Conditional (programming)">conditional statements</a>.<sup class="reference" id="cite_ref-27"><a href="#cite_note-27">[27]</a></sup> <a href="/wiki/GOSUB" title="GOSUB">GOSUB</a> and simple arithmetic is supported with the DR DOS COMMAND.COM as well as some with third-party shells like <a href="/wiki/4DOS" title="4DOS">4DOS</a>; however, no real form of programming is usually enabled.</p>
<p>The operating system offers an application programming interface that allows development of character-based applications, but not for accessing most of the <a href="/wiki/Computer_hardware" title="Computer hardware">hardware</a>, such as <a href="/wiki/Video_card" title="Video card">graphics cards</a>, <a class="mw-redirect" href="/wiki/Computer_printer" title="Computer printer">printers</a>, or <a href="/wiki/Computer_mouse" title="Computer mouse">mice</a>. This required programmers to access the hardware directly, usually resulting in each application having its own set of <a href="/wiki/Device_driver" title="Device driver">device drivers</a> for each hardware peripheral. Hardware manufacturers would release specifications to ensure device drivers for popular applications were available.<sup class="reference" id="cite_ref-28"><a href="#cite_note-28">[28]</a></sup></p>
<h3><span class="mw-headline" id="Boot_sequence">Boot sequence</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DOS&amp;action=edit&amp;section=8" title="Edit section: Boot sequence">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<ul>
<li>The <a class="mw-redirect" href="/wiki/Bootstrap_loader" title="Bootstrap loader">bootstrap loader</a> on PC-compatible computers (<a href="/wiki/Master_boot_record" title="Master boot record">MBR</a> or <a href="/wiki/Boot_sector" title="Boot sector">boot sector</a>) is located at <a href="/wiki/Track0" title="Track0">track zero</a>, the first sector on a disk. The <a href="/wiki/BIOS" title="BIOS">ROM BIOS</a> will load this sector into memory at address <span style="font-family: monospace, monospace;">0000h</span>:<span style="font-family: monospace, monospace;">7C00h</span>, and typically check for a signature "<span style="font-family: monospace, monospace;">55h AAh</span>" at offset <span style="font-family: monospace, monospace;">+1FEh</span>. If the sector is not considered to be valid, the ROM BIOS will try the next physical disk in the row, otherwise it will jump to the load address with certain registers set up.</li>
<li>If the loaded boot sector happens to be a <a class="mw-redirect" href="/wiki/Master_Boot_Record" title="Master Boot Record">Master Boot Record</a> (MBR), as found on partitioned media, it will relocate itself to <span style="font-family: monospace, monospace;">0000h</span>:<span style="font-family: monospace, monospace;">0600h</span> in memory,<sup class="reference" id="cite_ref-dewassoc_com-master_boot_record_29-0"><a href="#cite_note-dewassoc_com-master_boot_record-29">[29]</a></sup> otherwise this step is skipped. The MBR code will scan the partition table, which is located within this sector, for an active partition (modern MBRs check if bit 7 is set at offset <span style="font-family: monospace, monospace;">+1BEh+10h*n</span>, whereas old MBRs simply check for a value of <span style="font-family: monospace, monospace;">80h</span>), and, if found, load the first sector of the corresponding partition, which holds the <a class="mw-redirect" href="/wiki/Volume_Boot_Record" title="Volume Boot Record">Volume Boot Record</a> (VBR) of that volume, into memory at <span style="font-family: monospace, monospace;">0000h</span>:<span style="font-family: monospace, monospace;">7C00h</span> in the similar fashion as it had been loaded by the ROM BIOS itself. The MBR will then pass execution to the loaded portion with certain registers set up.</li>
<li>The sector content loaded at <span style="font-family: monospace, monospace;">0000h</span>:<span style="font-family: monospace, monospace;">7C00h</span> constitutes a VBR now. VBRs are operating system specific and cannot be exchanged between different DOS versions in general, as the exact behaviour differs between different DOS versions. In very old versions of DOS such as DOS 1.x, the VBR would load the whole IO.SYS/IBMBIO.COM file into memory at <span style="font-family: monospace, monospace;">0000h</span>:<span style="font-family: monospace, monospace;">0600h</span>.<sup class="reference" id="cite_ref-pagetable-165_30-0"><a href="#cite_note-pagetable-165-30">[30]</a></sup> For this to work, these sectors had to be stored in consecutive order on disk by SYS. In later issues, it would locate and store the contents of the first two entries in the root directory at <span style="font-family: monospace, monospace;">0000h</span>:<span style="font-family: monospace, monospace;">0500h</span> and if they happen to reflect the correct boot files as recorded in the VBR, the VBR would load the first 3 consecutive sectors of the IO.SYS/IBMBIO.COM file into memory at <span style="font-family: monospace, monospace;">0070h</span>:<span style="font-family: monospace, monospace;">0000h</span>. The VBR also has to take care to preserve the contents of the <a class="new" href="/w/index.php?title=Disk_Parameter_Table&amp;action=edit&amp;redlink=1" title="Disk Parameter Table (page does not exist)">Disk Parameter Table</a> (DPT). Finally, it passes control to the loaded portion by jumping to its entry point with certain registers set up (with considerable differences between different DOS versions).</li>
<li>In modern DOS versions, where the VBR has loaded only the first 3 sectors of the IO.SYS/IBMBIO.COM file into memory, the loaded portion contains another boot loader, which will then load the remainder of itself into memory, using the root directory information stored at <span style="font-family: monospace, monospace;">0000h</span>:<span style="font-family: monospace, monospace;">0500h</span>. For most versions, the file contents still need to be stored in consecutive order on disk. In older versions of DOS, which were still loaded as a whole, this step is skipped.</li>
<li>The DOS system initialization code will initial its builtin device drivers and then load the DOS kernel, located in <a href="/wiki/MSDOS.SYS" title="MSDOS.SYS">MSDOS.SYS</a> on MS-DOS systems, into memory as well. In Windows 9x, the DOS system initialization code and builtin device drivers and the DOS kernel are combined into a single IO.SYS file while MSDOS.SYS is used as a text configuration file.</li>
<li>The <a href="/wiki/CONFIG.SYS" title="CONFIG.SYS">CONFIG.SYS</a> file is then read to <a href="/wiki/Parsing#Programming_languages" title="Parsing">parse</a> configuration parameters. The <span style="font-family: monospace, monospace;">SHELL</span> variable specifies the location of the <a href="/wiki/Shell_(computing)" title="Shell (computing)">shell</a> which defaults to <a href="/wiki/COMMAND.COM" title="COMMAND.COM">COMMAND.COM</a>.</li>
<li>The shell is loaded and executed.</li>
<li>The startup batch file <a href="/wiki/AUTOEXEC.BAT" title="AUTOEXEC.BAT">AUTOEXEC.BAT</a> is then run by the shell.<sup class="reference" id="cite_ref-evergreen_edu-config-sys_31-0"><a href="#cite_note-evergreen_edu-config-sys-31">[31]</a></sup><sup class="reference" id="cite_ref-32"><a href="#cite_note-32">[32]</a></sup></li>
</ul>
<p>The DOS system files loaded by the boot sector must be <a class="mw-redirect" href="/wiki/Fragmentation_(computer)" title="Fragmentation (computer)">contiguous</a> and be the first two <a class="mw-redirect" href="/wiki/FAT_directory_table" title="FAT directory table">directory entries</a>.<sup class="reference" id="cite_ref-wustl_edu-misc_33-0"><a href="#cite_note-wustl_edu-misc-33">[33]</a></sup> As such, removing and adding this file is likely to render the media unbootable. It is, however, possible to replace the shell at will, a method that can be used to start the execution of dedicated applications faster. This limitation does not apply to any version of DR DOS, where the system files can be located anywhere in the root directory and do not need to be contiguous. Therefore, system files can be simply copied to a disk provided that the boot sector is DR DOS compatible already.</p>
<p>In PC DOS and DR DOS 5.0 and above, the DOS system files are named <a href="/wiki/IBMBIO.COM" title="IBMBIO.COM">IBMBIO.COM</a> instead of <a href="/wiki/IO.SYS" title="IO.SYS">IO.SYS</a> and <a href="/wiki/IBMDOS.COM" title="IBMDOS.COM">IBMDOS.COM</a> instead of <a href="/wiki/MSDOS.SYS" title="MSDOS.SYS">MSDOS.SYS</a>. Older versions of DR DOS used DRBIOS.SYS and DRBDOS.SYS instead.</p>
<p>Starting with <a class="mw-redirect" href="/wiki/MS-DOS_7.0" title="MS-DOS 7.0">MS-DOS 7.0</a> the binary system files IO.SYS and MSDOS.SYS were combined into a single file IO.SYS whilst MSDOS.SYS became a configuration file similar to CONFIG.SYS and AUTOEXEC.BAT. If the MSDOS.SYS <a class="new" href="/w/index.php?title=BootGUI&amp;action=edit&amp;redlink=1" title="BootGUI (page does not exist)">BootGUI</a> directive is set to <code>0</code>, the boot process will stop with the command processor (typically COMMAND.COM) loaded, instead of executing WIN.COM automatically.</p>
<h3><span class="mw-headline" id="File_system">File system</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DOS&amp;action=edit&amp;section=9" title="Edit section: File system">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>DOS uses a filesystem which supports <a href="/wiki/8.3_filename" title="8.3 filename">8.3 filenames</a>; 8 characters for the filename and 3 characters for the extension. Starting with DOS 2 hierarchical directories are supported. Each directory name is also 8.3 format but the maximum directory path length is 64 characters due to the internal current directory structure (CDS) tables that DOS maintains. Including the drive name, the maximum length of a fully qualified filename that DOS supports is 80 characters using the format drive:\path\filename.ext followed by a null byte.</p>
<p>DOS uses the <a href="/wiki/File_Allocation_Table" title="File Allocation Table">File Allocation Table</a> (FAT) filesystem. This was originally <a class="mw-redirect" href="/wiki/FAT12" title="FAT12">FAT12</a> which supported up to 4078 clusters per drive. DOS 3.0 added support for <a class="mw-redirect" href="/wiki/FAT16" title="FAT16">FAT16</a> which used 16-bit allocation entries and supported up to 65518 clusters per drive. DOS 3.31 added support for <a class="mw-redirect" href="/wiki/FAT16B" title="FAT16B">FAT16B</a> which removed the 32 MB drive limit and could support up to 2 GB. Finally MS-DOS 7.1 (the DOS component of Windows 9x) added support for <a class="mw-redirect" href="/wiki/FAT32" title="FAT32">FAT32</a> which used 32-bit allocation entries and could support hard drives up to 137 GB and beyond.</p>
<p>Starting with DOS 3.1, file redirector support was added to DOS. This was initially used to support networking but was later used to support CD-ROM drives with <a href="/wiki/MSCDEX" title="MSCDEX">MSCDEX</a>. IBM PC DOS 4.0 also had preliminary installable file system (IFS) support but this was unused and removed in DOS 5.0.</p>
<h4><span class="mw-headline" id="Drive_naming_scheme">Drive naming scheme</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DOS&amp;action=edit&amp;section=10" title="Edit section: Drive naming scheme">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<div class="hatnote" role="note">Main article: <a href="/wiki/Drive_letter_assignment" title="Drive letter assignment">Drive letter assignment</a></div>
<p>In DOS, drives are referred to by identifying letters. Standard practice is to reserve "A" and "B" for <a class="mw-redirect" href="/wiki/Floppy_drives" title="Floppy drives">floppy drives</a>. On systems with only one floppy drive DOS assigns both letters to the drive, prompting the user to swap disks as programs alternate access between them. This facilitates copying from floppy to floppy or having a program run from one floppy while accessing its data on another. <a class="mw-redirect" href="/wiki/Hard_drives" title="Hard drives">Hard drives</a> were originally assigned the letters "C" and "D". DOS could only support one active partition per drive. As support for more hard drives became available, this developed into first assigning a drive letter to each drive's active <a class="mw-redirect" href="/wiki/Primary_partition" title="Primary partition">primary partition</a>, then making a second pass over the drives to allocate letters to logical drives in the <a class="mw-redirect" href="/wiki/Extended_partition" title="Extended partition">extended partition</a>, then a third pass to give any other non-active <a class="mw-redirect" href="/wiki/Primary_partition" title="Primary partition">primary partitions</a> their names (where such additional partitions existed and contained a DOS-supported file system). Lastly, DOS allocates letters for <a href="/wiki/Optical_disc_drive" title="Optical disc drive">optical disc drives</a>, <a class="mw-redirect" href="/wiki/RAM_disk" title="RAM disk">RAM disks</a>, and other hardware. Letter assignments usually occur in the order the drivers are loaded, but the drivers can instruct DOS to assign a different letter; drivers for network drives, for example, typically assign letters nearer the end of the alphabet.<sup class="reference" id="cite_ref-driveletter_34-0"><a href="#cite_note-driveletter-34">[34]</a></sup></p>
<p>Because DOS applications use these drive letters directly (unlike the /dev directory in <a href="/wiki/Unix-like" title="Unix-like">Unix-like</a> systems), they can be disrupted by adding new hardware that needs a drive letter. An example is the addition of a new hard drive having a primary partition where a pre-existing hard drive contains logical drives in extended partitions; the new drive will be assigned a letter that was previously assigned to one of the extended partition logical drives. Moreover, even adding a new hard drive having only logical drives in an extended partition would still disrupt the letters of RAM disks and optical drives. This problem persisted through Microsoft's DOS-based 9x versions of Windows until they were replaced by versions based on the NT line, which preserves the letters of existing drives until the user changes them.<sup class="reference" id="cite_ref-driveletter_34-1"><a href="#cite_note-driveletter-34">[34]</a></sup> Under DOS, this problem can be worked around by defining a SUBST drive and installing the DOS program into this logical drive. The assignment of this drive would then be changed in a batch job whenever the application starts. Under some versions of <a class="mw-redirect" href="/wiki/Concurrent_DOS" title="Concurrent DOS">Concurrent DOS</a>, as well as under <a href="/wiki/Multiuser_DOS" title="Multiuser DOS">Multiuser DOS</a>, System Manager and <a class="mw-redirect" href="/wiki/REAL/32" title="REAL/32">REAL/32</a>, the reserved drive letter L: will automatically be assigned to the corresponding <a class="mw-redirect" href="/wiki/Load_drive" title="Load drive">load drive</a> whenever an application starts.</p>
<h4><span class="mw-headline" id="Reserved_device_names">Reserved device names</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DOS&amp;action=edit&amp;section=11" title="Edit section: Reserved device names">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<div class="hatnote" role="note">Main article: <a href="/wiki/Device_file" title="Device file">Device file</a></div>
<p>There are reserved device names in DOS that cannot be used as filenames regardless of extension as they are occupied by built-in character devices. These restrictions also affect several Windows versions, in some cases causing crashes and security vulnerabilities.<sup class="reference" id="cite_ref-35"><a href="#cite_note-35">[35]</a></sup></p>
<p>The reserved names are: <code>CON</code> (for console), <code>AUX</code> (for auxiliary), <code>PRN</code><sup class="reference" id="cite_ref-36"><a href="#cite_note-36">[36]</a></sup> (for printer) and <code>LST</code> (for lister), which were introduced with <a href="/wiki/86-DOS" title="86-DOS">86-DOS</a> 0.74. 86-DOS 1.10 and PC DOS 1.0 added <code>NUL</code>. Except for <code>LST</code> they continued to be supported in all versions of MS-DOS, PC DOS and DR-DOS ever since. <code>LST</code> was also available in some OEM versions of MS-DOS 1.25, whereas other OEM versions of MS-DOS 1.25 already used <code>LPT1</code> (first <a href="/wiki/Parallel_port" title="Parallel port">line printer</a>) and <code>COM1</code> (first <a href="/wiki/COM_(hardware_interface)" title="COM (hardware interface)">serial communication device</a>) instead, as introduced with PC DOS. In addition to <code>LPT1</code> and <code>LPT2</code> as well as <code>COM1</code> to <code>COM3</code>, Hewlett-Packard's <a class="mw-redirect" href="/wiki/MS-DOS_2.11" title="MS-DOS 2.11">MS-DOS 2.11</a> for the <a class="mw-redirect" href="/wiki/HP_Portable_Plus" title="HP Portable Plus">HP Portable Plus</a> also supported <code>LST</code> as alias for <code>LPT2</code> and <code>82164A</code> as alias for <code>COM2</code>;<sup class="reference" id="cite_ref-HP_1985_PP_37-0"><a href="#cite_note-HP_1985_PP-37">[37]</a></sup><sup class="reference" id="cite_ref-HP_1986_PP_38-0"><a href="#cite_note-HP_1986_PP-38">[38]</a></sup> it also supported <code>PLT</code> for <a href="/wiki/Plotter" title="Plotter">plotters</a>.<sup class="reference" id="cite_ref-HP_1985_PP_37-1"><a href="#cite_note-HP_1985_PP-37">[37]</a></sup><sup class="reference" id="cite_ref-HP_1986_PP_38-1"><a href="#cite_note-HP_1986_PP-38">[38]</a></sup> Otherwise, <code>COM2</code>, <code>LPT2</code>, <code>LPT3</code> and the <code>CLOCK$</code> (still named <code>CLOCK</code> in some issues of MS-DOS 2.11<sup class="reference" id="cite_ref-Microsoft_2014_Altos_39-0"><a href="#cite_note-Microsoft_2014_Altos-39">[39]</a></sup><sup class="reference" id="cite_ref-HP_1985_PP_37-2"><a href="#cite_note-HP_1985_PP-37">[37]</a></sup><sup class="reference" id="cite_ref-HP_1986_PP_38-2"><a href="#cite_note-HP_1986_PP-38">[38]</a></sup>) clock device were introduced with DOS 2.0, and <code>COM3</code> and <code>COM4</code> were added with DOS 3.3. Only the multitasking <a href="/wiki/MS-DOS_4.0_(multitasking)" title="MS-DOS 4.0 (multitasking)">MS-DOS 4</a> supported <code>KEYBD$</code> and <code>SCREEN$</code>. DR DOS 5.0 and higher and Multiuser DOS support an <code><a class="mw-redirect" href="/wiki/$IDLE$" title="$IDLE$">$IDLE$</a></code> device for dynamic idle detection to saving power and improve multitasking. <code>LPT4</code> is an optional built-in driver for a fourth line printer supported in some versions of DR-DOS since 7.02. <code>CONFIG$</code> constitutes the <a href="/wiki/Real_mode" title="Real mode">real mode</a> <a href="/wiki/Legacy_Plug_and_Play" title="Legacy Plug and Play">PnP</a> manager in MS-DOS 7.0-8.0.</p>
<p><code>AUX</code> typically defaults to <code>COM1</code>, and <code>PRN</code> to <code>LPT1</code> (<code>LST</code>), but these defaults can be changed in some versions of DOS to point to other serial or parallel devices.<sup class="reference" id="cite_ref-HP_1985_PP_37-3"><a href="#cite_note-HP_1985_PP-37">[37]</a></sup><sup class="reference" id="cite_ref-HP_1986_PP_38-3"><a href="#cite_note-HP_1986_PP-38">[38]</a></sup><sup class="reference" id="cite_ref-Paul_1997_OD-A3_40-0"><a href="#cite_note-Paul_1997_OD-A3-40">[40]</a></sup> <code>PLT</code> was reconfigurable as well.<sup class="reference" id="cite_ref-HP_1985_PP_37-4"><a href="#cite_note-HP_1985_PP-37">[37]</a></sup><sup class="reference" id="cite_ref-HP_1986_PP_38-4"><a href="#cite_note-HP_1986_PP-38">[38]</a></sup></p>
<p>Filenames ended with a <a href="/wiki/Colon_(punctuation)" title="Colon (punctuation)">colon</a> (<span style="font-family: monospace, monospace;">:</span>) such as <code>NUL:</code> conventionally indicate device names, but the colon is not actually a part of the name of the built-in device drivers. Colons are not necessary to be typed in some cases, for example:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">ECHO</span> This achieves nothing <span class="p">&gt;</span> NUL
</pre></div>
<p>It is still possible to create files or directories using these reserved device names, such as through direct editing of directory data structures in disk sectors. Such naming, such as starting a file name with a space, has sometimes been used by viruses or hacking programs to obscure files from users who do not know how to access these locations.</p>
<div class="hatnote" role="note">Further information: <a href="/wiki/Parallel_port" title="Parallel port">Parallel port</a> and <a href="/wiki/COM_(hardware_interface)" title="COM (hardware interface)">Serial port</a></div>
<h3><span class="mw-headline" id="Memory_management">Memory management</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DOS&amp;action=edit&amp;section=12" title="Edit section: Memory management">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/DOS_memory_management" title="DOS memory management">DOS memory management</a></div>
<p>DOS was originally designed for the Intel 8086/8088 processor and therefore could only directly access a maximum of 1 MB of RAM. Due to PC architecture only a maximum of 640 KB (known as <a href="/wiki/Conventional_memory" title="Conventional memory">conventional memory</a>) is available as the upper 384 KB is reserved.</p>
<p>Specifications were developed to allow access to additional memory. The first was the <a class="mw-redirect" href="/wiki/Expanded_Memory_Specification" title="Expanded Memory Specification">Expanded Memory Specification</a> (EMS) which originally allowed memory on an add-on card to be accessed via a 64 KB page frame in the reserved upper memory area. 80386 and later systems could use a <a href="/wiki/Virtual_8086_mode" title="Virtual 8086 mode">virtual 8086 mode</a> (V86) mode memory manager like <a href="/wiki/EMM386" title="EMM386">EMM386</a> to create expanded memory from extended memory without the need of an add-on card. The second specification was the <a class="mw-redirect" href="/wiki/Extended_Memory_Specification" title="Extended Memory Specification">Extended Memory Specification</a> (XMS) for 80286 and later systems. This provided a way to copy data to and from extended memory, access to the 65520-byte <a class="mw-redirect" href="/wiki/High_Memory_Area" title="High Memory Area">High Memory Area</a> (HMA) directly above the first megabyte of memory and the <a class="mw-redirect" href="/wiki/Upper_Memory_Block" title="Upper Memory Block">Upper Memory Block</a> (UMB) area. Generally XMS support was provided by <a href="/wiki/HIMEM.SYS" title="HIMEM.SYS">HIMEM.SYS</a> or a V86 mode memory manager like <a href="/wiki/QEMM" title="QEMM">QEMM</a> or <a href="/wiki/386MAX" title="386MAX">386MAX</a> which also supported EMS.</p>
<p>Starting with DOS 5, DOS could directly take advantage of the HMA by loading its kernel code and disk buffers there via the DOS=HIGH statement in CONFIG.SYS. DOS 5+ also allowed the use of available UMBs via the DOS=UMB statement in CONFIG.SYS.</p>
<h3><span class="mw-headline" id="DOS_under_OS.2F2_and_Windows">DOS under OS/2 and Windows</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DOS&amp;action=edit&amp;section=13" title="Edit section: DOS under OS/2 and Windows">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">See also: <a class="mw-redirect" href="/wiki/Virtual_DOS_Machine" title="Virtual DOS Machine">Virtual DOS Machine</a></div>
<p>The DOS emulation in OS/2 and Windows runs in much the same way as native applications do. They can access all of the drives and services, and can even use the host's clipboard services. Because the drivers for file systems and such forth reside in the host system, the DOS emulation needs only provide a DOS API translation layer which converts DOS calls to OS/2 or Windows system calls. The translation layer generally also converts BIOS calls and virtualizes common I/O port accesses which many DOS programs commonly use.</p>
<p>In Windows 3.1 and 9x, the DOS virtual machine is provided by WINOLDAP. WinOldAp creates a virtual machine based on the program's PIF file, and the system state when Windows was loaded. The DOS graphics mode, both character and graphic, can be captured and run in the window. DOS applications can use the Windows clipboard by accessing extra published calls in WinOldAp, and one can paste text through the WinOldAp graphics.</p>
<p>The emulated DOS in OS/2 and Windows NT is based upon DOS 5. Although there is a default configuration (config.sys and autoexec.bat), one can use alternate files on a session-by-session basis. It is possible to load drivers in these files to access the host system, although these are typically third-party.</p>
<p>Under OS/2 2.x and later, the DOS emulation is provided by DOSKRNL. This is a file that represents the combined IBMBIO.COM and IBMDOS.COM, the system calls are passed through to the OS/2 windowing services. DOS programs run in their own environment, the bulk of the DOS utilities are provided by bound DOS / OS2 applications in the \OS2 directory. OS/2 can run Windows 3.1 applications by using a modified copy of Windows (Win-OS/2). The modifications allow Windows 3.1 programs to run seamlessly on the OS/2 desktop, or one can start a WinOS/2 desktop, similar to starting Windows from DOS.</p>
<p>OS/2 allows for 'DOS from Drive A:', (VMDISK). This is a real DOS, like MS-DOS 6.22 or PC DOS 5.00. One makes a bootable floppy disk of the DOS, add a number of drivers from OS/2, and then creates a special image. The DOS booted this way has full access to the system, but provides its own drivers for hardware. One can use such a disk to access cdrom drives for which there is no OS/2 driver.</p>
<p>In Windows NT (2000, XP, Vista, 7), the DOS emulation is provided by way of a virtual DOS machine (NTVDM). The DOS files reside in NTIO.SYS and NTBIO.SYS as usual, but run in the virtual machine provided by NTVDM. The character input is passed to the console session that launched the DOS program. This allows one to use CLI features such as pipes and redirection between DOS and Windows NT. The OS/2 emulation in NT and 2000 is similar, but no virtual machine is loaded: it is handled by OS2SS.EXE and OS2.EXE.</p>
<p>64-bit versions of Windows do not support NTVDM and cannot run 16-bit DOS applications directly; Third-party emulators such as DOSbox can be used to run these programs.</p>
<h2><span class="mw-headline" id="User_interface">User interface</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DOS&amp;action=edit&amp;section=14" title="Edit section: User interface">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>DOS systems utilize a <a class="mw-redirect" href="/wiki/Command_line_interface" title="Command line interface">command line interface</a>. Programs are started by entering their filename at the command prompt. DOS systems include several programs as system utilities, and provides additional commands that don't correspond to programs (<i>internal commands</i>).<sup class="reference" id="cite_ref-41"><a href="#cite_note-41">[41]</a></sup></p>
<p>In an attempt to provide a more user-friendly environment, numerous software manufacturers wrote <a href="/wiki/File_manager" title="File manager">file management programs</a> that provided users with menu- and/or icon-based interfaces. Microsoft Windows is a notable example, eventually resulting in <a href="/wiki/Windows_9x" title="Windows 9x">Microsoft Windows 9x</a> becoming a self-contained program loader, and replacing DOS as the most-used PC-compatible program loader. <a class="mw-redirect" href="/wiki/Text_user_interface" title="Text user interface">Text user interface</a> programs included <a href="/wiki/Norton_Commander" title="Norton Commander">Norton Commander</a>, <a href="/wiki/DOS_Navigator" title="DOS Navigator">DOS Navigator</a>, <a href="/wiki/Volkov_Commander" title="Volkov Commander">Volkov Commander</a>, Quarterdesk <a href="/wiki/DESQview" title="DESQview">DESQview</a>, and <a href="/wiki/Borland_Sidekick" title="Borland Sidekick">Sidekick</a>. <a href="/wiki/Graphical_user_interface" title="Graphical user interface">Graphical user interface</a> programs included Digital Research's <a href="/wiki/Graphics_Environment_Manager" title="Graphics Environment Manager">Graphics Environment Manager</a> (originally written for CP/M) and <a href="/wiki/GEOS_(16-bit_operating_system)" title="GEOS (16-bit operating system)">GEOS</a>.</p>
<p>Eventually, the manufacturers of major DOS systems began to include their own environment managers. MS-DOS/IBM DOS 4 included <a href="/wiki/DOS_Shell" title="DOS Shell">DOS Shell</a>;<sup class="reference" id="cite_ref-42"><a href="#cite_note-42">[42]</a></sup> DR DOS 5.0, released the next year, included <a href="/wiki/ViewMAX" title="ViewMAX">ViewMAX</a>, based upon GEM.<sup class="reference" id="cite_ref-43"><a href="#cite_note-43">[43]</a></sup></p>
<h3><span class="mw-headline" id="Terminate_and_Stay_Resident">Terminate and Stay Resident</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DOS&amp;action=edit&amp;section=15" title="Edit section: Terminate and Stay Resident">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a class="mw-redirect" href="/wiki/Terminate_and_Stay_Resident" title="Terminate and Stay Resident">Terminate and Stay Resident</a></div>
<p>DOS is not a multitasking operating system. DOS did however provide a Terminate and Stay Resident (TSR) function which allowed programs to remain resident in memory. These programs could hook the system timer and/or keyboard interrupts to allow themselves to run tasks in the background or to be invoked at any time preempting the current running program effectively implementing a simple form of multitasking on a program-specific basis. The PRINT command did this to implement background print spooling. <a href="/wiki/Borland_Sidekick" title="Borland Sidekick">Borland Sidekick</a>, a popup <a href="/wiki/Personal_information_manager" title="Personal information manager">personal information manager</a> (PIM), also used this technique.</p>
<p>Terminate and Stay Resident programs were also used to provide additional features not available by default. Programs like CED and <a href="/wiki/DOSKEY" title="DOSKEY">DOSKEY</a> provided command line editing facilities beyond what was available in COMMAND.COM. Programs like the Microsoft CD-ROM Extensions (MSCDEX) provided access to files on CD-ROM disks.</p>
<p>Some TSRs could even perform a rudimentary form of task switching. For example, the <a href="/wiki/Shareware" title="Shareware">shareware</a> program Back and Forth (1990)<sup class="reference" id="cite_ref-44"><a href="#cite_note-44">[44]</a></sup> had a hotkey to save the state of the currently-running program to disk, load another program, and switch to it, hence it was possible to switch "back and forth" between programs, albeit slowly due to the disk access required. Back and Forth could not enable <a href="/wiki/Background_process" title="Background process">background processing</a> however; that needed <a href="/wiki/DESQview" title="DESQview">DESQview</a> (on at least a <a href="/wiki/Intel_80386" title="Intel 80386">386</a>).</p>

<li><a href="/wiki/Norton_Commander" title="Norton Commander">Norton Commander</a> and <a href="/wiki/XTree" title="XTree">XTree</a>, file management utilities.</li>
<li><a href="/wiki/Norton_Utilities" title="Norton Utilities">Norton Utilities</a>, a collection of disk and system utilities.</li>

<li>Vern Buerg's popular LIST utility, which displays the content of files in ASCII or HEX.</li>
<li><a href="/wiki/WordPerfect" title="WordPerfect">WordPerfect</a>, a <a href="/wiki/Word_processor" title="Word processor">word processor</a> that is currently produced for the <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Windows</a> platform.</li>
