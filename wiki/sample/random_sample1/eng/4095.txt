<h1 class="firstHeading" id="firstHeading" lang="en">DUAL table</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
</div>
<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><script>function mfTempOpenSection(id){var block=document.getElementById("mf-section-"+id);block.className+=" open-block";block.previousSibling.className+=" open-block";}</script><p>The <b>DUAL</b> table is a special one-row, one-column <a href="/wiki/Table_(database)" title="Table (database)">table</a> present by default in <a class="mw-redirect" href="/wiki/Oracle_database" title="Oracle database">Oracle</a> and other <a href="/wiki/Relational_database" title="Relational database">database</a> installations. In Oracle, the table has a single VARCHAR2(1) column called DUMMY that has a value of 'X'. It is suitable for use in selecting a pseudo column such as SYSDATE or USER.</p>

<h2><span class="mw-headline" id="Example_use">Example use</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DUAL_table&amp;action=edit&amp;section=1" title="Edit section: Example use">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Oracle's <a href="/wiki/SQL" title="SQL">SQL</a> syntax requires the <a href="/wiki/From_(SQL)" title="From (SQL)">FROM</a> clause but some queries don't require any tables - DUAL can be readily used in these cases.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">SELECT</span> <span class="mi">1</span><span class="o">+</span><span class="mi">1</span>
<span class="k">from</span> <span class="n">dual</span><span class="p">;</span>

<span class="k">SELECT</span> <span class="mi">1</span> 
<span class="k">from</span> <span class="n">dual</span><span class="p">;</span>

<span class="k">SELECT</span> <span class="k">USER</span> 
<span class="k">from</span> <span class="n">dual</span><span class="p">;</span>

<span class="k">SELECT</span> <span class="n">SYSDATE</span> 
<span class="k">from</span> <span class="n">dual</span><span class="p">;</span>

<span class="k">SELECT</span> <span class="o">*</span> 
<span class="k">from</span> <span class="n">dual</span><span class="p">;</span>
</pre></div>
<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DUAL_table&amp;action=edit&amp;section=2" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p><a href="/wiki/Charles_Weiss" title="Charles Weiss">Charles Weiss</a> explains why he created DUAL:</p>
<blockquote>
<p>I created the DUAL table as an underlying object in the Oracle Data Dictionary. It was never meant to be seen itself, but instead used inside a view that was expected to be queried. The idea was that you could do a JOIN to the DUAL table and create two rows in the result for every one row in your table. Then, by using GROUP BY, the resulting join could be summarized to show the amount of storage for the DATA extent and for the INDEX extent(s). The name, DUAL, seemed apt for the process of creating a pair of rows from just one.<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup></p>
</blockquote>
<p>The original DUAL table had two rows in it (hence its name), but subsequently it only had one row.</p>
<h2><span class="mw-headline" id="Optimization">Optimization</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DUAL_table&amp;action=edit&amp;section=3" title="Edit section: Optimization">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Beginning with 10g Release 1, Oracle no longer performs physical or logical I/O on the DUAL table, though the table still exists.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup></p>
<p>DUAL is readily available for all the users in database.</p>
<h2><span class="mw-headline" id="In_other_database_systems">In other database systems</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DUAL_table&amp;action=edit&amp;section=4" title="Edit section: In other database systems">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Several other databases (including Microsoft SQL Server, MySQL, PostgreSQL, SQLite, and Teradata) enable one to omit the FROM clause entirely if no table is needed. This avoids the need for any dummy table.</p>
<ul>
<li><a href="/wiki/Firebird_(database_server)" title="Firebird (database server)">Firebird</a> has a one-row system table RDB$DATABASE that is used in the same way as Oracle's DUAL, although it also has a meaning of its own.</li>
<li><a href="/wiki/IBM_DB2" title="IBM DB2">IBM DB2</a> has a view that resolves DUAL when using Oracle Compatibility.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup> It also has a table called sysibm.sysdummy1 that has similar properties to the Oracle DUAL one.</li>
<li><a href="/wiki/Microsoft_Access" title="Microsoft Access">Microsoft Access</a>: A table named DUAL may be created and the single-row constraint enforced via ADO (<a class="external text" href="https://stackoverflow.com/a/7933927/154439" rel="nofollow">Table-less UNION query in MS Access</a>)</li>
<li><a href="/wiki/MySQL" title="MySQL">MySQL</a> allows DUAL to be specified as a table in queries that do not need data from any tables.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup> It is suitable for use in selecting a result function such as <a class="external text" href="http://dev.mysql.com/doc/refman/5.5/en/date-and-time-functions.html#function_sysdate" rel="nofollow">SYSDATE()</a> or <a class="external text" href="http://dev.mysql.com/doc/refman/5.0/en/information-functions.html#function_user" rel="nofollow">USER()</a>, although it is not essential.</li>
<li><a href="/wiki/PostgreSQL" title="PostgreSQL">PostgreSQL</a>: A DUAL-view can be added to ease porting from Oracle.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup></li>
<li><a href="/wiki/SQLite" title="SQLite">SQLite</a>: A VIEW named "dual" that works the same as the Oracle "dual" table can be created as follows: <code class="mw-highlight" dir="ltr" id="" style=""><span class="k">CREATE</span> <span class="k">VIEW</span> <span class="n">dual</span> <span class="k">AS</span> <span class="k">SELECT</span> <span class="s1">'x'</span> <span class="k">AS</span> <span class="n">dummy</span><span class="p">;</span></code></li>
<li><a href="/wiki/SAP_HANA" title="SAP HANA">SAP HANA</a> has a table called DUMMY that works the same as the Oracle "dual" table.</li>
</ul>
