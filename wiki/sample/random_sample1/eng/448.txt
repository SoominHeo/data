<p><b>gzip</b> is a <a href="/wiki/File_format" title="File format">file format</a> and a <a class="mw-redirect" href="/wiki/Software_application" title="Software application">software application</a> used for <a href="/wiki/Data_compression" title="Data compression">file compression and decompression</a>. The program was created by <a href="/wiki/Jean-loup_Gailly" title="Jean-loup Gailly">Jean-loup Gailly</a> and <a href="/wiki/Mark_Adler" title="Mark Adler">Mark Adler</a> as a <a href="/wiki/Free_software" title="Free software">free software</a> replacement for the <a href="/wiki/Compress" title="Compress">compress</a> program used in early <a href="/wiki/Unix" title="Unix">Unix</a> systems, and intended for use by <a href="/wiki/GNU" title="GNU">GNU</a> (the "g" is from "GNU"). Version 0.1 was first publicly released on 31 October 1992, and version 1.0 followed in February 1993.</p>

<p>gzip is based on the <a href="/wiki/DEFLATE" title="DEFLATE">DEFLATE</a> algorithm, which is a combination of <a href="/wiki/LZ77_and_LZ78" title="LZ77 and LZ78">LZ77</a> and <a href="/wiki/Huffman_coding" title="Huffman coding">Huffman coding</a>. DEFLATE was intended as a replacement for <a class="mw-redirect" href="/wiki/LZW" title="LZW">LZW</a> and other <a href="/wiki/Patent" title="Patent">patent</a>-encumbered <a href="/wiki/Data_compression" title="Data compression">data compression</a> <a href="/wiki/Algorithm" title="Algorithm">algorithms</a> which, at the time, limited the usability of compress and other popular archivers.</p>
<p>"gzip" is often also used to refer to the gzip file format, which is:</p>
<ul>
<li>a 10-byte header, containing a <a href="/wiki/Magic_number_(programming)" title="Magic number (programming)">magic number</a> (1f 8b), a version number and a timestamp</li>
<li>optional extra headers, such as the original file name,</li>
<li>a body, containing a DEFLATE-compressed <a class="mw-redirect" href="/wiki/Payload_(communication_and_information_technology)" title="Payload (communication and information technology)">payload</a></li>
<li>an 8-byte footer, containing a <a class="mw-redirect" href="/wiki/CRC-32" title="CRC-32">CRC-32</a> checksum and the length of the original uncompressed data, modulo <span><span class="mwe-math-mathml-inline mwe-math-mathml-a11y mw-math-element" style="display: none;"><math xmlns="http://www.w3.org/1998/Math/MathML">
<semantics>
<mrow class="MJX-TeXAtom-ORD">
<mstyle displaystyle="true" scriptlevel="0">
<msup>
<mn>2</mn>
<mrow class="MJX-TeXAtom-ORD">
<mn>32</mn>
</mrow>
</msup>
</mstyle>
</mrow>
<annotation encoding="application/x-tex">{\displaystyle 2^{32}}</annotation>
</semantics>
</math></span><img alt="2^{32}" aria-hidden="true" class="mwe-math-fallback-image-inline mw-math-element" src="https://wikimedia.org/api/rest_v1/media/math/render/svg/a8c222ea8e5f187a2bb499395b6f4a6f38b43633" style="vertical-align: -0.338ex; width:3.064ex; height:2.676ex;"/></span>.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup></li>
</ul>
<p>Although its file format also allows for multiple such streams to be <a href="/wiki/Concatenation" title="Concatenation">concatenated</a> (zipped files are simply decompressed concatenated as if they were originally one file<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup>), gzip is normally used to compress just single files.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup> Compressed archives are typically created by assembling collections of files into a single <a class="mw-redirect" href="/wiki/Tar_(file_format)" title="Tar (file format)">tar</a> archive, and then compressing that archive with gzip. The final <span style="font-family: monospace, monospace;">.tar.gz</span> or <span style="font-family: monospace, monospace;">.tgz</span> file is usually called a <a class="mw-redirect" href="/wiki/Tar_(file_format)" title="Tar (file format)">tarball</a>.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup></p>
<p>gzip is not to be confused with the <a class="mw-redirect" href="/wiki/ZIP_(file_format)" title="ZIP (file format)">ZIP</a> archive format, which also uses DEFLATE. The ZIP format can hold collections of files without an external archiver, but is less compact than compressed <a class="mw-redirect" href="/wiki/Tar_(file_format)" title="Tar (file format)">tarballs</a> holding the same data, because it compresses files individually and cannot take advantage of redundancy between files (<a href="/wiki/Solid_compression" title="Solid compression">solid compression</a>).</p>

<p>Various implementations of the program have been written. The most commonly known is the GNU Project's implementation using <a class="mw-redirect" href="/wiki/LZ77" title="LZ77">Lempel-Ziv</a> coding (LZ77). <a href="/wiki/OpenBSD" title="OpenBSD">OpenBSD</a>'s version of gzip is actually the compress program, to which support for the gzip format was added in OpenBSD 3.4. The 'g' in this specific version stands for <i><a class="extiw" href="https://en.wiktionary.org/wiki/gratis" title="wikt:gratis">gratis</a></i>.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup> <a href="/wiki/FreeBSD" title="FreeBSD">FreeBSD</a>, <a class="mw-redirect" href="/wiki/DragonFlyBSD" title="DragonFlyBSD">DragonFlyBSD</a> and <a href="/wiki/NetBSD" title="NetBSD">NetBSD</a> use a BSD-licensed implementation instead of the GNU version; it is actually a <a href="/wiki/Command-line_interface" title="Command-line interface">command-line interface</a> for <a href="/wiki/Zlib" title="Zlib">zlib</a> intended to be compatible with the GNU implementation's options.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup> These implementations originally come from <a href="/wiki/NetBSD" title="NetBSD">NetBSD</a>, and supports decompression of <a href="/wiki/Bzip2" title="Bzip2">bzip2</a> and the Unix <a href="/wiki/Pack_(compression)" title="Pack (compression)">pack</a> format.</p>
<h2><span class="mw-headline" id="Derivatives_and_other_uses">Derivatives and other uses</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Gzip&amp;action=edit&amp;section=3" title="Edit section: Derivatives and other uses">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The <a href="/wiki/Tar_(computing)" title="Tar (computing)">tar</a> utility included in most Linux distributions can extract .tar.gz files by passing the <span style="font-family: monospace, monospace;">z</span> option, e.g., <span style="font-family: monospace, monospace;">tar -zxf file.tar.gz</span>.</p>
<p><a href="/wiki/Zlib" title="Zlib">zlib</a> is an abstraction of the DEFLATE algorithm in library form which includes support both for the gzip file format and a lightweight stream format in its API. The zlib stream format, DEFLATE, and the gzip file format were standardized respectively as <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc1950" rel="nofollow">RFC 1950</a>, <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc1951" rel="nofollow">RFC 1951</a>, and <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc1952" rel="nofollow">RFC 1952</a>.</p>
<p>The gzip format is used in <a href="/wiki/HTTP_compression" title="HTTP compression">HTTP compression</a>, a technique used to speed up the sending of <a href="/wiki/HTML" title="HTML">HTML</a> and other content on the <a href="/wiki/World_Wide_Web" title="World Wide Web">World Wide Web</a>. It is one of the three standard formats for HTTP compression as specified in <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc2616" rel="nofollow">RFC 2616</a>. This <a href="/wiki/Request_for_Comments" title="Request for Comments">RFC</a> also specifies a zlib format (called "DEFLATE"), which is equal to the gzip format except that gzip adds eleven bytes of overhead in the form of headers and trailers. Still, the gzip format is sometimes recommended over zlib because <a class="mw-redirect" href="/wiki/Microsoft_Internet_Explorer" title="Microsoft Internet Explorer">Microsoft Internet Explorer</a> does not implement the standard correctly and cannot handle the zlib format as specified in <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc1950" rel="nofollow">RFC 1950</a>.<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup></p>
<p>zlib DEFLATE is used internally by the <a href="/wiki/Portable_Network_Graphics" title="Portable Network Graphics">Portable Network Graphics</a> (PNG) format.</p>
<p>Since the late 1990s, <a href="/wiki/Bzip2" title="Bzip2">bzip2</a>, a file compression utility based on a block-sorting algorithm, has gained some popularity as a gzip replacement. It produces considerably smaller files (especially for source code and other structured text), but at the cost of memory and processing time (up to a factor of 4).<sup class="reference" id="cite_ref-benchmark_9-0"><a href="#cite_note-benchmark-9">[9]</a></sup></p>
<p><a href="/wiki/AdvanceCOMP" title="AdvanceCOMP">AdvanceCOMP</a> and <a href="/wiki/7-Zip" title="7-Zip">7-Zip</a> can produce gzip-compatible files, using an internal DEFLATE implementation with better compression ratios than gzip itself—at the cost of more processor time compared to the reference implementation.</p>
