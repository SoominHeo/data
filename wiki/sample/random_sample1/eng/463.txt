<p><b>JPEG XR</b><sup class="reference" id="cite_ref-ms-billcrow-intro_4-0"><a href="#cite_note-ms-billcrow-intro-4">[4]</a></sup> (abbr. for <b>JPEG</b> e<b>x</b>tended <b>r</b>ange<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup>) is a <a href="/wiki/Image_compression" title="Image compression">still-image compression</a> standard and file format for <a href="/wiki/Continuous_tone" title="Continuous tone">continuous tone</a> photographic images, based on technology originally developed and patented by <a href="/wiki/Microsoft" title="Microsoft">Microsoft</a> under the name <b>HD Photo</b> (formerly <b><a href="/wiki/Windows_Media" title="Windows Media">Windows Media</a> Photo</b>).<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup> It supports both <a href="/wiki/Lossy_compression" title="Lossy compression">lossy</a> and <a href="/wiki/Lossless_compression" title="Lossless compression">lossless compression</a>, and is the preferred image format for Ecma-388 <a href="/wiki/Open_XML_Paper_Specification" title="Open XML Paper Specification">Open XML Paper Specification</a> documents.</p>
<p>Support for the format is available in <a href="/wiki/Adobe_Flash_Player" title="Adobe Flash Player">Adobe Flash Player</a> 11.0, <a href="/wiki/Adobe_AIR" title="Adobe AIR">Adobe AIR</a> 3.0, <a href="/wiki/Sumatra_PDF" title="Sumatra PDF">Sumatra PDF</a> 2.1, <a href="/wiki/Windows_Imaging_Component" title="Windows Imaging Component">Windows Imaging Component</a>, <a href="/wiki/.NET_Framework" title=".NET Framework">.NET Framework</a> 3.0, <a href="/wiki/Windows_Vista" title="Windows Vista">Windows Vista</a>, <a href="/wiki/Windows_7" title="Windows 7">Windows 7</a>, <a href="/wiki/Windows_8" title="Windows 8">Windows 8</a>, <a href="/wiki/Internet_Explorer_9" title="Internet Explorer 9">Internet Explorer 9</a>, <a href="/wiki/Internet_Explorer_10" title="Internet Explorer 10">Internet Explorer 10</a>, <a href="/wiki/Internet_Explorer_11" title="Internet Explorer 11">Internet Explorer 11</a>.<sup class="reference" id="cite_ref-libjxr_7-0"><a href="#cite_note-libjxr-7">[7]</a></sup><sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> As of August 2014, there were still no cameras that shoot photos in the JPEG XR (.JXR) format.</p>

<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JPEG_XR&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Microsoft first announced Windows Media Photo at <a class="mw-redirect" href="/wiki/WinHEC" title="WinHEC">WinHEC</a> 2006,<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup> and then renamed it to HD Photo in November of that year. In July 2007, the <a href="/wiki/Joint_Photographic_Experts_Group" title="Joint Photographic Experts Group">Joint Photographic Experts Group</a> and <a href="/wiki/Microsoft" title="Microsoft">Microsoft</a> announced HD Photo to be under consideration to become a JPEG standard known as <b>JPEG XR</b>.<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup><sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup> On 16 March 2009, JPEG XR was given final approval as <b><a href="/wiki/ITU-T" title="ITU-T">ITU-T</a> Recommendation T.832</b> and starting in April 2009, it became available from the ITU-T in "pre-published" form.<sup class="reference" id="cite_ref-T832_1-1"><a href="#cite_note-T832-1">[1]</a></sup> On 19 June 2009, it passed an ISO/IEC Final Draft International Standard (FDIS) ballot, resulting in final approval as <b><a href="/wiki/International_standard" title="International standard">International Standard</a> ISO/IEC 29199-2</b>.<sup class="reference" id="cite_ref-July09PR_12-0"><a href="#cite_note-July09PR-12">[12]</a></sup><sup class="reference" id="cite_ref-ISOIEC291992v2009_13-0"><a href="#cite_note-ISOIEC291992v2009-13">[13]</a></sup> The ITU-T updated its publication with a <a class="mw-redirect" href="/wiki/Corrigendum" title="Corrigendum">corrigendum</a> approved in December 2009,<sup class="reference" id="cite_ref-T832_1-2"><a href="#cite_note-T832-1">[1]</a></sup> and ISO/IEC issued a new edition with similar corrections on 30 September 2010.<sup class="reference" id="cite_ref-ISOIEC291992v2010_14-0"><a href="#cite_note-ISOIEC291992v2010-14">[14]</a></sup></p>
<p>In 2010, after completion of the image coding specification, the ITU-T and ISO/IEC also published a motion format specification (ITU-T T.833 | ISO/IEC 29199-3), a conformance test set (ITU-T T.834 | ISO/IEC 29199-4), and reference software (ITU-T T.835 | ISO/IEC 29199-5) for JPEG XR. In 2011, they published a technical report describing the workflow architecture for the use of JPEG XR images in applications (ITU-T T.Sup2 | ISO/IEC TR 29199-1).</p>
<h2><span class="mw-headline" id="Description">Description</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JPEG_XR&amp;action=edit&amp;section=2" title="Edit section: Description">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Capabilities">Capabilities</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JPEG_XR&amp;action=edit&amp;section=3" title="Edit section: Capabilities">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>JPEG XR is an image file format that offers several key improvements over <a href="/wiki/JPEG" title="JPEG">JPEG</a>, including:<sup class="reference" id="cite_ref-ms-billcrow-jpegxr_15-0"><a href="#cite_note-ms-billcrow-jpegxr-15">[15]</a></sup></p>
<dl>
<dd>
<dl>
<dt>Better compression</dt>
<dd>JPEG XR file format supports higher compression ratios in comparison to JPEG for encoding an image with equivalent quality.</dd>
</dl>
</dd>
</dl>
<dl>
<dd>
<dl>
<dt>Lossless compression</dt>
<dd>JPEG XR also supports <a href="/wiki/Lossless_compression" title="Lossless compression">lossless compression</a>. The signal processing steps in JPEG XR are the same for both lossless and lossy coding. This makes the lossless mode simple to support and enables the "trimming" of some bits from a lossless compressed image to produce a lossy compressed image.</dd>
</dl>
</dd>
</dl>
<dl>
<dd>
<dl>
<dt>Tile structure support</dt>
<dd>A JPEG XR coded image can be segmented into <i>tile</i> regions. The data for each region can be decoded separately. This enables rapid access to parts of an image without needing to decode the entire image. When a type of tiling referred to as "soft tiling" is used, the tile region structuring can be changed without fully decoding the image and without introducing additional distortion.</dd>
</dl>
</dd>
</dl>
<dl>
<dd>
<dl>
<dt>Support for more color accuracy</dt>
</dl>
</dd>
</dl>
<dl>
<dd>
<dl>
<dd>JPEG XR supports a wide variety of image color representations in addition to the conventional 8-bit-per-sample <a href="/wiki/YUV" title="YUV">YUV</a> (formally <a href="/wiki/YCbCr" title="YCbCr">YCbCr</a>) <a class="mw-redirect" href="/wiki/4:2:0" title="4:2:0">4:2:0</a> encoding that is typically used for the original JPEG standard.</dd>
</dl>
</dd>
</dl>
<dl>
<dd>
<dl>
<dd>For support of images using an <a href="/wiki/RGB_color_space" title="RGB color space">RGB color space</a>, JPEG XR includes an internal conversion to the <a href="/wiki/YCgCo" title="YCgCo">YCgCo</a> color space, and supports a variety of bit depth and color representation packing schemes. These can be used with and without an accompanying alpha channel for shape masking and semi-transparency support, and some of them have much higher precision than what has typically been used for image coding. They include:
<ul>
<li>Low bit-depth packings of RGB into 16 bits per pixel using 5 bits for each channel or 5 bits for red and blue and 6 bits for green</li>
<li>8 bits per component (sometimes called <a class="mw-redirect" href="/wiki/True_Color" title="True Color">true color</a>) packed into 24 or 32 bits per pixel</li>
<li>10 bits per component in a 32 bit packed representation (one of several higher-precision varieties of color representation known as <a class="mw-redirect" href="/wiki/Deep_color" title="Deep color">deep color</a>)</li>
<li>16 bits per component as integers, fixed-point numbers, or half-precision floating-point numbers packed into 48 or 64 bits</li>
<li>32 bits per component as fixed-point numbers or full-precision floating point numbers packed into 96 or 128 bits (for which lossless coding is not supported due to the excessively high precision)</li>
</ul>
</dd>
</dl>
</dd>
</dl>
<dl>
<dd>
<dl>
<dd>JPEG XR also supports 16-bit per component (64-bit per pixel) integer <a href="/wiki/CMYK_color_model" title="CMYK color model">CMYK color model</a>.<sup class="reference" id="cite_ref-ms-billcrow-pixelformat1_16-0"><a href="#cite_note-ms-billcrow-pixelformat1-16">[16]</a></sup></dd>
</dl>
</dd>
</dl>
<dl>
<dd>
<dl>
<dd>16-bit and 32-bit fixed point color component codings are also supported in JPEG XR. In such encodings, the most-significant 4 bits of each color channel are treated as providing additional "headroom" and "toe room" beyond the range of values that represents the nominal black-to-white signal range.</dd>
</dl>
</dd>
</dl>
<dl>
<dd>
<dl>
<dd>Moreover, 16-bit and 32-bit floating point color component codings are supported in JPEG XR. In these cases the image is interpreted as floating point data, although the JPEG XR encoding and decoding steps are all performed using only integer operations (to simplify the compression processing).</dd>
</dl>
</dd>
</dl>
<dl>
<dd>
<dl>
<dd>The shared-exponent floating point color format known as <a href="/wiki/RGBE_image_format" title="RGBE image format">RGBE</a> (<a href="/wiki/Radiance_(software)" title="Radiance (software)">Radiance</a>) is also supported, enabling more faithful storage of <a class="mw-redirect" href="/wiki/High_dynamic_range_imaging" title="High dynamic range imaging">High Dynamic Range (HDR) images</a>.</dd>
</dl>
</dd>
</dl>
<dl>
<dd>
<dl>
<dd>In addition to RGB and CMYK formats, JPEG XR also supports <a href="/wiki/Grayscale" title="Grayscale">grayscale</a> and multi-<a href="/wiki/Channel_(digital_image)" title="Channel (digital image)">channel</a> color encodings with an arbitrary number of channels.</dd>
</dl>
</dd>
</dl>
<dl>
<dd>
<dl>
<dd>The color representations, in most cases, are transformed to an internal color representation. The transformation is entirely reversible, so that this color transformation step does not introduce distortion and thus lossless coding modes can be supported.</dd>
</dl>
</dd>
</dl>
<dl>
<dd>
<dl>
<dt>Transparency map support</dt>
<dd>An <a href="/wiki/Alpha_compositing" title="Alpha compositing">alpha channel</a> may be present to represent transparency, so that alpha blending overlay capability is enabled.</dd>
</dl>
</dd>
</dl>
<dl>
<dd>
<dl>
<dt>Compressed-domain image modification</dt>
<dd>In JPEG XR, full decoding of the image is unnecessary for converting an image from a lossless to lossy encoding, reducing the fidelity of a lossy encoding, or reducing the encoded image resolution.</dd>
</dl>
</dd>
</dl>
<dl>
<dd>
<dl>
<dd>Full decoding is also unnecessary for certain editing operations such as cropping, horizontal or vertical flips, or <a href="/wiki/Cardinal_direction" title="Cardinal direction">cardinal</a> rotations.</dd>
</dl>
</dd>
</dl>
<dl>
<dd>
<dl>
<dd>The tile structure for access to image regions can also be changed without full decoding and without introducing distortion.</dd>
</dl>
</dd>
</dl>
<dl>
<dd>
<dl>
<dt>Metadata support</dt>
<dd>A JPEG XR image file may optionally contain an embedded <a href="/wiki/International_Color_Consortium" title="International Color Consortium">ICC</a> color profile, to achieve consistent color representation across multiple devices.</dd>
</dl>
</dd>
</dl>
<dl>
<dd>
<dl>
<dd><a href="/wiki/Exif" title="Exif">Exif</a> and <a href="/wiki/Extensible_Metadata_Platform" title="Extensible Metadata Platform">XMP</a> metadata formats are also supported.</dd>
</dl>
</dd>
</dl>
<h3><span class="mw-headline" id="Container_format">Container format</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JPEG_XR&amp;action=edit&amp;section=4" title="Edit section: Container format">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>One file <a class="mw-redirect" href="/wiki/Container_format_(digital)" title="Container format (digital)">container format</a> that can be used to store JPEG XR image data is specified in Annex A of the JPEG XR standard. It is a <a href="/wiki/TIFF" title="TIFF">TIFF</a>-like format using a table of <i><a class="new" href="/w/index.php?title=Image_File_Directory&amp;action=edit&amp;redlink=1" title="Image File Directory (page does not exist)">Image File Directory</a></i> (IFD) tags. A JPEG XR file contains image data, optional alpha channel data, metadata, optional <a href="/wiki/Extensible_Metadata_Platform" title="Extensible Metadata Platform">XMP</a> metadata stored as <a href="/wiki/Resource_Description_Framework" title="Resource Description Framework">RDF/XML</a>, and optional <a href="/wiki/Exif" title="Exif">Exif</a> metadata, in IFD tags. The image data is a contiguous self-contained chunk of data. The optional alpha channel, if present, can be compressed as a separate image record, enabling decoding of the image data independently of transparency data in applications which do not support transparency. (Alternatively, JPEG XR also supports an "interleaved" alpha channel format in which the alpha channel data is encoded together with the other image data in a single compressed codestream.)</p>
<p>Being TIFF-based, this format inherits all of the limitations of the TIFF format including the 4 GB file-size limit, which according to the HD Photo specification "will be addressed in a future update".<sup class="reference" id="cite_ref-WMPspec_17-0"><a href="#cite_note-WMPspec-17">[17]</a></sup></p>
<p>New work has been started in the JPEG committee to enable the use of JPEG XR image coding within the JPX file storage format — enabling use of the JPIP protocol, which allows interactive browsing of networked images.<sup class="reference" id="cite_ref-July09PR_12-1"><a href="#cite_note-July09PR-12">[12]</a></sup> Additionally, a Motion JPEG XR specification was approved as an ISO standard for motion (video) compression in March 2010.<sup class="reference" id="cite_ref-18"><a href="#cite_note-18">[18]</a></sup></p>

<p>JPEG XR's design<sup class="reference" id="cite_ref-T832_1-3"><a href="#cite_note-T832-1">[1]</a></sup><sup class="reference" id="cite_ref-SPIEoverview_19-0"><a href="#cite_note-SPIEoverview-19">[19]</a></sup> is conceptually very similar to <a href="/wiki/JPEG" title="JPEG">JPEG</a>: the source image is optionally converted to a <a href="/wiki/Luma_(video)" title="Luma (video)">luma-chroma</a> colorspace, the chroma planes are optionally <a href="/wiki/Chroma_subsampling" title="Chroma subsampling">subsampled</a>, each plane is divided into fixed-size blocks, the blocks are transformed into the frequency domain, and the frequency coefficients are quantized and <a href="/wiki/Entropy_encoding" title="Entropy encoding">entropy coded</a>. Major differences include the following:</p>
<ul>
<li>JPEG supports bit depths of 8 and 12 bits; JPEG XR supports bit depths of up to 32 bits. JPEG XR also supports lossless and lossy compression of floating-point image data (by representing the floating-point values in an <a class="mw-redirect" href="/wiki/IEEE_754" title="IEEE 754">IEEE 754</a>-like format, and encoding them as though they were integers) and <a href="/wiki/RGBE_image_format" title="RGBE image format">RGBE</a> imagery.</li>
<li><a class="mw-redirect" href="/wiki/JFIF" title="JFIF">JFIF</a> and other typical image encoding practices specify a linear transformation from RGB to <a href="/wiki/YCbCr" title="YCbCr">YCbCr</a>, which is slightly lossy in practice because of <a class="mw-redirect" href="/wiki/Roundoff_error" title="Roundoff error">roundoff error</a>. JPEG XR specifies a lossless colorspace transformation, given (for RGB) by</li>
</ul>

<ul>
<li>While JPEG uses 8 × 8 blocks for its frequency transformation, JPEG XR primarily uses 4 × 4 block transforms. (2 × 4 and 2 × 2 transformations are also defined for special cases involving <a href="/wiki/Chroma_subsampling" title="Chroma subsampling">chroma subsampling</a>; encoder options include YUV_444, YUV_422, YUV_420, and a monochrome Y_only.)<sup class="reference" id="cite_ref-DPK_20-0"><a href="#cite_note-DPK-20">[20]</a></sup></li>
<li>While JPEG uses a single transformation stage, JPEG XR applies its 4 × 4 core transform in a two-level hierarchical fashion within 16 × 16 <i>macroblock</i> regions. This gives the transform a <a href="/wiki/Wavelet" title="Wavelet">wavelet</a>-like multi-resolution hierarchy and improves its compression capability.</li>
<li>The <a href="/wiki/Discrete_cosine_transform" title="Discrete cosine transform">DCT</a>, the frequency transformation used by JPEG, is slightly lossy because of roundoff error. JPEG XR uses a type of integer transform employing a <a href="/wiki/Lifting_scheme" title="Lifting scheme">lifting scheme</a>.<sup class="reference" id="cite_ref-LiftingTransform_21-0"><a href="#cite_note-LiftingTransform-21">[21]</a></sup> The required transform, called the Photo Core Transform (PCT), resembles a 4 × 4 DCT but is lossless (exactly invertible). In fact, it is a particular realization of a larger family of binary-friendly multiplier-less transforms called the binDCT.<sup class="reference" id="cite_ref-binDCT_22-0"><a href="#cite_note-binDCT-22">[22]</a></sup></li>
<li>JPEG XR allows an optional overlap prefiltering step, called the Photo Overlap Transform (POT), before each of its 4 × 4 core transform PCT stages.<sup class="reference" id="cite_ref-LiftingTransform_21-1"><a href="#cite_note-LiftingTransform-21">[21]</a></sup> The filter operates on 4 × 4 blocks which are offset by 2 samples in each direction from the 4 × 4 core transform blocks. Its purpose is to improve compression capability and reduce <a href="/wiki/Compression_artifact" title="Compression artifact">block-boundary artifacts</a> at low bitrates. At high bitrates, where such artifacts are typically not a problem, the prefiltering can be omitted to reduce encoding and decoding time. The overlap filtering is constructed using integer operations following a <a href="/wiki/Lifting_scheme" title="Lifting scheme">lifting scheme</a>, so that it is also lossless. When appropriately combined, the POT and the PCT in JPEG-XR form a lapped transform.<sup class="reference" id="cite_ref-prepost_23-0"><a href="#cite_note-prepost-23">[23]</a></sup></li>
<li>In JPEG, the image DC coefficients of the DCT blocks are predicted by applying DC prediction from the left neighbor transform block, and no other coeffients are predicted. In JPEG XR, 4 × 4 blocks are grouped into macroblocks of 16 × 16 samples, and the 16 DC coefficients from the 4 × 4 blocks of each macroblock are passed through another level of frequency transformation, leaving three types of coefficients to be entropy coded: the macroblock DC coefficients (called DC), macroblock-level AC coefficients (called "lowpass"), and lower-level AC coefficients (called AC). Prediction of coefficient values across transform blocks is applied to the DC coefficients and to an additional row or column of AC coefficients as well.</li>
<li>JPEG XR supports the encoding of an image by decomposing it into smaller individual rectangular <i>tile</i> area regions. Each tile area can be decoded independently from the other areas of the picture. This allows fast access to spatial areas of pictures without decoding the entire picture.</li>
<li>JPEG XR's entropy coding phase is more adaptive and complex than JPEG's, involving a DC and AC coefficient prediction scheme, adaptive coefficient reordering (in contrast to JPEG's fixed zigzag ordering), and a form of adaptive Huffman coding for the coefficients themselves.</li>
<li>JPEG uses a single quantization step size per DC/AC component per color plane per image. JPEG XR allows a selection of DC quantization step sizes on a <i>tile</i> region basis, and allows lowpass and AC quantization step sizes to vary from macroblock to macroblock.</li>
<li>Because all encoding phases except quantization are lossless, JPEG XR is lossless when all quantization coefficients are equal to 1. This is not true of JPEG. JPEG defines a separate lossless mode which does not use the DCT, but it is not implemented by <a href="/wiki/Libjpeg" title="Libjpeg">libjpeg</a> and therefore not widely supported.</li>
</ul>
<p>The HD Photo bitstream specification claims that "HD Photo offers image quality comparable to <a class="mw-redirect" href="/wiki/JPEG-2000" title="JPEG-2000">JPEG-2000</a> with computational and memory performance more closely comparable to JPEG", that it "delivers a lossy compressed image of better perceptive quality than JPEG at less than half the file size", and that "lossless compressed images … are typically 2.5 times smaller than the original uncompressed data".</p>

<h2><span class="mw-headline" id="Licensing">Licensing</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JPEG_XR&amp;action=edit&amp;section=7" title="Edit section: Licensing">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Microsoft has patents on the technology in JPEG XR. A Microsoft representative stated in a January 2007 interview that in order to encourage the adoption and use of HD Photo, the specification is made available under the <a href="/wiki/Microsoft_Open_Specification_Promise" title="Microsoft Open Specification Promise">Microsoft Open Specification Promise</a>, which asserts that Microsoft allows implementation of the specification for free, and will not file suits on the patented technology for its implementation,<sup class="reference" id="cite_ref-45"><a href="#cite_note-45">[45]</a></sup> as reportedly stated by Josh Weisberg, director of Microsoft's Rich Media Group. As of 15 August 2010, Microsoft made the resulting JPEG XR standard available under its <a class="mw-redirect" href="/wiki/Microsoft_Community_Promise" title="Microsoft Community Promise">Community Promise</a>.<sup class="reference" id="cite_ref-MCP_46-0"><a href="#cite_note-MCP-46">[46]</a></sup></p>
<p>In July 2010, reference software to implement the JPEG XR standard was published as ITU-T Recommendation T.835 and <a href="/wiki/International_standard" title="International standard">International Standard</a> ISO/IEC 29199-5. Microsoft included these publications in the list of specifications covered by its <a class="mw-redirect" href="/wiki/Microsoft_Community_Promise" title="Microsoft Community Promise">Community Promise</a>.<sup class="reference" id="cite_ref-MCP_46-1"><a href="#cite_note-MCP-46">[46]</a></sup></p>
<p>In April 2013, Microsoft released an open source JPEG XR library under the <a href="/wiki/BSD_licenses" title="BSD licenses">BSD licence</a>.<sup class="reference" id="cite_ref-47"><a href="#cite_note-47">[47]</a></sup><sup class="reference" id="cite_ref-48"><a href="#cite_note-48">[48]</a></sup> This resolved any licensing issues with the library being implemented in software packages distributed under popular open source licences such as the <a href="/wiki/GNU_General_Public_License" title="GNU General Public License">GNU General Public License</a>, with which the previously released "HD Photo Device Porting Kit"<sup class="reference" id="cite_ref-49"><a href="#cite_note-49">[49]</a></sup> was incompatible.</p>

<ul>
<li><a href="/wiki/JPEG" title="JPEG">JPEG</a>, an image format used for lossy compression, which JPEG XR lossy is comparable with</li>
<li><a href="/wiki/JPEG_2000" title="JPEG 2000">JPEG 2000</a>, an improvement intended to replace JPEG by the JPEG committee as of 2000</li>
<li><a href="/wiki/Portable_Network_Graphics" title="Portable Network Graphics">PNG</a>, a format for lossless compression, which JPEG XR lossless is comparable with</li>
<li><a href="/wiki/WebP" title="WebP">WebP</a>, a format with lossy (<a href="/wiki/WebM" title="WebM">WebM</a>) or lossless compression, proposed by <a href="/wiki/Google" title="Google">Google</a> in 2010</li>
<li><a href="/wiki/Better_Portable_Graphics" title="Better Portable Graphics">Better Portable Graphics</a>, a proposal by <a href="/wiki/Fabrice_Bellard" title="Fabrice Bellard">Fabrice Bellard</a> in 2014 based on <a href="/wiki/High_Efficiency_Video_Coding" title="High Efficiency Video Coding">HEVC</a></li>
</ul>
