<p><b>EAR</b> (<b><i>E</i>nterprise <i>A</i>pplication a<i>R</i>chive</b>) is a <a href="/wiki/File_format" title="File format">file format</a> used by <a class="mw-redirect" href="/wiki/Java_EE" title="Java EE">Java EE</a> for packaging one or more modules into a single archive so that the deployment of the various <a class="mw-redirect" href="/wiki/J2EE_application" title="J2EE application">modules</a> onto an <a href="/wiki/Application_server" title="Application server">application server</a> happens simultaneously and coherently. It also contains <a href="/wiki/XML" title="XML">XML</a> files called <a href="/wiki/Deployment_descriptor" title="Deployment descriptor">deployment descriptors</a> which describe how to deploy the modules.</p>
<p><a href="/wiki/Apache_Ant" title="Apache Ant">Ant</a>, <a href="/wiki/Apache_Maven" title="Apache Maven">Maven</a>, or <a href="/wiki/Gradle" title="Gradle">Gradle</a> can be used to build EAR files.</p>

<h2><span class="mw-headline" id="File_structure">File structure</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=EAR_(file_format)&amp;action=edit&amp;section=1" title="Edit section: File structure">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>An EAR file is a standard <a href="/wiki/JAR_(file_format)" title="JAR (file format)">JAR file</a> (and therefore a <a href="/wiki/Zip_(file_format)" title="Zip (file format)">Zip</a> file) with a .ear extension, with one or more entries representing the modules of the application, and a metadata directory called <code>META-INF</code> which contains one or more deployment descriptors.</p>
<h3><span class="mw-headline" id="Module">Module</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=EAR_(file_format)&amp;action=edit&amp;section=2" title="Edit section: Module">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Developers can embed various artifacts within an EAR file for deployment by application servers:</p>
<ul>
<li>A Web module has a <a class="mw-redirect" href="/wiki/Sun_WAR_(file_format)" title="Sun WAR (file format)">.war</a> extension. It is a deployable unit that consists of one or more web components, other resources, and a <a href="/wiki/Web_application" title="Web application">web application</a> <a href="/wiki/Deployment_descriptor" title="Deployment descriptor">deployment descriptor</a>. The web module is contained in a <a href="/wiki/Hierarchy" title="Hierarchy">hierarchy</a> of directories and files in a standard web application format.</li>
<li><a class="mw-redirect" href="/wiki/POJO" title="POJO">POJO</a> Java classes may be deployed in <a class="mw-redirect" href="/wiki/.jar" title=".jar">.jar</a> files.</li>
<li>An <a class="mw-redirect" href="/wiki/Enterprise_Java_Bean" title="Enterprise Java Bean">Enterprise Java Bean</a> module has a <a class="mw-redirect" href="/wiki/.jar" title=".jar">.jar</a> extension, and contains in its own <code>META-INF</code> directory descriptors describing the persistent classes deployed. Deployed entity beans become visible to other components and, if remotely exported, to remote clients. <a class="mw-redirect" href="/wiki/Enterprise_JavaBean#Message_driven_beans" title="Enterprise JavaBean">Message Beans</a> and <a class="mw-redirect" href="/wiki/Session_Beans" title="Session Beans">Session Beans</a> are available for remote access.</li>
<li>A <a class="mw-redirect" href="/wiki/Resource_Adapter" title="Resource Adapter">Resource Adapter</a> module has a <a class="mw-redirect" href="/wiki/Resource_Adapter_aRchive" title="Resource Adapter aRchive">.rar</a> extension.</li>
</ul>
<h3><span class="mw-headline" id="Class_isolation">Class isolation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=EAR_(file_format)&amp;action=edit&amp;section=3" title="Edit section: Class isolation">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Most application servers load classes from a deployed EAR file as an isolated tree of Java <a class="mw-redirect" href="/wiki/Classloader" title="Classloader">classloaders</a>, isolating the application from other applications, but sharing classes between deployed modules. For example, a deployed WAR file would be able to create instances of classes defined in a JAR file that was also included in the containing EAR file, but not necessarily those in JAR files in other EAR files. One key reason for this behavior is to allow complete separation between applications which use static singletons (e.g. Log4J), which would otherwise confuse the configuration between separate applications. This also enables different versions of applications and libraries to be deployed side-by-side.</p>
<p>The <a class="mw-redirect" href="/wiki/JBoss" title="JBoss">JBoss</a> application servers before Version 5 were notable in that it does not isolate deployed components. A web application deployed in one EAR file would have access to classes in other EAR and WAR files. This is a somewhat controversial policy. The <i>Unified Classloader</i> design reduces communications overhead between running applications, as class data can be shared by reference or simple copies. It also allows developers to avoid having to understand the problems that a tree of classloaders can create. However, it prevents different versions of dependent libraries from being deployed in separate applications. JBoss 4.0.2 switched to a hierarchical classloader, but in version 4.0.3 it reverted to a Unified Classloader for backwards compatibility reasons. There is now a configuration option to change this behavior. JBoss 5.x, 6.x and 7.x no longer use Unified Classloading.</p>

<h3><span class="mw-headline" id="META-INF_directory">META-INF directory</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=EAR_(file_format)&amp;action=edit&amp;section=4" title="Edit section: META-INF directory">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The <code>META-INF</code> directory contains at least the <code>application.xml</code> deployment descriptor, known as the <i><a class="external text" href="http://www.oracle.com/webfolder/technetwork/jsc/xml/ns/javaee/index.html" rel="nofollow">Java EE Deployment Descriptor</a></i>. It contains the following XML entities:</p>
<ul>
<li><code>icon</code>, which specifies the locations for the images that represent the application. A subdivision is made for <code>small-icon</code> and <code>large-icon</code>.</li>
<li><code>display-name</code>, which identifies the application</li>
<li><code>description</code></li>
<li>A <code>module</code> element for each module in the archive</li>
<li>Zero or more <code>security-role</code> elements for the global security roles in the application</li>
</ul>
<p>Each <code>module</code> element contains an <code>ejb</code>, <code>web</code> or <code>java</code> element which describes the individual modules within the application. Web modules also provide a <code>context-root</code> which identifies the web module by its URL.</p>
<p>Next to the Java EE deployment descriptor there can be zero or more <i>runtime deployment descriptors</i>. These are used to configure implementation-specific Java EE parameters.</p>
