<h1 class="firstHeading" id="firstHeading" lang="en">Year 2038 problem</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">

<p>The <b>Year 2038 problem</b> is an issue for computing and data storage situations in which time values are stored or calculated as a <a href="/wiki/Integer_(computer_science)" title="Integer (computer science)">signed 32-bit integer</a>, and this number is interpreted as the number of seconds since 00:00:00 <a href="/wiki/Coordinated_Universal_Time" title="Coordinated Universal Time">UTC</a> on 1 January 1970 (the <a href="/wiki/Epoch_(reference_date)" title="Epoch (reference date)">epoch</a>).<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> Such implementations cannot encode times after 03:14:07 UTC on 19 January 2038, a problem similar to but not entirely analogous to the <a href="/wiki/Year_2000_problem" title="Year 2000 problem">Y2K problem</a> (also known as the Millennium Bug), in which 2-digit values representing the number of years since 1900 could not encode the year 2000 or later. Most <a href="/wiki/32-bit" title="32-bit">32-bit</a> <a href="/wiki/Unix-like" title="Unix-like">Unix-like</a> systems store and manipulate time in this <a href="/wiki/Unix_time" title="Unix time">Unix time</a> format, so the year 2038 problem is sometimes referred to as the <b>Unix Millennium Bug</b> by association.</p>

<h2><span class="mw-headline" id="Technical_cause">Technical cause</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Year_2038_problem&amp;action=edit&amp;section=1" title="Edit section: Technical cause">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The latest time that can be represented in Unix's <a href="/wiki/Integer_(computer_science)" title="Integer (computer science)">signed 32-bit integer</a> time format is 03:14:07 <a href="/wiki/Coordinated_Universal_Time" title="Coordinated Universal Time">UTC</a> on Tuesday, 19 January 2038 (2,147,483,647 seconds after 1 January 1970).<sup class="reference" id="cite_ref-spinellis_2-0"><a href="#cite_note-spinellis-2">[2]</a></sup> Times beyond that will wrap around and be stored internally as a negative number, which these systems will interpret as having occurred on 13 December 1901 rather than 19 January 2038. This is caused by <a href="/wiki/Integer_overflow" title="Integer overflow">integer overflow</a>. The counter runs out of usable digit bits, flips the sign bit instead, and reports a maximally negative number (continuing to count <i>up</i>, toward zero). Resulting erroneous calculations on such systems are likely to cause problems for users and other relying parties.</p>
<p>Programs that work with future dates will begin to run into problems sooner; for example a program that works with dates 20 years in the future will have to be fixed no later than 2018.</p>
<h2><span class="mw-headline" id="Early_problems">Early problems</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Year_2038_problem&amp;action=edit&amp;section=2" title="Edit section: Early problems">edit</a><span class="mw-editsection-bracket">]</span></span></h2>

<p>In May 2006, reports surfaced of an early manifestation of the Y2038 problem in the <a href="/wiki/AOLserver" title="AOLserver">AOLserver</a> software. The software was designed with a <a href="/wiki/Kludge" title="Kludge">kludge</a> to handle a database request that should "never" time out. Rather than specifically handling this special case, the initial design simply specified an arbitrary <a href="/wiki/Timeout_(computing)" title="Timeout (computing)">time-out</a> date in the future. The default configuration for the server specified that the request should time out after one billion seconds. One billion seconds (approximately thirty-two years) after 9:27.28 pm on 12 May 2006 is beyond the 2038 cutoff date. Thus, after this time, the time-out calculation overflowed and returned a date that was actually in the past, causing the software to crash. When the problem was discovered, AOLServer operators had to edit the configuration file and set the time-out to a lower value.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup><sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></p>
<p>Players of games or apps which are programmed to impose waiting periods<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> are running into this problem when they attempt to work around the waiting period on devices which harbor the coding, by manually setting their devices (such as the <a class="mw-redirect" href="/wiki/Nexus_7_(2012_version)" title="Nexus 7 (2012 version)">Nexus 7</a><sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup>) to a date past 19 January 2038.</p>
<p>In 2011, a likely early incident of the Year 2038 problem was reported when the United States Congressional Budget Office's economic forecasting software was found to be incapable of running economic analysis past the year 2037 without crashing. Congressman Paul Ryan reported to the media "I asked CBO to run the model going out and they told me that their computer simulation crashes in 2037 because CBO can’t conceive of any way in which the economy can continue past the year 2037".<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup></p>
<h2><span class="mw-headline" id="Vulnerable_systems">Vulnerable systems</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Year_2038_problem&amp;action=edit&amp;section=3" title="Edit section: Vulnerable systems">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p><a class="mw-redirect" href="/wiki/Embedded_systems" title="Embedded systems">Embedded systems</a> that use dates for either computation or diagnostic logging are most likely to be affected by the 2038 bug.</p>
<p>Most worrisome are the command and control systems for nuclear weapon and power systems. These systems were largely designed and built starting in the 1970s and are still in use today. In The United States the General Accountability Office (GAO) released a May 2016 report <a class="external text" href="http://www.gao.gov/assets/680/677436.pdf" rel="nofollow">GAO Report on Legacy Systems</a> noting that the Strategic Automated Command and Control System, (SACCS) which is responsible for coordinating "the operational functions of the United States’ nuclear forces, such as intercontinental ballistic missiles, nuclear bombers, and tanker support craft" "runs on an <a href="/wiki/IBM_Series/1" title="IBM Series/1">IBM Series/1</a> Computer". Which is a computing system built in the 1970s and uses 8-inch floppy disks. Due to state secrecy it is not publicly knowable what a Unix epoch date reset might do to the control logic of the defense systems of nuclear states like the USA and the former USSR but the obvious, maximally negative outcome, is an automated <a href="/wiki/Dead_Hand_(nuclear_war)" title="Dead Hand (nuclear war)">Dead Hand</a> strike due to an epoch vulnerable component of the control system going off-line and that status being perceived by the system as indicating a successful attack that must be retaliated against.<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup><sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup></p>
<p>Many transportation systems from flight to automobiles use embedded systems extensively. In automotive systems, this may include anti-lock braking system (ABS), electronic stability control (ESC/ESP), traction control (TCS) and automatic four-wheel drive; aircraft may use inertial guidance systems and GPS receivers. However this does not imply that all these systems will suffer from the bug. Many such systems will not require access to dates. For those that do, those systems which only track the difference between times/dates and not absolute times/dates will, by the nature of the calculation, not experience a problem. This is the case for automotive diagnostics based on legislative standards such as CARB (<a href="/wiki/California_Air_Resources_Board" title="California Air Resources Board">California Air Resources Board</a>).<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup></p>
<p>Another major use of embedded systems is in communications devices, including cell phones and Internet appliances (routers, wireless access points, etc.) which rely on storing an accurate time and date and are increasingly based on UNIX-like operating systems. For example, the bug makes some <a href="/wiki/Android_(operating_system)" title="Android (operating system)">Android</a> devices crash and not restart when the time is changed to that date.<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup></p>
<p>As of 2012<sup class="plainlinks noprint asof-tag update" style="display:none;"><a class="external text" href="//en.wikipedia.org/w/index.php?title=Year_2038_problem&amp;action=edit">[update]</a></sup>, most embedded systems use <a href="/wiki/8-bit" title="8-bit">8-bit</a> or <a href="/wiki/16-bit" title="16-bit">16-bit</a> <a class="mw-redirect" href="/wiki/Microprocessors" title="Microprocessors">microprocessors</a>,<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup> even as desktop systems are transitioning to <a class="mw-redirect" href="/wiki/64-bit" title="64-bit">64-bit</a> systems.</p>
<p>Despite the modern <a class="mw-redirect" href="/wiki/Moore%27s_Law" title="Moore's Law">18–24-month</a> generational update in computer systems technology, embedded systems are designed to last the lifetime of the machine in which they are a component. It is conceivable that some of these systems may still be in use in 2038. It may be impractical or, in some cases, impossible to upgrade the software running these systems, ultimately requiring replacement if 32-bit <tt><a class="mw-redirect" href="/wiki/Time_t" title="Time t">time_t</a></tt> limitations are to be corrected.</p>
<p><a href="/wiki/MySQL" title="MySQL">MySQL</a> database's built-in functions like <tt>UNIX_TIMESTAMP()</tt> will return 0 after 03:14:07 <a class="mw-redirect" href="/wiki/UTC" title="UTC">UTC</a> on 19 January 2038.<sup class="reference" id="cite_ref-13"><a href="#cite_note-13">[13]</a></sup></p>
<h2><span class="mw-headline" id="Data_structures_with_time_problems">Data structures with time problems</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Year_2038_problem&amp;action=edit&amp;section=4" title="Edit section: Data structures with time problems">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Many data structures in use today have 32-bit time representations embedded into their structure. A full list of these data structures is virtually impossible to derive but there are well-known data structures that have the Unix time problem:</p>
<ul>
<li>file systems (many file systems use only 32 bits to represent times in <a href="/wiki/Inode" title="Inode">inodes</a>)</li>
<li>binary file formats (that use 32-bit time fields)</li>
<li>databases (that have 32-bit time fields)</li>
<li>database query languages, like <a href="/wiki/SQL" title="SQL">SQL</a> that have <tt>UNIX_TIMESTAMP()</tt> like commands</li>
</ul>
<p>Examples of systems utilizing data structures that may contain 32-bit time representations include:</p>
<ul>
<li><a href="/wiki/COBOL" title="COBOL">COBOL</a> systems of 1970s–1990s vintage that have not been replaced by 2038-compliant systems</li>
<li>embedded factory, refinery control and monitoring subsystems</li>
<li>assorted medical devices</li>
<li>assorted military devices</li>
</ul>
<p>Any system making use of data structures containing 32-bit time representations will present risk. The degree of risk is dependent on the mode of failure.</p>
<h2><span class="mw-headline" id="NTP_timestamps">NTP timestamps</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Year_2038_problem&amp;action=edit&amp;section=5" title="Edit section: NTP timestamps">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The <a href="/wiki/Network_Time_Protocol" title="Network Time Protocol">Network Time Protocol</a> has a related overflow issue, which manifests itself in 2036, rather than 2038. The 64-bit timestamps used by NTP consist of a 32-bit part for seconds and a 32-bit part for fractional second, giving NTP a time scale that <a href="/wiki/Integer_overflow" title="Integer overflow">rolls over</a> every 2<sup>32</sup> seconds (136 years) and a theoretical resolution of 2<sup>−32</sup> seconds (233 picoseconds). NTP uses an epoch of 1 January 1900. The first rollover occurs in 2036, prior to the UNIX year 2038 problem.</p>
<p>Implementations should disambiguate NTP time using a knowledge of the approximate time from other sources. Since NTP only works with the differences between timestamps and never their absolute values, the wraparound is invisible as long as the timestamps are within 68 years of each other.</p>
<p>This means that for NTP the rollover will be invisible for most running systems, since they will have the correct time to within a very small tolerance. However, systems that are starting up need to know the date within no more than 68 years. Given the large allowed error, it is not expected that this is too onerous a requirement. One suggested method is to set the clock to no earlier than the system build date or the release date of the current version of the NTP software. Many systems use a battery-powered hardware clock to avoid this problem.</p>
<p>Even so, future versions of NTP may extend the time representation to 128 bits: 64 bits for the second and 64 bits for the fractional-second. The current NTP4 format has support for <i>Era Number</i> and <i>Era Offset</i>, that when used properly should aid fixing date rollover issues. According to Mills, "The 64 bit value for the fraction is enough to resolve the amount of time it takes a <a href="/wiki/Photon" title="Photon">photon</a> to pass an <a href="/wiki/Electron" title="Electron">electron</a> at the speed of light. The 64 bit second value is enough to provide unambiguous time representation until the <a href="/wiki/Ultimate_fate_of_the_universe" title="Ultimate fate of the universe">universe goes dim</a>."<sup class="reference" id="cite_ref-14"><a href="#cite_note-14">[14]</a></sup><sup class="reference" id="cite_ref-15"><a href="#cite_note-15">[note 1]</a></sup> <span id="292.2C277.2C026.2C596"></span></p>
<h2><span class="mw-headline" id="Solutions">Solutions</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Year_2038_problem&amp;action=edit&amp;section=6" title="Edit section: Solutions">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>There is no universal solution for the Year 2038 problem. Any change to the definition of the <tt>time_t</tt> data type would result in <a href="/wiki/Binary_code_compatibility" title="Binary code compatibility">code compatibility</a> problems in any application in which date and time representations are dependent on the nature of the signed 32-bit <tt>time_t</tt> integer. For example, changing <tt>time_t</tt> to an unsigned 32-bit integer, which would extend the range to the year 2106, would adversely affect programs that store, retrieve, or manipulate dates prior to 1970, as such dates are represented by negative numbers. Increasing the size of the <a class="mw-redirect" href="/wiki/Time_t" title="Time t"><tt>time_t</tt></a> type to 64-bit in an existing system would cause incompatible changes to the layout of structures and the binary interface of functions.</p>
<p>There is also no universal solution for the issue with <a href="/wiki/Digital_Video_Broadcasting" title="Digital Video Broadcasting">DVB</a> and <a class="mw-redirect" href="/wiki/ATSC" title="ATSC">ATSC</a> real time transmitted dates due to issues with legacy receivers. The issue has yet to be acknowledged or resolved by either organization. The only work around would be to discontinue all time related <a href="/wiki/Metadata" title="Metadata">metadata</a> services such as programming guides and automatic date synchronization after the affected dates. One possible option would be to create new table types for the affected part of the specifications and use <a href="/wiki/ISO_8601" title="ISO 8601">ISO 8601</a> date strings rather than fixed integers as is used in <a href="/wiki/ISO_9660" title="ISO 9660">ISO 9660</a> and <a class="mw-redirect" href="/wiki/ISO_13346" title="ISO 13346">ISO 13346</a> filesystems.</p>
<p>Most operating systems designed to run on 64-bit <a href="/wiki/Hardware_architecture" title="Hardware architecture">hardware</a> already use signed 64-bit <tt>time_t</tt> integers. Using a signed 64-bit value introduces a new wraparound date that is over twenty times greater than the estimated <a href="/wiki/Age_of_the_universe" title="Age of the universe">age of the universe</a>: approximately 292 billion years from now, at 15:30:08 UTC on Sunday, 4 December 292,277,026,596. The ability to make computations on dates is limited by the fact that <tt>tm_year</tt> uses a signed 32 bit integer value starting at 1900 for the year. This limits the year to a maximum of 2,147,485,547 (2,147,483,647 + 1900).<sup class="reference" id="cite_ref-16"><a href="#cite_note-16">[15]</a></sup></p>
<p>Starting with <a href="/wiki/NetBSD" title="NetBSD">NetBSD</a> version 6.0 (released in October 2012), the NetBSD operating system uses a 64-bit <tt>time_t</tt> for both 32-bit and 64-bit architectures. Applications that were compiled for an older NetBSD release with 32-bit <tt>time_t</tt> are supported via a binary compatibility layer, but such older applications will still suffer from the Year 2038 problem.<sup class="reference" id="cite_ref-17"><a href="#cite_note-17">[16]</a></sup></p>
<p><a href="/wiki/OpenBSD" title="OpenBSD">OpenBSD</a> since version 5.5, released in May 2014, also uses a 64-bit <tt>time_t</tt> for both 32-bit and 64-bit architectures. In contrast to <a href="/wiki/NetBSD" title="NetBSD">NetBSD</a>, there is no binary compatibility layer. Therefore, applications expecting a 32-bit <tt>time_t</tt> and applications using anything different from <tt>time_t</tt> to store time values may break.<sup class="reference" id="cite_ref-18"><a href="#cite_note-18">[17]</a></sup></p>
<p><a href="/wiki/Linux" title="Linux">Linux</a> uses a 64-bit <tt>time_t</tt> for 64-bit architectures only; the pure 32-bit <a href="/wiki/Application_binary_interface" title="Application binary interface">ABI</a> is not changed due to backward compatibility.<sup class="reference" id="cite_ref-Pondering2038_19-0"><a href="#cite_note-Pondering2038-19">[18]</a></sup> There is ongoing work, mostly for <a href="/wiki/Linux_on_embedded_systems" title="Linux on embedded systems">embedded Linux</a> systems, to support 64-bit <tt>time_t</tt> on 32-bit architectures, too.<sup class="reference" id="cite_ref-20"><a href="#cite_note-20">[19]</a></sup><sup class="reference" id="cite_ref-21"><a href="#cite_note-21">[20]</a></sup></p>
<p>The <a href="/wiki/X32_ABI" title="X32 ABI">x32 ABI</a> for <a href="/wiki/Linux" title="Linux">Linux</a> (which defines an environment for programs with 32-bit addresses but running the processor in 64-bit mode) uses a 64-bit <tt>time_t</tt>. Since it was a new environment, there was no need for special compatibility precautions.<sup class="reference" id="cite_ref-Pondering2038_19-1"><a href="#cite_note-Pondering2038-19">[18]</a></sup></p>
<p>Alternative proposals have been made (some of which are in use), such as storing either <a href="/wiki/Millisecond" title="Millisecond">milliseconds</a> or <a href="/wiki/Microsecond" title="Microsecond">microseconds</a> since an epoch (typically either 1 January 1970 or 1 January 2000) in a signed 64-bit integer, providing a minimum range of 300,000 years.<sup class="reference" id="cite_ref-22"><a href="#cite_note-22">[21]</a></sup><sup class="reference" id="cite_ref-23"><a href="#cite_note-23">[22]</a></sup> Other proposals for new time representations provide different precisions, ranges, and sizes (almost always wider than 32 bits), as well as solving other related problems, such as the handling of <a href="/wiki/Leap_second" title="Leap second">leap seconds</a>. In particular, TAI64<sup class="reference" id="cite_ref-24"><a href="#cite_note-24">[23]</a></sup> is an implementation of the <a href="/wiki/International_Atomic_Time" title="International Atomic Time"><i>Temps Atomique International</i></a> standard, the current international real-time standard for defining a second and frame of reference.</p>
