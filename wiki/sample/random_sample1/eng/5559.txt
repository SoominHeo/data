<p>A <b>specification language</b> is a <a href="/wiki/Formal_language" title="Formal language">formal language</a> in <a href="/wiki/Computer_science" title="Computer science">computer science</a> used during <a href="/wiki/Systems_analysis" title="Systems analysis">systems analysis</a>, <a href="/wiki/Requirements_analysis" title="Requirements analysis">requirements analysis</a> and <a href="/wiki/Systems_design" title="Systems design">systems design</a> to describe a system at a much higher level than a <a href="/wiki/Programming_language" title="Programming language">programming language</a>, which is used to produce the executable code for a system.</p>
<p>Specification languages are generally not directly executed. They are meant to describe the <i>what</i>, not the <i>how</i>. Indeed, it is considered as an error if a requirement specification is cluttered with unnecessary implementation detail.</p>
<p>A common fundamental assumption of many specification approaches is that programs are modelled as <a href="/wiki/Algebra" title="Algebra">algebraic</a> or <a href="/wiki/Model_theory" title="Model theory">model-theoretic</a> structures that include a collection of <a href="/wiki/Set_(mathematics)" title="Set (mathematics)">sets</a> of data values together with <a href="/wiki/Function_(mathematics)" title="Function (mathematics)">functions</a> over those sets. This level of abstraction coincides with the view that the correctness of the input/output behaviour of a program takes precedence over all its other properties.</p>
<p>In the <i>property-oriented</i> approach to specification (taken e.g. by <a href="/wiki/Common_Algebraic_Specification_Language" title="Common Algebraic Specification Language">CASL</a>), specifications of programs consist mainly of logical <a href="/wiki/Axiom" title="Axiom">axioms</a>, usually in a <a class="mw-redirect" href="/wiki/Logical_system" title="Logical system">logical system</a> in which equality has a prominent role, describing the properties that the functions are required to satisfy - often just by their interrelationship. This is in contrast to so-called <a href="/wiki/Model-based_specification" title="Model-based specification">model-oriented specification</a> in frameworks like <a href="/wiki/Vienna_Development_Method" title="Vienna Development Method">VDM</a> and <a href="/wiki/Z_notation" title="Z notation">Z</a>, which consist of a simple realization of the required behaviour.</p>
<p>Specifications must be subject to a process of <i>refinement</i> (the filling-in of implementation detail) before they can actually be implemented. The result of such a refinement process is an executable algorithm, which is either formulated in a programming language, or in an executable subset of the specification language at hand. For example, <a class="mw-redirect" href="/wiki/Hartmann_pipeline" title="Hartmann pipeline">Hartmann pipelines</a>, when properly applied, may be considered a <a href="/wiki/Dataflow" title="Dataflow">dataflow</a> specification which <i>is</i> directly executable. Another example is the <a href="/wiki/Actor_model" title="Actor model">Actor model</a> which has no specific application content and must be <i>specialized</i> to be executable.</p>
<p>An important use of specification languages is enabling the creation of <a href="/wiki/Mathematical_proof" title="Mathematical proof">proofs</a> of <a class="mw-redirect" href="/wiki/Program_correctness" title="Program correctness">program correctness</a> (<i>see <a class="mw-redirect" href="/wiki/Automated_theorem_prover" title="Automated theorem prover">theorem prover</a></i>).</p>
