<h1 class="firstHeading" id="firstHeading" lang="en">Heuristic evaluation</h1>
<div class="mw-body-content" id="bodyContent">

<div id="contentSub"></div>

<p>A <b>heuristic evaluation</b> is a <a href="/wiki/Usability_inspection" title="Usability inspection">usability inspection</a> method for computer software that helps to identify <a href="/wiki/Usability" title="Usability">usability</a> problems in the <a class="mw-redirect" href="/wiki/User_Interface_Design" title="User Interface Design">user interface (UI) design</a>. It specifically involves evaluators examining the interface and judging its compliance with recognized usability principles (the "heuristics"). These evaluation methods are now widely taught and practiced in the <a href="/wiki/New_media" title="New media">new media</a> sector, where UIs are often designed in a short space of time on a budget that may restrict the amount of money available to provide for other types of interface testing.</p>

<h2><span class="mw-headline" id="Introduction">Introduction</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Heuristic_evaluation&amp;action=edit&amp;section=1" title="Edit section: Introduction">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The main goal of heuristic evaluations is to identify any problems associated with the design of user interfaces. Usability consultant <a href="/wiki/Jakob_Nielsen_(usability_consultant)" title="Jakob Nielsen (usability consultant)">Jakob Nielsen</a> developed this method on the basis of several years of experience in teaching and consulting about <a href="/wiki/Usability_engineering" title="Usability engineering">usability engineering</a>. <a href="/wiki/Heuristic" title="Heuristic">Heuristic</a> evaluations are one of the most informal methods<sup class="reference" id="cite_ref-multiple_1-0"><a href="#cite_note-multiple-1">[1]</a></sup> of usability inspection in the field of <a class="mw-redirect" href="/wiki/Human-computer_interaction" title="Human-computer interaction">human-computer interaction</a>. There are many sets of usability design heuristics; they are not mutually exclusive and cover many of the same aspects of user interface design. Quite often, usability problems that are discovered are categorized—often on a numeric scale—according to their estimated impact on user performance or acceptance. Often the heuristic evaluation is conducted in the context of <a class="mw-redirect" href="/wiki/Use_cases" title="Use cases">use cases</a> (typical user tasks), to provide <a href="/wiki/Feedback" title="Feedback">feedback</a> to the developers on the extent to which the interface is likely to be compatible with the intended users’ needs and preferences.</p>
<p>The simplicity of heuristic evaluation is beneficial at the early stages of design. This usability inspection method does not require user testing which can be burdensome due to the need for users, a place to test them and a payment for their time. Heuristic evaluation requires only one expert, reducing the complexity and expended time for evaluation. Most heuristic evaluations can be accomplished in a matter of days. The time required varies with the size of the artifact, its complexity, the purpose of the review, the nature of the usability issues that arise in the review, and the competence of the reviewers. Using heuristic evaluation prior to user testing will reduce the number and severity of design errors discovered by users. Although heuristic evaluation can uncover many major usability issues in a short period of time, a criticism that is often leveled is that results are highly influenced by the knowledge of the expert reviewer(s). This “one-sided” review repeatedly has different results than <a href="/wiki/Software_performance_testing" title="Software performance testing">software performance testing</a>, each type of testing uncovering a different set of problems.</p>
<h2><span class="mw-headline" id="Nielsen">Nielsen</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Heuristic_evaluation&amp;action=edit&amp;section=2" title="Edit section: Nielsen">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p><a href="/wiki/Jakob_Nielsen_(usability_consultant)" title="Jakob Nielsen (usability consultant)">Jakob Nielsen</a>'s heuristics are probably the most-used usability heuristics for user interface design. Nielsen developed the heuristics based on work together with <a class="new" href="/w/index.php?title=Rolf_Molich&amp;action=edit&amp;redlink=1" title="Rolf Molich (page does not exist)">Rolf Molich</a> in 1990.<sup class="reference" id="cite_ref-multiple_1-1"><a href="#cite_note-multiple-1">[1]</a></sup><sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> The final set of heuristics that are still used today were released by Nielsen in 1994.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup> The heuristics as published in Nielsen's book <i>Usability Engineering</i> are as follows:<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></p>
<ul>
<li class="mw-empty-elt"></li>
</ul>
<ol>
<li><b>Visibility of system status</b>:<br/>
The system should always keep users informed about what is going on, through appropriate feedback within reasonable time.</li>
<li><b>Match between system and the real world</b>:<br/>
The system should speak the user's language, with words, phrases and concepts familiar to the user, rather than system-oriented terms. Follow real-world conventions, making information appear in a natural and logical order.</li>
<li><b>User control and freedom</b>:<br/>
Users often choose system functions by mistake and will need a clearly marked "emergency exit" to leave the unwanted state without having to go through an extended dialogue. Support undo and redo.</li>
<li><b>Consistency and standards</b>:<br/>
Users should not have to wonder whether different words, situations, or actions mean the same thing. Follow platform conventions.</li>
<li><b>Error prevention</b>:<br/>
Even better than good error messages is a careful design which prevents a problem from occurring in the first place. Either eliminate error-prone conditions or check for them and present users with a confirmation option before they commit to the action.</li>
<li><b>Recognition rather than recall</b>:<br/>
Minimize the user's memory load by making objects, actions, and options visible. The user should not have to remember information from one part of the dialogue to another. Instructions for use of the system should be visible or easily retrievable whenever appropriate.</li>
<li><b>Flexibility and efficiency of use</b>:<br/>
Accelerators—unseen by the novice user—may often speed up the interaction for the expert user such that the system can cater to both inexperienced and experienced users. Allow users to tailor frequent actions.</li>
<li><b>Aesthetic and <a href="/wiki/Progressive_disclosure" title="Progressive disclosure">minimalist design</a></b>:<br/>
Dialogues should not contain information which is irrelevant or rarely needed. Every extra unit of information in a dialogue competes with the relevant units of information and diminishes their relative visibility.</li>
<li><b>Help users recognize, diagnose, and recover from errors</b>:<br/>
Error messages should be expressed in plain language (no codes), precisely indicate the problem, and constructively suggest a solution.</li>
<li><b>Help and documentation</b>:<br/>
Even though it is better if the system can be used without documentation, it may be necessary to provide help and documentation. Any such information should be easy to search, focused on the user's task, list concrete steps to be carried out, and not be too large.</li>
</ol>
<h2><span class="mw-headline" id="Gerhardt-Powals.E2.80.99_cognitive_engineering_principles">Gerhardt-Powals’ cognitive engineering principles</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Heuristic_evaluation&amp;action=edit&amp;section=3" title="Edit section: Gerhardt-Powals’ cognitive engineering principles">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Although Nielsen is considered the expert and field leader in heuristics, <a class="external text" href="http://loki.stockton.edu/~gerhardj/" rel="nofollow">Jill Gerhardt-Powals</a> also developed a set of <a class="mw-redirect" href="/wiki/Cognitive" title="Cognitive">cognitive</a> principles for enhancing computer performance.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> These heuristics, or principles, are similar to Nielsen’s heuristics but take a more holistic approach to evaluation. Gerhardt Powals’ principles<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup> are listed below.</p>
<h3><span class="mw-headline" id="Automate_unwanted_workload">Automate unwanted workload</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Heuristic_evaluation&amp;action=edit&amp;section=4" title="Edit section: Automate unwanted workload">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<ul>
<li>Free cognitive resources for high-level tasks.</li>
<li>Eliminate mental calculations, estimations, comparisons, and unnecessary thinking.</li>
</ul>
<h3><span class="mw-headline" id="Reduce_uncertainty">Reduce uncertainty</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Heuristic_evaluation&amp;action=edit&amp;section=5" title="Edit section: Reduce uncertainty">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<ul>
<li>Display data in a manner that is clear and obvious.</li>
</ul>
<h3><span class="mw-headline" id="Fuse_data">Fuse data</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Heuristic_evaluation&amp;action=edit&amp;section=6" title="Edit section: Fuse data">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<ul>
<li>Reduce <a href="/wiki/Cognitive_load" title="Cognitive load">cognitive load</a> by bringing together lower level data into a higher-level summation.</li>
</ul>
<h3><span class="mw-headline" id="Present_new_information_with_meaningful_aids_to_interpretation">Present new information with meaningful aids to interpretation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Heuristic_evaluation&amp;action=edit&amp;section=7" title="Edit section: Present new information with meaningful aids to interpretation">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<ul>
<li>Use a familiar framework, making it easier to absorb.</li>
<li>Use everyday terms, metaphors, etc.</li>
</ul>
<h3><span class="mw-headline" id="Use_names_that_are_conceptually_related_to_function">Use names that are conceptually related to function</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Heuristic_evaluation&amp;action=edit&amp;section=8" title="Edit section: Use names that are conceptually related to function">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<ul>
<li>Context-dependent.</li>
<li>Attempt to improve recall and recognition.</li>
<li>Group data in consistently meaningful ways to decrease search time.</li>
</ul>
<h3><span class="mw-headline" id="Limit_data-driven_tasks">Limit data-driven tasks</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Heuristic_evaluation&amp;action=edit&amp;section=9" title="Edit section: Limit data-driven tasks">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<ul>
<li>Reduce the time spent assimilating raw data.</li>
<li>Make appropriate use of color and graphics.</li>
</ul>
<h3><span class="mw-headline" id="Include_in_the_displays_only_that_information_needed_by_the_user_at_a_given_time">Include in the displays only that information needed by the user at a given time</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Heuristic_evaluation&amp;action=edit&amp;section=10" title="Edit section: Include in the displays only that information needed by the user at a given time">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<h3><span class="mw-headline" id="Provide_multiple_coding_of_data_when_appropriate">Provide multiple coding of data when appropriate</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Heuristic_evaluation&amp;action=edit&amp;section=11" title="Edit section: Provide multiple coding of data when appropriate">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<h3><span class="mw-headline" id="Practice_judicious_redundancy">Practice judicious redundancy</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Heuristic_evaluation&amp;action=edit&amp;section=12" title="Edit section: Practice judicious redundancy">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<h2><span class="mw-headline" id="Weinschenk_and_Barker_classification">Weinschenk and Barker classification</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Heuristic_evaluation&amp;action=edit&amp;section=13" title="Edit section: Weinschenk and Barker classification">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Susan Weinschenk and Dean Barker created a categorization of heuristics and guidelines by several major providers into the following twenty types:<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup></p>
<p>1. User control: heuristics that check whether the user has enough control of the interface.</p>
<p>2. Human limitations: the design takes into account human limitations, cognitive and sensorial, to avoid overloading them.</p>
<p>3. Modal integrity: the interface uses the most suitable <a href="/wiki/Modality_(human%E2%80%93computer_interaction)" title="Modality (human–computer interaction)">modality</a> for each task: auditory, visual, or motor/kinesthetic.</p>
<p>4. Accommodation: the design is adequate to fulfill the needs and behaviour of each targeted user group.</p>
<p>5. Linguistic clarity: the language used to communicate is efficient, clear and adequate to the audience.</p>
<p>6. Aesthetic integrity: the design is visually attractive and tailored to appeal to the target population.</p>
<p>7. Simplicity: the design does not use unnecessary complexity.</p>
<p>8. Predictability: users will be able to form a <a href="/wiki/Mental_model" title="Mental model">mental model</a> of how the system will behave in response to actions.</p>
<p>9. Interpretation: there are codified rules that try to guess the user <a href="/wiki/Intention" title="Intention">intentions</a> and anticipate the actions needed.</p>
<p>10. Accuracy: There are no errors, i.e. the result of user actions correspond to their goals.</p>
<p>11. Technical clarity: the concepts represented in the interface have the highest possible correspondence to the <a href="/wiki/Problem_domain" title="Problem domain">problem domain</a> they are modeling.</p>
<p>12. Flexibility: the design can be adjusted to the needs and behaviour of each particular user.</p>
<p>13. Fulfillment: the <a href="/wiki/User_experience" title="User experience">user experience</a> is adequate and the user feels good about the experience.</p>
<p>14. Cultural propriety: the user's cultural and social expectations are met.</p>
<p>15. Suitable tempo: the pace at which users works with the system is adequate.</p>
<p>16. Consistency: different parts of the system have the same style, so that there are no different ways to represent the same information or behavior.</p>
<p>17. User support: the design will support learning and provide the required assistance to usage.</p>
<p>18. Precision: the steps and results of a task will be what the user wants.</p>
<p>19. Forgiveness: the user will be able to recover to an adequate state after an error.</p>
<p>20. Responsiveness: the interface provides the user enough feedback information about the system status and their task completion.</p>
