<p><b>Defensive programming</b> is a form of <a href="/wiki/Defensive_design" title="Defensive design">defensive design</a> intended to ensure the continuing function of a piece of <a href="/wiki/Software" title="Software">software</a> under unforeseen circumstances. Defensive programming practices are often used where high availability, safety or security is needed.</p>
<p>Defensive programming is an approach to improve software and source code, in terms of:</p>
<ul>
<li>General quality – reducing the number of <a href="/wiki/Software_bug" title="Software bug">software bugs</a> and problems.</li>
<li>Making the source code comprehensible – the source code should be readable and understandable so it is approved in a <a href="/wiki/Code_audit" title="Code audit">code audit</a>.</li>
<li>Making the software behave in a predictable manner despite unexpected inputs or user actions.</li>
</ul>
<p>Overly defensive programming however introduces unnecessary code for errors impossible to even happen, thus wasting runtime and maintenance costs. There is also the risk that the code traps or prevents too many exceptions, potentially resulting in unnoticed, incorrect results.</p>

<h2><span class="mw-headline" id="Secure_programming">Secure programming</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Defensive_programming&amp;action=edit&amp;section=1" title="Edit section: Secure programming">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="hatnote" role="note">Main article: <a href="/wiki/Secure_coding" title="Secure coding">Secure coding</a></div>
<p>Secure programming is the subset of defensive programming concerned with <a href="/wiki/Computer_security" title="Computer security">computer security</a>. That is to say, security is the concern, not necessarily safety or availability (the software may be allowed to fail in certain ways). As with all kinds of defensive programming, avoiding bugs is a primary objective, however the motivation is not as much to reduce the likelihood of failure in normal operation (as if safety was the concern), but to reduce the attack surface – the programmer must assume that the software might be misused actively to reveal bugs, and that bugs could be exploited maliciously.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="kt">int</span> <span class="nf">risky_programming</span><span class="p">(</span><span class="kt">char</span> <span class="o">*</span><span class="n">input</span><span class="p">){</span>
  <span class="kt">char</span> <span class="n">str</span><span class="p">[</span><span class="mi">1000</span><span class="o">+</span><span class="mi">1</span><span class="p">];</span>     <span class="c1">// one more for the null character</span>
  <span class="c1">// ...</span>
  <span class="n">strcpy</span><span class="p">(</span><span class="n">str</span><span class="p">,</span> <span class="n">input</span><span class="p">);</span>   <span class="c1">// copy input</span>
  <span class="c1">// ...</span>
<span class="p">}</span>
</pre></div>
<p>The function will crash when the input is over 1000 characters. Some novice programmers may not feel that this is a problem, supposing that no user will enter such a long input. This particular bug demonstrates a vulnerability which enables <a href="/wiki/Buffer_overflow" title="Buffer overflow">buffer overflow</a> <a href="/wiki/Exploit_(computer_security)" title="Exploit (computer security)">exploits</a>. Here is a solution to this example:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="kt">int</span> <span class="nf">secure_programming</span><span class="p">(</span><span class="kt">char</span> <span class="o">*</span><span class="n">input</span><span class="p">){</span>
  <span class="kt">char</span> <span class="n">str</span><span class="p">[</span><span class="mi">1000</span><span class="p">];</span>
  <span class="c1">// ...</span>
  <span class="n">strncpy</span><span class="p">(</span><span class="n">str</span><span class="p">,</span> <span class="n">input</span><span class="p">,</span> <span class="k">sizeof</span><span class="p">(</span><span class="n">str</span><span class="p">));</span> <span class="c1">// copy input without exceeding the length of the destination</span>
  <span class="n">str</span><span class="p">[</span><span class="k">sizeof</span><span class="p">(</span><span class="n">str</span><span class="p">)</span> <span class="o">-</span> <span class="mi">1</span><span class="p">]</span> <span class="o">=</span> <span class="sc">'\0'</span><span class="p">;</span> <span class="c1">// if strlen(input) &gt;= sizeof(str) then strncpy won't NUL terminate</span>
  <span class="c1">// ...</span>
<span class="p">}</span>
</pre></div>
<h2><span class="mw-headline" id="Offensive_programming">Offensive programming</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Defensive_programming&amp;action=edit&amp;section=2" title="Edit section: Offensive programming">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="hatnote" role="note">Main article: <a href="/wiki/Offensive_programming" title="Offensive programming">Offensive programming</a></div>
<p>Offensive programming can be considered a category of defensive programming, with the added emphasis that certain errors should <i>not</i> be <a class="mw-redirect" href="/wiki/Graceful_degradation" title="Graceful degradation">handled defensively</a>. In this practice, only errors from outside the program's control are to be handled (such as user input); the software itself, as well as data from within the program's line of defense, are to be trusted in this methodology.</p>

<h2><span class="mw-headline" id="Techniques">Techniques</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Defensive_programming&amp;action=edit&amp;section=5" title="Edit section: Techniques">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Here are some defensive programming techniques:</p>
<h3><span class="mw-headline" id="Intelligent_source_code_reuse">Intelligent source code reuse</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Defensive_programming&amp;action=edit&amp;section=6" title="Edit section: Intelligent source code reuse">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>If existing code is tested and known to work, reusing it may reduce the chance of bugs being introduced.</p>
<p>However, reusing code is not <i>always</i> a good practice, because it also amplifies the damages of a potential attack on the initial code. <span class="clarify-content" style="padding-left:0.1em; padding-right:0.1em; color:#595959; border:1px solid #DDD;">Reuse in this case may cause serious <a href="/wiki/Business_process" title="Business process">business process</a> bugs.</span><sup class="noprint Inline-Template Template-Clarify" style="margin-left:0.1em; white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Please_clarify" title="Wikipedia:Please clarify"><span title="Business logic bugs and security bugs are different things. The original text here explicitly talked about business logic, and it's still not clear what the problem is with reuse in business logic. (December 2016)">clarify</span></a></i>]</sup></p>
<h4><span class="mw-headline" id="Legacy_problems">Legacy problems</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Defensive_programming&amp;action=edit&amp;section=7" title="Edit section: Legacy problems">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Before reusing old source code, libraries, APIs, configurations and so forth, it must be considered if the old work is valid for reuse, or if it is likely to be prone to <a href="/wiki/Legacy_system" title="Legacy system">legacy</a> problems.</p>
<p>Legacy problems are problems inherent when old designs are expected to work with today's requirements, especially when the old designs were not developed or tested with those requirements in mind.</p>
<p>Many software products have experienced problems with old legacy source code, for example:</p>
<ul>
<li><a href="/wiki/Legacy_code" title="Legacy code">Legacy code</a> may not have been designed under a Defensive programming initiative, and might therefore be of much lower quality than newly designed source code.</li>
<li>Legacy code may have been written and tested under conditions which no longer apply. The old quality assurance tests may have no validity any more.
<ul>
<li><b>Example 1</b>: legacy code may have been designed for ASCII input but now the input is UTF-8.</li>
<li><b>Example 2</b>: legacy code may have been compiled and tested on 32-bit architectures, but when compiled on 64-bit architectures new arithmetic problems may occur (e.g. invalid signedness tests, invalid type casts, etc.).</li>
<li><b>Example 3</b>: legacy code may have been targeted for offline machines, but becomes vulnerable once network connectivity is added.</li>
</ul>
</li>
<li>Legacy code is not written with new problems in mind. For example, source code written about 1990 is likely to be prone to many <a href="/wiki/Code_injection" title="Code injection">code injection</a> vulnerabilities, because most such problems were not widely understood at that time.</li>
</ul>
<p>Notable examples of the legacy problem:</p>
<ul>
<li><a href="/wiki/BIND" title="BIND">BIND 9</a>, presented by Paul Vixie and David Conrad as "BINDv9 is a <a href="/wiki/Rewrite_(programming)" title="Rewrite (programming)">complete rewrite</a>", "Security was a key consideration in design" <a class="external text" href="http://impressive.net/archives/fogo/20001005080818.O15286@impressive.net" rel="nofollow">*</a>, naming security, robustness, scalability and new protocols as key concerns for rewriting old legacy code.</li>
<li><a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a> suffered from "the" <a href="/wiki/Windows_Metafile_vulnerability" title="Windows Metafile vulnerability">Windows Metafile vulnerability</a> and other exploits related to the WMF format. Microsoft Security Response Center describes the WMF-features as <i>"Around 1990, WMF support was added... This was a different time in the security landscape... were all completely trusted"</i> <a class="external text" href="http://blogs.technet.com/msrc/archive/2006/01/13/417431.aspx" rel="nofollow">*</a>, not being developed under the security initiatives at Microsoft.</li>
<li><a href="/wiki/Oracle_Corporation" title="Oracle Corporation">Oracle</a> is combating legacy problems, such as old source code written without addressing concerns of <a href="/wiki/SQL_injection" title="SQL injection">SQL injection</a> and <a href="/wiki/Privilege_escalation" title="Privilege escalation">privilege escalation</a>, resulting in many security vulnerabilities which has taken time to fix and also generated incomplete fixes. This has given rise to heavy criticism from security experts such as <a href="/wiki/David_Litchfield" title="David Litchfield">David Litchfield</a>, <a class="new" href="/w/index.php?title=Alexander_Kornbrust&amp;action=edit&amp;redlink=1" title="Alexander Kornbrust (page does not exist)">Alexander Kornbrust</a>, <a class="new" href="/w/index.php?title=Cesar_Cerrudo&amp;action=edit&amp;redlink=1" title="Cesar Cerrudo (page does not exist)">Cesar Cerrudo</a> (<a class="external text" href="http://seclists.org/lists/bugtraq/2006/May/0039.html" rel="nofollow">1</a>, <a class="external text" href="http://seclists.org/lists/bugtraq/2006/May/0045.html" rel="nofollow">2</a>, <a class="external text" href="http://seclists.org/lists/bugtraq/2006/May/0083.html" rel="nofollow">3</a>). An additional criticism is that default installations (largely a legacy from old versions) are not aligned with their own security recommendations, such as <a class="external text" href="http://www.oracle.com/technology/deploy/security/database-security/pdf/twp_security_checklist_database.pdf" rel="nofollow">Oracle Database Security Checklist</a>, which is hard to amend as many applications require the less secure legacy settings to function correctly.</li>
</ul>
<h3><span class="mw-headline" id="Secure_input_and_output_handling">Secure input and output handling</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Defensive_programming&amp;action=edit&amp;section=8" title="Edit section: Secure input and output handling">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Secure_input_and_output_handling" title="Secure input and output handling">Secure input and output handling</a></div>
<h3><span class="mw-headline" id="Canonicalization">Canonicalization</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Defensive_programming&amp;action=edit&amp;section=9" title="Edit section: Canonicalization">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Crackers are likely to invent new kinds of representations of incorrect data.</p>
<p>For example, if you checked if a requested file is not "/etc/<a class="mw-redirect" href="/wiki/Passwd_(file)" title="Passwd (file)">passwd</a>", a cracker might pass another variant of this file name, like "/etc/./passwd".</p>
<p>To avoid bugs due to non-<a href="/wiki/Canonical_form" title="Canonical form">canonical</a> input, employ <a href="/wiki/Canonicalization" title="Canonicalization">canonicalization</a> libraries.</p>
<h3><span class="mw-headline" id="Low_tolerance_against_.22potential.22_bugs">Low tolerance against "potential" bugs</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Defensive_programming&amp;action=edit&amp;section=10" title='Edit section: Low tolerance against "potential" bugs'>edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Assume that code constructs that appear to be problem prone (similar to known vulnerabilities, etc.) are bugs and potential security flaws. The basic rule of thumb is: "I'm not aware of all types of <a class="mw-redirect" href="/wiki/Security_exploit" title="Security exploit">security exploits</a>. I must protect against those I <i>do</i> know of and then I must be proactive!".</p>
<h3><span class="mw-headline" id="Other_techniques">Other techniques</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Defensive_programming&amp;action=edit&amp;section=11" title="Edit section: Other techniques">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<ul>
<li>One of the most common problems is unchecked use of constant-size structures and functions for dynamic-size data (the <a href="/wiki/Buffer_overflow" title="Buffer overflow">buffer overflow</a> problem). This is especially common for <a class="mw-redirect" href="/wiki/String_(computer_programming)" title="String (computer programming)">string</a> data in <a href="/wiki/C_(programming_language)" title="C (programming language)">C</a>. C library functions like <tt>gets</tt> should never be used since the maximum size of the input buffer is not passed as an argument. C library functions like <tt>scanf</tt> can be used safely, but require the programmer to take care with the selection of safe format strings, by sanitising it before using it.</li>
<li>Encrypt/authenticate all important data transmitted over networks. Do not attempt to implement your own encryption scheme, but use a proven one instead.</li>
<li>All data is important until proven otherwise.</li>
<li>All data is tainted until proven otherwise.</li>
<li>All code is insecure until proven otherwise.
<ul>
<li>You cannot prove the security of any code in <a class="mw-redirect" href="/wiki/Userland_(computing)" title="Userland (computing)">userland</a>, or, more canonically: <i>"never trust the client"</i>.</li>
</ul>
</li>
<li>If data are to be checked for correctness, verify that they are correct, not that they are incorrect.</li>
<li><a href="/wiki/Design_by_contract" title="Design by contract">Design by contract</a>
<ul>
<li>Design by contract uses <a href="/wiki/Precondition" title="Precondition">preconditions</a>, <a href="/wiki/Postcondition" title="Postcondition">postconditions</a> and <a href="/wiki/Invariant_(computer_science)" title="Invariant (computer science)">invariants</a> to ensure that provided data (and the state of the program as a whole) is sanitized. This allows code to document its assumptions and make them safely. This may involve checking arguments to a function or method for validity before executing the body of the function. After the body of a function, doing a check of object state (in <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">Object-oriented programming</a> languages) or other held data and the return value before exits (break/return/throw/error code) is also wise.</li>
</ul>
</li>
<li><a class="mw-redirect" href="/wiki/Assertion_(computing)" title="Assertion (computing)">Assertions</a>
<ul>
<li>Within functions, you may want to check that you are not referencing something that is not valid (i.e., null) and that array lengths are valid before referencing elements, especially on all temporary/local instantiations. A good heuristic is to not trust the libraries you did not write either. So any time you call them, check what you get back from them. It often helps to create a small library of "asserting" and "checking" functions to do this along with a logger so you can trace your path and reduce the need for extensive <a href="/wiki/Debugging" title="Debugging">debugging</a> cycles in the first place. With the advent of logging libraries and <a class="mw-redirect" href="/wiki/Aspect_Oriented_Programming" title="Aspect Oriented Programming">Aspect Oriented Programming</a>, many of the tedious aspects of defensive programming are mitigated.</li>
</ul>
</li>
<li>Prefer <a href="/wiki/Exception_handling" title="Exception handling">exceptions</a> to return codes
<ul>
<li>Generally speaking, it is preferable to throw intelligible exception messages that enforce part of your <a href="/wiki/Application_programming_interface" title="Application programming interface">API</a> contract and guide the client <a href="/wiki/Programmer" title="Programmer">programmer</a> instead of returning values that a client programmer is likely to be unprepared for and hence minimize their complaints and increase robustness and security of your software.<sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Accuracy_dispute#Disputed_statement" title="Wikipedia:Accuracy dispute"><span title="The material near this tag is possibly inaccurate or nonfactual. (June 2015)">dubious</span></a> <span class="metadata">– <a href="/wiki/Talk:Defensive_programming#Dubious" title="Talk:Defensive programming">discuss</a></span></i>]</sup></li>
</ul>
</li>
</ul>
