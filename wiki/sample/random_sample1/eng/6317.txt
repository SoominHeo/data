<p><b>Bugzilla</b> is a <a href="/wiki/World_Wide_Web" title="World Wide Web">web</a>-based general-purpose <a class="mw-redirect" href="/wiki/Bugtracker" title="Bugtracker">bugtracker</a> and <a href="/wiki/Test_automation" title="Test automation">testing tool</a> originally developed and used by the <a href="/wiki/Mozilla" title="Mozilla">Mozilla</a> project, and <a href="/wiki/Software_license" title="Software license">licensed</a> under the <a href="/wiki/Mozilla_Public_License" title="Mozilla Public License">Mozilla Public License</a>.</p>
<p>Released as <a href="/wiki/Open-source_software" title="Open-source software">open-source software</a> by <a class="mw-redirect" href="/wiki/Netscape_Communications" title="Netscape Communications">Netscape Communications</a> in 1998, it has been adopted by a variety of organizations for use as a <a href="/wiki/Bug_tracking_system" title="Bug tracking system">bug tracking system</a> for both <a href="/wiki/Free_and_open-source_software" title="Free and open-source software">free and open-source software</a> and <a href="/wiki/Proprietary_software" title="Proprietary software">proprietary</a> projects and products. Bugzilla is used, among others, by the <a href="/wiki/Mozilla_Foundation" title="Mozilla Foundation">Mozilla Foundation</a>, <a href="/wiki/WebKit" title="WebKit">WebKit</a>, <a href="/wiki/Linux_kernel" title="Linux kernel">Linux kernel</a>, <a href="/wiki/FreeBSD" title="FreeBSD">FreeBSD</a>,<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup> <a href="/wiki/GNOME" title="GNOME">GNOME</a>, <a href="/wiki/KDE" title="KDE">KDE</a>, <a href="/wiki/Apache_Software_Foundation" title="Apache Software Foundation">Apache</a>, <a href="/wiki/Red_Hat" title="Red Hat">Red Hat</a>, <a href="/wiki/Eclipse_(software)" title="Eclipse (software)">Eclipse</a> and <a href="/wiki/LibreOffice" title="LibreOffice">LibreOffice</a>.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> It is also <a href="/wiki/Self-hosting" title="Self-hosting">self-hosting</a>.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup></p>

<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Bugzilla&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Bugzilla was originally devised by Terry Weissman in 1998 for the nascent <a class="mw-redirect" href="/wiki/Mozilla.org" title="Mozilla.org">Mozilla.org</a> project, as an <a class="mw-redirect" href="/wiki/Open_source" title="Open source">open source</a> application to replace the in-house system then in use at Netscape Communications for tracking defects in the <a href="/wiki/Netscape_Communicator" title="Netscape Communicator">Netscape Communicator</a> suite. Bugzilla was originally written in <a href="/wiki/Tcl" title="Tcl">Tcl</a>, but Weissman decided to port it to <a href="/wiki/Perl" title="Perl">Perl</a> before its release as part of Netscape's early open source code drops, in the hope that more people would be able to contribute to it, given that Perl seemed to be a more popular language at the time.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup></p>
<p>Bugzilla 2.0 was the result of that port to Perl, and the first version was released to the public via anonymous <a href="/wiki/Concurrent_Versions_System" title="Concurrent Versions System">CVS</a>. In April 2000, Weissman handed over control of the Bugzilla project to <a href="/wiki/Tara_Hernandez" title="Tara Hernandez">Tara Hernandez</a>. Under her leadership, some of the regular contributors were coerced into taking more responsibility, and Bugzilla development became more community-driven. In July 2001, facing distraction from her other responsibilities in Netscape, Hernandez handed control to Dave Miller, who was still in charge as of 2013<sup class="plainlinks noprint asof-tag update" style="display:none;"><a class="external text" href="//en.wikipedia.org/w/index.php?title=Bugzilla&amp;action=edit">[update]</a></sup>.<sup class="reference" id="cite_ref-bugzilla-dev-profiles_8-0"><a href="#cite_note-bugzilla-dev-profiles-8">[8]</a></sup></p>
<p>Bugzilla 3.0 was released on May 10, 2007 and brought refreshed <a href="/wiki/User_interface" title="User interface">UI</a>, <a href="/wiki/XML-RPC" title="XML-RPC">XML-RPC</a> interface, custom fields and resolutions, <a href="/wiki/Mod_perl" title="Mod perl">mod_perl</a> support, shared saved searches, and improved <a href="/wiki/UTF-8" title="UTF-8">UTF-8</a> support, along with other changes.</p>
<p>Bugzilla 4.0 was released on February 15, 2011.</p>

<h2><span class="mw-headline" id="Requirements">Requirements</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Bugzilla&amp;action=edit&amp;section=3" title="Edit section: Requirements">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Bugzilla's <a href="/wiki/System_requirements" title="System requirements">system requirements</a> include:</p>
<ul>
<li>A compatible <a class="mw-redirect" href="/wiki/Database_management_system" title="Database management system">database management system</a></li>
<li>A suitable release of <a href="/wiki/Perl" title="Perl">Perl</a> 5</li>
<li>An assortment of Perl modules</li>
<li>A compatible <a href="/wiki/Web_server" title="Web server">web server</a></li>
<li>A suitable <a class="mw-redirect" href="/wiki/Mail_transfer_agent" title="Mail transfer agent">mail transfer agent</a>, or any <a class="mw-redirect" href="/wiki/SMTP" title="SMTP">SMTP</a> server</li>
</ul>
<p>Currently supported database systems are <a href="/wiki/MySQL" title="MySQL">MySQL</a>, <a href="/wiki/PostgreSQL" title="PostgreSQL">PostgreSQL</a>, <a href="/wiki/Oracle_Database" title="Oracle Database">Oracle</a>, and <a href="/wiki/SQLite" title="SQLite">SQLite</a>. Bugzilla is usually installed on <a href="/wiki/Linux" title="Linux">Linux</a> using the <a href="/wiki/Apache_HTTP_Server" title="Apache HTTP Server">Apache HTTP Server</a>, but any web server that supports <a href="/wiki/Common_Gateway_Interface" title="Common Gateway Interface">CGI</a> such as <a href="/wiki/Lighttpd" title="Lighttpd">Lighttpd</a>, <a href="/wiki/Hiawatha_(web_server)" title="Hiawatha (web server)">Hiawatha</a>, <a href="/wiki/Cherokee_(web_server)" title="Cherokee (web server)">Cherokee</a> can be used. Bugzilla's installation process is <a class="mw-redirect" href="/wiki/Command_line" title="Command line">command line</a> driven and runs through a series of stages where system requirements and software capabilities are checked.</p>
<h2><span class="mw-headline" id="Design">Design</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Bugzilla&amp;action=edit&amp;section=4" title="Edit section: Design">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="thumb tright">

<p>While the potential exists in the code to turn Bugzilla into a <a href="/wiki/Issue_tracking_system" title="Issue tracking system">technical support ticket system</a>, task management tool, or project management tool, Bugzilla's developers have chosen to focus on the task of designing a system to track software defects. Mandated design requirements include:<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup></p>
<ul>
<li>The ability to run on freely available, open-source tools. While Bugzilla development includes work to support commercial databases, tools, and operating systems.</li>
<li>The maintenance of speed and efficiency at all costs. One of Bugzilla's major attractions to developers is its lightweight implementation and speed, so calls into the database are minimized whenever possible, data fetching is kept as light as possible, and generation of heavy <a href="/wiki/HTML" title="HTML">HTML</a> is avoided.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (September 2011)">citation needed</span></a></i>]</sup></li>
<li>Tickets. For instance, Mozilla.org uses it to track feature requests as well. In this case items (called bugs) can be submitted by anybody, and will be assigned to a particular developer. Various status updates for each bug are allowed, together with user notes and bug examples.</li>
</ul>
<p>In practice, most Bugzilla projects allowing the public to file bugs—such as the Bugzilla bug database for Bugzilla itself—assign all bugs to a gatekeeper, whose job it is to assign responsibility and priority level.</p>
<h2><span class="mw-headline" id="Zarro_Boogs">Zarro Boogs</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Bugzilla&amp;action=edit&amp;section=5" title="Edit section: Zarro Boogs">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>By design, Bugzilla is programmed to return the string "zarro boogs found" instead of "0 bugs found" when a search for bugs returns no results.<sup class="reference" id="cite_ref-zarro_11-0"><a href="#cite_note-zarro-11">[11]</a></sup> "Zarro Boogs" is a meta-statement about the nature of software <a href="/wiki/Software_bug" title="Software bug">debugging</a>. <a href="/wiki/Bug_tracking_system" title="Bug tracking system">Bug tracking systems</a> like Bugzilla readily describe how many <i>known</i> bugs are outstanding. The response "zarro boogs",<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup> is intended as a buggy statement itself (a misspelling of "zero bugs"), implying that even when no bugs have been identified, software is still likely to contain bugs that have not yet been identified.<sup class="reference" id="cite_ref-zarro_11-1"><a href="#cite_note-zarro-11">[11]</a></sup></p>
<p>The following comment is provided in the Bugzilla source code to developers who may be confused by this behaviour:</p>
<dl>
<dd><i><b>Zarro Boogs Found</b></i></dd>
<dd>This is just a goofy way of saying that there were no bugs found matching your query. When asked to explain this message, Terry Weissman (an early Bugzilla developer) had the following to say:</dd>
</dl>
<dl>
<dd>I've been asked to explain this ... way back when, when Netscape released version 4.0 of its browser, we had a release party. Naturally, there had been a big push to try and fix every known bug before the release. Naturally, that hadn't actually happened. (This is not unique to Netscape or to 4.0; the same thing has happened with every software project I've ever seen.) Anyway, at the release party, T-shirts were handed out that said something like "Netscape 4.0: Zarro Boogs". Just like the software, the T-shirt had no known bugs. Uh-huh. So, when you query for a list of bugs, and it gets no results, you can think of this as a friendly reminder. Of *course* there are bugs matching your query, they just aren't in the bugsystem yet...</dd>
<dd>— Terry Weissman</dd>
<dd><i>From <a class="external text" href="http://www.bugzilla.org/docs/2.16/html/glossary.html" rel="nofollow">The Bugzilla Guide – 2.16.10 Release: Glossary</a></i></dd>
</dl>
