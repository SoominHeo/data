<h1 class="firstHeading" id="firstHeading" lang="en">Universally unique identifier</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
</div>
<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><script>function mfTempOpenSection(id){var block=document.getElementById("mf-section-"+id);block.className+=" open-block";block.previousSibling.className+=" open-block";}</script><p>A <b>universally unique identifier</b> (<b>UUID</b>) is a 128-bit number used to identify information in computer systems. <a href="/wiki/Microsoft" title="Microsoft">Microsoft</a> uses the term <b>globally unique identifier</b> (<b>GUID</b>), either as a synonym for UUID or to refer to a particular UUID variant.</p>
<p>When generated according to the standard methods, UUIDs are for practical purposes unique, without depending for their uniqueness on a central registration authority or coordination between the parties generating them, unlike most other <a href="/wiki/Numbering_scheme" title="Numbering scheme">numbering schemes</a>. While the probability that a UUID will be duplicated is not zero, it is so close to zero as to be negligible.</p>
<p>Thus, anyone can create a UUID and use it to identify something with near certainty that the identifier does not duplicate one that has already been, or will be, created to identify something else. Information labeled with UUIDs by independent parties can therefore be later combined into a single database, or transmitted on the same channel, without needing to resolve conflicts between identifiers.</p>
<p>Adoption of UUIDs and GUIDs is widespread, with many computing platforms providing support for generating them, and for parsing their textual representation.</p>
<p></p>

<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Universally_unique_identifier&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>UUIDs were originally used in the <a href="/wiki/Apollo_Computer" title="Apollo Computer">Apollo</a> <a href="/wiki/Network_Computing_System" title="Network Computing System">Network Computing System</a> (NCS) and later in the <a href="/wiki/Open_Software_Foundation" title="Open Software Foundation">Open Software Foundation</a>'s (OSF) <a href="/wiki/Distributed_Computing_Environment" title="Distributed Computing Environment">Distributed Computing Environment</a> (DCE). The initial design of DCE UUIDs was based on the NCS UUIDs,<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> whose design was in turn inspired by the (<a href="/wiki/64-bit_computing" title="64-bit computing">64-bit</a>) unique identifiers defined and used pervasively in <a href="/wiki/Domain/OS" title="Domain/OS">Domain/OS</a>, an <a href="/wiki/Operating_system" title="Operating system">operating system</a> also designed by Apollo Computer. Later, the <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a> platforms adopted the DCE design as globally unique identifiers (GUIDs). <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc4122" rel="nofollow">RFC 4122</a> registered a <a class="mw-redirect" href="/wiki/URN" title="URN">URN</a> namespace for UUIDs,<sup class="reference" id="cite_ref-RFC_4122_2-0"><a href="#cite_note-RFC_4122-2">[2]</a></sup> and recapitulated the earlier specifications, with the same technical content. By the time RFC 4122 was adopted as a standard, the <a class="mw-redirect" href="/wiki/ITU" title="ITU">ITU</a> had also standardized UUIDs, based on the previous standards and early versions of RFC 4122.</p>
<h2><span class="mw-headline" id="Standards">Standards</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Universally_unique_identifier&amp;action=edit&amp;section=2" title="Edit section: Standards">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>UUIDs are standardized by the <a href="/wiki/Open_Software_Foundation" title="Open Software Foundation">Open Software Foundation</a> (OSF) as part of the <a href="/wiki/Distributed_Computing_Environment" title="Distributed Computing Environment">Distributed Computing Environment</a> (DCE).<sup class="reference" id="cite_ref-cde_spec_3-0"><a href="#cite_note-cde_spec-3">[3]</a></sup><sup class="reference" id="cite_ref-dce_spec_4-0"><a href="#cite_note-dce_spec-4">[4]</a></sup></p>
<p>UUIDs are documented as part of <a href="/wiki/International_Organization_for_Standardization" title="International Organization for Standardization">ISO</a>/<a href="/wiki/International_Electrotechnical_Commission" title="International Electrotechnical Commission">IEC</a> 11578:1996 "<a href="/wiki/Information_technology" title="Information technology">Information technology</a> – <a href="/wiki/Open_Systems_Interconnection" title="Open Systems Interconnection">Open Systems Interconnection</a> – <a class="mw-redirect" href="/wiki/Remote_Procedure_Call" title="Remote Procedure Call">Remote Procedure Call</a> (RPC)" and more recently in ITU-T Rec. X.667 | <a href="/wiki/International_Organization_for_Standardization" title="International Organization for Standardization">ISO</a>/<a href="/wiki/International_Electrotechnical_Commission" title="International Electrotechnical Commission">IEC</a> 9834-8:2005.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup></p>
<p>The <a href="/wiki/Internet_Engineering_Task_Force" title="Internet Engineering Task Force">Internet Engineering Task Force</a> (IETF) published the Standards-Track, RFC 4122,<sup class="reference" id="cite_ref-RFC_4122_2-1"><a href="#cite_note-RFC_4122-2">[2]</a></sup> technically equivalent to ITU-T Rec. X.667 | ISO/IEC 9834-8.</p>
<h2><span class="mw-headline" id="Format">Format</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Universally_unique_identifier&amp;action=edit&amp;section=3" title="Edit section: Format">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In its canonical textual representation, the sixteen <a href="/wiki/Octet_(computing)" title="Octet (computing)">octets</a> of a UUID are represented as 32 lowercase <a href="/wiki/Hexadecimal" title="Hexadecimal">hexadecimal</a> (base 16) digits, displayed in five groups separated by hyphens, in the form <code>8-4-4-4-12</code> for a total of 36 characters (32 alphanumeric characters and four hyphens). For example:</p>
<dl>
<dd><code>123e4567-e89b-12d3-a456-426655440000</code></dd>
<dd><code>xxxxxxxx-xxxx-Mxxx-Nxxx-xxxxxxxxxxxx</code></dd>
</dl>
<p>The one to three most significant bits of digit <code>N</code> indicate the UUID variant and the four bits of digit <code>M</code> indicate the UUID version. In the example, <i>M</i> is <code>1</code> and <i>N</i> is <code>a</code> (<code>10xx</code>), meaning that the UUID is a variant 1, version 1 UUID; that is, a time-based DCE/RFC 4122 UUID.</p>
<p>The canonical 8-4-4-4-12 format string is based on the "record layout" for the 16 bytes of the UUID:<sup class="reference" id="cite_ref-RFC_4122_2-2"><a href="#cite_note-RFC_4122-2">[2]</a></sup></p>
<ul>
<li>A 4-byte (8 hex digits) "time_low" integer giving the low 32 bits of the time</li>
<li>A 2-byte (4 hex digits) "time_mid" integer giving the middle 16 bits of the time</li>
<li>A 2-byte (4 hex digits) "time_hi_and_version", with the 4-bit "version" in the most significant bits, followed by the high 12 bits of the time</li>
<li>Two 1-byte fields (totalling 4 hex digits) called "clock_seq_hi_res" and "clock_seq_lo", with the "variant" multiplexed into the most significant 1 to 3 bits of clock_seq_hi_res</li>
<li>Six bytes (12 hex digits) with the 48-bit "node"</li>
</ul>
<p>These fields mainly reflect those in version 1 and 2 time-based UUIDs, but the same 8-4-4-4-12 representation is used for all UUIDs, even if the semantics of the UUID is different from the time-based versions 1 and 2, or is opaque.</p>
<p>Microsoft GUIDs are sometimes represented with surrounding braces:</p>
<dl>
<dd><code>{123e4567-e89b-12d3-a456-426655440000}</code></dd>
</dl>
<p>This format should not be confused with "registry format", which refers to the format <em>within</em> the curly braces.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup></p>
<p>RFC 4122 defines a <a href="/wiki/Uniform_Resource_Name" title="Uniform Resource Name">Uniform Resource Name</a> (URN) namespace for UUIDs. A UUID presented as a URN appears as follows:<sup class="reference" id="cite_ref-RFC_4122_2-3"><a href="#cite_note-RFC_4122-2">[2]</a></sup></p>
<dl>
<dd><code>urn:uuid:123e4567-e89b-12d3-a456-426655440000</code></dd>
</dl>
<h2><span class="mw-headline" id="Variants">Variants</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Universally_unique_identifier&amp;action=edit&amp;section=4" title="Edit section: Variants">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>One of the variants defined by RFC 4122, variant 0 (indicated by the one-bit pattern <code>0xxx N=0..7</code>), is for backwards compatibility with the now obsolete Apollo <a href="/wiki/Network_Computing_System" title="Network Computing System">Network Computing System</a> UUID format, and the 3-bit variant bit pattern <code>111x N=e..f</code> is reserved for possible future variants.<sup class="reference" id="cite_ref-RFC_4122_2-4"><a href="#cite_note-RFC_4122-2">[2]</a></sup></p>
<p>The other two variants, variants 1 and 2, are covered by the UUID specification. Variant 1 UUIDs (<code>10xx N=8..b</code>, 2 bits) are referred to as RFC 4122/DCE 1.1 UUIDs, or "Leach-Salz" UUIDs, after the authors of the original <a href="/wiki/Internet_Draft" title="Internet Draft">Internet Draft</a>. Variant 2 (<code>110x N=c..d</code>, 3 bits) is characterized in the RFC as "reserved, Microsoft Corporation backward compatibility", and was used for early GUIDs on the <a class="mw-redirect" href="/wiki/Windows" title="Windows">Windows</a> platform. Variant bits aside, the two variants are the same except that when reduced to a binary form for storage or transmission, variant 1 UUIDs use "network" (big-endian) byte order, while variant 2 GUIDs use "native" (little-endian) byte order. In their textual representations, variants 1 and 2 are the same except for the variant (<code>N</code>) digit.</p>
<p>When byte swapping is required to convert between the network byte order of variant 1 and the little-endian "native" order of variant 2, the fields above define the swapping. The first three fields are unsigned 32- and 16-bit integers and are subject to swapping, while the last two fields consist of uninterpreted bytes, not subject to swapping. This is so even for version 3, 4, and 5 UUID's where the canonical fields do not correspond to the content of the UUID.<sup class="reference" id="cite_ref-RFC_4122_2-5"><a href="#cite_note-RFC_4122-2">[2]</a></sup></p>
<p>Note that while some important GUIDs, such as the identifier for the <a href="/wiki/Component_Object_Model" title="Component Object Model">Component Object Model</a> <a href="/wiki/IUnknown" title="IUnknown">IUnknown</a> interface, are nominally variant 2 UUIDs, many identifiers generated and used in Microsoft <a class="mw-redirect" href="/wiki/Windows" title="Windows">Windows</a> software and referred to as "GUIDs" are standard variant 1 RFC 4122/DCE 1.1 network byte-order UUIDs, rather than little-endian variant 2 UUIDs. The current version of the Microsoft guidgen tool produces standard variant 1 UUIDs. Some Microsoft documentation states that "GUID" is a synonym for "UUID",<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup> as standardized in RFC 4122. RFC 4122 itself states that UUIDs "are also known as GUIDs". All this suggests that "GUID", while originally referring to a variant of UUID used by Microsoft, has become simply an alternative name for UUID, with both variant 1 and variant 2 GUIDs being extant.</p>
<h2><span class="mw-headline" id="Versions">Versions</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Universally_unique_identifier&amp;action=edit&amp;section=5" title="Edit section: Versions">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>For both variants 1 and 2, five "versions" are defined in the standards, and each version may be more appropriate than the others in specific use cases. Version is indicated by the <code>M</code> in the string representation.</p>
<p>Version 1 UUIDs are generated from a time and a node id (usually the MAC address); version 2 UUIDs are generated from an identifier (usually a group or user id), time, and a node id; versions 3 and 5 produce deterministic UUIDs generated by hashing a <a href="/wiki/Namespace" title="Namespace">namespace</a> identifier and name; and version 4 UUIDs are generated using a <a href="/wiki/Randomness" title="Randomness">random</a> or <a href="/wiki/Pseudorandomness" title="Pseudorandomness">pseudo-random</a> number.</p>
<h3><span class="mw-headline" id="Nil_UUID">Nil UUID</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Universally_unique_identifier&amp;action=edit&amp;section=6" title="Edit section: Nil UUID">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The "nil" UUID, a special case, is the UUID, <code>00000000-0000-0000-0000-00000000</code>; that is, all bits set to zero.<sup class="reference" id="cite_ref-RFC_4122_2-6"><a href="#cite_note-RFC_4122-2">[2]</a></sup></p>
<h3><span class="mw-headline" id="Version_1_.28date-time_and_MAC_address.29">Version 1 (date-time and MAC address)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Universally_unique_identifier&amp;action=edit&amp;section=7" title="Edit section: Version 1 (date-time and MAC address)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Version 1 concatenates the 48-bit <a href="/wiki/MAC_address" title="MAC address">MAC address</a> of the "node" (that is, the computer generating the UUID), with a 60-bit timestamp, being the number of 100-<a href="/wiki/Nanosecond" title="Nanosecond">nanosecond</a> intervals since midnight October 15, 1582 <a href="/wiki/Coordinated_Universal_Time" title="Coordinated Universal Time">Coordinated Universal Time</a> (UTC), the date on which the <a href="/wiki/Gregorian_calendar" title="Gregorian calendar">Gregorian calendar</a> was first adopted. RFC 4122 states that the time value rolls over around 3400 AD,<sup class="reference" id="cite_ref-RFC_4122_2-7"><a href="#cite_note-RFC_4122-2">[2]</a></sup> depending on the algorithm used, which implies that the 60-bit timestamp is a signed quantity. However some software, such as the libuuid library, treats the timestamp as unsigned, putting the rollover time in 5236 AD.<sup class="reference" id="cite_ref-e2fsprogs_8-0"><a href="#cite_note-e2fsprogs-8">[8]</a></sup></p>
<p>A 14-bit "uniqifying" clock sequence extends the timestamp in order to handle cases where the processor clock does not advance fast enough, or where there are multiple processors and UUID generators per node. With each version 1 UUID corresponding to a single point in space (the node) and time (intervals and clock sequence), the chance of two properly-generated version 1 UUID's being unintentionally the same is practically nil. Since the time and clock sequence total 74 bits, 2<sup>74</sup> (1.8x10<sup>22</sup> or 18 sextillion) version 1 UUIDs can be generated per node id, at a maximum average rate of 163 billion per second per node id.<sup class="reference" id="cite_ref-RFC_4122_2-8"><a href="#cite_note-RFC_4122-2">[2]</a></sup></p>
<p>In contrast to other UUID versions, version 1 and 2 UUIDs based on MAC addresses from <a class="mw-redirect" href="/wiki/Network_card" title="Network card">network cards</a> rely for their uniqueness in part on an identifier issued by a central registration authority, namely the <a class="mw-redirect" href="/wiki/Organizationally_Unique_Identifier" title="Organizationally Unique Identifier">Organizationally Unique Identifier</a> (OUI) part of the MAC address, which is issued by the <a class="mw-redirect" href="/wiki/IEEE" title="IEEE">IEEE</a> to manufacturers of networking equipment.<sup class="reference" id="cite_ref-IEEE-RA_9-0"><a href="#cite_note-IEEE-RA-9">[9]</a></sup></p>
<p>Usage of the the node's network card MAC address for the node id means that a version 1 UUID can be tracked back to the computer that created it. Documents can sometimes be traced to the computers where they were created or edited through UUIDs embedded into them by <a class="mw-redirect" href="/wiki/Word_processing" title="Word processing">word processing</a> software. This <a href="/wiki/Privacy" title="Privacy">privacy</a> hole was used when locating the creator of the <a href="/wiki/Melissa_(computer_virus)" title="Melissa (computer virus)">Melissa virus</a>.<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup></p>
<p>RFC 4122 does allow the MAC address in a version 1 (or 2) UUID to be replaced by a random 48-bit node id, either because the node does not have a MAC address, or because it is not desirable to expose it. In that case, the RFC requires that the least significant bit of the first octet of the node id should be set to 1.<sup class="reference" id="cite_ref-RFC_4122_2-9"><a href="#cite_note-RFC_4122-2">[2]</a></sup> This corresponds to the <a href="/wiki/Multicast" title="Multicast">multicast</a> bit in MAC addresses and setting it serves to differentiate UUIDs where the node id is randomly-generated from those based on MAC addresses from network cards, which typically have <a href="/wiki/Unicast" title="Unicast">unicast</a> MAC addresses.<sup class="reference" id="cite_ref-RFC_4122_2-10"><a href="#cite_note-RFC_4122-2">[2]</a></sup></p>
<h3><span class="mw-headline" id="Version_2_.28date-time_and_MAC_Address.2C_DCE_security_version.29">Version 2 (date-time and MAC Address, DCE security version)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Universally_unique_identifier&amp;action=edit&amp;section=8" title="Edit section: Version 2 (date-time and MAC Address, DCE security version)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>RFC 4122 reserves version 2 for "DCE security" UUIDs; but it does not provide any details. For this reason, many UUID implementatons omit version 2. However, the specification of version 2 UUIDs is provided by the DCE 1.1 Authentication and Security Services specification.<sup class="reference" id="cite_ref-dce_spec_4-1"><a href="#cite_note-dce_spec-4">[4]</a></sup></p>
<p>Version 2 UUIDs are similar to version 1, except that the least significant 8 bits of the clock sequence (clock_seq_low) are replaced by a "local domain" number, and the least significant 32 bits of the timestamp (time_low) are replaced by an integer identifier meaningful within the specified local domain. On <a href="/wiki/POSIX" title="POSIX">POSIX</a> systems, local domain numbers 1 and 2 are for user ids (UIDs), and group ids (GIDs), respectively, and other local domain numbers are site-defined. On non-POSIX systems, all local domain numbers are site-defined.</p>
<p>The ability to include a 40-bit domain/identifier in the UUID comes with a tradeoff. On the one hand, 40 bits allow about 1 trillion domain/identifier values. On the other hand, with the clock value truncated to the 28 most significant bits, compared to 60 bits in version 1, the clock in a version 2 UUID will "tick" only once every 429.49 seconds, a little more than 7 minutes, as opposed to every 100 nanoseconds for version 1. And with a clock sequence of only 6 bits, compared to 14 bits in version 1, only 64 unique UUID's per node/domain/id can be generated per 7 minute clock tick, compared to 16,384 clock sequence values for version 1.<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup></p>
<h3><span class="mw-headline" id="Versions_3_and_5_.28namespace_name-based.29">Versions 3 and 5 (namespace name-based)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Universally_unique_identifier&amp;action=edit&amp;section=9" title="Edit section: Versions 3 and 5 (namespace name-based)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Version 3 and 5 UUIDs are generated by <a href="/wiki/Cryptographic_hash_function" title="Cryptographic hash function">hashing</a> a <a href="/wiki/Namespace" title="Namespace">namespace</a> identifier and name. Version 3 uses <a href="/wiki/MD5" title="MD5">MD5</a> as the hashing algorithm, and version 5 uses <a class="mw-redirect" href="/wiki/SHA1" title="SHA1">SHA1</a>.<sup class="reference" id="cite_ref-RFC_4122_2-11"><a href="#cite_note-RFC_4122-2">[2]</a></sup></p>
<p>The namespace identifier is itself a UUID. The specification provides UUIDs to represent the namespaces for <a href="/wiki/Uniform_Resource_Locator" title="Uniform Resource Locator">URLs</a>, <a href="/wiki/Fully_qualified_domain_name" title="Fully qualified domain name">fully qualified domain names</a>, <a href="/wiki/Object_identifier" title="Object identifier">object identifiers</a>, and <a href="/wiki/X.500" title="X.500">X.500</a> <a class="mw-redirect" href="/wiki/LDAP" title="LDAP">distinguished names</a>; but other UUIDs may be used as namespace designators.</p>
<p>To determine the version 3 UUID corresponding to a given namespace and name, the UUID of the namespace is transformed to a string of bytes, concatenated with the input name, then hashed with MD5, yielding 128 bits. Six bits are then replaced by fixed values, the 4-bit version (e.g. <code>0011</code> for version 3), and the 2-bit UUID "variant" (e.g <code>10</code> indicating RFC 4122 UUIDs). Since 6 bits are thus predetermined, only 122 bits contribute to the uniqueness of the UUID.</p>
<p>Version 5 UUIDs are similar, but SHA1 is used instead of MD5. Since SHA1 generates 160-bit digests, the digest is truncated to 128-bits before the version and variant bits are inserted.</p>
<p>Version 3 and 5 UUIDs have the property that the same namespace and name will map to the same UUID. However, the namespace and name cannot be determined from the version 3 or 5 UUID alone.</p>
<p>RFC 4122 recommends version 5 (SHA1) over version 3 (MD5) and counsels against use of UUIDs of either version as security credentials.<sup class="reference" id="cite_ref-RFC_4122_2-12"><a href="#cite_note-RFC_4122-2">[2]</a></sup></p>
<h3><span class="mw-headline" id="Version_4_.28random.29">Version 4 (random)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Universally_unique_identifier&amp;action=edit&amp;section=10" title="Edit section: Version 4 (random)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Version 4 UUIDs are generated using <a href="/wiki/Random_number_generation" title="Random number generation">random or pseudo-random numbers</a>.<sup class="reference" id="cite_ref-RFC_4122_2-13"><a href="#cite_note-RFC_4122-2">[2]</a></sup></p>
<p>This method sets the version number (4 bits) and the UUID variant (2 bits) into a 128-bit randomly-generated integer. Therefore, since these 6 bits are predetermined, version 4 UUIDs have 122 random bits, and the total number of possible version 4 UUIDs is 2<sup>122</sup>, or 5.3x10<sup>36</sup> (5.3 undecillion).</p>
<p><a href="/wiki/Pseudorandom_number_generator" title="Pseudorandom number generator">Pseudorandom number generation</a> often lacks necessary entropy, and RFC 4122 recommends that when a high-grade source of randomness is not available, that one of the other UUID versions be used instead. Some implementations of version 4 UUIDs do not satisfy this requirement. For example, the <a class="mw-redirect" href="/wiki/WinAPI" title="WinAPI">WinAPI</a> GUID generator, which uses a pseudo-random number generator, has been shown to produce UUIDs which follow a predictable pattern.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (January 2017)">citation needed</span></a></i>]</sup></p>
<h2><span class="mw-headline" id="Collisions">Collisions</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Universally_unique_identifier&amp;action=edit&amp;section=11" title="Edit section: Collisions">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p><a href="/wiki/Collision_(computer_science)" title="Collision (computer science)">Collision</a> occurs when the same UUID is generated more than once and assigned to different referents. In the case of standard version 1 and 2 UUIDs using unique MAC addresses from network cards, collisions can occur only when an implementation varies from the standards, either inadvertently or intentionally.</p>
<p>In contrast, with version 1 and 2 UUIDs using randomly-generated node ids, hash-based version 3 and 5 UUIDs, and random version 4 UUIDs, collisions can occur even without implementation problems, albeit with a probability so small that it can normally be ignored. <h2><span class="mw-headline" id="Uses">Uses</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Universally_unique_identifier&amp;action=edit&amp;section=12" title="Edit section: Uses">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Significant uses include <a href="/wiki/Ext2" title="Ext2">ext2</a>/<a href="/wiki/Ext3" title="Ext3">ext3</a>/<a href="/wiki/Ext4" title="Ext4">ext4</a> filesystem userspace tools (<a href="/wiki/E2fsprogs" title="E2fsprogs">e2fsprogs</a> uses libuuid provided by <a href="/wiki/Util-linux" title="Util-linux">util-linux</a>), <a class="mw-redirect" href="/wiki/LUKS" title="LUKS">LUKS</a> encrypted partitions, <a href="/wiki/GNOME" title="GNOME">GNOME</a>, <a href="/wiki/KDE" title="KDE">KDE</a>, and <a class="mw-redirect" href="/wiki/Mac_OS_X" title="Mac OS X">Mac OS X</a>,<sup class="reference" id="cite_ref-15"><a href="#cite_note-15">[15]</a></sup> most of which are derived from the original implementation by <a href="/wiki/Theodore_Ts%27o" title="Theodore Ts'o">Theodore Ts'o</a>.<sup class="reference" id="cite_ref-e2fsprogs_8-1"><a href="#cite_note-e2fsprogs-8">[8]</a></sup></p>
<p>One of the uses of UUIDs in <a href="/wiki/Solaris_(operating_system)" title="Solaris (operating system)">Solaris</a> (using Open Software Foundation implementation) is identification of a running operating system instance for the purpose of pairing crash dump data with Fault Management Event in the case of kernel panic.<sup class="reference" id="cite_ref-16"><a href="#cite_note-16">[16]</a></sup></p>
<h3><span class="mw-headline" id="In_COM">In COM</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Universally_unique_identifier&amp;action=edit&amp;section=13" title="Edit section: In COM">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>There are several flavors of GUIDs used in Microsoft's <a href="/wiki/Component_Object_Model" title="Component Object Model">Component Object Model</a> (COM):</p>
<ul>
<li>IID – interface identifier; (The ones that are registered on a system are stored in the <a href="/wiki/Windows_Registry" title="Windows Registry">Windows Registry</a> at <code class="mw-highlight" dir="ltr" id="" style=""><span class="k">[</span><span class="nb">HKEY_CLASSES_ROOT</span><span class="k">\Interface]</span></code><sup class="reference" id="cite_ref-17"><a href="#cite_note-17">[17]</a></sup> )</li>
<li>CLSID – class identifier; (Stored at <code class="mw-highlight" dir="ltr" id="" style=""><span class="k">[</span><span class="nb">HKEY_CLASSES_ROOT</span><span class="k">\CLSID]</span></code>)</li>
<li>LIBID – type library identifier; (Stored at <code class="mw-highlight" dir="ltr" id="" style=""><span class="k">[</span><span class="nb">HKEY_CLASSES_ROOT</span><span class="k">\TypeLib]</span></code><sup class="reference" id="cite_ref-18"><a href="#cite_note-18">[18]</a></sup>)</li>
<li>CATID – category identifier; (its presence on a class identifies it as belonging to certain class categories, listed at <code class="mw-highlight" dir="ltr" id="" style=""><span class="k">[</span><span class="nb">HKEY_CLASSES_ROOT</span><span class="k">\Component Categories]</span></code><sup class="reference" id="cite_ref-19"><a href="#cite_note-19">[19]</a></sup>)</li>
</ul>
<h3><span class="mw-headline" id="As_database_keys">As database keys</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Universally_unique_identifier&amp;action=edit&amp;section=14" title="Edit section: As database keys">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>UUIDs are commonly used as a <a href="/wiki/Unique_key" title="Unique key">unique key</a> in <a href="/wiki/Database" title="Database">database</a> tables. The NEWID function in <a href="/wiki/Microsoft_SQL_Server" title="Microsoft SQL Server">Microsoft SQL Server</a> version 4 <a href="/wiki/Transact-SQL" title="Transact-SQL">Transact-SQL</a> returns standard random version 4 UUIDs, while the NEWSEQUENTIALID function returns non-standard UUIDs which are committed to ascend in sequence until the next system reboot.<sup class="reference" id="cite_ref-20"><a href="#cite_note-20">[20]</a></sup> The <a href="/wiki/Oracle_Database" title="Oracle Database">Oracle Database</a> SYS_GUID function returns a 16-byte 128-bit RAW value, based on a host identifier and a process or thread identifier, somewhat similar to a version 1 GUID.<sup class="reference" id="cite_ref-21"><a href="#cite_note-21">[21]</a></sup> <a href="/wiki/PostgreSQL" title="PostgreSQL">PostgreSQL</a> contains a UUID datatype,<sup class="reference" id="cite_ref-22"><a href="#cite_note-22">[22]</a></sup> though it leaves generation of the UUIDs themselves to the user. <a href="/wiki/MySQL" title="MySQL">MySQL</a> provides a UUID function, which generates standard version 1 UUIDs.<sup class="reference" id="cite_ref-23"><a href="#cite_note-23">[23]</a></sup></p>
<p>The random nature of standard version 3,4, and 5 UUIDs and the ordering of the fields within standard version 1 and 2 UUIDs may create problems with database <a href="/wiki/Locality_of_reference" title="Locality of reference">locality</a> or performance when UUIDs are used as <a class="mw-redirect" href="/wiki/Primary_key" title="Primary key">primary keys</a>. For example, in 2002 Jimmy Nilsson reported a significant improvement in performance with Microsoft SQL Server when the version 4 UUIDs being used as keys were modified to include a non-random suffix based on system time. This so-called "COMB" (combined time-GUID) approach made the UUIDs non-standard and significantly more likely to be duplicated, as Nilsson acknowledged, but Nilsson only required uniqueness within the application.<sup class="reference" id="cite_ref-24"><a href="#cite_note-24">[24]</a></sup></p>
