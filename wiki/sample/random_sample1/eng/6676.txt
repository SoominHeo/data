<h1 class="firstHeading" id="firstHeading" lang="en">Subset sum problem</h1>

<p>In <a href="/wiki/Computer_science" title="Computer science">computer science</a>, the <b>subset sum problem</b> is an important problem in <a href="/wiki/Computational_complexity_theory" title="Computational complexity theory">complexity theory</a> and <a href="/wiki/Cryptography" title="Cryptography">cryptography</a>. The problem is this: given a set (or <a href="/wiki/Multiset" title="Multiset">multiset</a>) of integers, is there a non-empty subset whose sum is zero? For example, given the set {−7, −3, −2, 5, 8}, the answer is <i>yes</i> because the subset {−3, −2, 5} sums to zero. The problem is <a class="mw-redirect" href="/wiki/NP-complete" title="NP-complete">NP-complete</a>.</p>
<p>An equivalent problem is this: given a set of integers and an integer <i>s</i>, does any non-empty subset sum to <i>s</i>? Subset sum can also be thought of as a special case of the <a href="/wiki/Knapsack_problem" title="Knapsack problem">knapsack problem</a>.<sup class="reference" id="cite_ref-MartelloToth_1-0"><a href="#cite_note-MartelloToth-1">[1]</a></sup> One interesting special case of subset sum is the <a href="/wiki/Partition_problem" title="Partition problem">partition problem</a>, in which <i>s</i> is half of the sum of all elements in the set.</p>

<h2><span class="mw-headline" id="Complexity">Complexity</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Subset_sum_problem&amp;action=edit&amp;section=1" title="Edit section: Complexity">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The <a href="/wiki/Computational_complexity_theory" title="Computational complexity theory">complexity</a> of the subset sum problem can be viewed as depending on two parameters, <i>N</i>, the number of decision variables, and <i>P</i>, the precision of the problem (stated as the number of binary place values that it takes to state the problem). (Note: here the letters <i>N</i> and <i>P</i> mean something different from what they mean in the <b><a href="/wiki/NP_(complexity)" title="NP (complexity)">NP</a></b> class of problems.)</p>
<p>The complexity of the best known algorithms is <a class="mw-redirect" href="/wiki/Exponential_time" title="Exponential time">exponential</a> in the smaller of the two parameters <i>N</i> and <i>P</i>. Thus, the problem is most difficult if <i>N</i> and <i>P</i> are of the same order. It only becomes easy if either <i>N</i> or <i>P</i> becomes very small.</p>
<p>If <i>N</i> (the number of variables) is small, then an <a class="mw-redirect" href="/wiki/Exhaustive_search" title="Exhaustive search">exhaustive search</a> for the solution is practical. If <i>P</i> (the number of place values) is a small fixed number, then there are <a href="/wiki/Dynamic_programming" title="Dynamic programming">dynamic programming</a> algorithms that can solve it exactly.</p>
<p>Efficient algorithms for both small <i>N</i> and small <i>P</i> cases are given below.</p>
<h2><span class="mw-headline" id="Exponential_time_algorithm">Exponential time algorithm</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Subset_sum_problem&amp;action=edit&amp;section=2" title="Edit section: Exponential time algorithm">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>There are several ways to solve subset sum in time exponential in <i>N</i>. The most <a class="mw-redirect" href="/wiki/Naive_solution" title="Naive solution">naïve algorithm</a> would be to cycle through all subsets of N numbers and, for every one of them, check if the subset sums to the right number. The running time is of order <i>O</i>(2<sup><i>N</i></sup><i>N</i>), since there are 2<sup><i>N</i></sup> subsets and, to check each subset, we need to sum at most <i>N</i> elements.</p>
<p>A better exponential time algorithm is known which runs in time <i>O</i>(2<sup><i>N</i>/2</sup>). The algorithm splits arbitrarily the <i>N</i> elements into two sets of <i>N</i>/2 each. For each of these two sets, it stores a list of the sums of all 2<sup><i>N</i>/2</sup> possible subsets of its elements. Each of these two lists is then sorted. Using a standard comparison sorting algorithm for this step would take time <i>O</i>(2<sup><i>N</i>/2</sup><i>N</i>). However, given a sorted list of sums for <i>k</i> elements, the list can be expanded to two sorted lists with the introduction of a (<i>k</i> + 1)st element, and these two sorted lists can be merged in time <i>O</i>(2<sup><i>k</i></sup>). Thus, each list can be generated in sorted form in time <i>O</i>(2<sup><i>N</i>/2</sup>). Given the two sorted lists, the algorithm can check if an element of the first array and an element of the second array sum up to <i>s</i> in time <i>O</i>(2<sup><i>N</i>/2</sup>). To do that, the algorithm passes through the first array in decreasing order (starting at the largest element) and the second array in increasing order (starting at the smallest element). Whenever the sum of the current element in the first array and the current element in the second array is more than <i>s</i>, the algorithm moves to the next element in the first array. If it is less than <i>s</i>, the algorithm moves to the next element in the second array. If two elements with sum <i>s</i> are found, it stops. Horowitz and <a href="/wiki/Sartaj_Sahni" title="Sartaj Sahni">Sahni</a> first published this algorithm in a technical report in 1972.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup></p>
<h2><span class="mw-headline" id="Pseudo-polynomial_time_dynamic_programming_solution">Pseudo-polynomial time dynamic programming solution</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Subset_sum_problem&amp;action=edit&amp;section=3" title="Edit section: Pseudo-polynomial time dynamic programming solution">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The problem can be solved in <a href="/wiki/Pseudo-polynomial_time" title="Pseudo-polynomial time">pseudo-polynomial time</a> using <a href="/wiki/Dynamic_programming" title="Dynamic programming">dynamic programming</a>. Suppose the sequence is</p>
<dl>
<dd><i>x</i><sub>1</sub>, ..., <i>x<sub>N</sub></i></dd>
</dl>
<p>and we wish to determine if there is a nonempty subset which sums to zero. Define the boolean-valued function <i>Q</i>(<i>i</i>, <i>s</i>) to be the value (<b>true</b> or <b>false</b>) of</p>
<dl>
<dd>"there is a nonempty subset of <i>x</i><sub>1</sub>, ..., <i>x<sub>i</sub></i> which sums to <i>s</i>".</dd>
</dl>
<p>Thus, the solution to the problem "Given a set of integers, is there a non-empty subset whose sum is zero?" is the value of <i>Q</i>(<i>N</i>, 0).</p>
<p>Let <i>A</i> be the sum of the negative values and <i>B</i> the sum of the positive values. Clearly, <span class="nowrap"><i>Q</i>(<i>i</i>, <i>s</i>) = <b>false</b></span>, if <span class="nowrap"><i>s</i> &lt; <i>A</i></span> or <span class="nowrap"><i>s</i> &gt; <i>B</i></span>. So these values do not need to be stored or computed.</p>
<p>Create an array to hold the values <i>Q</i>(<i>i</i>, <i>s</i>) for <span class="nowrap">1 ≤ <i>i</i> ≤ <i>N</i></span> and <span class="nowrap"><i>A</i> ≤ <i>s</i> ≤ <i>B</i>.</span></p>
<p>The array can now be filled in using a simple recursion. Initially, for <span class="nowrap"><i>A</i> ≤ <i>s</i> ≤ <i>B</i>,</span> set</p>
<dl>
<dd><i>Q</i>(1, <i>s</i>) := (<i>x</i><sub>1</sub> == <i>s</i>)</dd>
</dl>
<p>where <i>==</i> is a boolean function that returns true if <i>x</i><sub>1</sub> is equal to <i>s</i>, false otherwise.</p>
<p>Then, for <i>i</i> = 2, …, <i>N</i>, set</p>
<dl>
<dd><i>Q</i>(<i>i</i>, <i>s</i>) := <i>Q</i>(<i>i</i> − 1, <i>s</i>) <b>or</b> (<i>x<sub>i</sub></i> == <i>s</i>) <b>or</b> <i>Q</i>(<i>i</i> − 1, <i>s</i> − <i>x<sub>i</sub></i>),  for <i>A</i> ≤ <i>s</i> ≤ <i>B</i>.</dd>
</dl>
<p>For each assignment, the values of <i>Q</i> on the right side are already known, either because they were stored in the table for the previous value of <i>i</i> or because <span class="nowrap"><i>Q</i>(<i>i</i> − 1,<i>s</i> − <i>x<sub>i</sub></i>) = <b>false</b></span> if <span class="nowrap"><i>s</i> − <i>x<sub>i</sub></i> &lt; <i>A</i></span> or <span class="nowrap"><i>s</i> − <i>x<sub>i</sub></i> &gt; <i>B</i>.</span> Therefore, the total number of arithmetic operations is <span class="nowrap"><i>O</i>(<i>N</i>(<i>B</i> − <i>A</i>)).</span> For example, if all the values are <i>O</i>(<i>N<sup>k</sup></i>) for some <i>k</i>, then the time required is <i>O</i>(<i>N</i><sup><i>k</i>+2</sup>).</p>
<p>This algorithm is easily modified to return the subset with sum 0 if there is one.</p>

<h2><span class="mw-headline" id="Polynomial_time_approximate_algorithm">Polynomial time approximate algorithm</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Subset_sum_problem&amp;action=edit&amp;section=4" title="Edit section: Polynomial time approximate algorithm">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>An <a href="/wiki/Approximation_algorithm" title="Approximation algorithm">approximate</a> version of the subset sum would be: given a set of <i>N</i> numbers <i>x</i><sub>1</sub>, <i>x</i><sub>2</sub>, ..., <i>x<sub>N</sub></i> and a number <i>s</i>, output</p>
<ul>
<li>yes, if there is a subset that sums up to <i>s</i>;</li>
<li>no, if there is no subset summing up to a number between <span class="nowrap">(1 − <i>c</i>)<i>s</i></span> and <i>s</i> for some small <span class="nowrap"><i>c</i> &gt; 0;</span></li>
<li>any answer, if there is a subset summing up to a number between <span class="nowrap">(1 − <i>c</i>)<i>s</i></span> and <i>s</i> but no subset summing up to <i>s</i>.</li>
</ul>
<p>If all numbers are non-negative, the approximate subset sum is solvable in time polynomial in <i>N</i> and 1/<i>c</i>.</p>
<p>The solution for subset sum also provides the solution for the original subset sum problem in the case where the numbers are small (again, for nonnegative numbers). If any sum of the numbers can be specified with at most <i>P</i> bits, then solving the problem approximately with <span class="nowrap"><i>c</i> = 2<sup>−<i>P</i></sup></span> is equivalent to solving it exactly. Then, the polynomial time algorithm for approximate subset sum becomes an exact algorithm with running time polynomial in <i>N</i> and 2<sup><i>P</i></sup> (i.e., exponential in <i>P</i>).</p>
<p>The algorithm for the approximate subset sum problem is as follows:</p>
<pre>
 initialize a list <i>S</i> to contain one element 0.
 for each <i>i</i> from 1 to <i>N</i> do
   let <i>T</i> be a list consisting of <i>x<sub>i</sub></i> + <i>y</i>, for all <i>y</i> in <i>S</i>
   let <i>U</i> be the union of <i>T</i> and <i>S</i>
   sort <i>U</i>
   make <i>S</i> empty 
   let <i>y</i> be the smallest element of <i>U</i> 
   add <i>y</i> to <i>S</i> 
   for each element <i>z</i> of <i>U</i> in increasing order do
      //trim the list by eliminating numbers close to one another
      //and throw out elements greater than <i>s</i>
     if <i>y</i> + <i>cs</i>/<i>N</i> &lt; <i>z</i> ≤ <i>s</i>, set <i>y</i> = <i>z</i> and add <i>z</i> to <i>S</i> 
 if <i>S</i> contains a number between (1 − <i>c</i>)<i>s</i> and <i>s</i>, output <i>yes</i>, otherwise <i>no</i>
</pre>
<p>The algorithm is polynomial time because the lists <i>S</i>, <i>T</i> and <i>U</i> always remain of size polynomial in <i>N</i> and 1/<i>c</i> and, as long as they are of polynomial size, all operations on them can be done in polynomial time. The size of lists is kept polynomial by the trimming step, in which we only include a number <i>z</i> into <i>S</i> if it is greater than the previous one by <i>cs</i>/<i>N</i> and not greater than <i>s</i>.</p>
<p>This step ensures that each element in <i>S</i> is smaller than the next one by at least <i>cs</i>/<i>N</i> and do not contain elements greater than <i>s</i>. Any list with that property consists of no more than <i>N</i>/<i>c</i> elements.</p>
<p>The algorithm is correct because each step introduces an additive error of at most <i>cs</i>/<i>N</i> and <i>N</i> steps together introduce the error of at most <i>cs</i>.</p>
