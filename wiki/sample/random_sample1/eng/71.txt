<p><b>Portable Network Graphics</b> (<b>PNG</b> <span class="nowrap"><span class="IPA nopopups"><a href="/wiki/Help:IPA_for_English" title="Help:IPA for English">/<span style="border-bottom:1px dotted"><span title="/ˈ/ primary stress follows">ˈ</span><span title="'p' in 'pie'">p</span><span title="/ɪ/ short 'i' in 'bid'">ɪ</span><span title="/ŋ/ 'ng' in 'sing'">ŋ</span></span>/</a></span></span><sup class="reference" id="cite_ref-pnghist_2-0"><a href="#cite_note-pnghist-2">[2]</a></sup><sup class="reference" id="cite_ref-pngstd_3-0"><a href="#cite_note-pngstd-3">[3]</a></sup>) is a <a href="/wiki/Raster_graphics" title="Raster graphics">raster graphics</a> <a class="mw-redirect" href="/wiki/Graphics_file_format" title="Graphics file format">file format</a> that supports <a class="mw-redirect" href="/wiki/Lossless_data_compression" title="Lossless data compression">lossless data compression</a>. PNG was created as an improved, non-patented replacement for <a class="mw-redirect" href="/wiki/Graphics_Interchange_Format" title="Graphics Interchange Format">Graphics Interchange Format</a> (GIF), and is the most widely used lossless image compression format on <a class="mw-redirect" href="/wiki/The_Internet" title="The Internet">the Internet</a>.<sup class="reference" id="cite_ref-W3Techs_4-0"><a href="#cite_note-W3Techs-4">[4]</a></sup></p>
<p>PNG supports palette-based images (with palettes of 24-bit <a href="/wiki/RGB_color_model" title="RGB color model">RGB</a> or 32-bit <a class="mw-redirect" href="/wiki/RGBA" title="RGBA">RGBA</a> colors), <a href="/wiki/Grayscale" title="Grayscale">grayscale</a> images (with or without <a class="mw-redirect" href="/wiki/Alpha_channel" title="Alpha channel">alpha channel</a>), and full-color non-palette-based RGB[A] images (with or without alpha channel). PNG was designed for transferring images on the Internet, not for professional-quality print graphics, and therefore does not support non-RGB <a href="/wiki/Color_space" title="Color space">color spaces</a> such as <a href="/wiki/CMYK_color_model" title="CMYK color model">CMYK</a>.</p>
<p>PNG files nearly always use file extension <code>PNG</code> or <code>png</code> and are assigned <a class="mw-redirect" href="/wiki/MIME_type" title="MIME type">MIME</a> media type <code>image/png</code>. PNG was approved for this use by the <a href="/wiki/Internet_Engineering_Steering_Group" title="Internet Engineering Steering Group">Internet Engineering Steering Group</a> on 14 October 1996,<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> and was published as an <a class="mw-redirect" href="/wiki/ISO/IEC" title="ISO/IEC">ISO/IEC</a> standard in 2004.<sup class="reference" id="cite_ref-iso_1-1"><a href="#cite_note-iso-1">[1]</a></sup></p>

<h2><span class="mw-headline" id="History_and_development">History and development</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=1" title="Edit section: History and development">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="hatnote" role="note">See also: <a class="mw-redirect" href="/wiki/Graphics_Interchange_Format#Unisys_and_LZW_patent_enforcement" title="Graphics Interchange Format">Graphics Interchange Format § Unisys and LZW patent enforcement</a></div>
<p>The motivation for creating the PNG format was in early 1995, after it became known that the <a href="/wiki/Lempel%E2%80%93Ziv%E2%80%93Welch" title="Lempel–Ziv–Welch">Lempel–Ziv–Welch</a> (LZW) <a href="/wiki/Data_compression" title="Data compression">data compression</a> algorithm used in the <a class="mw-redirect" href="/wiki/Graphics_Interchange_Format" title="Graphics Interchange Format">Graphics Interchange Format</a> (GIF) format was <a href="/wiki/Patent" title="Patent">patented</a> by <a href="/wiki/Unisys" title="Unisys">Unisys</a>. There were also other problems with the GIF format that made a replacement desirable, notably its limit of 256 <a href="/wiki/Color" title="Color">colors</a> at a time when computers able to display far more than 256 colors were becoming common.</p>
<p>A January 1995 precursory discussion thread, on the <a href="/wiki/Usenet_newsgroup" title="Usenet newsgroup">usenet newsgroup</a> "comp.graphics" with the subject <i>Thoughts on a GIF-replacement file format</i>, had many propositions, which would later be part of the PNG file format. In this thread, Oliver Fromme, author of the popular <a href="/wiki/DOS" title="DOS">DOS</a> <a href="/wiki/JPEG" title="JPEG">JPEG</a> viewer <a href="/wiki/QPEG" title="QPEG">QPEG</a>, proposed the PING name, meaning <i><a href="/wiki/Recursive_acronym" title="Recursive acronym">PING is not GIF</a></i>, and also the PNG extension.<sup class="reference" id="cite_ref-fromme_6-0"><a href="#cite_note-fromme-6">[6]</a></sup></p>
<p>Although GIF allows for <a href="/wiki/Computer_animation" title="Computer animation">animation</a>, it was decided that PNG should be a single-image format.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup> In 2001, the developers of PNG published the <a href="/wiki/Multiple-image_Network_Graphics" title="Multiple-image Network Graphics">Multiple-image Network Graphics</a> (MNG) format, with support for animation. MNG achieved moderate application support, but not enough among mainstream web browsers and no usage among web site designers or publishers. In 2008, certain Mozilla developers published the <a href="/wiki/APNG" title="APNG">Animated Portable Network Graphics (APNG)</a> format with similar goals. APNG is a format that is natively supported by Gecko- and Presto-based web browsers and is also commonly used for thumbnails on Sony's <a href="/wiki/PlayStation_Portable" title="PlayStation Portable">PlayStation Portable</a> system (using the normal PNG file extension), but as of 2013, usage of APNG remains very minimal.</p>
<ul>
<li>1 October 1996: Version 1.0 of the PNG specification was released, and later appeared as <a href="/wiki/Request_for_Comments" title="Request for Comments">RFC</a> 2083. It became a <a href="/wiki/World_Wide_Web_Consortium" title="World Wide Web Consortium">W3C</a> Recommendation on 1 October 1996.</li>
<li>31 December 1998: Version 1.1, with some small changes and the addition of three new <a href="/wiki/Portable_Network_Graphics#.22Chunks.22_within_the_file" title="Portable Network Graphics">chunks</a>, was released.</li>
<li>11 August 1999: Version 1.2, adding one extra chunk, was released.</li>
<li>10 November 2003: PNG became an International Standard (<a href="/wiki/International_Organization_for_Standardization" title="International Organization for Standardization">ISO</a>/<a href="/wiki/International_Electrotechnical_Commission" title="International Electrotechnical Commission">IEC</a> 15948:2003). This version of PNG differs only slightly from version 1.2 and adds no new chunks.</li>
<li>3 March 2004: <a class="external text" href="http://www.iso.org/iso/en/CatalogueDetailPage.CatalogueDetail?CSNUMBER=29581&amp;scopelist=PROGRAMME" rel="nofollow">ISO/IEC 15948:2004</a>.<sup class="reference" id="cite_ref-iso_1-2"><a href="#cite_note-iso-1">[1]</a></sup></li>
</ul>

<h3><span class="mw-headline" id=".22Chunks.22_within_the_file">"Chunks" within the file</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=5" title='Edit section: "Chunks" within the file'>edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>After the header comes a series of <a href="/wiki/Chunk_(information)" title="Chunk (information)">chunks</a>, each of which conveys certain information about the image. Chunks declare themselves as <i>critical</i> or <i>ancillary</i>, and a program encountering an ancillary chunk that it does not understand can safely ignore it. This chunk-based storage layer structure, similar in concept to a <a class="mw-redirect" href="/wiki/Container_format_(digital)" title="Container format (digital)">container format</a>, is designed to allow the PNG format to be extended while maintaining compatibility with older versions—it provides <a href="/wiki/Forward_compatibility" title="Forward compatibility">forward compatibility</a>, and this same file structure (with different signature and chunks) is used in the associated <a class="mw-redirect" href="/wiki/MNG" title="MNG">MNG</a>, <a href="/wiki/JPEG_Network_Graphics" title="JPEG Network Graphics">JNG</a>, and <a href="/wiki/APNG" title="APNG">APNG</a> formats.</p>
<p>A chunk consists of four parts: length (4 bytes, big-endian), chunk type/name (4 bytes), chunk data (length bytes) and <a href="/wiki/Cyclic_redundancy_check" title="Cyclic redundancy check">CRC</a> (cyclic redundancy code/checksum; 4 bytes). The CRC is a network-byte-order CRC-32 computed over the chunk type and chunk data, but not the length.</p>
<table class="wikitable" style="text-align: center;">
<tr>
<th style="width: 8em">Length</th>
<th style="width: 8em">Chunk type</th>
<th style="width: 16em">Chunk data</th>
<th style="width: 8em">CRC</th>
</tr>
<tr>
<td>4 bytes</td>
<td>4 bytes</td>
<td><i>Length</i> bytes</td>
<td>4 bytes</td>

</table>
<p>Chunk types are given a four-letter <a href="/wiki/Case_sensitivity" title="Case sensitivity">case sensitive</a> ASCII type/name; compare <a href="/wiki/FourCC" title="FourCC">FourCC</a>. The case of the different letters in the name (bit 5 of the numeric value of the character) is a <a href="/wiki/Bit_field" title="Bit field">bit field</a> that provides the <a href="/wiki/Codec" title="Codec">decoder</a> with some information on the nature of chunks it does not recognize.</p>
<p>The case of the first letter indicates whether the chunk is critical or not. If the first letter is uppercase, the chunk is critical; if not, the chunk is ancillary. Critical chunks contain information that is necessary to read the file. If a decoder encounters a critical chunk it does not recognize, it must abort reading the file or supply the user with an appropriate warning.</p>
<p>The case of the second letter indicates whether the chunk is "public" (either in the specification or the registry of special-purpose public chunks) or "private" (not standardised). Uppercase is public and lowercase is private. This ensures that public and private chunk names can never conflict with each other (although two private chunk names could conflict).</p>
<p>The third letter must be uppercase to conform to the PNG specification. It is reserved for future expansion. Decoders should treat a chunk with a lower case third letter the same as any other unrecognised chunk.</p>
<p>The case of the fourth letter indicates whether the chunk is safe to copy by editors that do not recognize it. If lowercase, the chunk may be safely copied regardless of the extent of modifications to the file. If uppercase, it may only be copied if the modifications have not touched any critical chunks.</p>
<h4><span class="mw-headline" id="Critical_chunks">Critical chunks</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=6" title="Edit section: Critical chunks">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>A decoder must be able to interpret critical chunks to read and render a PNG file.</p>
<ul>
<li><code>IHDR</code> must be the first chunk; it contains (in this order) the image's width, height, bit depth, color type, compression method, filter method, and interlace method (13 data bytes total).<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup></li>
<li><code>PLTE</code> contains the <a href="/wiki/Palette_(computing)" title="Palette (computing)">palette</a>; list of colors.</li>
<li><code>IDAT</code> contains the image, which may be split among multiple IDAT chunks. Such splitting increases filesize slightly, but makes it possible to generate a PNG in a streaming manner. The IDAT chunk contains the actual image data, which is the output stream of the compression algorithm.<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup></li>
<li><code>IEND</code> marks the image end.</li>
</ul>
<p>The <code>PLTE</code> chunk is essential for color type 3 (<a href="/wiki/Indexed_color" title="Indexed color">indexed color</a>). It is optional for color types 2 and 6 (truecolor and truecolor with alpha) and it must not appear for color types 0 and 4 (grayscale and grayscale with alpha).</p>
<h4><span class="mw-headline" id="Ancillary_chunks">Ancillary chunks</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=7" title="Edit section: Ancillary chunks">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Other image attributes that can be stored in PNG files include <a href="/wiki/Gamma_correction" title="Gamma correction">gamma</a> values, background color, and textual <a href="/wiki/Metadata" title="Metadata">metadata</a> information. PNG also supports <a href="/wiki/Color_management" title="Color management">color management</a> through the inclusion of <a href="/wiki/International_Color_Consortium" title="International Color Consortium">ICC</a> <a href="/wiki/Color_space" title="Color space">color space</a> profiles.<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup></p>
<ul>
<li><tt>bKGD</tt> gives the default background color. It is intended for use when there is no better choice available, such as in standalone image viewers (but not web browsers; see below for more details).</li>
<li><tt>cHRM</tt> gives the <a href="/wiki/Chromaticity" title="Chromaticity">chromaticity</a> coordinates of the display <a href="/wiki/Primary_color" title="Primary color">primaries</a> and <a href="/wiki/White_point" title="White point">white point</a>.</li>
<li><tt>gAMA</tt> specifies <a href="/wiki/Gamma_correction" title="Gamma correction">gamma</a>.</li>
<li><tt>hIST</tt> can store the histogram, or total amount of each color in the image.</li>
<li><tt>iCCP</tt> is an <a class="mw-redirect" href="/wiki/ICC_color_profile" title="ICC color profile">ICC color profile</a>.</li>
<li><tt>iTXt</tt> contains <a href="/wiki/UTF-8" title="UTF-8">UTF-8</a> text, compressed or not, with an optional <a href="/wiki/IETF_language_tag" title="IETF language tag">language tag</a>. iTXt chunk with the keyword 'XML:com.adobe.xmp' can contain <a href="/wiki/Extensible_Metadata_Platform" title="Extensible Metadata Platform">Extensible Metadata Platform</a> (XMP).</li>
<li><tt>pHYs</tt> holds the intended pixel size and/or <a href="/wiki/Aspect_ratio" title="Aspect ratio">aspect ratio</a> of the image.</li>
<li><tt>sBIT</tt> (significant bits) indicates the color-accuracy of the source data.</li>
<li><tt>sPLT</tt> suggests a palette to use if the full range of colors is unavailable.</li>
<li><tt>sRGB</tt> indicates that the standard <a class="mw-redirect" href="/wiki/SRGB_color_space" title="SRGB color space">sRGB color space</a> is used.</li>
<li><tt>sTER</tt> stereo-image indicator chunk for <a class="mw-redirect" href="/wiki/Stereoscopic" title="Stereoscopic">stereoscopic</a> images.<sup class="reference" id="cite_ref-13"><a href="#cite_note-13">[13]</a></sup></li>
<li><tt>tEXt</tt> can store text that can be represented in <a href="/wiki/ISO/IEC_8859-1" title="ISO/IEC 8859-1">ISO/IEC 8859-1</a>, with one <a href="/wiki/Attribute%E2%80%93value_pair" title="Attribute–value pair">key-value</a> pair for each chunk. The "key" must be between 1 and 79 characters long. Separator is a null character. The "value" can be any length, including zero up to the maximum permissible chunk size minus the length of the keyword and separator. Neither "key" nor "value" can contain null character. Leading or trailing spaces are also disallowed.</li>
<li><tt>tIME</tt> stores the time that the image was last changed.</li>
<li><tt>tRNS</tt> contains transparency information. For indexed images, it stores alpha channel values for one or more palette entries. For truecolor and grayscale images, it stores a single pixel value that is to be regarded as fully transparent.</li>
<li><tt>zTXt</tt> contains compressed text with the same limits as <tt>tEXt.</tt></li>
</ul>
<p>The lowercase first letter in these chunks indicates that they are not needed for the PNG specification. The lowercase last letter in some chunks indicates that they are safe to copy, even if the application concerned does not understand them.</p>

<p>Pixels in PNG images may contain either a number indexing sample data in the separate table, the <a href="/wiki/Palette_(computing)" title="Palette (computing)">palette</a>, contained in the PLTE chunk or the sample data itself, encoded as between one and four numbers. In both cases the numbers are referred to as <a href="/wiki/Channel_(digital_image)" title="Channel (digital image)">channels</a> and every number in the image is encoded with an identical format.</p>
<p>The permitted formats encode each number as an unsigned integral value using a fixed number of bits, referred to in the PNG specification as the <i>bit depth</i>. Notice that this is not the same as <a href="/wiki/Color_depth" title="Color depth">color depth</a>, which is commonly used to refer to the total number of bits in each pixel, not each channel. The permitted bit depths are summarized in the table along with the total number of bits used for each pixel.</p>
<p>The number of channels will depend on whether the image is grayscale or color and whether it has an <a class="mw-redirect" href="/wiki/Alpha_channel" title="Alpha channel">alpha channel</a>. PNG allows the following combinations of channels, called the <i>color type</i>.</p>
<ul>
<li>0: grayscale</li>
<li>2: red, green and blue: rgb/truecolor</li>
<li>3: indexed: channel containing indices into a palette of colors</li>
<li>4: grayscale and alpha: level of <a href="/wiki/Opacity_(optics)" title="Opacity (optics)">opacity</a> for each pixel</li>
<li>6: red, green, blue and alpha</li>
</ul>

<p>With indexed color images, the palette always stores trichromatic colors at a depth of 8 bits per channel (24 bits per palette entry). Additionally, an optional list of 8-bit alpha values for the palette entries may be included; if not included, or if shorter than the palette, the remaining palette entries are assumed to be opaque. The palette must not have more entries than the image bit depth allows for, but it may have fewer (for example, if an image with 8-bit pixels only uses 90 colors then it does not need palette entries for all 256 colors). The palette must contain entries for all the pixel values present in the image.</p>
<p>The standard allows indexed color PNGs to have 1, 2, 4 or 8 bits per pixel; grayscale images with no alpha channel may have 1, 2, 4, 8 or 16 bits per pixel. Everything else uses a bit depth per channel of either 8 or 16. The combinations this allows are given in the table above. The standard requires that decoders can read all supported color formats, but many image editors can only produce a small subset of them.</p>
<h3><span class="mw-headline" id="Transparency_of_image">Transparency of image</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=9" title="Edit section: Transparency of image">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>PNG offers a variety of transparency options. With true-color and grayscale images either a single pixel value can be declared as transparent or an <a class="mw-redirect" href="/wiki/Alpha_channel" title="Alpha channel">alpha channel</a> can be added (enabling any percentage of partial transparency to be used). For paletted images, alpha values can be added to palette entries. The number of such values stored may be less than the total number of palette entries, in which case the remaining entries are considered fully opaque.</p>
<p>The scanning of pixel values for binary transparency is supposed to be performed before any color reduction to avoid pixels' becoming unintentionally transparent. This is most likely to pose an issue for systems that can decode 16-bits-per-channel images (as they must to be compliant with the specification) but only output at 8 bits per channel (the norm for all but the highest end systems).</p>
<p>Alpha storage can be "associated" ("<a href="/wiki/Alpha_compositing" title="Alpha compositing">premultiplied</a>") or "unassociated", but PNG standardized<sup class="reference" id="cite_ref-w3.org_15-0"><a href="#cite_note-w3.org-15">[15]</a></sup> on "unassociated" ("non-premultiplied") alpha so that images with separate transparency masks can be stored losslessly.</p>
<h3><span class="mw-headline" id="Compression">Compression</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=10" title="Edit section: Compression">edit</a><span class="mw-editsection-bracket">]</span></span></h3>

<p>PNG uses a 2-stage compression process:</p>
<ul>
<li>pre-compression: filtering (prediction)</li>
<li>compression: DEFLATE</li>
</ul>
<p>PNG uses a non-patented <a class="mw-redirect" href="/wiki/Lossless_data_compression" title="Lossless data compression">lossless data compression</a> method known as <a href="/wiki/DEFLATE" title="DEFLATE">DEFLATE</a>, which is the same algorithm used in the <a href="/wiki/Zlib" title="Zlib">zlib</a> compression library.</p>
<p>Compared to formats with <a href="/wiki/Lossy_compression" title="Lossy compression">lossy compression</a> such as <a class="mw-redirect" href="/wiki/JPG" title="JPG">JPG</a>, choosing a compression setting higher than average will delay processing but often not result in a significantly smaller file size.</p>
<h4><span class="mw-headline" id="Filtering">Filtering</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=11" title="Edit section: Filtering">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<div class="thumb tright">
<div class="thumbinner" style="width:130px;"><a class="image" href="/wiki/File:Pixel-prediction.svg"><img alt="" class="thumbimage" data-file-height="157" data-file-width="161" height="125" src="//upload.wikimedia.org/wikipedia/commons/thumb/3/39/Pixel-prediction.svg/128px-Pixel-prediction.svg.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/3/39/Pixel-prediction.svg/192px-Pixel-prediction.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/3/39/Pixel-prediction.svg/256px-Pixel-prediction.svg.png 2x" width="128"/></a>

</div>
</div>
<div class="thumb tright">
<div class="thumbinner" style="width:130px;"><a class="image" href="/wiki/File:PNG-Gradient.png"><img alt="" class="thumbimage" data-file-height="68" data-file-width="128" height="68" src="//upload.wikimedia.org/wikipedia/commons/8/89/PNG-Gradient.png" width="128"/></a>

<p>Before DEFLATE is applied, the data is transformed via a prediction method: a single <i>filter method</i> is used for the entire image, while for each image line, a <i>filter type</i> is chosen to transform the data to make it more efficiently compressible.<sup class="reference" id="cite_ref-16"><a href="#cite_note-16">[16]</a></sup> The filter type used for a scanline is prepended to the scanline to enable inline decompression.</p>
<p>There is only one filter method in the current PNG specification (denoted method 0), and thus in practice the only choice is which filter type to apply to each line. For this method, the filter predicts the value of each pixel based on the values of previous neighboring pixels, and subtracts the predicted color of the pixel from the actual value, as in <a class="mw-redirect" href="/wiki/DPCM" title="DPCM">DPCM</a>. An image line filtered in this way is often more compressible than the raw image line would be, especially if it is similar to the line above, since the differences from prediction will generally be clustered around 0, rather than spread over all possible image values. This is particularly important in relating separate rows, since DEFLATE has no understanding that an image is a 2D entity, and instead just sees the image data as a stream of bytes.</p>
<p>There are five filter types for filter method 0; each type predicts the value of each byte (of the image data before filtering) based on the corresponding byte of the pixel to the left (<i>A</i>), the pixel above (<i>B</i>), and the pixel above and to the left (<i>C</i>) or some combination thereof, and encodes the <i>difference</i> between the predicted value and the actual value. Filters are applied to byte values, not pixels; pixel values may be one or two bytes, or several values per byte, but never cross byte boundaries. The filter types are:<sup class="reference" id="cite_ref-17"><a href="#cite_note-17">[17]</a></sup></p>
<table class="wikitable">
<tr>
<th>Type byte</th>
<th>Filter name</th>
<th>Predicted value</th>
</tr>
<tr>
<td>0</td>
<td>None</td>
<td>Zero (so that the raw byte value passes through unaltered)</td>
</tr>
<tr>
<td>1</td>
<td>Sub</td>
<td>Byte <i>A</i> (to the left)</td>
</tr>
<tr>
<td>2</td>
<td>Up</td>
<td>Byte <i>B</i> (above)</td>
</tr>
<tr>
<td>3</td>
<td>Average</td>
<td>Mean of bytes <i>A</i> and <i>B</i>, rounded down</td>
</tr>
<tr>
<td>4</td>
<td>Paeth</td>
<td><i>A</i>, <i>B</i>, or <i>C</i>, whichever is closest to <span class="nowrap"><i>p</i> = <i>A</i> + <i>B</i> − <i>C</i></span></td>
</tr>
</table>
<p>The Paeth filter is based on an algorithm by Alan W. Paeth.<sup class="reference" id="cite_ref-18"><a href="#cite_note-18">[18]</a></sup> Compare to the version of <a class="mw-redirect" href="/wiki/DPCM" title="DPCM">DPCM</a> used in <a href="/wiki/Lossless_JPEG" title="Lossless JPEG">lossless JPEG</a>, and to the <a href="/wiki/Discrete_wavelet_transform" title="Discrete wavelet transform">discrete wavelet transform</a> using 1×2, 2×1, or (for the Paeth predictor) 2×2 windows and <a href="/wiki/Haar_wavelet" title="Haar wavelet">Haar wavelets</a>.</p>
<p>Compression is further improved by choosing filter types adaptively on a line-by-line basis. This improvement, and a heuristic method of implementing it commonly used by PNG-writing software, were created by <a href="/wiki/Lee_Daniel_Crocker" title="Lee Daniel Crocker">Lee Daniel Crocker</a>, who tested the methods on many images during the creation of the format;<sup class="reference" id="cite_ref-19"><a href="#cite_note-19">[19]</a></sup> the choice of filter is a component of file size optimization, as discussed below.</p>
<p>If interlacing is used, each stage of the interlacing is filtered separately, meaning that the image can be progressively rendered as each stage is received; however, interlacing generally makes compression less effective.</p>
<h3><span class="mw-headline" id="Interlacing">Interlacing</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=12" title="Edit section: Interlacing">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">
<div class="thumbinner" style="width:151px;"><a class="image" href="/wiki/File:Adam7_passes.gif"><img alt="" class="thumbimage" data-file-height="149" data-file-width="149" height="149" src="//upload.wikimedia.org/wikipedia/commons/2/27/Adam7_passes.gif" width="149"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Adam7_passes.gif" title="Enlarge"></a></div>
An illustration of Adam7 interlacing over a 16×16 image.</div>
</div>
</div>
<p>PNG offers an optional 2-dimensional, 7-pass <a href="/wiki/Interlacing_(bitmaps)" title="Interlacing (bitmaps)">interlacing</a> scheme—the <a href="/wiki/Adam7_algorithm" title="Adam7 algorithm">Adam7 algorithm</a>. This is more sophisticated than GIF's 1-dimensional, 4-pass scheme, and allows a clearer low-resolution image to be visible earlier in the transfer, particularly if interpolation algorithms such as <a href="/wiki/Bicubic_interpolation" title="Bicubic interpolation">bicubic interpolation</a> are used.<sup class="reference" id="cite_ref-20"><a href="#cite_note-20">[20]</a></sup></p>
<p>However, the 7-pass scheme tends to reduce the data's compressibility more than simpler schemes.</p>
<h3><span class="mw-headline" id="Animation">Animation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=13" title="Edit section: Animation">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">
<div class="thumbinner" style="width:102px;"><a class="image" href="/wiki/File:Animated_PNG_example_bouncing_beach_ball.png"><img alt="" class="thumbimage" data-file-height="100" data-file-width="100" height="100" src="//upload.wikimedia.org/wikipedia/commons/1/14/Animated_PNG_example_bouncing_beach_ball.png" width="100"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Animated_PNG_example_bouncing_beach_ball.png" title="Enlarge"></a></div>
An animated PNG (displays as static image in <a href="/wiki/Comparison_of_web_browsers#Image_format_support" title="Comparison of web browsers">some web browsers</a>)</div>
</div>
</div>
<p>PNG itself does not support animation at all. <a href="/wiki/Multiple-image_Network_Graphics" title="Multiple-image Network Graphics">MNG</a> is an extension to PNG that does; it was designed by members of the PNG Group. MNG shares PNG's basic structure and chunks, but it is significantly more complex and has a different file signature, which automatically renders it incompatible with standard PNG decoders.</p>
<p>The complexity of MNG led to the proposal of <a href="/wiki/APNG" title="APNG">APNG</a> by developers of the Mozilla Foundation. It is based on PNG, supports animation and is simpler than MNG. APNG offers fallback to single-image display for PNG decoders that do not support APNG. However, neither of these formats is currently widely supported. APNG is supported in <a class="mw-redirect" href="/wiki/Mozilla_Firefox" title="Mozilla Firefox">Firefox</a> 3.0 and <a href="/wiki/Opera_(web_browser)" title="Opera (web browser)">Opera</a> 9.5,<sup class="reference" id="cite_ref-21"><a href="#cite_note-21">[21]</a></sup> but since Opera changed its layout engine to <a class="mw-redirect" href="/wiki/Blink_(layout_engine)" title="Blink (layout engine)">Blink</a>, support was dropped. The latest version of <a href="/wiki/Safari_(web_browser)" title="Safari (web browser)">Safari</a> on <a href="/wiki/IOS_8" title="IOS 8">iOS 8</a> and Safari 8 for <a href="/wiki/OS_X_Yosemite" title="OS X Yosemite">OS X Yosemite</a> support APNG.<sup class="reference" id="cite_ref-22"><a href="#cite_note-22">[22]</a></sup> The PNG Group decided in April 2007 not to embrace APNG.<sup class="reference" id="cite_ref-23"><a href="#cite_note-23">[23]</a></sup> Several alternatives were under discussion, <a class="new" href="/w/index.php?title=Animated_Network_Graphics&amp;action=edit&amp;redlink=1" title="Animated Network Graphics (page does not exist)">ANG</a>, aNIM/<a class="new" href="/w/index.php?title=MPNG&amp;action=edit&amp;redlink=1" title="MPNG (page does not exist)">mPNG</a>, “<a class="new" href="/w/index.php?title=PNG_in_GIF&amp;action=edit&amp;redlink=1" title="PNG in GIF (page does not exist)">PNG in GIF</a>” and its subset “RGBA in GIF”.<sup class="reference" id="cite_ref-24"><a href="#cite_note-24">[24]</a></sup></p>
<h2><span class="mw-headline" id="Comparison_with_other_file_formats">Comparison with other file formats</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=14" title="Edit section: Comparison with other file formats">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="hatnote" role="note">Main article: <a href="/wiki/Comparison_of_graphics_file_formats" title="Comparison of graphics file formats">Comparison of graphics file formats</a></div>
<h3><span class="mw-headline" id="Graphics_Interchange_Format_.28GIF.29">Graphics Interchange Format (GIF)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=15" title="Edit section: Graphics Interchange Format (GIF)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<ul>
<li>On small images, GIF can achieve greater compression than PNG (see the <a href="#File_size_and_optimization_software">section on filesize</a>, below).</li>
<li>On most images, except for the above case, a GIF will have a larger file-size than an indexed PNG.</li>
<li>PNG gives a much wider range of transparency options than GIF, including <a class="mw-redirect" href="/wiki/Alpha_channel" title="Alpha channel">alpha channel</a> transparency.</li>
<li>Whereas GIF is limited to 8-bit <a href="/wiki/Indexed_color" title="Indexed color">indexed color</a>, PNG gives a much wider range of color depths, including 24-bit (8 bits per channel) and 48-bit (16 bits per channel) <a class="mw-redirect" href="/wiki/True_Color" title="True Color">truecolor</a>, allowing for greater color precision, smoother fades, etc.<sup class="reference" id="cite_ref-25"><a href="#cite_note-25">[25]</a></sup> When an alpha channel is added, up to 64 bits per pixel (before compression) are possible.</li>
<li>When converting an image from the PNG format to GIF, the image quality may suffer due to <a href="/wiki/Posterization" title="Posterization">posterization</a> if the PNG image has more than 256 colors.</li>
<li>GIF intrinsically supports animated images. PNG supports animation only via unofficial extensions (see the <a href="#Animation">section on animation</a>, above).</li>
</ul>
<p>PNG images are less widely supported by older browsers. In particular, IE6 has limited support for PNG.<sup class="reference" id="cite_ref-26"><a href="#cite_note-26">[26]</a></sup></p>
<h3><span class="mw-headline" id="JPEG">JPEG</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=16" title="Edit section: JPEG">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tleft">
<div class="thumbinner" style="width:202px;"><a class="image" href="/wiki/File:Comparison_of_JPEG_and_PNG.png"><img alt="" class="thumbimage" data-file-height="200" data-file-width="200" height="200" src="//upload.wikimedia.org/wikipedia/commons/a/a4/Comparison_of_JPEG_and_PNG.png" width="200"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Comparison_of_JPEG_and_PNG.png" title="Enlarge"></a></div>
Composite image comparing lossy compression in JPEG with lossless compression in PNG: the JPEG artifacts are easily visible in the background, where the PNG image has solid color.</div>
</div>
</div>
<p><a href="/wiki/JPEG" title="JPEG">JPEG</a> (Joint Photographic Experts Group) format can produce a smaller file than PNG for <a href="/wiki/Photography" title="Photography">photographic</a> (and photo-like) images, since JPEG uses a <a href="/wiki/Lossy_compression" title="Lossy compression">lossy encoding method</a> specifically designed for photographic image data, which is typically dominated by soft, low-contrast transitions, and an amount of noise or similar irregular structures. Using PNG instead of a high-quality JPEG for such images would result in a large increase in filesize with <a href="/wiki/Transparency_(data_compression)" title="Transparency (data compression)">negligible</a> gain in quality. In contrast, when storing images that contain text, line art, or graphics – images with sharp transitions and large areas of solid color – the PNG format can compress image data more than JPEG can. Additionally, PNG is lossless, while JPEG produces noticeable visual artifacts around high-contrast areas. Where an image contains both sharp transitions and photographic parts, a choice must be made between the two effects. JPEG does not support transparency.</p>
<p>Because JPEG uses lossy compression, it also suffers from <a href="/wiki/Generation_loss" title="Generation loss">generation loss</a>, where repeatedly decoding and re-encoding an image to save it again causes a loss of information each time, degrading the image. This does not happen with repeated viewing or copying, but only if the file is edited and saved over again. Because PNG is lossless, it is suitable for storing images to be edited. While PNG is reasonably efficient when compressing photographic images, there are lossless compression formats designed specifically for photographic images, lossless <a href="/wiki/WebP" title="WebP">WebP</a> and <a class="mw-redirect" href="/wiki/Adobe_DNG" title="Adobe DNG">Adobe DNG</a> (digital negative) for example. However these formats are either not widely supported, or are proprietary. An image can be stored losslessly and converted to JPEG format only for distribution, so that there is no generation loss.</p>
<p>The PNG specification does not include a standard for embedding <a class="mw-redirect" href="/wiki/Exchangeable_image_file_format" title="Exchangeable image file format">Exif</a> image data from sources such as digital cameras. Instead, PNG has various dedicated ancillary chunks for storing the metadata that other file formats (such as JPEG) would typically store in Exif format.</p>
<p>Early web browsers did not support PNG images; JPEG and GIF were the main image formats. JPEG was commonly used when exporting images containing gradients for web pages, because of GIF's limited color depth. However, JPEG compression causes a gradient to blur slightly. A PNG file will reproduce a gradient as accurately as possible for a given bit depth, while keeping the file size small. PNG became the optimal choice for small gradient images as web browser support for the format improved. No images at all are needed to display gradients in modern browsers, as gradients can be created using <a class="mw-redirect" href="/wiki/CSS" title="CSS">CSS</a>.</p>
<h3><span class="mw-headline" id="JPEG-LS">JPEG-LS</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=17" title="Edit section: JPEG-LS">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a class="mw-redirect" href="/wiki/JPEG-LS" title="JPEG-LS">JPEG-LS</a> is a "near-lossless" image format by the <a href="/wiki/Joint_Photographic_Experts_Group" title="Joint Photographic Experts Group">Joint Photographic Experts Group</a>, though far less widely known and supported than the other lossy JPEG format discussed above. It is directly comparable with PNG,<sup class="noprint Inline-Template" style="margin-left:0.1em; white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Please_clarify" title="Wikipedia:Please clarify"><span title="explain in which way it is directly comparable, or is this confused with Lossless JPEG? (March 2010)">clarification needed</span></a></i>]</sup> and has a standard set of test images.<sup class="reference" id="cite_ref-jpegstdimg_27-0"><a href="#cite_note-jpegstdimg-27">[27]</a></sup> On the Waterloo Repertoire ColorSet, a standard set of test images (unrelated to the JPEG-LS conformance test set), JPEG-LS generally performs better than PNG, by 10–15%, but on some images PNG performs substantially better, on the order of 50–75%.<sup class="reference" id="cite_ref-pngcf_28-0"><a href="#cite_note-pngcf-28">[28]</a></sup> Thus, if both of these formats are options and file size is an important criterion, they should both be considered, depending on the image.</p>
<h3><span class="mw-headline" id="TIFF">TIFF</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=18" title="Edit section: TIFF">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a class="mw-redirect" href="/wiki/Tagged_Image_File_Format" title="Tagged Image File Format">Tagged Image File Format</a> (TIFF) is a format that incorporates an extremely wide range of options. While this makes TIFF useful as a generic format for interchange between professional image editing applications, it makes adding support for it to applications a much bigger task and so it has little support in applications not concerned with image manipulation (such as web browsers). The high level of extensibility also means that most applications provide only a subset of possible features, potentially creating user confusion and compatibility issues.</p>
<p>The most common general-purpose, lossless compression algorithm used with TIFF is <a href="/wiki/Lempel%E2%80%93Ziv%E2%80%93Welch" title="Lempel–Ziv–Welch">Lempel–Ziv–Welch</a> (LZW). This compression technique, also used in GIF, was covered by patents until 2003. TIFF also supports the compression algorithm PNG uses (i.e. <a class="mw-redirect" href="/wiki/Tagged_Image_File_Format#TIFF_Compression_Tag" title="Tagged Image File Format">Compression Tag 0008<sub>16</sub></a> '<a href="/wiki/Adobe_Systems" title="Adobe Systems">Adobe</a>-style') with medium usage and support by applications. TIFF also offers special-purpose lossless compression algorithms like <a class="mw-redirect" href="/wiki/CCITT_Group_IV" title="CCITT Group IV">CCITT Group IV</a>, which can compress <a href="/wiki/Binary_image" title="Binary image">bilevel images</a> (e.g., faxes or black-and-white text) better than PNG's compression algorithm.</p>
<p>PNG supports non-premultiplied alpha only<sup class="reference" id="cite_ref-w3.org_15-1"><a href="#cite_note-w3.org-15">[15]</a></sup> whereas TIFF also supports "associated" (premultiplied) alpha.</p>
<h2><span class="mw-headline" id="Software_support">Software support</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=19" title="Edit section: Software support">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The official <a href="/wiki/Reference_implementation" title="Reference implementation">reference implementation</a> of the PNG format is the <a class="mw-redirect" href="/wiki/Programming_library" title="Programming library">programming library</a> <i><a href="/wiki/Libpng" title="Libpng">libpng</a></i>.<sup class="reference" id="cite_ref-29"><a href="#cite_note-29">[29]</a></sup> It is published as free software under the terms of a <a class="mw-redirect" href="/wiki/Permissive_free_software_licence" title="Permissive free software licence">permissive free software licence</a>. Therefore, it is usually found as an important system library in free operating systems.</p>

<h2><span class="mw-headline" id="File_size_and_optimization_software">File size and optimization software</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=23" title="Edit section: File size and optimization software">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>PNG file size can vary significantly depending on how it is encoded and compressed; this is discussed and a number of tips are given in <i>PNG: The Definitive Guide.</i><sup class="reference" id="cite_ref-pngcf_28-1"><a href="#cite_note-pngcf-28">[28]</a></sup></p>
<h3><span class="mw-headline" id="Compared_to_GIF">Compared to GIF</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=24" title="Edit section: Compared to GIF">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Compared to GIF files, a PNG file with the same information (256 colors, no ancillary chunks/metadata), compressed by an effective compressor will normally be smaller than GIF. Depending on the file and the compressor, PNG may range from somewhat smaller (10%) to significantly smaller (50%) to somewhat larger (5%), but is rarely significantly larger<sup class="reference" id="cite_ref-pngcf_28-2"><a href="#cite_note-pngcf-28">[28]</a></sup> for large images. This is attributed to the performance of PNG's <a href="/wiki/DEFLATE" title="DEFLATE">DEFLATE</a> compared to GIF's <a class="mw-redirect" href="/wiki/LZW" title="LZW">LZW</a>, and because the added precompression layer of PNG's predictive filters take account of the 2-dimensional image structure to further compress files; as filtered data encodes differences between pixels, they will tend to cluster closer to 0, rather than being spread across all possible values, and thus be more easily compressed by DEFLATE. However, some versions of <a href="/wiki/Adobe_Photoshop" title="Adobe Photoshop">Adobe Photoshop</a>, <a href="/wiki/CorelDRAW" title="CorelDRAW">CorelDRAW</a> and <a class="mw-redirect" href="/wiki/Paint_(software)" title="Paint (software)">MS Paint</a> provide poor PNG compression, creating the impression that GIF is more efficient.<sup class="reference" id="cite_ref-pngcf_28-3"><a href="#cite_note-pngcf-28">[28]</a></sup></p>
<h3><span class="mw-headline" id="File_size_factors">File size factors</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=25" title="Edit section: File size factors">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>PNG files vary in size due to a number of factors:</p>
<dl>
<dt>color depth</dt>
<dd>Color depth can range from 1 to 64 bits per pixel.</dd>
<dt>ancillary chunks</dt>
<dd>PNG supports metadata—this may be useful for editing, but unnecessary for viewing, as on websites.</dd>
<dt>interlacing</dt>
<dd>As each pass of the Adam7 algorithm is separately filtered, this can increase file size.<sup class="reference" id="cite_ref-pngcf_28-4"><a href="#cite_note-pngcf-28">[28]</a></sup></dd>
<dt>filter</dt>
<dd>As a precompression stage, each line is filtered by a predictive filter, which can change from line to line. As the ultimate DEFLATE step operates on the whole image's filtered data, one cannot optimize this row-by-row; the choice of filter for each row is thus potentially very variable, though heuristics exist.<sup class="reference" id="cite_ref-50"><a href="#cite_note-50">[note 1]</a></sup></dd>
<dt>compression</dt>
<dd>With additional computation, DEFLATE compressors can produce smaller files.</dd>
</dl>
<p>There is thus a filesize trade-off between high color depth, maximal metadata (including color space information, together with information that does not affect display), interlacing, and speed of compression, which all yield large files, with lower color depth, fewer or no ancillary chunks, no interlacing, and tuned but computationally intensive filtering and compression. For different purposes one will choose different trade-offs: a maximal file may be best for archiving and editing, while a stripped down file may be best for use on a website, and similarly fast but poor compression is preferred when repeatedly editing and saving a file, while slow but high compression is preferred when a file is stable: when archiving or posting. Interlacing is a trade-off: it dramatically speeds up early rendering of large files (improves latency), but may increase file size (decrease throughput) for little gain, particularly for small files.<sup class="reference" id="cite_ref-pngcf_28-5"><a href="#cite_note-pngcf-28">[28]</a></sup></p>
<h4><span class="mw-headline" id="Lossy_PNG_compression">Lossy PNG compression</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=26" title="Edit section: Lossy PNG compression">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Even though PNG has been designed as a lossless format, PNG encoders can pre-process image data in a lossy fashion (so as to reduce colors used) to improve PNG compression.<sup class="reference" id="cite_ref-51"><a href="#cite_note-51">[50]</a></sup></p>
<h3><span class="mw-headline" id="Image_editing_software">Image editing software</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=27" title="Edit section: Image editing software">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Some programs are more efficient than others when saving PNG files, this relates to implementation of the PNG compression used by the program.</p>
<p>Many graphics programs (such as Apple's <a class="mw-redirect" href="/wiki/Preview_(software)" title="Preview (software)">Preview</a> software) save PNGs with large amounts of <a href="/wiki/Metadata" title="Metadata">metadata</a> and color-correction data that are generally unnecessary for <a href="/wiki/World_Wide_Web" title="World Wide Web">Web</a> viewing. Unoptimized PNG files from <a href="/wiki/Adobe_Fireworks" title="Adobe Fireworks">Adobe Fireworks</a> are also notorious for this since they contain options to make the image editable in supported editors. Also CorelDRAW (at least version 11) sometimes produces PNGs which cannot be opened by Internet Explorer (versions 6–8).</p>
<p><a href="/wiki/Adobe_Photoshop" title="Adobe Photoshop">Adobe Photoshop</a>'s performance on PNG files has improved in the CS Suite when using the Save For Web feature (which also allows explicit PNG/8 use).</p>
<p>Adobe's Fireworks saves larger PNG files than many programs by default. This stems from the mechanics of its <i>Save</i> format: the images produced by Fireworks' save function include large, private chunks, containing complete layer and vector information. This allows further lossless editing. When saved with the <i>Export</i> option, Fireworks' PNGs are competitive with those produced by other image editors, but are no longer editable as anything but flattened bitmaps. Fireworks is unable to save size-optimized vector-editable PNGs.</p>
<p>Other notable examples of poor PNG compressors include:</p>
<ul>
<li>Microsoft's Paint for Windows XP</li>
<li>Microsoft Picture It! Photo Premium 9</li>
</ul>
<p>Poor compression increases the PNG file size but does not affect the image quality or compatibility of the file with other programs.</p>
<p>When the color depth of a <a class="mw-redirect" href="/wiki/True_Color" title="True Color">truecolor</a> image is reduced to an 8-bit palette (as in GIF), the resulting image data is typically much smaller. Thus a truecolor PNG will typically be larger than a color-reduced GIF, although PNG could store the color-reduced version as a palettized file of comparable size. Conversely, some tools, when saving images as PNGs, automatically save them as truecolor, even if the original data use only 8-bit color, thus bloating the file unnecessarily.<sup class="reference" id="cite_ref-pngcf_28-6"><a href="#cite_note-pngcf-28">[28]</a></sup> Both factors can lead to the misconception that PNG files are larger than equivalent GIF files.</p>
<h3><span class="mw-headline" id="Optimizing_tools">Optimizing tools</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=28" title="Edit section: Optimizing tools">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Various tools are available for optimizing PNG files; they do this by:</p>
<ul>
<li>(optionally) removing ancillary chunks,</li>
<li>reducing color depth, either:
<ul>
<li>use a palette (instead of RGB) if the image has 256 or fewer colors,</li>
<li>use a smaller palette, if the image has 2, 4, or 16 colors, or</li>
<li>(optionally) lossily discard some of the data in the original image,</li>
</ul>
</li>
<li>optimizing line-by-line filter choice, and</li>
<li>optimizing DEFLATE compression.</li>
</ul>
<h4><span class="mw-headline" id="Tool_list">Tool list</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=29" title="Edit section: Tool list">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<ul>
<li><a href="/wiki/Pngcrush" title="Pngcrush">pngcrush</a> is the oldest of the popular PNG optimizers. It allows for multiple trials on filter selection and compression arguments, and finally chooses the smallest one. This working model is used in almost every png optimizer.</li>
<li><span id="OptiPNG"></span>OptiPNG was inspired by pngcrush, but iterates over a wider range of compression parameters and performs trials in-memory for faster execution.<sup class="reference" id="cite_ref-pngoptguide_52-0"><a href="#cite_note-pngoptguide-52">[51]</a></sup> The main purpose of OptiPNG is to reduce the size of the PNG IDAT <a href="/wiki/Data_stream" title="Data stream">data stream</a> by trying various filtering and compression methods. It also performs automatic bit depth, color type and color palette reduction where possible, and can correct some data integrity errors in input files. (pngcrush has the ability to do color reduction in a later version.)</li>
<li>Advpng from package <a href="/wiki/AdvanceCOMP" title="AdvanceCOMP">AdvanceCOMP</a> was made to use <a class="mw-redirect" href="/wiki/7-zip#Others" title="7-zip">7-zip's deflater</a> (which is slower but have smaller output than zlib), to optimize png files. However, since PNG is filtered before deflate compression, while advpng uses filter 0 globally (in other words it only uses unfiltered data), it's not a good consideration for png optimization. (In most scenarios, filtering helps more than a good deflater.)<br/>
Advdef from the same package, however, is able to recompress the zlib stream, acting as a re-deflater.</li>
<li><a class="mw-redirect" href="/wiki/Pngout" title="Pngout">pngout</a> was made with the author's own deflater (same to the author's zip utility, kzip), while keeps all facilities of color reduction / filtering. However, pngout doesn't allow for using several trials on filters in a single run, so it's suggested to use its commercial GUI version, pngoutwin, or used with a <a href="/wiki/Portable_Network_Graphics#Wrapper_tools" title="Portable Network Graphics">wrapper</a> to automates the trials or to recompress using its own deflater while keep the filter line by line.<sup class="reference" id="cite_ref-pngoutreusefilter_53-0"><a href="#cite_note-pngoutreusefilter-53">[note 2]</a></sup></li>
<li><a href="/wiki/Zopfli" title="Zopfli">zopflipng</a> was also made with a self-own deflater, zopfli. It has all the optimizing features optipng/pngcrush have (including automating trials) while providing a good deflater.</li>
</ul>

<p>Before zopflipng was available, a good way in practice to perform a png optimization is to use a combination of 2 tools in sequence for optimal compression: one which optimizes filters (and removes ancillary chunks), and one which optimizes DEFLATE. Although pngout offers both, only one type of filter can be specified in a single run, therefore it can be used with a <a href="/wiki/Portable_Network_Graphics#Wrapper_tools" title="Portable Network Graphics">wrapper tool</a> or in combination with optipng or <a href="/wiki/Pngcrush" title="Pngcrush">pngcrush</a>,<sup class="reference" id="cite_ref-pngoutreusefilter_53-2"><a href="#cite_note-pngoutreusefilter-53">[note 2]</a></sup> acting as a re-deflater, like advdef.</p>
<h4><span class="mw-headline" id="Ancillary_chunk_removal">Ancillary chunk removal</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=30" title="Edit section: Ancillary chunk removal">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>For removing ancillary chunks, most PNG optimization tools have the ability to remove all color correction data from PNG files (gamma, white balance, ICC color profile, standard RGB color profile). This often results in much smaller file sizes. For example, the following command line options achieve this with pngcrush:</p>
<p><code>pngcrush -rem gAMA -rem cHRM -rem iCCP -rem sRGB <i>InputFile.png</i> <i>OutputFile.png</i></code></p>
<p>Ancillary chunks can also be losslessly removed using the free Win32 program <a class="external text" href="http://www.fieggen.com/software/pngextra.htm" rel="nofollow">PNGExtra</a>.</p>
<h4><span class="mw-headline" id="Filter_optimization">Filter optimization</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=31" title="Edit section: Filter optimization">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>OptiPNG, pngcrush, pngout, and zopflipng all offer options applying one of the filter types 0–4 globally (using the same filter type for all lines) or with a "pseudo filter" (numbered 5), which for each line chooses one of the filter types 0–4 using an adaptive algorithm. Zopflipng offers 3 different adaptive method, including a brute-force search that attempts to optimize the filtering.<sup class="reference" id="cite_ref-59"><a href="#cite_note-59">[note 8]</a></sup></p>
<p>pngout and zopflipng provide an option to preserve/reuse<sup class="reference" id="cite_ref-pngoutreusefilter_53-3"><a href="#cite_note-pngoutreusefilter-53">[note 2]</a></sup><sup class="reference" id="cite_ref-60"><a href="#cite_note-60">[note 9]</a></sup> the line-by-line filter set present in the input image.</p>
<p>OptiPNG, pngcrush and zopflipng provide options to try different filter strategies in a single run and choose the best. The freeware command line version of pngout doesn't offer this, but the commercial version, pngoutwin, does.<sup class="reference" id="cite_ref-61"><a href="#cite_note-61">[note 10]</a></sup></p>
<h4><span class="mw-headline" id="DEFLATE_optimization">DEFLATE optimization</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=32" title="Edit section: DEFLATE optimization">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p><a href="/wiki/AdvanceCOMP" title="AdvanceCOMP">AdvanceCOMP</a> <tt>advdef</tt>, <tt>advpng</tt>, <a href="/wiki/PNGOUT" title="PNGOUT">PNGOUT</a> and <a class="external text" href="https://code.google.com/p/zopfli/" rel="nofollow">zopflipng</a> employ <a href="/wiki/DEFLATE" title="DEFLATE">DEFLATE</a> compression algorithms that are more exhaustive and produce smaller files than the <a href="/wiki/Zlib" title="Zlib">zlib</a> reference implementation used by the other compressors.</p>
<p><tt>advpng</tt> doesn't have an option to apply filters and always uses filter 0 globally (leaving the image data unfiltered); therefore it should not be used where the image benefits significantly from filtering. By contrast, <tt>advdef</tt> from the same package doesn't deal with PNG structure and acts only as a re-deflater, retaining any existing filter settings.</p>
<h4><span class="mw-headline" id="Wrapper_tools">Wrapper tools</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=33" title="Edit section: Wrapper tools">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Most wrapper tools take several passes using different optimizers and then select the smallest file.</p>
<p>Wrapper tools that simplify this workflow include: <a class="external text" href="http://imageoptim.com" rel="nofollow">ImageOptim</a>, a GUI front-end for Mac OS X; Kashmir Web Optimizer- GUI front-end for Windows; <a class="external text" href="https://github.com/yumeyao/pngoptim/" rel="nofollow">pngoptim</a> a cmd batch script for Windows; <a class="external text" href="http://lyncd.com/2009/03/imgopt-lossless-optimize-png-jpeg/" rel="nofollow">imgopt</a>, a command-line shell script that also losslessly optimizes JPEG images, <a class="external text" href="http://smush.it/" rel="nofollow">Smush.it</a>, an image-optimizing web service; <a class="external text" href="http://tinypng.org/" rel="nofollow">TinyPNG</a>, which provides compression by reducing the number of colors in the image automatically, but preserving alpha transparency; and <a class="external text" href="http://compresspng.com/" rel="nofollow">Compress PNG</a> that allows users to pick the number of colors that should be used.</p>
<p>The <a class="external text" href="http://sourceforge.net/projects/littleutils" rel="nofollow">littleutils</a> is another open-source package, containing a wrapper script called opt-png that uses <a href="/wiki/Pngcrush" title="Pngcrush">pngcrush</a> and a variant of <a class="external text" href="http://entropymine.com/jason/pngrewrite/" rel="nofollow">pngrewrite</a> to reduce bit-depth when possible. <a href="/wiki/Perl" title="Perl">Perl</a> scripts might wish to employ <a class="new" href="/w/index.php?title=Image-Pngslimmer&amp;action=edit&amp;redlink=1" title="Image-Pngslimmer (page does not exist)">Image-Pngslimmer</a> which allows some dynamic optimization.</p>
<p>The current version of <a href="/wiki/IrfanView" title="IrfanView">IrfanView</a> can use PNGOUT as an external plug-in, obviating the need for a separate compressor.</p>
<p>An <a class="mw-redirect" href="/wiki/Open_source" title="Open source">open source</a> Windows program called <a class="external text" href="http://nikkhokkho.sourceforge.net/static.php?page=FileOptimizer" rel="nofollow">FileOptimizer</a> losslessly optimizes many filetypes, including PNG. It runs multiple PNG optimization programs: advpng, apngopt, optipng, PngOptimizer, pngout, pngrewrite, and pngwolf.</p>
<p>Another <a class="mw-redirect" href="/wiki/Open_source" title="Open source">open source</a> Windows tool, <a class="external text" href="https://github.com/yumeyao/pngoptim/" rel="nofollow">pngoptim</a>, uses a more opinionated approach, utilizing only pngout and zopflipng instead of taking the brute force approach of trying multiple optimizers.</p>
<h3><span class="mw-headline" id="Icon_optimization">Icon optimization</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portable_Network_Graphics&amp;action=edit&amp;section=34" title="Edit section: Icon optimization">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Since <a class="mw-redirect" href="/wiki/Computer_icon" title="Computer icon">icons</a> intended for Windows Vista and later versions may contain PNG subimages, the optimizations can be applied to them as well. At least one <a class="mw-redirect" href="/wiki/Icon_editor" title="Icon editor">icon editor</a>, <a class="external text" href="http://www.qualibyte.com/pixelformer/" rel="nofollow">Pixelformer</a>, is able to perform a special optimization pass while saving <a class="mw-redirect" href="/wiki/ICO_(icon_image_file_format)" title="ICO (icon image file format)">ICO</a> files, thereby reducing their sizes. <a class="new" href="/w/index.php?title=FileOptimizer&amp;action=edit&amp;redlink=1" title="FileOptimizer (page does not exist)">FileOptimizer</a> (mentioned above) can also handle ICO files.</p>
<p><a href="/wiki/Apple_Icon_Image_format" title="Apple Icon Image format">Icons for Mac OS X</a> may also contain PNG subimages, yet there isn't such tool available.</p>
