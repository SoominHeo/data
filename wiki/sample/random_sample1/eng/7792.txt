<p><b>Shellshock</b>, also known as <b>Bashdoor</b>,<sup class="reference" id="cite_ref-NYT-20140925-NP_1-0"><a href="#cite_note-NYT-20140925-NP-1">[1]</a></sup> is a family of <a href="/wiki/Security_bug" title="Security bug">security bugs</a><sup class="reference" id="cite_ref-TSM-20140927_2-0"><a href="#cite_note-TSM-20140927-2">[2]</a></sup> in the widely used <a href="/wiki/Unix" title="Unix">Unix</a> <a href="/wiki/Bash_(Unix_shell)" title="Bash (Unix shell)">Bash</a> <a href="/wiki/Shell_(computing)" title="Shell (computing)">shell</a>, the first of which was disclosed on 24 September 2014. Many Internet-facing services, such as some web server deployments, use Bash to process certain requests, allowing an attacker to cause vulnerable versions of Bash to <a href="/wiki/Arbitrary_code_execution" title="Arbitrary code execution">execute arbitrary commands</a>. This can allow an attacker to gain unauthorized access to a computer system.<sup class="reference" id="cite_ref-ZDN-20140929_3-0"><a href="#cite_note-ZDN-20140929-3">[3]</a></sup></p>
<p>Stéphane Chazelas contacted Bash's maintainer, Chet Ramey, on 12 September 2014<sup class="reference" id="cite_ref-NYT-20140925-NP_1-1"><a href="#cite_note-NYT-20140925-NP-1">[1]</a></sup> telling Ramey about his discovery of the original bug, which he called "Bashdoor". Working together with security experts, he soon had a patch as well.<sup class="reference" id="cite_ref-NYT-20140925-NP_1-2"><a href="#cite_note-NYT-20140925-NP-1">[1]</a></sup> The bug was assigned the <a class="mw-redirect" href="/wiki/CVE_identifier" title="CVE identifier">CVE identifier</a> <b>CVE-2014-6271</b>.<sup class="reference" id="cite_ref-seclist-q3-650_4-0"><a href="#cite_note-seclist-q3-650-4">[4]</a></sup> It was announced to the public on 24 September 2014 when Bash updates with the fix were ready for distribution.<sup class="reference" id="cite_ref-seclist-q3-666_5-0"><a href="#cite_note-seclist-q3-666-5">[5]</a></sup></p>
<p>The first bug causes Bash to unintentionally execute commands when the commands are concatenated to the end of <a href="/wiki/Subroutine" title="Subroutine">function definitions</a> stored in the values of <a href="/wiki/Environment_variable" title="Environment variable">environment variables</a>.<sup class="reference" id="cite_ref-NYT-20140925-NP_1-3"><a href="#cite_note-NYT-20140925-NP-1">[1]</a></sup><sup class="reference" id="cite_ref-TR-20140924_6-0"><a href="#cite_note-TR-20140924-6">[6]</a></sup> Within days of the publication of this, intense scrutiny of the underlying design flaws discovered a variety of related vulnerabilities, (<b>CVE-2014-6277</b>, <b>CVE-2014-6278</b>, <b>CVE-2014-7169</b>, <b>CVE-2014-7186</b>, and <b>CVE-2014-7187</b>); which Ramey addressed with a series of further patches.<sup class="reference" id="cite_ref-ITN-20140929_7-0"><a href="#cite_note-ITN-20140929-7">[7]</a></sup><sup class="reference" id="cite_ref-zdnet-betterbash_8-0"><a href="#cite_note-zdnet-betterbash-8">[8]</a></sup></p>
<p>Attackers exploited Shellshock within hours of the initial disclosure by creating <a href="/wiki/Botnet" title="Botnet">botnets</a> of compromised computers to perform <a href="/wiki/Denial-of-service_attack#Distributed_attack" title="Denial-of-service attack">distributed denial-of-service attacks</a> and <a href="/wiki/Vulnerability_scanner" title="Vulnerability scanner">vulnerability scanning</a>.<sup class="reference" id="cite_ref-Wired_9-0"><a href="#cite_note-Wired-9">[9]</a></sup><sup class="reference" id="cite_ref-IT-20140926-JS_10-0"><a href="#cite_note-IT-20140926-JS-10">[10]</a></sup> Security companies recorded millions of attacks and probes related to the bug in the days following the disclosure.<sup class="reference" id="cite_ref-NYT-20140926-NP_11-0"><a href="#cite_note-NYT-20140926-NP-11">[11]</a></sup><sup class="reference" id="cite_ref-businessweek_12-0"><a href="#cite_note-businessweek-12">[12]</a></sup></p>
<p>Shellshock could potentially compromise millions of unpatched servers and other systems. Accordingly, it has been compared to the <a href="/wiki/Heartbleed" title="Heartbleed">Heartbleed</a> bug in its severity.<sup class="reference" id="cite_ref-ZDN-20140929_3-1"><a href="#cite_note-ZDN-20140929-3">[3]</a></sup><sup class="reference" id="cite_ref-mit-tech_13-0"><a href="#cite_note-mit-tech-13">[13]</a></sup></p>
<p></p>

<h2><span class="mw-headline" id="Background">Background</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Shellshock_(software_bug)&amp;action=edit&amp;section=1" title="Edit section: Background">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The Shellshock bug affects <a href="/wiki/Bash_(Unix_shell)" title="Bash (Unix shell)">Bash</a>, a program that various <a href="/wiki/Unix" title="Unix">Unix</a>-based systems use to execute command lines and command scripts. It is often installed as the system's default <a href="/wiki/Command-line_interface" title="Command-line interface">command-line interface</a>. Analysis of the <a href="/wiki/Source_code" title="Source code">source code</a> history of Bash shows the bugs had existed since Bash version 1.03 released in September 1989.<sup class="reference" id="cite_ref-BASH105_CHANGELOG_14-0"><a href="#cite_note-BASH105_CHANGELOG-14">[14]</a></sup><sup class="reference" id="cite_ref-BASHBUG-20141010-SC_15-0"><a href="#cite_note-BASHBUG-20141010-SC-15">[15]</a></sup></p>
<p>Shellshock is a <a href="/wiki/Privilege_escalation" title="Privilege escalation">privilege escalation</a> vulnerability which offers a way for users of a system to execute commands that should be unavailable to them. This happens through the "function export" feature, whereby command scripts created in one running instance of Bash can be shared with subordinate instances.<sup class="reference" id="cite_ref-16"><a href="#cite_note-16">[16]</a></sup> This feature is implemented by encoding the scripts within a table that is shared between the instances, known as the <a href="/wiki/Environment_variable" title="Environment variable">environment variable</a> list. Each new instance of Bash scans this table for encoded scripts, assembles each one into a command that defines that script in the new instance, and executes that command.<sup class="reference" id="cite_ref-exported-function_17-0"><a href="#cite_note-exported-function-17">[17]</a></sup> The new instance assumes that the scripts found in the list come from another instance, but it cannot verify this, nor can it verify that the command that it has built is a properly formed script definition. Therefore, an attacker can execute arbitrary commands on the system or exploit other bugs that may exist in Bash's command interpreter, if the attacker has a way to manipulate the environment variable list and then cause Bash to run.</p>
<p>The presence of the bug was announced to the public on 24 September 2014, when Bash updates with the fix were ready for distribution,<sup class="reference" id="cite_ref-seclist-q3-666_5-1"><a href="#cite_note-seclist-q3-666-5">[5]</a></sup> though it took some time for computers to be updated to close the potential security issue.</p>
<h2><span class="mw-headline" id="Reports_of_attacks">Reports of attacks</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Shellshock_(software_bug)&amp;action=edit&amp;section=2" title="Edit section: Reports of attacks">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Within an hour of the announcement of the Bash vulnerability, there were reports of machines being compromised by the bug. By 25 September 2014, <a href="/wiki/Botnet" title="Botnet">botnets</a> based on computers compromised with exploits based on the bug were being used by attackers for <a href="/wiki/Denial-of-service_attack#Distributed_attack" title="Denial-of-service attack">distributed denial-of-service</a> (DDoS) attacks and <a href="/wiki/Vulnerability_scanner" title="Vulnerability scanner">vulnerability scanning</a>.<sup class="reference" id="cite_ref-Wired_9-1"><a href="#cite_note-Wired-9">[9]</a></sup><sup class="reference" id="cite_ref-IT-20140926-JS_10-1"><a href="#cite_note-IT-20140926-JS-10">[10]</a></sup><sup class="reference" id="cite_ref-bbconShellshock_18-0"><a href="#cite_note-bbconShellshock-18">[18]</a></sup> <a class="mw-redirect" href="/wiki/Kaspersky_Labs" title="Kaspersky Labs">Kaspersky Labs</a> reported that machines compromised in an attack, dubbed "Thanks-Rob", were conducting DDoS attacks against three targets, which they did not identify.<sup class="reference" id="cite_ref-Wired_9-2"><a href="#cite_note-Wired-9">[9]</a></sup> On 26 September 2014, a Shellshock-related botnet dubbed "wopbot" was reported, which was being used for a DDoS attack against <a href="/wiki/Akamai_Technologies" title="Akamai Technologies">Akamai Technologies</a> and to scan the <a href="/wiki/United_States_Department_of_Defense" title="United States Department of Defense">United States Department of Defense</a>.<sup class="reference" id="cite_ref-IT-20140926-JS_10-2"><a href="#cite_note-IT-20140926-JS-10">[10]</a></sup></p>
<p>On 26 September, the security firm <a href="/wiki/Incapsula" title="Incapsula">Incapsula</a> noted 17,400 attacks on more than 1,800 web domains, originating from 400 unique IP addresses, in the previous 24 hours; 55% of the attacks were coming from China and the United States.<sup class="reference" id="cite_ref-NYT-20140926-NP_11-1"><a href="#cite_note-NYT-20140926-NP-11">[11]</a></sup> By 30 September, the website performance firm <a class="mw-redirect" href="/wiki/CloudFlare" title="CloudFlare">CloudFlare</a> said it was tracking approximately 1.5 million attacks and probes per day related to the bug.<sup class="reference" id="cite_ref-businessweek_12-1"><a href="#cite_note-businessweek-12">[12]</a></sup></p>
<p>On 6 October, it was widely reported that <a href="/wiki/Yahoo!" title="Yahoo!">Yahoo!</a> servers had been compromised in an attack related to the Shellshock issue.<sup class="reference" id="cite_ref-19"><a href="#cite_note-19">[19]</a></sup><sup class="reference" id="cite_ref-20"><a href="#cite_note-20">[20]</a></sup> Yet the next day, it was denied that it had been <i>Shellshock</i> that specifically had allowed these attacks.<sup class="reference" id="cite_ref-21"><a href="#cite_note-21">[21]</a></sup></p>
<h2><span class="mw-headline" id="Specific_exploitation_vectors">Specific exploitation vectors</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Shellshock_(software_bug)&amp;action=edit&amp;section=3" title="Edit section: Specific exploitation vectors">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<dl>
<dt>CGI-based web server</dt>
<dd>When a <a href="/wiki/Web_server" title="Web server">web server</a> uses the <a href="/wiki/Common_Gateway_Interface" title="Common Gateway Interface">Common Gateway Interface</a> (CGI) to handle a document request, it passes various details of the request to a handler program in the environment variable list. For example, the variable HTTP_USER_AGENT has a value that, in normal usage, identifies the program sending the request. If the request handler is a Bash script, or if it executes one for example using the <a class="external text" href="http://linux.die.net/man/3/system" rel="nofollow">system(3)</a> call, Bash will receive the environment variables passed by the server and will process them as described above. This provides a means for an attacker to trigger the Shellshock vulnerability with a specially crafted server request.<sup class="reference" id="cite_ref-TR-20140924_6-1"><a href="#cite_note-TR-20140924-6">[6]</a></sup></dd>
<dd>Security documentation for the widely used <a href="/wiki/Apache_HTTP_Server" title="Apache HTTP Server">Apache</a> web server states: "CGI scripts can ... be extremely dangerous if they are not carefully checked."<sup class="reference" id="cite_ref-22"><a href="#cite_note-22">[22]</a></sup> and other methods of handling web server requests are often used. There are a number of online services which attempt to test the vulnerability against web servers exposed to the Internet.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (September 2014)">citation needed</span></a></i>]</sup></dd>
<dt>OpenSSH server</dt>
<dd><a href="/wiki/OpenSSH" title="OpenSSH">OpenSSH</a> has a "ForceCommand" feature, where a fixed command is executed when the user logs in, instead of just running an unrestricted command shell. The fixed command is executed even if the user specified that another command should be run; in that case the original command is put into the environment variable "SSH_ORIGINAL_COMMAND". When the forced command is run in a Bash shell (if the user's shell is set to Bash), the Bash shell will parse the SSH_ORIGINAL_COMMAND environment variable on start-up, and run the commands embedded in it. The user has used their restricted shell access to gain unrestricted shell access, using the Shellshock bug.<sup class="reference" id="cite_ref-qualys_23-0"><a href="#cite_note-qualys-23">[23]</a></sup></dd>
<dt>DHCP clients</dt>
<dd>Some <a href="/wiki/Dynamic_Host_Configuration_Protocol" title="Dynamic Host Configuration Protocol">DHCP</a> clients can also pass commands to Bash; a vulnerable system could be attacked when connecting to an open Wi-Fi network. A DHCP client typically requests and gets an IP address from a DHCP server, but it can also be provided a series of additional options. A malicious DHCP server could provide, in one of these options, a string crafted to execute code on a vulnerable workstation or laptop.<sup class="reference" id="cite_ref-mit-tech_13-1"><a href="#cite_note-mit-tech-13">[13]</a></sup></dd>
<dt>Qmail server</dt>
<dd>When using Bash to process email messages (e.g. through .forward or qmail-alias piping), the <a href="/wiki/Qmail" title="Qmail">qmail</a> mail server passes external input through in a way that can exploit a vulnerable version of Bash.<sup class="reference" id="cite_ref-24"><a href="#cite_note-24">[24]</a></sup><sup class="reference" id="cite_ref-25"><a href="#cite_note-25">[25]</a></sup></dd>
<dt>IBM HMC restricted shell</dt>
<dd>The bug can be exploited to gain access to Bash from the <a href="/wiki/Restricted_shell" title="Restricted shell">restricted shell</a> of the <a href="/wiki/IBM_Hardware_Management_Console" title="IBM Hardware Management Console">IBM Hardware Management Console</a>,<sup class="reference" id="cite_ref-26"><a href="#cite_note-26">[26]</a></sup> a tiny Linux variant for system administrators. IBM released a patch to resolve this.<sup class="reference" id="cite_ref-ibm-hmc_27-0"><a href="#cite_note-ibm-hmc-27">[27]</a></sup></dd>
</dl>
<h2><span class="mw-headline" id="Reported_vulnerabilities">Reported vulnerabilities</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Shellshock_(software_bug)&amp;action=edit&amp;section=4" title="Edit section: Reported vulnerabilities">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Shellshock_(software_bug)&amp;action=edit&amp;section=5" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The maintainer of Bash was warned about the first discovery of the bug on 12 September 2014; a fix followed soon.<sup class="reference" id="cite_ref-NYT-20140925-NP_1-4"><a href="#cite_note-NYT-20140925-NP-1">[1]</a></sup> A few companies and distributors were informed before the matter was publicly disclosed on 24 September 2014 with CVE identifier CVE-2014-6271.<sup class="reference" id="cite_ref-seclist-q3-650_4-1"><a href="#cite_note-seclist-q3-650-4">[4]</a></sup><sup class="reference" id="cite_ref-seclist-q3-666_5-2"><a href="#cite_note-seclist-q3-666-5">[5]</a></sup> However, after the release of the patch there were subsequent reports of different, yet related vulnerabilities.<sup class="reference" id="cite_ref-wheeler-summary_28-0"><a href="#cite_note-wheeler-summary-28">[28]</a></sup></p>
<p>On 26 September 2014, two open-source contributors, <a href="/wiki/David_A._Wheeler" title="David A. Wheeler">David A. Wheeler</a> and Norihiro Tanaka, noted that there were additional issues, even after patching systems using the most recently available patches. In an email addressed to the oss-sec list and the bash bug list, Wheeler wrote: "This patch just continues the 'whack-a-mole' job of fixing parsing errors that began with the first patch. Bash's parser is certain [to] have many many many other vulnerabilities".<sup class="reference" id="cite_ref-BASH_Whack-a-mole_29-0"><a href="#cite_note-BASH_Whack-a-mole-29">[29]</a></sup> However, this rather was some general reasoning without actually presenting exploitation examples and implied restricting Bash functionality with the effect that some Bash scripts won't work any more, even if <i>not</i> intended to harm other users.</p>
<p>On 27 September 2014, <a href="/wiki/Micha%C5%82_Zalewski" title="Michał Zalewski">Michał Zalewski</a> from <a class="mw-redirect" href="/wiki/Google_Inc." title="Google Inc.">Google Inc.</a> announced his discovery of other Bash vulnerabilities,<sup class="reference" id="cite_ref-ITN-20140929_7-1"><a href="#cite_note-ITN-20140929-7">[7]</a></sup> one based upon the fact that Bash is typically compiled without <a href="/wiki/Address_space_layout_randomization" title="Address space layout randomization">address space layout randomization</a>.<sup class="reference" id="cite_ref-HH-20140928_30-0"><a href="#cite_note-HH-20140928-30">[30]</a></sup> On 1 October, Zalewski released details of the final bugs and confirmed that a patch by Florian Weimer from <a href="/wiki/Red_Hat" title="Red Hat">Red Hat</a> posted on 25 September does indeed prevent them. He has done that using a <a href="/wiki/Fuzzing" title="Fuzzing">fuzzing</a> technique with the aid of software utility known as <i><a href="/wiki/American_fuzzy_lop_(fuzzer)" title="American fuzzy lop (fuzzer)">american fuzzy lop</a></i>.<sup class="reference" id="cite_ref-lcamtuf-oct-1_31-0"><a href="#cite_note-lcamtuf-oct-1-31">[31]</a></sup></p>
<h3><span class="mw-headline" id="Initial_report_.28CVE-2014-6271.29">Initial report (CVE-2014-6271)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Shellshock_(software_bug)&amp;action=edit&amp;section=6" title="Edit section: Initial report (CVE-2014-6271)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>This original form of the vulnerability involves a specially crafted environment variable containing an exported function definition, followed by arbitrary commands. Bash incorrectly executes the trailing commands when it imports the function.<sup class="reference" id="cite_ref-nvd6271_32-0"><a href="#cite_note-nvd6271-32">[32]</a></sup> The vulnerability can be tested with the following command:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
env <span class="nv">x</span><span class="o">=</span><span class="s1">'() { :;}; echo vulnerable'</span> bash -c <span class="s2">"echo this is a test"</span>
</pre></div>
<p>In systems affected by the vulnerability, the above commands will display the word "vulnerable" as a result of Bash executing the command <i><b>"echo vulnerable"</b></i>, which was embedded into the specially crafted environment variable named <i><b>"x"</b></i>.<sup class="reference" id="cite_ref-zdnet-betterbash_8-1"><a href="#cite_note-zdnet-betterbash-8">[8]</a></sup><sup class="reference" id="cite_ref-33"><a href="#cite_note-33">[33]</a></sup></p>
<h3><span class="mw-headline" id="CVE-2014-6277">CVE-2014-6277</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Shellshock_(software_bug)&amp;action=edit&amp;section=7" title="Edit section: CVE-2014-6277">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Discovered by <a href="/wiki/Micha%C5%82_Zalewski" title="Michał Zalewski">Michał Zalewski</a>.<sup class="reference" id="cite_ref-ITN-20140929_7-2"><a href="#cite_note-ITN-20140929-7">[7]</a></sup><sup class="reference" id="cite_ref-HH-20140928_30-1"><a href="#cite_note-HH-20140928-30">[30]</a></sup><sup class="reference" id="cite_ref-NIST-20140927_34-0"><a href="#cite_note-NIST-20140927-34">[34]</a></sup> this vulnerability relates to the parsing of function definitions in environment variables by Bash, and can cause a <a href="/wiki/Segmentation_fault" title="Segmentation fault">segfault</a>.<sup class="reference" id="cite_ref-PCW-20140929_35-0"><a href="#cite_note-PCW-20140929-35">[35]</a></sup></p>
<h3><span class="mw-headline" id="CVE-2014-6278">CVE-2014-6278</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Shellshock_(software_bug)&amp;action=edit&amp;section=8" title="Edit section: CVE-2014-6278">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Also discovered by <a href="/wiki/Micha%C5%82_Zalewski" title="Michał Zalewski">Michał Zalewski</a>.<sup class="reference" id="cite_ref-PCW-20140929_35-1"><a href="#cite_note-PCW-20140929-35">[35]</a></sup><sup class="reference" id="cite_ref-36"><a href="#cite_note-36">[36]</a></sup> this relates to the parsing of function definitions in environment variables by Bash.</p>
<h3><span class="mw-headline" id="CVE-2014-7169">CVE-2014-7169</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Shellshock_(software_bug)&amp;action=edit&amp;section=9" title="Edit section: CVE-2014-7169">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>On the same day the original vulnerability was published, Tavis Ormandy discovered this related bug<sup class="reference" id="cite_ref-qualys_23-1"><a href="#cite_note-qualys-23">[23]</a></sup> which is demonstrated in the following code:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
env <span class="nv">X</span><span class="o">=</span><span class="s1">'() { (a)=&gt;\'</span> bash -c <span class="s2">"echo date"</span><span class="p">;</span> cat <span class="nb">echo</span>
</pre></div>
<p>On a vulnerable system this would execute the command "date" unintentionally.<sup class="reference" id="cite_ref-qualys_23-2"><a href="#cite_note-qualys-23">[23]</a></sup></p>
<p>Here is an example of a system that has a patch for CVE-2014-6271 but not CVE-2014-7169:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="gp">$</span> <span class="nv">X</span><span class="o">=</span><span class="s1">'() { (a)=&gt;\'</span> bash -c <span class="s2">"echo date"</span>
<span class="go">bash: X: line 1: syntax error near unexpected token `='</span>
<span class="go">bash: X: line 1: `'</span>
<span class="go">bash: error importing function definition for `X'</span>
<span class="gp">$</span> cat <span class="nb">echo</span>
<span class="go">Fri Sep 26 01:37:16 UTC 2014</span>
</pre></div>
<p>The system displays syntax errors, notifying the user that CVE-2014-6271 has been prevented, but still writes a file named 'echo', into the working directory, containing the result of the 'date' call.</p>
<p>A system patched for both CVE-2014-6271 and CVE-2014-7169 will simply echo the word "date" and the file "echo" will <i>not</i> be created, as shown below:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="gp">$</span> <span class="nv">X</span><span class="o">=</span><span class="s1">'() { (a)=&gt;\'</span> bash -c <span class="s2">"echo date"</span>
<span class="go">date</span>
<span class="gp">$</span> cat <span class="nb">echo</span>
<span class="go">cat: echo: No such file or directory</span>
</pre></div>
<h3><span class="mw-headline" id="CVE-2014-7186">CVE-2014-7186</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Shellshock_(software_bug)&amp;action=edit&amp;section=10" title="Edit section: CVE-2014-7186">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Florian Weimer and Todd Sabin found this bug,<sup class="reference" id="cite_ref-zdnet-betterbash_8-2"><a href="#cite_note-zdnet-betterbash-8">[8]</a></sup><sup class="reference" id="cite_ref-lcamtuf-oct-1_31-1"><a href="#cite_note-lcamtuf-oct-1-31">[31]</a></sup> which relates to an <a href="/wiki/Buffer_overflow" title="Buffer overflow">out-of-bounds memory access error</a> in the Bash parser code.<sup class="reference" id="cite_ref-37"><a href="#cite_note-37">[37]</a></sup></p>
<p>An example of the vulnerability, which leverages the use of multiple "&lt;&lt;EOF" declarations:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
bash -c <span class="s1">'true &lt;&lt;EOF &lt;&lt;EOF &lt;&lt;EOF &lt;&lt;EOF &lt;&lt;EOF &lt;&lt;EOF &lt;&lt;EOF &lt;&lt;EOF &lt;&lt;EOF &lt;&lt;EOF &lt;&lt;EOF &lt;&lt;EOF &lt;&lt;EOF &lt;&lt;EOF'</span> <span class="o">||</span>
<span class="nb">echo</span> <span class="s2">"CVE-2014-7186 vulnerable, redir_stack"</span>
</pre></div>
<p>A vulnerable system will echo the text "CVE-2014-7186 vulnerable, redir_stack".</p>
<h3><span class="mw-headline" id="CVE-2014-7187">CVE-2014-7187</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Shellshock_(software_bug)&amp;action=edit&amp;section=11" title="Edit section: CVE-2014-7187">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Also found by Florian Weimer,<sup class="reference" id="cite_ref-zdnet-betterbash_8-3"><a href="#cite_note-zdnet-betterbash-8">[8]</a></sup> this is an <a href="/wiki/Off-by-one_error" title="Off-by-one error">off-by-one error</a> in the Bash parser code, allowing out-of-bounds memory access.<sup class="reference" id="cite_ref-38"><a href="#cite_note-38">[38]</a></sup></p>
<p>An example of the vulnerability, which leverages the use of multiple "done" declarations:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="o">(</span><span class="k">for</span> x in <span class="o">{</span>1..200<span class="o">}</span> <span class="p">;</span> <span class="k">do</span> <span class="nb">echo</span> <span class="s2">"for x</span>$<span class="s2">x in ; do :"</span><span class="p">;</span> <span class="k">done</span><span class="p">;</span> <span class="k">for</span> x in <span class="o">{</span>1..200<span class="o">}</span> <span class="p">;</span> <span class="k">do</span> <span class="nb">echo</span> <span class="k">done</span> <span class="p">;</span> <span class="k">done</span><span class="o">)</span> <span class="p">|</span> bash <span class="o">||</span>
<span class="nb">echo</span> <span class="s2">"CVE-2014-7187 vulnerable, word_lineno"</span>
</pre></div>
<p>A vulnerable system will echo the text "CVE-2014-7187 vulnerable, word_lineno". This test requires a shell that supports <a class="mw-redirect" href="/wiki/Brace_expansion" title="Brace expansion">brace expansion</a>.<sup class="reference" id="cite_ref-39"><a href="#cite_note-39">[39]</a></sup></p>
<h2><span class="mw-headline" id="Patches">Patches</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Shellshock_(software_bug)&amp;action=edit&amp;section=12" title="Edit section: Patches">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Until 24 September 2014, Bash maintainer Chet Ramey provided a patch version bash43-025 of Bash 4.3 addressing CVE-2014-6271,<sup class="reference" id="cite_ref-40"><a href="#cite_note-40">[40]</a></sup> which was already packaged by distribution maintainers. On 24 September, bash43-026 followed, addressing CVE-2014-7169.<sup class="reference" id="cite_ref-41"><a href="#cite_note-41">[41]</a></sup> Then CVE-2014-7186 was discovered. Florian Weimer from <a href="/wiki/Red_Hat" title="Red Hat">Red Hat</a> posted some patch code for this "unofficially" on 25 September,<sup class="reference" id="cite_ref-42"><a href="#cite_note-42">[42]</a></sup> which Ramey incorporated into Bash as bash43-027.<sup class="reference" id="cite_ref-43"><a href="#cite_note-43">[43]</a></sup><sup class="reference" id="cite_ref-44"><a href="#cite_note-44">[44]</a></sup>—These patches provided <i>code</i> only, helpful only for those who know how to <a class="mw-redirect" href="/wiki/Compile" title="Compile">compile</a> ("<a href="/wiki/Software_build" title="Software build">rebuild</a>") a new Bash <a class="mw-redirect" href="/wiki/Binary_executable" title="Binary executable">binary executable</a> file from the patch file and remaining source code files.</p>
<p>The next day, Red Hat officially presented according updates for <a href="/wiki/Red_Hat_Enterprise_Linux" title="Red Hat Enterprise Linux">Red Hat Enterprise Linux</a>,<sup class="reference" id="cite_ref-45"><a href="#cite_note-45">[45]</a></sup><sup class="reference" id="cite_ref-rh-art_46-0"><a href="#cite_note-rh-art-46">[46]</a></sup> after another day for <a href="/wiki/Fedora_(operating_system)" title="Fedora (operating system)">Fedora 21</a>.<sup class="reference" id="cite_ref-47"><a href="#cite_note-47">[47]</a></sup> <a class="mw-redirect" href="/wiki/Canonical_Ltd." title="Canonical Ltd.">Canonical Ltd.</a> presented updates for its <a href="/wiki/Ubuntu_(operating_system)" title="Ubuntu (operating system)">Ubuntu</a> <i>Long Term Support</i> versions on Saturday, 27 September;<sup class="reference" id="cite_ref-48"><a href="#cite_note-48">[48]</a></sup> on Sunday, there were updates for <a class="mw-redirect" href="/wiki/SUSE_Linux_Enterprise" title="SUSE Linux Enterprise">SUSE Linux Enterprise</a>.<sup class="reference" id="cite_ref-49"><a href="#cite_note-49">[49]</a></sup> The following Monday and Tuesday at the end of the month, <a class="mw-redirect" href="/wiki/Apple_OS_X" title="Apple OS X">Apple OS X</a> updates appeared.<sup class="reference" id="cite_ref-50"><a href="#cite_note-50">[50]</a></sup><sup class="reference" id="cite_ref-51"><a href="#cite_note-51">[51]</a></sup></p>
<p>On 1 October 2014, <a href="/wiki/Micha%C5%82_Zalewski" title="Michał Zalewski">Michał Zalewski</a> from <a class="mw-redirect" href="/wiki/Google_Inc." title="Google Inc.">Google Inc.</a> finally stated that Weimer's code and bash43-027 had fixed not only the first three bugs but even the remaining three that were published after bash43-027, including his own two discoveries.<sup class="reference" id="cite_ref-lcamtuf-oct-1_31-2"><a href="#cite_note-lcamtuf-oct-1-31">[31]</a></sup> This means that after the earlier distribution updates, no other updates have been required to cover all the six issues.<sup class="reference" id="cite_ref-rh-art_46-1"><a href="#cite_note-rh-art-46">[46]</a></sup></p>
<p>All of them have also been covered for the <a href="#IBM_HMC_restricted_shell">IBM <i>Hardware Management Console</i></a>.<sup class="reference" id="cite_ref-ibm-hmc_27-1"><a href="#cite_note-ibm-hmc-27">[27]</a></sup></p>
