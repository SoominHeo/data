<p><b>Windows Installer</b> (previously known as <b>Microsoft Installer</b><sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup>) is a <a class="mw-redirect" href="/wiki/Software_component" title="Software component">software component</a> and <a href="/wiki/Application_programming_interface" title="Application programming interface">application programming interface</a> (API) of <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a> used for the installation, maintenance, and removal of software. The installation information, and optionally the files themselves, are packaged in <a class="mw-redirect" href="/wiki/Package_(package_management_system)" title="Package (package management system)"><i>installation packages</i></a>, loosely <a href="/wiki/Relational_database" title="Relational database">relational databases</a> structured as <a href="/wiki/COM_Structured_Storage" title="COM Structured Storage">COM Structured Storages</a> and commonly known as "MSI files", from their default <a href="/wiki/Filename_extension" title="Filename extension">filename extensions</a>. Windows Installer contains significant changes from its predecessor, Setup API. New features include a <a class="mw-redirect" href="/wiki/GUI" title="GUI">GUI</a> <a href="/wiki/Software_framework" title="Software framework">framework</a> and automatic generation of the <a class="mw-redirect" href="/wiki/Uninstallation" title="Uninstallation">uninstallation</a> sequence. Windows Installer is positioned as an alternative to stand-alone executable installer frameworks such as older versions of <a href="/wiki/InstallShield" title="InstallShield">InstallShield</a> and <a href="/wiki/Nullsoft_Scriptable_Install_System" title="Nullsoft Scriptable Install System">NSIS</a>.</p>
<p>Before the introduction of <a href="/wiki/Windows_Store" title="Windows Store">Windows Store</a>, Microsoft encouraged third parties to use Windows Installer as the basis for installation frameworks, so that they synchronize correctly with other installers and keep the internal database of installed products consistent. Important features such as <a href="/wiki/Rollback_(data_management)" title="Rollback (data management)">rollback</a> and <a href="/wiki/Version_control" title="Version control">versioning</a> depend on a consistent internal database for reliable operation. Furthermore, Windows Installer facilitates the <a href="/wiki/Principle_of_least_privilege" title="Principle of least privilege">principle of least privilege</a> by performing software installations by proxy for unprivileged users.</p>

<h2><span class="mw-headline" id="Logical_structure_of_packages">Logical structure of packages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Installer&amp;action=edit&amp;section=1" title="Edit section: Logical structure of packages">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>A package describes the installation of one or more full <i>products</i> and is universally identified by a <a class="mw-redirect" href="/wiki/GUID" title="GUID">GUID</a>. A product is made up of <i>components</i>, grouped into <i>features</i>. Windows Installer does not handle dependencies between products.</p>
<h3><span class="mw-headline" id="Products">Products</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Installer&amp;action=edit&amp;section=2" title="Edit section: Products">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>A single, installed, working program (or set of programs) is a <i>product</i>. A product is identified by a unique GUID (the ProductCode property) providing an authoritative identity throughout the world. The GUID, in combination with the version number (ProductVersion property), allows for release management of the product's files and registry keys.</p>
<p>A <i>package</i> includes the package logic and other <a href="/wiki/Metadata" title="Metadata">metadata</a> that relates to how the package executes when running. For example, changing an <a class="mw-redirect" href="/wiki/EXE_file" title="EXE file">EXE file</a> in the product may require the ProductCode or ProductVersion to be changed for the release management. However, merely changing or adding a launch condition (with the product remaining exactly the same as the previous version) would still require the PackageCode to change for release management of the MSI file itself.</p>
<h3><span class="mw-headline" id="Features">Features</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Installer&amp;action=edit&amp;section=3" title="Edit section: Features">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>A <i>feature</i> is a hierarchical group of components. A feature may contain any number of components and other sub-features. Smaller packages can consist of a single feature. More complex installers may display a "custom setup" dialog box, from which the user can select which features to install or remove.</p>
<p>The package author defines the product features. A <a href="/wiki/Word_processor" title="Word processor">word processor</a>, for example, might place the program's core file into one feature, and the program's help files, optional spelling checker and stationery modules into additional features.</p>
<h3><span class="mw-headline" id="Components">Components</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Installer&amp;action=edit&amp;section=4" title="Edit section: Components">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>A <i>component</i> is the basic unit of a product. Each component is treated by Windows Installer as a unit. The installer cannot install just part of a component.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup> Components can contain program <a href="/wiki/Computer_file" title="Computer file">files</a>, <a class="mw-redirect" href="/wiki/Directory_(file_systems)" title="Directory (file systems)">folders</a>, <a href="/wiki/Component_Object_Model" title="Component Object Model">COM</a> components, <a class="mw-redirect" href="/wiki/Windows_registry" title="Windows registry">registry</a> keys, and <a class="mw-redirect" href="/wiki/Shortcut_file" title="Shortcut file">shortcuts</a>. The user does not directly interact with components.</p>
<p>Components are identified globally by GUIDs; thus the same component can be shared among several features of the same package or multiple packages, ideally through the use of <a href="/wiki/Merge_Module" title="Merge Module">Merge Modules</a>.</p>
<h3><span class="mw-headline" id="Key_paths">Key paths</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Installer&amp;action=edit&amp;section=5" title="Edit section: Key paths">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>A <i>key path</i> is a specific file, registry key, or <a href="/wiki/Open_Database_Connectivity" title="Open Database Connectivity">ODBC</a> data source that the package author specifies as critical for a given component. Because a file is the most common type of key path, the term <i>key file</i> is commonly used. A component can contain at most one key path; if a component has no explicit key path, the component's destination folder is taken to be the key path. When an MSI-based program is launched, Windows Installer checks the existence of key paths. If there is a mismatch between the current system state and the value specified in the MSI package (e.g., a key file is missing), the related feature is re-installed. This process is known as <i>self-healing</i> or <i>self-repair</i>. No two components should use the same key path.</p>
<h2><span class="mw-headline" id="Setup_phases">Setup phases</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Installer&amp;action=edit&amp;section=6" title="Edit section: Setup phases">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="User_interface">User interface</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Installer&amp;action=edit&amp;section=7" title="Edit section: User interface">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The <a href="/wiki/User_interface" title="User interface">user interface</a> phase typically queries the target system, displays an installation <a href="/wiki/Wizard_(software)" title="Wizard (software)">wizard</a> and enables the user to change various options that will affect the installation.</p>
<p>However, the user interface sequence should not make any changes to the system, for the following reasons:</p>
<ol>
<li>A user may install or uninstall an MSI package in passive mode or silent mode, bypassing this phase entirely. (Passive mode bypasses the user interface phase but shows a graphical progress bar. Silent mode shows nothing.) During installation, information gathered in this phase may be supplied beforehand through the <a href="/wiki/Command-line_interface" title="Command-line interface">command-line interface</a>.</li>
<li>The user interface sequence runs with user privileges, and not with the elevated privileges required during installation.</li>
</ol>
<h3><span class="mw-headline" id="Execute">Execute</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Installer&amp;action=edit&amp;section=8" title="Edit section: Execute">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>When the user clicks the "Install" button in a typical MSI installation wizard, installation proceeds to the Execute phase, in which software components are actually installed. The Execute phase makes system changes, but it does not display any user interface elements.</p>
<p>The Execute phase happens in two steps:<sup class="noprint Inline-Template" style="margin-left:0.1em; white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Please_clarify" title="Wikipedia:Please clarify"><span title="The text near this tag may need clarification or removal of jargon. (August 2015)">clarification needed</span></a> Two mode or two steps? Do both run?</i>]</sup></p>
<ul>
<li><i>Immediate mode.</i> In this phase, Windows Installer receives instructions, either from a user or an application, to install or uninstall features of a product. The requests cause the execution of <i>sequences</i> of <i>actions</i>, which query the installation database to build an internal <i>script</i> describing the execution phase in detail.</li>
<li><i>Deferred mode.</i> In this phase, the script built in immediate mode is executed in the context of the privileged Windows Installer <a class="mw-redirect" href="/wiki/Windows_Service" title="Windows Service">service</a>. The script must be executed by a privileged account because of the heterogeneity of the scenarios in which a setup operation is initiated. For example, elevated privileges are necessary to serve on-demand installation requests from non-privileged users. (To run with elevated privileges, however, the package must be deployed by a local administrator or advertised by a system administrator using <a href="/wiki/Group_Policy" title="Group Policy">Group Policy</a>.)</li>
</ul>
<h3><span class="mw-headline" id="Rollback">Rollback</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Installer&amp;action=edit&amp;section=9" title="Edit section: Rollback">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>All installation operations are transactional.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> In other words, for each operation that Windows Installer performs, it generates an equivalent undo operation that would revert the change made to the system. In case any script action fails during deferred execution, or the operation is cancelled by the user, all the actions performed until that point are rolled back, restoring the system to its original state. Standard Windows Installer actions automatically write information into a rollback script; package authors who create custom actions that change the target system should also create corresponding rollback actions (as well as uninstall actions and uninstallation-rollback actions). As a design feature, if applied correctly this mechanism will also rollback a failed uninstall of an application to a good working state.</p>
<h2><span class="mw-headline" id="Other_features">Other features</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Installer&amp;action=edit&amp;section=10" title="Edit section: Other features">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Advertisement">Advertisement</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Installer&amp;action=edit&amp;section=11" title="Edit section: Advertisement">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Windows Installer can <i>advertise</i> a product rather than actually install it.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup> The product will appear installed to the user, but it will not actually be installed until it is run for the first time by triggering an entry point (by means of a <a href="/wiki/Start_menu" title="Start menu">Start menu</a> shortcut, by opening a document that the product is configured to handle, or by invoking an advertised <a href="/wiki/Component_Object_Model" title="Component Object Model">COM</a> class). A package can be advertised by an administrator using <a href="/wiki/Group_Policy" title="Group Policy">Group Policy</a> or other deployment mechanism, or by running the msiexec executable with the /jm (for per-machine advertisement) or /ju (for per-user advertisement) switch. Some MSI packages authored in InstallShield may prevent the use of these and other native MSI features.</p>
<p>The user must have administrator privileges to complete the advertised installation.</p>
<h3><span class="mw-headline" id="Installation_on_demand">Installation on demand</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Installer&amp;action=edit&amp;section=12" title="Edit section: Installation on demand">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Similar to advertising, it installs a <i>feature</i> as soon as the user tries to use it.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup></p>
<h3><span class="mw-headline" id="Administrative_installation">Administrative installation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Installer&amp;action=edit&amp;section=13" title="Edit section: Administrative installation">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>An administrative installation creates an uncompressed source image for a product, typically to be used for installing or running an application from a network location.<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> An administrative installation is not a typical installation, in that it does not create any shortcuts, register COM servers, create an Add or Remove Programs entry, and so on. Often an administrative installation enables a user to install the product in such a way that its features run from the uncompressed installation source.</p>
<p>Administrative installations are also useful when creating a Windows Installer patch, as this requires uncompressed images of the earlier and current versions of a product in order to compute binary file differences. An administrative installation is performed by running the msiexec executable with the /a switch.</p>
<h3><span class="mw-headline" id="Custom_actions">Custom actions</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Installer&amp;action=edit&amp;section=14" title="Edit section: Custom actions">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The developer of an installer package may write code to serve their own purpose, delivered as a DLL, EXE, VBScript or JavaScript. This can be executed during the installation sequences, including when the user clicks a button in the user interface, or during the InstallExecuteSequence. Custom Actions typically validate product license keys, or initialize more complex services. Developers should normally provide inverse custom actions for use during uninstall.</p>
<p>Msiexec provides a way to break after loading a specified custom action DLL but before invoking the action.<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup></p>
<h3><span class="mw-headline" id="Merge_modules_and_nested_executables">Merge modules and nested executables</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Installer&amp;action=edit&amp;section=15" title="Edit section: Merge modules and nested executables">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>A Windows Installer package may contain another package to be installed at the same time. These are ideally provided as a .msm file component, but may also be a separate executable program which will be unpacked from the installer package during the InstallExecuteSequence and can be run immediately. The file can then optionally be deleted before the end of the InstallExecuteSequence, and so is ideal for using with older installers.</p>
<h3><span class="mw-headline" id="Interoperability_with_Windows_features">Interoperability with Windows features</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Installer&amp;action=edit&amp;section=16" title="Edit section: Interoperability with Windows features">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Windows Installer 4.0, which was shipped with <a href="/wiki/Windows_Vista" title="Windows Vista">Windows Vista</a>, incorporates capabilities to take advantage of the <a href="/wiki/User_Account_Control" title="User Account Control">User Account Control</a> feature. If an application can install without elevated privileges, its MSI package can be marked as such, thus allowing install without prompting the user for Administrator credentials. Windows Installer also works in conjunction with the Restart Manager; when installing or updating an application or system component with "full" user interface mode, the user will be displayed a list of affected applications that can be shut down, and then restarted after files have been updated. Installer actions running in silent mode perform these application restarts automatically. System services and tray applications can also be restarted in this manner.</p>
<h2><span class="mw-headline" id="Developing_installer_packages">Developing installer packages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Installer&amp;action=edit&amp;section=17" title="Edit section: Developing installer packages">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Creating an installer package for a new application is not trivial. It is necessary to specify which files must be installed, to where and with what registry keys. Any non-standard operations can be done using Custom Actions, which are typically developed in <a href="/wiki/Dynamic-link_library" title="Dynamic-link library">DLLs</a>. There are a number of commercial and freeware products to assist in creating MSI packages, including <a class="mw-redirect" href="/wiki/Visual_Studio" title="Visual Studio">Visual Studio</a> (up to VS 2010<sup class="reference" id="cite_ref-vs_msi_drop_10-0"><a href="#cite_note-vs_msi_drop-10">[10]</a></sup>), <a href="/wiki/InstallShield" title="InstallShield">InstallShield</a> and <a href="/wiki/WiX" title="WiX">WiX</a>. To varying degrees, the user interface and behavior may be configured for use in less common situations such as unattended installation. Once prepared, an installer package is "compiled" by reading the instructions and files from the developer's local machine, and creating the .msi file.</p>
<p>The user interface (dialog boxes) presented at the start of installation can be changed or configured by the setup engineer developing a new installer. There is a limited language of buttons, text fields and labels which can be arranged in a sequence of dialogue boxes. An installer package should be capable of running without any UI, for what is called "unattended installation".</p>
<h3><span class="mw-headline" id="ICE_validation">ICE validation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Windows_Installer&amp;action=edit&amp;section=18" title="Edit section: ICE validation">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Microsoft provides a set of Internal Consistency Evaluators (ICE), that can be used to detect potential problems with an MSI database.<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup> The ICE rules are combined into CUB files, which are stripped-down MSI files containing custom actions that test the target MSI database's contents for validation warnings and errors. ICE validation can be performed with the Platform SDK tools Orca and msival2, or with validation tools that ship with the various authoring environments.</p>
<p>For example, some of the ICE rules are:</p>
<ul>
<li>ICE09: Validates that any component destined for the System folder is marked as being permanent.</li>
<li>ICE24: Validates that the product code, product version, and product language have appropriate formats.</li>
<li>ICE33: Validates that the Registry table is not used for data better suited for another table (Class, Extension, Verb, and so on).</li>
</ul>
<p>Addressing ICE validation warnings and errors is an important step in the release process.</p>
