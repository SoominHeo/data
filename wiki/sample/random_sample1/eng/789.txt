<p>In <a href="/wiki/Software_development" title="Software development">software development</a>, <b>GNU Automake</b> is a <a href="/wiki/Programming_tool" title="Programming tool">programming tool</a> to automate parts of the compilation process. It eases usual compilation problems. For example, it points to needed dependencies.</p>
<p>It automatically generates one or more <i>Makefile.in</i> from files called <i>Makefile.am</i>. Each <i>Makefile.am</i> contains, among other things, useful variable definitions for the compiled software, such as <a href="/wiki/Compiler" title="Compiler">compiler</a> and <a href="/wiki/Linker_(computing)" title="Linker (computing)">linker</a> flags, dependencies and their versions, etc. The generated <i>Makefile.ins</i> are portable and compliant with the Makefile conventions in the <a class="mw-redirect" href="/wiki/GNU_Coding_Standards" title="GNU Coding Standards">GNU Coding Standards</a>, and may be used by <a href="/wiki/Configure_script" title="Configure script">configure scripts</a> to generate a working <a href="/wiki/Makefile" title="Makefile">Makefile</a>.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup></p>
<p>The <a href="/wiki/Free_Software_Foundation" title="Free Software Foundation">Free Software Foundation</a> maintains <tt>automake</tt> as one of the <a href="/wiki/GNU" title="GNU">GNU</a> programs, and as part of the <a class="mw-redirect" href="/wiki/GNU_build_system" title="GNU build system">GNU build system</a>. It is used to build several GNU applications and libraries, such as <a href="/wiki/GTK%2B" title="GTK+">GTK+</a>,<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup> as well as non-GNU software such as <a href="/wiki/XCircuit" title="XCircuit">XCircuit</a>.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></p>

<h2><span class="mw-headline" id="Process">Process</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Automake&amp;action=edit&amp;section=1" title="Edit section: Process">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="thumb tright">
<div class="thumbinner" style="width:332px;"><a class="image" href="/wiki/File:Autoconf-automake-process.svg"><img alt="" class="thumbimage" data-file-height="1170" data-file-width="644" height="600" src="//upload.wikimedia.org/wikipedia/commons/thumb/8/84/Autoconf-automake-process.svg/330px-Autoconf-automake-process.svg.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/8/84/Autoconf-automake-process.svg/495px-Autoconf-automake-process.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/8/84/Autoconf-automake-process.svg/660px-Autoconf-automake-process.svg.png 2x" width="330"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Autoconf-automake-process.svg" title="Enlarge"></a></div>
Flow diagram of autoconf and automake</div>
</div>
</div>
<p>Automake aims to allow the programmer to write a makefile in a higher-level language, rather than having to write the whole makefile manually. In simple cases, it suffices to give:</p>
<ul>
<li>A line that declares the name of the program to build</li>
<li>A list of source files</li>
<li>A list of command-line options to be passed to the <a href="/wiki/Compiler" title="Compiler">compiler</a> (for example, in which directories header files will be found)</li>
<li>A list of command-line options to be passed to the <a href="/wiki/Linker_(computing)" title="Linker (computing)">linker</a> (which libraries the program needs and in what directories they are to be found)</li>
</ul>
<p>From this information, Automake generates a makefile that allows the user to:</p>
<ul>
<li>Compile the program</li>
<li>Clean (i.e. remove the files resulting from the compilation)</li>
<li>Install the program in standard directories</li>
<li>Uninstall the program from where it was installed</li>
<li>Create a source distribution archive (commonly called a <i>tarball</i>)</li>
<li>Test that this archive is self-sufficient, and in particular that the program can be compiled in a directory other than the one where the sources are deployed</li>
</ul>
<p>The makefiles produced follow the <a class="mw-redirect" href="/wiki/GNU_Coding_Standards" title="GNU Coding Standards">GNU Coding Standards</a>.</p>
<p>Automake also takes care of automatically generating<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> the <a class="mw-redirect" href="/wiki/Coupling_(computer_science)" title="Coupling (computer science)">dependency</a> information, so that when a source file is modified, the next invocation of the make command will know which source files need to be recompiled. If the compiler allows it, Automake tries to make the dependency system dynamic: whenever a source file is compiled, that file's dependencies are updated by asking the compiler to regenerate the file's dependency list. In other words, dependency tracking is a side effect of the compilation process.</p>
<p>This attempts to avoid the problem with some static dependency systems, where the dependencies are detected only once when the programmer starts working on the project.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup> In such a case, if a source file gains a new dependency (e.g. if the programmer adds a new #include directive in a <a href="/wiki/C_(programming_language)" title="C (programming language)">C</a> source file), then a discrepancy is introduced between the real dependencies and those that are used by the compilation system. The programmer should then regenerate the dependencies, but runs the risk of forgetting to do so.</p>
<p>In the general case, automake generates dependencies via the bundled <a class="new" href="/w/index.php?title=Depcomp&amp;action=edit&amp;redlink=1" title="Depcomp (page does not exist)">depcomp</a> script, which will invoke the compiler appropriately or fall back to <a href="/wiki/Makedepend" title="Makedepend">makedepend</a>. If the compiler is a sufficiently recent version of <a href="/wiki/GNU_Compiler_Collection" title="GNU Compiler Collection">gcc</a>, however, automake will inline the dependency generation code to call gcc directly.</p>
<p>Automake can also help with the compilation of libraries by automatically generating makefiles that will invoke GNU <a class="mw-redirect" href="/wiki/Libtool" title="Libtool">Libtool</a>. The programmer is thus exempted from having to know how to call Libtool directly, and the project benefits from the use of a portable library creation tool.</p>
<p>Automake has been criticised for being complex compared to other tools like <a class="mw-redirect" href="/wiki/Cmake" title="Cmake">cmake</a> and plain old <a class="mw-redirect" href="/wiki/Make_file" title="Make file">make files</a>.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (December 2016)">citation needed</span></a></i>]</sup></p>
<h2><span class="mw-headline" id="Design">Design</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Automake&amp;action=edit&amp;section=2" title="Edit section: Design">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Automake is written in <a href="/wiki/Perl" title="Perl">Perl</a> and must be used with <a class="mw-redirect" href="/wiki/GNU_Autoconf" title="GNU Autoconf">GNU Autoconf</a>. Automake contains the following commands:</p>
<ul>
<li><i>aclocal</i></li>
<li><i>automake</i></li>
</ul>
<p><i>aclocal</i>, however, is a general-purpose program that can be useful to autoconf users. The <a href="/wiki/GNU_Compiler_Collection" title="GNU Compiler Collection">GNU Compiler Collection</a>, for example, uses <i>aclocal</i> even though its makefile is hand written.</p>
<p>Like Autoconf, Automake is not entirely <a class="mw-redirect" href="/wiki/Backward_compatible" title="Backward compatible">backward compatible</a>. For example, a project created with automake 1.13 will not necessarily work with automake 1.14.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup> This can require complex projects to include multiple versions.</p>
