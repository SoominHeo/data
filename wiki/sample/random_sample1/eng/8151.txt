<p>In <a href="/wiki/Computing" title="Computing">computing</a>, <b>scheduling</b> is the method by which work specified by some means is assigned to resources that complete the work. The work may be virtual computation elements such as <a class="mw-redirect" href="/wiki/Thread_(computer_science)" title="Thread (computer science)">threads</a>, <a href="/wiki/Process_(computing)" title="Process (computing)">processes</a> or data <a class="mw-redirect" href="/wiki/Flow_(computer_networking)" title="Flow (computer networking)">flows</a>, which are in turn scheduled onto hardware resources such as <a href="/wiki/Central_processing_unit" title="Central processing unit">processors</a>, network links or <a href="/wiki/Expansion_card" title="Expansion card">expansion cards</a>.</p>
<p>A scheduler is what carries out the scheduling activity. Schedulers are often implemented so they keep all computer resources busy (as in <a href="/wiki/Load_balancing_(computing)" title="Load balancing (computing)">load balancing</a>), allow multiple users to share system resources effectively, or to achieve a target <a href="/wiki/Quality_of_service" title="Quality of service">quality of service</a>. Scheduling is fundamental to computation itself, and an intrinsic part of the <a href="/wiki/Execution_model" title="Execution model">execution model</a> of a computer system; the concept of scheduling makes it possible to have <a href="/wiki/Computer_multitasking" title="Computer multitasking">computer multitasking</a> with a single <a href="/wiki/Central_processing_unit" title="Central processing unit">central processing unit</a> (CPU).</p>
<p>A scheduler may aim at one of many goals, for example, maximizing <i><a href="/wiki/Throughput" title="Throughput">throughput</a></i> (the total amount of work completed per time unit), minimizing <i><a href="/wiki/Responsiveness" title="Responsiveness">response time</a></i> (time from work becoming enabled until the first point it begins execution on resources), or minimizing <i><a href="/wiki/Latency_(engineering)" title="Latency (engineering)">latency</a></i> (the time between work becoming enabled and its subsequent completion),<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> maximizing <i>fairness</i> (equal CPU time to each process, or more generally appropriate times according to the priority and workload of each process). In practice, these goals often conflict (e.g. throughput versus latency), thus a scheduler will implement a suitable compromise. Preference is given to any one of the concerns mentioned above, depending upon the user's needs and objectives.</p>
<p>In <a href="/wiki/Real-time_computing" title="Real-time computing">real-time</a> environments, such as <a href="/wiki/Embedded_system" title="Embedded system">embedded systems</a> for <a href="/wiki/Automatic_control" title="Automatic control">automatic control</a> in industry (for example <a href="/wiki/Robotics" title="Robotics">robotics</a>), the scheduler also must ensure that processes can meet <a href="/wiki/Time_limit" title="Time limit">deadlines</a>; this is crucial for keeping the system stable. Scheduled tasks can also be distributed to remote devices across a network and <a class="mw-redirect" href="/wiki/Device_Management" title="Device Management">managed</a> through an administrative back end.</p>

<h2><span class="mw-headline" id="Types_of_operating_system_schedulers"><span id="SCHEDULER"></span>Types of operating system schedulers</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=1" title="Edit section: Types of operating system schedulers">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="hatnote" role="note">See also: <a href="/wiki/Network_scheduler" title="Network scheduler">Network scheduler</a>, <a href="/wiki/I/O_scheduling" title="I/O scheduling">I/O scheduling</a>, and <a href="/wiki/Job_scheduler" title="Job scheduler">Job scheduler</a></div>
<p>The scheduler is an operating system module that selects the next jobs to be admitted into the system and the next process to run. Operating systems may feature up to three distinct scheduler types: a <i>long-term scheduler</i> (also known as an admission scheduler or high-level scheduler), a <i>mid-term or medium-term scheduler</i>, and a <i>short-term scheduler</i>. The names suggest the relative frequency with which their functions are performed.</p>
<h3><span class="mw-headline" id="Process_scheduler"><span id="PROCESS"></span>Process scheduler</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=2" title="Edit section: Process scheduler">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The process scheduler is a part of the operating system that decides which process runs at a certain point in time. It usually has the ability to pause a running process, move it to the back of the running queue and start a new process; such a scheduler is known as <i><a href="/wiki/Preemption_(computing)" title="Preemption (computing)">preemptive</a> scheduler</i>, otherwise it is a <i><a class="mw-redirect" href="/wiki/Nonpreemptive_multitasking" title="Nonpreemptive multitasking">cooperative</a> scheduler</i>.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup></p>
<h4><span class="mw-headline" id="Long-term_scheduling"><span id="LONG-TERM"></span>Long-term scheduling</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=3" title="Edit section: Long-term scheduling">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>The <i>long-term scheduler</i>, or <i>admission scheduler</i>, decides which jobs or processes are to be admitted to the ready queue (in main memory); that is, when an attempt is made to execute a program, its admission to the set of currently executing processes is either authorized or delayed by the long-term scheduler. Thus, this scheduler dictates what processes are to run on a system, and the degree of concurrency to be supported at any one time –  whether many or few processes are to be executed concurrently, and how the split between I/O-intensive and CPU-intensive processes is to be handled. The long-term scheduler is responsible for controlling the degree of multiprogramming.</p>
<p>In general, most processes can be described as either <a class="mw-redirect" href="/wiki/I/O-bound" title="I/O-bound">I/O-bound</a> or <a href="/wiki/CPU-bound" title="CPU-bound">CPU-bound</a>. An I/O-bound process is one that spends more of its time doing I/O than it spends doing computations. A CPU-bound process, in contrast, generates I/O requests infrequently, using more of its time doing computations. It is important that a long-term scheduler selects a good process mix of I/O-bound and CPU-bound processes. If all processes are I/O-bound, the ready queue will almost always be empty, and the short-term scheduler will have little to do. On the other hand, if all processes are CPU-bound, the I/O waiting queue will almost always be empty, devices will go unused, and again the system will be unbalanced. The system with the best performance will thus have a combination of CPU-bound and I/O-bound processes. In modern operating systems, this is used to make sure that real-time processes get enough CPU time to finish their tasks.<sup class="reference" id="cite_ref-galvin_3-0"><a href="#cite_note-galvin-3">[3]</a></sup></p>
<p>Long-term scheduling is also important in large-scale systems such as <a href="/wiki/Batch_processing" title="Batch processing">batch processing</a> systems, <a href="/wiki/Computer_cluster" title="Computer cluster">computer clusters</a>, <a href="/wiki/Supercomputer" title="Supercomputer">supercomputers</a>, and <a href="/wiki/Render_farm" title="Render farm">render farms</a>. For example, in <a href="/wiki/Concurrent_computing" title="Concurrent computing">concurrent systems</a>, <a href="/wiki/Coscheduling" title="Coscheduling">coscheduling</a> of interacting processes is often required to prevent them from blocking due to waiting on each other. In these cases, special-purpose <a href="/wiki/Job_scheduler" title="Job scheduler">job scheduler</a> software is typically used to assist these functions, in addition to any underlying admission scheduling support in the operating system.</p>
<h4><span class="mw-headline" id="Medium-term_scheduling"><span id="MEDIUM-TERM"></span>Medium-term scheduling</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=4" title="Edit section: Medium-term scheduling">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>The <i>medium-term scheduler</i> temporarily removes processes from main memory and places them in secondary memory (such as a <a href="/wiki/Hard_disk_drive" title="Hard disk drive">hard disk drive</a>) or vice versa, which is commonly referred to as "swapping out" or "swapping in" (also incorrectly as "<a href="/wiki/Paging" title="Paging">paging</a> out" or "paging in"). The medium-term scheduler may decide to swap out a process which has not been active for some time, or a process which has a low priority, or a process which is <a href="/wiki/Page_fault" title="Page fault">page faulting</a> frequently, or a process which is taking up a large amount of memory in order to free up main memory for other processes, swapping the process back in later when more memory is available, or when the process has been unblocked and is no longer waiting for a resource. [Stallings, 396] [Stallings, 370]</p>
<p>In many systems today (those that support mapping virtual address space to secondary storage other than the swap file), the medium-term scheduler may actually perform the role of the long-term scheduler, by treating binaries as "swapped out processes" upon their execution. In this way, when a segment of the binary is required it can be swapped in on demand, or "lazy loaded". [Stallings, 394]</p>
<h4><span class="mw-headline" id="Short-term_scheduling"><span id="SHORT-TERM"></span>Short-term scheduling</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=5" title="Edit section: Short-term scheduling">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>The <i>short-term scheduler</i> (also known as the <i>CPU scheduler</i>) decides which of the ready, in-memory processes is to be executed (allocated a CPU) after a clock <a href="/wiki/Interrupt" title="Interrupt">interrupt</a>, an I/O interrupt, an operating <a href="/wiki/System_call" title="System call">system call</a> or another form of <a href="/wiki/Signal_programming" title="Signal programming">signal</a>. Thus the short-term scheduler makes scheduling decisions much more frequently than the long-term or mid-term schedulers – a scheduling decision will at a minimum have to be made after every time slice, and these are very short. This scheduler can be <a href="/wiki/Preemption_(computing)" title="Preemption (computing)">preemptive</a>, implying that it is capable of forcibly removing processes from a CPU when it decides to allocate that CPU to another process, or non-preemptive (also known as "voluntary" or "co-operative"), in which case the scheduler is unable to "force" processes off the CPU.</p>
<p>A preemptive scheduler relies upon a <a href="/wiki/Programmable_interval_timer" title="Programmable interval timer">programmable interval timer</a> which invokes an <a href="/wiki/Interrupt_handler" title="Interrupt handler">interrupt handler</a> that runs in <a class="mw-redirect" href="/wiki/Kernel_mode" title="Kernel mode">kernel mode</a> and implements the scheduling function.</p>
<h4><span class="mw-headline" id="Dispatcher"><span id="DISPATCH-LATENCY"></span>Dispatcher</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=6" title="Edit section: Dispatcher">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Another component that is involved in the CPU-scheduling function is the dispatcher, which is the module that gives control of the CPU to the process selected by the short-term scheduler. It receives control in kernel mode as the result of an interrupt or system call. The functions of a dispatcher involve the following:</p>
<ul>
<li><a href="/wiki/Context_switch" title="Context switch">Context switches</a>, in which the dispatcher saves the <a href="/wiki/State_(computer_science)" title="State (computer science)">state</a> (also known as <a href="/wiki/Context_(computing)" title="Context (computing)">context</a>) of the <a href="/wiki/Process_(computing)" title="Process (computing)">process</a> or <a href="/wiki/Thread_(computing)" title="Thread (computing)">thread</a> that was previously running; the dispatcher then loads the initial or previously saved state of the new process.</li>
<li>Switching to user mode.</li>
<li>Jumping to the proper location in the user program to restart that program indicated by its new state.</li>
</ul>
<p>The dispatcher should be as fast as possible, since it is invoked during every process switch. During the context switches, the processor is virtually idle for a fraction of time, thus unnecessary context switches should be avoided. The time it takes for the dispatcher to stop one process and start another is known as the <i>dispatch latency</i>.<sup class="reference" id="cite_ref-galvin_3-1"><a href="#cite_note-galvin-3">[3]</a></sup><sup class="reference" style="white-space:nowrap;">:155</sup></p>
<h2><span class="mw-headline" id="Scheduling_disciplines">Scheduling disciplines</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=7" title="Edit section: Scheduling disciplines">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Scheduling disciplines are algorithms used for distributing resources among parties which simultaneously and asynchronously request them. Scheduling disciplines are used in <a href="/wiki/Router_(computing)" title="Router (computing)">routers</a> (to handle packet traffic) as well as in <a href="/wiki/Operating_system" title="Operating system">operating systems</a> (to share <a href="/wiki/CPU_time" title="CPU time">CPU time</a> among both <a class="mw-redirect" href="/wiki/Thread_(computer_science)" title="Thread (computer science)">threads</a> and <a href="/wiki/Process_(computing)" title="Process (computing)">processes</a>), disk drives (<a href="/wiki/I/O_scheduling" title="I/O scheduling">I/O scheduling</a>), printers (<a class="mw-redirect" href="/wiki/Print_spooler" title="Print spooler">print spooler</a>), most embedded systems, etc.</p>
<p>The main purposes of scheduling algorithms are to minimize <a class="mw-redirect" href="/wiki/Resource_starvation" title="Resource starvation">resource starvation</a> and to ensure fairness amongst the parties utilizing the resources. Scheduling deals with the problem of deciding which of the outstanding requests is to be allocated resources. There are many different scheduling algorithms. In this section, we introduce several of them.</p>
<p>In <a class="mw-redirect" href="/wiki/Packet-switched" title="Packet-switched">packet-switched</a> <a class="mw-redirect" href="/wiki/Computer_networks" title="Computer networks">computer networks</a> and other <a class="mw-redirect" href="/wiki/Statistical_multiplexing" title="Statistical multiplexing">statistical multiplexing</a>, the notion of a <b>scheduling algorithm</b> is used as an alternative to <a href="/wiki/FIFO_(computing_and_electronics)" title="FIFO (computing and electronics)">first-come first-served</a> queuing of data packets.</p>
<p>The simplest best-effort scheduling algorithms are <a href="/wiki/Round-robin_scheduling" title="Round-robin scheduling">round-robin</a>, <a href="/wiki/Fair_queuing" title="Fair queuing">fair queuing</a> (a <a class="mw-redirect" href="/wiki/Max-min_fair" title="Max-min fair">max-min fair</a> scheduling algorithm), <a href="/wiki/Proportionally_fair" title="Proportionally fair">proportionally fair</a> scheduling and <a href="/wiki/Maximum_throughput_scheduling" title="Maximum throughput scheduling">maximum throughput</a>. If differentiated or guaranteed <a href="/wiki/Quality_of_service" title="Quality of service">quality of service</a> is offered, as opposed to best-effort communication, <a class="mw-redirect" href="/wiki/Weighted_fair_queuing" title="Weighted fair queuing">weighted fair queuing</a> may be utilized.</p>
<p>In advanced packet radio wireless networks such as <a class="mw-redirect" href="/wiki/HSDPA" title="HSDPA">HSDPA</a> (High-Speed Downlink Packet Access ) <a class="mw-redirect" href="/wiki/3.5G" title="3.5G">3.5G</a> cellular system, <b>channel-dependent scheduling</b> may be used to take advantage of <a href="/wiki/Channel_state_information" title="Channel state information">channel state information</a>. If the channel conditions are favourable, the <a href="/wiki/Throughput" title="Throughput">throughput</a> and <a class="mw-redirect" href="/wiki/System_spectral_efficiency" title="System spectral efficiency">system spectral efficiency</a> may be increased. In even more advanced systems such as <a href="/wiki/LTE_Advanced" title="LTE Advanced">LTE</a>, the scheduling is combined by channel-dependent packet-by-packet <a class="mw-redirect" href="/wiki/Dynamic_channel_allocation" title="Dynamic channel allocation">dynamic channel allocation</a>, or by assigning <a class="mw-redirect" href="/wiki/OFDMA" title="OFDMA">OFDMA</a> multi-carriers or other <a class="mw-redirect" href="/wiki/Frequency-domain_equalization" title="Frequency-domain equalization">frequency-domain equalization</a> components to the users that best can utilize them.<sup class="reference" id="cite_ref-Miao_4-0"><a href="#cite_note-Miao-4">[4]</a></sup></p>
<h3><span class="mw-headline" id="First_in.2C_first_out">First in, first out</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=8" title="Edit section: First in, first out">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/FIFO_(computing_and_electronics)" title="FIFO (computing and electronics)">FIFO (computing and electronics)</a></div>
<div class="thumb tright">
<div class="thumbinner" style="width:402px;"><a class="image" href="/wiki/File:Thread_pool.svg"><img alt="" class="thumbimage" data-file-height="300" data-file-width="580" height="207" src="//upload.wikimedia.org/wikipedia/commons/thumb/0/0c/Thread_pool.svg/400px-Thread_pool.svg.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/0/0c/Thread_pool.svg/600px-Thread_pool.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/0/0c/Thread_pool.svg/800px-Thread_pool.svg.png 2x" width="400"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/File:Thread_pool.svg" title="Enlarge"></a></div>
A sample <a href="/wiki/Thread_pool" title="Thread pool">thread pool</a> (green boxes) with a queue (FIFO) of waiting tasks (blue) and a queue of completed tasks (yellow)</div>
</div>
</div>
<p><i>First in, first out</i> (<a href="/wiki/FIFO_(computing_and_electronics)" title="FIFO (computing and electronics)">FIFO</a>), also known as <i>first come, first served</i> (FCFS), is the simplest scheduling algorithm. FIFO simply queues processes in the order that they arrive in the ready queue. This is commonly used for a <b><span id="task_queue">task queue</span></b>, for example as illustrated in this section.</p>
<ul>
<li>Since context switches only occur upon process termination, and no reorganization of the process queue is required, scheduling overhead is minimal.</li>
<li>Throughput can be low, since long processes can hold the CPU</li>
<li>Turnaround time, waiting time and response time can be high for the same reasons above</li>
<li>No prioritization occurs, thus this system has trouble meeting process deadlines.</li>
<li>The lack of prioritization means that as long as every process eventually completes, there is no starvation. In an environment where some processes might not complete, there can be starvation.</li>
<li>It is based on queuing.</li>
</ul>
<h3><span class="mw-headline" id="Earliest_deadline_first">Earliest deadline first</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=9" title="Edit section: Earliest deadline first">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Earliest_deadline_first_scheduling" title="Earliest deadline first scheduling">Earliest deadline first scheduling</a></div>
<p>Earliest deadline first (EDF) or <i>least time to go</i> is a dynamic scheduling algorithm used in real-time operating systems to place processes in a priority queue. Whenever a scheduling event occurs (a task finishes, new task is released, etc.), the queue will be searched for the process closest to its deadline, which will be the next to be scheduled for execution.</p>
<h3><span class="mw-headline" id="Shortest_remaining_time">Shortest remaining time</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=10" title="Edit section: Shortest remaining time">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Shortest_remaining_time" title="Shortest remaining time">Shortest remaining time</a></div>
<p>Similar to <a class="mw-redirect" href="/wiki/Shortest_job_first" title="Shortest job first">shortest job first</a> (SJF). With this strategy the scheduler arranges processes with the least estimated processing time remaining to be next in the queue. This requires advanced knowledge or estimations about the time required for a process to complete.</p>
<ul>
<li>If a shorter process arrives during another process' execution, the currently running process may be interrupted (known as preemption), dividing that process into two separate computing blocks. This creates excess overhead through additional context switching. The scheduler must also place each incoming process into a specific place in the queue, creating additional overhead.</li>
<li>This algorithm is designed for maximum throughput in most scenarios.</li>
<li>Waiting time and response time increase as the process's computational requirements increase. Since turnaround time is based on waiting time plus processing time, longer processes are significantly affected by this. Overall waiting time is smaller than FIFO, however since no process has to wait for the termination of the longest process.</li>
<li>No particular attention is given to deadlines, the programmer can only attempt to make processes with deadlines as short as possible.</li>
<li>Starvation is possible, especially in a busy system with many small processes being run.</li>
<li>This policy is rarely used, as of 2014.</li>
<li>To use this policy we should have at least two processes of different priority</li>
</ul>
<h3><span class="mw-headline" id="Fixed_priority_pre-emptive_scheduling">Fixed priority pre-emptive scheduling</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=11" title="Edit section: Fixed priority pre-emptive scheduling">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a class="mw-redirect" href="/wiki/Fixed_priority_pre-emptive_scheduling" title="Fixed priority pre-emptive scheduling">Fixed priority pre-emptive scheduling</a></div>
<p>The operating system assigns a fixed priority rank to every process, and the scheduler arranges the processes in the ready queue in order of their priority. Lower-priority processes get interrupted by incoming higher-priority processes.</p>
<ul>
<li>Overhead is not minimal, nor is it significant.</li>
<li>FPPS has no particular advantage in terms of throughput over FIFO scheduling.</li>
<li>If the number of rankings is limited, it can be characterized as a collection of FIFO queues, one for each priority ranking. Processes in lower-priority queues are selected only when all of the higher-priority queues are empty.</li>
<li>Waiting time and response time depend on the priority of the process. Higher-priority processes have smaller waiting and response times.</li>
<li>Deadlines can be met by giving processes with deadlines a higher priority.</li>
<li>Starvation of lower-priority processes is possible with large numbers of high-priority processes queuing for CPU time.</li>
</ul>
<h3><span class="mw-headline" id="Round-robin_scheduling">Round-robin scheduling</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=12" title="Edit section: Round-robin scheduling">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Round-robin_scheduling" title="Round-robin scheduling">Round-robin scheduling</a></div>
<p>The scheduler assigns a fixed time unit per process, and cycles through them.</p>
<ul>
<li>RR scheduling involves extensive overhead, especially with a small time unit.</li>
<li>Balanced throughput between FIFO and SJF, shorter jobs are completed faster than in FIFO and longer processes are completed faster than in SJF.</li>
<li>Good average response time, waiting time is dependent on number of processes, and not average process length.</li>
<li>Because of high waiting times, deadlines are rarely met in a pure RR system.</li>
<li>Starvation can never occur, since no priority is given. Order of time unit allocation is based upon process arrival time, similar to FIFO.</li>
</ul>
<h3><span class="mw-headline" id="Multilevel_queue_scheduling">Multilevel queue scheduling</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=13" title="Edit section: Multilevel queue scheduling">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Multilevel_queue" title="Multilevel queue">Multilevel queue</a></div>
<p>This is used for situations in which processes are easily divided into different groups. For example, a common division is made between foreground (interactive) processes and background (batch) processes. These two types of processes have different response-time requirements and so may have different scheduling needs. It is very useful for shared memory problems.</p>
<h3><span class="mw-headline" id="Work-conserving_schedulers">Work-conserving schedulers</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=14" title="Edit section: Work-conserving schedulers">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/Work-conserving_scheduler" title="Work-conserving scheduler">Work-conserving scheduler</a></div>
<p>A <a href="/wiki/Work-conserving_scheduler" title="Work-conserving scheduler">work-conserving scheduler</a> is a scheduler that always tries to keep the scheduled resources busy, if there are submitted jobs ready to be scheduled. In contrast, a non-work conserving scheduler is a scheduler that, in some cases, may leave the scheduled resources idle despite the presence of jobs ready to be scheduled.</p>
<h3><span class="mw-headline" id="Scheduling_optimization_problems">Scheduling optimization problems</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=15" title="Edit section: Scheduling optimization problems">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>There are several scheduling problems in which the goal is to decide which job goes to which station at what time, such that the total <a href="/wiki/Makespan" title="Makespan">makespan</a> is minimized:</p>
<ul>
<li><a href="/wiki/Job_shop_scheduling" title="Job shop scheduling">Job shop scheduling</a> –  there are <span class="texhtml mvar" style="font-style:italic;">n</span> jobs and <span class="texhtml mvar" style="font-style:italic;">m</span> identical stations. Each job should be executed on a single machine. This is usually regarded as an online problem.</li>
<li><a href="/wiki/Open-shop_scheduling" title="Open-shop scheduling">Open-shop scheduling</a> –  there are <span class="texhtml mvar" style="font-style:italic;">n</span> jobs and <span class="texhtml mvar" style="font-style:italic;">m</span> different stations. Each job should spend some time at each station, in a free order.</li>
<li><a href="/wiki/Flow_shop_scheduling" title="Flow shop scheduling">Flow shop scheduling</a> –  there are <span class="texhtml mvar" style="font-style:italic;">n</span> jobs and <span class="texhtml mvar" style="font-style:italic;">m</span> different stations. Each job should spend some time at each station, in a pre-determined order.</li>
</ul>
<h3><span class="mw-headline" id="Manual_scheduling">Manual scheduling</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=16" title="Edit section: Manual scheduling">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>A very common method in embedded systems is to manually schedule jobs. This can for example be done in a time-multiplexed fashion. Sometimes the kernel is divided in three or more parts: Manual scheduling, preemptive and interrupt level. Exact methods for scheduling jobs are often proprietary.</p>
<ul>
<li>No resource starvation problems</li>
<li>Very high predictability; allows implementation of hard real-time systems</li>
<li>Almost no overhead</li>
<li>May not be optimal for all applications</li>
<li>Effectiveness is completely dependent on the implementation</li>
</ul>
<h3><span class="mw-headline" id="Choosing_a_scheduling_algorithm">Choosing a scheduling algorithm</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=17" title="Edit section: Choosing a scheduling algorithm">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>When designing an operating system, a programmer must consider which scheduling algorithm will perform best for the use the system is going to see. There is no universal “best” scheduling algorithm, and many operating systems use extended or combinations of the scheduling algorithms above.</p>
<p>For example, <a href="/wiki/Windows_NT" title="Windows NT">Windows NT</a>/XP/Vista uses a <a href="/wiki/Multilevel_feedback_queue" title="Multilevel feedback queue">multilevel feedback queue</a>, a combination of fixed-priority preemptive scheduling, round-robin, and first in, first out algorithms. In this system, threads can dynamically increase or decrease in priority depending on if it has been serviced already, or if it has been waiting extensively. Every priority level is represented by its own queue, with <a href="/wiki/Round-robin_scheduling" title="Round-robin scheduling">round-robin scheduling</a> among the high-priority threads and <a href="/wiki/FIFO_(computing_and_electronics)" title="FIFO (computing and electronics)">FIFO</a> among the lower-priority ones. In this sense, response time is short for most threads, and short but critical system threads get completed very quickly. Since threads can only use one time unit of the round-robin in the highest-priority queue, starvation can be a problem for longer high-priority threads.</p>
<h2><span class="mw-headline" id="Operating_system_process_scheduler_implementations">Operating system process scheduler implementations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=18" title="Edit section: Operating system process scheduler implementations">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The algorithm used may be as simple as <a href="/wiki/Round-robin_scheduling" title="Round-robin scheduling">round-robin</a> in which each process is given equal time (for instance 1 ms, usually between 1 ms and 100 ms) in a cycling list. So, process A executes for 1 ms, then process B, then process C, then back to process A.</p>
<p>More advanced algorithms take into account process priority, or the importance of the process. This allows some processes to use more time than other processes. The kernel always uses whatever resources it needs to ensure proper functioning of the system, and so can be said to have infinite priority. In <a href="/wiki/Symmetric_multiprocessing" title="Symmetric multiprocessing">SMP</a>(symmetric multiprocessing) systems, <a href="/wiki/Processor_affinity" title="Processor affinity">processor affinity</a> is considered to increase overall system performance, even if it may cause a process itself to run more slowly. This generally improves performance by reducing <a class="mw-redirect" href="/wiki/Cache_thrashing" title="Cache thrashing">cache thrashing</a>.</p>
<h3><span class="mw-headline" id="OS.2F360_and_successors">OS/360 and successors</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=19" title="Edit section: OS/360 and successors">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>IBM <a href="/wiki/OS/360_and_successors" title="OS/360 and successors">OS/360</a> was available with three different schedulers. The differences were such that the variants were often considered three different operating systems:</p>
<ul>
<li>The <i>Single Sequential Scheduler</i> option, also known as the <i>Primary Control Program (PCP)</i> provided sequential execution of a single stream of jobs.</li>
<li>The <i>Multiple Sequential Scheduler</i> option, known as <i>Multiprogramming with a Fixed Number of Tasks (MFT)</i> provided execution of multiple concurrent jobs. Execution was governed by a priority which had a default for each stream or could be requested separately for each job. MFT version II added subtasks (threads), which executed at a priority based on that of the parent job. Each job stream defined the maximum amount of memory which could be used by any job in that stream.</li>
<li>The <i>Multiple Priority Schedulers</i> option, or <i>Multiprogramming with a Variable Number of Tasks (MVT)</i>, featured subtasks from the start; each job requested the priority and memory it required before execution.</li>
</ul>
<p>Later virtual storage versions of MVS added a <i><a href="/wiki/Workload_Manager" title="Workload Manager">Workload Manager</a></i> feature to the scheduler, which schedules processor resources according to an elaborate scheme defined by the installation.</p>
<h3><span class="mw-headline" id="Windows">Windows</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=20" title="Edit section: Windows">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Very early <a href="/wiki/MS-DOS" title="MS-DOS">MS-DOS</a> and Microsoft Windows systems were non-multitasking, and as such did not feature a scheduler. <a href="/wiki/Windows_3.1x" title="Windows 3.1x">Windows 3.1x</a> used a non-preemptive scheduler, meaning that it did not interrupt programs. It relied on the program to end or tell the OS that it didn't need the processor so that it could move on to another process. This is usually called cooperative multitasking. Windows 95 introduced a rudimentary preemptive scheduler; however, for legacy support opted to let 16 bit applications run without preemption.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup></p>
<p><a href="/wiki/Windows_NT" title="Windows NT">Windows NT</a>-based operating systems use a multilevel feedback queue. 32 priority levels are defined, 0 through to 31, with priorities 0 through 15 being "normal" priorities and priorities 16 through 31 being soft real-time priorities, requiring privileges to assign. 0 is reserved for the Operating System. Users can select 5 of these priorities to assign to a running application from the Task Manager application, or through thread management APIs. The kernel may change the priority level of a thread depending on its I/O and CPU usage and whether it is interactive (i.e. accepts and responds to input from humans), raising the priority of interactive and I/O bounded processes and lowering that of CPU bound processes, to increase the responsiveness of interactive applications.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup> The scheduler was modified in <a href="/wiki/Windows_Vista" title="Windows Vista">Windows Vista</a> to use the <a href="/wiki/Time_Stamp_Counter" title="Time Stamp Counter">cycle counter register</a> of modern processors to keep track of exactly how many CPU cycles a thread has executed, rather than just using an interval-timer interrupt routine.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup> Vista also uses a priority scheduler for the I/O queue so that disk defragmenters and other such programs do not interfere with foreground operations.<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup></p>
<h3><span class="mw-headline" id="Classic_Mac_OS_and_macOS">Classic Mac OS and macOS</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=21" title="Edit section: Classic Mac OS and macOS">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Mac OS 9 uses cooperative scheduling for threads, where one process controls multiple cooperative threads, and also provides preemptive scheduling for multiprocessing tasks. The kernel schedules multiprocessing tasks using a preemptive scheduling algorithm. All Process Manager processes run within a special multiprocessing task, called the "blue task". Those processes are scheduled cooperatively, using a <a href="/wiki/Round-robin_scheduling" title="Round-robin scheduling">round-robin scheduling</a> algorithm; a process yields control of the processor to another process by explicitly calling a blocking function such as <code>WaitNextEvent</code>. Each process has its own copy of the <a class="new" href="/w/index.php?title=Thread_Manager&amp;action=edit&amp;redlink=1" title="Thread Manager (page does not exist)">Thread Manager</a> that schedules that process's threads cooperatively; a thread yields control of the processor to another thread by calling <code>YieldToAnyThread</code> or <code>YieldToThread</code>.<sup class="reference" id="cite_ref-appletn2028_9-0"><a href="#cite_note-appletn2028-9">[9]</a></sup></p>
<p>macOS uses a multilevel feedback queue, with four priority bands for threads – normal, system high priority, kernel mode only, and real-time.<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup> Threads are scheduled preemptively; macOS also supports cooperatively scheduled threads in its implementation of the Thread Manager in <a href="/wiki/Carbon_(API)" title="Carbon (API)">Carbon</a>.<sup class="reference" id="cite_ref-appletn2028_9-1"><a href="#cite_note-appletn2028-9">[9]</a></sup></p>
<h3><span class="mw-headline" id="AIX">AIX</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=22" title="Edit section: AIX">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In AIX Version 4 there are three possible values for thread scheduling policy:</p>
<ul>
<li>First In, First Out: Once a thread with this policy is scheduled, it runs to completion unless it is blocked, it voluntarily yields control of the CPU, or a higher-priority thread becomes dispatchable. Only fixed-priority threads can have a FIFO scheduling policy.</li>
<li>Round Robin: This is similar to the AIX Version 3 scheduler round-robin scheme based on 10ms time slices. When a RR thread has control at the end of the time slice, it moves to the tail of the queue of dispatchable threads of its priority. Only fixed-priority threads can have a Round Robin scheduling policy.</li>
<li>OTHER: This policy is defined by POSIX1003.4a as implementation-defined. In AIX Version 4, this policy is defined to be equivalent to RR, except that it applies to threads with non-fixed priority. The recalculation of the running thread's priority value at each clock interrupt means that a thread may lose control because its priority value has risen above that of another dispatchable thread. This is the AIX Version 3 behavior.</li>
</ul>
<p>Threads are primarily of interest for applications that currently consist of several asynchronous processes. These applications might impose a lighter load on the system if converted to a multithreaded structure.</p>
<p>AIX 5 implements the following scheduling policies: FIFO, round robin, and a fair round robin. The FIFO policy has three different implementations: FIFO, FIFO2, and FIFO3. The round robin policy is named SCHED_RR in AIX, and the fair round robin is called SCHED_OTHER.<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup></p>

<h4><span class="mw-headline" id="Linux_2.4">Linux 2.4</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=24" title="Edit section: Linux 2.4">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>In <a href="/wiki/Linux" title="Linux">Linux</a> 2.4, an <a href="/wiki/O(n)_scheduler" title="O(n) scheduler">O(n) scheduler</a> with a <a href="/wiki/Multilevel_feedback_queue" title="Multilevel feedback queue">multilevel feedback queue</a> with priority levels ranging from 0 to 140 was used; 0–99 are reserved for real-time tasks and 100–140 are considered <a href="/wiki/Nice_(Unix)" title="Nice (Unix)">nice</a> task levels. For real-time tasks, the time quantum for switching processes was approximately 200 ms, and for nice tasks approximately 10 ms.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (December 2011)">citation needed</span></a></i>]</sup> The scheduler ran through the <a href="/wiki/Run_queue" title="Run queue">run queue</a> of all ready processes, letting the highest priority processes go first and run through their time slices, after which they will be placed in an expired queue. When the active queue is empty the expired queue will become the active queue and vice versa.</p>
<p>However, some enterprise <a class="mw-redirect" href="/wiki/Linux_distributions" title="Linux distributions">Linux distributions</a> such as <a href="/wiki/SUSE_Linux_Enterprise_Server" title="SUSE Linux Enterprise Server">SUSE Linux Enterprise Server</a> replaced this scheduler with a backport of the <a href="/wiki/O(1)_scheduler" title="O(1) scheduler">O(1) scheduler</a> (which was maintained by <a href="/wiki/Alan_Cox" title="Alan Cox">Alan Cox</a> in his Linux 2.4-ac Kernel series) to the Linux 2.4 kernel used by the distribution.</p>
<h4><span class="mw-headline" id="Linux_2.6.0_to_Linux_2.6.22">Linux 2.6.0 to Linux 2.6.22</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=25" title="Edit section: Linux 2.6.0 to Linux 2.6.22">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>In versions 2.6.0 to 2.6.22, the kernel used an <a href="/wiki/O(1)_scheduler" title="O(1) scheduler">O(1) scheduler</a> developed by <a class="mw-redirect" href="/wiki/Ingo_Molnar" title="Ingo Molnar">Ingo Molnar</a> and many other kernel developers during the Linux 2.5 development. For many kernel in time frame, <a href="/wiki/Con_Kolivas" title="Con Kolivas">Con Kolivas</a> developed patch sets which improved interactivity with this scheduler or even replaced it with his own schedulers.</p>
<h4><span class="mw-headline" id="Since_Linux_2.6.23">Since Linux 2.6.23</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=26" title="Edit section: Since Linux 2.6.23">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Con Kolivas's work, most significantly his implementation of "<a href="/wiki/Fair-share_scheduling" title="Fair-share scheduling">fair scheduling</a>" named "Rotating Staircase Deadline", inspired Ingo Molnár to develop the <a href="/wiki/Completely_Fair_Scheduler" title="Completely Fair Scheduler">Completely Fair Scheduler</a> as a replacement for the earlier <a href="/wiki/O(1)_scheduler" title="O(1) scheduler">O(1) scheduler</a>, crediting Kolivas in his announcement.<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup> CFS is the first implementation of a fair queuing <a class="mw-redirect" href="/wiki/Process_scheduler" title="Process scheduler">process scheduler</a> widely used in a general-purpose operating system.<sup class="reference" id="cite_ref-dwrr_13-0"><a href="#cite_note-dwrr-13">[13]</a></sup></p>
<p>The <a href="/wiki/Completely_Fair_Scheduler" title="Completely Fair Scheduler">Completely Fair Scheduler</a> (CFS) uses a well-studied, classic scheduling algorithm called <a href="/wiki/Fair_queuing" title="Fair queuing">fair queuing</a> originally invented for <a class="mw-redirect" href="/wiki/Packet_network" title="Packet network">packet networks</a>. Fair queuing had been previously applied to CPU scheduling under the name <a href="/wiki/Stride_scheduling" title="Stride scheduling">stride scheduling</a>. The fair queuing CFS scheduler has a scheduling complexity of O(<a href="/wiki/Logarithm" title="Logarithm">log</a> N), where N is the number of tasks in the <a href="/wiki/Run_queue" title="Run queue">runqueue</a>. Choosing a task can be done in constant time, but reinserting a task after it has run requires O(log N) operations, because the <a href="/wiki/Run_queue" title="Run queue">run queue</a> is implemented as a <a class="mw-redirect" href="/wiki/Red-black_tree" title="Red-black tree">red-black tree</a>.</p>
<p>The <a href="/wiki/Brain_Fuck_Scheduler" title="Brain Fuck Scheduler">Brain Fuck Scheduler</a> (BFS), also created by Con Kolivas, is an alternative to the CFS.</p>
<h3><span class="mw-headline" id="FreeBSD">FreeBSD</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=27" title="Edit section: FreeBSD">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/FreeBSD" title="FreeBSD">FreeBSD</a> uses a multilevel feedback queue with priorities ranging from 0–255. 0–63 are reserved for interrupts, 64–127 for the top half of the kernel, 128–159 for real-time user threads, 160–223 for time-shared user threads, and 224–255 for idle user threads. Also, like Linux, it uses the active queue setup, but it also has an idle queue.<sup class="reference" id="cite_ref-opensolaris-queue_14-0"><a href="#cite_note-opensolaris-queue-14">[14]</a></sup></p>
<h3><span class="mw-headline" id="NetBSD">NetBSD</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=28" title="Edit section: NetBSD">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/NetBSD" title="NetBSD">NetBSD</a> uses a multilevel feedback queue with priorities ranging from 0–223. 0–63 are reserved for time-shared threads (default, SCHED_OTHER policy), 64–95 for user threads which entered <a class="mw-redirect" href="/wiki/Kernel_space" title="Kernel space">kernel space</a>, 96-128 for kernel threads, 128–191 for user real-time threads (SCHED_FIFO and SCHED_RR policies), and 192–223 for <a href="/wiki/Interrupt" title="Interrupt">software interrupts</a>.</p>
<h3><span class="mw-headline" id="Solaris">Solaris</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Scheduling_(computing)&amp;action=edit&amp;section=29" title="Edit section: Solaris">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/Solaris_(operating_system)" title="Solaris (operating system)">Solaris</a> uses a multilevel feedback queue with priorities ranging between 0 and 169. Priorities 0–59 are reserved for time-shared threads, 60–99 for system threads, 100–159 for real-time threads, and 160–169 for low priority interrupts. Unlike Linux,<sup class="reference" id="cite_ref-opensolaris-queue_14-1"><a href="#cite_note-opensolaris-queue-14">[14]</a></sup> when a process is done using its time quantum, it is given a new priority and put back in the queue. Solaris 9 introduced two new scheduling classes, namely fixed priority class and fair share class. The threads with fixed priority have the same priority range as that of the time-sharing class, but their priorities are not dynamically adjusted. The fair scheduling class uses CPU <i>shares</i> to prioritize threads for scheduling decisions. CPU shares indicate the entitlement to CPU resources. They are allocated to a set of processes, which are collectively known as a project.<sup class="reference" id="cite_ref-galvin_3-2"><a href="#cite_note-galvin-3">[3]</a></sup></p>
