<p><b>Btrfs</b> (<b>B-tree file system</b>, pronounced as "butter F S",<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[11]</a></sup> "better F S",<sup class="reference" id="cite_ref-CM090622_9-1"><a href="#cite_note-CM090622-9">[8]</a></sup> "b-tree F S",<sup class="reference" id="cite_ref-13"><a href="#cite_note-13">[12]</a></sup> or simply by spelling it out) is a <a href="/wiki/File_system" title="File system">file system</a> based on the <a href="/wiki/Copy-on-write" title="Copy-on-write">copy-on-write</a> (COW) principle, initially designed at <a href="/wiki/Oracle_Corporation" title="Oracle Corporation">Oracle Corporation</a> for use in <a href="/wiki/Linux" title="Linux">Linux</a>. The development of Btrfs began in 2007, and by August 2014, the file system's on-disk format has been marked as stable.<sup class="reference" id="cite_ref-14"><a href="#cite_note-14">[13]</a></sup></p>
<p>Btrfs is intended to address the lack of <a href="/wiki/Pool_(computer_science)" title="Pool (computer science)">pooling</a>, <a href="/wiki/Snapshot_(computer_storage)" title="Snapshot (computer storage)">snapshots</a>, <a href="/wiki/Checksum" title="Checksum">checksums</a>, and integral multi-device spanning in <a class="mw-redirect" href="/wiki/Linux_file_systems" title="Linux file systems">Linux file systems</a>.<sup class="reference" id="cite_ref-CM090622_9-2"><a href="#cite_note-CM090622-9">[8]</a></sup> Chris Mason, the principal Btrfs author, has stated that its goal was "to let Linux scale for the storage that will be available. Scaling is not just about addressing the storage but also means being able to administer and to manage it with a clean interface that lets people see what's being used and makes it more reliable."<sup class="reference" id="cite_ref-15"><a href="#cite_note-15">[14]</a></sup></p>

<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Btrfs&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The core data structure of Btrfs—<wbr></wbr>​​the copy-on-write <a href="/wiki/B-tree" title="B-tree">B-tree</a>—<wbr></wbr>​​was originally proposed by IBM researcher Ohad Rodeh at a presentation at <a href="/wiki/USENIX" title="USENIX">USENIX</a> 2007.<sup class="reference" id="cite_ref-rodeh-1_16-0"><a href="#cite_note-rodeh-1-16">[15]</a></sup> Chris Mason, an engineer working on <a href="/wiki/ReiserFS" title="ReiserFS">ReiserFS</a> for <a href="/wiki/SUSE_Linux_distributions" title="SUSE Linux distributions">SUSE</a> at the time, joined Oracle later that year and began work on a new file system based on these B-trees.<sup class="reference" id="cite_ref-joinfb_17-0"><a href="#cite_note-joinfb-17">[16]</a></sup></p>
<p>In 2008, the principal developer of the <a href="/wiki/Ext3" title="Ext3">ext3</a> and <a href="/wiki/Ext4" title="Ext4">ext4</a> file systems, <a href="/wiki/Theodore_Ts%27o" title="Theodore Ts'o">Theodore Ts'o</a>, stated that although ext4 has improved features, it is not a major advance; it uses old technology and is a stop-gap. Ts'o said that Btrfs is the better direction because "it offers improvements in scalability, reliability, and ease of management".<sup class="reference" id="cite_ref-18"><a href="#cite_note-18">[17]</a></sup> Btrfs also has "a number of the same design ideas that <a href="/wiki/ReiserFS" title="ReiserFS">reiser3</a>/<a href="/wiki/Reiser4" title="Reiser4">4</a> had".<sup class="reference" id="cite_ref-19"><a href="#cite_note-19">[18]</a></sup></p>
<p>Btrfs 1.0, with finalized on-disk format, was originally slated for a late-2008 release,<sup class="reference" id="cite_ref-20"><a href="#cite_note-20">[19]</a></sup> and was finally accepted into the <a class="mw-redirect" href="/wiki/Linux_kernel_mainline" title="Linux kernel mainline">Linux kernel mainline</a> in 2009.<sup class="reference" id="cite_ref-21"><a href="#cite_note-21">[20]</a></sup> Several <a href="/wiki/Linux_distribution" title="Linux distribution">Linux distributions</a> began offering Btrfs as an experimental choice of <a class="mw-redirect" href="/wiki/Root_file_system" title="Root file system">root file system</a> during installation.<sup class="reference" id="cite_ref-22"><a href="#cite_note-22">[21]</a></sup><sup class="reference" id="cite_ref-23"><a href="#cite_note-23">[22]</a></sup><sup class="reference" id="cite_ref-24"><a href="#cite_note-24">[23]</a></sup></p>
<p>In July 2011, Btrfs automatic <a href="/wiki/Defragmentation" title="Defragmentation">defragmentation</a> and <a href="/wiki/Data_scrubbing" title="Data scrubbing">scrubbing</a> features were merged into version 3.0 of the <a class="mw-redirect" href="/wiki/Linux_kernel_mainline" title="Linux kernel mainline">Linux kernel mainline</a>.<sup class="reference" id="cite_ref-defragandscrubbing_25-0"><a href="#cite_note-defragandscrubbing-25">[24]</a></sup> Besides Mason at Oracle, Miao Xie at Fujitsu contributed performance improvements.<sup class="reference" id="cite_ref-26"><a href="#cite_note-26">[25]</a></sup> In June 2012, Chris Mason left Oracle for <a href="/wiki/Fusion-io" title="Fusion-io">Fusion-io</a>, which he left a year later with Josef Bacik to join <a href="/wiki/Facebook" title="Facebook">Facebook</a>; in both companies, Mason continued to work on Btrfs.<sup class="reference" id="cite_ref-27"><a href="#cite_note-27">[26]</a></sup><sup class="reference" id="cite_ref-joinfb_17-1"><a href="#cite_note-joinfb-17">[16]</a></sup></p>
<p>In summer 2012, several Linux distributions moved Btrfs from experimental to production or supported status.<sup class="reference" id="cite_ref-28"><a href="#cite_note-28">[27]</a></sup><sup class="reference" id="cite_ref-29"><a href="#cite_note-29">[28]</a></sup> In 2015, Btrfs was adopted as the default filesystem for <a href="/wiki/SUSE_Linux_Enterprise_Server" title="SUSE Linux Enterprise Server">SUSE Linux Enterprise Server</a> 12.<sup class="reference" id="cite_ref-30"><a href="#cite_note-30">[29]</a></sup></p>
<h2><span class="mw-headline" id="Features">Features</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Btrfs&amp;action=edit&amp;section=2" title="Edit section: Features">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>As of version 3.14 of the Linux kernel, Btrfs implements the following features:<sup class="reference" id="cite_ref-btrfs-features-list_31-0"><a href="#cite_note-btrfs-features-list-31">[30]</a></sup><sup class="reference" id="cite_ref-32"><a href="#cite_note-32">[31]</a></sup></p>
<ul>
<li>Mostly self-healing in some configurations due to the nature of <a href="/wiki/Copy-on-write" title="Copy-on-write">copy-on-write</a></li>
<li>Online defragmentation and an <i>autodefrag</i> mount option <sup class="reference" id="cite_ref-defragandscrubbing_25-1"><a href="#cite_note-defragandscrubbing-25">[24]</a></sup></li>
<li>Online volume growth and shrinking</li>
<li>Online <a class="mw-redirect" href="/wiki/Block_device" title="Block device">block device</a> addition and removal</li>

<h3><span class="mw-headline" id="Cloning">Cloning</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Btrfs&amp;action=edit&amp;section=3" title="Edit section: Cloning">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Btrfs provides a <i>clone</i> operation that <a class="mw-redirect" href="/wiki/Atomicity_(programming)" title="Atomicity (programming)">atomically</a> creates a copy-on-write snapshot of a <a href="/wiki/Computer_file" title="Computer file">file</a>. Such cloned files are sometimes referred to as <i>reflinks</i>, in light of the associated Linux kernel <a href="/wiki/System_call" title="System call">system calls</a>.<sup class="reference" id="cite_ref-50"><a href="#cite_note-50">[49]</a></sup></p>
<p>By cloning, the file system does not create a new link pointing to an existing <a href="/wiki/Inode" title="Inode">inode</a>; instead, it creates a new inode that initially shares the same disk blocks with the original file. As a result, cloning works only within the boundaries of the same Btrfs file system, but since version 3.6 of the Linux kernel it may cross the boundaries of subvolumes under certain circumstances.<sup class="reference" id="cite_ref-btrfs-usecases_51-0"><a href="#cite_note-btrfs-usecases-51">[50]</a></sup><sup class="reference" id="cite_ref-52"><a href="#cite_note-52">[51]</a></sup> The actual data blocks are not duplicated; at the same time, due to the copy-on-write (CoW) nature of Btrfs, modifications to any of the cloned files are not visible in the original file and vice versa.<sup class="reference" id="cite_ref-oracle-reflinks_53-0"><a href="#cite_note-oracle-reflinks-53">[52]</a></sup></p>
<p>Cloning should not be confused with <a class="mw-redirect" href="/wiki/Hard_links" title="Hard links">hard links</a>, which are directory entries that associate multiple file names with actual files on a file system. While hard links can be taken as different names for the same file, cloning in Btrfs provides independent files that share their disk blocks.<sup class="reference" id="cite_ref-oracle-reflinks_53-1"><a href="#cite_note-oracle-reflinks-53">[52]</a></sup><sup class="reference" id="cite_ref-54"><a href="#cite_note-54">[53]</a></sup></p>
<p>Support for this Btrfs feature was added in version 7.5 of the <a class="mw-redirect" href="/wiki/Coreutils" title="Coreutils">GNU coreutils</a>, via the <code>--reflink</code> option to the <code><a href="/wiki/Cp_(Unix)" title="Cp (Unix)">cp</a></code> command.<sup class="reference" id="cite_ref-55"><a href="#cite_note-55">[54]</a></sup><sup class="reference" id="cite_ref-56"><a href="#cite_note-56">[55]</a></sup></p>
<h3><span class="mw-headline" id="Subvolumes_and_snapshots">Subvolumes and snapshots</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Btrfs&amp;action=edit&amp;section=4" title="Edit section: Subvolumes and snapshots">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>A Btrfs subvolume can be thought of as a separate POSIX file <a href="/wiki/Namespace" title="Namespace">namespace</a>, <a href="/wiki/Mount_(computing)" title="Mount (computing)">mountable</a> separately by passing <code>subvol</code> or <code>subvolid</code> options to the <span class="plainlinksneverexpand"><code><a class="external text" href="http://man.cx/?page=mount(8)" rel="nofollow">mount(8)</a></code></span> utility. It can also be accessed by mounting the top-level subvolume, in which case subvolumes are visible and accessible as its subdirectories.<sup class="reference" id="cite_ref-btrfs-sysadmin-guide_57-0"><a href="#cite_note-btrfs-sysadmin-guide-57">[56]</a></sup></p>
<p>Subvolumes can be created at any place within the file system hierarchy, and they can also be nested. Nested subvolumes appear as subdirectories within their parent subvolumes, similarly to the way top-level subvolume presents its subvolumes as subdirectories. Deleting a subvolume deletes all subvolumes below it in the nesting hierarchy; as a result, top-level subvolumes cannot be deleted.<sup class="reference" id="cite_ref-oracle-btrfs-subvolumes_58-0"><a href="#cite_note-oracle-btrfs-subvolumes-58">[57]</a></sup></p>
<p>Any Btrfs file system always has a default subvolume, which is initially set to be the top-level subvolume, and is mounted by default if no subvolume selection option is passed to <code>mount</code>. The default subvolume can be changed as required.<sup class="reference" id="cite_ref-oracle-btrfs-subvolumes_58-1"><a href="#cite_note-oracle-btrfs-subvolumes-58">[57]</a></sup></p>
<p>A Btrfs <a href="/wiki/Snapshot_(computer_storage)" title="Snapshot (computer storage)">snapshot</a> is actually a subvolume that shares its data (and metadata) with some other subvolume, using Btrfs' copy-on-write capabilities, and modifications to a snapshot are not visible in the original subvolume. Once a writable snapshot is made, it can be treated as an alternate version of the original file system. For example, to rollback to a snapshot, a modified original subvolume needs to be unmounted and the snapshot needs to be mounted in its place. At that point, the original subvolume may also be deleted.<sup class="reference" id="cite_ref-btrfs-sysadmin-guide_57-1"><a href="#cite_note-btrfs-sysadmin-guide-57">[56]</a></sup></p>
<p>The copy-on-write (CoW) nature of Btrfs means that snapshots are quickly created, while initially consuming very little disk space. Since a snapshot is a subvolume, creating nested snapshots is also possible. Taking snapshots of a subvolume is not a recursive process; thus, if a snapshot of a subvolume is created, every subvolume or snapshot that the subvolume already contains is mapped to an empty directory of the same name inside the snapshot.<sup class="reference" id="cite_ref-btrfs-sysadmin-guide_57-2"><a href="#cite_note-btrfs-sysadmin-guide-57">[56]</a></sup><sup class="reference" id="cite_ref-oracle-btrfs-subvolumes_58-2"><a href="#cite_note-oracle-btrfs-subvolumes-58">[57]</a></sup></p>
<p>Taking snapshots of a directory is not possible, as only subvolumes can have snapshots. However, there is a workaround that involves reflinks spread across subvolumes: a new subvolume is created, containing cross-subvolume reflinks to the content of the targeted directory. Having that available, a snapshot of this new volume can be created.<sup class="reference" id="cite_ref-btrfs-usecases_51-1"><a href="#cite_note-btrfs-usecases-51">[50]</a></sup></p>
<p>A subvolume in Btrfs is quite different from a traditional <a class="mw-redirect" href="/wiki/Logical_Volume_Manager" title="Logical Volume Manager">Logical Volume Manager</a> (LVM) logical volume. With LVM, a logical volume is a separate <a class="mw-redirect" href="/wiki/Block_device" title="Block device">block device</a>, while a Btrfs subvolume is not and it cannot be treated or used that way.<sup class="reference" id="cite_ref-btrfs-sysadmin-guide_57-3"><a href="#cite_note-btrfs-sysadmin-guide-57">[56]</a></sup></p>
<h3><span class="mw-headline" id="Send.2Freceive">Send/receive</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Btrfs&amp;action=edit&amp;section=5" title="Edit section: Send/receive">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Given any pair of subvolumes (or snapshots), Btrfs can generate a binary <a class="mw-redirect" href="/wiki/Diff" title="Diff">diff</a> between them (by using the <code>btrfs send</code> command) that can be replayed later (by using <code>btrfs receive</code>), possibly on a different Btrfs file system. The send/receive feature effectively creates (and applies) a set of data modifications required for converting one subvolume into another.<sup class="reference" id="cite_ref-corbet-jul2011_40-1"><a href="#cite_note-corbet-jul2011-40">[39]</a></sup><sup class="reference" id="cite_ref-oracle-btrfs-send-receive_59-0"><a href="#cite_note-oracle-btrfs-send-receive-59">[58]</a></sup></p>
<p>The send/receive feature can be used with regularly scheduled snapshots for implementing a simple form of file system master/slave <a href="/wiki/Replication_(computing)" title="Replication (computing)">replication</a>, or for the purpose of performing <a href="/wiki/Incremental_backup" title="Incremental backup">incremental backups</a>.<sup class="reference" id="cite_ref-corbet-jul2011_40-2"><a href="#cite_note-corbet-jul2011-40">[39]</a></sup><sup class="reference" id="cite_ref-oracle-btrfs-send-receive_59-1"><a href="#cite_note-oracle-btrfs-send-receive-59">[58]</a></sup></p>
<h3><span class="mw-headline" id="Quota_groups">Quota groups</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Btrfs&amp;action=edit&amp;section=6" title="Edit section: Quota groups">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>A <i>quota group</i> (or <i>qgroup</i>) imposes an upper limit to the space a subvolume or snapshot may consume. A new snapshot initially consumes no quota because its data is shared with its parent, but thereafter incurs a charge for new files and copy-on-write operations on existing files. When quotas are active, a quota group is automatically created with each new subvolume or snapshot. These initial quota groups are building blocks which can be grouped (with the <code>btrfs qgroup</code> command) into hierarchies to implement quota pools.<sup class="reference" id="cite_ref-jansen-oct2011_41-1"><a href="#cite_note-jansen-oct2011-41">[40]</a></sup></p>
<p>Quota groups only apply to subvolumes and snapshots, while having quotas enforced on individual subdirectories, users, or user groups is not possible. However, workarounds are possible by using different subvolumes for all users or user groups that require a quota to be enforced.</p>
<h3><span class="mw-headline" id="In-place_ext2.2F3.2F4_conversion">In-place ext2/3/4 conversion</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Btrfs&amp;action=edit&amp;section=7" title="Edit section: In-place ext2/3/4 conversion">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>As the result of having very little metadata anchored in fixed locations, Btrfs can warp to fit unusual spatial layouts of the backend storage devices. The <code>btrfs-convert</code> tool exploits this ability to do an in-place conversion of any ext2/3/4 file system, by nesting the equivalent Btrfs metadata in its unallocated space — while preserving an unmodified copy of the original file system.<sup class="reference" id="cite_ref-ext3_conversion_38-1"><a href="#cite_note-ext3_conversion-38">[37]</a></sup></p>
<p>The conversion involves creating a copy of the whole ext2/3/4 metadata, while the Btrfs files simply point to the same blocks used by the ext2/3/4 files. This makes the bulk of the blocks shared between the two filesystems before the conversion becomes permanent. Thanks to the copy-on-write nature of Btrfs, the original versions of the file data blocks are preserved during all file modifications. Until the conversion becomes permanent, only the blocks that were marked as free in ext2/3/4 are used to hold new Btrfs modifications, meaning that the conversion can be undone at any time (although doing so will erase any changes made after the conversion to Btrfs).<sup class="reference" id="cite_ref-ext3_conversion_38-2"><a href="#cite_note-ext3_conversion-38">[37]</a></sup></p>
<p>All converted files are available and writable in the default subvolume of the Btrfs. A sparse file holding all of the references to the original ext2/3/4 filesystem is created in a separate subvolume, which is mountable on its own as a read-only disk image, allowing both original and converted file systems to be accessed at the same time. Deleting this sparse file frees up the space and makes the conversion permanent.<sup class="reference" id="cite_ref-ext3_conversion_38-3"><a href="#cite_note-ext3_conversion-38">[37]</a></sup></p>
<p>As of June 2015<sup class="plainlinks noprint asof-tag update" style="display:none;"><a class="external text" href="//en.wikipedia.org/w/index.php?title=Btrfs&amp;action=edit">[update]</a></sup> and 4.x versions of the Linux kernel mainline, the in-place ext3/4 conversion is considered untested and rarely used.<sup class="reference" id="cite_ref-ext3_conversion_38-4"><a href="#cite_note-ext3_conversion-38">[37]</a></sup></p>
<h3><span class="mw-headline" id="Union_mounting_.2F_seed_devices">Union mounting / seed devices</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Btrfs&amp;action=edit&amp;section=8" title="Edit section: Union mounting / seed devices">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>When creating a new Btrfs, an existing Btrfs can be used as a read-only "seed" file system. The new file system will then act as a copy-on-write overlay on the seed, as a form of <a href="/wiki/Union_mount" title="Union mount">union mounting</a>. The seed can be later detached from the Btrfs, at which point the rebalancer will simply copy over any seed data still referenced by the new file system before detaching. Mason has suggested this may be useful for a <a href="/wiki/Live_CD" title="Live CD">Live CD</a> installer, which might boot from a read-only Btrfs seed on an optical disc, rebalance itself to the target partition on the install disk in the background while the user continues to work, then eject the disc to complete the installation without rebooting.<sup class="reference" id="cite_ref-mason-apr2012_60-0"><a href="#cite_note-mason-apr2012-60">[59]</a></sup></p>
<h3><span class="mw-headline" id="Encryption">Encryption</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Btrfs&amp;action=edit&amp;section=9" title="Edit section: Encryption">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In his 2009 interview, Chris Mason stated that support for encryption was planned for Btrfs.<sup class="reference" id="cite_ref-61"><a href="#cite_note-61">[60]</a></sup> In the meantime, a workaround for combining encryption with Btrfs is to use a full-disk encryption mechanism such as <a href="/wiki/Dm-crypt" title="Dm-crypt">dm-crypt</a>/<a class="mw-redirect" href="/wiki/LUKS" title="LUKS">LUKS</a> on the underlying devices and to create the Btrfs filesystem on top of that layer.</p>
<h3><span class="mw-headline" id="Checking_and_recovery">Checking and recovery</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Btrfs&amp;action=edit&amp;section=10" title="Edit section: Checking and recovery">edit</a><span class="mw-editsection-bracket">]</span></span></h3>

<td class="mbox-text"><span class="mbox-text-span">This section's <b><a href="/wiki/Wikipedia:Accuracy_dispute" title="Wikipedia:Accuracy dispute">factual accuracy</a> may be compromised due to out-of-date information</b>. <span class="hide-when-compact">Please update this article to reflect recent events or newly available information.</span> <small><i>(November 2016)</i></small></span></td>

<p>Unix systems traditionally rely on "<a href="/wiki/Fsck" title="Fsck">fsck</a>" programs to check and repair filesystems. The <code>btrfsck</code> program is now available but, as of May 2012, it is described by the authors as "relatively new code"<sup class="reference" id="cite_ref-btrfsck_62-0"><a href="#cite_note-btrfsck-62">[61]</a></sup> which has "not seen widespread testing on a large range of real-life breakage", and that "may cause additional damage in the process of repair".<sup class="reference" id="cite_ref-btrfsck_62-1"><a href="#cite_note-btrfsck-62">[61]</a></sup></p>
<p>There is another tool, named <code>btrfs-restore</code>, that can be used to recover files from an unmountable filesystem, without modifying the broken filesystem itself (i.e., non-destructively).<sup class="reference" id="cite_ref-63"><a href="#cite_note-63">[62]</a></sup></p>
<p>In normal use, Btrfs is mostly self-healing and can recover from broken root trees at mount time, thanks to making periodic data flushes to permanent storage every 30 seconds (which is the default period). Thus, isolated errors will cause a maximum of 30 seconds of filesystem changes to be lost at the next mount.<sup class="reference" id="cite_ref-64"><a href="#cite_note-64">[63]</a></sup> This period can be changed by specifying a desired value (in seconds) for the <code>commit</code> mount option.<sup class="reference" id="cite_ref-65"><a href="#cite_note-65">[64]</a></sup><sup class="reference" id="cite_ref-66"><a href="#cite_note-66">[65]</a></sup></p>
<h2><span class="mw-headline" id="Design">Design</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Btrfs&amp;action=edit&amp;section=11" title="Edit section: Design">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Ohad Rodeh's original proposal at USENIX 2007 noted that <a href="/wiki/B%2B_tree" title="B+ tree">B+ trees</a>, which are widely used as on-disk data structures for databases, could not efficiently allow copy-on-write-based snapshots because its leaf nodes were linked together: if a leaf was copy-on-written, its siblings and parents would have to be as well, as would <i>their</i> siblings and parents and so on until the entire tree was copied. He suggested instead a modified <a href="/wiki/B-tree" title="B-tree">B-tree</a> (which has no leaf linkage), with a <a href="/wiki/Reference_counting" title="Reference counting">refcount</a> associated to each tree node but stored in an ad-hoc free map structure and certain relaxations to the tree's balancing algorithms to make them copy-on-write friendly. The result would be a data structure suitable for a high-performance object store that could perform copy-on-write snapshots, while maintaining good <a href="/wiki/Concurrent_computing" title="Concurrent computing">concurrency</a>.<sup class="reference" id="cite_ref-rodeh-1_16-1"><a href="#cite_note-rodeh-1-16">[15]</a></sup></p>
<p>At Oracle later that year, Chris Mason began work on a snapshot-capable file system that would use this data structure almost exclusively—not just for metadata and file data, but also recursively to track space allocation of the trees themselves. This allowed all traversal and modifications to be funneled through a single code path, against which features such as copy-on-write, checksumming and mirroring needed to be implemented only once to benefit the entire file system.<sup class="reference" id="cite_ref-aurora-1_48-1"><a href="#cite_note-aurora-1-48">[47]</a></sup></p>
<p>Btrfs is structured as several layers of such trees, all using the same B-tree implementation. The trees store generic <i>items</i> sorted on a 136-bit key. The first 64 bits of the key are a unique <i>object id</i>. The middle 8 bits are an item type field; its use is hardwired into code as an item filter in tree lookups. <i>Objects</i> can have multiple items of multiple types. The remaining right-hand 64 bits are used in type-specific ways. Therefore, items for the same object end up adjacent to each other in the tree, ordered by type. By choosing certain right-hand key values, objects can further put items of the same type in a particular order.<sup class="reference" id="cite_ref-aurora-1_48-2"><a href="#cite_note-aurora-1-48">[47]</a></sup><sup class="reference" id="cite_ref-btrfs-wiki-1_67-0"><a href="#cite_note-btrfs-wiki-1-67">[66]</a></sup></p>
<p>Interior tree nodes are simply flat lists of key-pointer pairs, where the pointer is the logical block number of a child node. Leaf nodes contain item keys packed into the front of the node and item data packed into the end, with the two growing toward each other as the leaf fills up.<sup class="reference" id="cite_ref-aurora-1_48-3"><a href="#cite_note-aurora-1-48">[47]</a></sup></p>
<h3><span class="mw-headline" id="Root_tree">Root tree</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Btrfs&amp;action=edit&amp;section=12" title="Edit section: Root tree">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<table class="plainlinks metadata ambox ambox-content ambox-Unreferenced" role="presentation">

<p>Every tree appears as an object in the <i>root tree</i> (or <i>tree of tree roots</i>). Some trees, such as file system trees and log trees, have a variable number of instances, each of which is given its own object id. Trees which are <a href="/wiki/Singleton_pattern" title="Singleton pattern">singletons</a> (the data relocation, extent and chunk trees) are assigned special, fixed object ids ≤256. The root tree appears in itself as a tree with object id 1.</p>
<p>Trees refer to each other by object id. They may also refer to individual nodes in other trees as a triplet of the tree's object id, the node's level within the tree and its leftmost key value. Such references are independent of where the tree is actually stored.</p>
<h3><span class="mw-headline" id="File_system_tree">File system tree</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Btrfs&amp;action=edit&amp;section=13" title="Edit section: File system tree">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>User-visible files and directories are contained in a <i>file system tree</i>. There is one file system tree per subvolume. Subvolumes can nest, in which case they appear as a directory item (described below) whose data is a reference to the nested subvolume's file system tree.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (January 2017)">citation needed</span></a></i>]</sup></p>
<p>Within the file system tree, each file and directory object has an <i><a href="/wiki/Inode" title="Inode">inode</a> item</i>. <a class="mw-redirect" href="/wiki/Extended_attribute" title="Extended attribute">Extended attributes</a> and <a href="/wiki/Access_control_list" title="Access control list">ACL</a> entries are stored alongside in separate items.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (January 2017)">citation needed</span></a></i>]</sup></p>
<p>Within each directory, directory entries appear as <i>directory items</i>, whose right-hand key values are a <a href="/wiki/Cyclic_redundancy_check" title="Cyclic redundancy check">CRC32C</a> hash of their filename. Their data is a <i>location key</i>, or the key of the inode item it points to. Directory items together can thus act as an index for path-to-inode lookups, but are not used for iteration because they are sorted by their hash, effectively <a href="/wiki/Random_permutation" title="Random permutation">randomly permuting</a> them. This means user applications iterating over and opening files in a large directory would thus generate many more disk seeks between non-adjacent files—a notable performance drain in other file systems with hash-ordered directories such as <a href="/wiki/ReiserFS" title="ReiserFS">ReiserFS</a>,<sup class="reference" id="cite_ref-68"><a href="#cite_note-68">[67]</a></sup> ext3 (with Htree-indexes enabled<sup class="reference" id="cite_ref-69"><a href="#cite_note-69">[68]</a></sup>) and ext4, all of which have <a href="/wiki/Tiny_Encryption_Algorithm" title="Tiny Encryption Algorithm">TEA</a>-hashed filenames. To avoid this, each directory entry has a <i>directory index item</i>, whose right-hand key value of the item is set to a per-directory counter that increments with each new directory entry. Iteration over these index items thus returns entries in roughly the same order as they are stored on disk.</p>
<p>Besides inode items, files and directories also have a <i>reference item</i> whose right-hand key value is the object id of their parent directory. The data part of the reference item is the filename that inode is known by in that directory. This allows upward traversal through the directory hierarchy by providing a way to map inodes back to paths.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (January 2017)">citation needed</span></a></i>]</sup></p>
<p>Files with hard links in multiple directories have multiple reference items, one for each parent directory. Files with multiple hard links in the <i>same</i> directory pack all of the links' filenames into the same reference item. This was a design flaw that limited the number of same-directory hard links to however many could fit in a single tree block. (On the default block size of 4 KiB, an average filename length of 8 bytes and a per-filename header of 4 bytes, this would be less than 350.) Applications which made heavy use of multiple same-directory hard links, such as <a class="mw-redirect" href="/wiki/Git_(software)" title="Git (software)">git</a>, <a href="/wiki/Gnus" title="Gnus">GNUS</a>, <a href="/wiki/MAME" title="MAME">GMame</a> and <a href="/wiki/BackupPC" title="BackupPC">BackupPC</a> were later observed to fail after hitting this limit.<sup class="reference" id="cite_ref-hard_link_limit_70-0"><a href="#cite_note-hard_link_limit-70">[69]</a></sup> The limit was eventually removed<sup class="reference" id="cite_ref-71"><a href="#cite_note-71">[70]</a></sup> (and as of October 2012 has been merged<sup class="reference" id="cite_ref-72"><a href="#cite_note-72">[71]</a></sup> pending release in Linux 3.7) by introducing spillover <i>extended reference items</i> to hold hard link filenames which could not otherwise fit.</p>
<h4><span class="mw-headline" id="Extents">Extents</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Btrfs&amp;action=edit&amp;section=14" title="Edit section: Extents">edit</a><span class="mw-editsection-bracket">]</span></span></h4>

<p>File data are kept outside the tree in <i>extents</i>, which are contiguous runs of disk blocks. Extent blocks default to 4KiB in size, do not have headers and contain only (possibly compressed) file data. In compressed extents, individual blocks are not compressed separately; rather, the compression stream spans the entire extent.</p>
<p>Files have <i>extent data items</i> to track the extents which hold their contents. The item's right-hand key value is the starting byte offset of the extent. This makes for efficient seeks in large files with many extents, because the correct extent for any given file offset can be computed with just one tree lookup.</p>
<p>Snapshots and cloned files share extents. When a small part of a large such extent is overwritten, the resulting copy-on-write may create three new extents: a small one containing the overwritten data, and two large ones with unmodified data on either side of the overwrite. To avoid having to re-write unmodified data, the copy-on-write may instead create <i>bookend extents</i>, or extents which are simply slices of existing extents. Extent data items allow for this by including an offset into the extent they are tracking: items for bookends are those with non-zero offsets.<sup class="reference" id="cite_ref-btrfs-wiki-1_67-1"><a href="#cite_note-btrfs-wiki-1-67">[66]</a></sup></p>
<p>If the file data is small enough to fit inside a tree node, it is instead pulled in-tree and stored inline in the extent data item. Each tree node is stored in its own <i>tree block</i>—a single uncompressed block with a header. The tree block is regarded as a free-standing, single-block extent.</p>
<h3><span class="mw-headline" id="Extent_allocation_tree">Extent allocation tree</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Btrfs&amp;action=edit&amp;section=15" title="Edit section: Extent allocation tree">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<table class="plainlinks metadata ambox ambox-content ambox-Refimprove" role="presentation">
<tr>
<td class="mbox-image">

<p>The <i>extent allocation tree</i> acts as an allocation map for the file system. Unlike other trees, items in this tree do not have object ids and represent regions of space: their left-hand and right-hand key values are the starting offsets and lengths of the regions they represent.</p>
<p>The file system zones its allocated space into <i>block groups</i>, which are variable-sized allocation regions that alternate successively between preferring metadata extents (tree nodes) and data extents (file contents). The default ratio of data to metadata block groups is 1:2. They are intended to work like the <a href="/wiki/Orlov_block_allocator" title="Orlov block allocator">Orlov block allocator</a> and block groups in ext3 in allocating related files together and resisting fragmentation by leaving allocation gaps between groups. (ext3 block groups, however, have fixed locations computed from the size of the file system, whereas those in Btrfs are dynamic and are created as needed.) Each block group is associated with a <i>block group item</i>. Inode items in the file system tree include a reference to their current block group.<sup class="reference" id="cite_ref-btrfs-wiki-1_67-2"><a href="#cite_note-btrfs-wiki-1-67">[66]</a></sup></p>
<p><i>Extent items</i> contain a back-reference to the tree node or file occupying that extent. There may be multiple back-references if the extent is shared between snapshots. If there are too many back-references to fit in the item, they spill out into individual <i>extent data reference items</i>. Tree nodes, in turn, have back-references to their containing trees. This makes it possible to find which extents or tree nodes are in any region of space by doing a B-tree range lookup on a pair of offsets bracketing that region, then following the back-references. For relocating data, this allows an efficient upwards traversal from the relocated blocks to quickly find and fix all downwards references to those blocks, without having to walk the entire file system. This, in turn, allows the file system to efficiently shrink, migrate and defragment its storage online.</p>
<p>The extent allocation tree, as with all other trees in the file system, is copy-on-write. Writes to the file system may thus cause a cascade whereby changed tree nodes and file data result in new extents being allocated, causing the extent tree to itself change. To avoid creating a <a class="mw-redirect" href="/wiki/Feedback_loop" title="Feedback loop">feedback loop</a>, extent tree nodes which are still in memory but not yet committed to disk may be updated in-place to reflect new copy-on-written extents.</p>
<p>In theory, the extent allocation tree makes a conventional <a class="mw-redirect" href="/wiki/Free-space_bitmap" title="Free-space bitmap">free-space bitmap</a> unnecessary because the extent allocation tree acts as a B-tree version of a <a href="/wiki/Binary_space_partitioning" title="Binary space partitioning">BSP tree</a>. In practice, however, an in-memory <a class="mw-redirect" href="/wiki/Red-black_tree" title="Red-black tree">red-black tree</a> of <a class="mw-redirect" href="/wiki/Page_(computing)" title="Page (computing)">page</a>-sized bitmaps is used to speed up allocations. These bitmaps are persisted to disk (starting in Linux 2.6.37, via the <code>space_cache</code> mount option<sup class="reference" id="cite_ref-73"><a href="#cite_note-73">[72]</a></sup>) as special extents that are exempt from checksumming and copy-on-write. The extent items tracking these extents are stored in the root tree.</p>
<h3><span class="mw-headline" id="Checksum_tree_and_scrubbing">Checksum tree and scrubbing</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Btrfs&amp;action=edit&amp;section=16" title="Edit section: Checksum tree and scrubbing">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">See also: <a class="mw-redirect" href="/wiki/Silent_data_corruption" title="Silent data corruption">Silent data corruption</a></div>
<p><a class="mw-redirect" href="/wiki/CRC-32C" title="CRC-32C">CRC-32C</a> checksums are computed for both data and metadata and stored as <i>checksum items</i> in a <i>checksum tree</i>. There is room of 256 bits for metadata checksums and up to a full leaf block (roughly 4 KB or more) for data checksums. More checksum algorithm options are planned for the future.<sup class="reference" id="cite_ref-btrfs-features-list_31-2"><a href="#cite_note-btrfs-features-list-31">[30]</a></sup><sup class="reference" id="cite_ref-74"><a href="#cite_note-74">[73]</a></sup></p>
<p>There is one checksum item per contiguous run of allocated blocks, with per-block checksums packed end-to-end into the item data. If there are more checksums than can fit, they spill rightwards over into another checksum item in a new leaf. If the file system detects a checksum mismatch while reading a block, it first tries to obtain (or create) a good copy of this block from another device –  if internal mirroring or RAID techniques are in use.<sup class="reference" id="cite_ref-oracle-advanced-btrfs_75-0"><a href="#cite_note-oracle-advanced-btrfs-75">[74]</a></sup><sup class="reference" id="cite_ref-76"><a href="#cite_note-76">[75]</a></sup></p>
<p>Btrfs can initiate an online check of the entire file system by triggering a file system scrub job that is performed in the background. The scrub job scans the entire file system for integrity and automatically attempts to report and repair any bad blocks it finds along the way.<sup class="reference" id="cite_ref-oracle-advanced-btrfs_75-1"><a href="#cite_note-oracle-advanced-btrfs-75">[74]</a></sup><sup class="reference" id="cite_ref-77"><a href="#cite_note-77">[76]</a></sup></p>
<h3><span class="mw-headline" id="Log_tree">Log tree</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Btrfs&amp;action=edit&amp;section=17" title="Edit section: Log tree">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>An <a class="mw-redirect" href="/wiki/Fsync" title="Fsync">fsync</a> is a request to commit modified data immediately to stable storage. fsync-heavy workloads (such as databases) could potentially generate a great deal of redundant write I/O by forcing the file system to repeatedly copy-on-write and flush frequently modified parts of trees to storage. To avoid this, a temporary per-subvolume <i>log tree</i> is created to <a href="/wiki/Journaling_file_system" title="Journaling file system">journal</a> fsync-triggered copy-on-writes. Log trees are self-contained, tracking their own extents and keeping their own checksum items. Their items are replayed and deleted at the next full tree commit or (if there was a system crash) at the next remount.</p>
<h3><span class="mw-headline" id="Chunk_and_device_trees">Chunk and device trees</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Btrfs&amp;action=edit&amp;section=18" title="Edit section: Chunk and device trees">edit</a><span class="mw-editsection-bracket">]</span></span></h3>

<p><a class="mw-redirect" href="/wiki/Block_device" title="Block device">Block devices</a> are divided into <i>chunks</i> of 256 MB or more. Chunks may be mirrored or striped across multiple devices. The mirroring/striping arrangement is transparent to the rest of the file system, which simply sees the single, logical address space that chunks are mapped into.</p>
<p>This is all tracked by the <i>chunk tree</i>, where each device is represented as a <i>device item</i> and each mapping from a logical chunk to its underlying physical chunks is stored in a <i>chunk map item</i>. The <i>device tree</i> is the inverse of the chunk tree, and contains <i>device extent items</i> which map byte ranges of block devices back to individual chunks. As in the extent allocation tree, this allows Btrfs to efficiently shrink or remove devices from volumes by locating the chunks they contain (and relocating their contents).</p>
<p>The file system, chunks and devices are all assigned a <a class="mw-redirect" href="/wiki/Universally_Unique_Identifier" title="Universally Unique Identifier">Universally Unique Identifier</a> (UUID). The header of every tree node contains both the UUID of its containing chunk and the UUID of the file system. The chunks containing the chunk tree, the root tree, device tree and extent tree are always mirrored—even on single-device volumes. These are all intended to improve the odds of successful data salvage in the event of <a href="/wiki/Bad_sector" title="Bad sector">media errors</a>.</p>
<h3><span class="mw-headline" id="Relocation_trees">Relocation trees</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Btrfs&amp;action=edit&amp;section=19" title="Edit section: Relocation trees">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Defragmentation, shrinking and rebalancing operations require extents to be relocated. However, doing a simple copy-on-write of the relocating extent will break sharing between snapshots and consume disk space. To preserve sharing, an update-and-swap algorithm is used, with a special <i>relocation tree</i> serving as scratch space for affected metadata. The extent to be relocated is first copied to its destination. Then, by following backreferences upward through the affected subvolume's file system tree, metadata pointing to the old extent is progressively updated to point at the new one; any newly updated items are stored in the relocation tree. Once the update is complete, items in the relocation tree are swapped with their counterparts in the affected subvolume, and the relocation tree is discarded.<sup class="reference" id="cite_ref-78"><a href="#cite_note-78">[77]</a></sup></p>
<h3><span class="mw-headline" id="Superblock">Superblock</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Btrfs&amp;action=edit&amp;section=20" title="Edit section: Superblock">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>All the file system's trees—including the chunk tree itself—are stored in chunks, creating a potential <a href="/wiki/Chicken_or_the_egg" title="Chicken or the egg">chicken-and-egg</a> problem when <a href="/wiki/Mount_(computing)" title="Mount (computing)">mounting</a> the file system. To <a class="mw-redirect" href="/wiki/Bootstrapping_(computing)" title="Bootstrapping (computing)">bootstrap</a> into a mount, a list of physical addresses of chunks belonging to the chunk and root trees must be stored in the <i><a class="mw-redirect" href="/wiki/Block_device" title="Block device">superblock</a></i>.<sup class="reference" id="cite_ref-79"><a href="#cite_note-79">[78]</a></sup></p>
<p><i>Superblock mirrors</i> are kept at fixed locations:<sup class="reference" id="cite_ref-80"><a href="#cite_note-80">[79]</a></sup> 64 KiB into every block device, with additional copies at 64 MiB, 256 GiB and 1 PiB. When a superblock mirror is updated, its <i>generation number</i> is incremented. At mount time, the copy with the highest generation number is used. All superblock mirrors are updated in tandem, except in <a href="/wiki/Solid-state_drive" title="Solid-state drive">SSD</a> mode which alternates updates among mirrors to provide some <a class="mw-redirect" href="/wiki/Wear_levelling" title="Wear levelling">wear levelling</a>.</p>
