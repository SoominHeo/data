<p>The <b>Spring Framework</b> is an <a href="/wiki/Application_framework" title="Application framework">application framework</a> and <a href="/wiki/Inversion_of_control" title="Inversion of control">inversion of control</a> <a class="mw-redirect" href="/wiki/Servlet_container" title="Servlet container">container</a> for the <a class="mw-redirect" href="/wiki/Java_platform" title="Java platform">Java platform</a>. The framework's core features can be used by any Java application, but there are extensions for building web applications on top of the <a class="mw-redirect" href="/wiki/Java_EE" title="Java EE">Java EE</a> platform. Although the framework does not impose any specific <a href="/wiki/Programming_model" title="Programming model">programming model</a>, it has become popular in the Java community as an alternative to, replacement for, or even addition to the <a href="/wiki/Enterprise_JavaBeans" title="Enterprise JavaBeans">Enterprise JavaBeans</a> (EJB) model. The Spring Framework is <a class="mw-redirect" href="/wiki/Open_source" title="Open source">open source</a>.</p>

<h2><span class="mw-headline" id="Version_history">Version history</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Spring_Framework&amp;action=edit&amp;section=1" title="Edit section: Version history">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The first version was written by <a href="/wiki/Rod_Johnson_(programmer)" title="Rod Johnson (programmer)">Rod Johnson</a>, who released the framework with the publication of his book <i>Expert One-on-One J2EE Design and Development</i> in October 2002. The framework was first released under the <a href="/wiki/Apache_License" title="Apache License">Apache 2.0 license</a> in June 2003. The first milestone release, 1.0, was released in March 2004, with further milestone releases in September 2004 and March 2005. The Spring 1.2.6 framework won a <a class="mw-redirect" href="/wiki/Jolt_Awards" title="Jolt Awards">Jolt productivity award</a> and a <a class="new" href="/w/index.php?title=JAX_Innovation_Award&amp;action=edit&amp;redlink=1" title="JAX Innovation Award (page does not exist)">JAX Innovation Award</a> in 2006.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup><sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup> Spring 2.0 was released in October 2006, Spring 2.5 in November 2007, Spring 3.0 in December 2009, Spring 3.1 in December 2011, and Spring 3.2.5 in November 2013.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup> Spring Framework 4.0 was released in December 2013.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> Notable improvements in Spring 4.0 included support for Java SE 8, <a href="/wiki/Groovy_(programming_language)" title="Groovy (programming language)">Groovy</a> 2, some aspects of Java EE7, and <a href="/wiki/WebSocket" title="WebSocket">WebSocket</a>.</p>
<p>Spring Framework 4.2.0 was released on 31 July 2015 and was immediately upgraded to version 4.2.1, which was released on 01 Sept 2015.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup> It is <i>"compatible with Java 6, 7 and 8, with a focus on core refinements and modern web capabilities"</i>.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup></p>
<p>Spring Framework 4.3 has been released on 10 June 2016. The 4.3.0.RC1 <sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> version is available. It <i>"will be the final generation within the general Spring 4 system requirements (Java 6+, Servlet 2.5+), getting prepared for an extended 4.3.x support life until 2019"</i>.<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup></p>
<p>Spring 5 is announced to be built upon <a href="/wiki/Reactive_Streams" title="Reactive Streams">Reactive Streams</a> compatible Reactor Core.<sup class="reference" id="cite_ref-10"><a href="#cite_note-10">[10]</a></sup></p>
<h2><span class="mw-headline" id="Modules">Modules</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Spring_Framework&amp;action=edit&amp;section=2" title="Edit section: Modules">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The Spring Framework includes several modules that provide a range of services:</p>
<ul>
<li>Spring Core Container: this is the base module of Spring and provides spring containers (BeanFactory and ApplicationContext).<sup class="reference" id="cite_ref-11"><a href="#cite_note-11">[11]</a></sup></li>
<li><a href="/wiki/Aspect-oriented_programming" title="Aspect-oriented programming">Aspect-oriented programming</a>: enables implementing <a href="/wiki/Cross-cutting_concern" title="Cross-cutting concern">cross-cutting concerns</a>.</li>
<li><a href="/wiki/Authentication" title="Authentication">Authentication</a> and <a href="/wiki/Authorization" title="Authorization">authorization</a>: configurable security processes that support a range of standards, protocols, tools and practices via the <a href="/wiki/Spring_Security" title="Spring Security">Spring Security</a> sub-project (formerly Acegi Security System for Spring).</li>
<li><a href="/wiki/Convention_over_configuration" title="Convention over configuration">Convention over configuration</a>: a rapid application development solution for Spring-based enterprise applications is offered in the <a href="/wiki/Spring_Roo" title="Spring Roo">Spring Roo</a> module</li>
<li><a href="/wiki/Data_access" title="Data access">Data access</a>: working with <a class="mw-redirect" href="/wiki/RDBMS" title="RDBMS">relational database management systems</a> on the Java platform using <a class="mw-redirect" href="/wiki/JDBC" title="JDBC">JDBC</a> and <a href="/wiki/Object-relational_mapping" title="Object-relational mapping">object-relational mapping</a> tools and with <a href="/wiki/NoSQL" title="NoSQL">NoSQL</a> databases</li>
<li><a href="/wiki/Inversion_of_control" title="Inversion of control">Inversion of control</a> container: configuration of application components and lifecycle management of Java objects, done mainly via <a href="/wiki/Dependency_injection" title="Dependency injection">dependency injection</a></li>
<li>Messaging: configurative registration of message listener objects for transparent message-consumption from <a href="/wiki/Message_queue" title="Message queue">message queues</a> via <a href="/wiki/Java_Message_Service" title="Java Message Service">JMS</a>, improvement of message sending over standard JMS APIs</li>
<li><a href="/wiki/Model%E2%80%93view%E2%80%93controller" title="Model–view–controller">Model–view–controller</a>: an <a class="mw-redirect" href="/wiki/HTTP" title="HTTP">HTTP</a>- and <a class="mw-redirect" href="/wiki/Java_Servlet_API" title="Java Servlet API">servlet</a>-based framework providing hooks for extension and customization for web applications and <a class="mw-redirect" href="/wiki/REST" title="REST">RESTful</a> Web services.</li>
<li>Remote access framework: configurative <a href="/wiki/Remote_procedure_call" title="Remote procedure call">RPC</a>-style <a href="/wiki/Marshalling_(computer_science)" title="Marshalling (computer science)">marshalling</a> of Java objects over networks supporting <a href="/wiki/Java_remote_method_invocation" title="Java remote method invocation">RMI</a>, <a class="mw-redirect" href="/wiki/CORBA" title="CORBA">CORBA</a> and <a class="mw-redirect" href="/wiki/HTTP" title="HTTP">HTTP</a>-based protocols including <a class="mw-redirect" href="/wiki/Web_services" title="Web services">Web services</a> (<a class="mw-redirect" href="/wiki/SOAP_(protocol)" title="SOAP (protocol)">SOAP</a>)</li>
<li><a href="/wiki/Transaction_processing" title="Transaction processing">Transaction management</a>: unifies several transaction management APIs and coordinates transactions for Java objects</li>
<li>Remote management: configurative exposure and management of Java objects for local or remote configuration via <a href="/wiki/Java_Management_Extensions" title="Java Management Extensions">JMX</a></li>
<li><a href="/wiki/Software_testing" title="Software testing">Testing</a>: support classes for writing unit tests and integration tests</li>
</ul>
<h3><span class="mw-headline" id="Inversion_of_control_container_.28dependency_injection.29">Inversion of control container (dependency injection)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Spring_Framework&amp;action=edit&amp;section=3" title="Edit section: Inversion of control container (dependency injection)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Central to the Spring Framework is its inversion of control (IoC) container, which provides a consistent means of configuring and managing Java objects using <a class="mw-redirect" href="/wiki/Reflection_(computer_science)" title="Reflection (computer science)">reflection</a>. The container is responsible for managing <a href="/wiki/Object_lifetime" title="Object lifetime">object lifecycles</a> of specific objects: creating these objects, calling their initialization methods, and configuring these objects by wiring them together.</p>
<p>Objects created by the container are also called managed objects or <a href="/wiki/JavaBeans" title="JavaBeans">beans</a>. The container can be configured by loading XML files or detecting specific <a href="/wiki/Java_annotation" title="Java annotation">Java annotations</a> on configuration classes. These data sources contain the bean definitions that provide the information required to create the beans.</p>
<p>Objects can be obtained by means of either dependency lookup or dependency injection.<sup class="reference" id="cite_ref-12"><a href="#cite_note-12">[12]</a></sup> Dependency lookup is a pattern where a caller asks the container object for an object with a specific name or of a specific type. Dependency injection is a pattern where the container passes objects by name to other objects, via either <a class="mw-redirect" href="/wiki/Constructor_(computer_science)" title="Constructor (computer science)">constructors</a>, <a href="/wiki/Property_(programming)" title="Property (programming)">properties</a>, or <a href="/wiki/Factory_method_pattern" title="Factory method pattern">factory methods</a>.</p>
<p>In many cases one need not use the container when using other parts of the Spring Framework, although using it will likely make an application easier to configure and customize. The Spring container provides a consistent mechanism to configure applications and integrates with almost all Java environments, from small-scale applications to large enterprise applications.</p>
<p>The container can be turned into a partially compliant <a href="/wiki/Enterprise_JavaBeans#EJB_3.0.2C_final_release_.282006-05-11.29" title="Enterprise JavaBeans">EJB 3.0</a> container by means of the Pitchfork project. Some<sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Manual_of_Style/Words_to_watch#Unsupported_attributions" title="Wikipedia:Manual of Style/Words to watch"><span title="The material near this tag possibly uses too-vague attribution or weasel words. (May 2011)">who?</span></a></i>]</sup> criticize the Spring Framework for not complying with standards.<sup class="reference" id="cite_ref-13"><a href="#cite_note-13">[13]</a></sup> However, SpringSource doesn't see EJB 3 compliance as a major goal, and claims that the Spring Framework and the container allow for more powerful programming models.<sup class="reference" id="cite_ref-14"><a href="#cite_note-14">[14]</a></sup> You do not create an object, but describe how they should be created, by defining it in the Spring configuration file. You do not call services and components, but tell which services and components must be called, by defining them in the Spring configuration files. This makes the code easy to maintain and easier to test through IoC.</p>
<h3><span class="mw-headline" id="Aspect-oriented_programming_framework">Aspect-oriented programming framework</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Spring_Framework&amp;action=edit&amp;section=4" title="Edit section: Aspect-oriented programming framework">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The Spring Framework has its own Aspect-oriented programming (AOP) framework that modularizes cross-cutting concerns in <a class="mw-redirect" href="/wiki/Aspect_(computer_science)" title="Aspect (computer science)">aspects</a>. The motivation for creating a separate AOP framework comes from the belief that it would be possible to provide basic AOP features without too much complexity in either design, implementation, or configuration. The Spring AOP framework also takes full advantage of the Spring container.</p>
<p>The Spring AOP framework is <a href="/wiki/Proxy_pattern" title="Proxy pattern">proxy pattern-based</a>, and is configured at <a href="/wiki/Run_time_(program_lifecycle_phase)" title="Run time (program lifecycle phase)">run time</a>. This removes the need for a compilation step or load-time weaving. On the other hand, interception only allows for public method-execution on existing objects at a <a href="/wiki/Join_point" title="Join point">join point</a>.</p>
<p>Compared to the <a href="/wiki/AspectJ" title="AspectJ">AspectJ</a> framework, Spring AOP is less powerful, but also less complicated. Spring 1.2 includes support to configure AspectJ aspects in the container. Spring 2.0 added more integration with AspectJ; for example, the <a href="/wiki/Pointcut" title="Pointcut">pointcut</a> language is reused and can be mixed with Spring AOP-based aspects. Further, Spring 2.0 added a Spring Aspects library that uses AspectJ to offer common Spring features such as declarative transaction management and dependency injection via AspectJ compile-time or load-time weaving. <a href="/wiki/SpringSource" title="SpringSource">SpringSource</a> also uses AspectJ AOP in other Spring projects such as Spring Roo and Spring Insight, with Spring Security also offering an AspectJ-based aspect library.</p>
<p>Spring AOP has been designed to make it able to work with cross-cutting concerns inside the Spring Framework. Any object which is created and configured by the container can be enriched using Spring AOP.</p>
<p>The Spring Framework uses Spring AOP internally for transaction management, security, remote access, and <a href="/wiki/Java_Management_Extensions" title="Java Management Extensions">JMX</a>.</p>
<p>Since version 2.0 of the framework, Spring provides two approaches to the AOP configuration:</p>
<ul>
<li>schema-based approach<sup class="reference" id="cite_ref-15"><a href="#cite_note-15">[15]</a></sup> and</li>
<li>@AspectJ-based annotation style.<sup class="reference" id="cite_ref-16"><a href="#cite_note-16">[16]</a></sup></li>
</ul>

<p>The Spring team decided not to introduce new AOP-related terminology; therefore, in the Spring reference documentation and API, terms such as <a href="/wiki/Aspect_(computer_programming)" title="Aspect (computer programming)">aspect</a>, join point, <a class="mw-redirect" href="/wiki/Advice_(computer_science)" title="Advice (computer science)">advice</a>, pointcut, introduction, <a class="new" href="/w/index.php?title=Target_object&amp;action=edit&amp;redlink=1" title="Target object (page does not exist)">target object</a> (<a class="new" href="/w/index.php?title=Advised_object&amp;action=edit&amp;redlink=1" title="Advised object (page does not exist)">advised object</a>), <a class="new" href="/w/index.php?title=AOP_proxy&amp;action=edit&amp;redlink=1" title="AOP proxy (page does not exist)">AOP proxy</a>, and weaving all have the same meanings as in most other AOP frameworks (particularly AspectJ).</p>
<h3><span class="mw-headline" id="Data_access_framework">Data access framework</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Spring_Framework&amp;action=edit&amp;section=5" title="Edit section: Data access framework">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Spring's data access framework addresses common difficulties developers face when working with databases in applications. Support is provided for all popular data access frameworks in Java: JDBC, <a href="/wiki/IBATIS" title="IBATIS">iBatis</a>/<a href="/wiki/MyBatis" title="MyBatis">MyBatis</a>, <a class="mw-redirect" href="/wiki/Hibernate_(Java)" title="Hibernate (Java)">Hibernate</a>, <a href="/wiki/Java_Data_Objects" title="Java Data Objects">JDO</a>, <a href="/wiki/Java_Persistence_API" title="Java Persistence API">JPA</a>, <a href="/wiki/TopLink" title="TopLink">Oracle TopLink</a>, <a class="mw-redirect" href="/wiki/Ojb" title="Ojb">Apache OJB</a>, and <a href="/wiki/Apache_Cayenne" title="Apache Cayenne">Apache Cayenne</a>, among others.</p>
<p>For all of these supported frameworks, Spring provides these features</p>
<ul>
<li>Resource management - automatically acquiring and releasing database resources</li>
<li>Exception handling - translating data access related exception to a Spring data access hierarchy</li>
<li>Transaction participation - transparent participation in ongoing transactions</li>
<li>Resource unwrapping - retrieving database objects from connection pool wrappers</li>
<li>Abstraction for <a href="/wiki/Binary_large_object" title="Binary large object">BLOB</a> and <a href="/wiki/Character_large_object" title="Character large object">CLOB</a> handling</li>
</ul>
<p>All these features become available when using <a href="/wiki/Template_method_pattern" title="Template method pattern">template classes</a> provided by Spring for each supported framework. Critics have said these template classes are intrusive and offer no advantage over using (for example) the Hibernate API directly.<sup class="reference" id="cite_ref-17"><a href="#cite_note-17">[17]</a></sup><sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Verifiability" title="Wikipedia:Verifiability"><span title="The material near this tag failed verification of its source citation(s). (January 2012)">not in citation given</span></a></i>]</sup> In response, the Spring developers have made it possible to use the Hibernate and JPA APIs directly. This however requires transparent transaction management, as application code no longer assumes the responsibility to obtain and close database resources, and does not support exception translation.</p>
<p>Together with Spring's transaction management, its data access framework offers a flexible abstraction for working with data access frameworks. The Spring Framework doesn't offer a common data access API; instead, the full power of the supported APIs is kept intact. The Spring Framework is the only framework available in Java that offers managed data access environments outside of an application server or container.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (December 2007)">citation needed</span></a></i>]</sup></p>
<p>While using Spring for transaction management with Hibernate, the following beans may have to be configured:</p>
<ul>
<li>A <a href="/wiki/Datasource" title="Datasource">Data Source</a> like <code>com.mchange.v2.c3p0.ComboPooledDataSource</code> or <code>org.apache.commons.dbcp.BasicDataSource</code></li>
<li>A SessionFactory like <code>org.springframework.orm.hibernate3.LocalSessionFactoryBean</code> with a DataSource attribute</li>
<li>A HibernateProperties like <code>org.springframework.beans.factory.config.PropertiesFactoryBean</code></li>
<li>A TransactionManager like <code>org.springframework.orm.hibernate3.HibernateTransactionManager</code> with a SessionFactory attribute</li>
</ul>
<p>Other points of configuration include:</p>
<ul>
<li>An AOP configuration of cutting points.</li>
<li><span class="clarify-content" style="padding-left:0.1em; padding-right:0.1em; color:#595959; border:1px solid #DDD;">Transaction semantics of AOP advice</span><sup class="noprint Inline-Template Template-Clarify" style="margin-left:0.1em; white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Please_clarify" title="Wikipedia:Please clarify"><span title="This passage needs to be better explained. (February 2013)">clarify</span></a></i>]</sup>.</li>
</ul>
<h3><span class="mw-headline" id="Transaction_management_framework">Transaction management framework</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Spring_Framework&amp;action=edit&amp;section=6" title="Edit section: Transaction management framework">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Spring's transaction management framework brings an abstraction mechanism to the Java platform. Its abstraction is capable of:</p>
<ul>
<li>working with local and <a href="/wiki/Distributed_transaction" title="Distributed transaction">global transactions</a> (local transaction does not require an <a href="/wiki/Application_server" title="Application server">application server</a>)</li>
<li>working with <a href="/wiki/Nested_transaction" title="Nested transaction">nested transactions</a></li>
<li>working with <a href="/wiki/Savepoint" title="Savepoint">savepoints</a></li>
<li>working in almost all environments of the Java platform</li>
</ul>
<p>In comparison, <a href="/wiki/Java_Transaction_API" title="Java Transaction API">JTA</a> only supports nested transactions and global transactions, and requires an application server (and in some cases also deployment of applications in an application server).</p>
<p>The Spring Framework ships a PlatformTransactionManager for a number of transaction management strategies:</p>
<ul>
<li>Transactions managed on a JDBC Connection</li>
<li>Transactions managed on Object-relational mapping Units of Work</li>
<li>Transactions managed via the JTA TransactionManager and UserTransaction</li>
<li>Transactions managed on other resources, like <a href="/wiki/Object_database" title="Object database">object databases</a></li>
</ul>
<p>Next to this abstraction mechanism the framework also provides two ways of adding transaction management to applications:</p>
<ul>
<li>Programmatically, by using Spring's TransactionTemplate</li>
<li>Configuratively, by using <a href="/wiki/Metadata" title="Metadata">metadata</a> like XML or Java annotations (@Transactional, etc.)</li>
</ul>
<p>Together with Spring's data access framework — which integrates the transaction management framework — it is possible to set up a transactional system through configuration without having to rely on JTA or <a class="mw-redirect" href="/wiki/EJB" title="EJB">EJB</a>. The transactional framework also integrates with messaging and <a href="/wiki/Cache_(computing)" title="Cache (computing)">caching</a> engines.</p>
<h3><span class="mw-headline" id="Model.E2.80.93view.E2.80.93controller_framework">Model–view–controller framework</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Spring_Framework&amp;action=edit&amp;section=7" title="Edit section: Model–view–controller framework">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The Spring Framework features its own <a href="/wiki/Model%E2%80%93view%E2%80%93controller" title="Model–view–controller">MVC</a> <a class="mw-redirect" href="/wiki/Web_application_framework" title="Web application framework">web application framework</a>, which wasn't originally planned. The Spring developers decided to write their own Web framework as a reaction to what they perceived as the poor design of the (then) popular <a class="mw-redirect" href="/wiki/Jakarta_Struts" title="Jakarta Struts">Jakarta Struts</a> Web framework,<sup class="reference" id="cite_ref-18"><a href="#cite_note-18">[18]</a></sup> as well as deficiencies in other available frameworks. In particular, they felt there was insufficient separation between the presentation and request handling layers, and between the request handling layer and the model.<sup class="reference" id="cite_ref-19"><a href="#cite_note-19">[19]</a></sup></p>
<p>Like Struts, Spring MVC is a request-based framework. The framework defines <a href="/wiki/Strategy_pattern" title="Strategy pattern">strategy</a> interfaces for all of the responsibilities that must be handled by a modern request-based framework. The goal of each interface is to be simple and clear so that it's easy for Spring MVC users to write their own implementations, if they so choose. MVC paves the way for cleaner front end code. All interfaces are tightly coupled to the <a class="mw-redirect" href="/wiki/Java_Servlet" title="Java Servlet">Servlet API</a>. This tight coupling to the Servlet API is seen by some as a failure on the part of the Spring developers to offer a high-level abstraction for Web-based applications<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (February 2007)">citation needed</span></a></i>]</sup>. However, this coupling makes sure that the features of the Servlet API remain available to developers while offering a high abstraction framework to ease working with said API.</p>
<p>The DispatcherServlet class is the <a href="/wiki/Front_controller" title="Front controller">front controller</a><sup class="reference" id="cite_ref-20"><a href="#cite_note-20">[20]</a></sup> of the framework and is responsible for delegating control to the various interfaces during the execution phases of an <a href="/wiki/Hypertext_Transfer_Protocol" title="Hypertext Transfer Protocol">HTTP request</a>.</p>
<p>The most important interfaces defined by Spring MVC, and their responsibilities, are listed below:</p>
<ul>
<li>Controller: comes between Model and View to manage incoming requests and redirect to proper response. It acts as a gate that directs the incoming information. It switches between going into model or view.</li>
<li>HandlerAdapter: execution of objects that handle incoming requests</li>
<li>HandlerInterceptor: interception of incoming requests comparable, but not equal to Servlet filters (use is optional and not controlled by DispatcherServlet).</li>
<li>HandlerMapping: selecting objects that handle incoming requests (handlers) based on any attribute or condition internal or external to those requests</li>
<li>LocaleResolver: resolving and optionally saving of the <a href="/wiki/Locale_(computer_software)" title="Locale (computer software)">locale</a> of an individual user</li>
<li>MultipartResolver: facilitate working with file uploads by wrapping incoming requests</li>
<li>View: responsible for returning a response to the client. Some requests may go straight to view without going to the model part; others may go through all three.</li>
<li>ViewResolver: selecting a View based on a logical name for the view (use is not strictly required)</li>
</ul>
<p>Each strategy interface above has an important responsibility in the overall framework. The abstractions offered by these interfaces are powerful, so to allow for a set of variations in their implementations, Spring MVC ships with implementations of all these interfaces and together offers a feature set on top of the Servlet API. However, developers and vendors are free to write other implementations. Spring MVC uses the Java <code>java.util.Map</code> interface as a data-oriented abstraction for the Model where keys are expected to be string values.</p>
<p>The ease of testing the implementations of these interfaces seems one important advantage of the high level of abstraction offered by Spring MVC. DispatcherServlet is tightly coupled to the Spring inversion of control container for configuring the web layers of applications. However, web applications can use other parts of the Spring Framework—including the container—and choose not to use Spring MVC.</p>
<h3><span class="mw-headline" id="Remote_access_framework">Remote access framework</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Spring_Framework&amp;action=edit&amp;section=8" title="Edit section: Remote access framework">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Spring's Remote Access framework is an abstraction for working with various RPC-based technologies available on the Java platform both for client connectivity and marshalling objects on servers. The most important feature offered by this framework is to ease configuration and usage of these technologies as much as possible by combining inversion of control and AOP.</p>
<p>The framework also provides fault-recovery (automatic reconnection after connection failure) and some optimizations for client-side use of <span class="clarify-content" style="padding-left:0.1em; padding-right:0.1em; color:#595959; border:1px solid #DDD;">EJB remote stateless session beans</span><sup class="noprint Inline-Template Template-Clarify" style="margin-left:0.1em; white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Please_clarify" title="Wikipedia:Please clarify"><span title="This passage needs to be better explained. (February 2013)">clarify</span></a></i>]</sup>.</p>
<p>Spring provides support for these protocols and products out of the box</p>
<ul>
<li>HTTP-based protocols
<ul>
<li><a href="/wiki/Hessian_(Web_service_protocol)" title="Hessian (Web service protocol)">Hessian</a>: binary serialization protocol, open-sourced and maintained by <a href="/wiki/Common_Object_Request_Broker_Architecture" title="Common Object Request Broker Architecture">CORBA</a>-based protocols</li>
<li><a href="/wiki/Java_remote_method_invocation" title="Java remote method invocation">RMI</a> (1): method invocations using RMI infrastructure yet specific to Spring</li>
<li><a href="/wiki/Java_remote_method_invocation" title="Java remote method invocation">RMI</a> (2): method invocations using RMI interfaces complying with regular RMI usage</li>
<li><a href="/wiki/RMI-IIOP" title="RMI-IIOP">RMI-IIOP</a> (<a href="/wiki/Common_Object_Request_Broker_Architecture" title="Common Object Request Broker Architecture">CORBA</a>): method invocations using RMI-IIOP/CORBA</li>
</ul>
</li>
<li><a class="mw-redirect" href="/wiki/Enterprise_JavaBean" title="Enterprise JavaBean">Enterprise JavaBean</a> client integration
<ul>
<li>Local EJB stateless session bean connectivity: connecting to local stateless session beans</li>
<li>Remote EJB stateless session bean connectivity: connecting to remote stateless session beans</li>
</ul>
</li>
<li>SOAP
<ul>
<li>Integration with the <a href="/wiki/Apache_Axis" title="Apache Axis">Apache Axis</a> Web services framework</li>
</ul>
</li>
</ul>
<p><a href="/wiki/Apache_CXF" title="Apache CXF">Apache CXF</a> provides integration with the Spring Framework for RPC-style exporting of object on the server side.</p>
<p>Both client and server setup for all RPC-style protocols and products supported by the Spring Remote access framework (except for the Apache Axis support) is configured in the Spring Core container.</p>
<p>There is alternative open-source implementation (Cluster4Spring) of a remoting subsystem included into Spring Framework that is intended to support various schemes of remoting (1-1, 1-many, dynamic services discovering)…</p>
<h3><span class="mw-headline" id="Convention-over-configuration_rapid_application_development">Convention-over-configuration rapid application development</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Spring_Framework&amp;action=edit&amp;section=9" title="Edit section: Convention-over-configuration rapid application development">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<h4><span class="mw-headline" id="Spring_Boot">Spring Boot</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Spring_Framework&amp;action=edit&amp;section=10" title="Edit section: Spring Boot">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Spring Boot is Spring's <a href="/wiki/Convention_over_configuration" title="Convention over configuration">convention-over-configuration</a> solution for creating stand-alone, production-grade Spring-based Applications that you can "just run".<sup class="reference" id="cite_ref-21"><a href="#cite_note-21">[21]</a></sup> It takes an opinionated view<sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a class="mw-redirect" href="/wiki/MOS:JARGON" title="MOS:JARGON"><span title="The material near this tag may be using jargon that limits the article's accessibility. (September 2016)">jargon</span></a></i>]</sup> of the Spring platform and third-party libraries so you can get started with minimum fuss. Most Spring Boot applications need very little Spring configuration. Features:</p>
<ul>
<li>Create stand-alone Spring applications</li>
<li>Embed Tomcat or Jetty directly (no need to deploy WAR files)</li>
<li>Provide opinionated 'starter' POMs to simplify your Maven configuration</li>
<li>Automatically configure Spring whenever possible</li>
<li>Provide production-ready features such as metrics, health checks and externalized configuration</li>
<li>Absolutely no code generation and no requirement for XML configuration</li>
</ul>
<h4><span class="mw-headline" id="Spring_Roo">Spring Roo</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Spring_Framework&amp;action=edit&amp;section=11" title="Edit section: Spring Roo">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Spring Roo provides an alternative, code-generation based approach at using convention-over-configuration to rapidly build applications in <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>. It currently supports Spring Framework, Spring Security and <a href="/wiki/Spring_Web_Flow" title="Spring Web Flow">Spring Web Flow</a>. Roo differs from other <a href="/wiki/Rapid_application_development" title="Rapid application development">rapid application development</a> frameworks by focusing on:</p>
<ul>
<li>Extensibility (via add-ons)</li>
<li>Java platform productivity (as opposed to other languages)</li>
<li>Lock-in avoidance (Roo can be removed within a few minutes from any application)</li>
<li>Runtime avoidance (with associated deployment advantages)</li>
<li>Usability (particularly via the shell features and usage patterns)</li>
</ul>
<h3><span class="mw-headline" id="Batch_framework">Batch framework</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Spring_Framework&amp;action=edit&amp;section=12" title="Edit section: Batch framework">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/Spring_Batch" title="Spring Batch">Spring Batch</a> is a framework for <a href="/wiki/Batch_processing" title="Batch processing">batch processing</a> that provides reusable functions that are essential in processing large volumes of records, including:</p>
<ul>
<li>logging/tracing</li>
<li>transaction management</li>
<li>job processing statistics</li>
<li>job restart</li>
</ul>
<p>It also provides more advanced technical services and features that will enable extremely high-volume and high performance batch jobs through optimizations and partitioning techniques. Spring Batch is a framework for batch processing – execution of a series of jobs. In Spring Batch, A job consists of many steps and each step consists of a READ-PROCESS-WRITE task or single operation task (tasklet).</p>
<p>For "READ-PROCESS-WRITE" process, it means "read" data from the resources (csv, xml or database), "process" it and "write" it to other resources (csv, xml and database). For example, a step may read data from a CSV file, process it and write it into the database. Spring Batch provides many made Classes to read/write CSV, XML and database.</p>
<p>For "single" operation task (tasklet), it means doing single task only, like clean up the resources after or before a step is started or completed.</p>
<p>And the steps can be chained together to run as a job.</p>
<h3><span class="mw-headline" id="Integration_framework">Integration framework</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Spring_Framework&amp;action=edit&amp;section=13" title="Edit section: Integration framework">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/Spring_Integration" title="Spring Integration">Spring Integration</a> is a framework for <a href="/wiki/Enterprise_application_integration" title="Enterprise application integration">Enterprise application integration</a> that provides reusable functions that are essential in messaging, or event-driven architectures.</p>
<ul>
<li>routers - routes a message to a message channel based on conditions</li>
<li>transformers - converts/transforms/changes the message payload and creates a new message with transformed payload</li>
<li>adapters - to integrate with other technologies and systems (HTTP, <a class="mw-redirect" href="/wiki/AMQP" title="AMQP">AMQP</a>, JMS, <a href="/wiki/XMPP" title="XMPP">XMPP</a>, <a class="mw-redirect" href="/wiki/SMTP" title="SMTP">SMTP</a>, <a class="mw-redirect" href="/wiki/IMAP" title="IMAP">IMAP</a>, <a class="mw-redirect" href="/wiki/FTP" title="FTP">FTP</a> (as well as FTPS/SFTP), file systems, etc.)</li>
<li>filters - filters message based on criteria. If the criteria are not met, message is dropped</li>
<li>service activators - invoke an operation on a service object</li>
<li>management and auditing</li>
</ul>
<p>Spring Integration supports pipe-and-filter based architectures.</p>
<h2><span class="mw-headline" id="Criticisms">Criticisms</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Spring_Framework&amp;action=edit&amp;section=14" title="Edit section: Criticisms">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The Spring Framework has received some criticism for what some developers perceive to be an over-reliance on <a href="/wiki/XML" title="XML">XML</a> by Spring's container. Since version 3.0.0, however, developers have been able to specify all or part of an application context through annotations or Java code. Spring Boot makes heavy use of this to minimize the amount of configuration that must be written. Furthermore, the Spring Tool Suite (STS), built on top of <a href="/wiki/Eclipse_(software)" title="Eclipse (software)">Eclipse</a>, provides code-completion, validation, contextual information, and graphical visualizations when editing Spring XML configuration files.</p>
