<h1 class="firstHeading" id="firstHeading" lang="en">Provable security</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">From Wikipedia, the free encyclopedia</div>
<div id="contentSub"></div>
<div class="mw-jump" id="jump-to-nav">
					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>
</div>
<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en"><script>function mfTempOpenSection(id){var block=document.getElementById("mf-section-"+id);block.className+=" open-block";block.previousSibling.className+=" open-block";}</script><p><b>Provable security</b> refers to any type or level of security that can be proved. It is used in different ways by different fields.</p>
<p>Usually, this refers to mathematical proofs, which are common in cryptography. In such a proof, the capabilities of the attacker are defined by an <a href="/wiki/Adversary_(cryptography)" title="Adversary (cryptography)">adversarial</a> model (also referred to as attacker model): the aim of the proof is to show that the attacker must solve the underlying <a href="/wiki/Computational_complexity_theory#Hard" title="Computational complexity theory">hard problem</a> in order to break the security of the modelled system. Such a proof does not consider <a class="mw-redirect" href="/wiki/Side-channel_attacks" title="Side-channel attacks">side-channel attacks</a> or other implementation-specific attacks, because they are usually impossible to model without implementing the system (and thus, the proof only applies to this implementation).</p>
<p>Outside of cryptography, the term is often used in conjunction with <a href="/wiki/Secure_coding" title="Secure coding">secure coding</a> and <a class="mw-redirect" href="/wiki/Security_by_design" title="Security by design">security by design</a>, both of which can rely on proofs to show the security of a particular approach. As with the cryptographic setting, this involves an attacker model and a model of the system. For example, code can be verified to match the intended functionality, described by a model: this can be done through <a href="/wiki/Static_program_analysis" title="Static program analysis">static checking</a>. These techniques are sometimes used for evaluating products (see <a href="/wiki/Common_Criteria" title="Common Criteria">Common Criteria</a>): the security here depends not only on the correctness of the attacker model, but also on the model of the code.</p>
<p>Finally, the term provable security is sometimes used by sellers of <a class="mw-redirect" href="/wiki/Security_software" title="Security software">security software</a> that are attempting to sell security products like <a href="/wiki/Firewall_(computing)" title="Firewall (computing)">firewalls</a>, <a href="/wiki/Antivirus_software" title="Antivirus software">antivirus software</a> and <a class="mw-redirect" href="/wiki/Intrusion_detection_systems" title="Intrusion detection systems">intrusion detection systems</a>. As these products are typically not subject to scrutiny, many <a class="mw-redirect" href="/wiki/Hacker_(computer_security)" title="Hacker (computer security)">security researchers</a> consider this type of claim to be selling <a href="/wiki/Snake_oil_(cryptography)" title="Snake oil (cryptography)">snakeoil</a>.</p>

<h2><span class="mw-headline" id="In_cryptography">In cryptography</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Provable_security&amp;action=edit&amp;section=1" title="Edit section: In cryptography">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In <a href="/wiki/Cryptography" title="Cryptography">cryptography</a>, a system has <b>provable security</b> if its security requirements can be stated formally in an <a href="/wiki/Adversary_(cryptography)" title="Adversary (cryptography)">adversarial</a> model, as opposed to heuristically, with clear assumptions that the adversary has access to the system as well as enough computational resources. The proof of security (called a "reduction") is that these security requirements are met provided the assumptions about the adversary's access to the system are satisfied and some clearly stated assumptions about the <a href="/wiki/Computational_hardness_assumption" title="Computational hardness assumption">hardness</a> of certain computational tasks hold. An early example of such requirements and proof was given by <a href="/wiki/Shafi_Goldwasser" title="Shafi Goldwasser">Goldwasser</a> and <a href="/wiki/Silvio_Micali" title="Silvio Micali">Micali</a> for <a href="/wiki/Semantic_security" title="Semantic security">semantic security</a> and the construction based on the <a href="/wiki/Quadratic_residuosity_problem" title="Quadratic residuosity problem">quadratic residuosity problem</a>.</p>
<p>There are several lines of research in provable security. One is to establish the "correct" definition of security for a given, intuitively understood task. Another is to suggest constructions and proofs based on general assumptions as much as possible, for instance the existence of a <a href="/wiki/One-way_function" title="One-way function">one-way function</a>. A major open problem is to establish such proofs based on P ≠ NP, since the existence of one-way functions is not known to follow from the P ≠ NP conjecture.</p>
<p>Some proofs of the security are in given theoretical models such as the <a class="mw-redirect" href="/wiki/Random_oracle_model" title="Random oracle model">random oracle model</a>, where real cryptographic hash functions are represented by an idealization. "Exact security" or "<a href="/wiki/Concrete_security" title="Concrete security">concrete security</a>" is the name given to provable security reductions where one quantifies security by computing precise bounds on computational effort, rather than an asymptotic bound which is guaranteed to hold for "sufficiently large" values of the <a href="/wiki/Security_parameter" title="Security parameter">security parameter</a>.</p>
<h3><span class="mw-headline" id="Controversies">Controversies</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Provable_security&amp;action=edit&amp;section=2" title="Edit section: Controversies">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/Neal_Koblitz" title="Neal Koblitz">Koblitz</a> and <a href="/wiki/Alfred_Menezes" title="Alfred Menezes">Menezes</a> have criticized aspects of provable security research in their papers Another Look at "Provable Security"<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> and Another Look at "Provable Security" II.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> These views have been controversial in the community. A rebuttal, titled On Post-Modern Cryptography<sup class="reference" id="cite_ref-weizmann.ac.il_3-0"><a href="#cite_note-weizmann.ac.il-3">[3]</a></sup> was posted by <a href="/wiki/Oded_Goldreich" title="Oded Goldreich">Oded Goldreich</a>, who argues that the rigorous analysis methodology of provable security is the only one compatible with science.</p>
<p>In 2007, <a href="/wiki/Neal_Koblitz" title="Neal Koblitz">Koblitz</a> published "The Uneasy Relationship Between Mathematics and Cryptography"<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup> in the <i><a href="/wiki/Notices_of_the_American_Mathematical_Society" title="Notices of the American Mathematical Society">Notices of the American Mathematical Society</a></i>. Several rebuttals have been written by <a href="/wiki/Oded_Goldreich" title="Oded Goldreich">Oded Goldreich</a>, <a href="/wiki/Avi_Wigderson" title="Avi Wigderson">Avi Wigderson</a> and other researchers in the field.<sup class="reference" id="cite_ref-weizmann.ac.il_3-1"><a href="#cite_note-weizmann.ac.il-3">[3]</a></sup><sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> <a href="/wiki/Ivan_Damg%C3%A5rd" title="Ivan Damgård">Ivan Damgård</a> later wrote <a href="/wiki/Position_paper" title="Position paper">position paper</a> at ICALP 2007 on the technical issues,<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup> and it was recommended by <a href="/wiki/Scott_Aaronson" title="Scott Aaronson">Scott Aaronson</a> as a good in-depth analysis.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup></p>
<h3><span class="mw-headline" id="Practice_oriented_provable_security">Practice oriented provable security</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Provable_security&amp;action=edit&amp;section=3" title="Edit section: Practice oriented provable security">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Classical provable security primarily aimed at studying the relationship between asymptotically defined objects. Instead, practice-oriented provable security is concerned with concrete objects of cryptographic practice, such as hash functions, block ciphers, and protocols as they are deployed and used. <sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup></p>
<p>Practice oriented provable security uses <a href="/wiki/Concrete_security" title="Concrete security">concrete security</a> to analyse practical constructions with fixed key sizes. It emphasizes the use of models such as the <a class="mw-redirect" href="/wiki/Random_oracle_model" title="Random oracle model">random oracle model</a> as tools, that are assessed with regards to their usefulness rather than being either true or false. Thus largely avoiding the controversy around provable security.</p>
