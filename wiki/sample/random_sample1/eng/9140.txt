<h2><span class="mw-headline" id="Input_handling">Input handling</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_input_and_output_handling&amp;action=edit&amp;section=1" title="Edit section: Input handling">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Input handling is how an application, server or other computing system handles the <a class="mw-redirect" href="/wiki/Input/Output" title="Input/Output">input</a> supplied from users, clients, or a computer network.</p>
<p>Secure input handling is often required to prevent vulnerabilities related to <a href="/wiki/Code_injection" title="Code injection">Code injection</a>, <a class="mw-redirect" href="/wiki/Directory_traversal" title="Directory traversal">Directory traversal</a> and so on.</p>
<h3><span class="mw-headline" id="Input_validation">Input validation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_input_and_output_handling&amp;action=edit&amp;section=2" title="Edit section: Input validation">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Validating (or sanitizing) user input is to ensure that input is safe prior to use.</p>
<p>The most secure way<sup class="noprint Inline-Template" style="margin-left:0.1em; white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Manual_of_Style/Words_to_watch#Unsupported_attributions" title="Wikipedia:Manual of Style/Words to watch"><span title="The material near this tag may use weasel words or too-vague attribution. (December 2013)">according to whom?</span></a></i>]</sup> to do this is to <b>Terminate on suspicious input</b> and use a <b>Whitelist</b> strategy to determine if execution should be terminated or not. This behavior is however not always preferred<sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Manual_of_Style/Words_to_watch#Unsupported_attributions" title="Wikipedia:Manual of Style/Words to watch"><span title="The material near this tag may use weasel words or too-vague attribution. (December 2013)">by whom?</span></a></i>]</sup> from a usability point of view.</p>
<h4><span class="mw-headline" id="Whitelists_and_blacklists">Whitelists and blacklists</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_input_and_output_handling&amp;action=edit&amp;section=3" title="Edit section: Whitelists and blacklists">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>In computer security, there are often known good inputs — input the developer is completely certain is safe. There are also known bad characters; input the developer is certain is unsafe (can cause <a href="/wiki/Code_injection" title="Code injection">Code injection</a> etc.). Based on this, two different approaches to how input should be managed exists:</p>
<dl>
<dt>Whitelist (known goods)</dt>
<dd>A <a href="/wiki/Whitelist" title="Whitelist">Whitelist</a> is a list of "known good inputs". A Whitelist is basically a list which says "A, B and C is good (and everything else is bad)".</dd>
<dt>Blacklist (known bads)</dt>
<dd>A <a class="mw-redirect" href="/wiki/Blacklist" title="Blacklist">Blacklist</a> is a list of "known bad inputs". A Blacklist is basically a list which says "A, B and C is bad (and everything else is good)".</dd>
</dl>
<p>Security professionals<sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Manual_of_Style/Words_to_watch#Unsupported_attributions" title="Wikipedia:Manual of Style/Words to watch"><span title="The material near this tag possibly uses too-vague attribution or weasel words. (December 2013)">who?</span></a></i>]</sup> tend to prefer Whitelists, because Blacklists may accidentally treat bad input as safe. However, in some cases a whitelist solution may not be easily implemented.</p>
<h4><span class="mw-headline" id="Terminate.2Fstop.2Fabort_on_input_problems">Terminate/stop/abort on input problems</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_input_and_output_handling&amp;action=edit&amp;section=4" title="Edit section: Terminate/stop/abort on input problems">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>This is a very safe strategy. If unexpected characters occur in input, abort execution. But if implemented poorly, it can lead to a <a href="/wiki/Denial-of-service_attack" title="Denial-of-service attack">denial-of-service attack</a> in which the attacker floods the system with unexpected input, forcing the system to expend scarce processing and communication resources on rejecting it.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (December 2013)">citation needed</span></a></i>]</sup></p>
<h4><span class="mw-headline" id="Filter_input">Filter input</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_input_and_output_handling&amp;action=edit&amp;section=5" title="Edit section: Filter input">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Filtering input is used as a less orthodox security principle than <b>Terminate/stop/abort on input problems</b>.</p>
<ul>
<li>The benefit of the filter approach is that to end-users, the security mechanism often behaves in a less intrusive manner. For example, if "<code>*</code>" is illegal, then "<code>I ***LOVE*** you</code>" will just become "<code>I LOVE you</code>", which is experienced as a minor but acceptable oddity.</li>
<li>The downside is that the filter approach is a bit difficult to get right — in practice many applications have the filter applied at one place in the code, but the programmer accidentally uses the unfiltered input at another place.</li>
</ul>
<h5><span class="mw-headline" id="Filter_input:_Automatic_taint_checking">Filter input: Automatic taint checking</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_input_and_output_handling&amp;action=edit&amp;section=6" title="Edit section: Filter input: Automatic taint checking">edit</a><span class="mw-editsection-bracket">]</span></span></h5>
<p>Some <a class="mw-redirect" href="/wiki/Programming_languages" title="Programming languages">programming languages</a><sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citing_sources" title="Wikipedia:Citing sources"><span title="Statement needs to be more specific about the content to which it refers. (December 2013)">specify</span></a></i>]</sup> have built-in support for <a href="/wiki/Taint_checking" title="Taint checking">taint checking</a>. These languages throw <a href="/wiki/Compile_time" title="Compile time">compile time</a> or <a href="/wiki/Run_time_(program_lifecycle_phase)" title="Run time (program lifecycle phase)">run time</a> <a href="/wiki/Exception_handling" title="Exception handling">exceptions</a> whenever a variable derived from user input is used in a risky way, e.g. to execute a <a class="new" href="/w/index.php?title=Shell_command&amp;action=edit&amp;redlink=1" title="Shell command (page does not exist)">shell command</a>.</p>
<h5><span class="mw-headline" id="Filter_input:_Whitelist_filters_.28Filter_in_known_goods.29">Filter input: Whitelist filters (Filter in known goods)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_input_and_output_handling&amp;action=edit&amp;section=7" title="Edit section: Filter input: Whitelist filters (Filter in known goods)">edit</a><span class="mw-editsection-bracket">]</span></span></h5>
<p>Example:</p>
<ul>
<li>An input filter which expects all characters to be of <a class="mw-redirect" href="/wiki/Charset" title="Charset">charset</a> <code>A-Za-z</code> is used to protect a UNIX application from <a href="/wiki/Code_injection" title="Code injection">shell injection</a>.</li>
<li>Attacker supplies input <code>; ls -l /</code> to attempt shell injection.</li>
<li>Filter is applied to input.</li>
<li>Characters <code>; - /</code> are thrown away by filter because they are not in whitelist.</li>
<li>Characters <code>lsl</code> are kept by filter because they are in whitelist.</li>
<li>Exploit attempt fails because only safe input remains.</li>
</ul>
<h5><span class="mw-headline" id="Filter_input:_Blacklist_filters_.28Filter_out_known_bads.29">Filter input: Blacklist filters (Filter out known bads)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_input_and_output_handling&amp;action=edit&amp;section=8" title="Edit section: Filter input: Blacklist filters (Filter out known bads)">edit</a><span class="mw-editsection-bracket">]</span></span></h5>
<p>A strategy that is usually insufficient is to <i>filter out known bads</i>. If the characters in the set [:;.-/] are known to be bad, but <b>; ls -l /</b> is received, the original input is replaced with <b>ls l</b> (<b>;-/</b> are thrown away). This strategy has several problems:</p>
<ul>
<li>It does not protect against <b>unknown</b> threats. There may be other "bad" inputs that the developer did not consider.</li>
<li>It does not protect against <b>future</b> threats. Inputs that are safe at present may obtain a dangerous interpretation if the underlying language changes. For example, a UNIX command line security filter designed to stop attacks against <a href="/wiki/C_shell" title="C shell">C shell</a> will be insecure if the software is moved to an environment using <a href="/wiki/Bash_(Unix_shell)" title="Bash (Unix shell)">bash</a>.</li>
</ul>
<h3><span class="mw-headline" id="Encode_.28escape.29_input">Encode (escape) input</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_input_and_output_handling&amp;action=edit&amp;section=9" title="Edit section: Encode (escape) input">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>To keep malicious inputs contained, any inputs written to the database need to be encoded.</p>
<p>SQL encoding: <i>' OR 1=1 --'</i> is encoded to <i>\ \'\ OR\ 1\=1\ \-\-'</i></p>
<p><br/>
In <a href="/wiki/PHP" title="PHP">PHP</a> this can be done with the function <tt>mysql_real_escape_string()</tt><sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> or with <tt>PDO::quote()</tt><sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup></p>
<h3><span class="mw-headline" id="Other_solutions">Other solutions</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_input_and_output_handling&amp;action=edit&amp;section=10" title="Edit section: Other solutions">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>There may be other solutions, depending on which programming language is used and what type of code injection is being prevented. E.g., the <a class="external text" href="http://www.bioinformatics.org/phplabware/internal_utilities/htmLawed/index.php" rel="nofollow">htmLawed</a> PHP script can be used to remove cross-site scripting code.</p>
<p>In particular, to prevent <a href="/wiki/SQL_injection" title="SQL injection">SQL injection</a>, <b>parameterized queries</b> (also known as <b>prepared statements</b> and <b>bind variables</b>) are excellent for improving security while also improving code clarity and performance.</p>
<h2><span class="mw-headline" id="Output_handling">Output handling</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_input_and_output_handling&amp;action=edit&amp;section=11" title="Edit section: Output handling">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Output handling is how an application, server or system handles the output (e.g. generating HTML, printing, logging, ...). It is important to keep in mind output often contains input supplied from users, clients, network, databases etc.</p>
<p>Secure output handling is primarily associated with preventing <a class="mw-redirect" href="/wiki/Cross-site_Scripting" title="Cross-site Scripting">Cross-site Scripting</a> (XSS) vulnerabilities, but could also prove to be important in similar areas (e.g. if generating Microsoft Office documents with some API, output management could potentially be required to prevent macro-injections).</p>
<h3><span class="mw-headline" id="Encode_.28escape.29_output">Encode (escape) output</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_input_and_output_handling&amp;action=edit&amp;section=12" title="Edit section: Encode (escape) output">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>"Encoding" processes content that is about to be output so that any characters which have potentially special meanings to the receiving application are made safe. Characters from a typical known safe <a class="mw-redirect" href="/wiki/Charset" title="Charset">charset</a> for the particular destination medium are often left as they are. A simple encoding might leave alone alphanumerics <i>a–z</i>, <i>A–Z</i> and <i>0–9</i>. Any other characters could be possibly interpreted in an unexpected manner, and are therefore replaced with the appropriate "encoded" representation.</p>
<p>HTML encoding: <i>&lt;script&gt;</i> is encoded to <i>&amp;lt;script&amp;gt;</i></p>
<p>In <a href="/wiki/PHP" title="PHP">PHP</a> this can be done with the function <tt>htmlspecialchars()</tt><sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup></p>
