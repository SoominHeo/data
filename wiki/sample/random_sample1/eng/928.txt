<p><b>DragonFly BSD</b> is a <a class="mw-redirect" href="/wiki/Free_and_open_source_software" title="Free and open source software">free and open source</a> <a href="/wiki/Unix-like" title="Unix-like">Unix-like</a> operating system created as a <a class="mw-redirect" href="/wiki/Fork_(software)" title="Fork (software)">fork</a> of <a href="/wiki/FreeBSD" title="FreeBSD">FreeBSD</a> 4.8. Matthew Dillon, an <a href="/wiki/Amiga" title="Amiga">Amiga</a> developer in the late 1980s and early 1990s and a FreeBSD developer between 1994 and 2003, began work on DragonFly BSD in June 2003 and announced it on the FreeBSD mailing lists on 16 July 2003.<sup class="reference" id="cite_ref-announcing-dfbsd_3-0"><a href="#cite_note-announcing-dfbsd-3">[3]</a></sup></p>
<p>Dillon started DragonFly in the belief that the methods and techniques being adopted for <a href="/wiki/Thread_(computing)" title="Thread (computing)">threading</a> and <a href="/wiki/Symmetric_multiprocessing" title="Symmetric multiprocessing">symmetric multiprocessing</a> in FreeBSD 5<sup class="reference" id="cite_ref-fbsd-smp_4-0"><a href="#cite_note-fbsd-smp-4">[4]</a></sup> would lead to poor system performance and cause maintenance difficulties. He sought to correct these suspected problems within the FreeBSD project.<sup class="reference" id="cite_ref-df-released-for-users_5-0"><a href="#cite_note-df-released-for-users-5">[5]</a></sup> Due to ongoing conflicts with other FreeBSD developers over the implementation of his ideas,<sup class="reference" id="cite_ref-behind-dfbsd_6-0"><a href="#cite_note-behind-dfbsd-6">[6]</a></sup> his ability to directly change the FreeBSD codebase was eventually revoked. Despite this, the DragonFly BSD and FreeBSD projects still work together contributing bug fixes, driver updates, and other system improvements to each other.</p>
<p>Intended to be the logical continuation of the FreeBSD 4.x series, DragonFly's development has diverged significantly from FreeBSD's, including a new <a href="/wiki/Light_Weight_Kernel_Threads" title="Light Weight Kernel Threads">Light Weight Kernel Threads</a> (LWKT) implementation, a lightweight ports/messaging system, and feature-rich <a href="/wiki/HAMMER" title="HAMMER">HAMMER</a> file system.<sup class="reference" id="cite_ref-osnews-interview-2004_7-0"><a href="#cite_note-osnews-interview-2004-7">[7]</a></sup> Many concepts planned for DragonFly were inspired by the <a href="/wiki/AmigaOS" title="AmigaOS">AmigaOS</a> operating system.<sup class="reference" id="cite_ref-unix-for-clusters_8-0"><a href="#cite_note-unix-for-clusters-8">[8]</a></sup></p>

<h2><span class="mw-headline" id="System_design">System design</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DragonFly_BSD&amp;action=edit&amp;section=1" title="Edit section: System design">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Kernel">Kernel</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DragonFly_BSD&amp;action=edit&amp;section=2" title="Edit section: Kernel">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>DragonFly's <a class="mw-redirect" href="/wiki/Kernel_(computing)" title="Kernel (computing)">kernel</a> is a <a href="/wiki/Hybrid_kernel" title="Hybrid kernel">hybrid</a>, containing features of both monolithic and microkernels, such as the message passing capability of microkernels enabling larger portions of the OS to benefit from <a class="mw-redirect" href="/wiki/Protected_memory" title="Protected memory">protected memory</a>, as well as retaining the speed of monolithic kernels for certain critical tasks. The messaging subsystem being developed is similar to those found in microkernels such as <a class="mw-redirect" href="/wiki/Mach_kernel" title="Mach kernel">Mach</a>, though it is less complex by design. DragonFly's messaging subsystem has the ability to act in either a synchronous or asynchronous fashion, and attempts to use this capability to achieve the best performance possible in any given situation.<sup class="reference" id="cite_ref-dfbsd-os-wp_9-0"><a href="#cite_note-dfbsd-os-wp-9">[9]</a></sup></p>
<p>According to developer Matthew Dillon, progress is being made to provide both device <a href="/wiki/Input/output" title="Input/output">input/output</a> (I/O) and <a href="/wiki/Virtual_file_system" title="Virtual file system">virtual file system</a> (VFS) messaging capabilities that will enable the remainder of the project goals to be met. The new infrastructure will allow many parts of the kernel to be migrated out into userspace; here they will be more easily debugged as they will be smaller, isolated programs, instead of being small parts entwined in a larger chunk of code. Additionally, the migration of select kernel code into userspace has the benefit of making the system more robust; if a userspace driver crashes, it will not crash the kernel.<sup class="reference" id="cite_ref-kt-interview_10-0"><a href="#cite_note-kt-interview-10">[10]</a></sup></p>
<p><a href="/wiki/System_call" title="System call">System calls</a> are being split into userland and kernel versions and being encapsulated into messages. This will help reduce the size and complexity of the kernel by moving variants of standard system calls into a userland <a href="/wiki/Compatibility_layer" title="Compatibility layer">compatibility layer</a>, and help maintain forwards and backwards compatibility between DragonFly versions. <a href="/wiki/Linux" title="Linux">Linux</a> and other <a href="/wiki/Unix-like" title="Unix-like">Unix-like</a> OS compatibility code is being migrated out similarly.<sup class="reference" id="cite_ref-unix-for-clusters_8-1"><a href="#cite_note-unix-for-clusters-8">[8]</a></sup></p>
<h4><span class="mw-headline" id="Threading">Threading</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DragonFly_BSD&amp;action=edit&amp;section=3" title="Edit section: Threading">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>As support for multiple <a class="mw-redirect" href="/wiki/Processor_architecture" title="Processor architecture">processor architectures</a> complicates <a href="/wiki/Symmetric_multiprocessing" title="Symmetric multiprocessing">symmetric multiprocessing</a> (SMP) support,<sup class="reference" id="cite_ref-behind-dfbsd_6-1"><a href="#cite_note-behind-dfbsd-6">[6]</a></sup> DragonFly BSD now limits its support to the <a href="/wiki/X86-64" title="X86-64">x86-64</a> platform.<sup class="reference" id="cite_ref-osnews-mp-inproved_11-0"><a href="#cite_note-osnews-mp-inproved-11">[11]</a></sup> DragonFly originally ran on the <a href="/wiki/X86" title="X86">x86</a> architecture, however as of version 4.0 it is no longer supported. Since version 1.10, DragonFly supports 1:1 userland threading (one kernel thread per userland thread),<sup class="reference" id="cite_ref-luciani_dcbsdcon2009_12-0"><a href="#cite_note-luciani_dcbsdcon2009-12">[12]</a></sup> which is regarded as a relatively simple solution that is also easy to maintain.<sup class="reference" id="cite_ref-unix-for-clusters_8-2"><a href="#cite_note-unix-for-clusters-8">[8]</a></sup> Inherited from FreeBSD, DragonFly also supports multi-threading.<sup class="reference" id="cite_ref-paying-off-already_13-0"><a href="#cite_note-paying-off-already-13">[13]</a></sup></p>
<p>In DragonFly, each <a href="/wiki/Central_processing_unit" title="Central processing unit">CPU</a> has its own thread scheduler. Upon creation, threads are assigned to processors and are never preemptively switched from one processor to another; they are only migrated by the passing of an <a href="/wiki/Inter-processor_interrupt" title="Inter-processor interrupt">inter-processor interrupt</a> (IPI) message between the CPUs involved. Inter-processor thread scheduling is also accomplished by sending asynchronous IPI messages. One advantage to this clean compartmentalization of the threading subsystem is that the processors' on-board <a href="/wiki/CPU_cache" title="CPU cache">caches</a> in <a class="mw-redirect" href="/wiki/Symmetric_Multiprocessor_System" title="Symmetric Multiprocessor System">Symmetric Multiprocessor Systems</a> do not contain duplicated data, allowing for higher performance by giving each processor in the system the ability to use its own cache to store different things to work on.<sup class="reference" id="cite_ref-unix-for-clusters_8-3"><a href="#cite_note-unix-for-clusters-8">[8]</a></sup></p>
<p>The <a href="/wiki/Light_Weight_Kernel_Threads" title="Light Weight Kernel Threads"><i>LWKT</i></a> subsystem is being employed to partition work among multiple kernel threads (for example in the networking code there is one thread per protocol per processor), reducing competition by removing the need to share certain resources among various kernel tasks.<sup class="reference" id="cite_ref-behind-dfbsd_6-2"><a href="#cite_note-behind-dfbsd-6">[6]</a></sup></p>
<h4><span class="mw-headline" id="Shared_resources_protection">Shared resources protection</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DragonFly_BSD&amp;action=edit&amp;section=4" title="Edit section: Shared resources protection">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>In order to run safely on multiprocessor machines, access to shared resources (like files, data structures) must be <a href="/wiki/Serialization" title="Serialization">serialized</a> so that threads or processes do not attempt to modify the same resource at the same time. In order to prevent multiple threads from accessing or modifying a shared resource simultaneously, DragonFly employs <a href="/wiki/Critical_section" title="Critical section">critical sections</a>, and serializing tokens to prevent concurrent access. While both Linux and FreeBSD 5 employ fine-grained <a href="/wiki/Lock_(computer_science)" title="Lock (computer science)">mutex</a> models to achieve higher performance on <a class="mw-redirect" href="/wiki/Multiprocessor" title="Multiprocessor">multiprocessor</a> systems, DragonFly does not.<sup class="reference" id="cite_ref-behind-dfbsd_6-3"><a href="#cite_note-behind-dfbsd-6">[6]</a></sup> Until recently, DragonFly also employed <a href="/wiki/Spl_(Unix)" title="Spl (Unix)">spls</a>, but these were replaced with critical sections.</p>
<p>Much of the system's core, including the <i>LWKT</i> subsystem, the IPI messaging subsystem and the new kernel memory allocator, are lockless, meaning that they work without using mutexes, with each process operating on a single CPU. Critical sections are used to protect against local interrupts, individually for each CPU, guaranteeing that a thread currently being executed will not be preempted.<sup class="reference" id="cite_ref-luciani_dcbsdcon2009_12-1"><a href="#cite_note-luciani_dcbsdcon2009-12">[12]</a></sup></p>
<p><a href="/wiki/Serializing_tokens" title="Serializing tokens">Serializing tokens</a> are used to prevent concurrent accesses from other CPUs and may be held simultaneously by multiple threads, ensuring that only one of those threads is running at any given time. Blocked or sleeping threads therefore do not prevent other threads from accessing the shared resource unlike a thread that is holding a mutex. Among other things, the use of serializing tokens prevents many of the situations that could result in <a href="/wiki/Deadlock" title="Deadlock">deadlocks</a> and <a href="/wiki/Priority_inversion" title="Priority inversion">priority inversions</a> when using mutexes, as well as greatly simplifying the design and implementation of a many-step procedure that would require a resource to be shared among multiple threads. The serializing token code is evolving into something quite similar to the "<a href="/wiki/Read-copy-update" title="Read-copy-update">Read-copy-update</a>" feature now available in Linux. Unlike Linux's current RCU implementation, DragonFly's is being implemented such that only processors competing for the same token are affected rather than all processors in the computer.<sup class="reference" id="cite_ref-serializing_token_14-0"><a href="#cite_note-serializing_token-14">[14]</a></sup></p>
<p>DragonFly switched to multiprocessor safe <a class="mw-redirect" href="/wiki/Slab_allocator" title="Slab allocator">slab allocator</a>, which requires neither mutexes nor blocking operations for memory assignment tasks.<sup class="reference" id="cite_ref-slab-allocator-fbsd_15-0"><a href="#cite_note-slab-allocator-fbsd-15">[15]</a></sup> It was eventually ported into standard C library in the userland, where it replaced FreeBSD's malloc implementation.<sup class="reference" id="cite_ref-new-libc-malloc_16-0"><a href="#cite_note-new-libc-malloc-16">[16]</a></sup></p>
<h4><span class="mw-headline" id="Virtual_kernel">Virtual kernel</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DragonFly_BSD&amp;action=edit&amp;section=5" title="Edit section: Virtual kernel">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Since release 1.8 DragonFly has a virtualization mechanism similar to <a href="/wiki/User-mode_Linux" title="User-mode Linux">User-mode Linux</a>,<sup class="reference" id="cite_ref-towards-free-clustering_17-0"><a href="#cite_note-towards-free-clustering-17">[17]</a></sup> allowing a user to run another kernel in the userland. The virtual kernel (<i>vkernel</i>) is run in completely isolated environment with emulated network and storage interfaces, thus simplifying testing kernel subsystems and clustering features.<sup class="reference" id="cite_ref-unix-for-clusters_8-4"><a href="#cite_note-unix-for-clusters-8">[8]</a></sup><sup class="reference" id="cite_ref-kt-interview_10-1"><a href="#cite_note-kt-interview-10">[10]</a></sup></p>
<p>The vkernel has two important differences from the real kernel: it lacks many routines for dealing with the low-level hardware management and it uses <a href="/wiki/C_standard_library" title="C standard library">C standard library</a> (libc) functions in place of in-kernel implementations wherever possible. As both real and virtual kernel are compiled from the same code base, this effectively means that platform-dependent routines and re-implementations of libc functions are clearly separated in a source tree.<sup class="reference" id="cite_ref-peek-vkernel-1_18-0"><a href="#cite_note-peek-vkernel-1-18">[18]</a></sup></p>
<p>The virtualized platform vkernel runs on is built on top of high-level abstractions provided by the real kernel. These abstractions include the <a href="/wiki/Kqueue" title="Kqueue">kqueue</a>-based timer, the console (mapped to the <a href="/wiki/Virtual_terminal" title="Virtual terminal">virtual terminal</a> where vkernel is executed), the disk image and virtual kernel Ethernet device (<i>VKE</i>), tunneling all packets to the host's <a href="/wiki/TUN/TAP" title="TUN/TAP">tap</a> interface.<sup class="reference" id="cite_ref-peek-vkernel-2_19-0"><a href="#cite_note-peek-vkernel-2-19">[19]</a></sup></p>
<h3><span class="mw-headline" id="Package_management">Package management</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DragonFly_BSD&amp;action=edit&amp;section=6" title="Edit section: Package management">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Third-party software is available on DragonFly as binary packages via <code>pkgng</code> or from a native <a href="/wiki/Ports_collection" title="Ports collection">ports collection</a> – <a href="/wiki/FreeBSD_Ports#DPorts" title="FreeBSD Ports">DPorts</a>.<sup class="reference" id="cite_ref-howto-dports_20-0"><a href="#cite_note-howto-dports-20">[20]</a></sup></p>
<p>DragonFly originally used the <a href="/wiki/FreeBSD_Ports" title="FreeBSD Ports">FreeBSD Ports</a> collection as its official <a class="mw-redirect" href="/wiki/Package_management_system" title="Package management system">package management system</a>, but starting with the 1.4 release switched to NetBSD's <a href="/wiki/Pkgsrc" title="Pkgsrc">pkgsrc</a> system, which was perceived as a way of lessening the amount of work needed for third-party software availability.<sup class="reference" id="cite_ref-df-released-for-users_5-1"><a href="#cite_note-df-released-for-users-5">[5]</a></sup><sup class="reference" id="cite_ref-10-years-of-pkgsrc_21-0"><a href="#cite_note-10-years-of-pkgsrc-21">[21]</a></sup> Eventually, maintaining compatibility with <code>pkgsrc</code> proved to require more effort than was initially anticipated, so the project created DPorts, an overlay on top of the <a href="/wiki/FreeBSD_Ports" title="FreeBSD Ports">FreeBSD Ports</a> collection.<sup class="reference" id="cite_ref-why-dports_22-0"><a href="#cite_note-why-dports-22">[22]</a></sup><sup class="reference" id="cite_ref-any-new-packages_23-0"><a href="#cite_note-any-new-packages-23">[23]</a></sup></p>
<h3><span class="mw-headline" id="CARP_support">CARP support</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DragonFly_BSD&amp;action=edit&amp;section=7" title="Edit section: CARP support">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The initial implementation of <a href="/wiki/Common_Address_Redundancy_Protocol" title="Common Address Redundancy Protocol">Common Address Redundancy Protocol</a> (commonly referred to as <i>CARP</i>) was finished in March 2007.<sup class="reference" id="cite_ref-carp-first-patch_24-0"><a href="#cite_note-carp-first-patch-24">[24]</a></sup> As of 2011, CARP support is integrated into DragonFly BSD.<sup class="reference" id="cite_ref-man-carp_25-0"><a href="#cite_note-man-carp-25">[25]</a></sup></p>
<h3><span class="mw-headline" id="HAMMER_file_system">HAMMER file system</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DragonFly_BSD&amp;action=edit&amp;section=8" title="Edit section: HAMMER file system">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="hatnote" role="note">Main article: <a href="/wiki/HAMMER" title="HAMMER">HAMMER</a></div>
<p>Alongside the <a href="/wiki/Unix_File_System" title="Unix File System">Unix File System</a>, which is typically the default <a href="/wiki/File_system" title="File system">file system</a> on BSDs, DragonFly BSD supports HAMMER file system. It was developed specifically for DragonFly BSD to provide a feature-rich yet better designed analogue of the increasingly popular <a href="/wiki/ZFS" title="ZFS">ZFS</a>.<sup class="reference" id="cite_ref-unix-for-clusters_8-5"><a href="#cite_note-unix-for-clusters-8">[8]</a></sup><sup class="reference" id="cite_ref-kt-interview_10-2"><a href="#cite_note-kt-interview-10">[10]</a></sup><sup class="reference" id="cite_ref-hammer-design-document_26-0"><a href="#cite_note-hammer-design-document-26">[26]</a></sup> HAMMER supports configurable file system history, <a href="/wiki/Snapshot_(computer_storage)" title="Snapshot (computer storage)">snapshots</a>, <a href="/wiki/Checksum" title="Checksum">checksumming</a>, <a href="/wiki/Data_deduplication" title="Data deduplication">data deduplication</a> and other features typical for file systems of its kind.<sup class="reference" id="cite_ref-towards-free-clustering_17-1"><a href="#cite_note-towards-free-clustering-17">[17]</a></sup><sup class="reference" id="cite_ref-can-hammer-compete_27-0"><a href="#cite_note-can-hammer-compete-27">[27]</a></sup></p>
<p>The next generation of HAMMER file system (<i>HAMMER2</i>) is being developed by Dillon.<sup class="reference" id="cite_ref-hammer2-design_28-0"><a href="#cite_note-hammer2-design-28">[28]</a></sup> DragonFly BSD 3.8.0 was the first released to include support for HAMMER2, though it is declared as not ready for general use in release notes.<sup class="reference" id="cite_ref-dfly-release38_29-0"><a href="#cite_note-dfly-release38-29">[29]</a></sup></p>
<h3><span class="mw-headline" id="devfs">devfs</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DragonFly_BSD&amp;action=edit&amp;section=9" title="Edit section: devfs">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In 2007 DragonFly BSD received a new <a class="mw-redirect" href="/wiki/Device_file_system" title="Device file system">device file system</a> (devfs), which dynamically adds and removes device nodes, allows accessing devices by connection paths, recognises drives by <a class="mw-redirect" href="/wiki/Serial_number" title="Serial number">serial numbers</a> and removes the need for pre-populated <code>/dev</code> file system hierarchy. It was implemented as a <a href="/wiki/Google_Summer_of_Code" title="Google Summer of Code">Google Summer of Code</a> 2009 project.<sup class="reference" id="cite_ref-bsdtalk184_30-0"><a href="#cite_note-bsdtalk184-30">[30]</a></sup></p>
<h3><span class="mw-headline" id="Application_snapshots">Application snapshots</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DragonFly_BSD&amp;action=edit&amp;section=10" title="Edit section: Application snapshots">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>DragonFly BSD supports <a href="/wiki/Amiga" title="Amiga">Amiga</a>-style <i>resident applications</i> feature: it takes a snapshot of a large, dynamically linked program's <a href="/wiki/Virtual_memory" title="Virtual memory">virtual memory</a> space after loading, allowing future instances of the program to start much more quickly than it otherwise would have. This replaces the <a class="mw-redirect" href="/wiki/Prelinking" title="Prelinking">prelinking</a> capability that was being worked on earlier in the project's history, as the resident support is much more efficient. Large programs like those found in <a href="/wiki/KDE_Software_Compilation" title="KDE Software Compilation">KDE Software Compilation</a> with many <a class="mw-redirect" href="/wiki/Library_(computer_science)" title="Library (computer science)">shared libraries</a> will benefit the most from this support.<sup class="reference" id="cite_ref-dfbsd-diary_31-0"><a href="#cite_note-dfbsd-diary-31">[31]</a></sup></p>
<h2><span class="mw-headline" id="Development_and_distribution">Development and distribution</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DragonFly_BSD&amp;action=edit&amp;section=11" title="Edit section: Development and distribution">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>As with <a href="/wiki/FreeBSD" title="FreeBSD">FreeBSD</a> and <a href="/wiki/OpenBSD" title="OpenBSD">OpenBSD</a>, the developers of DragonFly BSD are slowly replacing <a href="/wiki/Indent_style#K.26R_style" title="Indent style">K&amp;R</a> style <a href="/wiki/C_(programming_language)" title="C (programming language)">C</a> code with more modern, <a href="/wiki/ANSI_C" title="ANSI C">ANSI</a> equivalents. Similar to other operating systems, DragonFly's version of the <a href="/wiki/GNU_Compiler_Collection" title="GNU Compiler Collection">GNU Compiler Collection</a> has an enhancement called the <a href="/wiki/Buffer_overflow_protection#GCC_Stack-Smashing_Protector_.28ProPolice.29" title="Buffer overflow protection">Stack-Smashing Protector (ProPolice)</a> enabled by default, providing some additional protection against <a href="/wiki/Buffer_overflow" title="Buffer overflow">buffer overflow</a> based attacks. It should be noted that as of 23 July 2005<sup class="plainlinks noprint asof-tag update" style="display:none;"><a class="external text" href="//en.wikipedia.org/w/index.php?title=DragonFly_BSD&amp;action=edit">[update]</a></sup>, the kernel is no longer built with this protection by default.<sup class="reference" id="cite_ref-dfbsd-diary_31-1"><a href="#cite_note-dfbsd-diary-31">[31]</a></sup></p>
<p>Being a derivative of FreeBSD, DragonFly has inherited an easy-to-use integrated build system that can rebuild the entire base system from source with only a few commands. The DragonFly developers use the <a class="mw-redirect" href="/wiki/Git_(software)" title="Git (software)">Git</a> version control system to manage changes to the DragonFly <a href="/wiki/Source_code" title="Source code">source code</a>. Unlike its parent FreeBSD, DragonFly has both stable and unstable releases in a single source tree, due to a smaller developer base.<sup class="reference" id="cite_ref-behind-dfbsd_6-4"><a href="#cite_note-behind-dfbsd-6">[6]</a></sup></p>
<p>Like the other BSD kernels (and those of most modern operating systems), DragonFly employs a built-in <a href="/wiki/Kernel_debugger" title="Kernel debugger">kernel debugger</a> to help the developers find kernel bugs. Furthermore, as of October 2004<sup class="plainlinks noprint asof-tag update" style="display:none;"><a class="external text" href="//en.wikipedia.org/w/index.php?title=DragonFly_BSD&amp;action=edit">[update]</a></sup>, a debug kernel, which makes bug reports more useful for tracking down kernel-related problems, is installed by default, at the expense of a relatively small quantity of disk space. When a new kernel is installed, the backup copy of the previous kernel and its modules are stripped of their debugging symbols to further minimize disk space usage.</p>
<h3><span class="mw-headline" id="Distribution_media">Distribution media</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=DragonFly_BSD&amp;action=edit&amp;section=12" title="Edit section: Distribution media">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The operating system is distributed as a <a href="/wiki/Live_CD" title="Live CD">Live CD</a> and <a href="/wiki/Live_USB" title="Live USB">Live USB</a> (<i>full <a href="/wiki/X_Window_System" title="X Window System">X11</a></i> flavour available) that boots into a complete DragonFly system.<sup class="reference" id="cite_ref-towards-free-clustering_17-2"><a href="#cite_note-towards-free-clustering-17">[17]</a></sup><sup class="reference" id="cite_ref-bsdtalk184_30-1"><a href="#cite_note-bsdtalk184-30">[30]</a></sup> It includes the base system and a complete set of manual pages, and may include source code and useful packages in future versions. The advantage of this is that with a single CD you can install the software onto a computer, use a full set of tools to repair a damaged installation, or demonstrate the capabilities of the system without installing it. Daily snapshots are available from the master site for those who want to install the most recent versions of DragonFly without building from source.</p>
<p>Like the other free open source BSDs, DragonFly is distributed under the terms of the modern version of the <a class="mw-redirect" href="/wiki/BSD_license" title="BSD license">BSD license</a>.</p>
