<p><b>bzip2</b> is a <a class="mw-redirect" href="/wiki/Free_and_open-source" title="Free and open-source">free and open-source</a> <a href="/wiki/Computer_file" title="Computer file">file</a> <a class="mw-redirect" href="/wiki/Compression_program" title="Compression program">compression program</a> that uses the <a href="/wiki/Burrows%E2%80%93Wheeler_transform" title="Burrows–Wheeler transform">Burrows–Wheeler algorithm</a>. It only compresses single files and is not a <a href="/wiki/File_archiver" title="File archiver">file archiver</a>. It is developed and maintained by <a href="/wiki/Julian_Seward" title="Julian Seward">Julian Seward</a>. Seward made the first public release of bzip2, version 0.15, in July 1996. The compressor's stability and popularity grew over the next several years, and Seward released version 1.0 in late 2000.<sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources in the body of the article. (February 2014)">not verified in body</span></a></i>]</sup></p>

<h2><span class="mw-headline" id="Compression_efficiency">Compression efficiency</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Bzip2&amp;action=edit&amp;section=1" title="Edit section: Compression efficiency">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>bzip2 compresses most files more effectively than the older <a href="/wiki/Lempel%E2%80%93Ziv%E2%80%93Welch" title="Lempel–Ziv–Welch">LZW</a> (<a href="/wiki/Compress" title="Compress">.Z</a>) and <a href="/wiki/DEFLATE" title="DEFLATE">Deflate</a> (<a class="mw-redirect" href="/wiki/ZIP_(file_format)" title="ZIP (file format)">.zip</a> and <a href="/wiki/Gzip" title="Gzip">.gz</a>) compression algorithms, but is considerably slower. <a href="/wiki/Lempel%E2%80%93Ziv%E2%80%93Markov_chain_algorithm" title="Lempel–Ziv–Markov chain algorithm">LZMA</a> is generally more space-efficient than bzip2 at the expense of even slower compression speed, while having much faster decompression.<sup class="reference" id="cite_ref-benchmark_3-0"><a href="#cite_note-benchmark-3">[3]</a></sup></p>
<p>bzip2 compresses data in blocks of size between 100 and 900 <a href="/wiki/Kilobyte" title="Kilobyte">kB</a> and uses the <a href="/wiki/Burrows%E2%80%93Wheeler_transform" title="Burrows–Wheeler transform">Burrows–Wheeler transform</a> to convert frequently-recurring character sequences into strings of identical letters. It then applies <a href="/wiki/Move-to-front_transform" title="Move-to-front transform">move-to-front transform</a> and <a href="/wiki/Huffman_coding" title="Huffman coding">Huffman coding</a>. bzip2's ancestor <b>bzip</b> used <a href="/wiki/Arithmetic_coding" title="Arithmetic coding">arithmetic coding</a> instead of Huffman. The change was made because of a <a href="/wiki/Software_patent" title="Software patent">software patent</a> restriction.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup></p>
<p>bzip2 performance is asymmetric, as decompression is relatively fast. Motivated by the large CPU time required for compression, a modified version was created in 2003 called pbzip2 that supported <a class="mw-redirect" href="/wiki/Thread_(computer_science)" title="Thread (computer science)">multi-threading</a>, giving almost linear speed improvements on multi-CPU and multi-core computers.<sup class="reference" id="cite_ref-compressioncomparison_5-0"><a href="#cite_note-compressioncomparison-5">[5]</a></sup> As of May 2010<sup class="plainlinks noprint asof-tag update" style="display:none;"><a class="external text" href="//en.wikipedia.org/w/index.php?title=Bzip2&amp;action=edit">[update]</a></sup>, this functionality has not been incorporated into the main project.</p>
<p>Like gzip, bzip2 is only a data compressor. It is not an archiver like <a class="mw-redirect" href="/wiki/Tar_(file_format)" title="Tar (file format)">tar</a> or ZIP; the program itself has no facilities for multiple files, encryption or archive-splitting, but, in the <a href="/wiki/Unix_philosophy" title="Unix philosophy">UNIX tradition</a>, relies instead on separate external utilities such as tar and <a href="/wiki/GNU_Privacy_Guard" title="GNU Privacy Guard">GnuPG</a> for these tasks.</p>
<h2><span class="mw-headline" id="Compression_stack">Compression stack</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Bzip2&amp;action=edit&amp;section=2" title="Edit section: Compression stack">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Bzip2 uses several layers of compression techniques stacked on top of each other, which occur in the following order during compression and the reverse order during decompression:</p>
<ol>
<li><a href="/wiki/Run-length_encoding" title="Run-length encoding">Run-length encoding</a> (RLE) of initial data</li>
<li><a href="/wiki/Burrows%E2%80%93Wheeler_transform" title="Burrows–Wheeler transform">Burrows–Wheeler transform</a> (BWT) or block sorting</li>
<li><a class="mw-redirect" href="/wiki/Move_to_front" title="Move to front">Move to front</a> (MTF) transform</li>
<li>Run-length encoding (RLE) of MTF result</li>
<li><a href="/wiki/Huffman_coding" title="Huffman coding">Huffman coding</a></li>
<li>Selection between multiple Huffman tables</li>
<li>Unary <a class="mw-redirect" href="/wiki/Base_1" title="Base 1">base 1</a> encoding of Huffman table selection</li>
<li><a href="/wiki/Delta_encoding" title="Delta encoding">Delta encoding</a> (Δ) of Huffman code bit-lengths</li>
<li>Sparse <a href="/wiki/Bit_array" title="Bit array">bit array</a> showing which symbols are used</li>
</ol>
<h3><span class="mw-headline" id="Initial_run-length_encoding">Initial run-length encoding</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Bzip2&amp;action=edit&amp;section=3" title="Edit section: Initial run-length encoding">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Any sequence of 4 to 255 consecutive duplicate symbols is replaced by the first four symbols and a repeat length between 0 and 251. Thus the sequence "<code>AAAAAAABBBBCCCD</code>" is replaced with "<code>AAAA\3BBBB\0CCCD</code>", where "<code>\3</code>" and "<code>\0</code>" represent byte values 3 and 0 respectively. Runs of symbols are always transformed after four consecutive symbols, even if the run-length is set to zero, to keep the transformation reversible.</p>
<p>In the worst case, it can cause an expansion of 1.25 and best case a reduction to &lt;0.02 . While the specification theoretically allows for runs of length 256–259 to be encoded, the reference encoder will not produce such output.</p>
<p>The author of bzip2 has stated that the RLE step was a historical mistake and was only intended to protect the original BWT implementation from pathological cases.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup></p>
<h3><span class="mw-headline" id="Burrows.E2.80.93Wheeler_transform">Burrows–Wheeler transform</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Bzip2&amp;action=edit&amp;section=4" title="Edit section: Burrows–Wheeler transform">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>This is the reversible block-sort that is at the core of bzip2. <h3><span class="mw-headline" id="Move_to_front_transform">Move to front transform</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Bzip2&amp;action=edit&amp;section=5" title="Edit section: Move to front transform">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Again, this transform does not alter the size of the processed block. Each of the symbols in use in the document is placed in an array. When a symbol is processed, it is replaced by its location (index) in the array and that symbol is shuffled to the front of the array. The effect is that immediately recurring symbols are replaced by zero symbols (long runs of <i>any</i> arbitrary symbol thus become runs of zero symbols), while other symbols are remapped according to their local frequency.</p>
<p>Much "natural" data contains identical symbols that recur within a limited range (text is a good example). As the MTF transform assigns low values to symbols that reappear frequently, this results in a data stream which contains many symbols in the low integer range, many of them being identical (different recurring input symbols can actually map to the same output symbol). Such data can be very efficiently encoded by any legacy compression method.</p>
<h3><span class="mw-headline" id="Run-length_encoding_of_MTF_result">Run-length encoding of MTF result</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Bzip2&amp;action=edit&amp;section=6" title="Edit section: Run-length encoding of MTF result">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Long strings of zeros in the output of the move-to-front transform (which come from repeated symbols in the output of the BWT) are replaced by a sequence of two special codes, RUNA and RUNB, which represent the run-length as a binary number. Actual zeros are never encoded in the output; a lone zero becomes RUNA. (This step in fact is done at the same time as MTF is; whenever MTF would produce zero, it instead increases a counter to then encode with RUNA and RUNB.)</p>
<p>The sequence <code>0,0,0,0,0,1</code> would be represented as <code>RUNA,RUNB,1</code>; <code>RUNA,RUNB</code> represents the value 5 as described below. The run-length code is terminated by reaching another normal symbol. This RLE process is more flexible than the initial RLE step, as it is able to encode arbitrarily long integers (in practice, this is usually limited by the block size, so that this step does not encode a run of more than 900000 bytes). The run-length is encoded in this fashion: assigning place values of 1 to the first bit, 2 to the second, 4 to the third, etc. in the sequence, multiply each place value in a RUNB spot by 2, and add all the resulting place values (for RUNA and RUNB values alike) together. This is similar to base-2 <a href="/wiki/Bijective_numeration" title="Bijective numeration">bijective numeration</a>. Thus, the sequence <code>RUNA,RUNB</code> results in the value (1 + 2 × 2) = 5. As a more complicated example:</p>
<pre>
RUNA RUNB RUNA RUNA RUNB (ABAAB)
   1    2    4    8   16
   1    4    4    8   32 = 49
</pre>
<h3><span class="mw-headline" id="Huffman_coding">Huffman coding</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Bzip2&amp;action=edit&amp;section=7" title="Edit section: Huffman coding">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>This process replaces fixed length symbols in the range 0–258 with variable length codes based on the frequency of use. More frequently used codes end up shorter (2–3 bits) whilst rare codes can be allocated up to 20 bits. The codes are selected carefully so that no sequence of bits can be confused for a different code.</p>
<p>The end-of-stream code is particularly interesting. If there are <i>n</i> different bytes (symbols) used in the uncompressed data, then the Huffman code will consist of two RLE codes (RUNA and RUNB), <i>n</i> − 1 symbol codes and one end-of-stream code. Because of the combined result of the MTF and RLE encodings in the previous two steps, there is never any need to explicitly reference the first symbol in the MTF table (would be zero in the ordinary MTF), thus saving one symbol for the end-of-stream marker (and explaining why only <i>n</i> − 1 symbols are coded in the Huffman tree). In the extreme case where only one symbol is used in the uncompressed data, there will be no symbol codes at all in the Huffman tree, and the entire block will consist of RUNA and RUNB (implicitly repeating the single byte) and an end-of-stream marker with value 2.</p>
<dl>
<dd>0: RUNA<br/>
1: RUNB<br/>
2–257: byte values 0–255<br/>
258: end of stream, finish processing. (could be as low as 2).</dd>
</dl>
<h3><span class="mw-headline" id="Multiple_Huffman_tables">Multiple Huffman tables</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Bzip2&amp;action=edit&amp;section=8" title="Edit section: Multiple Huffman tables">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Several identically-sized Huffman tables can be used with a block if the gain from using them is greater than the cost of including the extra table. At least two (2) and up to six (6) tables can be present, with the most appropriate table being reselected before every 50 symbols processed. This has the advantage of having very responsive Huffman dynamics without having to continuously supply new tables, as would be required in <a href="/wiki/DEFLATE" title="DEFLATE">DEFLATE</a>. Run-length encoding in the previous step is designed to take care of codes that have an inverse probability of use higher than the shortest code Huffman code in use.</p>
<h3><span class="mw-headline" id="Unary_encoding_of_Huffman_table_selection">Unary encoding of Huffman table selection</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Bzip2&amp;action=edit&amp;section=9" title="Edit section: Unary encoding of Huffman table selection">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>If multiple Huffman tables are in use, the selection of each table (numbered 0..5) is done from a list by a zero-terminated bit run between one (1) and six (6) bits in length. The selection is into a <a href="/wiki/Move-to-front_transform" title="Move-to-front transform">MTF</a> list of the tables. Using this feature results in a maximum expansion of around 1.015, but generally less. This expansion is likely to be greatly over-shadowed by the advantage of selecting more appropriate Huffman tables and the common-case of continuing to use the same Huffman table is represented as a single bit. Rather than unary encoding, effectively this is an extreme form of a Huffman tree where each code has half the probability of the previous code.</p>
<h3><span class="mw-headline" id="Delta_encoding">Delta encoding</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Bzip2&amp;action=edit&amp;section=10" title="Edit section: Delta encoding">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Huffman code bit-lengths are required to reconstruct each of the used <a href="/wiki/Canonical_Huffman_code" title="Canonical Huffman code">canonical Huffman tables</a>. Each bit-length is stored as an encoded difference against the previous code bit-length. A zero-bit (0) means that the previous bit-length should be duplicated for the current code, whilst a one-bit (1) means that a further bit should be read and the bit-length incremented or decremented based on that value.</p>
<p>In the common case a single bit is used per symbol per table and the worst case—going from length one (1) to length twenty (20)—would require approximately 37 bits. As a result of the earlier MTF encoding, code lengths would start at 2–3 bits long (very frequently used codes) and gradually increase, meaning that the delta format is fairly efficient—requiring around 300 bits (38 bytes) per full Huffman table.</p>
<h3><span class="mw-headline" id="Sparse_bit_array">Sparse bit array</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Bzip2&amp;action=edit&amp;section=11" title="Edit section: Sparse bit array">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>A bitmap is used to show which symbols are used inside the block and should be included in the Huffman trees. Binary data is likely to use all 256 symbols representable by a byte, whereas textual data may only use a small subset of available values, perhaps covering the <a href="/wiki/ASCII" title="ASCII">ASCII</a> range between 32 and 126. Storing 256 zero bits would be inefficient if they were mostly unused. A <i>sparse</i> method is used: the 256 symbols are divided up into 16 ranges and only if symbols are used within that block is a 16-bit array included. The presence of each of these 16 ranges is indicated by an additional 16-bit bit array at the front.</p>
<p>The total bitmap uses between 32 and 272 bits of storage (4–34 bytes). For contrast, the <a href="/wiki/DEFLATE" title="DEFLATE">DEFLATE</a> algorithm would show the absence of symbols by encoding the symbols as having a (zero) bit-length with Run Length Encoding and additional Huffman coding.</p>

<p>A <code>.bz2</code> stream consists of a 4-byte header, followed by zero or more compressed blocks, immediately followed by an end-of-stream marker containing a 32-bit CRC for the plaintext whole stream processed. The compressed blocks are bit-aligned and no padding occurs.</p>
<pre>
.magic:16                       = 'BZ' signature/magic number
.version:8                      = 'h' for Bzip2 ('H'uffman coding), '0' for Bzip1 (deprecated)
.hundred_k_blocksize:8          = '1'..'9' block-size 100 kB-900 kB (uncompressed)

.compressed_magic:48            = 0x314159265359 (BCD (pi))
.crc:32                         = checksum for this block
.randomised:1                   = 0=&gt;normal, 1=&gt;randomised (deprecated)
.origPtr:24                     = starting pointer into BWT for after untransform
.huffman_used_map:16            = bitmap, of ranges of 16 bytes, present/not present
.huffman_used_bitmaps:0..256    = bitmap, of symbols used, present/not present (multiples of 16)
.huffman_groups:3               = 2..6 number of different Huffman tables in use
.selectors_used:15              = number of times that the Huffman tables are swapped (each 50 bytes)
*.selector_list:1..6            = zero-terminated bit runs (0..62) of MTF'ed Huffman table (*selectors_used)
.start_huffman_length:5         = 0..20 starting bit length for Huffman deltas
*.delta_bit_length:1..40        = 0=&gt;next symbol; 1=&gt;alter length
                                                { 1=&gt;decrement length;  0=&gt;increment length } (*(symbols+2)*groups)
.contents:2..∞                  = Huffman encoded data stream until end of block (max. 7372800 bit)

.eos_magic:48                   = 0x177245385090 (BCD sqrt(pi))
.crc:32                         = checksum for whole stream
.padding:0..7                   = align to whole byte
</pre>
<p>Because of the first-stage RLE compression (see above), the maximum length of plaintext that a single 900 kB bzip2 block can contain is around 46 MB (45,899,236 bytes). This can occur if the whole plaintext consists entirely of repeated values (the resulting <code>.bz2</code> file in this case is 46 bytes long). An even smaller file of 40 bytes can be achieved by using an input containing entirely values of 251, an apparent compression ratio of 1147480.9:1.</p>
<p>The compressed blocks in bzip2 can be independently decompressed, without having to process earlier blocks. This means that bzip2 files can be decompressed in parallel, making it a good format for use in <a href="/wiki/Big_data" title="Big data">big data</a> applications with cluster computing frameworks like <a class="mw-redirect" href="/wiki/Hadoop" title="Hadoop">Hadoop</a> and <a href="/wiki/Apache_Spark" title="Apache Spark">Apache Spark</a>.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup></p>
<h2><span class="mw-headline" id="Implementations">Implementations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Bzip2&amp;action=edit&amp;section=13" title="Edit section: Implementations">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In addition to <a href="/wiki/Julian_Seward" title="Julian Seward">Julian Seward</a>'s original reference implementation, the following programs support bzip2 format.</p>
<ul>
<li><a href="/wiki/7-Zip" title="7-Zip">7-Zip</a>: Written by <a class="mw-redirect" href="/wiki/Igor_Pavlov_(Computer_programmer)" title="Igor Pavlov (Computer programmer)">Igor Pavlov</a> in <a href="/wiki/C%2B%2B" title="C++">C++</a>, the 7-Zip suite contains a bzip2 encoder/decoder which is freely licensed. 7-Zip comes with multi-threading support.</li>
<li><a class="external text" href="http://www.landley.net/code/" rel="nofollow">micro-bzip2</a>: A version by <a class="new" href="/w/index.php?title=Rob_Landley&amp;action=edit&amp;redlink=1" title="Rob Landley (page does not exist)">Rob Landley</a> designed for reduced compiled code size and available under the GNU <a class="mw-redirect" href="/wiki/LGPL" title="LGPL">LGPL</a>.</li>
<li><a class="external text" href="http://compression.ca/pbzip2/" rel="nofollow">PBZIP2</a>: Parallel <a href="/wiki/POSIX_Threads" title="POSIX Threads">pthreads</a>-based implementation in <a href="/wiki/C%2B%2B" title="C++">C++</a> by Jeff Gilchrist (and <a class="external text" href="http://www.leszer.net/index.php/my-software.html" rel="nofollow">Windows version</a>).</li>
<li><a class="external text" href="http://bzip2smp.sourceforge.net/" rel="nofollow">bzip2smp</a>: A modification to <code>libbzip2</code> that has <a href="/wiki/Symmetric_multiprocessing" title="Symmetric multiprocessing">SMP</a> parallelisation <cite>"hacked in"</cite> by Konstantin Isakov.</li>
<li><a class="external text" href="http://home.student.utwente.nl/n.werensteijn/smpbzip2/" rel="nofollow">smpbzip2</a>: Another go at parallel bzip2, by Niels Werensteijn.</li>
<li><a class="external text" href="http://www.paul.sladen.org/projects/pyflate/" rel="nofollow">pyflate</a>: A pure-<a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a> stand-alone bzip2 and <a href="/wiki/DEFLATE" title="DEFLATE">DEFLATE</a> (<a href="/wiki/Gzip" title="Gzip">gzip</a>) decoder by Paul Sladen. Probably useful for research and prototyping, made available under the <a class="mw-redirect" href="/wiki/BSD_license" title="BSD license">BSD</a>/<a href="/wiki/GNU_General_Public_License" title="GNU General Public License">GPL</a>/<a href="/wiki/GNU_Lesser_General_Public_License" title="GNU Lesser General Public License">LGPL</a>, or any other <a href="/wiki/Debian_Free_Software_Guidelines" title="Debian Free Software Guidelines">DFSG</a>-compatible license.</li>

<li><a class="external text" href="http://bouchez.info/delphi-bzip.html" rel="nofollow">Arnaud Bouchez's BZip</a>: Implementation using C library or optimized i386 assembler code.</li>
<li><a class="external text" href="http://lbzip2.org" rel="nofollow">lbzip2</a>: Parallel <a href="/wiki/POSIX_Threads" title="POSIX Threads">pthreads</a>-based bzip2/bunzip2 (bzip2 compressor/decompressor) filter for sequential access input/output, by László Érsek.</li>
<li><a class="external text" href="http://compression.ca/mpibzip2/" rel="nofollow">mpibzip2</a>: An <a href="/wiki/Message_Passing_Interface" title="Message Passing Interface">MPI</a>-enabled implementation that compresses/decompresses in parallel, by Jeff Gilchrist, available under a BSD-style license.</li>
<li><a class="external text" href="http://commons.apache.org/compress/" rel="nofollow">Apache Commons Compress</a> Apache Commons Compress project contains Java implementations of Bzip2 compressor and decompressor.</li>
<li><a class="external text" href="https://code.google.com/p/jbzip2/" rel="nofollow">jbzip2</a>: A Java implementation of bzip2 made available under the <a class="mw-redirect" href="/wiki/MIT_license" title="MIT license">MIT license</a></li>
<li><a class="external text" href="http://dotnetzip.codeplex.com/" rel="nofollow">DotNetZip</a>: Includes a C# implementation of bzip2, derived from the Java implementation in Apache Commons Compress. Includes a multi-threaded .NET bzip2 encoder/decoder library, and a bzip2 command-line tool in managed code.</li>
<li><a class="external text" href="https://www.noemax.com/dotnetcompression/" rel="nofollow">DotNetCompression</a>: A streaming implementation of bzip2 in managed C# that conforms to the API of System.IO.Compression and includes assemblies for <a href="/wiki/.NET_Framework" title=".NET Framework">.NET Framework</a>, <a href="/wiki/.NET_Compact_Framework" title=".NET Compact Framework">.NET Compact Framework</a>, <a href="/wiki/Xamarin" title="Xamarin">Xamarin</a>.<a href="/wiki/IOS" title="IOS">iOS</a>, <a href="/wiki/Xamarin" title="Xamarin">Xamarin</a>.<a href="/wiki/Android_(operating_system)" title="Android (operating system)">Android</a>, <a href="/wiki/Xamarin" title="Xamarin">Xamarin</a>.<a href="/wiki/MacOS" title="MacOS">Mac</a>, <a href="/wiki/Windows_Phone" title="Windows Phone">Windows Phone</a>, <a href="/wiki/Xbox_360" title="Xbox 360">Xbox 360</a>, <a class="mw-redirect" href="/wiki/Silverlight" title="Silverlight">Silverlight</a>, <a href="/wiki/Mono_(software)" title="Mono (software)">Mono</a> and as a Portable Class Library.</li>
