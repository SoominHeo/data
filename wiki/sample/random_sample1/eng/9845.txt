<h1 class="firstHeading" id="firstHeading" lang="en">Enterprise service bus</h1>
<div class="mw-body-content" id="bodyContent">

<p>An <b>enterprise service bus</b> (<b>ESB</b>) is implementing a communication system between mutually interacting software applications in a <a href="/wiki/Service-oriented_architecture" title="Service-oriented architecture">service-oriented architecture</a> (SOA). It implements a <a href="/wiki/Software_architecture" title="Software architecture">software architecture</a> as depicted on the right. As it implements a software architecture for distributed computing, it implements a special variant of the more general <a class="mw-redirect" href="/wiki/Client-server" title="Client-server">client-server</a> model therefore also. Whereas in general any application using ESB can behave as server or client in turns. ESB promotes agility and flexibility with regard to high protocol-level communication between applications. The primary goal of the high protocol-level communication is <a href="/wiki/Enterprise_application_integration" title="Enterprise application integration">enterprise application integration</a> (EAI) of heterogeneous and complex landscapes.</p>

<h2><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Enterprise_service_bus&amp;action=edit&amp;section=1" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The concept is analogous to the <a href="/wiki/Bus_(computing)" title="Bus (computing)">bus</a> concept found in <a href="/wiki/Computer_architecture" title="Computer architecture">computer hardware architecture</a> combined with the modular and concurrent design of high-performance computer operating systems. The motivation was to find a standard, structured, and general purpose concept for describing implementation of loosely coupled software components (called <a href="/wiki/Service_(systems_architecture)" title="Service (systems architecture)">services</a>) that are expected to be independently deployed, running, heterogeneous, and disparate within a network. ESB is also a common implementation pattern for <a href="/wiki/Service-oriented_architecture" title="Service-oriented architecture">service-oriented architecture</a>.</p>
<h3><span class="mw-headline" id="Working">Working</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Enterprise_service_bus&amp;action=edit&amp;section=2" title="Edit section: Working">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>An ESB transports the design concept of modern <a href="/wiki/Operating_system" title="Operating system">operating systems</a> to networks of disparate and independent computers. Like concurrent operating systems an ESB caters for commodity services in addition to adoption, translation and routing of a client request to the appropriate answering service.</p>
<p>The primary duties of an ESB are:</p>
<ul>
<li>Monitor and control routing of message exchange between services</li>
<li>Resolve contention between communicating service components</li>
<li>Control deployment and versioning of services</li>
<li>Marshal use of redundant services</li>
<li>Cater for commodity services like event handling, data transformation and mapping, message and event queuing and sequencing, security or <a href="/wiki/Exception_handling" title="Exception handling">exception handling</a>, protocol conversion and enforcing proper quality of communication service</li>
</ul>
<h2><span class="mw-headline" id="Ambiguous_use_of_the_term_ESB_in_commerce">Ambiguous use of the term ESB in commerce</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Enterprise_service_bus&amp;action=edit&amp;section=3" title="Edit section: Ambiguous use of the term ESB in commerce">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>There is no global standard for enterprise service bus concepts or implementations.<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> Most providers of <a href="/wiki/Message-oriented_middleware" title="Message-oriented middleware">message-oriented middleware</a> have adopted the enterprise service bus concept as de facto standard for a service-oriented architecture. The implementations of ESB use <a href="/wiki/Event-driven_architecture" title="Event-driven architecture">event-driven</a> and standards-based message-oriented middleware in combination with <a href="/wiki/Message_queue" title="Message queue">message queues</a> as technology frameworks.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> However, some software manufacturers relabel their existing middleware and communication solutions as ESB without adopting the crucial aspect of a bus concept.</p>
<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Enterprise_service_bus&amp;action=edit&amp;section=4" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The first published usage of the term "enterprise service bus" is attributed to Roy W. Schulte from the Gartner Group 2002 and the book <i>The Enterprise Service Bus</i> by David Chappell.</p>
<ul>
<li>Service - denotes non-iterative and autonomously executing programs that communicate with other services through message exchange</li>
<li>Bus - is used in analogy to a computer hardware <a href="/wiki/Bus_(computing)" title="Bus (computing)">bus</a></li>
<li>Enterprise - the concept has been originally invented to reduce complexity of enterprise application integration within an enterprise; the restriction has become obsolete since modern Internet communication is no longer limited to a corporate entity</li>
</ul>
<p>In fact, the term "bus" was created in the 1980s by Teknekron Software Systems<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (January 2013)">citation needed</span></a></i>]</sup>. Frustrated by how software seemed to always under-deliver, while hardware was always on time and under budget, <a href="/wiki/Vivek_Ranadiv%C3%A9" title="Vivek Ranadivé">Vivek Ranadivé</a> set out to build software based on the premise of a "Software Bus" (which later became known as "The Information Bus" or TIB), where a "bus" is the standard data highway by which various elements—such as a computer system such as the CPU, the memory, the I/O devices, etc.—communicate. This concept would allow for the "tight" coupling of applications.</p>
<p>In 1986 <a href="/wiki/Teknekron_Corporation" title="Teknekron Corporation">Teknekron Corporation</a> embarked on a consulting project with <a href="/wiki/Goldman_Sachs" title="Goldman Sachs">Goldman Sachs</a> to redefine the "trading floor of the future" applying this approach. In 1987 the first TIB—for the integration and delivery of market data such as stock quotes, news, and other financial information—went live at <a href="/wiki/Fidelity_Investments" title="Fidelity Investments">Fidelity</a><sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (January 2013)">citation needed</span></a></i>]</sup>, followed by <a href="/wiki/First_Interstate_Bancorp" title="First Interstate Bancorp">First Interstate Bank</a><sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (January 2013)">citation needed</span></a></i>]</sup>, then <a href="/wiki/Salomon_Group" title="Salomon Group">Salomon</a>, eventually digitizing all of <a href="/wiki/Wall_Street" title="Wall Street">Wall Street</a><sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (January 2013)">citation needed</span></a></i>]</sup>. Teknekron was later <a href="/wiki/Takeover" title="Takeover">acquired</a> by <a href="/wiki/Reuters" title="Reuters">Reuters</a> in 1994<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (January 2013)">citation needed</span></a></i>]</sup> to expand its use of the Information Bus in the financial services markets. In January 1997, Ranadivé founded <a class="mw-redirect" href="/wiki/Tibco_Software" title="Tibco Software">Tibco Software</a> Inc. to create and market software for use in the integration of business applications outside the financial services sector. In 1998 Tibco Software released TIB/ActiveEnterprise suite. In July 1999 Tibco <a href="/wiki/Initial_public_offering" title="Initial public offering">went public</a> on the <a href="/wiki/NASDAQ" title="NASDAQ">NASDAQ</a> Stock Market<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (January 2013)">citation needed</span></a></i>]</sup> under the <a href="/wiki/Ticker_symbol" title="Ticker symbol">ticker symbol</a> TIBX. Tibco stands for The Information Bus Company.</p>

<p>The ESB is implemented in software that operates between the business applications, and enables communication among them. Ideally, the ESB should be able to replace all direct contact with the applications on the bus, so that all communication takes place via the ESB. To achieve this objective, the ESB must <a href="/wiki/Information_hiding" title="Information hiding">encapsulate</a> the functionality offered by its component applications in a meaningful way. This typically occurs through the use of an <a href="/wiki/Enterprise_messaging_system" title="Enterprise messaging system">enterprise message model</a>. The message model defines a standard set of messages that the ESB transmits and receives. When the ESB receives a message, it routes the message to the appropriate application. Often, because that application evolved without the same message model, the ESB has to transform the message into a format that the application can interpret. A software adapter fulfills the task of effecting these transformations, analogously to a physical <a href="/wiki/Adapter" title="Adapter">adapter</a>.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup></p>
<p>ESBs rely on accurately constructing the enterprise message model and properly designing the functionality offered by applications. If the message model does not completely <a class="mw-redirect" href="/wiki/Encapsulation_(computer_science)" title="Encapsulation (computer science)">encapsulate</a> the application functionality, then other applications that desire that functionality may have to bypass the bus, and <a href="/wiki/Invocational_media" title="Invocational media">invoke</a> the mismatched applications directly. Doing so violates the principles of the ESB model, and negates many of the advantages of using this architecture.</p>
<p>The beauty of the ESB lies in its platform-agnostic nature and the ability to integrate with anything at any condition. It is important that <a class="mw-redirect" href="/wiki/Application_Lifecycle_Management" title="Application Lifecycle Management">Application Lifecycle Management</a> vendors truly apply all the ESB capabilities in their integration products while adopting <a href="/wiki/Service-oriented_architecture" title="Service-oriented architecture">SOA</a>. Therefore, the challenges and opportunities for <a href="/wiki/Enterprise_application_integration" title="Enterprise application integration">EAI</a> vendors are to provide an integration solution that is low-cost, easily configurable, intuitive, user-friendly, and open to any tools customers choose.</p>

<p>² <i>While process choreography supports implementation of complex business processes that require coordination of multiple <a href="/wiki/Business" title="Business">business</a> services (usually using <a class="mw-redirect" href="/wiki/BPEL" title="BPEL">BPEL</a>), <a class="mw-redirect" href="/wiki/Service_orchestration" title="Service orchestration">service orchestration</a> enables coordination of multiple implementation services (most suitably exposed as an aggregate service) to serve individual requests..</i></p>
<p>These solutions often focus on low-level ESB functions, such as connectivity, routing and transformation, and require coding or scripting to implement orchestration.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup> Developers operating at a project or tactical level, e.g., just trying to fix a problem, often gravitate toward lightweight service bus technologies, but there is often ongoing tension between these initiatives and an enterprise architecture whose goal it is to optimize infrastructure across multiple projects.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup></p>
<p>If the message broker, the ESB software, translates a message from one format to another, then as with any translation, there is the issue of semantics of the message. For example, a record can be translated from JSON to XML, but the same set of fields can be interpreted differently by different applications, specially in the case of the various corner cases that are usually known only to developers that have extensive experience with the application that is connected to the ESB. For the known corner cases the number of tests that cover all corner cases increases exponentially with every application that is connected to the ESB, because every ESB-connected application must be tested against every other application that is connected to the ESB.</p>
<h2><span class="mw-headline" id="Key_benefits">Key benefits</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Enterprise_service_bus&amp;action=edit&amp;section=7" title="Edit section: Key benefits">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<ul>
<li>Scales from point-solutions to enterprise-wide deployment (distributed bus)</li>
<li>More configuration rather than integration coding</li>
<li>No central rules-engine, no central broker</li>
<li>Easy plug-in and plug-out and loosely coupling system</li>
<li>Incremental <a href="/wiki/Patch_(computing)" title="Patch (computing)">patching</a> with zero down-time; enterprise becomes "<a href="/wiki/Code_refactoring" title="Code refactoring">refactorable</a>"</li>
</ul>
<h2><span class="mw-headline" id="Key_disadvantages">Key disadvantages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Enterprise_service_bus&amp;action=edit&amp;section=8" title="Edit section: Key disadvantages">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<ul>
<li>Slower communication speed, especially for those already compatible services</li>
<li>Single point of failure, can bring down all communications in the Enterprise</li>
<li>High configuration and maintenance complexity</li>
</ul>
