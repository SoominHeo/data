<h1 class="firstHeading" id="firstHeading" lang="ko">Null (SQL)</h1>
<div class="mw-body-content" id="bodyContent">
<div id="siteSub">위키백과, 우리 모두의 백과사전.</div>

<p><b>NULL</b>은 <a class="mw-redirect" href="/wiki/%EA%B5%AC%EC%A1%B0%EC%A0%81_%EC%A7%88%EC%9D%98_%EC%96%B8%EC%96%B4" title="구조적 질의 언어">구조적 질의 언어</a> (SQL)에서 <a href="/wiki/%EB%8D%B0%EC%9D%B4%ED%84%B0%EB%B2%A0%EC%9D%B4%EC%8A%A4" title="데이터베이스">데이터베이스</a> 내의 데이터 값이 존재하지 않는다는 것을 지시하는데 사용되는 특별한 표시어이다. 데이터베이스 <a class="mw-redirect" href="/wiki/%EA%B4%80%EA%B3%84_%EB%AA%A8%EB%8D%B8" title="관계 모델">관계 모델</a>의 창시자인 E. F. 콧(E. F. Codd)에 만들어진 개념으로, SQL NULL은 모든 true RDBMS가 빠진 정보와 적용할 수 없는 정보를 지원해야 한다는 요구사항을 충족시키는 역할을 한다. 콧은 또한 데이터베이스 이론에서 Null을 표현하기 위해 <a href="/wiki/%EA%B7%B8%EB%A6%AC%EC%8A%A4%EC%96%B4" title="그리스어">그리스어</a>의 <a class="mw-redirect" href="/wiki/%EC%98%A4%EB%A9%94%EA%B0%80" title="오메가">오메가</a>의 소문자(ω)를 이용할 것을 도입했다. NULL은 또한 SQL에서 Null 특수 표시를 구분하기 위해 사용되는 유보된 키워드이다.</p>
<p>Null은 SQL 조인에서 사용하기 위해 필수적인 <a class="new" href="/w/index.php?title=3%EC%B9%98_%EC%9E%90%EA%B6%81&amp;action=edit&amp;redlink=1" title="3치 자궁 (없는 문서)">3치 자궁</a>로 인해 논란의 중심이 되어 왔다. 컴퓨터 과학자인 론 밴 더 메이든 교수는 그러한 여러 문제를 다음과 같이 요약했다. SQL 표준에서 비일관성은 SQL에서 널의 취급 문제를 직관적인 논리 어휘 탓으로 돌리는 것은 가능하지 않다는 것을 의미하는 것이다.<sup class="reference" id="cite_ref-Meyden_1-0"><a href="#cite_note-Meyden-1">[1]</a></sup> 이 문제를 해결하기 위해 여러 해법들이 제시되었지만, 대안의 복잡성은 폭넓은 이론 확산의 걸림돌이 되었다.</p>
<p>데이터베이스 비전문가들에게, 널이 뜻하는 바가 무엇인지 이해하는 좋은 방법은 정보적 측면을 기억하는 것이다. “값의 부족”은 0값과 동일한 의미가 아니며, 유사하게 ‘정답의 부족’이라는 말도 정답이 없다는 말과 같은 의미가 아니다. 예를 들어, 다음과 같은 질문을 예로 들어보자. “철수가 책을 몇 권 가지고 있지?” 답은 0(하나도 가지고 있지 않다는 것을 안다면) 또는 null(그가 가지고 있는지, 아닌지 또는 얼마나 가지고 있는 지 모름)일 것이다. 데이터베이스 테이블에서, 이러한 답을 가진 컬럼은 null 값으로 시작할 것이다. 그리고 철수가 책을 가지고 있다는 것이 확인되기 전까지는 0으로 수정되지 않을 것이다. 유사하게, 질문이 “철수가 차를 가지고 있어?”라는 질문에도 “모르겠는데”라는 정답은 “아니!(No)”라는 것과 같은 의미가 아니다. 전자는 데이터베이스의 Null 목록을 생성하며, 후자는 데이터베이스 엔트리에서 No를 생성한다.</p>

<h2><span class="mw-headline" id=".EC.97.AD.EC.82.AC">역사</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Null_(SQL)&amp;action=edit&amp;section=1" title="부분 편집: 역사">편집</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>E. F. 콧(E. F. Codd)은 1975년 그의 논문 《<i>FDT Bulletin of ACM-SIGMOD</i>》에서 널을 빠진 데이터를 나타내는 수단으로서 언급했다. SQL에서 채택된 Null을 설명하기 위해 가장 보편적으로 인용되는 콧의 논문은 1979년 논문이었던 《<i><a class="extiw" href="https://en.wikipedia.org/wiki/ACM_Transactions_on_Database_Systems" title="w:ACM Transactions on Database Systems">ACM Transactions on Database Systems</a></i>》으로 비록 이후의 논문에서 많은 제안들 대부분이 모호한 채로 남아있지만, 여기에서 그는 또는 <a class="mw-redirect" href="/wiki/%EA%B4%80%EA%B3%84_%EB%AA%A8%EB%8D%B8" title="관계 모델">관계 모델</a>/태즈매니아를 소개했다. 1979년 논문 2.3장에는 수학적 연산에서의 널 보급의 세부적인 사항과 널 값을 비교할 때, <a class="new" href="/w/index.php?title=3%EA%B0%80_%EB%85%BC%EB%A6%AC&amp;action=edit&amp;redlink=1" title="3가 논리 (없는 문서)">3가 논리</a>를 이용하는 비교도 소개를 했다. 여기에는 또한 다른 연산자 셋과 Null을 취급하는 것도 소개를 했으며, 후자의 이슈는 오늘 날에도 여전히 논란이 많은 문제이다. 데이터베이 이론계에서는, 1975년과 1979년 제안한 콧의 원래 이론들이 지금은 ‘콧의 테이블’로 언급되고 있다.<sup class="reference" id="cite_ref-Meyden_1-1"><a href="#cite_note-Meyden-1">[1]</a></sup> 콧은 이후에 모든 RDBMS가 빠진 데이터를 지시하기 위해 Null에 지원해야할 그의 요구사항을 <a href="/wiki/1985%EB%85%84" title="1985년">1985년</a> 《컴퓨터월드 매거진》에 기고한 2개의 글을 통해 보강했다.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup><sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup></p>
<p>1986년 SQL 표준은 근본적으로 <a class="new" href="/w/index.php?title=IBM_%EC%8B%9C%EC%8A%A4%ED%85%9C_R&amp;action=edit&amp;redlink=1" title="IBM 시스템 R (없는 문서)">IBM 시스템 R</a>에서 수행 포로토타입 이후 콧의 제안을 채택했다. 비록 돈 챔벌린은 널 값을 SQL에서 가장 논쟁적인 특징으로 규정했지만, 그는 빠진 정보를 설명하는 가장 비용이 적게 드는 시스템 지원이라는 실용적인 측면을 들어 SQL에서 널의 설계를 옹호했다.</p>
<h2><span class="mw-headline" id="Null_.EB.B3.B4.EA.B8.89">Null 보급</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Null_(SQL)&amp;action=edit&amp;section=2" title="부분 편집: Null 보급">편집</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id=".EC.88.98.ED.95.99.EC.A0.81_.EC.97.B0.EC.82.B0">수학적 연산</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Null_(SQL)&amp;action=edit&amp;section=3" title="부분 편집: 수학적 연산">편집</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>널은 데이터 값이 아니라, 미지의 값에 대한 표시일 뿐이기 때문에, Null에 수학적 연산을 사용하는 것은 미지의 값으로 나타난다.<sup class="reference" id="cite_ref-SQL2003-Part2-Sec626_4-0"><a href="#cite_note-SQL2003-Part2-Sec626-4">[4]</a></sup> 아래의 예에서, 10을 Null에 곱하면 결과값은 Null이 된다.:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="mi">10</span> <span class="o">*</span> <span class="k">NULL</span>          <span class="c1">-- 결과는 NULL</span>
</pre></div>
<p>이것은 예기치 않은 결과를 야기한다. 예를 들어, 널을 0으로 나누려 한다면, 플랫폼은 ‘0으로 나눈’ 예상된 데이터 예외값을 던지는 대신, 널 값을 반환한다.<sup class="reference" id="cite_ref-SQL2003-Part2-Sec626_4-1"><a href="#cite_note-SQL2003-Part2-Sec626-4">[4]</a></sup> 비록 이러한 행위가 ISO SQL 표준으로 정의되어 있지는 않지만, 많은 DBMS 벤더들이 이러한 연산을 유사하게 다룬다. 예를 들어, 오라클, PostgreSQL, MySQL 서버, 그리고 마이크로소프트 SQL 서버 플랫폼은 모두 널 값을 아래와 같이 반환한다 :</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">NULL</span> <span class="o">/</span> <span class="mi">0</span>
</pre></div>
<h3><span class="mw-headline" id=".EB.AC.B8.EC.9E.90.EC.97.B4_.EC.97.B0.EC.86.8D">문자열 연속</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Null_(SQL)&amp;action=edit&amp;section=4" title="부분 편집: 문자열 연속">편집</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>SQL에서 흔히 볼 수 있는 문자열 연속 연산 또한 연산자 중 하나가 Null 값일 때 Null 값으로 나타난다.<sup class="reference" id="cite_ref-SQL2003-Part2-Sec628_5-0"><a href="#cite_note-SQL2003-Part2-Sec628-5">[5]</a></sup> 아래의 예제는 SQL || 문자열 연속 연산자에 Null을 사용함으로써 널 값이 리턴되는 것을 보여준다.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="s1">'생선'</span> <span class="o">||</span> <span class="k">NULL</span> <span class="o">||</span> <span class="s1">'김치'</span>   <span class="c1">-- 결과는 NULL</span>
</pre></div>
<p>모든 데이터베이스에서 이러한 것은 아니다. 오라클 RDBMS를 예로 들면, NULL 과 빈 문자열은 동일한 것으로 취급되어, '생선' || NULL || '김치'는 ‘생선 김치’로 나타난다.</p>
<h2><span class="mw-headline" id="NULL.EA.B3.BC_3.EA.B0.80_.EB.85.BC.EB.A6.AC.283VL.29.EC.9D.98_.EB.B9.84.EA.B5.90">NULL과 3가 논리(3VL)의 비교</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Null_(SQL)&amp;action=edit&amp;section=5" title="부분 편집: NULL과 3가 논리(3VL)의 비교">편집</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="rellink boilerplate seealso"><img alt="" data-file-height="128" data-file-width="128" height="16" src="//upload.wikimedia.org/wikipedia/commons/thumb/e/ec/Crystal_Clear_app_xmag.svg/16px-Crystal_Clear_app_xmag.svg.png" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/e/ec/Crystal_Clear_app_xmag.svg/24px-Crystal_Clear_app_xmag.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/e/ec/Crystal_Clear_app_xmag.svg/32px-Crystal_Clear_app_xmag.svg.png 2x" width="16"/> <a class="new" href="/w/index.php?title=3%EA%B0%80_%EB%85%BC%EB%A6%AC&amp;action=edit&amp;redlink=1" title="3가 논리 (없는 문서)">3가 논리</a> 문서를 참고하십시오.</div>
<p>널은 데이터 도메인의 한 부분이 아니기 때문에, 값으로 여기지지는 않지만, 값의 부재를 표시하는 표시자서의 역할을 한다. 이러한 이유로 인해, 널과 비교를 하는 것은 ‘참’(T)도 ‘거짓(F)’도 아니며, 항상 제3의 값, ‘미지’(unknown)로 나타난다.<sup class="reference" id="cite_ref-SQL2003-Part1-Sec442_6-0"><a href="#cite_note-SQL2003-Part1-Sec442-6">[6]</a></sup> 10과 Null을 비교하는 아래 표현의 논리적 결과는 Unknown이다:</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">SELECT</span> <span class="mi">10</span> <span class="o">=</span> <span class="k">NULL</span>       <span class="c1">-- 결과는 Unknown</span>
</pre></div>
<p>그러나, 만약 Null 값이 연산의 결과물에 적합하지 않을 때 Null에 대한 어떤 연산은 값을 리턴할 수 있다. 아래의 예제를 보자 :</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="k">SELECT</span> <span class="k">NULL</span> <span class="k">OR</span> <span class="k">TRUE</span>   <span class="c1">-- 결과는 True</span>
</pre></div>
<p>이 경우, OR의 왼쪽에 있는 값이 알 수 없다는 사실이 부적절하다. 왜냐하면, OR 연산의 결과가 왼쪽 값에 상관없이 참일 것이기 때문이다.</p>
