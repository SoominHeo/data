<p><b>RISC</b>(Reduced Instruction Set Computer)는 CPU 명령어의 개수를 줄여 하드웨어 구조를 좀 더 간단하게 만드는 방식으로, <a href="/wiki/%EB%A7%88%EC%9D%B4%ED%81%AC%EB%A1%9C%ED%94%84%EB%A1%9C%EC%84%B8%EC%84%9C" title="마이크로프로세서">마이크로프로세서</a>를 설계하는 방법 가운데 하나이며, <a href="/wiki/SPARC" title="SPARC">SPARC</a>, <a href="/wiki/MIPS_%EC%95%84%ED%82%A4%ED%85%8D%EC%B2%98" title="MIPS 아키텍처">MIPS</a> 등의 아키텍처에서 사용된다. ll</p>
<p>전통적인 <a href="/wiki/CISC" title="CISC">CISC</a> <a href="/wiki/%EC%A4%91%EC%95%99_%EC%B2%98%EB%A6%AC_%EC%9E%A5%EC%B9%98" title="중앙 처리 장치">CPU</a>에는 <a href="/wiki/%EC%BB%B4%ED%93%A8%ED%84%B0_%ED%94%84%EB%A1%9C%EA%B7%B8%EB%9E%98%EB%B0%8D" title="컴퓨터 프로그래밍">프로그래밍</a>을 돕기 위한 많은 수의 명령어와 주소 모드가 존재했다. 그러나 그중에서 실제로 쓰이는 명령어는 몇 개 되지 않는다는 사실을 바탕으로, 적은 수의 명령어만으로 명령어 집합을 구성한 것이 RISC이다. 그래서, RISC는 CISC보다 구조가 더 단순하다. 복잡한 연산도 적은 수의 명령어들을 조합하는 방식으로 수행이 가능하다.</p>
<p>그리고 <a href="/wiki/CISC" title="CISC">CISC</a> 형식의 <a href="/wiki/%EC%A4%91%EC%95%99_%EC%B2%98%EB%A6%AC_%EC%9E%A5%EC%B9%98" title="중앙 처리 장치">CPU</a>내 <a href="/wiki/%EA%B3%A0%EC%A0%95_%EA%B8%B0%EC%96%B5_%EC%9E%A5%EC%B9%98" title="고정 기억 장치">ROM</a>에 <a href="/wiki/%EC%86%8C%ED%94%84%ED%8A%B8%EC%9B%A8%EC%96%B4" title="소프트웨어">소프트웨어</a>적으로 적재된 내부 명령어들을 <a href="/wiki/%EC%BB%B4%ED%93%A8%ED%84%B0_%ED%95%98%EB%93%9C%EC%9B%A8%EC%96%B4" title="컴퓨터 하드웨어">하드웨어</a>적으로 구성하여 제어기가 제거된 부분에 <a href="/wiki/%ED%94%84%EB%A1%9C%EC%84%B8%EC%84%9C_%EB%A0%88%EC%A7%80%EC%8A%A4%ED%84%B0" title="프로세서 레지스터">프로세서 레지스터</a> 뱅크와 <a href="/wiki/%EC%BA%90%EC%8B%9C" title="캐시">캐시</a>를 둔다. 이렇게 함으로써 CPU가, 상대적으로 느린 메인 메모리에 접근하는 횟수를 줄여주어 파이프라이닝 등 시스템 수행속도가 전체적으로 향상된다.</p>
<h2><span class="mw-headline" id=".ED.8A.B9.EC.A7.95">특징</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=RISC&amp;action=edit&amp;section=1" title="부분 편집: 특징">편집</a><span class="mw-editsection-bracket">]</span></span></h2>
<ul>
<li>고정 길이의 명령어를 사용하여 더욱 빠르게 해석할 수 있다.</li>
<li>모든 연산은 하나의 클럭으로 실행되므로 파이프라인을 기다리게 하지 않는다.</li>
<li>­레지스터 사이의 연산만 실행하며, 메모리 접근은 로드(load), 스토어(store) 명령어로 제한된다. 이렇게 함으로써 회로가 단순해지고, 불필요한 메모리 접근을 줄일 수 있다.</li>
<li><a class="new" href="/w/index.php?title=%EB%A7%88%EC%9D%B4%ED%81%AC%EB%A1%9C%EC%BD%94%EB%93%9C_%EB%85%BC%EB%A6%AC&amp;action=edit&amp;redlink=1" title="마이크로코드 논리 (없는 문서)">마이크로코드 논리</a>를 사용하지 않아 높은 클럭을 유지할 수 있다.</li>
<li>많은 수의 레지스터를 사용하여 메모리 접근을 줄인다.</li>
<li>지연 실행 기법을 사용하여 파이프라인의 위험을 줄인다.</li>
</ul>
<p>CISC에서는 하드웨어가 <a href="/wiki/%EC%8A%A4%ED%83%9D" title="스택">스택</a>을 지원하지만, RISC에는 없다. 스택 제어(데이터의 PUSH, POP이 발생할 때 레지스터의 퇴피, 서브루틴에 점프했을 때의 리턴 주소의 보존, 복귀)의 처리는 단순한 명령을 조합하여 소프트웨어로 구현된다. 명령어의 순서로 인해 발생할 수 있는 파이프라인의 위험은 <a class="mw-redirect" href="/wiki/%EC%BB%B4%ED%8C%8C%EC%9D%BC" title="컴파일">컴파일</a>하는 동안 최적화되어 사라진다.</p>
<p>명령어의 길이를 고정하면 파이프라인 처리의 고속화를 꾀할 수 있지만, <a href="/wiki/%EC%BB%B4%ED%8C%8C%EC%9D%BC%EB%9F%AC" title="컴파일러">컴파일러</a>의 최적화 과정이 복잡해지기 쉽다.</p>
<p>RISC는 대부분의 현대 프로세서 디자인에 채택되고 있고, 또 비교적 전력 소모가 적기 때문에 <a class="mw-redirect" href="/wiki/%EC%9E%84%EB%B2%A0%EB%94%94%EB%93%9C_%ED%94%84%EB%A1%9C%EC%84%B8%EC%84%9C" title="임베디드 프로세서">임베디드 프로세서</a>에도 채택되고 있다.</p>
<p><a href="/wiki/%ED%8E%9C%ED%8B%B0%EC%97%84" title="펜티엄">펜티엄</a>과 같은 CISC CPU도 내부적으로는 복잡한 명령들을 단순한 명령들로 나누어 파이프라인에서 처리하기 때문에, 실제 작동 원리는 RISC와 같다.</p>
