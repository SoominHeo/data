<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="ko"><script>function mfTempOpenSection(id){var block=document.getElementById("mf-section-"+id);block.className+=" open-block";block.previousSibling.className+=" open-block";}</script><p><b>정적 라이브러리</b>는 루틴들과 외부 함수들 그리고 변수들의 집합으로서, <a href="/wiki/%EC%BB%B4%ED%8C%8C%EC%9D%BC_%ED%83%80%EC%9E%84" title="컴파일 타임">컴파일 타임</a>에 호출자에 의해 리졸브되며 컴파일러와 링커에 의해 <a href="/wiki/%EB%AA%A9%EC%A0%81_%ED%8C%8C%EC%9D%BC" title="목적 파일">목적 파일</a>과 독립된 실행 파일을 생성하기 위해 대상 애플리케이션에 복사된다.<sup class="reference" id="cite_ref-Static_Libraries_1-0"><a href="#cite_note-Static_Libraries-1">[1]</a></sup>이 실행 파일과 이것을 컴파일링 하는 과정 모두 프로그램의 <a class="new" href="/w/index.php?title=%EC%A0%95%EC%A0%81_%EB%B9%8C%EB%93%9C&amp;action=edit&amp;redlink=1" title="정적 빌드 (없는 문서)">정적 빌드</a>로 알려져 있다. 역사적으로, 라이브러리들은 단지 정적이었었다. 정적 라이브러리들은 다른 정적 라이브러리와 <a href="/wiki/%EB%AA%A9%EC%A0%81_%ED%8C%8C%EC%9D%BC" title="목적 파일">목적 파일</a>들과 빌딩/링킹 시에 단일 <a href="/wiki/%EC%8B%A4%ED%96%89_%ED%8C%8C%EC%9D%BC" title="실행 파일">실행 파일</a>을 생성하기 위해 통합되거나 <a href="/wiki/%EB%9F%B0%ED%83%80%EC%9E%84" title="런타임">런타임</a> 시에 메모리에 올라온 실행 파일의 <a href="/wiki/%EC%A3%BC%EC%86%8C_%EA%B3%B5%EA%B0%84" title="주소 공간">주소 공간</a>(컴파일 타임/<a href="/wiki/%EB%A7%81%ED%81%AC_%ED%83%80%EC%9E%84" title="링크 타임">링크 타임</a>에 결정되는 정적 메모리 오프셋)에 로드된다.</p>

<h2><span class="mw-headline" id=".EC.9E.A5.EC.A0.90.EA.B3.BC_.EB.8B.A8.EC.A0.90">장점과 단점</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=%EC%A0%95%EC%A0%81_%EB%9D%BC%EC%9D%B4%EB%B8%8C%EB%9F%AC%EB%A6%AC&amp;action=edit&amp;section=1" title="부분 편집: 장점과 단점">편집</a><span class="mw-editsection-bracket">]</span></span></h2>
<p><a href="/wiki/%EB%8F%99%EC%A0%81_%EB%9D%BC%EC%9D%B4%EB%B8%8C%EB%9F%AC%EB%A6%AC" title="동적 라이브러리">동적으로 링크</a>하는 대신, 정적으로 라이브러리를 실행 파일에 링크하는 것은 여러가지 장점이 있다. 가장 눈에 띄는 점은 애플리케이션이 자신의 모든 라이브러리들이 존재하고 그것이 정확한 버전인 것을 확신할 수 있다는 점이다. 이것은 <a href="/wiki/DLL_%EC%A7%80%EC%98%A5" title="DLL 지옥">DLL 지옥</a> 또는 그 이상의 의존성 지옥으로 알려진, 의존성과 관련된 문제들을 피할 수 있다. 정적 링킹은 또한 애플리케이션이 단일 실행 파일에 포함되게 함으로써 배포와 설치를 간단하게 만든다.</p>
<p>정적 링킹을 통해서 직접 또는 간접적으로 대상 실행 파일(또는 라이브러리)에 참조되는 라이브러리의 부분들을 충분히 포함할 수 있다. 동적 라이브러리를 통해서는 전체 라이브러리가 로드되며, 함수가 애플리케이션에 의해 호출되는 이 방식의 장점은 알려져 있지 않다. 이 장점이 중요한지는 라이브러리의 실제 구조에 달려있다.</p>
<p>정적 링킹에서는 라이브러리 코드가 실행 파일 내에 저장되기 때문에, 실행 파일의 크기는 분리된 파일에 저장하는 동적 링킹 때 보다 커진다. 그러나 라이브러리 파일들이 애플리케이션의 부분으로서 세어진다면, 전체 크기는 비슷하거나 컴파일러가 사용되지 않은 심볼들을 제거한 경우에는 더 작아질 수 있다. <a href="/wiki/%EB%A7%88%EC%9D%B4%ED%81%AC%EB%A1%9C%EC%86%8C%ED%94%84%ED%8A%B8_%EC%9C%88%EB%8F%84%EC%9A%B0" title="마이크로소프트 윈도우">마이크로소프트 윈도우</a>에서 애플리케이션이 필요한 라이브러리 파일들을 포함하는 것은 흔한 일이다.<sup class="reference" id="cite_ref-endofdllhell_2-0"><a href="#cite_note-endofdllhell-2">[2]</a></sup> <a href="/wiki/%EC%9C%A0%EB%8B%89%EC%8A%A4_%EA%B3%84%EC%97%B4" title="유닉스 계열">유닉스 계열</a> 시스템들에서는 패키지 관리 시스템들이 사용 가능한 정확한 라이브러리 파일들을 보장하기 때문에 윈도우보다는 덜 흔한 일이다. 이것은 라이브러리 파일들이 많은 애플리케이션들 사이에서 공유될 수 있게 하며, 공간을 절약한다. 또한 라이브러리를 사용하는 애플리케이션의 업데이트 대신, 라이브러리만을 업데이트함으로써 버그와 보안 결함을 고칠수 있게 한다. 실제로 많은 실행 파일들이(특히 윈도우 용) 정적 라이브러리와 동적 라이브러리들을 모두 사용한다.</p>
<h2><span class="mw-headline" id=".EB.A7.81.ED.82.B9.EA.B3.BC_.EB.A1.9C.EB.94.A9">링킹과 로딩</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=%EC%A0%95%EC%A0%81_%EB%9D%BC%EC%9D%B4%EB%B8%8C%EB%9F%AC%EB%A6%AC&amp;action=edit&amp;section=2" title="부분 편집: 링킹과 로딩">편집</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>어떤 정적 라이브러리 함수도 다른 정적 라이브러리에서 함수나 프로시저를 호출할 수 있다. <a href="/wiki/%EB%A7%81%EC%BB%A4_(%EC%BB%B4%ED%93%A8%ED%8C%85)" title="링커 (컴퓨팅)">링커</a>와 로더는 다른 목적 파일들과 같이 이것을 같은 방식으로 처리한다. 정적 라이브러리 파일들은 링킹 로더에 의해서 <a href="/wiki/%EB%9F%B0%ED%83%80%EC%9E%84" title="런타임">런타임</a> 시에 링크될 것이다.(예를 들면, <a href="/wiki/X_%EC%9C%88%EB%8F%84_%EC%8B%9C%EC%8A%A4%ED%85%9C" title="X 윈도 시스템">X11</a> 모듈 로더). 그러나 이러한 과정이 정적 링킹으로 불릴수 있는지에 관해서는 논란이 있다.</p>
<h2><span class="mw-headline" id="C.2FC.2B.2B.EC.97.90.EC.84.9C_.EC.A0.95.EC.A0.81_.EB.9D.BC.EC.9D.B4.EB.B8.8C.EB.9F.AC.EB.A6.AC_.EC.83.9D.EC.84.B1">C/C++에서 정적 라이브러리 생성</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=%EC%A0%95%EC%A0%81_%EB%9D%BC%EC%9D%B4%EB%B8%8C%EB%9F%AC%EB%A6%AC&amp;action=edit&amp;section=3" title="부분 편집: C/C++에서 정적 라이브러리 생성">편집</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>정적 라이브러리들은 <a href="/wiki/C_(%ED%94%84%EB%A1%9C%EA%B7%B8%EB%9E%98%EB%B0%8D_%EC%96%B8%EC%96%B4)" title="C (프로그래밍 언어)">C</a>나 <a href="/wiki/C%2B%2B" title="C++">C++</a>에서 쉽게 만들어질 수 있다. 이 두 언어들은 외부 또는 내부 링크를 나타내고, 또한 다른 특징들을 제공하기 위해 기억 영역 분류를 제공한다. 이러한 라이브러리를 생성하기 위해서, 익스포트되는 함수/프로시저들과 다른 객체 변수들은 반드시 <i>external linkage</i>로 명시되어야 한다(즉, C <code>static</code> 키워드를 사용해서). 정적 라이브러리 파일 이름들은 보통 <a href="/wiki/%EC%9C%A0%EB%8B%89%EC%8A%A4_%EA%B3%84%EC%97%B4" title="유닉스 계열">유닉스 계열</a> 시스템에서는 ".a" 확장을 갖고<sup class="reference" id="cite_ref-Static_Libraries_1-1"><a href="#cite_note-Static_Libraries-1">[1]</a></sup> <a href="/wiki/%EB%A7%88%EC%9D%B4%ED%81%AC%EB%A1%9C%EC%86%8C%ED%94%84%ED%8A%B8_%EC%9C%88%EB%8F%84%EC%9A%B0" title="마이크로소프트 윈도우">마이크로소프트 윈도우</a>에서는 ".lib"을 갖는다.</p>
<p>예를 들면 파일들 <tt>class1.o</tt>, <tt>class2.o</tt>, <tt>class3.o</tt>를 사용해서 아카이브를 새성하기 위해서는 아래와 같은 명령어가 사용될 것이다:<sup class="reference" id="cite_ref-Static_Libraries_1-2"><a href="#cite_note-Static_Libraries-1">[1]</a></sup></p>
<pre>
ar rcs libclass.a class1.o class2.o class3.o
</pre>
<p><tt>class1.o</tt>, <tt>class2.o</tt>, and <tt>class3.o</tt> 를 사용한 프로그램을 컴파일하기 위해서는:</p>
<pre>
cc main.c libclass.a
</pre>
<p>또는 (만약 libclass.a 가 표준 라이브러리 경로 즉,  <tt>/usr/local/lib 같은 곳에 위치한다면</tt>)</p>
<pre>
cc main.c -lclass
</pre>
<p>또는 (링킹 동안)</p>
<pre>
ld ... main.o -lclass ...
</pre>
<p>이것 대신에:</p>
<pre>
cc main.c class1.o class2.o class3.o
</pre>
