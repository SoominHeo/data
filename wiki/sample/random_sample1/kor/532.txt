<h1 class="firstHeading" id="firstHeading" lang="ko">2038년 문제</h1>

<p><b>2038년 문제</b>(-年 問題, <span style="font-size: smaller;"><a href="/wiki/%EC%98%81%EC%96%B4" title="영어">영어</a>:</span> <span lang="en" xml:lang="en">year 2038 problem, Unix Millennium bug, Y2K38</span>)란 컴퓨터 분야에서 <a href="/wiki/2038%EB%85%84" title="2038년">2038년</a> 또는 전후에 특정한 컴퓨터 소프트웨어에 이상이 발생하는 문제를 말한다. 이 문제는 <a class="new" href="/w/index.php?title=POSIX_%EC%8B%9C%EA%B0%84&amp;action=edit&amp;redlink=1" title="POSIX 시간 (없는 문서)">POSIX 시간</a> 표기법을 이용하는 프로그램에서 발생하는데, 이 표기법은 시간을 <a href="/wiki/1970%EB%85%84" title="1970년">1970년</a> <a href="/wiki/1%EC%9B%94_1%EC%9D%BC" title="1월 1일">1월 1일</a> 자정 <a class="mw-redirect" href="/wiki/UTC" title="UTC">UTC</a> 이후 경과된 초 시간을 이용하여 표현한다. 이 표기법은 <a href="/wiki/%EC%9C%A0%EB%8B%89%EC%8A%A4_%EA%B3%84%EC%97%B4" title="유닉스 계열">유닉스 계열</a>의 운영 체제에서 표준이며, <a href="/wiki/C_(%ED%94%84%EB%A1%9C%EA%B7%B8%EB%9E%98%EB%B0%8D_%EC%96%B8%EC%96%B4)" title="C (프로그래밍 언어)">C</a>가 널리 이용되기 때문에 다른 대부분의 운영 체제에서 작동하는 프로그램에도 영향을 준다. 대부분의 32비트 시스템에서 초 시간을 저장하는 데 이용되는 <tt>time_t</tt> 자료 형식은 부호 있는 <a href="/wiki/32%EB%B9%84%ED%8A%B8" title="32비트">32비트</a> <a href="/wiki/%EC%A0%95%EC%88%98%ED%98%95" title="정수형">정수형</a>이다. POSIX 표준에 따르면, 이 형식을 이용하여 나타낼 수 있는 최후의 시각은 1970년 1월 1일 자정에서 정확히 <a href="/wiki/2147483647" title="2147483647">2147483647</a>초가 지난 <a href="/wiki/2038%EB%85%84" title="2038년">2038년</a> <a href="/wiki/1%EC%9B%94_19%EC%9D%BC" title="1월 19일">1월 19일</a> <a href="/wiki/%ED%99%94%EC%9A%94%EC%9D%BC" title="화요일">화요일</a> 03:14:07 <a class="mw-redirect" href="/wiki/UTC" title="UTC">UTC</a>이다. 이 시각 이후의 시각은 범위를 초과하여 내부적으로 음수로 표현되며, 프로그램의 이상 작동을 유발하는데, 왜냐하면 이러한 값은 2038년 대신 프로그램의 구현 방법에 따라 1970년 또는 1901년을 가리키기 때문이다. 따라서 계산 오류와 결정 오류가 발생할 수 있다.</p>

<h2><span class="mw-headline" id=".ED.95.B4.EA.B2.B0.EC.B1.85">해결책</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=2038%EB%85%84_%EB%AC%B8%EC%A0%9C&amp;action=edit&amp;section=1" title="부분 편집: 해결책">편집</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>현재의 <a href="/wiki/%EC%A4%91%EC%95%99_%EC%B2%98%EB%A6%AC_%EC%9E%A5%EC%B9%98" title="중앙 처리 장치">CPU</a>와 <a href="/wiki/%EC%9A%B4%EC%98%81_%EC%B2%B4%EC%A0%9C" title="운영 체제">OS</a> 조합에서 이 문제의 간단한 해결책은 없다. 표현 범위를 확장하려 32비트 대신 64비트를 이용하도록 <tt>time_t</tt>의 정의를 변경할 수 있지만, 이렇게 할 경우 소프트웨어, 저장 장치, 그리고 일반적으로는 시간의 이진 표현을 이용하는 모든 곳에서 바이너리 수준의 호환성에 이상이 생길 수 있다. <tt>time_t</tt>를 부호 없는 32비트 정수형으로 바꿀 경우 <a class="mw-redirect" href="/wiki/2106%EB%85%84" title="2106년">2106년</a> <a href="/wiki/2%EC%9B%94_7%EC%9D%BC" title="2월 7일">2월 7일</a> 06:28:15 UTC까지는 다룰 수 있게 되지만, 프로그램이 시간 차이를 다루거나 상대적인 시간을 표현할 때 음수를 다룰 수 없어 문제가 생긴다.</p>
<p><a href="/wiki/64%EB%B9%84%ED%8A%B8" title="64비트">64비트</a> 아키텍처를 이용하는 대부분의 운영 체제는 이미 <tt>time_t</tt>에 64비트 정수를 이용하고 있다. 이미 이러한 아키텍처로의 이전이 진행 중이며, 많은 사람들은 이것이 2038년 전에 완료될 것으로 예상하고 있다. 하지만 2012년 기준으로 수억대의 32비트 시스템이 있으며, 많은 수가 <a href="/wiki/%EC%9E%84%EB%B2%A0%EB%94%94%EB%93%9C_%EC%8B%9C%EC%8A%A4%ED%85%9C" title="임베디드 시스템">임베디드 시스템</a>에 내장되어 있는데, 이 모두가 2038년까지 교체될 수 있을지는 불분명하다. 또한 <tt>time_t</tt>는 <a class="mw-disambig" href="/wiki/ZIP" title="ZIP">ZIP</a> 파일 포맷을 비롯한 많은 파일 포맷들에 사용되고 있다.</p>
<p>64비트 정수형 체제 하에서는 이 문제를 약 3000억 년 정도 연기시킬 수 있다. 만약 그때까지 <a href="/wiki/%EA%B7%B8%EB%A0%88%EA%B3%A0%EB%A6%AC%EB%A0%A5" title="그레고리력">그레고리력</a>이 단 한번도 개정되지 않고 쓰이고 있다면, 정확히 2922억 7702만 6596년 12월 4일 일요일 15:30:08 UTC 이후의 시각을 다루려 할 때 문제가 발생한다.<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> 그러나 <a href="/wiki/%ED%83%9C%EC%96%91" title="태양">태양</a>의 수명은 약 123억 년 정도로 알려져 있으므로 지구상의 컴퓨터에서는 문제가 발생하지 않는다.</p>
<h2><span class="mw-headline" id=".EC.95.8C.EB.A0.A4.EC.A7.84_.EB.AC.B8.EC.A0.9C">알려진 문제</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=2038%EB%85%84_%EB%AC%B8%EC%A0%9C&amp;action=edit&amp;section=2" title="부분 편집: 알려진 문제">편집</a><span class="mw-editsection-bracket">]</span></span></h2>
<p><a href="/wiki/2006%EB%85%84" title="2006년">2006년</a> 5월에 발생한 <a class="extiw" href="https://en.wikipedia.org/wiki/AOLServer" title="en:AOLServer">AOLserver</a> <a href="/wiki/%EC%86%8C%ED%94%84%ED%8A%B8%EC%9B%A8%EC%96%B4" title="소프트웨어">소프트웨어</a>의 문제는 최초로 나타난 2038년 문제의 예로 알려져 있다. 소프트웨어 설정에서 데이터베이스 연결의 지속 시간을 10억 초로 지정하여 ‘절대로 연결이 끊기지 않음’을 나타내는 경우가 있었는데, 2006년 5월 13일 01:27:28 <a class="mw-redirect" href="/wiki/UTC" title="UTC">UTC</a>에서 10억 초 지난 시간은 <tt>time_t</tt>로 표시할 수 없었고, 따라서 이 시점부터 지속 시간이 과거의 시간으로 계산되어 소프트웨어에 문제를 일으켰다.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup></p>
