<p><b>포터블 네트워크 그래픽스</b>(Portable Network Graphics; PNG)는 <a href="/wiki/%EB%B9%84%EC%86%90%EC%8B%A4_%EC%95%95%EC%B6%95" title="비손실 압축">비손실</a> <a class="mw-redirect" href="/wiki/%EA%B7%B8%EB%9E%98%ED%94%BD_%ED%8C%8C%EC%9D%BC_%ED%8F%AC%EB%A7%B7" title="그래픽 파일 포맷">그래픽 파일 포맷</a>의 하나이다. <a href="/wiki/%ED%8A%B9%ED%97%88" title="특허">특허</a> 문제가 얽힌 <a href="/wiki/GIF" title="GIF">GIF</a> 포맷의 문제를 해결하고 개선하기 위해서 고안되었다. PNG는 공식적으로는 "핑"(<span class="IPA" title="IPA 발음">/pɪŋ/</span>)이라고 읽지만, 대부분은 "피엔지"라고 영어 철자 그대로 읽는다.</p>
<p>PNG 포맷은 <a href="/wiki/%ED%8C%94%EB%A0%88%ED%8A%B8_(%EC%BB%B4%ED%93%A8%ED%8C%85)" title="팔레트 (컴퓨팅)">컬러 팔레트</a> 화상과 <a href="/wiki/%EA%B7%B8%EB%A0%88%EC%9D%B4%EC%8A%A4%EC%BC%80%EC%9D%BC" title="그레이스케일">그레이스케일</a> 화상, 그리고 풀 컬러 화상 방식을 모두 지원한다. 그러나 <a href="/wiki/%EC%9D%B8%ED%84%B0%EB%84%B7" title="인터넷">인터넷</a> 상의 이미지 표시를 염두에 두고 개발되었기 때문에 <a class="mw-redirect" href="/wiki/CMYK" title="CMYK">CMYK</a> 등의 <a href="/wiki/%EC%83%89_%EA%B3%B5%EA%B0%84" title="색 공간">색 공간</a>은 지원하지 않는다.</p>
<p>파일 확장자는 <code>PNG</code> 또는 <code>png</code>를 쓰며, MIME 타입은 <code>image/png</code>으로 적는다.</p>

<h2><span class="mw-headline" id=".EC.97.AD.EC.82.AC">역사</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=PNG&amp;action=edit&amp;section=1" title="부분 편집: 역사">편집</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>PNG 포맷을 만들게 된 배경은 <a href="/wiki/1995%EB%85%84" title="1995년">1995년</a>, <a href="/wiki/%EC%9C%A0%EB%8B%88%EC%8B%9C%EC%8A%A4" title="유니시스">유니시스</a> 사가 <a href="/wiki/GIF" title="GIF">GIF</a>에 사용되는 <a href="/wiki/LZW" title="LZW">LZW</a> <a href="/wiki/%EB%8D%B0%EC%9D%B4%ED%84%B0_%EC%95%95%EC%B6%95" title="데이터 압축">데이터 압축</a> 알고리즘에 대해 <a href="/wiki/%EC%86%8C%ED%94%84%ED%8A%B8%EC%9B%A8%EC%96%B4_%ED%8A%B9%ED%97%88" title="소프트웨어 특허">소프트웨어 특허</a>를 적용할 것이라고 공고하면서이다. 이 알고리즘은 미국 특허 4,558,302번으로 등록되어 있고, 다른 여러 나라에도 등록되어 있다. 또한 256 색만을 저장할 수 있는 GIF는 한계가 있으므로 컴퓨터 성능이 좋아지면서 문제가 되어 왔다. <a href="/wiki/1999%EB%85%84" title="1999년">1999년</a> 8월, 유니시스가 <a href="/wiki/%EC%9E%90%EC%9C%A0_%EC%86%8C%ED%94%84%ED%8A%B8%EC%9B%A8%EC%96%B4" title="자유 소프트웨어">자유 소프트웨어</a>(프리웨어)와 비상업 소프트웨어(Non-Commercial License Software, FMOD)에 대한 무료 특허 정책을 거둬들이면서 PNG는 인기를 끌기 시작했다.</p>
<ul>
<li>버전 1.0: <a href="/wiki/1996%EB%85%84" title="1996년">1996년</a> <a href="/wiki/7%EC%9B%94_1%EC%9D%BC" title="7월 1일">7월 1일</a> 발표되었고, <a href="/wiki/W3C" title="W3C">W3C</a>에서도 <a href="/wiki/1996%EB%85%84" title="1996년">1996년</a> <a href="/wiki/10%EC%9B%94_1%EC%9D%BC" title="10월 1일">10월 1일</a> 표준으로 지정했다.</li>
<li>버전 1.1: <a href="/wiki/1998%EB%85%84" title="1998년">1998년</a> <a href="/wiki/12%EC%9B%94_31%EC%9D%BC" title="12월 31일">12월 31일</a></li>
<li>버전 1.2: <a href="/wiki/1999%EB%85%84" title="1999년">1999년</a> <a href="/wiki/8%EC%9B%94_11%EC%9D%BC" title="8월 11일">8월 11일</a></li>
<li>PNG는 국제 표준(<a class="mw-redirect" href="/wiki/%EA%B5%AD%EC%A0%9C%ED%91%9C%EC%A4%80%ED%99%94%EA%B8%B0%EA%B5%AC" title="국제표준화기구">ISO</a>/IEC 15948:2003)과 W3C 표준으로 <a href="/wiki/2003%EB%85%84" title="2003년">2003년</a> <a href="/wiki/11%EC%9B%94_10%EC%9D%BC" title="11월 10일">11월 10일</a> 발표되었다. 이 버전은 1.2판과 약간의 차이만이 있다.</li>
</ul>
<h2><span class="mw-headline" id=".EA.B8.B0.EC.88.A0_.EA.B0.9C.EC.9A.94">기술 개요</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=PNG&amp;action=edit&amp;section=2" title="부분 편집: 기술 개요">편집</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id=".ED.8C.8C.EC.9D.BC_.ED.97.A4.EB.8D.94">파일 헤더</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=PNG&amp;action=edit&amp;section=3" title="부분 편집: 파일 헤더">편집</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>PNG 파일은 8바이트의 신호로 시작한다.<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup></p>

<td>통신 등에서 8비트 데이터를 지원하지 않는 시스템을 찾거나 텍스트 파일과의 구분을 위해 사용된다.</td>

</tr>

</tr>
<tr>

<h3><span class="mw-headline" id=".ED.8C.8C.EC.9D.BC.EC.9D.98_.EC.B2.AD.ED.81.AC">파일의 청크</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=PNG&amp;action=edit&amp;section=4" title="부분 편집: 파일의 청크">편집</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>헤더 뒤에는 이미지에 대한 정보를 담고 있는 일련의 청크(chunk)가 온다. 청크는 그 자신을 <i>중요</i> 또는 <i>보조</i>로 선언한다. 프로그램이 받아들이지 못하는 <i>보조</i> 청크는 그냥 무시된다. 이런 식으로 청크를 통한 계층 구조는 <a href="/wiki/%EB%94%94%EC%A7%80%ED%84%B8_%EC%BB%A8%ED%85%8C%EC%9D%B4%EB%84%88_%ED%8F%AC%EB%A7%B7" title="디지털 컨테이너 포맷">디지털 컨테이너 포맷</a>과 같은 개념으로, PNG 포맷이 구버전과 호환되면서 쉽게 확장할 수 있도록 해준다. 이와 같은 구조가 MNG, JNG, APNG 포맷에도 사용된다. 청크는 네 가지 부분으로 구성된다. 길이(4바이트), 청크 타입(또는 이름)(4바이트), 청크 데이터(길이 바이트), 그리고 <a class="mw-disambig" href="/wiki/CRC" title="CRC">순환 중복 검사</a>(순환 중복검사/체크섬, 4바이트). CRC는 길이를 제외한 청크 타입과 데이터로 생성된 network-byte-order CRC-32이다.</p>

<p>청크 타입은 ASCII 문자 네개로 구성된다. 대소문자는 구분된다. 이들의 대소문자 여부가 <a class="mw-redirect" href="/wiki/%EB%94%94%EC%BD%94%EB%8D%94" title="디코더">디코더</a>에게 청크를 인식해야 할지에 대한 정보를 준다. 첫번째 문자는 이 청크가 <i>중요</i>인지 <i>보조</i>인지 알려준다. 대문자일 경우 <i>중요</i>이고, 그렇지 않을 경우 <i>보조</i>이다. 중요 청크는 파일을 읽는데 필수적인 정보를 갖고 있다. 디코더가 해석할 수 없는 중요 청크를 받으면 파일 읽기를 중단하고 사용자에게 경고를 전달해야 한다. 두번째 문자는 청크가 "퍼블릭"(PNG파일 표준에 포함되어 있거나 특수목적 퍼블릭 청크의 레지스트리에 포함되어 있음)인지 "프라이빗"(표준이 아님)인지 판별한다. 대문자는 퍼블릭이고 소문자는 프라이빗이다. 이는 프라이빗과 퍼블릭 청크의 이름이 충돌하는 것을 막아준다.(단, 같은 이름의 프라이빗 청크는 충돌할 수 있다) 세번째 문자는 PNG 표준에 따라 대문자여야 한다. 미래의 확장을 위해 남겨둔 문자로, 디코더는 이 문자가 소문자일 경우 해석 못하는 것으로 처리해야 한다. 네번째 문자는 청크를 해석하지 못할 경우 복사가 안전한지 판별한다. 소문자일 경우 파일에 변경이 있어도 안전하게 복사될 수 있다. 대문자일 경우, 파일 변경이 중요 청크를 변경하지 않았을 경우에만 복사될 수 있다.</p>
<h3><span class="mw-headline" id=".EC.A4.91.EC.9A.94_.EC.B2.AD.ED.81.AC">중요 청크</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=PNG&amp;action=edit&amp;section=5" title="부분 편집: 중요 청크">편집</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>디코더는 PNG 파일을 읽고 렌더링하기 위해서 중요 청크를 해석할 수 있어야 한다.</p>
<ul>
<li><code>IHDR</code>는 첫번째 청크로 와야 한다. 이것은 순서대로 이미지의 넓이, 높이, 비트 수와 컬러 타입을 표시한다.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup></li>
<li><code>PLTE</code>는 <a href="/wiki/%ED%8C%94%EB%A0%88%ED%8A%B8_(%EC%BB%B4%ED%93%A8%ED%8C%85)" title="팔레트 (컴퓨팅)">팔레트 (컴퓨팅)</a>, 즉 색공간을 표시한다.</li>
<li><code>IDAT</code>는 여러 개의 IDAT 청크로 쪼개질 수 있는 이미지를 표시한다. 파일 사이즈가 약간 커지긴 하지만 PNG를 스트리밍 방식으로 전달할 수 있게 만든다. IDAT는 압축 알고리즘의 출력 스트림을 통한 실제 이미지 파일을 갖고 있다.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup></li>
<li><code>IEND</code>는 이미지의 끝을 표시한다.</li>
</ul>
<p><code>PLTE</code>는 컬러 타입 3(인덱스드 컬러, 설정된 색만을 표시한다)에는 필수적이다. 컬러 타입 2와 6(트루 컬러와 트루컬러 + 알파 채널)에는 선택사항이다 그리고 컬러 타입 0와 4(그레이스케일과 그레이스케일 + 알파 채널)에는 나타내서는 안된다.</p>
<h2><span class="mw-headline" id="GIF.EC.99.80.EC.9D.98_.EB.B9.84.EA.B5.90">GIF와의 비교</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=PNG&amp;action=edit&amp;section=6" title="부분 편집: GIF와의 비교">편집</a><span class="mw-editsection-bracket">]</span></span></h2>
<ul>
<li>대부분의 경우 PNG는 <a href="/wiki/GIF" title="GIF">GIF</a>보다 압축률이 더 높다.</li>
<li>GIF의 단색 투명층과 달리 PNG는 8비트 알파 채널을 이용한 다양한 투명층을 지원한다.</li>
<li>256색까지 지원하는 GIF와 달리 PNG는 <a class="mw-redirect" href="/wiki/%ED%8A%B8%EB%A3%A8_%EC%BB%AC%EB%9F%AC" title="트루 컬러">트루 컬러</a>를 지원한다.</li>
<li>GIF에서는 제공되는 애니메이션을 PNG는 지원하지 않는다. (대안으로 PNG에 기반한 <a href="/wiki/APNG" title="APNG">APNG</a>, <a class="new" href="/w/index.php?title=JNG&amp;action=edit&amp;redlink=1" title="JNG (없는 문서)">JNG</a>, <a href="/wiki/MNG" title="MNG">MNG</a>와 같은 파일 형식이 제안되었다.)</li>
</ul>
<h3><span class="mw-headline" id=".ED.8C.8C.EC.9D.BC_.ED.81.AC.EA.B8.B0">파일 크기</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=PNG&amp;action=edit&amp;section=7" title="부분 편집: 파일 크기">편집</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>PNG가 GIF보다 최신의 압축 알고리즘을 사용하지만, GIF보다 더 큰 파일을 만든다고 알고 있는 사람이 있다. 여기에는 몇 가지 까닭이 있는데,</p>
<ul>
<li>GIF는 256색만을 지원한다. 트루 컬러 그림을 PNG로 압축할 때는 원본의 색을 다 저장하는 반면, GIF로 저장할 때는 256 색으로 수를 줄인 다음에 저장한다. 만약 원본도 256색만을 사용한다면 이런 차이는 나오지 않는다.</li>
<li>PNG 파일 형식에는 <a href="/wiki/%EB%A9%94%ED%83%80%EB%8D%B0%EC%9D%B4%ED%84%B0" title="메타데이터">메타데이터</a>가 추가로 붙어 있는 경우가 있다. (어도비 사의 파이어웍스 등).</li>
<li>어도비 포토샵의 일부 옛 버전에서는 PNG 압축 알고리즘을 잘 구현해 내지 못해서 큰 파일을 만들곤 했다.</li>
</ul>
<p>PNG 파일의 크기를 줄이는 <a class="external text" href="http://optipng.sourceforge.net/" rel="nofollow">OptiPNG</a>나 <a class="external text" href="http://pmt.sourceforge.net/pngcrush/" rel="nofollow">pngcrush</a>와 같은 <a href="/wiki/%EC%98%A4%ED%94%88_%EC%86%8C%EC%8A%A4" title="오픈 소스">오픈 소스</a>로 <a href="/wiki/MS-DOS" title="MS-DOS">MS-DOS</a>에서 <a href="/wiki/%EC%9C%A0%EB%8B%89%EC%8A%A4" title="유닉스">유닉스</a>나 <a href="/wiki/%EB%A6%AC%EB%88%85%EC%8A%A4" title="리눅스">리눅스</a> 등의 다양한 환경을 지원하여 제공하고 있다.</p>
<h2><span class="mw-headline" id="JPEG.EC.99.80.EC.9D.98_.EB.B9.84.EA.B5.90">JPEG와의 비교</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=PNG&amp;action=edit&amp;section=8" title="부분 편집: JPEG와의 비교">편집</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="thumb tright">
<div class="thumbinner" style="width:202px;"><a class="image" href="/wiki/%ED%8C%8C%EC%9D%BC:Comparison_of_JPEG_and_PNG.png"><img alt="" class="thumbimage" data-file-height="200" data-file-width="200" height="200" src="//upload.wikimedia.org/wikipedia/commons/a/a4/Comparison_of_JPEG_and_PNG.png" width="200"/></a>
<div class="thumbcaption">
<div class="magnify"><a class="internal" href="/wiki/%ED%8C%8C%EC%9D%BC:Comparison_of_JPEG_and_PNG.png" title="실제 크기로"></a></div>
왼쪽의 JPEG 파일에 비해 오른쪽의 PNG 파일이 훨씬 더 깔끔하다.</div>
</div>
</div>
<p>사진과 같은 이미지에 대해서는, <a href="/wiki/JPEG" title="JPEG">JPEG</a>가 사진에 특화된 <a href="/wiki/%EC%86%90%EC%8B%A4_%EC%95%95%EC%B6%95" title="손실 압축">손실 압축</a> 알고리즘을 사용하므로 PNG에 비해 더 작은 파일을 만들 수 있다. <span class="fix" title="이 부분은 출처가 필요합니다.">경우에 따라 5-10배 차이를 보이기도 한다.</span><sup class="정리_필요 noprint">[<a href="/wiki/%EC%9C%84%ED%82%A4%EB%B0%B1%EA%B3%BC:%EC%B6%9C%EC%B2%98_%EB%B0%9D%ED%9E%88%EA%B8%B0" title="위키백과:출처 밝히기">출처 필요</a>]</sup> 하지만 JPEG 압축은 양자화의 영향으로, 바라지 않던 잡티가 낄 수 있다. 문자나 날카로운 경계가 있는 그림은, JPEG에서 생기기 쉬운 뭉개짐 없이 JPEG보다 압축을 더 잘 할 수 있는 PNG를 쓰는 것이 더 낫다.</p>
<p>또한, PNG는 비손실 압축이므로, 나중에 고화질의 재편집을 해야 한다면 PNG로 저장해 놓는 것이 낫다. JPEG를 사용할 때는 저장을 하면 할수록 계속 손실이 누적될 수 있다.</p>
